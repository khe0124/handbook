풀스택 성장 로드맵

풀스택 역량은 프론트엔드와 백엔드 문법을 모두 아는 것이 아니라 사용자 행동, API 계약, 데이터 정합성, 배포, 운영 신호를 하나의 변경 흐름으로 연결해 설명하고 실행하는 능력입니다.

TARGET : 프론트엔드 경험을 기반으로 백엔드·인프라·운영까지 확장하는 개발자
STANCE : 읽는 순서 · 실전 과제 · 품질 기준을 함께 제공
FS-01

목적과 기준

이 핸드북의 목적은 지식 목록을 넓히는 것이 아니라, 하나의 기능을 제품 경험에서 운영까지 끝까지 책임지는 판단 체계를 만드는 것입니다.

관점단편 지식풀스택 판단
기능 구현화면과 API를 각각 만든다사용자 흐름, API 계약, 데이터 상태 전이를 함께 설계한다
데이터CRUD가 동작한다트랜잭션, 동시성, 마이그레이션, 감사 추적을 고려한다
배포서버에 올린다빌드 산출물, 설정, secret, rollback, health check를 설계한다
운영오류가 나면 로그를 본다로그·메트릭·트레이스·알림으로 실패 위치와 영향 범위를 좁힌다
제품 품질요구사항대로 만든다접근성, 성능, 보안, 사용성, 회복 경로까지 검증한다
핵심 전환

풀스택 학습의 중심 질문은 “무엇을 더 배울까”가 아니라 “이 변경이 사용자에게 도달할 때 어떤 계약, 상태, 장애 모드를 만드는가”입니다. 이 질문을 기준으로 각 핸드북을 연결해 읽습니다.

FS-02

역량 지도

풀스택 역량은 여섯 축으로 나눠 익히면 누락이 줄어듭니다. 각 축은 독립 과목이 아니라 하나의 변경 안에서 동시에 만납니다.

축핵심 질문연결 문서
제품·디자인사용자가 어떤 목표를 어떤 순서로 달성하는가디자인 개요, 사용자 흐름, 폼과 입력 경험
프론트엔드상태, 렌더링, 접근성, 에러 UX가 안정적인가프론트엔드 개요, 성능과 지표, 테스트 전략
API·백엔드계약, 권한, 실패, 멱등성, 동시성을 어떻게 다루는가백엔드 개요, 인증과 보안, 아키텍처 패턴
데이터정합성, 조회 성능, 스키마 변경을 어떻게 보장하는가DB, JPA, DB 스키마·시딩
인프라·DevOps빌드·배포·설정·네트워크가 재현 가능한가Docker와 배포, DevOps 개요, 네트워크 인프라
운영문제가 생겼을 때 감지·완화·복구가 가능한가관측 가능성, 로그·커스텀 명령, 실무 플레이북
FS-03

학습 순서

순서는 관심사가 아니라 의존 관계를 따릅니다. 먼저 한 기능을 끝까지 만들고, 그 다음 신뢰성과 운영성을 얹습니다.

제품 흐름→ API 계약→ 데이터 모델→ 구현·테스트→ 배포·운영
단계목표읽을 문서산출물
1. 흐름 정의사용자 목표와 화면 상태를 정한다디자인 개요, 사용자 흐름, 폼사용자 시나리오, 상태 전이표
2. 계약 설계API, 에러, 권한, 페이지네이션을 정한다백엔드 개요, 프론트엔드 API 연동OpenAPI 초안, error code 표
3. 데이터 설계정합성과 조회 패턴을 맞춘다DB, JPA, 마이그레이션ERD, migration, index 후보
4. 구현 검증프론트·백엔드 테스트를 나눈다프론트엔드 테스트, 백엔드 테스트unit, integration, E2E 최소 세트
5. 배포 준비설정·secret·이미지·파이프라인을 분리한다Docker와 배포, CI/CD, 설정 관리Dockerfile, compose, pipeline
6. 운영 준비로그·메트릭·알림·롤백을 확인한다관측 가능성, DevOps, 실무 플레이북health check, dashboard, runbook
FS-04

경계면 체크리스트

풀스택 실무의 대부분의 문제는 기술 내부보다 경계면에서 납니다. 화면과 서버, 서버와 DB, 앱과 인프라 사이의 계약을 먼저 봅니다.

경계확인 질문흔한 실패
UI ↔ APIloading, empty, error, permission denied가 모두 정의됐는가500만 처리하고 400/401/403/409 UX가 없음
API ↔ DB트랜잭션 경계와 unique 제약이 비즈니스 규칙을 보호하는가프론트 중복 방지만 믿어 중복 데이터 생성
Auth ↔ Browser쿠키, SameSite, CORS, CSRF, 토큰 만료가 일관적인가로컬은 되지만 배포 후 쿠키가 안 붙음
Build ↔ Runtime같은 이미지가 환경변수만 바꿔 여러 환경에서 도는가환경별로 다른 빌드를 만들어 재현성 붕괴
Service ↔ NetworkDNS, route, firewall, TLS, health check가 맞는가앱 문제로 보였지만 Security Group 차단
Change ↔ Operation배포 실패 시 rollback 또는 roll-forward 경로가 있는가마이그레이션과 코드가 한 번에 깨짐
정확도 기준

프론트엔드 검증, 백엔드 검증, DB 제약은 서로 대체하지 않습니다. UI 검증은 사용성을 높이고, 서버 검증은 보안과 규칙을 지키며, DB 제약은 마지막 정합성 방어선입니다. 같은 규칙이 여러 층에 반복되는 것은 중복이 아니라 방어 심도입니다.

FS-05

캡스톤 과제

읽은 내용을 몸에 붙이려면 작은 기능 하나를 끝까지 책임지는 과제가 필요합니다. 아래 과제를 반복하면 전체 연결이 보입니다.

과제포함해야 할 것검증 기준
관리자 목록·상세·수정검색, 필터, 페이지네이션, 수정 폼, 권한URL 상태, optimistic/pessimistic update 판단, 400/409 오류 처리
파일 업로드 기능용량 제한, MIME 검증, 진행률, 실패 재시도서버 검증, 저장소 분리, 업로드 중 이탈 처리
초대·권한 관리초대 발송, 만료, 재발송, 역할 변경 이력감사 로그, idempotency, 권한 테스트
배치 작업 상태판작업 요청, 진행 상태, 실패 사유, 재실행polling/SSE 판단, 상태 전이, 운영 로그 연결
배포 파이프라인빌드, 테스트, 이미지 push, staging 배포환경변수 분리, secret 관리, rollback 문서
FS-06

품질 기준

풀스택 기능은 “동작한다”에서 끝나지 않습니다. 아래 기준을 PR 체크리스트처럼 사용합니다.

계약contract
요청·응답·에러·권한·페이지네이션이 문서화되어 있고, breaking change가 관리됩니다.
정합성consistency
트랜잭션, unique 제약, 멱등성, 동시성 충돌 처리가 설계되어 있습니다.
사용성usability
loading, empty, error, disabled, success 상태가 사용자 행동과 연결됩니다.
보안security
인증과 인가가 분리되어 있고, 서버가 모든 입력과 권한을 최종 검증합니다.
운영성operability
로그, 메트릭, traceId, health check, rollback/runbook이 준비되어 있습니다.
성능performance
초기 로드, API latency, 쿼리 실행계획, p95/p99 지연을 측정하고 병목을 설명합니다.
읽는 방법

각 영역 문서를 순서대로 완독하는 것보다, 하나의 기능을 잡고 이 기준을 통과시키는 방식이 빠릅니다. 부족한 기준이 보이면 해당 핸드북으로 들어가 깊이를 보강합니다.

FS-07

고급 개발자 루브릭

고급 개발자는 더 많은 기술명을 아는 사람이 아니라, 불확실한 조건에서 시스템을 안전하게 바꾸고 그 결정을 설명할 수 있는 사람입니다. 아래 기준은 각 장을 읽은 뒤 스스로 점검할 최소 루브릭입니다.

역량통과 기준증거
문제 정의요구사항, 제약, 비기능 목표, 모르는 것을 분리한다기능 명세, SLO, 데이터 규모, 권한 범위가 먼저 적힌다
설계 판단선택지와 포기한 것을 함께 설명한다ADR, trade-off table, rollback plan이 남는다
구현 품질해피패스와 실패 경로를 함께 구현한다400/401/403/409/5xx, timeout, retry, idempotency 테스트가 있다
운영 책임배포 후 문제를 감지하고 완화할 수 있다로그, 메트릭, traceId, dashboard, runbook, alert 기준이 있다
리뷰 역량코드 스타일보다 위험과 계약을 먼저 본다동시성, 보안, migration, 성능, 접근성, 장애 모드가 리뷰된다
학습 전이한 영역의 원리를 다른 영역에 옮겨 적용한다UI 상태와 DB 상태 전이, API 계약과 컴포넌트 API를 연결해 설명한다
핵심 기준

“이 기능을 만들 수 있는가”보다 “이 기능이 실패할 때 어디서 감지되고, 어떻게 제한되며, 어떻게 복구되는가”를 답할 수 있어야 합니다. 이 질문에 답하지 못하면 아직 운영 가능한 설계가 아닙니다.

FS-08

실전 훈련 루프

핸드북은 읽는 것만으로 완성되지 않습니다. 하나의 capstone을 선택해 설계, 구현, 운영, 회고까지 반복해야 지식이 판단으로 바뀝니다.

훈련할 일검증 질문
설계 리뷰ADR 1장을 작성하고 대안을 2개 이상 비교왜 이 구조가 현재 규모와 팀에 맞는가
성능 리허설목록/검색/API에 부하를 주고 p95, p99를 기록병목이 DB, 네트워크, 렌더링, GC 중 어디인가
장애 주입DB 지연, 외부 API timeout, queue 적체, chunk 404를 일부러 만든다사용자 영향이 제한되고 알림이 울리는가
보안 점검STRIDE 기준으로 신뢰 경계와 abuse case를 적는다인가 누락, XSS, CSRF, secret 노출을 어떻게 막는가
복구 훈련rollback, restore drill, hotfix 절차를 실제로 실행RTO/RPO 안에 복구 가능한가
포스트모템원인보다 재발 방지 액션을 남긴다같은 실패가 다시 와도 더 빨리 감지·완화되는가
시니어급 학습 방식

문서를 한 번에 끝까지 읽는 것보다, capstone 하나를 잡고 이 루프를 반복하세요. 같은 기능을 성능, 보안, 접근성, 배포, 장애 대응 관점으로 다시 보는 과정이 고급 개발자의 사고 훈련입니다.
