프론트엔드 인터랙션 ·
애니메이션과 3D

모션은 화면을 화려하게 만드는 장식이 아니라 상태 변화, 공간 관계, 사용자 행동의 결과를 설명하는 피드백입니다. 좋은 애니메이션은 덜 보이고, 실패한 애니메이션은 사용자의 시간을 빼앗습니다.

AUDIENCE : FRONTEND ENGINEERING · PRODUCT UI · DESIGN SYSTEM · UPDATED 2026-06
MOTION-01

모션의 역할과 원칙

모션의 1차 목적은 즐거움이 아니라 이해입니다. 사용자가 방금 누른 버튼이 어떤 상태를 만들었는지, 새 패널이 어디서 왔는지, 목록 항목이 왜 이동했는지 설명하지 못하는 모션은 대부분 제거하는 편이 낫습니다.

목적좋은 사용실패 모드숫자 가이드
상태 피드백hover, pressed, loading, success, error를 짧게 구분클릭 후 300ms 이상 아무 변화가 없어 중복 클릭 유발입력 피드백은 100ms 안에 시작
공간 관계drawer, popover, detail panel이 트리거와 연결되어 보임화면 중앙에서 갑자기 나타나 사용자가 위치를 잃음open 160-240ms, close 120-180ms
변화 위치 안내추가, 삭제, 정렬, 필터 결과가 어디서 바뀌었는지 드러냄layout shift를 부드럽게 포장해 원인을 숨김목록 재배치 180-300ms
주의 환기위험, 충돌, 미완료 필드를 한 번만 강조반복 흔들림, 깜빡임, 색상만으로 경고 전달강조는 1회, 400ms 이하
브랜드 감각제품의 속도감과 밀도를 반영한 작은 signature motion모든 화면 전환에 같은 과장 효과를 붙임핵심 flow 외 장식 모션은 예산 밖
판단 질문통과 기준탈락 신호
이 모션이 없어도 의미가 유지되는가의미는 텍스트, 구조, 상태로 유지되고 모션은 보조한다모션을 끄면 완료/오류/위치 변화를 알 수 없다
사용자 작업 속도보다 느리지 않은가반복 업무의 다음 입력을 막지 않는다닫기 애니메이션을 기다려야 다음 작업 가능
성능 예산 안에 있는가transform/opacity 중심, long task 없음, frame drop 제한height, top, box-shadow, filter가 큰 영역에서 반복
접근성 설정을 존중하는가prefers-reduced-motion에서 대체 상태가 존재패럴랙스, 자동 재생, 흔들림을 끌 수 없음
MOTION FIRST PRINCIPLES1. 모션은 사용자의 원인과 결과를 연결한다
2. 빠른 작업 흐름에서는 짧게, 학습이 필요한 변화에서는 명확하게 둔다
3. show보다 hide가 보통 더 짧다
4. layout을 움직이기보다 composite 가능한 속성을 우선한다
5. motion off 상태에서도 제품 의미와 조작 가능성이 유지되어야 한다
PRACTICAL RULE

애니메이션 duration을 정할 때 “예뻐 보이는가”보다 “사용자가 다음 행동을 하기 전에 필요한 정보를 받는가”를 먼저 봅니다. 80ms 이하는 변화가 거의 감지되지 않고, 500ms 이상은 반복 업무에서 지연으로 느껴지기 쉽습니다.

MOTION-02

CSS transition/keyframes

CSS 애니메이션은 UI 상태 전환의 기본값입니다. hover, focus, open/close, skeleton, 간단한 reveal처럼 DOM 상태와 스타일 변화가 일대일로 연결되는 경우에는 CSS가 가장 단순하고 안정적입니다.

도구적합한 상황피해야 할 상황실무 기준
transition상태 class, pseudo-class, 속성 변화가 명확함중간 단계 제어, sequence, interrupt가 복잡함버튼, 카드 hover, panel opacity/transform
@keyframes반복 로딩, skeleton, 정해진 entrance 효과사용자 입력에 따라 경로가 달라짐무한 반복은 reduced motion에서 정지 또는 단순화
animation시작과 끝이 자동이고 JS 상태가 필요 없음React state와 정확히 동기화해야 함완료 후 상태는 class로 확정
transition-behaviordisplay 같은 discrete 전환을 보완할 때브라우저 지원이 제품 기준과 맞지 않음progressive enhancement로만 사용
속성권장도이유주의
transform높음layout을 다시 계산하지 않고 composite 단계에서 처리 가능텍스트가 흐려지거나 hit area와 시각 위치가 어긋나지 않게 확인
opacity높음fade, overlay, entrance에 안정적0이어도 focusable/clickable일 수 있어 visibility와 pointer-events 관리
height, width낮음layout을 유발해 주변 요소까지 영향을 줌accordion은 grid row, max-height, FLIP, measured animation 검토
top, left낮음positioned element라도 layout/paint 비용이 커질 수 있음대부분 translate로 대체
box-shadow, filter제한큰 영역 repaint와 GPU 비용 증가작은 요소에만 사용하고 Performance trace 확인
CSS MOTION TOKEN 예시:root {
  --motion-fast: 120ms;
  --motion-base: 180ms;
  --motion-slow: 280ms;
  --ease-standard: cubic-bezier(0.2, 0, 0, 1);
  --ease-enter: cubic-bezier(0, 0, 0.2, 1);
  --ease-exit: cubic-bezier(0.4, 0, 1, 1);
}
TRANSITION 예시.panel {
  opacity: 0;
  transform: translateY(8px);
  transition: opacity var(--motion-base) var(--ease-standard), transform var(--motion-base) var(--ease-standard);
}
.panel[data-open="true"] {
  opacity: 1;
  transform: translateY(0);
}
COMMON FAILURE

transition: all은 편하지만 위험합니다. 나중에 width, color, shadow, filter가 추가되면 의도하지 않은 속성까지 움직여 성능과 예측 가능성을 망칩니다. 움직일 속성은 명시합니다.

MOTION-03

WAAPI와 JS animation

JS 애니메이션은 “CSS보다 고급”이라서 쓰는 것이 아닙니다. 애니메이션의 시작, 중단, 재생 방향, 사용자 입력, Promise 흐름, 동적 측정값이 제품 상태와 연결될 때 사용합니다. 그렇지 않다면 CSS가 더 적은 코드와 낮은 위험을 줍니다.

방식좋은 사용실패 모드검증 포인트
WAAPI element.animate()JS에서 play/cancel/reverse/finish를 제어하고 싶음React unmount 뒤 애니메이션이 남아 memory leak 또는 flicker 발생cleanup에서 animation.cancel()
requestAnimationFramescroll, pointer, canvas처럼 frame별 계산이 필요함state set을 매 frame 호출해 React render 폭발DOM read/write 분리, ref 기반 업데이트
Timer 기반단순 delay, staged feedbacktab background, throttling, unmount 뒤 setStatetimer cleanup과 visibility 상태
Physics 직접 구현도메인 제약이 단순하고 dependency를 피해야 함spring 튜닝, interrupt, accessibility를 매번 새로 품라이브러리 대비 유지보수 비용
문제JS가 필요한 신호CSS로 충분한 신호
Accordion내용 높이를 측정해 정확히 접고 펼쳐야 하며 중간 중단이 필요고정 높이 또는 opacity/transform reveal만 필요
Dragpointer 위치, velocity, threshold, snap point가 필요hover/focus/pressed 상태만 필요
Route transition이전 route exit 완료 후 다음 route enter를 조율단순 fade in만 필요
Chart데이터 값 변화에 따라 path, axis, tooltip을 보간legend hover나 point highlight 수준
WAAPI CLEANUP 예시const animation = node.animate(
  [{ opacity: 0, transform: "translateY(8px)" }, { opacity: 1, transform: "translateY(0)" }],
  { duration: 180, easing: "cubic-bezier(0.2, 0, 0, 1)", fill: "both" }
);

return () => animation.cancel();
FRAME BUDGET

60fps에서 한 frame 예산은 약 16.7ms입니다. 이 안에 JS, style, layout, paint, composite가 모두 들어와야 합니다. pointermove마다 React state를 바꾸거나 DOM read/write를 섞으면 금방 예산을 넘습니다.

JS ANIMATION REVIEWanimation lifecycle이 component lifecycle과 연결되어 있는가
unmount, route change, rapid toggle에서 cancel되는가
prefers-reduced-motion일 때 즉시 완료 또는 단순 전환되는가
동일 interaction을 10회 반복해도 memory와 DOM node가 증가하지 않는가
Performance trace에서 long task와 forced reflow가 없는가
MOTION-04

Framer Motion/GSAP/Lottie 선택 기준

애니메이션 라이브러리 선택은 취향 문제가 아니라 제품 표면, 팀 역량, 번들 예산, 디자이너 산출물, 상호작용 복잡도의 문제입니다. 간단한 상태 전환에 대형 라이브러리를 넣는 것도 문제고, 복잡한 timeline을 CSS로 억지로 관리하는 것도 문제입니다.

선택지강점좋은 사용주의
CSS only가볍고 예측 가능, 디자인 토큰과 결합 쉬움대부분의 상태 전환, hover, panel, skeleton복잡한 sequencing과 interrupt가 늘면 class 상태가 지저분해짐
WAAPI브라우저 기본 API, imperative control 가능작은 JS 제어 animation, 라이브러리 없이 cancel/reverse 필요팀 표준 wrapper 없이 흩어지면 패턴이 갈라짐
Framer MotionReact 친화, layout animation, presence, gestureReact 앱의 modal, route, card reorder, drag, spring 기반 UIServer Component 경계, bundle, layout animation 비용 확인
GSAP강력한 timeline, scroll, SVG, 복잡한 sequence마케팅 페이지, 데이터 스토리텔링, 고밀도 timeline제품 CRUD UI에 남발하면 코드 소유권이 어려워짐
LottieAfter Effects 산출물 재생, 브랜드 일러스트 전달 쉬움empty state, success, onboarding, 설명용 루프상호작용 로직에는 부적합. JSON 크기와 렌더러 성능 확인
Three.js/R3F실제 3D scene, camera, material, interaction제품 3D preview, configurator, spatial demo, game-like UI접근성 대체 UI, GPU 비용, asset pipeline 필요
상황추천이유검토 질문
button, tooltip, drawerCSS상태 기반 전환으로 충분하고 비용이 낮음토큰으로 duration/easing이 통일됐는가
modal enter/exit가 unmount와 충돌Framer Motion 또는 작은 presence wrapperexit 완료 전 DOM 유지가 필요focus trap과 scroll lock 수명주기가 맞는가
카드 reordering, shared layoutFramer Motionlayout 변화 보간과 interrupt 처리 부담 감소목록이 커질 때 layout 측정 비용은 감당 가능한가
스크롤 기반 긴 브랜드 페이지GSAPtimeline과 scroll trigger 관리가 핵심reduced motion과 모바일 성능 대체안이 있는가
디자이너가 AE로 만든 성공 애니메이션Lottie벡터 산출물을 재현하기 쉬움용량, loop 여부, 의미 전달 대체 텍스트가 있는가
3D 제품 커스터마이저Three.js 또는 React Three Fibercamera, material, lighting, hit testing이 필요저사양 기기 fallback과 asset lazy loading이 있는가
DEPENDENCY GATE

새 모션 라이브러리는 “할 수 있다”가 아니라 “제품이 이 복잡도를 지속적으로 필요로 한다”는 증거가 있을 때 도입합니다. PR에는 bundle delta, 사용 surface, reduced motion 대응, 소유 팀을 함께 남깁니다.

LIBRARY DECISION LOGProblem: route exit animation과 modal presence가 반복 구현되고 있음
Options: CSS timeout wrapper / WAAPI wrapper / Framer Motion
Decision: React surface가 많고 layout/presence 요구가 반복되므로 Framer Motion 도입
Guardrail: shared primitives에만 사용, page별 임의 timeline 금지, reduced motion variant 필수
Review: bundle analyzer와 핵심 interaction trace를 PR에 첨부
MOTION-05

Motion system

모션 시스템은 개별 애니메이션 모음이 아니라 제품 전체의 시간, easing, 거리, 상태 언어를 표준화한 계약입니다. 토큰이 없으면 화면마다 다른 속도감이 생기고, 토큰만 있고 사용 기준이 없으면 장식 모음이 됩니다.

토큰예시 값사용처주의
Duration fast80-120mshover, pressed, small feedback의미 있는 변화 설명에는 너무 짧을 수 있음
Duration base160-220mspopover, dropdown, small panel반복 업무 기본값으로 적합
Duration slow260-360mspage section, drawer, 큰 layout 변화닫기나 취소에는 짧은 값을 별도 적용
Distance xs2-4pxbutton press, focus emphasis작아도 layout shift처럼 보이면 실패
Distance sm8-12pxmenu, toast, panel reveal모바일에서 과장되어 보이지 않는지 확인
Easing standardcubic-bezier(0.2, 0, 0, 1)일반 상태 전환brand easing을 모든 곳에 강제하지 않음
Easing exitcubic-bezier(0.4, 0, 1, 1)사라지는 요소사용자가 기다릴 이유가 없으면 빠르게
컴포넌트motion contract금지 패턴QA 기준
Buttonpress 80ms, loading 즉시 표시, success 120ms클릭 후 spinner 표시 지연, bounce키보드 Space/Enter에서도 동일 피드백
Toastenter 160ms, exit 120ms, 자동 소멸 4-6s필수 오류를 자동 소멸만으로 처리hover/focus 시 pause 또는 충분한 지속 시간
Modaloverlay fade + dialog scale/translate, focus 이동dialog만 움직이고 focus trap 지연Escape, backdrop, rapid open/close 테스트
Drawer방향성 있는 translate, close는 open보다 짧게width 애니메이션으로 layout 밀기스크롤 잠금과 animation lifecycle 일치
List iteminsert/delete/reorder에 위치 변화 표시정렬 때 전체 리스트 fade out100개 이상에서 frame drop 확인
MOTION SYSTEM CHECKLISTduration, easing, distance, delay token이 있는가
enter, exit, emphasis, loading, reorder의 쓰임이 구분되는가
component별 motion contract가 문서화되어 있는가
reduced motion variant가 token 수준에서 제공되는가
새 animation PR이 token 밖 임의 값을 만들지 않는가
디자인과 코드의 naming이 같은가
SYSTEM

모션 시스템의 성공 기준은 “멋진 animation demo”가 아니라 제품 팀이 새 컴포넌트를 만들 때 duration과 easing을 새로 고민하지 않아도 되는 상태입니다.

MOTION-06

고급 모션과 전환

고급 모션의 어려움은 효과 자체가 아니라 상태 전환, unmount, 빠른 반복 입력, route 변경, focus 복귀가 동시에 일어나는 데 있습니다. 이 장은 3D 구현이 아니라 제품 UI의 복잡한 전환을 안정적으로 제어하는 기준을 다룹니다. WebGL·Three.js 구현은 그래픽·3D 파트에서 별도로 다룹니다.

패턴사용 시점주요 위험검증
Presence transitionmodal, toast, drawer가 사라질 때 exit motion이 필요DOM이 너무 빨리 unmount되어 exit·focus restore가 깨짐빠른 열기/닫기, Esc, route 이동 중 닫기
Shared layout카드가 상세 화면으로 확장되거나 목록 순서가 바뀜layout 측정 비용, scroll 위치, focus 위치 혼란큰 목록, 모바일, reduced motion, back navigation
Gesture motiondrag, swipe, reorder, pull-to-refreshpointer capture, scroll 충돌, keyboard 대체 누락mouse/touch/keyboard 각각의 완료 경로
Route transition화면 간 공간 관계를 설명해야 할 때loading state, data fetch, scroll restoration과 충돌느린 네트워크, back/forward, 중복 navigation
Interrupted animation사용자가 animation 중 반대 행동을 할 수 있음중간 상태가 DOM·state와 불일치rapid toggle, double click, cancel, retry

    const dx = prev.left - last.left;
    const dy = prev.top - last.top;

scroll container offset을 고려보이는 위치와 DOMRect 차이를 보정스크롤된 목록에서 이동 방향이 틀림
reduced motion에서는 위치 전환을 제거큰 이동을 피하면서 상태 의미를 유지모션 축소 설정에서도 카드가 날아다님
ANIMATION CONTROLLER CONTRACTstart(): 현재 animation을 cancel하고 새 animation 시작
cancel(): 시각 상태와 제품 상태를 마지막 확정 상태로 복구
finish(): DOM style을 최종 상태로 고정하고 임시 style 제거
reduced(): motion off 또는 단순 fade 경로 제공
cleanup(): unmount, route leave, hidden tab에서 listener와 animation 제거
observe(): duration, cancelled, interrupted, long frame을 측정
CODE REVIEW RULE

모션 PR에서 가장 먼저 볼 것은 easing 값이 아니라 cleanup입니다. 빠른 토글, route 이동, unmount, reduced motion에서 남는 animation이나 inline style이 없어야 합니다.

useReducedMotion 예시function useReducedMotion() {
  const [reduce, setReduce] = useState(false);
  useEffect(() => {
    const query = window.matchMedia("(prefers-reduced-motion: reduce)");
    setReduce(query.matches);
    const onChange = () => setReduce(query.matches);

MOTION-15

근거 기반 모션 예산

모션은 감각 품질이지만 운영 기준은 숫자로 관리해야 합니다. duration, frame budget, input delay, layer 수, reduced motion 경로를 함께 보지 않으면 “보기에는 부드러운” 효과가 실제로는 입력 지연, 배터리 소모, 접근성 저하를 만들 수 있습니다.

항목실질 숫자왜 중요한가부수효과검증 방법

MOTION-16

자주 봉착하는 문제와 해결

모션 버그는 “애니메이션이 이상하다”로 보이지만 실제 원인은 layout 속성, animation 수명, focus trap, layer 비용, reduced motion 누락인 경우가 많습니다. 해결은 easing을 바꾸는 것이 아니라 어떤 속성을 언제 읽고 쓰며, 취소와 종료를 어떻게 처리하는지 명확히 하는 것입니다.
