사용자 흐름과
태스크 설계

사용자 흐름은 화면 목록이 아니라 목표 달성을 위한 행동의 순서입니다. 좋은 흐름은 다음 행동을 예측 가능하게 하고, 예외 상황에서도 사용자가 길을 잃지 않게 합니다.

FLOW-01

Flow의 종류

구분설명예시
User flow사용자 목표 달성 경로상품 선택 → 장바구니 → 결제
Task flow특정 업무 수행 단계승인 요청 작성 → 검토자 선택 → 제출
System flow시스템 상태 전이pending → approved → completed
Service blueprint사용자 행동과 내부 운영 연결고객 요청과 운영 처리 흐름
FLOW-02

Happy path와 예외

흐름 설계는 정상 경로만 그리면 부족합니다. 실제 제품은 예외와 중단, 재개가 더 중요할 때가 많습니다.

상황설계 질문
입력 오류어디가 왜 잘못됐고 어떻게 고치는가
권한 없음왜 접근 불가이며 누구에게 요청해야 하는가
네트워크 실패재시도, 임시 저장, 데이터 손실 방지가 있는가
중도 이탈나중에 이어서 할 수 있는가
중복 제출idempotency와 버튼 상태가 설계되어 있는가
FLOW-03

상태 설계

상태필요한 UI놓치기 쉬운 점
Empty왜 비었는지, 무엇을 하면 되는지권한 없음과 데이터 없음 구분
Loadingskeleton, progress, disabled action무한 대기 방지
Error원인, 복구 행동, 문의 경로기술 에러 그대로 노출 금지
Success완료 확인과 다음 행동토스트만 띄우고 맥락 잃게 하지 않음
Dirty저장 전 변경 있음 표시이탈 경고와 자동 저장 기준
FLOW-04

마찰과 안전장치

모든 마찰이 나쁜 것은 아닙니다. 위험한 행동에는 의도적인 마찰이 필요하고, 반복 업무에는 마찰을 줄여야 합니다.

행동권장 패턴이유
삭제undo 또는 명확한 confirmation복구 가능성에 따라 선택
결제/제출요약 확인, disabled 중복 방지실수와 중복 처리 방지
반복 승인bulk action, keyboard shortcut업무 효율
위험 설정 변경영향 범위 표시, 감사 로그운영 리스크 관리
FLOW-05

체크리스트

Entry point진입
사용자가 어디서 이 흐름을 시작하는지 명확한가.
Exit point종료
완료, 취소, 실패, 중단 후 상태가 정의되어 있는가.
Recovery복구
오류와 이탈 후 다시 이어갈 수 있는가.
Feedback피드백
사용자 행동 뒤 시스템 반응이 즉시 보이는가.
FLOW-06

실패와 중복을 포함한 흐름

실무 플로우는 happy path보다 재시도, 중복 제출, 권한 변경, 장시간 작업의 복구가 더 중요합니다.

리스크설계 대응개발 계약
중복 제출제출 중 버튼 잠금, 결과 화면 명확화idempotency key, 중복 요청 응답 규칙
권한 변경작업 중 권한 만료 안내와 저장 보호403 응답 메시지, draft 보존
부분 성공성공/실패 항목 분리와 재시도 액션batch API 결과 구조
긴 처리작업 큐 상태, 알림, 다시 방문 경로job id, polling 또는 webhook 상태
Case study

대량 권한 변경 화면에서 “저장” 후 토스트만 보여주면 일부 실패를 놓칩니다. 100명 중 92명 성공, 8명 실패처럼 결과를 분리하고 실패 사유별 재시도 버튼을 제공해야 운영자가 엑셀로 다시 관리하지 않습니다.
