AI 제품 UX·신뢰 설계

AI 제품의 UX는 예쁜 입력창을 만드는 일이 아니다. 불확실성, 근거, 검토, 거절, 사람 개입을 사용자가 이해하게 만드는 일이다. 모델은 확률적으로 답하고, 자신이 틀렸는지 스스로 모른다. 그 간극을 인터페이스가 메운다. 이 문서는 규제가 강한 공공·B2B 플랫폼에 RAG·에이전트 기능을 붙이는 개발자를 독자로 상정한다. 공공 도메인에서는 틀린 답 하나가 정책 판단이나 금액 정산으로 이어질 수 있어, 신뢰 설계가 미관보다 먼저다.

MAP

AI 제품 UX의 핵심 문제

일반 소프트웨어 UX는 "버튼을 누르면 정해진 결과가 나온다"는 결정론 위에 서 있다. AI 제품은 그 전제가 깨진다. 같은 질문에 다른 답이 나오고, 그럴듯하지만 틀린 답이 나오고, 근거 없이 확신한다. 그래서 AI UX의 일은 결과를 예쁘게 보여주는 게 아니라, 결과를 얼마나 믿어야 하는지를 사용자가 판단하도록 돕는 것이다. 아래 다섯 문제가 이 문서 전체의 뼈대다.

문제UX가 답해야 할 질문실패 신호
확률적 출력사용자는 이 결과를 어느 정도 믿어도 되는가모든 답이 똑같이 자신만만한 톤으로 나온다
근거 부족주장의 출처와 한계를 확인할 수 있는가출처 링크가 장식이고 실제 주장과 무관하다
권한·정책 거절거절 이유와 다음 행동이 명확한가"죄송합니다, 도와드릴 수 없습니다"로 막다른 골목
검토 필요생성 결과를 승인·수정·폐기하는 흐름이 있는가모델 출력이 곧바로 발송·실행된다
실패 복구오류를 신고하고 대체 경로로 갈 수 있는가에러 화면에서 사용자가 할 수 있는 게 새로고침뿐이다
실무 기준

기능을 설계하기 전에 이 표의 다섯 줄에 각각 "이 화면에서는 어떻게?"를 먼저 답해 본다. 답이 안 나오는 줄이 있으면 그 기능은 아직 출시 준비가 안 된 것이다. 특히 공공 B2B에서는 "검토"와 "실패 복구"가 비어 있는 채로 나가면 감사(audit) 요구를 통과하지 못한다.

질문·요청→ 근거 검색·생성→ 불확실성 판정→ 근거와 함께 제시→ 검토·승인 또는 핸드오프
UNCERT

불확실성 표시

가장 흔한 실수는 모델의 confidence 점수(예: 0.87)를 그대로 화면에 노출하는 것이다. 이 숫자는 두 가지 이유로 위험하다. 첫째, 대부분의 LLM confidence는 보정(calibration)되어 있지 않다 — 0.9라고 해서 실제로 90% 맞는 게 아니다. 둘째, 사용자는 숫자를 보고도 "그래서 내가 뭘 해야 하지?"를 모른다. 불확실성은 숫자가 아니라 행동으로 번역해서 보여준다. "이 답은 근거가 약하니 발송 전 담당자 확인이 필요합니다"가 "confidence 62%"보다 백배 유용하다.

상태표현(행동 중심)사용자가 할 수 있는 행동
근거 충분근거 문서와 함께 답변, 출처 chip 노출바로 사용, 출처 열어 검증
근거 일부"일부만 확인됨 — 검토 권장" 배지질문 좁히기, 담당자 검토 요청
근거 없음답변을 만들지 않고 "찾지 못함" 명시문서 추가, 사람에게 문의
정책 제한거절 이유 + 허용 범위 안내권한 요청, 대체 자료 보기
안티패턴

"AI가 생성한 콘텐츠는 부정확할 수 있습니다"라는 전역 고지 하나로 모든 불확실성을 퉁치는 것. 이건 법적 방어일 뿐 UX가 아니다. 사용자는 그 문구를 3초 만에 눈에서 지운다. 불확실성은 각 답변의 맥락에서, 그 답이 실제로 얼마나 위험한지에 비례해서 표시해야 한다.

보조 노트

불확실성 표현의 강도는 결과의 되돌릴 수 있는 정도에 맞춘다. 읽고 마는 요약은 가벼운 배지로 충분하지만, 금액·정산 수치가 들어간 리포트 문안은 근거가 일부만 있어도 강한 검토 요청을 붙인다. 위험이 낮은 곳에 경고를 남발하면 경고 피로(alert fatigue)로 정작 중요한 경고가 무시된다.

CITE

출처·근거 UI

RAG 제품의 신뢰는 출처에서 만들어지고 출처에서 무너진다. 핵심은 "링크가 있다"가 아니라 "그 링크가 실제로 이 주장을 뒷받침하는가"다. 출처 UI의 계약은 아래와 같다. 주장 → 출처 chip → 근거 미리보기 → 문서 내 위치 → 최종 수정일 → 접근 권한 상태 → 잘못된 출처 신고. 이 사슬 중 하나라도 끊기면 사용자는 검증을 포기하고, 검증을 포기하면 over-trust로 넘어간다.

CITATION UX CONTRACT 1. answer claim — 근거가 붙는 주장 단위를 명확히 분리
2. source chip — 주장 옆 인라인 출처 표식(클릭 가능)
3. evidence preview — 원문 발췌를 하이라이트해 보여줌
4. document location — 문서명 + 섹션/페이지/문단 앵커
5. last updated — 근거 문서의 최종 수정일(오래되면 경고)
6. permission state — 사용자가 원문을 볼 권한이 있는지 표시
7. report wrong citation — 출처가 주장과 안 맞을 때 신고 경로
금지 패턴

세 가지를 절대 하지 마라. (1) 출처 링크가 실제 주장을 뒷받침하지 않는데 붙어 있는 것 — 가짜 근거가 근거 없음보다 나쁘다. (2) "AI가 그렇게 판단했습니다"를 근거처럼 제시하는 것 — 그건 근거가 아니라 동어반복이다. (3) 사용자가 볼 권한 없는 문서를 출처로 걸어 미리보기에서 403이 뜨는 것 — 신뢰가 즉시 붕괴한다.

고위험

공공·규제 산업 맥락에서 근거 문서가 이미 개정·폐지된 고시나 지침일 수 있다. 최종 수정일과 문서 버전을 출처 chip에 함께 노출하고, 일정 기간 이상 지난 근거는 "최신 여부 확인 필요"로 표시한다. "출처는 있는데 낡은 출처"는 감사에서 가장 자주 걸리는 지점이다.

실무 기준

근거 미리보기(evidence preview)에서 원문 중 실제로 인용된 문장을 하이라이트한다. 문서 전체를 열어 주기만 하면 사용자는 어디를 봐야 할지 몰라 검증을 안 한다. "이 주장은 이 문장에서 나왔다"를 시각적으로 연결하는 것이 citation UI의 실제 가치다.

REVIEW

생성 결과 검토·승인

모델 출력을 사용자·시스템에 곧바로 반영할지, 사람 승인을 거칠지는 결과의 영향 범위와 되돌릴 수 있는 정도로 정한다. 읽고 마는 요약과 외부로 나가는 공문, 시스템을 바꾸는 실행은 같은 정책일 수 없다. 위험이 커질수록 확인 단계를 늘리고, 감사 로그를 남긴다.

결과 유형기본 정책UI 요소
읽기 전용 요약자동 표시 + 출처·수정 제안 제공copy, cite, report wrong
외부 발송 문안(공문·이메일)사람 승인 필수, 발송 전 검토review diff, approve, reject, edit
코드·설정 변경검증 결과 통과 후 승인test 결과, risk 요약, rollback
결제·삭제·권한 변경고위험 — 2단계 확인 + 감사2-step confirmation, audit log
실무 기준

검토 UI의 기본값은 diff다. "새 문안 전체"를 보여주는 것보다 "기존 대비 무엇이 바뀌었나"를 보여줄 때 사람이 실수를 훨씬 잘 잡는다. 외부 발송 문안이라면 모델이 넣은 수치·날짜·고유명사를 시각적으로 구분해, 검토자가 사실 확인해야 할 지점을 바로 찾게 한다.

고위험 — 실전 시나리오

규제 정산 리포트 초안을 AI가 만든다고 하자. before: 모델이 만든 리포트에 "승인" 버튼 하나만 있고 누르면 대외 공개된다. after: (1) 수치 항목마다 원장 데이터와의 대사(reconcile) 결과를 옆에 표시, (2) 근거가 없는 수치는 빨간 배지로 강제 표시, (3) 승인은 2단계 + 승인자 이름·시각을 audit log에 기록, (4) 공개 후에도 정정 이력을 남기는 rollback 경로 유지. 공공 정산에서 "누가 언제 무엇을 승인했나"는 UX가 아니라 규정 요건이다.

GEN

생성 UX 패턴

생성은 시간이 걸리고, 중간 상태가 있고, 결과가 완성형이 아니다. 이 세 성질을 인터페이스가 정직하게 드러내야 한다. 스트리밍으로 진행을 보여주고, 복잡한 결과는 점진적으로 펼치고, 결과는 고칠 수 있게 하고, 사용자가 무엇을 물어봐도 되는지를 알려 준다.

패턴목적실무 판단·실패 신호
streaming 출력대기 체감 시간 단축, 진행 중임을 전달토큰이 뚝뚝 끊겨 읽기 방해되면 문장 단위 버퍼링
progressive disclosure핵심 먼저, 근거·상세는 접어 둠모든 걸 한 번에 펼쳐 인지 과부하
loading / thinking 상태"멈춘 것"과 "생각 중"을 구분무한 스피너로 실패인지 진행인지 모름
결과 editability초안임을 전제, 사용자가 직접 수정출력이 읽기 전용이라 재생성 외 방법 없음
prompt affordance무엇을 물어봐도 되는지 예시로 안내빈 입력창만 있어 사용자가 뭘 할지 모름
regeneration / variation마음에 안 들면 다시·다르게재생성 시 이전 결과가 통째로 사라져 비교 불가
실무 기준

streaming은 공짜 신뢰가 아니다. 스트리밍 도중 출처가 아직 안 붙은 상태의 문장을 사용자가 복사해 가면, 근거 없는 텍스트가 밖으로 나간다. 본문이 다 나온 뒤 출처를 소급해 붙이는 경우, 스트리밍 중에는 복사·발송 버튼을 비활성화한다. thinking 상태는 가짜로 길게 끌지 말고, 실제로 오래 걸리는 단계(검색 중 / 근거 대조 중 / 초안 작성 중)를 단계명으로 보여 주는 편이 정직하다.

보조 노트

prompt affordance는 "AI에게 무엇이든 물어보세요"가 아니라 도메인에 맞는 구체 예시여야 한다. 운영 분석 대시보드라면 "이번 분기 매출 실적을 전년 대비 요약해 줘", "이 프로젝트의 방법론 근거를 찾아 줘" 같은 실제 쓰이는 질문을 3~4개 제시한다. 이는 기능 범위(무엇을 할 수 있고 없는지)를 은근히 알려 주는 역할도 한다 — TRUST 섹션의 능력·한계 노출과 이어진다.

FAIL

에러·거절·복구

AI 제품의 에러는 두 종류를 반드시 구분한다. expected failure(예상된 실패 — 근거 없음, 정책상 거절, rate limit, timeout)와 bug(진짜 오류 — null, 파싱 실패, 서버 500)다. 전자는 정상 흐름의 일부로 설계하고, 후자는 신고·복구 경로를 준다. 둘을 같은 "오류가 발생했습니다" 화면으로 뭉개면 사용자는 자기가 뭘 잘못했는지, 기다리면 되는지, 포기해야 하는지 알 수 없다.

상황분류사용자에게 줄 것
근거를 못 찾음expected"찾지 못함" 명시 + 문서 추가·사람 문의 경로
정책상 거절expected거절 이유 + 허용 범위 + 권한 요청 경로
rate limitexpected재시도 가능 시각, 대기 안내(에러 아님)
timeout / partialexpected부분 결과 표시 + "이어서 생성" 또는 재시도
파싱·서버 오류bug사과 1회 + 신고 경로 + 대체 수단
거절 설계

거절은 막다른 골목이 아니라 분기점이어야 한다. 나쁜 거절: "요청을 처리할 수 없습니다." 좋은 거절: "이 문서는 열람 권한이 필요해 요약할 수 없습니다. 권한을 요청하거나, 공개된 요약본을 대신 볼 수 있습니다." 항상 왜 거절했는지 + 다음에 할 수 있는 것 두 가지를 함께 준다.

고위험 — partial 처리

timeout으로 답변이 중간에 끊긴 partial 결과는 특히 위험하다. 문장이 절반에서 잘린 리포트가 완성본처럼 보이면 사용자가 그대로 쓴다. partial일 때는 명시적으로 "생성이 중단됨" 표시 + 미완성 상태 배지 + 발송·저장 잠금을 건다. 자동 재시도는 결제·발송 같은 비멱등(non-idempotent) 작업에는 절대 걸지 않는다 — 중복 실행이 partial보다 더 큰 사고다.

HANDOFF

Human handoff

좋은 AI UX는 사람이 개입해야 하는 순간을 숨기지 않는다. 오히려 그 순간을 먼저 감지해 넘긴다. 핸드오프는 실패가 아니라 설계된 안전장치다. 넘겨야 할 신호는 대체로 네 가지 — 근거가 약할 때, 위험이 높을 때, 정책이 모호할 때, 사용자가 같은 지적을 반복할 때. 넘길 때는 담당자가 맥락을 다시 캐지 않도록 handoff packet을 함께 넘긴다.

handoff_trigger:
  - low_evidence          # 근거 부족·상충
  - high_risk_action      # 결제·삭제·대외 발송
  - policy_ambiguity      # 규정 해석이 갈리는 요청
  - repeated_correction   # 사용자가 같은 오류를 2회+ 지적

handoff_packet:
  - user_question         # 사용자가 실제로 물은 것
  - model_answer          # 모델이 낸 답(초안)
  - retrieved_evidence    # 검색된 근거 + 출처 위치
  - confidence_reason     # 왜 불확실하다고 판정했는지
  - user_feedback         # 사용자의 지적·수정 이력
  - suggested_next_action # 담당자가 바로 할 수 있는 다음 행동
실무 기준

"repeated correction"은 놓치기 쉬운 강력한 신호다. 사용자가 같은 오류를 두 번 이상 고쳐 준다는 건 모델이 그 맥락을 못 잡고 있다는 뜻이다. 이때 계속 재생성하게 두지 말고 "담당자에게 연결할까요?"를 제안한다. 핸드오프 경로가 없는 AI 제품은 사용자를 무한 재시도 루프에 가둔다.

보조 노트

핸드오프 후에도 사용자에게 상태를 보여 준다 — "담당자 검토 중", "예상 응답 시각". 넘기고 침묵하면 사용자는 요청이 사라진 줄 안다. 담당자가 답하면 그 답을 원래 대화 맥락에 이어 붙여, 사용자가 두 채널을 오가지 않게 한다.

TRUST

신뢰·기대 설정

사용자는 AI 제품을 쓰면서 머릿속에 "이건 뭘 할 수 있고 뭘 못 하는가"라는 mental model을 만든다. 이 모델이 틀리면 두 방향으로 사고가 난다. over-trust(automation bias — 틀린 답도 검증 없이 믿음)와 under-trust(맞는 답도 못 믿어 안 씀). 둘 다 위험하고, 해법은 하나다 — 능력과 한계를 정직하게 노출하는 것. 과장하면 over-trust를, 겸손을 가장한 회피는 under-trust를 부른다.

위험증상완화
over-trust (automation bias)근거를 안 열고 결과를 그대로 발송고위험 결과에 검토 마찰(friction) 의도적 삽입
under-trust맞는 답도 매번 처음부터 다시 확인근거·정확도 이력을 보여 신뢰 축적
범위 오해안 되는 걸 계속 시도능력·한계를 prompt affordance로 미리 노출
실무 기준

신뢰는 단발 문구가 아니라 일관된 행동으로 쌓인다. 모델이 모를 때 정직하게 "모른다"고 한 경험이 있는 사용자는, 답할 때의 확신도 더 믿는다. 반대로 한 번이라도 그럴듯한 거짓말(confident hallucination)에 당하면 그 제품 전체의 신뢰가 무너진다. 정직성은 이 도메인에서 미관보다 상위의 가치다 — 특히 공공 B2B에서 한 번의 신뢰 붕괴는 조직 단위 이탈로 이어진다.

안티패턴

제품 소개에서 "AI가 알아서 다 해 드립니다"류의 과장. 이건 초기 채택은 올릴지 몰라도 첫 실패에서 신뢰를 통째로 잃는다. 능력을 정직하게 좁혀 약속하고(under-promise), 그 안에서 확실히 지키는(over-deliver) 편이 장기 신뢰에 낫다.

A11Y

접근성

AI UX의 동적 특성 — 스트리밍, 상태 변화, 실시간 근거 표시 — 은 접근성을 특히 어렵게 만든다. 시각적으로 나타나는 변화가 스크린리더 사용자에게 전달되지 않으면, 그 사용자에게는 화면이 조용히 멈춰 있는 것과 같다. 대비·명도 같은 정적 접근성은 별도 문서의 원칙(WCAG 대비 기준 등)을 따르고, 여기서는 AI 특유의 동적 접근성에 집중한다.

요소접근성 요구구현 판단
스트리밍 출력진행·완료를 소리로 알림live region으로, 단 polite로 토큰마다 말하지 않게
상태 변화(생성 중→완료→오류)상태 전환을 announce단계 완료 시점에만 aria-live 갱신
citation chip키보드 접근 + 의미 전달버튼으로, 레이블에 "출처: 문서명" 명시
근거 preview 팝오버포커스 이동·복귀 관리열 때 preview로 focus, 닫으면 트리거로 복귀
로딩 중 입력비활성 상태를 명확히aria-busy + 버튼 disabled 상태 announce
실무 기준

스트리밍을 aria-live="assertive"로 토큰마다 읽게 하면 스크린리더가 끝없이 떠들어 오히려 못 쓴다. 본문은 조용히 채우되(polite, 혹은 완료 시 한 번), "생성 완료" 같은 상태 전환만 announce한다. citation은 반드시 실제 버튼/링크 요소로 만들어 키보드 탭으로 닿게 하고, 근거 preview를 열면 focus를 그 안으로 옮겨 스크린리더 사용자가 근거를 읽을 수 있게 한다 — 닫으면 원래 위치로 되돌린다.

보조 노트

공공 서비스는 접근성이 법적 요건(전자정부 웹 접근성 등)인 경우가 많다. AI 기능이라고 예외가 아니다. 오히려 새로 도입되는 만큼 감사에서 먼저 지적된다. 동적 상태의 announce, 키보드 전용 조작, focus 관리 세 가지를 출시 전 체크리스트에 반드시 넣는다.

QA

AI UX QA Matrix

일반 QA는 "기능이 동작하는가"를 본다. AI UX QA는 거기에 더해 불확실성·근거·거절·피드백·접근성이 제대로 노출되는가를 검증한다. happy path만 통과하고 실패 경로를 안 보면, 정작 사고가 나는 경로가 검증 안 된 채 나간다. 아래를 출시 전 체크리스트로 쓴다.

차원검증 질문합격 기준
불확실성근거 부족 상태가 happy path와 다른 UI로 보이는가근거 없음/일부/충분이 시각적으로 구분됨
출처모든 핵심 주장에 확인 가능한 근거가 있는가chip → preview → 원문 위치가 끊김 없이 연결
거절정책상 답할 수 없을 때 다음 행동이 있는가모든 거절에 이유 + 대체 경로 포함
피드백사용자가 오류를 수정·신고할 수 있는가report·edit 경로가 모든 결과에 존재
실패partial·timeout이 완성본처럼 보이지 않는가미완성 상태 배지 + 발송 잠금 동작

2. 출처 chip을 눌러 실제 원문의 해당 문장에 닿는가
3. 권한 없는 문서를 출처로 걸었을 때 우아하게 처리되는가
4. 고위험 결과에 2단계 확인·감사 로그가 걸리는가
5. timeout을 강제해 partial 결과가 잠기는가
6. 핸드오프 트리거가 실제로 발동하고 packet이 채워지는가
7. 스크린리더로 생성→완료→오류 전환이 들리는가
GLOSSARY

용어집

이 문서에서 쓴 핵심 용어를 정리한다. 팀 안에서 이 단어들의 뜻을 맞춰 두면, "신뢰 설계"를 감각이 아니라 공통 어휘로 논의할 수 있다.

Grounding근거 접지
모델 답변을 실제 문서·데이터에 연결해 검증 가능하게 만드는 것. RAG의 핵심 목적.
RAG검색 증강 생성
Retrieval-Augmented Generation. 답을 만들기 전 관련 문서를 검색해 근거로 삼는 방식.
Hallucination환각
모델이 근거 없이 그럴듯한 거짓을 만들어 내는 현상. confident hallucination이 가장 위험.
Confidence확신도
모델이 자기 답에 매기는 점수. 보통 calibration이 안 돼 실제 정확도와 어긋나므로 그대로 노출 금지.
Calibration보정
confidence 숫자가 실제 정답률과 일치하도록 맞추는 것. 대부분의 LLM은 미보정 상태.
Citation출처 표기
주장을 뒷받침하는 근거 문서·위치를 사용자가 검증할 수 있게 제시하는 UI 계약.
Human handoff사람 인계
모델이 감당하기 어려운 요청을 맥락(packet)과 함께 담당자에게 넘기는 것.
Guardrail가드레일
모델이 해선 안 될 행동(정책 위반·고위험 실행)을 사전에 막는 안전장치.
Automation bias자동화 편향
시스템이 낸 결과를 검증 없이 신뢰하는 인지 편향. over-trust의 원인.
Expected failure예상된 실패
근거 없음·거절·rate limit처럼 정상 흐름의 일부인 실패. bug와 구분해 설계.
Progressive disclosure점진적 공개
핵심을 먼저 보이고 상세·근거는 필요할 때 펼치는 정보 노출 방식.
Live region라이브 영역
내용이 동적으로 바뀔 때 스크린리더가 그 변화를 읽도록 표시하는 ARIA 영역.
