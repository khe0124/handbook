FE-CS-01

핵심 명령·스니펫

화면을 고치기 전에 증거부터 고정한다. DevTools로 병목을 나누고, CLI로 번들·성능·리소스 크기를 측정한다.

DevTools 실전 순서

# 1) Performance 탭 — 느린 상호작용/로딩 원인 분리
#    Record 시작 → 문제 동작 재현 → Stop
#    Main thread flame chart에서 Scripting/Rendering/Painting 비중 확인
#    Long Task(>50ms) 노란 삼각형, INP 후보 interaction 클릭해 breakdown

# 2) Network 탭 — waterfall로 로딩 순서 확인
#    Disable cache 체크, throttling을 Fast 4G로 두고 새로고침
#    LCP 이미지가 initial JS 뒤에 늦게 시작하는지, TTFB가 큰지 확인
#    Priority 컬럼(우클릭 헤더로 표시)으로 우선순위 오지정 탐지

# 3) Coverage 탭 — 안 쓰는 JS/CSS 비율
#    Cmd+Shift+P → "Show Coverage" → reload → Unused Bytes 확인

# 4) React DevTools Profiler — 렌더 원인
#    Profiler 탭 → 녹화 → 상호작용 1회 → 멈춤
#    "Ranked" 뷰에서 commit 비싼 컴포넌트, "Why did this render?" 활성화

빌드 · 번들 분석

# 프로덕션 빌드 (번들 에러/사이즈 먼저 확인)
npm run build

# 빌드 결과를 로컬에서 프로덕션과 유사하게 서빙
npx serve dist          # Vite 계열
npx serve build         # CRA 계열

# 번들에서 뭐가 큰지 시각화 (source map 필요)
npx source-map-explorer 'dist/**/*.js'

# gzip/brotli 실제 전송 크기 확인
npx vite-bundle-visualizer          # Vite
ANALYZE=true npm run build          # next.js (@next/bundle-analyzer 설정 시)

성능 측정 (Lighthouse / Web Vitals)

# 성능 카테고리만 로컬 audit
npx lighthouse http://localhost:3000 --only-categories=performance

# 헤드리스로 JSON 저장 (CI 비교용)
npx lighthouse http://localhost:3000 \
  --only-categories=performance \
  --output=json --output-path=./lh.json --chrome-flags="--headless"

# 모바일 프리셋 대신 데스크톱으로
npx lighthouse http://localhost:3000 --preset=desktop

응답 헤더 · 리소스 크기 확인

# 캐시/압축 헤더만 빠르게 (cache-control, etag, content-encoding)
curl -sI https://example.com/app.js

# 압축이 실제 적용되는지 (Accept-Encoding 보내고 헤더 확인)
curl -sI -H "Accept-Encoding: br,gzip" https://example.com/app.js

# 실제 전송 바이트 (전송 크기 = size_download)
curl -s -o /dev/null -w "%{size_download} bytes / %{time_total}s\n" https://example.com/app.js

# 이미지 파일 크기 확인
ls -lh public/*.{png,jpg,webp,avif} 2>/dev/null

접근성 빠른 점검 (코드 없이 손으로)

# 1) Tab 키만으로 핵심 흐름 완주 — 마우스 금지
#    포커스 순서가 시각 순서와 같은가? 갇히는 곳(trap) 없는가?
# 2) focus visible — Tab 이동 시 포커스 링이 보이는가? (outline:none 사고)
# 3) accessible name — 아이콘 버튼에 이름이 있는가?
#    DevTools Elements → Accessibility 패널 → Computed Properties의 Name 확인
# 4) 대비 — DevTools에서 텍스트 요소 클릭 → color picker의 Contrast ratio (AA≥4.5:1)
명령용도주의

curl -sI <url>캐시·압축 헤더 확인-I는 HEAD 요청 — 일부 서버는 GET과 헤더가 다름
FE-CS-02

판단 한 줄

선택언제 쓰나주의
CSR로그인 뒤 내부 도구, SEO 무관, 인터랙션 중심초기 빈 화면 + 큰 JS 번들
SSR초기 표시·SEO 중요 + 사용자별 데이터서버 비용, cache 전략, hydration 안정성
SSG콘텐츠 거의 고정, CDN 빠른 제공이 핵심데이터 변경 주기 → 재생성(ISR) 전략 필요
memoization측정으로 렌더 비용/props 안정성 확인 뒤에만측정 전 남발 = 복잡도만 증가, 병목이 네트워크면 무효
URL state공유·뒤로가기·새로고침 복원 필요한 필터/검색입력 중 값과 확정 query는 분리
client state모달·탭·입력 중 값처럼 UI 즉시성 상태공유 필요하면 URL로, 아니면 컴포넌트 로컬
server state서버가 소유한 캐시 가능 비동기 데이터UI state로 복사하지 말 것 — freshness/invalidate가 핵심
FE-CS-03

자주 터지는 것 · 즉시 확인

신호왜 위험즉시 확인
측정 전 memo/useMemo/useCallback 남발병목이 네트워크·이미지면 효과 0, 복잡도만 증가React Profiler로 렌더 원인, Network waterfall 먼저
API 응답을 컴포넌트 state로 복사server/client 상태 혼합 → stale·동기화 버그새로고침·탭 전환 후 값이 어긋나는지
hydration mismatch서버 HTML과 client 첫 렌더 불일치 → 화면 튐날짜·locale·random·localStorage 접근을 effect로 미룸
div에 onClick만 붙인 가짜 버튼키보드 도달 불가, accessible name 없음Tab 도달 여부, semantic button/a로 교체
loading/empty/error/permission 상태 누락운영 데이터에서 UI가 쉽게 깨짐상태별 스냅샷·story, 실패 경로 2개 이상
dangerouslySetInnerHTML / innerHTML사용자 입력이 escape 없이 DOM 실행 → XSSsanitize, 입력 출처, CSP 확인
FE-CS-04

장애 증거 1카드

BROKEN SCREEN TRIAGE코드부터 고치지 말고 route/role/flag → Console → Network → API/cache → hydration 순서로 증거를 고정한다. 원인 확정 전 복구는 flag off · rollback · CDN purge 중 사용자 영향이 가장 작은 것.
순서질문증거
1어떤 route·role·browser·viewport·flag에서만 재현?재현 조건, session, feature flag 상태
2Console error가 먼저인가?DevTools Console stack trace, 첫 에러 시각
3Network 실패가 있는가?status code, response body, deploy SHA
4API contract 변경 또는 cache stale인가?response schema, etag/cache-control
5hydration·date/locale/random·browser-only API 차이?SSR log, client stack, hydration warning
복구/중단

영향 범위와 데이터 변경 여부부터 확인. 데이터를 바꾸지 않는 flag off를 우선, 배포 회귀면 rollback, 정적 자산 stale이면 CDN purge. 원인 미확정 상태로 hotfix 배포 금지.
