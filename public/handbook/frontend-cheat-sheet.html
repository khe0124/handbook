Cheat Sheet

프론트엔드 판단을 빠르게 복기하기 위한 압축 참조입니다. 렌더링, 상태 경계, 성능, 접근성, 보안, 테스트를 긴 설명 대신 실무 체크 기준으로 정리합니다.

COMPILED : 2026-06 · FORMAT : QUICK REFERENCE
FE-CS-01

개요

프론트엔드 문제는 “화면이 안 나옴”으로 보이지만 실제 원인은 렌더링, 네트워크, 상태 소유권, 접근성, 보안 경계 중 하나입니다. 먼저 증상을 계층으로 나눠야 빠르게 고칩니다.

상황먼저 확인할 것실패 신호
화면이 느림렌더 횟수, 큰 DOM, 네트워크 waterfall, 이미지와 번들 크기interaction마다 전체 목록 재렌더, LCP 이미지 지연
상태가 꼬임서버 상태와 UI 상태가 섞였는지, derived state를 저장했는지새로고침 후 값이 바뀌거나 탭 간 값이 어긋남
SSR 오류서버/클라이언트 렌더 결과 차이, 브라우저 전용 API 접근 시점hydration mismatch, 날짜·랜덤값·localStorage 차이
접근성 누락키보드 이동, focus, label, semantic element, contrastTab으로 닿지 않거나 screen reader 이름이 없음
보안 리뷰HTML 삽입, token 저장, CSRF, dependency, CSP사용자 입력이 escape 없이 DOM에 들어감
FE-CS-02

핵심 명령/개념

개념압축 설명실무 해석
Reflow레이아웃 계산. 크기와 위치 변경이 원인이라 비용이 큼반복 측정/쓰기, 큰 table/list에서 체감됨
Repaint색, 그림자 같은 픽셀 다시 그리기box-shadow, filter, 큰 배경 변경은 비용이 커질 수 있음
Composite이미 그린 레이어를 합성animation은 가능하면 transform/opacity 위주로 설계
Server state서버가 소유한 캐시 가능한 비동기 데이터freshness, invalidate, retry, loading/error 상태가 핵심
Client state모달, 탭, 입력 중 값처럼 UI가 소유한 즉시성 상태URL에 남길지, 컴포넌트에 둘지, 전역화할지 결정
Hydration mismatch서버 HTML과 클라이언트 첫 렌더 결과 불일치서버에서 결정 불가능한 값은 effect 이후로 미룸
Core Web VitalsLCP, INP, CLS 중심 사용자 체감 지표LCP는 로딩, INP는 반응성, CLS는 레이아웃 안정성 문제
FAST CHECKnpm run build로 번들 오류를 먼저 잡고, DevTools Performance와 Network 탭으로 병목을 나눠 봅니다.
FE-CS-03

판단 기준

선택기준주의점
CSR로그인 뒤 내부 도구, SEO 영향 작음, 인터랙션 중심초기 빈 화면과 큰 JS 번들 관리 필요
SSR초기 표시와 SEO가 중요하고 사용자별 데이터가 있음서버 비용, cache 전략, hydration 안정성 필요
SSG콘텐츠가 거의 고정이고 빠른 CDN 제공이 중요함데이터 변경 주기와 재생성 전략 확인
memoization계산 비용이 크거나 props 안정성이 보장될 때만 적용측정 없이 남발하면 코드 복잡도만 증가
URL state공유, 뒤로가기, 새로고침 복원이 필요한 필터/검색입력 중 값과 확정된 query를 분리
E2E test사용자 흐름 자체가 더 중요한 핵심 경로모든 edge를 E2E로 밀면 느리고 불안정해짐
FE-CS-04

자주 나는 실수

  • API 응답 전체를 컴포넌트 상태로 복사해 서버 상태와 UI 상태를 동시에 관리한다. stale data와 동기화 버그가 생긴다.
  • 성능 문제를 보기 전에 무조건 memo, useMemo, useCallback을 추가한다. 실제 병목이 네트워크나 이미지면 효과가 없다.
  • 버튼 역할을 하는 div를 만들고 keyboard/focus 처리를 빼먹는다. 접근성과 테스트 가능성이 동시에 나빠진다.
  • 토큰 저장 위치를 보안 위협 모델 없이 관성으로 결정한다. XSS, CSRF, 세션 폐기 요구가 선택을 바꾼다.
  • hydration 문제를 suppress 옵션으로 숨기고 원인을 제거하지 않는다. 날짜, locale, 랜덤값, 브라우저 API 접근을 먼저 의심한다.
  • loading, empty, error, permission denied, offline 상태를 나중에 붙인다. 핵심 UI 구조가 뒤늦게 흔들린다.
FE-CS-05

면접/리뷰용 한 줄 답변

상태 분리state boundary
서버 상태는 캐시와 동기화 문제이고, 클라이언트 상태는 즉시 UI 반응 문제라 lifecycle과 소유자가 다릅니다.
성능 최적화performance
먼저 측정으로 병목을 렌더링, 네트워크, 번들, 이미지 중 어디인지 나눈 뒤 가장 큰 비용부터 줄입니다.
접근성a11y
접근성은 추가 기능이 아니라 키보드, 스크린리더, semantic markup까지 포함한 기본 사용성입니다.
SSRhydration
SSR의 핵심 위험은 서버가 만든 HTML과 클라이언트 첫 렌더가 달라지는 것이므로 비결정적 값은 분리해야 합니다.
보안browser risk
브라우저는 사용자가 조작 가능한 환경이므로 입력 escape, 권한 재검증, 토큰 저장, 의존성 위험을 함께 봐야 합니다.
FE-CS-06

Red Flags

프론트엔드 품질은 정상 화면보다 빈 화면, 실패 화면, 느린 상호작용, 접근성, 브라우저 보안에서 드러납니다.

신호왜 위험한가즉시 확인
loading, empty, error, permission 상태가 없음실제 운영 데이터에서는 UI가 쉽게 깨집니다.상태별 스냅샷, story, E2E 경로를 추가합니다.
dangerouslySetInnerHTML 또는 innerHTML 사용XSS가 DOM 실행으로 이어질 수 있습니다.escape/sanitize, CSP, 입력 출처를 확인합니다.
토큰 저장 위치를 관성으로 결정XSS와 CSRF 위협 모델에 따라 선택이 달라집니다.HttpOnly, SameSite, refresh 흐름, logout/revoke를 비교합니다.
측정 전 memoization 남발렌더 원인보다 참조 안정성 코드가 복잡해집니다.React Profiler와 Network waterfall을 먼저 봅니다.
hydration 경고를 suppress로 숨김서버·클라이언트 초기 상태 차이가 계속 남습니다.날짜, locale, random, browser API 접근을 분리합니다.
FE-CS-07

Review Gate

변경 유형merge/deploy 차단 기준통과 증거
화면 추가빈 상태, 오류 상태, 권한 없음, 모바일 폭에서 레이아웃이 없음상태별 캡처와 주요 viewport 확인이 있습니다.
폼·입력서버 validation 오류를 사용자 행동으로 복구할 수 없음필드별 오류, submit 중복 방지, 재시도 흐름이 있습니다.
성능 변경LCP/INP/CLS 또는 번들 영향이 측정되지 않음전후 수치와 병목 근거가 있습니다.
접근성키보드로 핵심 액션에 도달할 수 없음focus order, label, semantic element, contrast를 확인했습니다.
보안 경계사용자 입력 HTML, 토큰, redirect URL 검증이 불명확함sanitize, allowlist, SameSite/CSP 정책이 명시돼 있습니다.
FE-CS-08

Failure Playbook

BROKEN SCREEN TRIAGE깨진 화면은 코드부터 고치기보다 route, role, API, runtime error, hydration, cache 순서로 증거를 모읍니다.
순서질문증거
1어떤 route, user role, browser, viewport에서만 재현되는가?재현 조건, session, feature flag
2Console error와 Network 실패가 먼저인가?DevTools Console, Network status, response body
3API contract 변경 또는 cache stale이 있는가?response schema, ETag/cache-control, deploy SHA
4hydration, date/locale/random, browser-only API 차이인가?SSR log, client stack trace, hydration warning
5flag off, rollback, CDN purge 중 어떤 복구가 가장 안전한가?사용자 영향 범위와 데이터 변경 여부
FE-CS-09

실전 실행 카드

프론트엔드 치트시트는 화면을 고치기 전에 증거를 고정하는 용도로 씁니다. 아래 카드는 PR 리뷰나 장애 triage에서 그대로 복사해 체크합니다.

상황바로 확인정상 신호다음 행동
첫 화면 느림Network waterfall, LCP element, Coverage, bundle sizeHTML, CSS, LCP image, initial JS의 시작 시각과 크기를 설명 가능TTFB, render-blocking CSS, 이미지 우선순위, initial JS 중 하나만 실험
입력 지연Performance trace, React Profiler commit, INP 후보 interactionhandler, render, layout, paint 비용이 분리됨state fan-out 축소, virtualization, worker, transition 중 최소 변경 선택
화면 깨짐Console stack, Network response body, route/role/flag, deploy SHA재현 조건과 실패 응답이 같은 패킷에 있음API contract, cache, hydration, feature flag 순서로 원인 좁힘
폼 실패필드 오류, 서버 validation code, 중복 submit, focus 이동사용자가 어떤 값을 고칠지 알 수 있음error contract와 UI 상태를 맞추고 pending/disabled/retry를 점검
접근성 의심Keyboard only, accessible name, focus visible, aria-describedby마우스 없이 핵심 흐름 완료 가능공통 컴포넌트 API에 label, description, focus restore를 강제
FE-CS-10

출력 판정표

도구/증거위험 신호해석중단 기준
Network waterfallLCP 이미지가 JS 뒤에 늦게 시작됨리소스 우선순위 또는 lazy/preload 설정 문제전후 waterfall 없이 이미지 최적화 완료 처리 금지
React Profiler입력 1회에 큰 parent와 목록 전체가 commit됨상태 소유권이나 props identity 문제memo 추가 전 render 원인 기록 필요
Consolehydration mismatch, uncontrolled/controlled 전환서버/클라이언트 초기 상태 또는 폼 상태 모델 불일치suppress 경고로 숨기고 배포 금지
Lighthouse점수만 개선되고 LCP/INP/CLS 핵심 route 수치가 없음점수 추격으로 실제 UX 개선 증거 부족field 또는 동일 조건 lab 비교 없으면 성능 PR 보류
E2E/수동 테스트happy path만 있고 error/empty/permission/offline 없음운영 데이터 상태에서 쉽게 깨짐핵심 화면은 최소 실패 상태 2개 이상 검증
