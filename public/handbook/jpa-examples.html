JPA
예시 사례

엔티티 매핑부터 N+1 회피까지 — 바로 복붙해 쓰는 JPA 레시피. 개념의 왜는 JPA 핸드북에 넘기고, 여기선 실제로 어떻게 쓰는지만 코드로 봅니다. 각 섹션은 "언제·왜" 한두 줄 + 복붙 가능한 코드 + 발 걸리는 곳 경고.

NOTE : 패키지는 jakarta.persistence.* (Spring Boot 3 / Jakarta EE 9+) 기준 — 구버전은 javax.persistence.*
COMPILED : 2026-06
JPA-01

엔티티 기본

테이블 하나를 객체 하나에 매핑하는 출발점. 엔티티 = DB 테이블에 대응하는 영속 객체이고, JPA가 추적할 수 있는 형태로 만드는 게 1번 작업입니다.

Member.java — 가장 기본형 엔티티 @Entity
@Table(name = "member")
public class Member {
  @Id
  @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;

  @Column(name = "email", nullable = false, unique = true, length = 100)
  private String email;

  @Column(nullable = false)
  private String name;

  protected Member() {}  // JPA 기본 생성자 (필수)

  public Member(String email, String name) {
    this.email = email;
    this.name = name;
  }
  // getter ...
}
어노테이션역할
@Entity이 클래스를 영속 엔티티로 등록. 클래스명이 기본 엔티티명
@Table(name=...)매핑할 테이블명 지정. 생략하면 엔티티명을 테이블명으로
@IdPK 필드 표시 (엔티티당 하나는 필수)
@GeneratedValue(strategy=IDENTITY)PK를 DB의 AUTO_INCREMENT에 위임 (MySQL·PostgreSQL 흔함)
@Column(nullable, unique, length)컬럼 제약. 생략하면 필드명·기본 제약으로 매핑
기본 생성자는 빼먹지 말 것

JPA는 reflection으로 엔티티를 만들기 때문에 파라미터 없는 기본 생성자가 반드시 있어야 합니다. 접근 제어는 protected까지 허용 — 외부에서 빈 객체를 못 만들게 막으면서 JPA·프록시는 쓸 수 있습니다. final 클래스/필드, @Id 없는 엔티티는 안 됩니다. (→ JPA 핸드북: 엔티티 생명주기)

JPA-02

연관관계 매핑

테이블의 외래키(FK)를 객체 참조로 바꾸는 작업. 핵심 두 가지 — FK를 가진 쪽이 연관관계의 주인(owner)이고, fetch는 거의 항상 LAZY로 시작합니다.

N:1 — 주인 쪽 (FK 보유: Order → Member) @Entity
public class Order {
  @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;

  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "member_id")  // FK 컬럼명
  private Member member;
}
1:N — 반대편 (mappedBy: 거울만, FK 없음) @Entity
public class Member {
  // ... @Id 등
  @OneToMany(mappedBy = "member", cascade = CascadeType.ALL, orphanRemoval = true)
  private List<Order> orders = new ArrayList<>();

  // 양방향 연관관계 편의 메서드
  public void addOrder(Order order) {
    this.orders.add(order);
    order.setMember(this);  // 양쪽 동기화
  }
}
요소의미
@ManyToOne + @JoinColumnFK를 가진 주인 쪽. name이 실제 FK 컬럼명
@OneToMany(mappedBy="member")주인이 아닌 거울. mappedBy는 상대 엔티티의 필드명
cascade = ALL부모 영속화/삭제가 자식에 전파 (수명주기 종속일 때만)
orphanRemoval = true컬렉션에서 빠진 자식을 DELETE
편의 메서드를 쓰는 이유

양방향에선 객체 양쪽을 직접 맞춰줘야 합니다 — JPA가 자동으로 동기화하지 않습니다. order.setMember(member)만 하고 member.getOrders()에 안 넣으면, 같은 트랜잭션 안에서 컬렉션이 비어 보이는 버그가 납니다. 그래서 한쪽에 addOrder 같은 편의 메서드를 두고 양쪽을 한 번에 맞춥니다. DB에 FK를 쓰는 건 주인 쪽(@ManyToOne)뿐이라는 점도 기억하세요.

JPA-03

Repository

Spring Data JPA의 핵심 편의 — 인터페이스만 선언하면 구현체는 런타임에 자동 생성됩니다. 메서드 이름 규칙만 지키면 쿼리가 만들어집니다.

MemberRepository.java — 쿼리 메서드 public interface MemberRepository extends JpaRepository<Member, Long> {

  // SELECT ... WHERE email = ?
  Optional<Member> findByEmail(String email);

  // SELECT ... WHERE name LIKE %?%
  List<Member> findByNameContaining(String keyword);

  // SELECT EXISTS ... WHERE email = ?
  boolean existsByEmail(String email);

  // SELECT COUNT(*) WHERE name = ?
  long countByName(String name);

  // 조건 조합 — AND / OrderBy
  List<Member> findByNameAndEmail(String name, String email);
  List<Member> findByNameOrderByIdDesc(String name);
}
키워드생성되는 조건
findBy필드WHERE 필드 = ? — 단건은 Optional<T>, 다건은 List<T>
Containing / StartingWithLIKE %?% / LIKE ?%
existsBy필드boolean 반환 — 존재 여부만 (count보다 가벼움)
countBy필드long 반환 — 개수
And / Or / OrderBy조건 조합·정렬
JpaRepository가 공짜로 주는 것

JpaRepository<Member, Long>만 상속해도 save, findById, findAll, delete, count 등이 이미 들어 있습니다. 제네릭은 <엔티티타입, PK타입> 순서. 단건 조회는 null 대신 Optional로 받아 orElseThrow로 명시적으로 처리하는 패턴을 권합니다.

JPA-04

JPQL과 @Query

메서드 이름으로 표현이 안 되는 쿼리는 직접 씁니다. JPQL은 테이블이 아니라 엔티티·필드를 대상으로 하는 쿼리입니다 — FROM Member m처럼 클래스명을 씁니다.

@Query — JPQL + @Param 바인딩 @Query("SELECT m FROM Member m WHERE m.email = :email")
Optional<Member> findByEmailJpql(@Param("email") String email);

// 일부 컬럼만 — DTO projection (생성자 표현식)
@Query("SELECT new com.example.dto.MemberView(m.id, m.name) FROM Member m")
List<MemberView> findAllViews();

// 네이티브 SQL — DB 함수가 필요할 때
@Query(value = "SELECT * FROM member WHERE email = :email", nativeQuery = true)
Optional<Member> findByEmailNative(@Param("email") String email);
방식언제
JPQL @Query기본. 엔티티·필드 기준, DB 독립적
@Param 바인딩:name 형태 이름 파라미터. SQL 인젝션 방지에도 필수
DTO projectionSELECT new ...로 필요한 컬럼만 — 조회 전용에 유리
nativeQuery = trueDB 고유 함수·복잡 SQL이 필요할 때만. 이식성 포기
문자열 연결로 파라미터 넣지 말 것

"WHERE email = '" + email + "'" 같은 문자열 조립은 SQL 인젝션입니다. 항상 :param + @Param 바인딩을 쓰세요 — JPA가 parameterized query로 처리합니다. 네이티브 쿼리도 마찬가지로 바인딩으로만 값을 넣습니다.

JPA-05

N+1 해결

JPA에서 가장 자주 만나는 성능 함정. 목록 1번 조회 후, 각 행의 연관 엔티티를 N번 추가 조회하는 현상입니다. LAZY 연관을 루프에서 건드릴 때 터집니다.

문제 재현 — orders 10건이면 쿼리 1 + 10 = 11번 List<Order> orders = orderRepository.findAll();  // SELECT * FROM orders (1번)
for (Order o : orders) {
  o.getMember().getName();  // member마다 SELECT (N번) ← 문제
}
해결 1 — fetch join (한 방에 조인 조회) @Query("SELECT o FROM Order o JOIN FETCH o.member")
List<Order> findAllWithMember();

해결 2 — @EntityGraph (메서드 이름 규칙 유지하며 fetch)
@EntityGraph(attributePaths = {"member"})
@Query("SELECT o FROM Order o")
List<Order> findAllGraph();
해결책특징
fetch join JOIN FETCH가장 명시적. @ManyToOne·단건 컬렉션에 적합
@EntityGraph(attributePaths)JPQL 없이 어노테이션으로 fetch 지정. 쿼리 메서드와 조합 가능
default_batch_fetch_size컬렉션 N+1을 IN (?, ?, ...)로 묶어 N → 1~몇 번으로
해결 3 — 컬렉션엔 batch size (application.yml) spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
컬렉션 fetch join + 페이징은 함께 못 씀

@OneToMany 컬렉션에 JOIN FETCH와 Pageable을 같이 쓰면, Hibernate가 메모리에서 페이징하며 경고를 띄웁니다(전체를 다 읽음). 컬렉션 N+1은 fetch join 대신 default_batch_fetch_size로 푸는 게 정석입니다. @ManyToOne·@OneToOne(to-one) 연관은 fetch join + 페이징이 안전합니다. (→ JPA 핸드북: N+1 심화)

JPA-06

페이징·정렬

목록을 통째로 읽지 말고 페이지 단위로. Spring Data가 Pageable 하나로 LIMIT·OFFSET·정렬·전체 개수를 한 번에 처리합니다.

Repository — Pageable 파라미터만 추가 Page<Member> findByNameContaining(String keyword, Pageable pageable);

호출 — PageRequest.of(page, size, sort)
Pageable pageable = PageRequest.of(0, 20, Sort.by(Sort.Direction.DESC, "id"));
Page<Member> page = memberRepository.findByNameContaining("kim", pageable);

List<Member> content = page.getContent();  // 현재 페이지 데이터
long total = page.getTotalElements();  // 전체 개수 (count 쿼리 자동)
int pages = page.getTotalPages();  boolean last = page.isLast();
요소의미
Pageable페이지 번호·크기·정렬을 담는 요청 객체
PageRequest.of(page, size)0-based 페이지 번호. of(0, 20)은 첫 20개
Page<T>데이터 + 전체 개수 + 페이지 메타. count 쿼리가 같이 나감
Slice<T>전체 개수 불필요할 때 (count 쿼리 생략, "더보기"식)
Sort.by(...)정렬. PageRequest.of(p, s, sort)로 합쳐 전달
page는 0부터

PageRequest.of의 페이지 번호는 0-based입니다 — 첫 페이지가 0. 프론트의 1-based 페이지와 어긋나기 쉬우니 변환을 한 곳에서 처리하세요. 전체 개수가 굳이 필요 없는 무한 스크롤은 Page 대신 Slice를 쓰면 count 쿼리를 아낍니다.

JPA-07

트랜잭션·영속성 컨텍스트

JPA의 마법이자 함정. 트랜잭션 안에서 조회한 엔티티는 영속 상태이고, 값을 바꾸면 커밋 시점에 자동으로 UPDATE가 나갑니다 — 이게 dirty checking입니다.

변경감지 — save() 호출이 없어도 UPDATE @Transactional
public void changeName(Long id, String newName) {
  Member member = memberRepository.findById(id)
    .orElseThrow(() -> new IllegalArgumentException("없는 회원"));

  member.changeName(newName);  // 필드만 변경
  // save() 호출 불필요 — 커밋 시 dirty checking으로 UPDATE 자동
}
상황해야 할 일
이미 영속 상태인 엔티티 수정변경감지 — 필드만 바꾸면 끝. save() 불필요
새 엔티티 저장 (PK 없음)save() 명시 호출 → INSERT
준영속/분리된 엔티티 반영save()(merge) 호출 필요
읽기 전용 메서드@Transactional(readOnly = true) — 스냅샷 생략, 최적화
변경감지는 트랜잭션 경계 안에서만

dirty checking은 영속성 컨텍스트가 살아 있는 동안, 즉 @Transactional 경계 안에서만 동작합니다. 트랜잭션 밖에서 받은 엔티티는 준영속(detached) 상태라 필드를 바꿔도 DB에 반영되지 않습니다. 그래서 수정 로직은 트랜잭션 안의 서비스 메서드에 두고, 거기서 조회한 엔티티를 바꿔야 합니다. LAZY 연관 접근도 같은 이유로 트랜잭션 밖이면 LazyInitializationException이 납니다. (→ JPA 핸드북: 영속성 컨텍스트)

JPA-08

낙관적 락

두 트랜잭션이 같은 행을 동시에 수정할 때, 마지막 저장이 앞 저장을 덮어쓰는(lost update) 문제를 막습니다. 락을 미리 걸지 않고 "충돌하면 그때 실패"시키는 방식.

@Version 필드 하나면 끝 @Entity
public class Product {
  @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;

  private int stock;

  @Version
  private Long version;  // UPDATE 때마다 자동 +1, WHERE version=? 로 충돌 검출
}
충돌 처리 — 예외 잡아 재시도/안내 @Transactional
public void decreaseStock(Long id, int qty) {
  Product p = productRepository.findById(id).orElseThrow();
  p.decrease(qty);  // 커밋 시 UPDATE ... SET version=version+1 WHERE id=? AND version=?
}
// 다른 트랜잭션이 먼저 바꿨다면 → OptimisticLockingFailureException
동작 원리와 한계

@Version 필드가 있으면 Hibernate가 UPDATE에 WHERE id=? AND version=?를 붙입니다. 그 사이 누가 먼저 수정했으면 version이 안 맞아 영향받은 행 0건 → OptimisticLockingFailureException(JPA 표준은 OptimisticLockException). 충돌 빈도가 낮을 때 적합하고, 충돌 시 재시도하거나 사용자에게 다시 시도를 안내해야 합니다. 충돌이 잦으면 비관적 락(@Lock(PESSIMISTIC_WRITE))을 검토하세요. (→ JPA 핸드북: 동시성 제어)

JPA-09

Auditing

생성·수정 시각을 매번 수동으로 세팅하지 마세요. JPA Auditing이 INSERT·UPDATE 시점에 자동으로 채워줍니다. 거의 모든 엔티티에 공통으로 들어가는 패턴.

1) 설정 활성화 (@Configuration 또는 메인 클래스) @Configuration
@EnableJpaAuditing
public class JpaConfig {}
2) 공통 베이스 클래스 — 상속해서 재사용 @MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseTimeEntity {

  @CreatedDate
  @Column(updatable = false)
  private LocalDateTime createdAt;

  @LastModifiedDate
  private LocalDateTime updatedAt;
}

// 사용: public class Member extends BaseTimeEntity { ... }
요소역할
@EnableJpaAuditingAuditing 기능 켜기 (설정에 한 번)
@EntityListeners(AuditingEntityListener.class)해당 엔티티에 Auditing 리스너 연결
@CreatedDateINSERT 시각 자동 기록 (updatable=false로 고정)
@LastModifiedDateUPDATE 시각 자동 갱신
@MappedSuperclass테이블 없는 공통 매핑 부모 — 필드만 상속
생성자·수정자도 자동으로

시각뿐 아니라 누가 만들고 바꿨는지도 @CreatedBy/@LastModifiedBy로 자동 기록할 수 있습니다 — AuditorAware<String> 빈을 등록해 현재 사용자를 공급하면 됩니다. 시간대는 LocalDateTime이 기본이지만, 다국가/UTC 저장이 필요하면 Instant를 검토하세요.

JPA-10

치트시트 — 자주 쓰는 한 줄

현장에서 반복되는 어노테이션·API. 외우기보다 "이게 있다"를 알아두고 필요할 때 찾으세요.

엔티티 선언@Entity
@Entity + @Id + 기본 생성자(protected)는 최소 3종 세트. @Table(name=...)로 테이블명 지정
PK 자동 생성@GeneratedValue
@GeneratedValue(strategy = GenerationType.IDENTITY) — DB AUTO_INCREMENT 위임. MySQL·PostgreSQL 기본 선택
N:1 (FK 주인)@ManyToOne
@ManyToOne(fetch = FetchType.LAZY) + @JoinColumn(name="member_id"). fetch는 항상 LAZY부터
1:N (거울)@OneToMany
@OneToMany(mappedBy="member") — mappedBy는 상대 엔티티의 필드명. 양방향 편의 메서드로 동기화
Repository 상속JpaRepository
interface MemberRepository extends JpaRepository<Member, Long> — <엔티티, PK타입> 순서
쿼리 메서드findBy / existsBy / countBy
findByEmail, findByNameContaining, existsByEmail, countByName — 이름 규칙으로 쿼리 자동 생성
직접 쿼리@Query + @Param
@Query("SELECT m FROM Member m WHERE m.email = :email") + @Param("email"). 네이티브는 nativeQuery = true
N+1 (to-one)fetch join
@Query("SELECT o FROM Order o JOIN FETCH o.member") 또는 @EntityGraph(attributePaths={"member"})
N+1 (컬렉션)batch_fetch_size
hibernate.default_batch_fetch_size: 100 — 컬렉션을 IN (?,?,...)으로 묶음. 페이징과 fetch join 동시 금지
페이징Pageable / Page
PageRequest.of(0, 20, Sort.by("id")) → Page<Member>. 페이지는 0-based, count 쿼리 자동
트랜잭션 경계@Transactional
서비스 메서드에 부착. 안에서 조회한 엔티티는 필드만 바꿔도 변경감지(dirty checking)로 UPDATE 자동. 읽기는 readOnly = true
동시 수정 충돌@Version
@Version 필드 추가 → UPDATE에 WHERE version=?. 충돌 시 OptimisticLockingFailureException → 재시도/안내
시각 자동 기록Auditing
@EnableJpaAuditing + @EntityListeners(AuditingEntityListener.class) + @CreatedDate/@LastModifiedDate
공통 필드 상속@MappedSuperclass
@MappedSuperclass로 베이스 클래스 만들어 createdAt/updatedAt 등 공통 필드를 모든 엔티티에 상속
학습 순서 제안

전부 한 번에 외우려 하지 마세요. 손에 먼저 붙일 것: @Entity+@Id · JpaRepository 상속 · 쿼리 메서드(findBy) · @Transactional 안에서의 변경감지. 이 네 가지로 foundational CRUD는 끝납니다. 그다음 실무에서 거의 반드시 만나는 게 N+1(JPA-05)이니, 여기까지 잡으면 실전 80%입니다. 각 항목의 왜가 궁금하면 JPA 핸드북에서 영속성 컨텍스트·프록시·flush 타이밍을 보세요.
