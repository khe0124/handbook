라우팅·NAT·Internet Gateway

라우팅은 패킷의 내비게이션입니다. 방화벽이 허용해도 route table에 목적지로 가는 경로가 없으면 패킷은 도착하지 않습니다.

RT-02

Longest Prefix Match

여러 route가 동시에 맞으면 가장 구체적인 CIDR이 선택됩니다.

Longest Prefix Match 
172.16.0.0/16  → vpn-gw-a
172.16.1.0/24  → vpn-gw-b

목적지 172.16.1.10은 /24가 더 구체적이므로
vpn-gw-b로 라우팅된다.
RT-03

IGW와 NAT

요소역할대표 사용
Internet GatewayVPC와 공인 인터넷을 연결public subnet의 inbound/outbound
NAT Gatewayprivate subnet의 outbound 인터넷 통신 대리패키지 설치, 외부 API 호출
공인 IP인터넷에서 도달 가능한 주소LB, bastion, NAT outbound IP
NAT는 inbound가 아니다

NAT Gateway는 private 서버가 인터넷으로 나갈 때 source IP를 바꿔주는 장치입니다. 인터넷에서 private 서버로 직접 들어오는 경로를 만들어주지 않습니다.

RT-04

return path

네트워크는 왕복입니다. 요청 경로만 맞고 응답 경로가 없으면 통신은 실패합니다.

상황요청 경로응답 경로 확인
우리 → 고객사 DBPrivate App → VPN → 고객 DB고객사 라우터가 우리 CIDR을 VPN으로 돌려보내는가
Private App → 외부 APIPrivate App → NAT → Internet외부 API가 NAT 공인 IP를 허용하는가
사용자 → LBInternet → IGW → LBLB 응답 경로와 보안 정책이 맞는가
RT-05

장애 패턴

Case study

VPN 터널이 UP이고 우리 route table도 고객사 CIDR을 VPN으로 보내는데 DB 접속이 안 될 수 있습니다. 고객사 쪽 return path가 우리 CIDR을 인터넷 기본 경로로 보내면 응답이 돌아오지 않습니다. VPN 장애에서는 양쪽 route table을 동시에 확인해야 합니다.

default route0.0.0.0/0
더 구체적인 경로가 없을 때 사용하는 마지막 경로입니다.
return path응답 경로
요청이 도착한 뒤 응답 패킷이 다시 source로 돌아가는 경로입니다.
