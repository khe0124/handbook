OP-00

전체 흐름

인프라·운영은 네트워크, 배포, 모니터링을 따로 외우는 영역이 아니다. 사용자의 요청은 DNS → CDN/WAF → Load Balancer → App → DB 경로를 지나고, 변경은 CI/CD → artifact → environment → runtime → observability → incident response로 검증된다. 이 두 흐름을 한 장에 연결해야 장애 질문과 시스템 설계 질문에 깊이가 생긴다.

판단 질문검증 증거운영 리스크
전체 흐름이 실제 서비스 경로에서 어떤 책임을 갖는가? 요청 경로 다이어그램, DNS record, WAF/LB rule, ingress route, DB connection path, trace sample 장애 시 네트워크, 앱, 데이터 계층 중 어디서 증거를 모을지 정해지지 않음
변경되면 어떤 사용자 영향과 rollback 조건이 생기는가? 변경 티켓, artifact digest, config diff, 배포 후 SLI, rollback drill 기록, runbook 링크 복구 절차가 담당자 기억에 의존하고, 되돌린 뒤 검증 기준이 비어 있음
운영 경계가 팀 사이에서 어떻게 인수인계되는가? owner matrix, on-call escalation, RACI, 서비스 카탈로그, 운영 인수 체크리스트 플랫폼, 애플리케이션, 보안 팀의 책임 경계가 모호해 MTTR이 길어짐
OP-01

학습 순서

먼저 요청 경로를 그린다. 그다음 VPC, subnet, route table, NAT, ingress, egress를 붙인다. 이후 DNS, TLS, VPN, Private Endpoint 같은 연결 경계를 이해하고, CI/CD, 컨테이너, IaC, SLO, rollback, DR을 운영 루프로 연결한다.

판단 질문검증 증거운영 리스크
네트워크 지식이 실제 요청 실패를 설명하는가? route table, security group, NACL, private DNS, TLS certificate chain, LB target health 보안 그룹을 열었는데도 private DNS, NAT, route, certificate 중 어디가 문제인지 분리하지 못함
배포 지식이 운영 안전장치까지 연결되는가? pipeline stage, artifact digest, approval gate, canary metric, rollback command, post-deploy report CI/CD는 설명하지만 실패한 배포를 어떤 지표로 멈출지 답하지 못함
관측 가능성이 의사결정 기준까지 이어지는가? SLI 정의, dashboard panel, alert rule, runbook step, incident review action item dashboard는 많지만 사용자 영향과 우선순위를 결정하는 핵심 지표가 없음
OP-02

운영자가 보는 산출물

운영 문서는 아키텍처 다이어그램만으로 끝나지 않는다. owner, SLO, dashboard, alert, runbook, rollback plan, RPO/RTO, 변경 이력, 운영 인수 체크리스트가 있어야 다음 사람이 같은 판단을 반복할 수 있다.

판단 질문검증 증거운영 리스크
서비스를 넘겨받은 운영자가 15분 안에 상태를 판단할 수 있는가? 서비스 카탈로그, owner, dependency map, SLO, dashboard, alert, runbook, 최근 변경 링크 신규 담당자가 첫 장애에서 Slack 검색과 구두 질문으로만 대응함
알림은 사람을 깨울 만큼 정확한가? alert rule, threshold 근거, paging 정책, 지난 30일 noise/false positive, mute 이력 증상 알림과 원인 알림이 섞여 on-call이 매번 dashboard를 처음부터 탐색함
복구 절차가 리허설 가능한 수준인가? runbook 명령, 접근 권한, 예상 소요 시간, 되돌림 조건, 마지막 drill 결과 문서에는 "재시작"만 있고 어느 노드, 어떤 순서, 어떤 검증을 할지 없음
OPERATIONS ARTIFACT SET 서비스별 최소 산출물은 owner, dependency map, SLO, dashboard, alert, runbook, rollback plan, DR plan, cost owner, access owner입니다. 문서가 있다는 사실보다 중요한 것은 각 산출물이 같은 서비스 이름과 같은 책임자로 연결되어 있는지입니다.
산출물포함해야 할 내용운영 리뷰 질문
Dashboard사용자 영향 SLI, saturation, dependency error, deploy marker, cost signal이 화면만 보고 장애 심각도와 다음 행동을 정할 수 있는가?
Alert증상 기준, burn-rate 기준, paging 대상, suppress 조건, runbook 링크알림을 받은 사람이 왜 지금 깨워졌는지 설명할 수 있는가?
Runbook확인 명령, 완화 명령, rollback/failover, 검증 쿼리, escalation 기준권한이 있는 다른 엔지니어가 담당자 없이 실행할 수 있는가?
Post-incidenttimeline, impact, detection gap, decision log, corrective action, owner/date재발 방지가 코드, 설정, 프로세스 중 어디에 반영되는가?
OP-03

면접 답변 구조

인프라 질문을 받으면 도구 이름보다 경로, 경계, 실패 모드, 검증 증거를 먼저 말한다. 예를 들어 '배포를 어떻게 운영했나요?'에는 pipeline stage, artifact digest, environment config, smoke test, metric gate, rollback 조건을 순서대로 답한다.

판단 질문검증 증거운영 리스크
답변이 도구 나열이 아니라 운영 판단으로 이어지는가? 요청 경로, 실패 모드, 관측 지표, 변경 증거, 복구 절차, 재발 방지 사례 Kubernetes, Terraform, Grafana 이름은 말하지만 어떤 장애를 줄였는지 설명하지 못함
trade-off를 수치와 근거로 말하는가? SLO target, error budget, RTO/RPO, cost delta, latency percentile, incident timeline 안정성, 비용, 보안을 모두 좋게 만들었다고만 말하고 선택의 비용을 숨김
Senior 레벨의 책임 범위를 보여주는가? 운영 표준화, review gate, drill 운영, cross-team escalation, action item 추적 개별 작업 경험은 있지만 팀이 반복 가능한 운영 체계를 갖추게 한 증거가 없음
MODEL

개념 모델

인프라·운영 로드맵은 단일 지식 항목이 아니라 운영 판단의 일부다. 아래 표처럼 개념을 책임, 확인 지표, 실패 신호로 나누면 면접 답변과 실제 장애 대응이 모두 선명해진다.

CHECK

실무 체크리스트

작업 전에는 무엇을 바꿀지보다 무엇을 깨뜨릴 수 있는지를 먼저 본다. 체크리스트는 리뷰와 운영 인수에서 그대로 사용할 수 있어야 한다.

  • 요청 흐름과 배포 흐름을 한 장에 연결한다.
  • owner, SLO, dashboard, alert, runbook, rollback plan을 확인한다.
  • 단일 실패 지점과 수동 복구 지점을 표시한다.
  • 운영 인수 전에 Rollback drill과 Restore drill을 실행한다.
작업 단계필수 산출물차단 기준
설계목적, 영향 범위, 비목표, owner영향 범위가 사용자·시스템·비용 기준으로 설명되지 않음
변경diff, 승인 기록, 검증 명령production 변경과 staging 검증의 차이가 설명되지 않음
복구rollback 또는 forward fix, 담당자, 예상 시간실패 시 되돌릴 수 없는데 사전 승인과 공지가 없음
SLO / Burn-rate evidence

SLO는 "좋아 보이는 평균"이 아니라 우선순위를 정하는 계약이다. 장애 판단에서는 현재 오류율이 예산을 얼마나 빠르게 태우는지, 그 속도가 몇 시간 안에 월 예산을 소진하는지, 어떤 완화책이 burn-rate를 낮췄는지를 남긴다.

항목운영 기준증거 예시
SLI사용자가 체감하는 성공률, 지연, freshness, durability 중 하나로 정의한다.HTTP 5xx 비율, p95 latency, queue age, payment success rate
SLO기간과 제외 조건을 포함한다. 예: 30일 rolling 99.9% 성공률.SLO config, 제외 이벤트 승인, 제품/고객 영향 설명
Fast burn짧은 창에서 예산을 급격히 태우면 즉시 paging한다.5분/1시간 burn-rate, 관련 deploy marker, dependency error
Slow burn긴 창에서 지속적으로 예산을 태우면 근무 시간 내 owner가 처리한다.6시간/3일 burn-rate, trend chart, backlog action item
Error budget policy예산 소진 시 feature release, risky migration, capacity 축소를 멈출 조건을 정한다.release freeze 기록, exception 승인, 후속 action owner/date

CASE

장애/운영 시나리오

신규 서비스를 런칭하기 전 운영 준비도 리뷰를 진행한다. 요청 경로는 DNS, CDN/WAF, Load Balancer, App, DB로 그려 보고, 변경 경로는 CI/CD, artifact, environment, runtime, observability, incident response로 확인한다. 빠진 항목은 런칭 후 장애가 아니라 런칭 전 차단 이슈로 다룬다.

Triage frame

증상 → 영향 범위 → 최근 변경 → 계층별 증거 → 완화책 → 복구 검증 → 재발 방지 순서로 기록한다. 이 순서가 없으면 원인 분석이 사람마다 달라진다.

처음 10분다음 30분종료 조건
사용자 영향과 최근 변경을 확인한다.계층별 증거를 모아 완화책을 실행한다.지표 회복, 재현 불가, owner 확인, 기록 완료
Incident timeline template기록할 내용판단 기준

T+05 영향 범위고객군, 지역, endpoint, error rate, latency, 데이터 손실 가능성Sev 등급과 커뮤니케이션 채널을 확정한다.
T+10 최근 변경배포, config, feature flag, infra change, dependency incident상관관계와 원인을 구분하고 rollback 후보를 정한다.
T+20 완화책rollback, traffic shift, queue pause, rate limit, failover, feature disable완화책마다 예상 효과와 되돌림 조건을 남긴다.
T+40 검증SLI 회복, synthetic check, business KPI, support ticket trend지표가 회복되어도 누락된 고객 영향이 없는지 확인한다.
T+60 종료incident commander 승인, customer update, action item owner/date재발 방지 항목이 측정 가능한 변경으로 닫히는지 확인한다.
DR DRILL / FAILBACK CHECKLIST 1. RTO/RPO와 drill 범위를 선언한다.
2. backup snapshot, replica lag, DNS TTL, traffic switch 권한을 확인한다.
3. read-only 또는 degraded mode 전환 기준을 정한다.
4. failover 후 쓰기 경로, queue 재처리, idempotency를 검증한다.
5. failback 전 데이터 차이, sequence, cache, search index, external webhook을 대조한다.
6. 원복 후 원래 region의 write 차단 해제와 monitoring 기준선을 갱신한다.

RISK

자주 틀리는 판단

운영 품질은 아는 개념보다 틀린 가정을 빨리 찾는 능력에서 갈린다. 아래 신호가 보이면 설계나 답변을 다시 점검한다.

  • 아키텍처 다이어그램은 있지만 알림 owner가 없다.
  • 배포 절차는 있지만 rollback 검증이 없다.
  • SLO가 없어서 장애와 단순 오류를 같은 우선순위로 처리한다.
  • 운영 인수 문서가 사람 이름과 구두 설명에 의존한다.
잘못된 판단대신 볼 증거수정 방향
CPU가 낮으니 capacity 문제는 아니다.p95/p99 latency, queue depth, connection pool, throttling, dependency saturation리소스별 병목이 아니라 사용자 경로의 saturation을 본다.
알림이 울렸으니 관측은 충분하다.false positive rate, unhandled incident, alert-to-action mapping, runbook success알림마다 사람이 취할 행동과 종료 조건을 연결한다.
백업이 있으니 DR은 준비됐다.restore duration, restored data validation, failback plan, dependency readiness복구 가능한 백업인지 drill로 증명한다.
권한은 최소 권한으로 보인다.last used, wildcard, condition, exception expiry, audit trail정책 문구가 아니라 실제 사용과 예외 만료를 확인한다.
Cost spike analysis

비용 증가는 청구서가 아니라 운영 이벤트로 다룬다. 배포, 트래픽, 재시도, 로그 폭증, 데이터 전송, managed service autoscaling, 예약 할인 누락을 같은 timeline에서 대조해야 원인을 좁힐 수 있다.

분석 축확인할 증거조치 기준
시간 상관관계cost anomaly 시각, deploy marker, traffic spike, incident timeline비용 상승이 변경 직후면 rollback 또는 feature flag disable 후보로 둔다.
사용량 단위request당 비용, tenant당 비용, job당 비용, GB 전송당 비용총액보다 unit cost가 변했는지 먼저 본다.
재시도/루프5xx, retry count, queue re-drive, cron overlap, batch duration장애 완화책이 비용 폭증을 만들고 있으면 rate limit 또는 pause를 건다.
로그/관측log volume, trace sample rate, cardinality, retention changehigh-cardinality label과 debug logging을 긴급 차단한다.
데이터 전송cross-zone/region egress, NAT gateway bytes, CDN origin fetch캐시 미스, region mismatch, private endpoint 누락을 점검한다.
구매/용량reservation coverage, savings plan, idle resource, rightsizing report지속 사용량은 구매 전략으로, 일시 폭증은 guardrail로 처리한다.

면접 답변 템플릿

인프라·운영은 요청 경로와 변경 경로를 함께 보는 영역이라고 답합니다. 사용자의 요청은 DNS부터 DB까지 계층별로 흐르고, 코드 변경은 artifact와 환경 설정, 런타임, 관측, 장애 대응 절차를 거쳐 안전해집니다. 저는 각 단계의 실패 모드와 검증 증거를 연결해 운영 준비도를 판단하겠습니다.

ANSWER SHAPE 1. 한 줄 결론
2. 핵심 개념과 책임 경계
3. 실무에서 깨지는 지점
4. 검증 명령 또는 관측 증거
5. rollback, runbook, 운영 인수 기준
질문좋은 답변의 뼈대강한 증거
장애 대응을 어떻게 운영했나요?영향 범위, incident commander, timeline, 완화책, SLO 회복, 재발 방지 순서로 말한다.MTTD/MTTR 변화, timeline 예시, action item closure, 알림 개선 결과
SLO를 어떻게 잡았나요?사용자 여정에서 SLI를 고르고, business risk에 맞춰 target과 error budget policy를 정했다고 답한다.burn-rate alert, 제외 조건, release freeze 기준, 고객 영향 사례
DR을 어떻게 검증했나요?백업 존재가 아니라 복구 시간, 데이터 정합성, failback까지 drill로 검증했다고 답한다.실측 RTO/RPO, restore 검증 쿼리, failback 체크리스트, 권한 gap 수정
비용 최적화를 어떻게 했나요?총액 절감보다 unit cost, workload pattern, guardrail, ownership을 기준으로 설명한다.비용 anomaly timeline, unit cost 개선, idle 제거, reservation coverage
보안/네트워크 운영을 어떻게 증명하나요?정책 선언이 아니라 접근 경로, 권한 사용, audit log, 예외 만료로 증명한다고 답한다.IAM diff, flow log, access review, exception ticket, 만료 자동화
