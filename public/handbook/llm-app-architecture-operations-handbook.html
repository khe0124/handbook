LLM 앱 아키텍처와
운영

LLM 앱은 일반 API보다 지연과 비용 변동이 크고, 품질 실패가 500 에러로 드러나지 않습니다. 그래서 관측과 평가가 운영의 일부가 됩니다.

OPS-01

운영 가능한 LLM 앱의 정의

LLM 앱의 성공 기준은 응답 성공 여부가 아니라 latency, cost, quality, safety, reliability를 동시에 만족하는 것입니다. “실패는 500만이 아니다”가 운영의 출발점입니다.

LLM APP ARCHITECTURE ONLINE PATH: client → API gateway → auth/tenant check → request validator → prompt builder → retrieval/tool layer → model_gateway → output validator → response store
ASYNC PATH: client → job create → queue → worker → model_gateway → artifact store → progress update → webhook
CONTROL PATH: prompt registry → eval runner → eval_gate → canary_rollout → observability pipeline → alert → runbook → prompt_rollback
컴포넌트책임운영 지표
Model gatewayprovider 추상화, retry, timeout, fallback_routingprovider error rate, p95 latency
Prompt registryprompt_version, owner, changelogrelease success, rollback count
Retrieval service권한 filter, reranking, citationretrieval_empty_rate, recall@k
Output validatorschema parse, policy checkschema_failure_rate
Eval pipelinegolden set, regression, online feedbackgroundedness, judge score

OPS-03

model_gateway와 Resilience

모델 제공자 장애, 지연, rate limit은 정상적으로 발생합니다. gateway는 timeout, retry_budget, circuit_breaker, fallback_routing을 한곳에서 관리해야 합니다.

DEPTH

운영 가능한 LLM 앱의 증거 설계

MARKER: concept-depth — Request lifecycle as reconstructable evidence

request envelope는 로그 장식이 아닙니다. 답변이 틀렸을 때 같은 입력, 같은 prompt_version, 같은 model_version, 같은 retrieval index로 사고를 재현하기 위한 최소 증거입니다. 비용이 급증했을 때 어떤 feature와 tenant에서 token이 늘었는지 추적하고, 권한 사고가 났을 때 어떤 document_acl_version과 cache_key가 쓰였는지 확인하며, prompt 변경 후 품질이 떨어졌을 때 어떤 release가 원인인지 찾는 기반입니다.

Online path, async path, control path는 상태 소유권도 달라야 합니다. Online path는 사용자의 대기 시간과 partial UI 상태를 관리하고, async path는 job state와 idempotency를 관리하며, control path는 prompt registry, eval gate, canary, rollback을 관리합니다. 이 경계를 섞으면 사용자는 완료되지 않은 결과를 완료로 보거나, worker 재시도로 같은 tool action이 두 번 실행되거나, prompt 변경이 평가 없이 배포될 수 있습니다.

MARKER: failure-driven — LLM failure taxonomy

LLM 운영에서 모든 실패를 retry로 해결하려 하면 비용 폭주가 납니다. transport failure나 일시적 provider failure는 제한된 retry가 가능하지만, schema failure는 targeted repair가 필요하고, grounding failure는 retrieval/context/prompt를 봐야 하며, safety failure는 즉시 차단하고 regression case를 추가해야 합니다. cost failure는 quota와 routing 문제이고, latency failure는 provider 시간·queue depth·token length를 분해해야 합니다.

MARKER: artifact-ready — Failure class decision table

transport_failure: retry=yes, fallback=maybe, alert=error_rate
provider_failure: retry=limited, fallback=yes, alert=circuit_breaker
schema_failure: retry=repair_once, fallback=review_queue, alert=schema_failure_rate
grounding_failure: retry=no, fallback=no_answer, alert=groundedness_drop
safety_failure: retry=no, fallback=block, alert=policy_violation
cost_failure: retry=no, fallback=cheaper_model, alert=cost_per_successful_answer
latency_failure: retry=no, fallback=async_or_streaming, alert=p95_latency

Observability는 지표를 많이 모으는 일이 아니라 의사결정으로 연결하는 일입니다. `retrieval_empty_rate`가 오르고 `groundedness`가 내려가면 모델보다 인덱스 freshness나 권한 필터를 먼저 의심해야 합니다. `schema_failure_rate`가 오르면 max output, schema evolution, repair prompt를 봐야 합니다. `retry_count`와 `cache_hit_rate`가 동시에 나빠지면 비용 알림이 오기 전에 quota와 cache key를 확인해야 합니다.

MARKER: failure-driven — 비용 최적화가 품질 퇴행을 숨기는 실패

운영 단계에서 비용을 줄이려고 retrieved chunk 수, context compression, model tier, completion budget을 동시에 낮추면 겉으로는 latency와 cost가 좋아질 수 있습니다. 하지만 citation coverage가 줄고 no-answer가 늘거나, 반대로 근거 없는 짧은 답변이 많아지면 제품 품질은 퇴행합니다. 이 실패는 비용 대시보드만 보면 개선처럼 보이는 점이 위험합니다. LLM 앱 운영에서는 cost, latency, quality, safety를 같은 release gate에서 봐야 합니다. 비용 절감 변경은 반드시 golden dataset, groundedness, refusal accuracy, 사용자 fallback 비율과 함께 비교되어야 하며, p95 latency 개선만으로 릴리스하면 안 됩니다.
