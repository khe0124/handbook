PKG-01

두 범주를 구분하라

제일 먼저 잡을 것: "라이브러리를 설치하는 도구"와 "Node 자체의 버전을 바꾸는 도구"는 다른 범주입니다. 둘 다 "매니저"라 불려서 섞이지만, 푸는 문제가 완전히 다릅니다.

범주하는 일도구설정 파일
패키지 매니저프로젝트 의존성(라이브러리) 설치·관리npm · pnpm · yarnpackage.json + lockfile
런타임 버전 매니저Node(런타임) 자체의 버전 설치·전환nvm · mise · fnm · Volta.nvmrc · .mise.toml

React 프로젝트로 비유하면 이렇습니다 — npm install react는 react 라이브러리를 끌어오는 일(패키지 매니저), mise use node@22는 이 프로젝트를 Node 22에서 돌리겠다는 일(버전 매니저). 한쪽은 node_modules를 채우고, 다른 쪽은 node 명령이 가리키는 실행 파일을 바꿉니다.

JAVA 대응 — Gradle 가이드와 짝

Java 세계로 옮기면 정확히 같은 두 축입니다. 패키지 매니저 = Gradle/Maven(build.gradle의 dependencies 블록이 라이브러리를 끌어옴). 런타임 버전 매니저 = mise/SDKMAN(JDK 17 vs 21 전환). 그래서 mise 하나로 Node와 Java 버전을 동시에 고정할 수 있습니다(→ PKG-08). 의존성 해석과 런타임 전환은 별개의 일이라는 감각이 핵심입니다.

PKG-02

npm 기본

npm은 Node에 기본 내장된 패키지 매니저입니다. 새 도구를 배우기 전에, 모든 매니저가 공유하는 기반인 package.json·semver·lockfile부터 정확히 잡으세요.

2.1package.json과 scripts

package.json은 프로젝트의 명세서입니다 — 의존성 목록, 실행 스크립트, 메타데이터. 익숙하겠지만 dependencies(런타임에 필요)와 devDependencies(빌드·테스트에만 필요)의 구분이 배포 산출물 크기를 좌우합니다.

PACKAGE.JSON — 핵심 필드 {
  "scripts": { "dev": "next dev", "build": "next build" },
  "dependencies": { "react": "^18.3.0" },
  "devDependencies": { "typescript": "~5.4.0" }
}
실행: npm run dev  (dev/start/test 는 run 생략 가능 — npm test)

2.2semver — ^ 와 ~ 의 차이

버전 앞 기호는 "업데이트를 어디까지 허용하는가"의 범위 지정입니다. semver는 MAJOR.MINOR.PATCH (예: 18.3.1) — MAJOR는 호환성 깨짐, MINOR는 기능 추가, PATCH는 버그 수정.

표기허용 범위의미
^18.3.018.3.0 ~ 18.x.x (19 미만)MINOR·PATCH 업데이트 허용 (npm 기본)
~18.3.018.3.0 ~ 18.3.x (18.4 미만)PATCH만 허용 (더 보수적)
18.3.018.3.0 고정정확히 그 버전만
* / latest아무 버전위험 — 재현성 깨짐, 쓰지 말 것

2.3install vs ci

명령동작 / 언제
npm installpackage.json 기준으로 설치 — 필요하면 lockfile을 갱신. 로컬 개발에서 의존성 추가·변경할 때
npm install pkg패키지 추가. -D는 devDependencies로, -g는 전역
npm cilockfile 그대로 설치 — node_modules를 지우고 lock대로만. 빠르고 결정적. CI·배포 전용
npm run <script>scripts에 정의한 명령 실행
npx <pkg>설치 없이 패키지 실행 (예: npx create-next-app)
install과 ci를 헷갈리지 말 것

npm install은 상황에 따라 package-lock.json을 고쳐 씁니다 — semver 범위 안에 새 버전이 나왔으면 올려버릴 수 있습니다. CI에서 이걸 쓰면 "어제는 됐는데 오늘 빌드가 깨짐"이 생깁니다. CI·배포에서는 항상 npm ci — lockfile과 package.json이 어긋나면 아예 에러를 내고, lock에 적힌 버전만 정확히 설치합니다.

PKG-03

pnpm

pnpm은 npm과 같은 일을 하되 디스크와 속도에서 다른 설계를 택했습니다. 명령어는 거의 호환돼서 갈아타는 비용이 작습니다.

3.1왜 빠르고 디스크를 아끼나

핵심은 content-addressable store + hardlink입니다. npm은 프로젝트마다 node_modules에 패키지를 통째로 복사하지만, pnpm은 전역 store(~/.local/share/pnpm/store) 한 곳에만 실제 파일을 두고, 각 프로젝트의 node_modules에는 그 파일로의 hardlink만 겁니다. 같은 버전을 10개 프로젝트가 써도 디스크에는 한 벌만 존재합니다.

DISK MODEL — npm vs pnpm npm : projectA/node_modules/react (복사본)
       projectB/node_modules/react (또 복사본)

pnpm: ~/.pnpm-store/react (원본 1벌)
       projectA/node_modules/react ──▶ hardlink
       projectB/node_modules/react ──▶ hardlink

3.2엄격한 node_modules — 유령 의존성 차단

npm은 의존성을 node_modules 최상단에 평평하게(flat) 깝니다. 그 부작용으로, package.json에 직접 선언하지 않은 패키지도 import가 됩니다 — phantom dependency(유령 의존성). 어느 날 의존성 트리가 바뀌면 그 import가 소리 없이 깨집니다. pnpm은 선언한 패키지만 접근 가능하게 막아서 이 함정을 원천 차단합니다.

3.3npm → pnpm 명령 대응

npmpnpm의미
npm installpnpm install전체 설치
npm install reactpnpm add react패키지 추가
npm install -D tspnpm add -D tsdevDependency 추가
npm uninstall reactpnpm remove react패키지 제거
npm run devpnpm dev스크립트 실행 (run 생략 가능)
npm cipnpm install --frozen-lockfilelockfile 고정 설치
lockfile은 pnpm-lock.yaml

pnpm은 pnpm-lock.yaml을 씁니다 — npm의 package-lock.json과 형식이 다릅니다. 둘은 호환되지 않으니 한 프로젝트에 둘이 공존하면 안 됩니다(→ PKG-04). 기존 npm 프로젝트를 옮길 때는 pnpm import로 package-lock.json을 읽어 pnpm-lock.yaml을 만들면 버전이 보존됩니다.

PKG-04

npm vs pnpm vs yarn

셋 다 같은 일을 합니다 — 라이브러리 설치. 선택은 취향이 아니라 팀 합의의 문제입니다. 한 프로젝트엔 반드시 하나만.

기준npmpnpmyarn (berry)
설치 속도보통빠름빠름
디스크 사용많음적음 (hardlink)중간 (PnP는 적음)
엄격성느슨 (유령 의존성)엄격엄격 (PnP)
모노레포workspaces 지원강력강력
설치Node 내장별도 (corepack)별도 (corepack)
lockfilepackage-lock.jsonpnpm-lock.yamlyarn.lock
lockfile은 섞이지 않는다

세 매니저의 lockfile은 서로 호환되지 않습니다. 한 사람은 npm install, 다른 사람은 pnpm install을 쓰면 package-lock.json과 pnpm-lock.yaml이 둘 다 생기고 의존성 버전이 갈라집니다 — "내 PC에선 되는데"의 전형적 원인. 팀은 매니저를 하나로 통일하고, package.json의 packageManager 필드(예: "packageManager": "pnpm@9.1.0")에 박아 corepack이 강제하게 하세요.

실무 권고

새 프로젝트라면 pnpm을 기본값으로 두는 게 합리적입니다 — 빠르고, 디스크를 아끼고, 유령 의존성을 막아줍니다. 기존 npm 프로젝트를 굳이 바꿀 필요는 없습니다. 중요한 건 도구의 우열이 아니라 팀 전체가 같은 하나를 쓰는 것입니다.

PKG-05

lockfile과 재현성

lockfile은 "누가 언제 설치해도 같은 의존성 트리가 나오게" 만드는 장치입니다. "내 PC만 됨"을 막는 가장 직접적인 도구.

package.json의 ^18.3.0은 범위일 뿐 정확한 버전이 아닙니다. 그래서 설치 시점에 따라 18.3.0이 올 수도, 18.4.2가 올 수도 있습니다. lockfile은 실제로 설치된 정확한 버전과 전체 의존성 트리(transitive dependency 포함)를 기록해, 다음 설치가 그대로 재현되게 합니다.

package.json
^18.3.0 (범위) → 해석 → lockfile
18.3.1 (정확) → 재현 → 모든 환경
동일 트리
해야 할 것이유
lockfile을 git에 커밋팀·CI가 같은 버전을 받게 — 재현성의 전제
CI에선 npm ci / pnpm install --frozen-lockfilelockfile을 고치지 않고 그대로만 설치 — 결정적 빌드
의존성 추가·변경 후 lockfile도 함께 커밋package.json과 lock의 어긋남 방지
lockfile을 무시하면

lockfile을 .gitignore에 넣거나 CI에서 npm install을 쓰면, transitive dependency의 PATCH 하나가 조용히 올라와 빌드를 깨거나 — 더 나쁘게는 — 공급망 공격(악성 버전 주입)에 노출됩니다. frozen-lockfile은 lockfile과 package.json이 불일치하면 설치를 거부해 이런 표류를 막습니다. lockfile은 빌드 산출물이 아니라 소스의 일부로 다루세요.

PKG-06

워크스페이스(모노레포) 간단

한 저장소 안에 여러 패키지(앱·라이브러리)를 두고 함께 관리하는 구조가 모노레포입니다. npm·pnpm 모두 workspaces로 지원합니다 — 여기서는 개념만.

예를 들어 웹 앱, 모바일 앱, 둘이 공유하는 UI 컴포넌트 라이브러리를 한 저장소에 둡니다. workspaces를 쓰면 의존성을 루트에서 한 번에 설치하고, 패키지끼리 로컬 경로로 서로를 참조할 수 있습니다(npm publish 없이).

MONOREPO — 구조와 선언 my-repo/
  package.json ← 루트 (workspaces 선언)
  packages/
    ui/ package.json
    web/ package.json (ui 를 참조)

# npm: 루트 package.json
"workspaces": ["packages/*"]
# pnpm: 별도 pnpm-workspace.yaml 파일에 선언
매니저워크스페이스 선언
npmpackage.json의 "workspaces": ["packages/*"]
pnpm별도 pnpm-workspace.yaml 파일에 packages: 목록
언제 쓰나 — 그리고 언제 멈추나

여러 패키지가 긴밀히 함께 바뀔 때(공유 컴포넌트·내부 SDK) 모노레포가 빛납니다. 규모가 커지면 빌드 캐시·태스크 그래프를 다루는 Turborepo·Nx 같은 도구를 얹습니다 — 다만 그건 진짜 필요해질 때 도입하세요. 작은 프로젝트에 처음부터 모노레포 도구를 까는 건 과설계입니다.

PKG-07

nvm — Node 버전 매니저

여기부터는 다른 범주 — 라이브러리가 아니라 Node 런타임 자체의 버전을 바꾸는 도구입니다. nvm이 가장 오래되고 널리 알려진 선택지입니다.

프로젝트 A는 Node 18, 프로젝트 B는 Node 22를 요구할 때, 시스템에 Node를 하나만 깔아두면 충돌합니다. nvm은 여러 Node 버전을 깔아두고 셸에서 전환하게 해줍니다.

명령의미
nvm install 22Node 22 설치
nvm use 22현재 셸에서 22로 전환
nvm alias default 22새 셸의 기본 버전을 22로
nvm ls설치된 버전 목록
nvm use현재 디렉터리의 .nvmrc에 적힌 버전으로
.NVMRC — 프로젝트 버전 고정 # 프로젝트 루트의 .nvmrc 파일 내용:
22

# 진입 후 직접 적용해야 함:
$ nvm use → Now using node v22.x
nvm의 단점 — 느림·수동

nvm은 셸 함수로 구현돼 있어 ~/.zshrc에서 매번 로드되며 셸 시작이 눈에 띄게 느려집니다. 또 .nvmrc가 있어도 디렉터리에 들어갈 때 자동 전환이 안 됩니다 — 매번 nvm use를 직접 쳐야 합니다(자동화하려면 별도 셸 훅 설정 필요). 이 두 가지 불편 때문에 fnm·mise 같은 더 빠른 대안으로 옮기는 흐름이 있습니다(→ PKG-08).

PKG-08

mise — 폴리글랏 버전 매니저

mise는 Node만이 아니라 여러 언어의 런타임을 하나로 관리합니다 — nvm·pyenv·rbenv·SDKMAN을 한 도구로 대체. 디렉터리 진입 시 자동 전환이 기본이라 nvm의 두 단점을 동시에 해결합니다. 풀스택 프로젝트에서 Node와 Java 같은 여러 런타임을 함께 고정할 때 특히 유용합니다.

8.1설치·전환 명령

명령의미
mise use node@22현재 프로젝트에 Node 22 고정 (.mise.toml에 기록)
mise install.mise.toml에 적힌 모든 런타임 설치
mise use -g node@22전역 기본값으로 (-g)
mise ls설치·활성 버전 목록
mise exec -- node -vmise 환경에서 일회성 실행

8.2.mise.toml — Node·Java 한 파일에

nvm은 Node만, pyenv는 Python만 봅니다. mise는 한 파일에 여러 런타임을 선언해 — 풀스택으로 확장 중이라면 Node와 Java 버전을 한 곳에 고정할 수 있습니다.

.MISE.TOML — 프로젝트 루트 [tools]
node = "22"
java = "temurin-21"

# 이 디렉터리에 들어가면 node·java 가 자동으로 이 버전

8.3자동 전환 — activate

핵심 한 줄. ~/.zshrc에 아래를 넣으면, 디렉터리에 들어가는 순간 그 프로젝트의 .mise.toml을 읽어 버전이 자동으로 바뀝니다 — nvm처럼 매번 명령을 칠 필요가 없습니다.

~/.ZSHRC — 자동 전환 활성화 eval "$(mise activate zsh)"

# chezmoi 로 .zshrc 관리 중이면:
$ chezmoi edit ~/.zshrc → 위 줄 추가 → chezmoi apply

덤으로 mise tasks로 프로젝트 스크립트(빌드·린트 등)도 관리할 수 있습니다 — npm scripts와 비슷하지만 언어에 구애받지 않습니다. 여기서는 존재만 알아두세요.

nvm 정리하고 mise로

nvm과 mise를 동시에 켜두면 둘 다 node PATH를 건드려 충돌합니다. mise로 옮기면 ~/.zshrc에서 nvm 로드 줄을 빼세요. mise는 nvm의 .nvmrc와 .node-version도 읽어주므로, 기존 프로젝트의 .nvmrc를 굳이 지우지 않아도 동작합니다 — 다만 새 프로젝트는 .mise.toml로 통일하는 게 깔끔합니다.
