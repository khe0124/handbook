응용 수학과 측정: 엔지니어링 판단의 숫자화

측정은 그래프를 만드는 일이 아니라 불확실성을 줄이고 결정을 방어하는 일입니다. 확률, 표본, 신뢰구간, latency percentile, 실험 설계, 벡터 유사도, fuzz/benchmark 검증을 실무 증거 관점으로 정리합니다.

AUDIENCE : 제품·플랫폼 엔지니어 · 실험/성능/검증 공통 · UPDATED 2026-06
AM-01

APPLIED MEASUREMENT SPINE

좋은 측정은 먼저 질문을 좁힙니다. "빨라졌나"보다 "어떤 사용자군의 p95 latency가 얼마나 줄었고, 오차와 회귀 위험은 얼마인가"가 더 나은 질문입니다.

APPLIED MEASUREMENT SPINE
decision question → metric definition → sampling plan → baseline
→ experiment/control → confidence interval → error analysis
→ regression guard → decision record
개념핵심 질문산출물
확률불확실한 사건을 어떻게 수량화하는가분포, 기대값, 분산
표본전체를 대표하는 데이터를 어떻게 뽑는가sampling frame, bias note
통계량표본에서 무엇을 요약하는가mean, median, percentile
추론표본 결과로 모집단을 어디까지 말할 수 있는가confidence interval
검증코드가 넓은 입력에서 성질을 지키는가property/fuzz/benchmark report
AM-02

PROBABILITY SAMPLING DRILL

확률은 장기 빈도 또는 믿음의 정도를 다루는 언어입니다. 엔지니어링에서는 "이 관찰이 우연인지", "이 표본이 전체를 대표하는지", "위험이 얼마나 자주 발생하는지"를 묻는 데 쓰입니다.

PROBABILITY SAMPLING DRILL
population: 알고 싶은 전체
sampling frame: 실제로 뽑을 수 있는 목록
sample: 관측한 일부
statistic: sample에서 계산한 값
parameter: population의 진짜 값
분포/개념사용 감각실무 예
Bernoulli성공/실패 하나전환 여부, 에러 여부
Binomialn번 중 성공 횟수1000 요청 중 실패 수
Poisson시간/공간당 사건 수분당 에러, 초당 가입
Normal평균 주변 대칭 변동측정 오차 근사, 평균 추정
Heavy-tailed극단값이 자주 큼latency, 파일 크기, 사용자별 트래픽
표본 편향

로그가 남은 요청만 보면 실패해서 도달하지 못한 요청은 빠집니다. 설문 응답자만 보면 강한 의견을 가진 사용자가 과대표집됩니다. sampling frame이 모집단과 어떻게 다른지 먼저 적어야 합니다.

AM-03

신뢰구간과 실험 설계

신뢰구간은 "진짜 값이 있을 법한 범위"를 표현합니다. 단일 숫자보다 더 정직한 이유는 표본 변동성을 함께 드러내기 때문입니다.

항목의미실무 기준
Baseline변경 전 기준값요일/시간대 효과를 포함해 충분히 관측
Control변경을 받지 않는 비교군동시 비교로 외부 요인을 줄임
Randomization사용자/요청을 무작위 배정군 간 차이를 우연화
Confidence interval추정값의 불확실성 범위효과 크기와 함께 보고
Power실제 효과를 발견할 가능성작은 효과를 보려면 큰 표본 필요
실험 판정

p-value만 보고 결정하지 않습니다. 효과 크기가 제품적으로 의미 있는지, 신뢰구간이 실질 손실 영역을 포함하는지, guardrail metric이 깨지지 않았는지를 함께 봅니다.

  • 실험 시작 전에 primary metric과 guardrail metric을 정한다.
  • 중간에 유리한 순간만 보고 멈추는 peeking을 피한다.
  • 사용자 단위 실험인지 요청 단위 실험인지 명확히 한다.
  • 계절성, 배포, 마케팅 이벤트 같은 교란 요인을 기록한다.
AM-04

LATENCY PERCENTILE EVIDENCE

latency는 평균보다 percentile로 봐야 합니다. 사용자는 평균 요청이 아니라 자신이 겪은 한 요청을 경험하고, 시스템은 꼬리 지연에서 먼저 깨집니다.

LATENCY PERCENTILE EVIDENCE
p50: typical experience
p95: bad but common enough
p99: tail risk and saturation signal
bucketed histogram required for aggregation
지표읽는 법오해
Average전체 합 / 개수소수의 매우 느린 요청을 숨김
Median p50절반은 이 값 이하꼬리 문제를 거의 보여주지 않음
p9595%가 이 값 이하샘플 수가 적으면 흔들림
p99최악 1% 경계인스턴스별 p99 평균은 잘못된 집계
Histogram분포를 bucket으로 저장bucket 설계가 나쁘면 해상도 부족
백분위 집계

백분위는 평균 낼 수 없습니다. 여러 서버의 p99를 평균 내면 전체 p99가 아닙니다. 원본 샘플이나 histogram bucket을 합친 뒤 다시 percentile을 계산해야 합니다.

AM-05

VECTOR SIMILARITY BASICS

벡터 유사도는 객체를 숫자 공간에 놓고 가까움을 계산합니다. 검색, 추천, 임베딩 기반 분류는 대부분 "표현을 만들고 거리 함수를 고르는" 문제입니다.

개념의미주의
Embedding텍스트/이미지/사용자를 벡터로 표현모델과 데이터 분포에 의미가 의존
Cosine similarity방향의 유사도길이보다 의미 방향을 볼 때 사용
Dot product방향과 크기를 함께 반영벡터 norm 차이가 ranking에 영향
Euclidean distance공간상 거리차원과 scale에 민감
ANN index근사 최근접 검색recall/latency/storage tradeoff
VECTOR SIMILARITY BASICS
normalize vectors → choose metric → build index
→ evaluate recall@k and latency → inspect false positives
→ add filters, reranker, and drift monitoring
유사도는 정답이 아니다

가까운 벡터가 항상 좋은 결과라는 보장은 없습니다. 평가셋, negative examples, 도메인별 false positive 비용을 정하고 recall@k, precision@k, nDCG 같은 검색 품질 지표를 함께 봅니다.

AM-06

측정 오차와 편향

측정값은 현실의 복사본이 아니라 관측 시스템이 만든 결과입니다. 계측 위치, 시간 동기화, 샘플링, 누락, 단위 변환이 모두 오차를 만듭니다.

오차설명대응
Selection bias관측 대상이 전체를 대표하지 않음sampling frame 점검
Survivorship bias살아남은 사례만 분석실패/이탈/timeout 포함
Instrumentation error계측 코드가 잘못 측정단위 테스트, shadow metric
Clock skew서버 간 시간이 다름monotonic clock, trace span 기준
Observer effect측정 자체가 시스템에 영향sampling, low overhead instrumentation
측정 리뷰 질문

이 숫자는 어디서 생성됐고, 무엇을 제외했고, 어떤 단위이며, 어떤 지연으로 도착했고, 재처리하면 같은 값이 나오는가. 이 질문에 답하지 못하면 숫자는 결정 근거가 되기 어렵습니다.

AM-07

VERIFICATION EVIDENCE PACKET

검증은 예제 몇 개가 통과했다는 보고가 아닙니다. 속성 기반 테스트, fuzzing, benchmark, 회귀 임계값을 조합해 코드가 넓은 입력과 시간 예산에서 성질을 지킨다는 증거를 만듭니다.

VERIFICATION EVIDENCE PACKET
examples → properties → generators → fuzz corpus
→ benchmark fixture → percentile result → regression threshold
→ failure seed and replay command
기법검증하는 것좋은 산출물
Example test대표 사례 동작명확한 기대값
Property-based test많은 입력에서 유지되는 성질generator, invariant, shrinked counterexample
Fuzz test예상 밖 입력에서 crash/security bugcorpus, seed, minimized input
Benchmark시간/메모리 비용p50/p95/p99, allocation, environment
Regression guard다시 느려지거나 깨지는지CI threshold, alert, baseline
Confidence Intervaluncertainty
표본으로 추정한 값의 불확실성을 범위로 표현한 것입니다.
Percentiledistribution
분포에서 특정 비율이 그 값 이하에 놓이는 경계입니다.
Propertyinvariant
개별 예제가 아니라 모든 입력군에서 유지되어야 하는 성질입니다.
