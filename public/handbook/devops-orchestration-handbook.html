오케스트레이션 기초

오케스트레이션은 여러 컨테이너를 어느 노드에 띄우고, 어떻게 발견하고, 어떻게 교체하고, 어떻게 권한을 제한할지 다루는 계층입니다. Kubernetes는 그 대표 구현입니다.

ORC-01

클러스터 구조

구성요소역할
Control plane원하는 상태를 저장하고 스케줄링·조정
Node워크로드가 실제 실행되는 서버
SchedulerPod를 어느 node에 배치할지 결정
Controller현재 상태를 원하는 상태로 수렴
kubelet노드에서 컨테이너 실행 상태 관리
핵심 모델

Kubernetes는 명령 실행기가 아니라 desired state controller입니다. “Pod 3개가 떠 있어야 한다”고 선언하면, 실제 상태가 달라질 때 계속 맞추려 합니다.

ORC-02

워크로드 모델

리소스의미예시
Pod배포 가능한 최소 실행 단위앱 컨테이너 + sidecar
Deploymentstateless app 배포와 rolling updateAPI 서버
StatefulSet고정 식별자와 저장소가 필요한 appDB, broker
Job/CronJob일회성 또는 주기 작업batch, cleanup
DaemonSet모든 노드에 하나씩 실행log agent, node exporter
ORC-03

서비스 디스커버리

Pod IP는 바뀔 수 있습니다. Service는 바뀌는 Pod 집합 앞에 안정적인 접근 지점을 제공합니다.

Service Type역할
ClusterIP클러스터 내부 접근
NodePort노드 포트로 외부 노출
LoadBalancer클라우드 로드밸런서 연동
IngressHTTP host/path 기반 라우팅
DNS와 readiness

Service가 있다고 트래픽을 받을 준비가 된 것은 아닙니다. readiness probe가 실패하는 Pod는 endpoint에서 제외되어야 합니다.

ORC-04

스케일링과 업데이트

Replica수평 확장
동일한 Pod 수를 늘려 트래픽을 나눕니다.
HPA자동 확장
CPU, memory, custom metric 기준으로 replica 수를 조정합니다.
request와 limit스케줄링 기준
request는 스케줄러가 배치에 사용하는 보장량이고, limit은 런타임 상한입니다. request가 없으면 과밀 배치가 쉽고, limit이 과하면 throttle/OOM이 장애로 보입니다.
Rolling update점진 교체
새 Pod를 띄우고 준비되면 이전 Pod를 줄입니다.
PodDisruptionBudget가용성
자발적 중단 중에도 최소 가용 Pod 수를 유지합니다.

정책과 권한

영역기준

RBAC사용자와 service account 권한 최소화
NetworkPolicyPod 간 통신 허용 범위 제한
