CS 기초: 알고리즘을 실무 판단으로 바꾸기

CS 기초는 면접용 암기가 아니라 데이터 크기, 접근 패턴, 불변식, 실패 비용을 근거로 구현을 고르는 언어입니다. 이 문서는 복잡도를 자릿수 계산으로, 자료구조를 실코드로, 증명을 PR 리뷰 문장으로, 벤치마크를 복붙 가능한 harness로 바꿉니다.

CS-01

CS FOUNDATION SPINE

알고리즘 선택은 "가장 빠른 알고리즘"을 찾는 일이 아닙니다. 실제 입력 분포, 메모리 한계, 캐시 지역성, 구현 위험, 운영 관측성을 함께 보며 충분히 좋은 선택을 고정하는 일입니다.

CS FOUNDATION SPINE
problem contract → input scale → operation mix → data structure invariant
→ asymptotic bound → constant factor → memory layout
→ correctness sketch → benchmark evidence → failure playbook
질문확인할 것산출물
입력은 얼마나 커지는가현재 p50/p99 크기, 1년 성장률, 최악 요청 크기크기 표와 제한값 (예: "태그 룰은 현재 120개, 연 2배 성장, 상한 5,000")
무슨 연산이 지배적인가읽기/쓰기/탐색/정렬/삭제 비율operation mix 표 (예: 조회 10⁴/일 vs 갱신 10/일 → 읽기 최적화)
정확성의 핵심은 무엇인가정렬 순서, 중복 금지, 연결성, 도달성, 단조성불변식 한 문장
성능 병목은 어디인가CPU, 메모리, 할당, I/O, lock 경합프로파일링 기록

RULE판단 부등식 세 개

이 문서 전체를 관통하는 판단은 부등식 세 개로 요약됩니다. 감이 아니라 이 부등식의 양변에 숫자를 넣을 수 있어야 결정입니다.

  • 인덱스 구축: 조회 횟수를 q라 할 때 n + q·1 (구축 + hash 조회) < q·n (매번 스캔) — q ≥ 2면 거의 항상 인덱스가 이긴다. q = 1이면 스캔이 이긴다.
  • 정렬 후 이분탐색: n log n + q log n < q·n — q가 log n보다 크면 정렬이 회수된다.
  • 캐시 도입: hit율 × 절감 시간 > miss 비용 + 무효화 버그 위험 + 메모리 상한 관리 비용 — 우변을 0으로 치는 순간 stale 장애가 시작된다.
실무 기준

복잡도는 출발점이고 벤치마크는 결론입니다. O(n log n)이 O(n)보다 느리다는 말은 같은 상수와 같은 메모리 접근 조건에서만 대략 맞습니다. n이 수백이면 연속 메모리를 도는 단순 배열 스캔이 포인터를 쫓는 "이론적 최적해"를 이기는 경우가 흔합니다. 교차점은 CS-06의 harness로 직접 잽니다.

CS-02

복잡도와 비용 모델

Big-O는 증가율의 언어입니다. 실무에서 쓸모 있으려면 두 가지 환산이 필요합니다: 표기를 연산 수 자릿수로, 연산 수를 시간 예산으로. 이 절의 모든 숫자는 "단순 연산 1회 ≈ 10ns (10⁸ ops/s)"라는 보수적 비용 모델 기준의 자릿수 감각이며, 실제 상수는 연산 종류에 따라 10~100배 달라집니다.

표기n = 1,000n = 1,000,00010ms 예산에서 허용되는 n실무 예
O(1)11n과 무관 (단, 원격 호출이면 상수가 10⁵배)hash map lookup, 배열 인덱스
O(log n)~10~20사실상 무제한binary search, B-tree index
O(n)10³10⁶ (≈ 10ms)~10⁶filter, validation pass
O(n log n)~10⁴~2×10⁷ (≈ 200ms)~5×10⁴sort, merge
O(n²)10⁶ (≈ 10ms)10¹² (≈ 3시간)~1,000중첩 루프 join, 거리 행렬
예산 계산 절차
1. 시간 예산을 정한다 — 예: API p99 100ms 중 이 함수 몫 10ms
2. 예산을 연산 수로 바꾼다 — 10ms ÷ 10ns = 10⁶ 회
3. 복잡도 식을 뒤집어 허용 n을 구한다 — O(n²)이면 n ≤ √10⁶ = 1,000
4. p99 입력 크기와 비교한다 — p99 n = 5,000이면 이 설계는 예산을 25배 초과
5. 초과하면 알고리즘을 바꾸거나(CS-05), 입력을 제한한다(CS-07)

MODEL숨은 비용: 연산은 평등하지 않다

복잡도 계산에서 "연산 1회"는 보통 CPU 명령 몇 개를 가정합니다. 하지만 루프 본문이 원격 호출이면 상수가 자릿수로 다섯 개 뛰고, Big-O가 같아도 성능은 완전히 다른 문제가 됩니다.

단순 연산 / L1 cache 참조~1–10ns기준 ×1
main memory 접근 (cache miss)~100ns×10
SSD random read~100µs×10⁴
같은 데이터센터 round trip + 인덱스 hit DB 쿼리~0.5–2ms×10⁵ — 루프 안 DB 호출 1회 ≈ 로컬 연산 수십만 회

TS · N+1 — Big-O가 아니라 상수가 죽이는 경우
// BAD — N+1 query: 복잡도는 O(n)이지만 상수가 1ms.
// n = 200이면 네트워크 왕복만 200 × 1ms = 200ms. p99 예산 100ms 초과.
for (const order of orders) {
  const customer = await db.customer.findUnique({
    where: { id: order.customerId },
  });
  enrich(order, customer);
}

// GOOD — 2 query + 메모리 join: 왕복 2회 ≈ 2ms, 나머지는 로컬 O(n).
const customerIds = [...new Set(orders.map((o) => o.customerId))];
const customers = await db.customer.findMany({
  where: { id: { in: customerIds } },
});
const customersById = new Map(customers.map((c) => [c.id, c]));
for (const order of orders) {
  enrich(order, customersById.get(order.customerId));
}
숨은 비용 리뷰 규칙

코드 리뷰에서 Big-O보다 먼저 표시할 것: 루프 안 await·원격 호출, 루프 안 정렬, 요청별 전체 스캔, 큰 자료구조 복사(spread로 매 반복 새 배열), 무제한 재귀. 이 다섯은 표기상 복잡도가 낮아도 상수 또는 실효 복잡도가 폭발하는 지점입니다. 예: 루프 안 [...acc, item] 누적은 O(n)처럼 보이지만 복사 때문에 O(n²)입니다.

  • 상한 입력 크기를 API, DB, UI 경계에서 모두 제한한다 (CS-07의 guard 코드 참조).
  • 평균이 아니라 p95/p99 입력 크기로 시간 예산을 계산한다 — 예산 계산 절차 2~4단계.
  • 중첩 루프는 한쪽 크기 상한이 코드나 스키마로 강제될 때만 허용한다 ("보통 작다"는 근거가 아니다).
  • 캐시를 추가할 때 무효화 비용과 메모리 상한을 같이 쓴다 (CS-01 캐시 부등식).
CS-03

자료구조 선택

자료구조는 불변식을 저장하는 방식입니다. 어떤 연산을 빠르게 만들면 보통 다른 연산, 메모리, 구현 복잡도가 비용으로 따라옵니다. 선택 기준은 "무슨 연산이 몇 번 일어나는가"이지 익숙함이 아닙니다.

자료구조핵심 연산 비용불변식실패 모드
Array / Vector인덱스 O(1), 끝 삽입 amortized O(1), 중간 삽입/탐색 O(n)연속 메모리, 순서 유지includes/indexOf를 루프 안에서 호출해 O(n²) (아래 코드)
Hash Map / Set조회·삽입·삭제 평균 O(1), 최악 O(n)키 해시와 동등성이 안정적mutable key, 나쁜 hashCode, 무제한 성장(→ LRU로 상한)
Tree / B-tree조회·삽입 O(log n), 범위 검색 O(log n + k)정렬 순서, 균형비교자 불일치, 포인터 추적으로 cache miss 누적

CODE중복 제거 — 루프 안 includes는 O(n²)

가장 흔한 실무 O(n²)는 중첩 for가 아니라 루프 안에 숨은 선형 탐색입니다. includes, indexOf, find, Java의 List.contains가 모두 O(n)이라는 사실이 코드 모양에 드러나지 않습니다.

TS · dedupe — O(n²) 함정과 O(n) 교정
// BAD — out.includes가 매번 out을 처음부터 스캔한다 → 전체 O(n²).
// n = 50,000이면 최악 비교 횟수 ≈ n²/2 = 1.25 × 10⁹ 회 → 초 단위.
function dedupeSlow(ids: string[]): string[] {
  const out: string[] = [];
  for (const id of ids) {
    if (!out.includes(id)) out.push(id); // 루프 안 O(n) 선형 탐색
  }
  return out;
}

// GOOD — Set 조회는 평균 O(1) → 전체 O(n).
// 불변식: k번째 반복이 끝나면 seen === ids[0..k)의 고유값 집합이고
//         out은 그 고유값을 첫 등장 순서로 담는다.
function dedupe(ids: string[]): string[] {
  const seen = new Set<string>();
  const out: string[] = [];
  for (const id of ids) {
    if (seen.has(id)) continue;
    seen.add(id);
    out.push(id);
  }
  return out;
}
JAVA · 같은 함정, 같은 교정 + 반복 조회용 인덱스
// BAD — List.contains는 O(n) → 전체 O(n²)

for (String id : ids) {

// GOOD — LinkedHashSet: 평균 O(1) 조회 + 입력 순서 유지 → 전체 O(n)
// 불변식: unique는 ids의 고유값을 첫 등장 순서로 담는다

// 반복 조회가 있다면 스캔하지 말고 인덱스를 한 번 구축한다.
// 구축 O(n), 이후 조회 평균 O(1) — CS-01 부등식: 조회 2회 이상이면 이득.

Map<Long, List<Order>> index = orders.stream()
    .collect(Collectors.groupingBy(Order::getCustomerId));

CODELRU cache — 자료구조 성질을 그대로 쓰는 20줄

LRU는 "hash 조회 O(1) + 사용 순서 유지"라는 두 요구의 결합입니다. JS Map과 Java LinkedHashMap은 둘 다 이 결합을 언어 차원에서 제공하므로, 직접 doubly linked list를 짤 이유가 없습니다.

TS · Map 기반 LRU — 삽입 순서가 곧 사용 순서
// Map은 삽입 순서를 유지한다 — 이 성질이 구현의 전부다.
// 불변식 1: map의 key 순서는 "가장 오래 안 씀 → 가장 최근 씀".
// 불변식 2: map.size ≤ capacity (set이 반환되는 시점마다 성립).
class LruCache<K, V> {
  private readonly map = new Map<K, V>();
  constructor(private readonly capacity: number) {}

  get(key: K): V | undefined {
    if (!this.map.has(key)) return undefined;
    const value = this.map.get(key)!;
    this.map.delete(key);     // 삭제 후 재삽입 = 맨 뒤(가장 최근)로 이동
    this.map.set(key, value); // → 불변식 1 유지
    return value;
  }

  set(key: K, value: V): void {
    if (this.map.has(key)) this.map.delete(key);
    this.map.set(key, value);
    if (this.map.size > this.capacity) {
      // 불변식 1에 의해 keys()의 첫 원소가 eviction 대상
      this.map.delete(this.map.keys().next().value!);
    }
  }
}
JAVA · LinkedHashMap 기반 LRU
// accessOrder = true: get/put마다 entry를 순서 맨 뒤로 보낸다.
// removeEldestEntry가 true를 반환하면 맨 앞(가장 오래 안 쓴) entry 제거.
// 불변식: size() ≤ capacity, iteration 순서 = LRU → MRU.
// 주의: 동기화 없음 — 동시 접근 경로면 Caffeine 같은 검증된 라이브러리로.
class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(16, 0.75f, true); // 세 번째 인자가 accessOrder
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}

RULE운영 데이터 구조

생산 코드에서 자료구조는 성능만 담당하지 않습니다. 중복 요청 방지용 set, 위의 LRU cache, rate limit counter, 작업 큐, topology graph처럼 운영 정책을 코드로 고정합니다. 이때 선택 문서에는 용량 상한(위 코드의 capacity), TTL, eviction 정책, 동시성 정책이 함께 있어야 합니다. "cache를 쓴다"는 결정이 아니고 "capacity 10,000, LRU eviction, 무효화는 write-through"가 결정입니다.

CS-04

INVARIANT PROOF DRILL

증명은 형식 논문이 아니라 "왜 이 코드가 깨지지 않는지"를 짧게 설명하는 훈련입니다. 실무에서는 불변식, 초기화, 보존, 종료를 PR 설명에 넣을 수 있으면 충분합니다. 아래 binary search는 이 네 단계가 코드 주석으로 그대로 들어간 예입니다.

INVARIANT PROOF DRILL
1. invariant: 루프/상태 전이 전후로 항상 참인 문장을 쓴다
2. initialization: 시작 전에 invariant가 참임을 보인다
3. preservation: 한 단계 실행 후에도 참임을 보인다
4. termination: 끝났을 때 invariant가 원하는 결과를 의미함을 보인다
5. counterexample: 깨지는 입력을 하나 만들어 테스트로 고정한다

CODEbinary search — 반열림 [lo, hi) 불변식

TS · lowerBound — 불변식이 주석으로 증명되는 코드
// lowerBound: target 이상인 값이 처음 나오는 인덱스 (없으면 arr.length).
// 전제: arr는 오름차순 정렬.
// 불변식 (루프 전후 항상 참):
//   (a) arr[0..lo)  의 모든 값 < target   — lo 왼쪽은 전부 탈락 확정
//   (b) arr[hi..n)  의 모든 값 ≥ target   — hi부터 오른쪽은 전부 정답 후보 이상
// 초기화: lo=0, hi=n이면 두 구간 모두 빈 구간이므로 (a)(b) 자명하게 참.
function lowerBound(arr: number[], target: number): number {
  let lo = 0;
  let hi = arr.length; // 반열림: hi 자체는 후보 구간에 포함되지 않는다
  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1); // (lo+hi)/2의 overflow 방어 습관
    if (arr[mid] < target) {
      lo = mid + 1; // 보존: arr[mid] < target이므로 (a)가 mid까지 확장
    } else {
      hi = mid;     // 보존: arr[mid] ≥ target이므로 (b)가 mid까지 확장
    }
    // 종료 보장: mid ∈ [lo, hi-1]이므로 두 분기 모두 구간을 최소 1 줄인다
  }
  // 종료: lo === hi. (a)와 (b)가 그 지점에서 만나므로
  // lo는 "target 이상이 시작되는 첫 인덱스" — 곧 삽입 위치다.
  return lo;
}
TS · 나쁜 예 — 닫힌 구간 초기화 + 반열림 축소의 오프바이원
// BAD — 구간 규약이 섞였다: 초기화는 닫힌 구간 [lo, hi],
// 축소는 반열림 규칙(hi = mid). 불변식 (b) "arr[hi..n) ≥ target"이
// 초기화 시점부터 거짓이다 — arr[n-1]이 target 미만일 수 있으므로.
function lowerBoundBroken(arr: number[], target: number): number {
  let lo = 0;
  let hi = arr.length - 1; // 닫힌 구간으로 시작해 놓고
  while (lo < hi) {
    const mid = (lo + hi) >> 1;
    if (arr[mid] < target) lo = mid + 1;
    else hi = mid;         // 반열림 규칙으로 줄인다 → 규약 불일치
  }
  return lo;
}
// counterexample: lowerBoundBroken([1, 3], 5)
//   기대값 2 ("없음" = arr.length) → 실제 1을 반환.
//   hi = length - 1 초기화가 "마지막 원소 뒤"라는 정답 후보를
//   시작부터 배제했다. 오프바이원 버그의 뿌리는 ±1이 아니라
//   구간 규약(닫힘/반열림)을 한 함수 안에서 섞은 것이다.
오프바이원의 진짜 원인

binary search 버그는 mid + 1이냐 mid냐를 외워서 고치는 게 아닙니다. 구간 규약을 하나 정하고(반열림 권장), 초기화·축소·종료가 모두 그 규약을 따르는지 확인하면 끝납니다. 테스트는 counterexample 계열을 고정합니다: 빈 배열, 한 원소, 전부 target 미만, 전부 target 이상, 중복값 경계.

CODEtwo pointers — "탈락은 안전하다"를 증명하기

TS · 정렬 배열에서 합이 target인 쌍 — O(n)
// 전제: arr는 오름차순 정렬.
// 불변식: 합이 target인 쌍이 존재한다면, 그 쌍의 두 인덱스는
//         항상 현재 구간 [lo, hi] 안에 있다.
// lo 탈락의 근거: arr[lo] + arr[hi] < target일 때 hi는 lo가 만날 수 있는
//   최대 파트너다. 최대와도 부족하면 arr[lo]는 어떤 쌍도 못 만든다 → 안전.
function findPairSum(arr: number[], target: number): [number, number] | null {
  let lo = 0;
  let hi = arr.length - 1;
  while (lo < hi) {
    const sum = arr[lo] + arr[hi];
    if (sum === target) return [lo, hi];
    if (sum < target) lo++; // arr[lo] 탈락 — 위 근거로 불변식 보존
    else hi--;              // 대칭 논리: 최소 파트너와도 초과 → arr[hi] 탈락
  }
  return null; // 종료: 후보 구간 소진 = "쌍이 없다"가 증명된 상태
}
// 이중 루프 O(n²) 대비: n = 100,000이면 5×10⁹ 회 → 10⁵ 회. 자릿수 4개 차이.
패턴불변식 예검증 테스트 (counterexample 계열)
binary search정답은 항상 반열림 [lo, hi) 안에 있다빈 배열, 한 원소, 전부 미만/이상, 중복 경계
two pointers탈락한 원소는 어떤 정답에도 참여할 수 없다모두 통과, 모두 실패, 포인터 교차 직전
BFS큐에서 꺼낸 노드의 거리는 최단 거리로 확정됐다cycle, disconnected, 여러 최단 경로
transaction state machine저장된 상태는 허용된 전이의 결과뿐이다중복 요청, 역전이, 동시 전이
리뷰 문장

"이 함수는 정렬된 배열을 전제로 하며, 각 반복에서 정답 후보 구간을 절반으로 줄입니다. 구간은 반열림 [lo, hi)로 유지하고, 종료 시 lo == hi이므로 삽입 위치가 됩니다." — PR 설명에 이 세 문장이 있으면 리뷰어는 구현 의도, 전제, 경계 조건을 한 번에 검증할 수 있습니다.

CS-05

PRODUCTION ALGORITHM DECISION

프로덕션 알고리즘 선택은 "교과서상 최적"보다 "충분히 빠르고, 설명 가능하며, 장애 시 완화 가능한가"가 중요합니다. 그리고 대부분의 회귀는 새 알고리즘이 아니라 데이터가 자라서 예전 결정이 무효가 되는 것에서 옵니다.

CASE태그 매칭 — 이중 루프가 데이터 성장에 깔리는 시나리오

CASE · 사례 시나리오 (비용 모델 추정, 실측 아님)
증상: 주문-태그 룰 매칭 API의 p99가 배포 없이 몇 달에 걸쳐 30ms → 400ms로 상승. p50은 거의 그대로.
증거: 트래픽은 정체. 주문 배치 크기 n과 태그 룰 수 m이 각각 3배 성장. profiler에서 matchOrders가 CPU의 85%.
원인: O(n·m) 이중 루프. n = 10,000, m = 100이면 내부 반복 10⁶회 — 10ns/op 모델로 수십 ms. 3배 성장 후 n·m = 9×10⁶ → 9배인 수백 ms. p99만 튀는 이유: 가장 큰 배치가 꼬리를 만든다.
조치: 룰을 tagId로 인덱싱해 O(n·k + m)로 (k = 주문당 태그 수 ≈ 5). 연산 수 ~5×10⁴ → ms 미만 규모. 자릿수 두 개 감소.
TS · Before — O(n·m·k): 주문마다 전체 룰 스캔
// n = 주문 수, m = 룰 수, k = 주문당 태그 수.
// includes까지 세면 n·m·k. 출시 시점(n=1,000, m=30)엔 3×10⁴로 무해했다.
// 문제는 코드가 아니라 "n과 m이 함께 자란다"는 사실이 결정에 없던 것.
function matchOrders(orders: Order[], rules: TagRule[]): Match[] {
  const matches: Match[] = [];
  for (const order of orders) {          // n
    for (const rule of rules) {          // × m
      if (order.tagIds.includes(rule.tagId)) { // × k
        matches.push({ orderId: order.id, ruleId: rule.id });
      }
    }
  }
  return matches;
}
TS · After — O(n·k + m): 룰을 tagId로 한 번 인덱싱
// 불변식: rulesByTag.get(tagId)는 해당 tagId를 갖는 룰 전체, 입력 순서 유지.
// CS-01 인덱스 부등식: 구축 m회 + 조회 n·k회 ≪ 스캔 n·m·k회 (m ≫ 1이면 항상).
function matchOrders(orders: Order[], rules: TagRule[]): Match[] {
  const rulesByTag = new Map<string, TagRule[]>();
  for (const rule of rules) {
    const bucket = rulesByTag.get(rule.tagId) ?? [];
    bucket.push(rule);
    rulesByTag.set(rule.tagId, bucket);
  }
  const matches: Match[] = [];
  for (const order of orders) {
    for (const tagId of order.tagIds) {          // n × k
      for (const rule of rulesByTag.get(tagId) ?? []) {
        matches.push({ orderId: order.id, ruleId: rule.id });
      }
    }
  }
  return matches;
}
수치의 지위

위 ms 수치는 10ns/op 비용 모델로 계산한 자릿수 추정이지 실측이 아닙니다. 결정 기록에 넣을 최종 수치는 CS-06의 harness로 실제 입력 분포에서 잰 p50/p95여야 합니다. 추정은 "재볼 가치가 있는가"를 정하는 필터입니다.

선택지고르는 상황피해야 할 상황운영 증거
단순 스캔n 상한이 강제되고(예: ≤ 500) 코드 경로가 드묾요청마다 큰 컬렉션 스캔입력 상한, p99 시간
정렬 후 처리조회 q > log n (CS-01 부등식)매 요청마다 같은 정렬 반복정렬 키, 안정성 필요 여부
인덱스/맵 구성반복 lookup ≥ 2회 — 위 CASE한 번만 조회하고 버림메모리 사용량, 갱신 정책
근사 알고리즘정확도보다 latency와 비용이 중요 (검색 추천, count 추정)정산, 권한, 보안 판정오차 범위, 재현 가능한 평가셋

면접식 최적화의 함정

작은 팀 서비스에서 복잡한 lock-free 구조나 직접 구현한 balanced tree를 넣는 것은 성능보다 유지보수 리스크가 큽니다. 표준 라이브러리, DB 인덱스, 검증된 라이브러리를 먼저 쓰고, 직접 구현은 측정된 병목 + 대안 부재 두 조건이 모두 확인됐을 때만 선택합니다. 위 CASE의 해법도 Map 하나였지 자료구조 발명이 아니었습니다.

CS-06

BENCHMARK EVIDENCE PACKET

벤치마크는 숫자를 만드는 절차가 아니라 결정을 방어하는 증거 패킷입니다. 재현 가능하고, 비교 조건이 공정하며, 실제 입력을 닮아야 합니다. 아래 harness는 그대로 파일에 붙여 node bench.mjs로 실행됩니다.

BENCHMARK EVIDENCE PACKET
hypothesis → fixture data → warmup → measured run
→ p50/p95/max → allocation/GC → profiler snapshot
→ regression threshold → decision note

CODE최소 harness — warmup + 반복 + percentile

JS · bench.mjs — node bench.mjs 로 실행
import { performance } from 'node:perf_hooks';

function bench(name, fn, { warmup = 50, runs = 200 } = {}) {
  for (let i = 0; i < warmup; i++) fn(); // JIT/캐시 예열 — 측정에서 제외
  const samples = [];
  for (let i = 0; i < runs; i++) {
    const t0 = performance.now();
    fn();
    samples.push(performance.now() - t0);
  }
  samples.sort((a, b) => a - b);
  const at = (q) => samples[Math.min(samples.length - 1, Math.floor(samples.length * q))];
  console.log(
    `${name.padEnd(20)} p50=${at(0.5).toFixed(3)}ms` +
    `  p95=${at(0.95).toFixed(3)}ms  max=${samples[samples.length - 1].toFixed(3)}ms`
  );
}

// fixture: 실제 분포를 닮게 — 여기선 고유값 1,000종이 반복되는 50,000건
const input = Array.from({ length: 50_000 }, (_, i) => `id-${i % 1_000}`);
let sink = 0; // 결과를 소비해 dead code elimination 방지

bench('includes-dedupe', () => {
  const out = []; for (const v of input) if (!out.includes(v)) out.push(v);
  sink += out.length;
});
bench('set-dedupe', () => { sink += new Set(input).size; });
console.log('sink =', sink); // sink를 출력해야 최적화로 증발하지 않는다

관찰 (예시 수치)해석다음 행동
includes p50 = 38ms vs set p50 = 0.6ms~60배 — 알고리즘 등급 차이(O(n²) vs O(n)). 상수 튜닝으로 못 좁힌다set 채택. Decision record에 두 수치와 fixture 조건 기록
p95/p50 비율 ≤ 1.3측정 노이즈 적정 — 분포가 안정적그대로 판정에 사용
p95/p50 비율 > 2GC, 절전 모드, 백그라운드 프로세스 개입 의심runs 증가, 전원 연결·부하 없는 상태에서 재측정

두 후보 차이가 p50 기준 20% 미만이 fixture에서는 사실상 동급단순한 쪽 채택 — 복잡도는 그 자체가 비용
MICROBENCHMARK 함정

결과를 소비하지 않으면(위 코드의 sink) 엔진이 계산 자체를 제거해 "0.001ms의 기적"이 나옵니다. 그 외 필수 조건: 같은 프로세스에서 후보 순서를 바꿔도 결과가 유지되는지 확인, 노트북 전원 연결, fixture는 랜덤 균등 분포가 아니라 실제 로그에서 샘플. Java라면 이 수제 harness 대신 JMH를 씁니다 — JVM warmup과 dead code 문제를 도구가 처리합니다.

CS-07

실패 모드와 운영 방어선

CODE입력 폭탄 방어 — "무제한 n"을 경계에서 끊기

CS-02의 예산 계산이 "O(n) 경로는 n ≤ 10⁶까지 허용"이라고 말했다면, 그 상한을 신뢰 경계에서 코드로 강제합니다. 클라이언트가 보내는 값은 입력이지 계약이 아닙니다.

TS · pagination guard — 통과한 값의 범위를 불변식으로 보장
// 불변식: 이 함수가 반환한 값은 항상
//   1 ≤ limit ≤ MAX_PAGE_SIZE, 0 ≤ offset ≤ MAX_OFFSET 를 만족한다.
// 이후 코드(쿼리, 루프, 직렬화)는 이 범위를 전제로 예산을 계산할 수 있다.
const MAX_PAGE_SIZE = 100;
const DEFAULT_PAGE_SIZE = 20;
const MAX_OFFSET = 10_000; // deep offset은 DB가 그만큼 읽고 버린다 — 아래 callout

function parsePagination(query: Record<string, unknown>): {
  limit: number;
  offset: number;
} {
  const rawLimit = Number(query.limit);
  const rawOffset = Number(query.offset);
  // 초과 요청은 400 대신 clamp — 에러냐 clamp냐는 API 정책으로 명시해 합의
  const limit =
    Number.isInteger(rawLimit) && rawLimit > 0
      ? Math.min(rawLimit, MAX_PAGE_SIZE)
      : DEFAULT_PAGE_SIZE;
  const offset =
    Number.isInteger(rawOffset) && rawOffset >= 0
      ? Math.min(rawOffset, MAX_OFFSET)
      : 0;
  return { limit, offset };
}

// body 폭탄은 파싱 전에 끊는다 — 100MB JSON은 parse 자체가 CPU·메모리 공격
app.use(express.json({ limit: '1mb' })); // 초과 시 413 Payload Too Large
DEEP OFFSET

OFFSET 1000000 LIMIT 20은 결과가 20행이어도 DB가 100만 행을 읽고 버립니다 — 복잡도가 페이지 번호에 비례해 자라는 O(offset) 경로입니다. offset 상한을 두고, 그 너머는 cursor 기반 pagination(WHERE id > :lastId ORDER BY id LIMIT :n)으로 전환합니다. cursor는 인덱스를 타므로 페이지 깊이와 무관하게 O(log n + k)입니다.

  • 모든 리스트 API에 limit 상한이 코드로 강제되는가 (기본값 존재 포함).
  • 모든 cache·큐·버퍼에 용량 상한과 eviction 정책이 있는가.
  • hash key로 쓰는 객체가 immutable인가, comparator가 대칭·추이적인가 테스트했는가.
  • 재귀에 깊이 상한이 있는가 — 입력이 만든 cycle이 stack overflow가 되지 않는가.
Amortized평균화 비용
개별 연산은 비싸도 긴 연산열 전체로 나누면 낮은 비용을 갖는다는 분석입니다. 동적 배열의 끝 삽입이 대표 예 — 가끔 O(n) 재할당이 있어도 평균 O(1)입니다.
Invariant항상 참
상태 전이 전후에 유지되어야 하는 조건입니다. CS-04의 [lo, hi)처럼 코드 주석과 PR 설명에 넣을 수 있는 한 문장이 목표입니다.
Tail Latencyp95/p99
소수의 느린 요청이 사용자 경험과 장애 예산을 지배하는 현상입니다. CS-05 CASE처럼 "가장 큰 입력"이 꼬리를 만들므로, 평균이 아니라 p99 입력으로 예산을 계산합니다.
N+1 Query루프 안 원격 호출
목록 1회 조회 후 각 항목마다 추가 쿼리를 날리는 패턴입니다. 복잡도는 O(n)이지만 상수가 로컬 연산의 10⁵배라 CS-02 비용 모델에서 최우선 리뷰 대상입니다.
