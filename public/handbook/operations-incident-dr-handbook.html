INC-00

역할

장애가 커지면 Incident Commander, scribe, communications owner, subject matter expert를 나눈다. 한 사람이 원인 분석, 의사결정, 공지를 모두 맡으면 시간이 갈수록 판단 품질이 떨어진다.

운영 질문확인할 증거판단 기준
장애 역할이 의사결정, 기록, 커뮤니케이션, 전문 분석으로 나뉘는가?Incident Commander, scribe, comms owner, SME, decision log, channel owner한 사람이 모든 역할을 맡으면 판단 품질이 떨어진다.
역할을 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?incident channel, role assignment, decision log, comms timeline역할과 권한이 초반에 고정되어야 완화가 빨라진다.
역할 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?장애 채널에서 여러 명이 동시에 명령IC가 우선순위를 정하고 scribe가 timeline을 남긴다.
INC-01

초기 대응

처음 15분에는 사용자 영향, 최근 변경, error budget 소진, 핵심 지표, 우회 가능성을 확인한다. 원인 확정 전에는 rollback, traffic shift, feature flag off 같은 사용자 영향 완화책을 우선 고려한다.

운영 질문확인할 증거판단 기준
처음 15분에 사용자 영향과 완화책을 원인 분석보다 먼저 고정했는가?impact scope, SLI, recent deploy, dependency status, support ticket trend, mitigation candidate원인 확정 전에도 사용자 영향 완화를 검토해야 한다.
초기 대응을 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?SLO dashboard, recent deploy, status page, support trend, rollback/flag stateSev와 comms channel, 완화 후보가 초기에 정해져야 한다.
초기 대응 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?원인 분석만 하다 rollback 타이밍을 놓침rollback, traffic shift, feature flag off를 비교해 실행한다.
INC-02

rollback과 restore

rollback은 이전 artifact로 돌아가는 것이고 restore는 데이터나 상태를 복구하는 것이다. migration이 포함된 변경은 rollback이 어려울 수 있으므로 expand-contract, backup, restore drill이 필요하다.

운영 질문확인할 증거판단 기준
rollback과 restore의 대상과 위험을 구분했는가?artifact version, migration version, backup age, restore point, data loss risk, reconciliation need코드 rollback과 데이터 restore는 다른 복구 전략이다.
rollback과 restore를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?release history, migration table, backup/restore status, reconciliation query데이터 변경이 있으면 이전 artifact만으로 복구되지 않을 수 있다.
rollback과 restore 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?rollback 후 데이터 불일치 발생restore drill, reconciliation, forward fix 계획을 실행한다.
INC-03

DR과 사후 분석

DR은 disaster recovery이며 RPO/RTO로 목표를 말한다. RPO는 허용 가능한 데이터 손실, RTO는 복구 시간 목표다. 운영 리허설은 Rollback drill, Restore drill, traffic shift drill로 나누어 실행한다. postmortem은 비난이 아니라 timeline, contributing factors, action item, runbook 개선을 남기는 절차다.

운영 질문확인할 증거판단 기준
DR 목표와 사후 분석이 실제 시스템 개선으로 닫히는가?RPO/RTO, replica lag, failover/failback result, postmortem action owner/date, runbook diff백업 존재가 아니라 복구와 원복 성공이 증거다.
DR과 사후 분석을 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?restore drill log, traffic shift result, replica lag, postmortem action trackerpostmortem은 timeline과 contributing factor, action item으로 닫혀야 한다.
DR과 사후 분석 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?복구는 됐지만 failback 절차가 없어 장기 임시 상태 지속failback drill과 runbook diff를 action item으로 남긴다.

CASE

장애/운영 시나리오

배포 직후 결제 성공률이 급락한다. Incident Commander가 역할을 나누고, 배포 artifact와 feature flag를 확인한다. 원인 분석과 별개로 rollback 또는 flag off로 사용자 영향을 줄인 뒤, 결제 누락 데이터가 있는지 restore와 reconciliation 절차를 실행한다.

Triage frame

Incident Response·Rollback·DR 장애는 증상 → 영향 범위 → 최근 변경 → Incident command 증거 → 완화책 → 복구 검증 → 재발 방지 순서로 기록한다. 이 흐름이 있어야 담당자가 바뀌어도 같은 결론에 도달한다.

RISK

자주 틀리는 판단

Incident Response·Rollback·DR에서 자주 틀리는 판단은 역할와 DR과 사후 분석의 책임 경계를 흐리는 데서 시작된다. 아래 신호가 보이면 설계, runbook, 답변 구조를 다시 점검한다.

  • 원인 확정 전까지 완화 조치를 미룬다.
  • 장애 채널에서 모두가 동시에 명령한다.
  • 백업은 있지만 Restore drill을 해본 적이 없다.
  • postmortem이 비난이나 사건 요약으로 끝난다.
Q&A

면접 답변 템플릿

장애 대응은 역할 분리, 영향 완화, 원인 분석, 복구 검증, 사후 개선의 순서로 답합니다. DR은 RPO/RTO로 목표를 정의하고 Rollback drill과 Restore drill로 실제 복구 가능성을 검증해야 합니다.

OPERATIONS ANSWER FRAME 1. 장애 대응은 역할 분리와 사용자 영향 완화가 먼저다.
2. rollback, restore, failover는 서로 다른 복구 전략이다.
3. postmortem은 action owner/date와 runbook diff까지 남겨야 한다.
