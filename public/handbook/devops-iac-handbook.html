IaC와
인프라 변경 관리

IaC는 인프라를 클릭이 아니라 코드로 정의하는 방식입니다. 핵심은 자동 생성이 아니라 변경을 리뷰 가능하고 재현 가능하게 만드는 것입니다.

IAC-01

선언적 관점

선언형 IaC는 “어떻게 만들지”보다 “원하는 최종 상태가 무엇인지”를 적습니다.

명령형선언형
서버 만들기 → 보안그룹 연결 → 태그 추가서버, 보안그룹, 태그가 이런 상태여야 함
절차 누락에 취약현재 상태와 목표 상태 차이를 계산
작업자 숙련도 의존리뷰와 자동화 가능
IAC-02

State와 Drift

개념의미위험
StateIaC 도구가 알고 있는 실제 리소스 상태손상되면 잘못된 변경 계산
remote state팀이 공유하는 원격 state 저장소암호화, locking, 접근 통제 필요
Drift코드와 실제 인프라가 달라진 상태다음 apply에서 예상 밖 변경
Plan목표 상태로 가기 위한 변경 미리보기검토 없이 apply하면 장애 가능
콘솔 변경 금지의 이유

콘솔에서 직접 리소스를 바꾸면 IaC 코드와 실제 상태가 어긋납니다. 긴급 변경이 필요했다면 사후에 반드시 코드와 state를 맞춰야 합니다.

drift detection

운영 환경은 정기적으로 drift detection을 수행합니다. plan을 읽기 전용으로 실행해 코드와 실제 인프라의 차이를 찾고, 허용된 긴급 변경인지 미승인 변경인지 분류한 뒤 코드 반영 또는 원복을 결정합니다.

IAC-03

의존 관계

인프라 리소스는 독립적이지 않습니다. VPC, subnet, route, security group, compute, database는 의존 관계를 가집니다.

VPC→ Subnet→ Route→ Security Group→ Compute
의존 관계실패 예시
DB subnet → DB instance서브넷 변경이 DB 재생성을 유발
Security group → service규칙 삭제로 트래픽 차단
IAM role → deployment권한 축소로 배포 실패
IAC-04

안전장치

Plan review변경 검토
apply 전에 생성, 수정, 삭제 리소스를 사람이 검토합니다.
State lock동시 실행 방지
여러 apply가 동시에 state를 바꾸지 못하게 합니다.
Policy check규칙 검증
public bucket, wide-open security group 같은 위험을 배포 전에 차단합니다.
Environment promotion승격
dev에서 검증한 모듈과 패턴을 staging/prod로 승격합니다.
