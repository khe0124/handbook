로그 조회·커스텀 명령
실무 가이드

로그에서 원인을 빠르게 찾고, 반복 작업을 명령 한 줄로 만드는 실무 가이드. 명령·스크립트가 중심이고, 각 단계엔 왜 그렇게 하는지 한 줄을 붙였습니다. macOS·zsh·Docker·Spring 환경 기준 — 리눅스 핸드북·관측가능성 핸드북과 함께 보세요.

ENV : macOS(Darwin) · zsh · Docker · Spring Boot · Gradle
COMPILED : 2026-06
LOGTL-01

로그를 어디서 보나

디버깅의 첫 질문은 항상 같습니다 — "이 로그는 어디로 나가는가." 콘솔인가, 파일인가, 컨테이너 안인가, systemd가 받아가는가, 클라우드로 흘러가는가. 출처를 먼저 정해야 다음 명령이 정해집니다.

실행 환경어디서 보나 / 명령왜
로컬 IDE·터미널표준 출력 콘솔. 파일로 쓰면 tail -f logs/app.log가장 직접적 — Spring은 기본이 콘솔, logging.file.name 설정 시 파일로
Docker 컨테이너(단일)docker logs -f <c>컨테이너 stdout/stderr를 Docker가 캡처 — -f로 실시간 추적
Docker Compose(다중)docker compose logs -f <svc>서비스 이름으로 골라봄 — 서비스 여러 개를 한 번에 또는 하나만
systemd 서비스journalctl -u <svc> -f리눅스 서버에서 서비스로 등록된 프로세스의 로그는 journal이 모음
클라우드(ECS/Cloud Run 등)CloudWatch · GCP Logging 콘솔, 또는 CLI컨테이너가 사라져도 로그는 남음 — 운영 사고 분석의 원본
실행 환경별 로그 추적 — 첫 명령 # 단일 컨테이너 — 마지막 100줄부터 실시간으로
docker logs -f --tail 100 myapp

# compose — api 서비스만, 타임스탬프 붙여서
docker compose logs -f -t api

# systemd — 부팅 이후 + 실시간
journalctl -u myapp.service -b -f

# 컨테이너 ID/이름을 모를 때 — 먼저 목록부터
docker ps | grep api
먼저 정할 것

로그를 못 찾는 가장 흔한 이유는 엉뚱한 출처를 보는 것입니다. 컨테이너로 돌리면서 로컬 파일을 tail하거나, compose로 띄워놓고 docker logs로 단일 컨테이너를 찾는 식. "지금 이 프로세스가 어떻게 떠 있나"(IDE / docker / compose / systemd)를 먼저 확정하고, 그에 맞는 한 줄을 고르세요. 컨테이너 안에서 직접 봐야 하면 docker exec -it <c> sh로 들어가 파일을 tail합니다.

LOGTL-02

로그 필터링

로그는 보는 게 아니라 좁히는 것입니다. 수천 줄에서 의미 있는 줄은 보통 몇 개 — grep/rg로 깎고, 실시간이면 tail -f | grep, 탐색이면 less +F로 흐름과 검색을 오갑니다.

목적명령왜
에러만 추리기grep -i error app.log-i로 ERROR/Error/error 모두 — 대소문자에 안 흔들림
앞뒤 맥락까지grep -C 3 "NullPointer" app.log에러 한 줄만 보면 원인을 모름 — -C 3으로 전후 3줄
특정 요청 추적grep "userId=42" app.log한 사용자·한 주문의 흐름만 — 노이즈 제거
제외하며 보기grep -v "health" app.log헬스체크 같은 반복 노이즈를 빼고 봄(-v=반대)
실시간 + 필터tail -f app.log | grep -i error새로 찍히는 에러만 흘려보냄 — 운영 디버깅 단골
빠른 재귀 검색rg -n "timeout" logs/여러 로그 파일을 한 번에, 줄번호와 함께(→ 리눅스 핸드북 LX-04)
실시간 흐름 vs 탐색 — less +F # 큰 로그를 열어 끝에서 실시간으로 따라가기
less +F app.log
# Ctrl-C → 흐름 멈춤(탐색 모드) / /패턴 → 검색 / Shift-F → 다시 실시간

# 시간 범위로 좁히기 — 09:13~09:14 구간의 에러만
grep "2026-06-23 09:1[34]" app.log | grep -i error

# docker 로그도 같은 방식 — 표준출력을 grep으로 받음
docker logs api 2>&1 | grep -i error
함정 — 에러는 stderr로 나간다

docker logs api | grep error가 비어 나오면, 에러가 stderr로 나가는데 파이프는 기본적으로 stdout만 넘기기 때문일 수 있습니다. docker logs api 2>&1 | grep error로 stderr를 stdout에 합쳐서 받으세요(2>&1의 의미는 → 리눅스 핸드북 LX-08). tail -f | grep은 실시간이지만 과거 로그를 거슬러 검색하진 못합니다 — 지나간 구간을 뒤지려면 less +F로 멈춰서 / 검색을 쓰세요.

LOGTL-03

구조화(JSON) 로그 조회

로그가 JSON 한 줄(JSON Lines)이면 grep으로 문자열을 긁는 대신 필드로 묻습니다 — "level이 ERROR이고 traceId가 X인 줄". jq가 그 도구입니다. 상관관계 ID 하나로 한 요청의 전 구간을 꿰는 것이 핵심.

목적명령왜
필드만 추출jq '.message' app.logJSON 전체 대신 보고 싶은 필드만 — 눈이 덜 피로함
레벨로 필터jq 'select(.level=="ERROR")' app.log구조화 필드로 정확히 거름 — 문자열 매칭의 오탐 없음
여러 필드 조립jq -r '[.ts,.level,.msg]|@tsv' app.log-r로 따옴표 없이, 탭 구분 표로 — 읽기 좋게
요청 하나 추적jq 'select(.traceId=="abc123")' app.log상관관계 ID로 한 요청의 모든 로그를 모음
docker 로그를 jq로docker logs api | jq 'select(.level=="ERROR")'컨테이너가 JSON으로 찍으면 바로 파이프
상관관계 ID로 한 요청을 꿰기 # 1) 에러 줄에서 traceId를 먼저 뽑는다
jq -r 'select(.level=="ERROR") | .traceId' app.log

# 2) 그 ID로 한 요청의 전 구간을 시간순으로 — 왜 이 흐름이 실패했나
jq -c 'select(.traceId=="abc123")' app.log

# 3) 여러 서비스 로그를 합쳐서 한 요청을 본다 (timestamp로 정렬)
cat api.log gateway.log | jq -c 'select(.traceId=="abc123")' | sort
왜 구조화 로그인가 → 관측가능성 핸드북

문자열 로그는 사람이 읽기엔 편하지만 기계가 묻기엔 약합니다. 구조화 로그(JSON)는 필드로 질의·집계·알람이 가능해집니다 — jq로 로컬에서 묻는 것이 클라우드 로그 쿼리(CloudWatch Insights 등)의 사고 모델과 같습니다. traceId / spanId 같은 상관관계 ID를 모든 로그에 심는 것이 분산 추적의 출발점입니다(이 ID를 서비스 경계를 넘어 전파하는 방법, 로그·메트릭·트레이스의 관계는 관측가능성 핸드북 참고).

LOGTL-04

런타임 로그 레벨 변경

DEBUG 로그를 보려고 재배포하는 건 느리고 위험합니다. Spring Boot는 재시작 없이 특정 패키지의 로그 레벨을 바꿀 수 있습니다 — Actuator의 /loggers 엔드포인트. 평소엔 INFO, 문제 추적할 때만 그 패키지를 DEBUG로 올렸다 내립니다.

방법형태왜
설정 파일(정적)logging.level.com.greenery=DEBUG in application.yml기본 레벨 — 재시작 시 적용. 영구 설정에 적합
현재 레벨 조회GET /actuator/loggers/com.greenery지금 이 패키지가 무슨 레벨인지 확인부터
런타임 변경POST /actuator/loggers/com.greenery + body재시작 없이 즉시 적용 — 운영 중 추적
원복body의 configuredLevel을 null로임시 DEBUG는 끝나면 반드시 되돌림
재시작 없이 DEBUG로 — Actuator /loggers # 1) application.yml — Actuator 엔드포인트 노출 (운영은 인증 뒤에)
management.endpoints.web.exposure.include: loggers

# 2) 현재 레벨 확인
curl -s localhost:8080/actuator/loggers/com.greenery | jq

# 3) 해당 패키지만 DEBUG로 — 즉시 적용, 재시작 불필요
curl -X POST localhost:8080/actuator/loggers/com.greenery \
  -H "Content-Type: application/json" -d '{"configuredLevel":"DEBUG"}'

# 4) 추적 끝나면 원복 — null이면 기본 레벨로 돌아감
curl -X POST localhost:8080/actuator/loggers/com.greenery \
  -H "Content-Type: application/json" -d '{"configuredLevel":null}'
운영에서 DEBUG는 양날의 검

전체를 DEBUG로 올리면 로그가 폭증해 디스크·비용·성능을 때리고, 민감정보(요청 바디, 쿼리 파라미터)가 로그로 새어나올 수 있습니다. 꼭 의심 가는 패키지만, 짧게, 끝나면 즉시 원복. Actuator 엔드포인트는 운영에서 반드시 인증·네트워크 제한 뒤에 둬야 합니다 — 노출되면 누구나 로그 레벨을 바꾸고 내부 정보를 봅니다. root 로거(/actuator/loggers/ROOT) 변경은 특히 신중히.

LOGTL-05

셸 alias·함수

같은 긴 명령을 하루에 열 번 친다면 그건 alias나 function으로 만들 신호입니다. 인자 없이 고정이면 alias, 인자를 받아야 하면 function. 그리고 ~/.zshrc는 chezmoi로 관리합니다 — 손으로 직접 고치지 마세요(→ 리눅스 핸드북 LX-13).

형태예시왜
alias(고정 명령)alias dl='docker compose logs -f'인자 없이 늘 같은 명령 — 가장 단순
function(인자 받음)dlog() { docker logs -f "$1"; }컨테이너 이름이 매번 달라짐 — $1로 받음
여러 단계 묶기function 안에서 파이프·여러 명령alias로는 못 하는 조합 로직을 function으로
현재 셸에 로드source ~/.zshrc새 alias는 새 셸이나 source 후에 보임
~/.zshrc — 로그 작업용 alias·함수 # 자주 치는 명령은 alias로
alias dps='docker ps --format "table {{.Names}}\t{{.Status}}"'
alias dcl='docker compose logs -f'

# 인자가 필요하면 function — 컨테이너의 에러만 실시간으로
elog() {
  docker logs -f "$1" 2>&1 | grep --line-buffered -i error
}
# 사용: elog api ← $1 자리에 컨테이너 이름

# 특정 traceId로 JSON 로그 추적
trace() {
  docker logs "$1" 2>&1 | jq -c "select(.traceId==\"$2\")"
}
# 사용: trace api abc123
chezmoi로 관리 · grep 버퍼링

~/.zshrc를 직접 고치면 chezmoi 관리와 어긋납니다 — chezmoi edit ~/.zshrc로 고치고 chezmoi apply, 그다음 source ~/.zshrc로 현재 셸에 반영하세요. 그리고 tail -f | grep이 실시간인데 출력이 안 나오면 grep의 버퍼링 탓일 수 있습니다 — grep --line-buffered로 줄 단위로 흘려보내세요. function의 인자는 "$1"처럼 반드시 따옴표로 감싸 공백·특수문자에 깨지지 않게 합니다.

LOGTL-06

스크립트화

function이 한 줄로 안 끝나고 단계가 여럿이거나 팀과 공유해야 하면 .sh 파일로 만듭니다. 핵심은 세 가지 — shebang으로 인터프리터 명시, set -euo pipefail로 실패 시 멈춤, 위험한 작업엔 가드. 스크립트는 조용히 실패하면 안 됩니다.

요소형태왜
shebang#!/usr/bin/env bash어떤 인터프리터로 실행할지 명시 — 이식성
안전 가드set -euo pipefail에러·미정의 변수·파이프 실패에서 즉시 중단 — 사고 전파 차단
인자 받기"$1", "$@", ${1:?메시지}인자 없으면 메시지 내고 종료(:?) — 빈 인자로 폭주 방지
실행 권한chmod +x reset-db.sh실행 비트가 있어야 ./reset-db.sh로 돌아감
운영 거부 가드환경변수 확인 후 운영이면 exit 1파괴적 스크립트가 운영에서 도는 사고를 코드로 막음
reset-db.sh — 가드를 갖춘 파괴적 스크립트 #!/usr/bin/env bash
# 에러/미정의변수/파이프실패에서 즉시 멈춤 — 절반만 실행되는 사고 방지
set -euo pipefail

# 운영에서는 절대 못 돌게 — 가장 먼저 거부 가드
if [[ "${APP_ENV:-}" == "production" ]]; then
  echo "refusing to reset production DB" >&2; exit 1
fi

# 인자 필수 — 없으면 메시지 내고 종료
DB_NAME="${1:?usage: reset-db.sh <dbname>}"

echo "resetting ${DB_NAME} ..."
dropdb --if-exists "$DB_NAME"
createdb "$DB_NAME"
echo "done."
파괴적 스크립트의 규칙 → 실무 플레이북

DB를 지우거나 데이터를 옮기는 스크립트는 가드를 먼저, 작업을 나중에 짭니다 — set -euo pipefail은 기본이고, 운영 환경 거부 가드와 필수 인자 검증(${1:?})을 맨 위에 둡니다. set -e가 있어도 파괴적 명령 전에 echo로 무엇을 할지 출력하고, 가능하면 확인 프롬프트(read -p)를 넣으세요. 운영 절차로 굳히는 법(승인·롤백·dry-run)은 실무 플레이북 참고.

LOGTL-07

프로젝트 커스텀 명령

alias는 내 셸에만 있습니다. 팀이 공유해야 하는 명령은 저장소 안에 둡니다 — Gradle custom task, 또는 Makefile/just로 명령 모음을. 새 팀원이 make up 한 줄로 환경을 띄울 수 있게 하는 것이 목표입니다.

방법형태왜
Gradle custom task./gradlew printLogs빌드 도구 안에서 — JVM·의존성 컨텍스트가 필요한 작업에
Makefilemake up, make logs어디에나 있는 도구 — 언어 무관 명령 모음의 표준
justjust logs apiMake의 함정(탭·암묵 규칙) 없는 모던 대체 — 인자 받기 깔끔
팀 공유이 파일들을 저장소에 커밋명령이 코드와 함께 버전 관리됨 — 온보딩이 한 줄
Makefile — 팀 공용 명령 모음 # .PHONY: 같은 이름의 파일이 있어도 항상 명령으로 실행
.PHONY: up down logs

up: ## 개발 스택 기동
    docker compose up -d

logs: ## 서비스 로그 실시간 (예: make logs svc=api)
    docker compose logs -f $(svc)

# 사용: make up / make logs svc=api

build.gradle(.kts) — 커스텀 task tasks.register("printRoutes") {
  group = "help"
  doLast { println("GET /api/credits ...") }
}
# 사용: ./gradlew printRoutes
선택 기준

JVM 빌드 컨텍스트(클래스패스·Spring 설정)가 필요하면 Gradle task. 그냥 명령 묶음(docker·db·migration 호출)이면 Makefile이나 just가 가볍고 빠릅니다. Make는 레시피 줄 들여쓰기가 반드시 탭이어야 하는 함정이 있어, 인자를 자주 받는다면 just(justfile)가 더 편합니다. 어느 쪽이든 핵심은 저장소에 커밋해 팀이 공유하는 것 — alias는 개인용, 이건 팀용입니다.

LOGTL-08

치트시트 — 자주 쓰는 한 줄

로그 추적과 명령 자동화에서 반복되는 조합들. 외우기보다 "이런 게 가능하다"를 알아두고 필요할 때 찾으세요.

컨테이너 로그 실시간docker logs
