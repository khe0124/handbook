디자인 시스템과 토큰

디자인 시스템은 UI 키트가 아니라 제품 경험을 일관되게 만드는 운영 체계입니다. 토큰, 컴포넌트, 문서, 코드, 변경 절차가 함께 있어야 실제 제품 속도와 품질을 높입니다.

SYS-01

시스템의 목적

목적설명측정 신호
일관성같은 문제에 같은 해결 방식을 사용페이지별 UI 차이 감소
속도반복 UI를 빠르게 설계하고 구현디자인/개발 리드타임 감소
품질접근성, 반응형, 상태를 기본 품질로 포함QA 반복 지적 감소
확장성브랜드, 테마, 제품 확장에 대응토큰 변경으로 다수 화면 업데이트

SYS-05

성숙도

Level 1kit
Figma 컴포넌트와 일부 코드 컴포넌트가 존재합니다.
Level 2shared
디자인과 코드가 같은 토큰과 variant를 공유합니다.
Level 3governed
기여, 버전, 마이그레이션, 품질 기준이 운영됩니다.
Level 4measured
사용률, 회귀, 접근성, 제품 속도를 지표로 관리합니다.
SYS-06

토큰 파이프라인과 거버넌스

디자인 시스템이 커질수록 Figma, 코드, 문서, 릴리스가 어긋나는 문제가 생깁니다. 이를 줄이려면 design token pipeline과 변경 관리가 필요합니다.

단계역할실무 기준
SourceFigma variables 또는 token repository에서 원천 관리primitive와 semantic token을 분리
Transform플랫폼별 CSS variable, TS constant, Android/iOS token 생성단위, 색상 포맷, theme mapping 자동화
Validatecontrast, naming, breaking change 검사WCAG 기준과 deprecated token 사용 검사
Release버전 태그와 migration note 제공제품팀이 언제 어떤 변경을 적용할지 판단 가능
Case study

다크 모드를 추가할 때 hex 색상을 화면별로 바꾸면 유지보수가 불가능해집니다. semantic token을 먼저 정의하고 component token이 이를 참조하게 만들면 테마 전환은 매핑 변경으로 처리되고, contrast 검증도 파이프라인에서 반복할 수 있습니다.

SYS-07

멀티 브랜드·테마 전략

토큰 시스템의 전문성은 다크 모드나 브랜드 색상 변경에서 드러납니다. primitive 값을 직접 화면에 쓰면 테마 확장이 불가능하고, semantic token을 통해 의미를 유지해야 합니다.

전략설명적합한 상황위험
Single theme하나의 브랜드와 모드만 운영초기 제품, 내부 도구나중에 확장 시 hard-coded style 부채
Mode themelight/dark/high contrast를 semantic token으로 전환장시간 사용, 접근성 요구contrast 자동 검증 없으면 회귀 발생
Brand theme브랜드별 alias mapping화이트라벨, 멀티 제품브랜드별 예외가 component token을 오염
Density themecompact/comfortable spacing과 typography업무형 UI와 일반 UI 병행동일 컴포넌트의 정보 구조가 깨질 수 있음
SYS-08

시스템 지표와 ROI

디자인 시스템은 미학 프로젝트가 아니라 제품 인프라입니다. 따라서 채택률, 리드타임, 회귀 감소, 접근성 품질처럼 운영 지표로 가치를 증명해야 합니다.

지표의미측정 방법개선 액션
Adoption rate제품 화면이 시스템 컴포넌트를 쓰는 비율코드 검색, design auditmigration guide, codemod, 교육
Design-dev parityFigma와 코드가 같은 상태/variant를 갖는지component inventory diffvariant 정리, Storybook 문서화
Lead time반복 UI 제작 속도디자인/개발 cycle time패턴 문서와 템플릿 보강
Quality defects접근성, 반응형, 상태 누락 버그QA ticket taxonomy컴포넌트 내부 품질 기준 강화
System ROI

시스템 지표는 “얼마나 예쁜가”가 아니라 제품팀이 더 적은 의사결정으로 더 일관된 품질을 낼 수 있는지를 보여줘야 합니다. 채택되지 않는 시스템은 존재해도 효과가 없습니다.
