디자인 시스템과 토큰

디자인 시스템은 UI 키트가 아니라 제품 경험을 일관되게 만드는 운영 체계입니다. 토큰, 컴포넌트, 문서, 코드, 변경 절차가 함께 있어야 실제 제품 속도와 품질을 높입니다.

SYS-01

시스템의 목적

목적설명측정 신호
일관성같은 문제에 같은 해결 방식을 사용페이지별 UI 차이 감소
속도반복 UI를 빠르게 설계하고 구현디자인/개발 리드타임 감소
품질접근성, 반응형, 상태를 기본 품질로 포함QA 반복 지적 감소
확장성브랜드, 테마, 제품 확장에 대응토큰 변경으로 다수 화면 업데이트
SYS-02

토큰 구조

토큰은 값의 별칭이 아니라 디자인 결정의 이름입니다.

SYS-04

운영 모델

영역운영 기준위험 신호
Ownership디자인과 프론트엔드 공동 소유한쪽만 관리해 문서와 코드가 분리
Contribution요청, 검토, 승인, 배포 흐름 정의제품팀이 임의 컴포넌트를 계속 생성
Versioningbreaking change와 migration 안내변경 후 기존 화면이 조용히 깨짐
Audit사용률, 중복, 접근성 회귀 점검시스템보다 예외 CSS가 더 많아짐
SYS-05

성숙도

Level 1kit
Figma 컴포넌트와 일부 코드 컴포넌트가 존재합니다.
Level 2shared
디자인과 코드가 같은 토큰과 variant를 공유합니다.
Level 3governed
기여, 버전, 마이그레이션, 품질 기준이 운영됩니다.
Level 4measured
사용률, 회귀, 접근성, 제품 속도를 지표로 관리합니다.
SYS-06

토큰 파이프라인과 거버넌스

디자인 시스템이 커질수록 Figma, 코드, 문서, 릴리스가 어긋나는 문제가 생깁니다. 이를 줄이려면 design token pipeline과 변경 관리가 필요합니다.

단계역할실무 기준
SourceFigma variables 또는 token repository에서 원천 관리primitive와 semantic token을 분리
Transform플랫폼별 CSS variable, TS constant, Android/iOS token 생성단위, 색상 포맷, theme mapping 자동화
Validatecontrast, naming, breaking change 검사WCAG 기준과 deprecated token 사용 검사
Release버전 태그와 migration note 제공제품팀이 언제 어떤 변경을 적용할지 판단 가능
Case study

다크 모드를 추가할 때 hex 색상을 화면별로 바꾸면 유지보수가 불가능해집니다. semantic token을 먼저 정의하고 component token이 이를 참조하게 만들면 테마 전환은 매핑 변경으로 처리되고, contrast 검증도 파이프라인에서 반복할 수 있습니다.
