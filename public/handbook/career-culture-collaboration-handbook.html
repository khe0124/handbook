컬처·압박 면접

BP-01

Q. 자기소개를 해주세요.

핵심 개념

자기소개는 경력 요약이 아니라 면접 전체의 프레임을 여는 답변입니다. 내가 어떤 문제를 잘 푸는 개발자인지 한 문장으로 잡아야 합니다.

개념 해부

자기소개는 모든 경력을 압축하는 시간이 아니라 면접관이 이후 질문을 걸 수 있는 좌표를 제공하는 시간입니다. 강점 한 문장, 대표 문제, 사용한 방법, 앞으로 맡고 싶은 역할이 이어져야 합니다. 기술명 나열보다 “어떤 문제를 반복적으로 잘 해결했는가”가 남아야 합니다.

30초 답변

저는 복잡한 업무 도메인을 사용 가능한 제품 UI와 안정적인 데이터 흐름으로 구조화하는 데 강점이 있는 개발자입니다. 화면 구현뿐 아니라 입력, 검증, 상태관리, API 계약, 운영 이슈까지 연결해 기능이 실제 사용자 흐름 안에서 동작하도록 만드는 데 관심이 있습니다.

90초 답변

구체 프로젝트를 하나 붙입니다. 예를 들어 “복잡한 입력 화면에서 Excel, 직접 입력, grid edit를 공통 모델로 정리하고 validation과 저장 흐름을 분리했다”처럼 강점을 증거와 연결합니다. 마지막에는 앞으로 확장하고 싶은 방향을 말하되, 현재 경험의 범위를 과장하지 않습니다.

질문답변 방향
본인의 차별점은?도메인 규칙을 화면 상태와 API 계약으로 번역하는 능력으로 답합니다.
풀스택이라고 볼 수 있나요?프론트엔드 중심 경험과 백엔드/API 설계 이해를 구분해 말합니다.
가장 강한 프로젝트는?문제, 구조, 결과가 선명한 프로젝트 하나로 연결합니다.

오답/주의점

주의

기술 스택을 모두 나열하지 마세요. 면접관이 기억할 중심 문장 하나가 필요합니다.

BP-02

Q. 왜 이직하려고 하나요?

핵심 개념

이직 사유는 불만 설명이 아니라 다음 환경에서 더 잘 발휘하고 싶은 역량과 성장 방향을 말하는 질문입니다.

개념 해부

이직 사유는 이전 회사를 평가하는 답변이 아니라 다음 환경에서 어떤 문제를 더 잘 풀고 싶은지 설명하는 답변입니다. 불만을 말하면 방어적으로 들리고, 성장 욕구만 말하면 추상적으로 들립니다. 현재 경험에서 확장하고 싶은 책임 범위와 지원 회사의 문제를 연결해야 합니다.

30초 답변

이전 경험에서 복잡한 도메인과 제품 UI를 구조화하는 일을 해왔고, 이제는 그 강점을 API, 데이터 모델, 운영 관점까지 더 넓게 연결하는 환경에서 일하고 싶습니다. 단순히 다른 기술을 써보고 싶다는 것보다, 제품과 시스템 경계를 더 깊게 책임지는 방향으로 성장하고 싶습니다.

90초 답변

부정적인 이유는 짧게 정리하고, 앞으로의 방향을 길게 말합니다. “더 큰 트래픽”, “더 깊은 백엔드”, “더 좋은 문화”처럼 추상적으로 말하기보다 어떤 문제를 더 잘 풀고 싶은지 말해야 합니다. 회사의 제품/도메인/팀 방식과 연결하면 설득력이 올라갑니다.

꼬리질문

질문답변 방향
현재 회사에서 해결할 수 없나요?현재 환경에서 배운 점을 인정하고, 다음 단계의 문제 범위가 다르다고 설명합니다.
우리 회사여야 하는 이유는?제품, 기술 문제, 사용자, 팀 방식 중 2개 이상과 본인 경험을 연결합니다.
금방 또 옮기지 않을까요?찾는 환경과 기여 방식을 구체화해 일관성을 보여줍니다.

오답/주의점

주의

이전 회사 비판으로 들리면 손해입니다. 배운 점과 다음 목표를 중심에 두세요.

BP-03

Q. 실패했던 기술 선택은 무엇인가요?

핵심 개념

실패 경험은 약점 고백이 아니라 판단 기준이 어떻게 발전했는지 보여주는 질문입니다.

개념 해부

실패 경험은 잘못을 고백하는 항목이 아니라 판단 기준이 어떻게 개선됐는지 보는 항목입니다. 당시 제약, 선택 이유, 놓친 신호, 피해를 줄인 방법, 이후 바꾼 기준이 있어야 합니다. “다음부터 조심했다”가 아니라 재발 방지 장치가 남아야 합니다.

30초 답변

당시에는 빠른 구현을 우선해 단순한 상태 구조로 시작했지만, 입력 경로와 validation이 늘어나면서 원본 데이터와 편집 중 draft가 섞이는 문제가 생겼습니다. 이후 serverData, draftData, validationErrors를 분리하고, 다음부터는 입력 채널과 저장 모델을 먼저 나누는 기준을 갖게 됐습니다.

90초 답변

좋은 실패 답변은 “당시 전제는 합리적이었다 → 어떤 조건이 바뀌며 문제가 드러났다 → 영향은 무엇이었다 → 어떻게 수정했다 → 다음 판단 기준은 무엇인가”로 구성됩니다. 무조건 후회하거나 남 탓하지 말고, 판단 기준의 변화가 보여야 합니다.

꼬리질문

질문답변 방향
왜 처음부터 몰랐나요?당시 정보와 일정 제약을 설명하되, 놓친 신호를 인정합니다.
피해는?사용자/QA/개발 비용 측면에서 솔직하게 말합니다.
지금은?초기 설계에서 확인할 체크리스트로 전환됐다고 말합니다.

오답/주의점

주의

“큰 실패는 없었습니다”는 약합니다. 작은 실패라도 판단이 바뀐 사례가 더 좋습니다.

BP-04

Q. 협업 갈등이나 설득 경험을 말해보세요.

핵심 개념

갈등 질문은 성격을 묻는 것이 아니라 목표 충돌을 사실과 선택지로 정리해 합의로 이끌 수 있는지 보는 질문입니다.

개념 해부

갈등 답변의 핵심은 누가 맞았는지가 아니라 목표가 달랐을 때 어떻게 사실과 기준으로 다시 정렬했는지입니다. 감정, 직급, 취향이 아니라 사용자 영향, 유지보수 비용, 일정 위험 같은 공유 기준으로 설득해야 합니다. 결과가 내 의견대로 되지 않았어도 합리적 합의 과정을 말할 수 있어야 합니다.

30초 답변

갈등이 생기면 먼저 서로의 목표가 무엇인지 분리합니다. 일정, 품질, 사용자 영향, 기술 부채 중 어떤 기준이 충돌하는지 정리하고, 선택지별 비용과 리스크를 표로 보여준 뒤 지금 선택할 것과 나중에 보완할 것을 나눠 합의하려고 합니다.

90초 답변

예시는 STAR로 말합니다. 상황과 목표, 본인이 한 행동, 결과를 분리합니다. 중요한 것은 “제가 맞았습니다”가 아니라 팀이 더 좋은 결정을 내리도록 어떤 정보와 기준을 제공했는지입니다. 설득이 안 됐던 경험도 배운 점이 명확하면 좋은 답변이 됩니다.

꼬리질문

질문답변 방향
과한 설계라는 피드백은?현재 규모에 맞는 최소 구조와 나중 확장 지점을 분리해 조정합니다.
PM과 우선순위 충돌은?사용자 영향, 일정, 기술 리스크를 드러내고 선택지를 만듭니다.
리더십은?사람을 지시한 경험보다 기준과 프로세스를 만든 경험을 말합니다.

오답/주의점

주의

상대가 틀렸다는 식으로 말하지 마세요. 좋은 협업 답변은 공동 목표와 판단 기준을 보여줍니다.

BP-05

Q. 압박 질문에는 어떻게 대응하나요?

핵심 개념

압박 질문은 공격이 아니라 답변의 신뢰도와 경계를 확인하는 과정입니다. 짧게 인정하고 기준을 다시 세운 뒤 증거 또는 검증 방법으로 이동합니다.

개념 해부

압박 질문은 면접관이 공격하는 장면이라기보다 답변의 과장, 경계, 검증 가능성을 확인하는 장면입니다. 먼저 인정할 것은 인정하고, 경험의 범위를 분리한 뒤, 판단 기준과 확인 방법을 말하면 신뢰가 올라갑니다. 방어적으로 길어지는 순간 핵심 메시지가 흐려집니다.

30초 답변

압박 질문을 받으면 먼저 전제를 확인하고, 직접 경험한 부분과 설계 지식인 부분을 나눠 말합니다. 수치가 없으면 꾸미지 않고 재현 조건, 변경 전후 증거, QA 감소 같은 대체 증거를 말합니다. 모르는 부분은 모른다고 말하되 어떻게 확인할지까지 답합니다.

90초 답변

압박 질문대응 문장
직접 해본 게 맞나요?“직접 담당한 범위는 여기까지이고, 이 부분은 구조를 이해하고 협업했습니다.”
수치가 있나요?“정량 지표는 남기지 못했습니다. 대신 재현 조건과 QA 포인트 감소, 수정 범위 축소를 증거로 말할 수 있습니다.”
그 구조가 과한 것 아닌가요?“현재 규모에서는 과할 수 있습니다. 그래서 최소 구조와 확장 시점을 나눠 판단합니다.”
백엔드 경험이 부족한 것 아닌가요?“깊은 운영 경험은 제한적입니다. 다만 API 계약, validation, 데이터 흐름을 이해하고 프론트 구조에 반영해왔습니다.”

모르면?모른다고 말하고, 어떤 문서/로그/실험으로 확인할지 답합니다.
방어적으로 느껴지면?인정 → 기준 → 증거 순서로 짧게 답합니다.
마지막 한마디는?본인이 풀 수 있는 문제와 팀에 기여할 방식을 한 문장으로 정리합니다.

오답/주의점

주의

압박 질문에서 즉흥적으로 경험을 부풀리면 이후 꼬리질문에서 무너집니다. 경계를 지키는 것이 더 강합니다.

BP-06

Q. 왜 우리 회사인가요?

핵심 개념

지원 동기는 회사 칭찬이 아니라 회사가 풀고 있는 문제와 내 경험이 만나는 지점을 말해야 합니다.

개념 해부

지원 동기는 회사 이름을 칭찬하는 답변이 아니라 회사의 문제와 내 경험의 접점을 제시하는 답변입니다. 제품, 고객, 채용 JD, 기술 과제를 근거로 삼아야 합니다. 범용 문장을 줄이고 “이 회사라서 꺼낼 수 있는 내 경험”을 앞에 둬야 합니다.

30초 답변

이 회사의 제품은 복잡한 사용자 흐름과 데이터 신뢰성이 중요해 보였습니다. 저는 복잡한 입력, validation, API 계약, 운영 이슈를 사용자 경험으로 연결해온 경험이 있어 이 문제에 기여할 수 있다고 봤습니다.

90초 답변

답변은 제품/기술/팀 방식 중 2개 이상과 본인 경험을 연결합니다. 예를 들어 B2B 업무 도구라면 “정확한 입력, 권한, audit, 실패 복구가 중요하다”로 연결하고, 실시간/대용량이라면 성능과 관측성 경험으로 연결합니다. 막연히 성장하고 싶다는 말보다 내가 기여할 문제를 먼저 말합니다.

꼬리질문

질문답변 방향
다른 회사도 같은데요?이 회사 제품의 사용자, 도메인, 기술 문제를 구체화합니다.
무엇을 기여?첫 3개월에 파악하고 개선할 영역을 말합니다.
관심만 있는 것 아닌가요?과거 경험과 연결해 근거를 둡니다.

면접에서 말할 증거

회사 제품에서 관찰한 문제 2개, 내 프로젝트 경험 2개, 입사 후 확인할 질문 3개를 준비합니다.

오답/주의점

주의

복지, 유명세, 성장만 말하면 회사와 내 역량의 접점이 약합니다.

BP-07

Q. 가장 자랑스러운 프로젝트는 무엇인가요?

핵심 개념

자랑스러운 프로젝트는 규모보다 본인의 판단과 결과가 선명한 프로젝트여야 합니다.

개념 해부

자랑스러운 프로젝트는 가장 큰 프로젝트가 아니라 본인의 판단 흔적이 가장 선명한 프로젝트여야 합니다. 문제 정의, 제약, 선택지, 본인 역할, 결과, 다시 한다면 바꿀 점이 있어야 깊은 질문을 견딥니다. 팀 성과를 말하되 내가 실제로 책임진 단위를 흐리지 않아야 합니다.

30초 답변

가장 자랑스러운 프로젝트는 복잡한 입력 흐름을 사용자가 수정 가능한 구조로 바꾼 경험입니다. 여러 입력 채널을 공통 모델로 정리하고 validation과 오류 복구를 분리해 QA와 사용자 혼란을 줄였습니다.

90초 답변

STAR로 말합니다. 상황은 복잡한 입력과 잦은 오류, 과제는 사용자가 오류를 찾고 저장할 수 있게 하는 것, 행동은 상태 분리와 validation 구조화, 결과는 오류 위치 명확화와 수정 범위 축소입니다. 수치가 있다면 말하고, 없으면 재현 조건과 품질 개선 증거를 말합니다.

꼬리질문

질문답변 방향
본인 기여는?설계/구현/조율/검증 중 직접 한 범위를 분리합니다.
어려웠던 점은?도메인 규칙, 상태 구조, API 계약 중 하나를 깊게 설명합니다.
결과 수치는?정량 또는 대체 증거를 솔직히 말합니다.

면접에서 말할 증거

상태 구조도, validation rule, QA 포인트 감소, 오류 재현/수정 기록을 준비합니다.

오답/주의점

주의

프로젝트 규모만 크고 본인 판단이 드러나지 않으면 답변이 약합니다.

BP-08

Q. 일정이 부족할 때 어떻게 대응하나요?

핵심 개념

일정 압박 답변은 야근 의지가 아니라 범위 조정, 위험 공개, 품질 기준 유지 능력을 보는 질문입니다.

개념 해부

일정 압박 답변은 희생을 많이 했다는 이야기가 아니라 범위, 품질, 위험을 어떻게 조정했는지 보는 질문입니다. 마감이 짧을수록 필수 기능, 나중에 미룰 기능, 실패 시 피해가 큰 기능을 나눠야 합니다. 야근보다 중요한 것은 조기 공유와 trade-off 합의입니다.

30초 답변

먼저 반드시 지켜야 할 사용자 흐름과 나중으로 미룰 수 있는 범위를 나눕니다. 리스크를 숨기지 않고 선택지를 제시하며, 최소 품질 기준인 데이터 정합성, 보안, 장애 가능성은 타협하지 않습니다.

90초 답변

일정이 부족하면 기능을 모두 얕게 끝내기보다 핵심 경로를 안정적으로 닫는 것이 중요합니다. 선택지는 “범위 축소”, “단계 배포”, “feature flag”, “임시 manual operation”, “후속 보완”으로 나눠 제안합니다. 기술 부채를 만들었다면 기록하고 제거 시점을 합의합니다.

꼬리질문

질문답변 방향
품질과 일정 충돌은?사용자 영향과 되돌릴 수 없는 위험을 기준으로 판단합니다.
PM 설득은?선택지별 비용과 리스크를 표로 보여줍니다.
부채는?기록, owner, 제거 일정을 둡니다.

면접에서 말할 증거

scope cut 목록, risk table, feature flag 계획, 후속 부채 제거 ticket을 말합니다.

오답/주의점

주의

무조건 밤새서 맞췄다는 답은 지속 가능성과 판단 기준이 약해 보일 수 있습니다.

BP-09

Q. 과한 설계라는 피드백을 받으면 어떻게 하나요?

핵심 개념

과한 설계 피드백은 방어할 문제가 아니라 현재 규모와 변경 가능성에 맞는 최소 구조를 다시 찾는 과정입니다.

개념 해부

과한 설계 피드백은 자존심 문제가 아니라 현재 문제 크기에 비해 추상화 비용이 큰지 다시 보는 신호입니다. 추상화는 미래 변경을 싸게 만들 때만 이득입니다. 피드백을 받으면 어떤 변경 가능성을 가정했는지, 지금은 무엇을 단순화할지 분리해 답해야 합니다.

30초 답변

먼저 피드백이 맞는지 사용 사례와 변경 가능성을 다시 봅니다. 현재 반복이 적고 불확실성이 크다면 단순화하고, 반복되는 규칙과 실패 비용이 명확한 부분만 추상화합니다.

90초 답변

좋은 설계는 복잡한 구조가 아니라 변경 비용을 낮추는 구조입니다. 그래서 “지금 필요한 최소 구조”, “확장 시 분리할 지점”, “되돌리기 어려운 결정”을 나눠 설명합니다. 피드백을 받으면 코드 사용처와 실제 중복, 테스트 비용을 확인해 조정합니다.

꼬리질문

질문답변 방향
본인이 틀렸다면?근거를 확인하고 단순화한 경험을 말합니다.
어디까지 추상화?반복과 변경 가능성이 확인된 부분만 합니다.
설계 고집은?팀의 이해 비용까지 설계 비용으로 봅니다.

면접에서 말할 증거

삭제한 추상화, 단순화 전후 코드, 유지한 경계의 이유를 말합니다.

오답/주의점

주의

내 설계가 맞았다는 방어보다 어떤 기준으로 조정했는지가 중요합니다.

BP-10

Q. 멘토링이나 리더십 경험이 있나요?

핵심 개념

리더십은 직급이 아니라 팀이 더 나은 결정을 반복할 수 있게 기준과 피드백 루프를 만드는 행동입니다.

개념 해부

멘토링과 리더십은 지시가 아니라 팀원이 더 나은 판단을 반복하도록 기준과 피드백 루프를 만드는 일입니다. 코드리뷰 기준, 문서화, 페어링, 문제 분해 방식이 실제 증거가 됩니다. ‘도와줬다’보다 상대가 이후 독립적으로 해낼 수 있게 만든 구조를 말해야 합니다.

30초 답변

공식 리더가 아니더라도 복잡한 이슈를 정리해 팀이 같은 기준으로 판단하도록 도운 경험이 있습니다. 리뷰 기준, 질문 목록, 재현 절차, 문서화를 통해 개인 지식이 팀 지식이 되게 만들었습니다.

90초 답변

멘토링은 답을 대신 주는 것이 아니라 문제를 쪼개는 방법을 알려주는 것이라고 봅니다. 예를 들어 버그 재현, API 계약 확인, 상태 위치 판단 같은 체크리스트를 공유하고, 리뷰에서는 단순 취향보다 사용자 영향과 유지보수 기준으로 피드백했습니다.

꼬리질문

질문답변 방향
성과는?리뷰 반복 감소, 온보딩 시간 단축, 재현 절차 정착 같은 증거를 말합니다.
갈등은?사람보다 기준을 논의했다고 말합니다.
리더 경험이 없으면?공식 권한 없이 만든 영향력을 말합니다.

면접에서 말할 증거

체크리스트, 리뷰 템플릿, 문서, 온보딩 자료, 반복 버그 감소 사례를 준비합니다.

오답/주의점

주의

사람을 많이 도와줬다는 말보다 어떤 시스템을 남겼는지가 강합니다.

BP-11

Q. 본인을 상급 개발자라고 볼 수 있는 근거는 무엇인가요?

핵심 개념

상급 개발자 근거는 연차가 아니라 문제 정의, trade-off 판단, 운영 책임, 타인에게 설명 가능한 기준입니다.

개념 해부

상급 개발자 근거는 연차나 기술 스택 수가 아니라 모호한 문제를 정의하고 위험을 줄이는 방식에서 나옵니다. 구현, 리뷰, 운영, 커뮤니케이션, 의사결정 기준을 함께 보여줘야 합니다. 특히 내가 만든 기준이 팀에 남았는지가 강한 증거입니다.

30초 답변

저는 기능 구현을 넘어 데이터 흐름, 상태 경계, API 계약, 실패 복구, 운영 증거까지 함께 봅니다. 복잡한 문제를 작은 결정으로 나누고, 팀이 유지할 수 있는 구조와 문서로 남기는 점이 강점입니다.

90초 답변

근거는 프로젝트 하나로 연결합니다. 어떤 복잡성을 발견했고, 어떤 대안을 비교했으며, 어떤 trade-off를 선택했고, 결과적으로 어떤 품질이나 협업 개선이 있었는지 말합니다. 직접 경험하지 않은 영역은 상급이라고 포장하지 않고 학습/설계 지식으로 분리합니다.

꼬리질문

질문답변 방향
정량 근거는?수치 또는 대체 증거를 말합니다.
부족한 점은?운영/백엔드 등 개선 중인 영역과 학습 방식을 말합니다.
팀 영향은?문서, 리뷰, 재사용 구조, 장애 대응 기준을 말합니다.

면접에서 말할 증거

대표 프로젝트 1개, 의사결정 표, 산출물, 결과 수치 또는 대체 증거를 준비합니다.

오답/주의점

주의

상급이라는 단어를 스스로 강하게 주장하기보다 근거가 그렇게 보이게 해야 합니다.

BP-12

Q. 직접 해본 게 맞나요?

핵심 개념

이 질문은 공격이 아니라 경험의 경계를 확인하는 질문입니다. 직접 담당 범위와 협업/학습 범위를 분리해야 합니다.

개념 해부

직접 해봤는지 묻는 질문은 경험 범위를 검증하는 질문입니다. 직접 구현, 설계 참여, 리뷰, 학습 지식을 섞어 말하면 신뢰가 무너집니다. 각 범위를 분리하고 직접 하지 않은 영역은 어떤 기준으로 이해하고 있는지 말하는 편이 안전합니다.

30초 답변

직접 담당한 범위는 화면 상태 구조, validation, API 연결, 오류 복구 흐름입니다. 백엔드 내부 구현은 협업하며 contract와 error format을 맞췄고, 운영 지표는 로그와 Network trace 중심으로 확인했습니다.

90초 답변

답변은 “제가 직접 한 것”, “리뷰/협업한 것”, “개념으로 이해한 것”을 나눕니다. 직접 한 일은 코드 수준으로 설명하고, 협업한 일은 어떤 질문과 합의를 했는지 말합니다. 모르는 부분은 숨기지 말고 확인 방법을 제시합니다.

꼬리질문

질문답변 방향
코드 설명 가능?상태 구조, 함수 책임, 테스트를 설명합니다.
협업 범위는?API contract, error mapping, 배포 확인 등을 말합니다.
모르는 부분은?문서, 로그, 실험으로 확인할 방법을 말합니다.

면접에서 말할 증거

내가 쓴 모듈, PR, 테스트, 문서, 합의 기록을 준비합니다.

오답/주의점

주의

모든 것을 직접 했다고 말하면 깊은 꼬리질문에서 신뢰가 무너집니다.

BP-13

Q. 수치가 있나요?

핵심 개념

수치 질문은 성과 과장을 잡는 질문입니다. 수치가 없으면 없다고 말하고 대체 증거를 제시해야 합니다.

개념 해부

수치 질문은 성과를 부풀렸는지 확인하는 장치입니다. 정량 수치가 없으면 없다고 말하고, 대신 재현 조건, 전후 비교, QA 감소, 장애 감소, 코드 삭제량, 사용자 피드백 같은 대체 증거를 제시할 수 있습니다. 근거 없는 숫자보다 정직한 증거가 낫습니다.

30초 답변

정량 수치를 남긴 경우에는 전후 조건과 함께 말합니다. 수치가 없으면 꾸미지 않고, 재현 조건, QA 포인트 감소, 오류 위치 명확화, 수정 범위 축소, 사용자 문의 감소 같은 대체 증거를 말합니다.

90초 답변

수치는 맥락이 없으면 위험합니다. 어떤 환경에서, 어떤 기간에, 어떤 기준으로 측정했는지 말해야 합니다. 수치가 없는 프로젝트라면 다음에는 어떤 metric을 심을지까지 답하면 학습 능력이 드러납니다.

꼬리질문

질문답변 방향
왜 안 남겼나요?당시 한계와 다음 개선 기준을 말합니다.
대체 증거는?QA 기록, 오류 재현, 로그, 리뷰 피드백을 말합니다.
다음에는?성능, 오류율, 처리 시간, 사용자 행동 metric을 제안합니다.

면접에서 말할 증거

측정 조건, before/after, QA ticket, 로그, profiler 캡처를 준비합니다.

오답/주의점

주의

근거 없는 퍼센트 개선 수치를 말하는 것은 가장 위험합니다.

BP-14

Q. 백엔드 경험이 부족한 것 아닌가요?

핵심 개념

부족한 영역은 인정하되, 서버와 데이터 흐름을 이해하고 협업에서 어떤 가치를 냈는지 보여줘야 합니다.

개념 해부

GIT-01

Q. 개발 협업을 잘한다는 것은 무엇이라고 보나요?

핵심 개념

협업은 태도 문제가 아니라 변경을 다른 사람이 이해하고 검증하고 되돌릴 수 있게 만드는 엔지니어링 능력입니다.

개념 해부

협업을 잘한다는 것은 코드를 많이 쓰는 능력보다 변경의 의도, 범위, 위험을 팀이 이해할 수 있게 만드는 능력입니다. Git 기록, PR 설명, 리뷰 응답, 릴리즈 메모는 모두 미래의 디버깅 비용을 줄이는 자료입니다. 좋은 협업은 개인의 속도보다 팀의 재현성과 회복성을 높입니다.

30초 답변

좋은 협업은 변경 목적, 위험, 검증 근거를 명확히 남기는 것입니다. commit과 PR을 작게 만들고, 리뷰어가 봐야 할 의사결정과 위험을 먼저 설명하며, 배포 후 문제가 생겼을 때 rollback이나 feature flag로 되돌릴 수 있게 준비합니다.

90초 답변

면접에서는 “소통을 잘합니다”보다 구체적 루틴을 말해야 합니다. 예를 들어 요구사항이 불명확하면 API 계약, 화면 상태, 데이터 규칙으로 쪼개고, 큰 변경은 리팩터링·스키마·기능 변경을 분리합니다. 리뷰에서는 스타일보다 버그, 호환성, 권한, 성능, 테스트 공백을 우선 봅니다. 의사결정은 이슈나 ADR에 남겨 나중에 맥락을 복구할 수 있게 합니다.

GIT-02

Q. 좋은 commit 단위는 어떻게 정하나요?

핵심 개념

좋은 commit은 나중에 변경 이유를 추적하고, 문제가 생겼을 때 revert하거나 bisect할 수 있는 의미 단위입니다.

개념 해부

좋은 commit은 저장 지점이 아니라 검토 가능한 의미 단위입니다. 한 commit이 하나의 의도와 되돌림 단위를 가지면 bisect, revert, review가 쉬워집니다. 포맷 변경과 동작 변경을 섞으면 문제가 생겼을 때 원인을 찾기 어렵습니다.

30초 답변

commit은 “한 가지 의도” 단위로 나눕니다. 포맷팅, 리팩터링, 기능 변경, 테스트 추가, 마이그레이션을 섞지 않고, 메시지는 무엇을 바꿨는지보다 왜 바꿨는지 드러나게 작성합니다. 되돌렸을 때 시스템이 깨지지 않는 단위인지도 봅니다.

90초 답변

작은 commit은 리뷰와 rollback에 유리하지만 너무 잘게 쪼개져 빌드가 깨지는 commit이 많으면 추적성이 떨어집니다. 기준은 “이 commit만 봐도 의도가 이해되고, 이 commit까지 checkout했을 때 검증 가능한가”입니다. 대규모 변경은 준비 리팩터링, 호환성 계층 추가, 기능 전환, cleanup commit으로 나눕니다.

질문답변 방향

GIT-03

Q. merge와 rebase는 언제 선택하나요?

핵심 개념

merge와 rebase의 차이는 히스토리 모양과 이력 보존 방식입니다. 정답은 팀의 워크플로와 공유 여부에 따라 달라집니다.

개념 해부

merge는 브랜치의 실제 병합 맥락을 남기고, rebase는 커밋을 새 base 위에 재적용해 선형 기록을 만듭니다. 공유 브랜치의 공개 커밋을 rebase하면 다른 사람의 기준점이 바뀔 수 있습니다. 선택 기준은 기록 가독성, 팀 정책, 충돌 해결 책임입니다.

30초 답변

rebase는 개인 feature branch를 최신 main 위로 정리해 선형 히스토리를 만들 때 유리합니다. merge는 브랜치 통합 맥락을 보존하고 여러 사람의 작업 흐름을 유지할 때 유리합니다. 이미 공유된 commit은 rebase로 rewrite하지 않는 것이 원칙입니다.

90초 답변

면접에서는 명령어 차이보다 리스크를 말해야 합니다. rebase는 깔끔한 history를 만들지만 commit hash가 바뀌므로 공유 브랜치에서 쓰면 동료 작업을 꼬이게 할 수 있습니다. merge는 맥락을 보존하지만 merge commit이 많으면 흐름이 복잡해질 수 있습니다. 팀이 squash merge를 쓰는지, release branch를 어떻게 관리하는지도 함께 봅니다.

GIT-04

Q. revert, reset, restore는 어떻게 구분하나요?

핵심 개념

복구 명령은 이력을 보존할지, 브랜치 포인터를 옮길지, 작업 디렉터리 파일만 되돌릴지에 따라 선택합니다.

개념 해부

revert는 새 commit으로 변경을 되돌려 공개 기록을 보존하고, reset은 현재 브랜치 포인터와 index/working tree를 이동시킵니다. restore는 파일 단위 작업 복원에 가깝습니다. 협업 브랜치에서는 기록을 지우는 동작보다 되돌림을 기록으로 남기는 방식이 안전합니다.

30초 답변

revert는 기존 commit을 취소하는 새 commit을 만들어 이력을 보존합니다. reset은 브랜치 포인터를 옮겨 이력을 바꾸므로 공유 브랜치에서는 위험합니다. restore는 특정 파일의 작업 디렉터리나 staging 상태를 되돌릴 때 씁니다. 운영에 올라간 변경은 보통 revert나 hotfix commit이 안전합니다.

90초 답변

장애 대응에서는 추적성이 중요합니다. 이미 main에 merge되고 배포된 변경을 없애야 한다면 revert로 반대 commit을 남기는 편이 안전합니다. reset은 로컬에서 commit을 정리하거나 아직 공유되지 않은 브랜치를 되돌릴 때 사용합니다. 잘못된 명령을 막기 위해 보호 브랜치, required review, CI check도 필요합니다.

꼬리질문

질문답변 방향
revert가 충돌나면?원래 변경 이후 코드가 바뀐 것이므로 의도를 확인하며 수동 해결합니다.
reset --hard는 언제?로컬 변경을 완전히 버릴 때만 쓰며 공유 변경에는 쓰지 않습니다.
reflog는?브랜치가 가리켰던 이전 위치를 찾아 실수 복구에 도움을 줍니다.

면접에서 말할 증거

장애 rollback 때 왜 revert를 골랐는지, 어떤 검증 후 배포했는지 말하면 운영 감각이 보입니다.

GIT-05

Q. PR은 얼마나 작아야 하고 무엇을 써야 하나요?

핵심 개념

PR 크기는 라인 수가 아니라 리뷰어가 한 번에 이해해야 하는 의사결정 수로 판단합니다.

개념 해부

PR은 코드 묶음이 아니라 reviewer에게 주는 변경 설명서입니다. 문제, 접근, 영향 범위, 검증, 위험, 배포/rollback 고려가 있어야 reviewer가 핵심을 볼 수 있습니다. PR이 너무 크면 리뷰어는 스타일만 보거나 중요한 설계 위험을 놓치기 쉽습니다.

30초 답변

좋은 PR은 목적, 배경, 변경 범위, 위험, 검증 결과가 첫 화면에서 보입니다. schema 변경, 리팩터링, 기능 변경, UI 변경, 테스트 변경을 한꺼번에 섞지 않고 리뷰 가능한 단위로 나눕니다. 리뷰어가 집중해야 할 파일이나 위험도 표시합니다.

90초 답변

PR 설명에는 “왜 이 변경이 필요한가”, “대안은 무엇이었나”, “호환성/권한/성능/마이그레이션 위험은 무엇인가”, “어떻게 검증했나”가 들어가야 합니다. 스크린샷, 테스트 결과, 쿼리 계획, migration dry-run, feature flag 상태 같은 증거를 남기면 리뷰 품질이 올라갑니다.

꼬리질문

질문답변 방향
큰 PR을 피할 수 없으면?리뷰 가이드, 파일별 의도, 선행 PR 분리, walkthrough를 제공합니다.
리뷰어가 놓치기 쉬운 부분은?권한, migration, backward compatibility, error handling, race condition입니다.
검증 증거는?테스트, 스크린샷, 로그, 성능 수치, 배포 체크리스트를 남깁니다.

오답/주의점

주의

“작게 만들겠습니다”보다 어떤 기준으로 쪼갤지 말해야 합니다.

GIT-06

Q. 코드리뷰에서 무엇을 우선순위로 보나요?

핵심 개념

코드리뷰는 취향 교정이 아니라 결함, 요구사항 누락, 운영 위험, 유지보수 비용을 줄이는 활동입니다.

개념 해부

코드리뷰는 취향 교정 시간이 아니라 버그, 보안, 성능, 유지보수성, 계약 변경을 찾는 품질 활동입니다. 먼저 동작과 위험을 보고, 스타일은 자동화 가능한 도구로 넘기는 편이 낫습니다. 좋은 리뷰는 대안을 제시하거나 질문으로 의도를 확인합니다.

30초 답변

우선 요구사항을 만족하는지, 권한과 데이터 정합성 문제가 없는지, 실패 케이스와 테스트가 충분한지 봅니다. 그 다음 복잡도, 네이밍, 중복, 성능, 접근성 같은 품질을 봅니다. 스타일 논쟁은 formatter와 lint에 맡기는 편이 좋습니다.

90초 답변

좋은 리뷰 코멘트는 문제와 근거와 제안을 분리합니다. “이건 별로입니다”가 아니라 “이 경로에서 null이면 저장이 실패하고 사용자에게 조용히 사라질 수 있습니다. guard를 추가하거나 서버 validation error를 표시하는 게 필요합니다”처럼 말합니다. 확신이 낮으면 질문 형태로 남기고, blocking과 non-blocking을 구분합니다.

꼬리질문

질문답변 방향
리뷰 의견이 충돌하면?기준, 위험, 데이터, 팀 컨벤션으로 판단하고 필요하면 ADR로 남깁니다.
리뷰가 공격적으로 느껴지면?사람이 아니라 코드와 사용자 영향에 초점을 맞추고 예시를 듭니다.
리뷰를 받는 태도는?방어보다 의도 설명, 근거 제시, 필요하면 작은 수정으로 답합니다.

면접에서 말할 증거

실제로 리뷰에서 잡은 결함, 테스트 보강, 장애 예방 사례를 하나 준비하면 답변이 살아납니다.

GIT-07

Q. Git conflict가 나면 어떻게 해결하나요?

핵심 개념

충돌 해결은 양쪽 코드를 기계적으로 합치는 일이 아니라 두 변경의 의도를 이해하고 최종 동작을 다시 검증하는 일입니다.

개념 해부

conflict는 Git이 같은 문맥의 변경을 자동으로 합칠 근거가 부족하다는 신호입니다. 둘 중 하나를 고르는 일이 아니라 양쪽 변경 의도를 이해해 새로운 결과를 만들어야 합니다. 해결 후에는 테스트와 관련 파일 검토로 semantic conflict까지 확인해야 합니다.

30초 답변

먼저 충돌 파일에서 양쪽 변경 의도를 확인합니다. 단순 텍스트 병합이 아니라 요구사항이 충돌하는지 봅니다. 해결 후에는 관련 테스트와 빌드를 돌리고, 충돌이 난 영역의 동작을 직접 검증합니다. 이해가 안 되는 변경은 작성자에게 확인합니다.

90초 답변

복잡한 충돌은 최신 main을 자주 반영하지 않았거나 PR 범위가 너무 큰 신호일 수 있습니다. 해결 과정에서는 ours/theirs를 무작정 선택하지 않고, 각 변경이 추가한 테스트나 의도를 비교합니다. schema, API contract, generated file 충돌은 특히 조심해야 하며 자동 생성 파일은 원본을 수정한 뒤 다시 생성하는 편이 안전합니다.

꼬리질문

질문답변 방향
generated file 충돌은?산출물을 손으로 맞추기보다 원본을 병합하고 재생성합니다.
충돌을 줄이려면?작은 PR, 파일 소유권, 공통 리팩터링 선행, 자주 main 반영을 씁니다.
semantic conflict는?텍스트 충돌이 없어도 동작이 충돌할 수 있어 테스트와 리뷰가 필요합니다.

오답/주의점

주의

충돌 마커만 없애고 끝내면 위험합니다. 최종 동작과 테스트를 확인해야 합니다.

GIT-08

Q. 브랜치 전략은 어떻게 선택하나요?

핵심 개념

브랜치 전략은 팀 규모, 배포 빈도, 테스트 자동화 수준, 릴리즈 안정화 방식에 따라 달라집니다.

개념 해부

브랜치 전략은 조직의 배포 빈도와 안정성 요구를 반영해야 합니다. trunk-based는 작은 변경과 빠른 CI가 전제이고, GitFlow는 release branch와 장기 지원이 필요한 환경에 맞을 수 있습니다. 전략보다 중요한 것은 변경이 작고 언제든 되돌릴 수 있는 상태를 유지하는 것입니다.

30초 답변

작고 자주 배포하는 팀은 trunk-based development가 유리하고, 릴리즈 안정화 기간이 필요한 조직은 release branch를 둘 수 있습니다. Git Flow는 명확하지만 브랜치가 많아 느려질 수 있습니다. 어떤 전략이든 main 보호, CI, 리뷰, feature flag, rollback 경로가 필요합니다.

90초 답변

브랜치 전략은 개발 속도와 안정성의 trade-off입니다. trunk-based는 작은 변경과 강한 자동화가 전제이고, feature branch 장기화는 충돌과 통합 리스크를 키웁니다. 릴리즈 브랜치를 쓴다면 hotfix가 main과 release에 어떻게 반영되는지 명확해야 합니다.

꼬리질문

질문답변 방향
feature branch가 오래되면?통합 비용과 충돌이 커지므로 작은 단위로 main에 숨겨진 상태로 합칩니다.
main이 깨지면?보호 규칙, required CI, 빠른 revert, incident 기록이 필요합니다.
hotfix 흐름은?운영 수정 후 main/release 양쪽에 반영하고 테스트와 release note를 남깁니다.

면접에서 말할 증거

팀의 배포 주기, CI 품질, release branch 운영 경험을 연결하면 현실적인 답변이 됩니다.

GIT-09

Q. release와 rollback을 Git 관점에서 어떻게 관리하나요?

핵심 개념

배포는 코드가 main에 merge되는 순간이 아니라 어떤 commit, artifact, config, migration이 운영에 반영됐는지 추적하는 과정입니다.

개념 해부

release 관리는 어떤 코드가 어떤 환경에 어떤 설정으로 나갔는지 추적하는 일입니다. Git tag, commit SHA, artifact digest, migration version이 연결되어야 rollback 판단이 가능합니다. rollback은 Git revert만이 아니라 DB, config, cache, feature flag 호환성까지 포함합니다.

30초 답변

release는 commit SHA, tag, artifact version, config version, migration version을 묶어 추적해야 합니다. rollback은 단순히 이전 commit으로 돌아가는 것이 아니라 DB migration, feature flag, cache, external dependency 호환성까지 확인해야 합니다.

90초 답변

운영에서는 “어떤 버전이 배포됐나”를 즉시 알 수 있어야 합니다. tag와 changelog, CI artifact, image digest를 남기고, 배포 후 health check와 metric을 확인합니다. DB 변경이 포함된 release는 expand-contract 방식으로 구버전과 신버전이 동시에 동작하도록 만들어야 rollback이 가능합니다.

꼬리질문

질문답변 방향
rollback 기준은?5xx, latency, 핵심 기능 오류, business metric, error budget 기준을 둡니다.
DB migration rollback은?파괴적 변경은 되돌리기 어려우므로 호환 배포와 feature flag가 중요합니다.
release note에는?변경 요약, 영향 범위, migration, 운영자 action, rollback 방법을 남깁니다.

오답/주의점

주의

“git revert하면 rollback입니다”로 끝내면 운영 현실을 놓칩니다. 데이터와 설정 호환성을 같이 봐야 합니다.

GIT-10

Q. ADR은 언제 쓰고 무엇을 남기나요?

핵심 개념

ADR은 Architecture Decision Record로, 나중에 되돌아볼 비용이 큰 기술 결정의 맥락과 trade-off를 짧게 남기는 문서입니다.

개념 해부

ADR은 모든 결정을 문서화하는 것이 아니라 나중에 다시 논쟁될 만한 중요한 선택의 맥락을 남기는 문서입니다. 선택지, 결정 이유, 포기한 대안, 예상 비용을 남겨야 미래 팀원이 같은 배경 없이도 판단을 이해할 수 있습니다. ADR은 코드와 함께 바뀌는 살아 있는 기록이어야 합니다.

30초 답변

ADR은 모든 토론을 문서화하는 것이 아니라 되돌리기 어렵거나 팀 전체에 영향을 주는 결정을 남길 때 씁니다. context, options, decision, consequences를 짧게 적고, 나중에 조건이 바뀌면 superseded 상태로 새 결정을 남깁니다.

90초 답변

예를 들어 상태관리 라이브러리 선택, 인증 방식, DB migration 전략, monorepo 전환 같은 결정은 시간이 지나면 왜 그렇게 했는지 잊힙니다. ADR에는 선택한 방법뿐 아니라 포기한 대안과 리스크를 남겨야 합니다. 그래야 새 팀원이 “왜 이렇게 복잡하지?”라고 느낄 때 맥락을 복구할 수 있습니다.

꼬리질문

질문답변 방향
ADR이 너무 많아지면?큰 결정만 남기고 template을 짧게 유지합니다.
결정이 틀렸다면?기존 ADR을 지우지 않고 새 ADR로 변경 배경과 결과를 남깁니다.
이슈와 ADR 차이는?이슈는 작업 추적, ADR은 장기 의사결정 맥락 보존입니다.

면접에서 말할 증거

직접 남긴 의사결정 문서나 PR 설명에서 대안과 trade-off를 적었던 경험을 말합니다.

GIT-11

Q. 기술부채는 어떻게 설명하고 관리하나요?

핵심 개념

기술부채는 마음에 들지 않는 코드가 아니라 미래 변경 비용과 장애 가능성을 증가시키는 선택의 누적입니다.

개념 해부

기술부채는 지저분한 코드라는 감정 표현이 아니라 현재 구조가 미래 변경 비용을 얼마나 증가시키는지에 대한 설명입니다. 좋은 부채 관리는 영향, 발생 원인, 방치 비용, 제거 계획, 검증 방법을 같이 말합니다. 모든 부채를 즉시 제거하는 것도 제품 속도 관점에서는 나쁜 선택일 수 있습니다.

30초 답변

기술부채는 영향 범위, 비용, 위험, 제거 조건으로 설명합니다. “나중에 고치겠습니다”가 아니라 어떤 변경을 어렵게 만드는지, 지금 고치지 않는 이유는 무엇인지, 언제 어떤 단계로 줄일지 계획을 세웁니다.

90초 답변

모든 부채를 즉시 갚는 것은 현실적이지 않습니다. 사용자 가치와 안정성을 기준으로 우선순위를 정하고, 기능 개발 중 함께 줄일 수 있는 부채와 별도 투자해야 하는 부채를 나눕니다. 큰 리팩터링은 feature flag, compatibility layer, characterization test, 단계적 migration으로 위험을 줄입니다.

꼬리질문

GIT-16

협업·변경관리 인과관계

협업 면접에서 좋은 답변은 성격이 좋다는 말이 아니라 변경 위험을 작게 만들고 검증 가능한 상태로 전달하는 과정을 설명하는 것입니다.

COLLABORATION CAUSAL MODELPR이 커지면 리뷰어가 전체 맥락을 머리에 올릴 수 없고, 그러면 중요한 회귀와 보안 경계가 누락됩니다. commit이 뒤섞이면 revert 범위가 커지고, 배포 단위가 커지면 장애 원인 식별 시간이 늘어납니다. rebase는 선형 히스토리를 만들어 추적을 쉽게 하지만 공유 branch에서 쓰면 다른 사람의 기준점을 바꿀 수 있습니다. merge는 맥락 보존이 좋지만 히스토리가 복잡해질 수 있습니다. 그래서 협업 답변은 도구 취향보다 변경의 blast radius와 복구 가능성을 기준으로 말해야 합니다.
상황원인결과좋은 답변

릴리즈 사고변경 단위와 feature flag, rollback 조건이 없다장애 중 원인 식별과 복구가 늦다canary, metric, feature flag, revert/forward-fix 판단 기준을 준비한다
