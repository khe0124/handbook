보안과 신뢰성

DevOps에서 보안과 신뢰성은 배포 마지막에 붙이는 검사가 아닙니다. 권한, 취약점, 백업, 복구, SLO를 전달 파이프라인과 운영 체계 안에 넣어야 합니다.

REL-01

최소 권한

대상원칙
사용자역할 기반 접근, MFA, 주기적 권한 리뷰
CI/CD tokenrepo, environment, registry 권한 최소화
Runtime service account서비스가 필요한 API만 접근
Break-glass비상 권한은 기록과 사후 리뷰 필수
REL-02

취약점 관리

탐지→ 분류→ 우선순위→ 수정→ 검증
기준설명
SeverityCVSS만 보지 않고 실제 노출 경로를 함께 봄
Reachability취약 코드가 런타임에서 도달 가능한지 확인
Exploitability인증 필요, 네트워크 노출, 입력 제어 여부
Fix impact업그레이드 호환성과 회귀 위험
REL-03

공급망 보안

위험대응
악성 패키지lockfile review, registry scope, install script 주의
이미지 변조digest pinning, image signing, trusted registry
빌드 환경 탈취ephemeral runner, 최소 권한 token
artifact 불명확SBOM, provenance, release attestation
실무 감각

취약점 scan은 시작점일 뿐입니다. 운영에 실제로 포함되는 dependency, image layer, runtime path까지 연결해야 우선순위를 제대로 정할 수 있습니다.

SLSA 관점

SLSA는 source, build, provenance, dependency를 추적해 공급망 신뢰 수준을 높이는 프레임워크입니다. 실무에서는 protected source, hermetic에 가까운 빌드, 서명된 provenance, 검증 가능한 artifact를 단계적으로 적용합니다.

REL-04

백업과 복구

항목확인 기준
Backup무엇을, 얼마나 자주, 어디에 저장하는가
Restore test복구가 실제로 되는지 주기적으로 검증
RPO얼마나 많은 데이터 손실을 허용하는가
RTO얼마나 빨리 복구해야 하는가
DR리전/계정/클러스터 장애 시 대체 경로
백업은 복구 전까지 미완성

백업 파일이 있다는 것과 복구가 가능하다는 것은 다릅니다. restore drill 없이 백업 정책을 신뢰하면 실제 장애 때 시간을 잃습니다.

REL-05

신뢰성 목표

SLIindicator
사용자 경험을 나타내는 측정값입니다. 예: 성공률, latency, freshness.
SLOobjective
SLI가 만족해야 하는 목표입니다. 예: 30일 동안 요청 성공률 99.9%.
SLAagreement
고객과 약속한 계약 수준입니다. 보상과 책임이 따를 수 있습니다.
Error budget변경 여유
허용 가능한 실패량입니다. 소진 속도에 따라 배포 속도를 조절합니다.
