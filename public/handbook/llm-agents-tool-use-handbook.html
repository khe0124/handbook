Agent·Tool Use·Workflow

Agent는 마법의 자율성이 아니라 목표, 상태, 도구, 권한, 검증 루프를 가진 workflow입니다. 안정적인 제품은 agent보다 먼저 좁고 검증 가능한 workflow를 설계합니다.

AGT-01

agent-workflow-rule

기본값은 workflow입니다. 모델이 다음 행동을 선택해야 할 만큼 불확실성이 높고, 실패를 검증할 수 있으며, 위험 행동을 승인 게이트로 막을 수 있을 때만 agent를 사용합니다.

autonomy-gradient설명추천
Single prompt한 번의 요약·분류·추출출력 계약이 명확한 작업
Chain정해진 순서의 여러 모델 호출분석 후 요약처럼 단계가 고정된 작업
Workflow코드가 상태와 분기를 제어프로덕션 기본값
Tool-using assistant모델이 도구 호출을 제안하고 앱이 검증검색, 조회, 초안 작성
Autonomous agent모델이 계획·실행·검증을 반복낮은 위험의 탐색·분석, 강한 중단 조건 필요
AGENT WORKFLOW BOUNDARY user goal → planner proposes steps → app validates allowed actions
→ executor calls tools → verifier checks evidence
→ human approval gate for write actions → audit log
AGT-02

PLANNER EXECUTOR · planner-output-contract와 role-contracts

Planner와 Executor를 분리하면 모델이 세운 계획과 실제 도구 실행을 다르게 통제할 수 있습니다. Verifier는 결과가 목표를 충족했는지 증거 기반으로 판단합니다.

AGT-05

memory-lifecycle

Agent memory는 모든 대화를 저장하는 것이 아니라 다음 행동에 필요한 상태를 구조화해 보존하는 것입니다. memory도 prompt injection과 privacy의 공격면이 됩니다.

DEPTH

Agent를 안전한 실행 시스템으로 만들기

MARKER: concept-depth — Workflow에서 Agent로 넘어가는 결정 기준

Agent는 자동화의 상위 단계가 아니라 불확실성을 모델에게 일부 위임하는 설계입니다. 그래서 agent를 쓰면 비용, 지연, 권한 검증, 감사 로그, 중단 조건이 모두 어려워집니다. 단계가 고정되어 있고 입력과 출력이 예측 가능하다면 deterministic workflow가 기본값입니다. agent는 도구 선택이 동적이고, 중간 결과에 따라 다음 행동이 달라지며, 모든 side effect를 검증하거나 승인할 수 있을 때만 정당화됩니다.

agent를 쓰지 말아야 하는 경우도 명확해야 합니다. 결제, 권한 변경, 삭제, 외부 발송처럼 되돌리기 어렵거나 법적 책임이 큰 작업은 모델의 자율 반복에 맡기면 안 됩니다. 이런 작업은 draft나 dry-run까지는 agent가 도울 수 있지만, 실제 write는 human approval gate와 서버 권한 검사를 통과해야 합니다. agent 설계의 목표는 “스스로 다 하게 만들기”가 아니라, 불확실한 탐색은 맡기되 위험한 실행은 통제하는 것입니다.

MARKER: failure-driven — 잘못된 Planner가 만드는 사고

Planner가 “고객에게 갱신 안내 메일을 보내라”는 목표를 받고 곧바로 `send_email`을 계획에 넣는다면 실패입니다. 좋은 planner는 먼저 문서 검색, 초안 작성, preview 생성, 승인 요청을 분리하고, 승인 전에는 외부 전송을 계획하지 않아야 합니다. Executor는 planner가 제안한 단계도 다시 검증해야 하며, Verifier는 “메일이 잘 작성됨”이 아니라 citation, preview, approval, idempotency, audit log가 모두 있는지를 확인해야 합니다.

MARKER: artifact-ready — Tool call execution pipeline

model_proposal -> JSON schema parse -> feature allowlist -> actor permission check
-> risk tier -> dry-run preview -> human approval -> idempotency key
-> tool execution -> audit log -> verifier evidence check

MARKER: artifact-ready — Good planner output

{
  "goal": "계약 갱신 안내 메일 초안 작성",
  "steps": [
    {"action": "search_documents", "risk": "read-only"},
    {"action": "draft_email", "risk": "draft"},
    {"action": "request_approval", "risk": "approval-required"}
  ],
  "forbidden_until_approved": ["send_email"],
  "success_evidence": ["citations", "preview", "approval_id"]
}

Verifier는 또 다른 요약 모델이 아닙니다. verifier는 결과가 목표를 만족한다는 증거를 확인하는 계층입니다. schema가 유효한지, tool side effect가 승인 범위 안인지, citation이 실제 근거를 가리키는지, 테스트나 dry-run 결과가 있는지, stop condition을 무시하지 않았는지 확인합니다. evidence가 없으면 “그럴듯한 완료 메시지”를 실패로 처리해야 agent loop가 안전해집니다.

MARKER: artifact-ready — Failure traces

approval_denied: {tool:"send_email", approval:"denied", final_state:"draft_only"}
idempotency_collision: {key:"renewal-42", action:"skip_duplicate", audit:"deduped"}
verifier_blocked: {reason:"missing citation for renewal clause", next:"search_documents"}

MARKER: failure-driven — Retry가 같은 위험 행동을 반복하는 실패

Agent 시스템에서 retry는 단순 재시도가 아닙니다. planner가 잘못된 tool을 골랐거나, 입력 근거가 부족하거나, approval token이 없는 상태라면 같은 호출을 반복해도 성공 확률은 올라가지 않고 위험만 커집니다. 특히 write tool, external notification, 결제·계약 변경처럼 부작용이 있는 작업은 retry가 중복 실행 사고로 이어질 수 있습니다. 따라서 retry policy는 실패 원인별로 달라야 합니다. rate limit은 backoff, schema error는 repair, missing approval은 human gate, unsafe plan은 abort로 분기해야 하며, 같은 plan_id가 같은 write action을 반복할 때 idempotency key와 audit trace가 반드시 남아야 합니다.
