언어와 런타임: 문법 뒤의 실행 계약

언어 선택은 문법 취향이 아니라 타입 경계, 컴파일/해석 파이프라인, 메모리 관리, 동시성 의미론, 직렬화 계약을 고르는 일입니다. 이 문서는 언어가 코드에 제공하는 보장과 런타임이 실제로 치르는 비용을 함께 봅니다.

AUDIENCE : 제품 엔지니어 · Java/JavaScript/Python/Go 비교 관점 · UPDATED 2026-06
LR-01

LANGUAGE RUNTIME SPINE

언어 런타임을 이해하면 "왜 타입 에러가 컴파일 때 잡히는가", "왜 JSON 경계에서 깨지는가", "왜 GC가 latency를 흔드는가", "왜 async 코드가 병렬이 아닌가"를 설명할 수 있습니다.

LANGUAGE RUNTIME SPINE
source text → lexer/parser → AST → type check
→ IR/bytecode/native/JIT → module loading → runtime memory
→ concurrency scheduler → serialization boundary → observability
영역핵심 질문실무 판단
Type system어떤 오류를 실행 전에 막는가API DTO, domain model, null 처리
Compiler/interpreter코드는 언제 어떤 형태로 바뀌는가빌드 시간, startup, warmup
Runtime memory객체 생명주기를 누가 관리하는가GC pause, ownership, leak
Concurrency실행 단위와 공유 메모리 규칙은 무엇인가race, lock, actor, async
Serialization프로세스 밖으로 나갈 때 어떤 의미가 사라지는가schema, version, compatibility
LR-02

TYPE SYSTEM BOUNDARY

타입 시스템은 값의 모양과 연산 가능성을 제한합니다. 강한 타입은 런타임 오류를 줄이고, 약한 경계는 빠른 실험을 돕지만 계약을 테스트와 검증으로 보강해야 합니다.

구분의미실무 영향
Static typing실행 전 타입 검사리팩터링, API 계약, IDE 지원이 강함
Dynamic typing실행 중 타입 확인빠른 작성, 테스트와 런타임 검증 중요
Nominal typing이름/선언이 같아야 호환도메인 모델 구분이 명확
Structural typing구조가 맞으면 호환프론트엔드 DTO와 duck typing에 유리
Null safetynull 가능성을 타입으로 표현NPE를 컴파일 단계로 이동
TYPE SYSTEM BOUNDARY
trusted inside: typed domain objects and invariants
untrusted outside: JSON, DB rows, env vars, CLI args
boundary rule: parse → validate → normalize → construct typed value
타입의 바깥

네트워크, 파일, DB, 브라우저 storage, 환경변수는 타입 시스템 밖입니다. TypeScript 타입이나 Java DTO가 있어도 런타임 입력은 거짓말할 수 있으므로 경계에서 schema validation을 해야 합니다.

LR-03

AST COMPILER PIPELINE

컴파일러와 인터프리터는 소스 텍스트를 의미 있는 구조로 바꿉니다. AST를 이해하면 lint, formatter, codemod, transpiler, query planner 같은 도구가 같은 계열 문제라는 것이 보입니다.

AST COMPILER PIPELINEcharacters → tokens → AST → symbol table → type check
→ IR optimization → bytecode/native code → runtime execution
단계하는 일실무 예
Lexing문자열을 token으로 자름keyword, identifier, literal
Parsing문법에 맞는 AST 생성syntax error, precedence
Semantic analysis이름, scope, 타입 의미 확인undefined symbol, access check
Lowering고수준 문법을 낮은 표현으로 변환async/await transform, JSX transform
Optimization의미를 유지하며 비용 줄임dead code elimination, inlining
Code generationbytecode/native/JS 등 실행 형태 생성JVM bytecode, V8 bytecode
도구 작성 감각

정규식으로 코드를 고치면 문자열에는 맞지만 문법에는 틀릴 수 있습니다. import 정리, API rename, deprecated 사용 탐지는 AST 기반 도구로 처리하는 것이 안전합니다.

LR-04

모듈과 패키지

모듈 시스템은 이름 충돌을 막고 의존성 그래프를 관리합니다. 패키지 매니저는 버전, lockfile, transitive dependency, 공급망 위험을 관리합니다.

개념역할실패 모드
Moduleexport/import 경계circular dependency, hidden side effect
Package배포 가능한 의존성 단위semver 오해, transitive breakage
Lockfile정확한 의존성 버전 고정로컬/CI 버전 불일치
Class path / module path런타임 로딩 경로같은 클래스 다른 버전 충돌
Tree shaking사용하지 않는 코드 제거side effect 선언 누락
  • 공개 API와 내부 모듈을 분리한다.
  • 패키지 업데이트는 lockfile diff와 changelog를 함께 본다.
  • 초기화 side effect가 import 시점에 실행되는지 확인한다.
  • circular dependency는 테스트보다 설계 신호로 다룬다.
LR-05

GC JIT MEMORY MODEL

런타임은 객체를 할당하고 회수하며, 자주 실행되는 코드를 더 빠른 형태로 바꿉니다. 이 자동화는 생산성을 주지만 latency, warmup, pause, visibility 비용을 만듭니다.

주제의미관측할 것
GC root도달 가능한 객체의 시작점누수 객체가 어디서 참조되는가
Generational GC짧게 사는 객체와 오래 사는 객체 분리young/old collection 비율
Stop-the-world회수 중 애플리케이션 일시 정지pause p95/p99
JIT warmuphot path를 런타임에 최적화배포 직후 latency, profile 안정화
Memory model스레드 간 읽기/쓰기 가시성 규칙volatile, atomic, happens-before
GC JIT MEMORY MODEL
allocation rate ↑ → young GC frequency ↑
retained references ↑ → old generation pressure ↑
JIT warmup not complete → early requests slower
missing happens-before → stale read or data race
메모리 모델

멀티스레드 코드에서 "한 스레드가 썼으니 다른 스레드가 볼 것"은 자동 보장이 아닙니다. lock, volatile, atomic, channel, actor 같은 동기화 장치가 happens-before 관계를 만들어야 합니다.

LR-06

동시성 의미론

언어마다 동시성을 표현하는 단위가 다릅니다. OS thread, green thread, coroutine, event loop, actor는 같은 문제를 다르게 모델링합니다.

모델특징주의
Thread + lock공유 메모리와 명시적 lockdeadlock, race, contention
Event loop단일 실행 루프와 non-blocking I/OCPU 작업이 전체 루프를 막음
Coroutine/async중단 가능한 함수structured concurrency, cancellation 전파
Actor상태를 메시지로 격리메일박스 증가, 메시지 순서
CSP/channel공유보다 통신channel close, backpressure
취소와 timeout

동시성 코드는 시작보다 취소가 어렵습니다. 부모 요청이 취소되면 하위 작업, DB 쿼리, 외부 API, 큐 작업이 어떤 순서로 멈추는지 정의해야 resource leak을 막을 수 있습니다.

LR-07

SERIALIZATION CONTRACT

객체가 프로세스 밖으로 나가는 순간 포인터, 메서드, 타입 정보, 시간대, 정밀도, null 의미가 일부 사라집니다. 직렬화는 단순 변환이 아니라 독립 시스템 사이의 계약입니다.

SERIALIZATION CONTRACTdomain object → DTO → schema validation → encoded bytes
→ transport/storage → decode → validate → compatible domain object
경계잃기 쉬운 의미방어선

Database row도메인 불변식, enum 진화constraint, migration, adapter
Message queue버전, 순서, 중복 여부schema version, idempotency key
Binary protocol호환성 규칙field numbering, backward compatibility tests
Cachestale data, type driftTTL, namespace version
