프론트엔드 개요

FE-01

실무 프론트엔드의 역할

프론트엔드의 성공 기준은 "화면이 보인다"가 아닙니다. 사용자가 의도한 일을 끝낼 수 있고, 실패했을 때 이유를 이해할 수 있으며, 팀이 기능을 계속 바꿔도 UI가 무너지지 않는 것입니다.

관점기본 구현제품 기능시스템 품질
성공 기준디자인과 비슷하게 구현상태·에러·권한·빈 화면까지 흐름을 완성제품과 팀이 유지 가능한 UI 시스템 설계
컴포넌트props를 받아 화면 출력역할·소유 상태·합성 경계를 명확히 함도메인 변화와 디자인 시스템 확장을 흡수
네트워크API 호출 후 데이터 표시로딩·재시도·낙관 업데이트·캐시 무효화 설계사용자 경험과 백엔드 계약을 함께 조율
품질로컬에서 동작 확인접근성·성능·테스트·관측가능성을 기본값으로 둠품질 기준을 자동화하고 팀의 반복 비용을 줄임
CORE STANCE

프론트엔드 실력은 CSS 트릭이나 라이브러리 개수가 아니라 상태와 사용자 흐름을 안정적으로 설계하는 능력에서 갈립니다. "클릭하면 모달이 열린다"가 아니라 "권한이 없고, 저장 중이고, 네트워크가 실패하고, 새로고침 후에도 사용자가 맥락을 잃지 않는가"까지 보는 사람이 실무형입니다.

FE-02

브라우저를 알아야 하는 이유

React나 Vue 이전에 브라우저가 있습니다. 이 장은 상세 원리 설명이 아니라, 프론트엔드 의사결정이 브라우저 제약과 어떻게 연결되는지 보여주는 입구입니다. 이벤트 루프, 렌더링 파이프라인, 네트워크 waterfall의 상세 진단은 브라우저 동작 원리 파트에서 다룹니다.

개념의미실무 영향
이벤트 루프콜스택, 태스크, 마이크로태스크, 렌더링 기회를 순서대로 처리긴 JS 작업은 입력 반응성을 막음. 무거운 계산은 쪼개거나 worker로 이동
DOM문서 객체 모델. 브라우저가 화면 구조를 표현하는 트리빈번한 레이아웃 읽기·쓰기 섞임은 reflow를 유발
CSSOMCSS 규칙 트리복잡한 selector와 전역 cascade는 예측 비용을 키움
렌더링 파이프라인style → layout → paint → compositetransform/opacity는 비교적 싸고, width/height/top 변경은 layout 비용이 큼

실무 성능 최적화는 대부분 "무엇이 비싼지"를 구분하는 일입니다. 렌더링을 줄인다고 무조건 빨라지는 것이 아니라, 사용자가 체감하는 병목이 JS 실행인지, 네트워크인지, layout인지, 이미지인지 먼저 측정해야 합니다.

MEASURE FIRST

렌더링 병목을 React memo로 해결하려 하기 전에 Chrome Performance, React Profiler, Network 탭을 먼저 봅니다. "느린 것 같다"는 근거가 아니고, "입력 후 main thread가 220ms 막힌다" 또는 "초기 JS가 900KB라 hydration 전까지 버튼이 죽어 있다"가 근거입니다.

FE-03

HTML·CSS UI 계약

HTML과 CSS는 화면을 꾸미는 재료가 아니라 브라우저, 보조기술, 테스트가 읽는 UI 계약입니다. 이 장은 기본 요소 선택과 스타일 경계만 다루고, accessible name, ARIA, 키보드 패턴은 웹접근성 파트에서 상세히 다룹니다.

선택좋은 기본값피해야 할 패턴
버튼<button type="button"><div onClick>에 role만 붙이기
링크이동은 <a href>페이지 이동을 button으로 처리
폼label, name, 오류 메시지 연결placeholder만 라벨처럼 사용
레이아웃semantic landmark, heading 순서 유지시각 크기 때문에 heading 레벨 건너뛰기

CSS는 "꾸미기"가 아니라 상태와 구조의 표현입니다. 전역 cascade, specificity 경쟁, 임의 spacing이 쌓이면 작은 변경도 위험해집니다. 토큰, 컴포넌트 경계, 반응형 제약을 일관되게 두면 디자인 품질과 유지보수성이 같이 올라갑니다.

UI CONTRACT

개요 문서에서는 기본 계약만 잡습니다. 버튼은 버튼으로, 이동은 링크로, 폼은 label과 name으로 시작합니다. 세부 접근성 판단은 별도 접근성 파트에서 한 번만 깊게 다룹니다.

FE-04

컴포넌트 설계와 합성

컴포넌트는 재사용 가능한 조각이기 전에 책임 경계입니다. 좋은 컴포넌트는 "무엇을 하는지, 무엇을 몰라도 되는지, 어디까지 바꿔도 소비자가 깨지지 않는지"가 명확합니다.

패턴언제 쓰나주의점
Presentational데이터를 받아 그리기만 할 때비즈니스 로직이 섞이면 테스트와 재사용성이 떨어짐
Container데이터 조회, mutation, 권한, 라우팅 맥락을 묶을 때너무 커지면 페이지 전체가 한 컴포넌트가 됨
CompoundTabs, Dialog, Menu처럼 내부 부품이 협력할 때암묵적 규칙이 많아지면 사용법이 어려움
Headless동작과 UI 스타일을 분리해야 할 때접근성 책임이 호출자에게 넘어가지 않게 주의
COMPOSITION OVER CONFIGURATION 좋은 방향
<Card>
  <CardHeader>프로젝트</CardHeader>
  <CardBody><ProjectList /></CardBody>
</Card>

위험 신호
<Card title="프로젝트" body={list} padded bordered compact interactive />

props가 늘어나는 것은 컴포넌트가 너무 많은 결정을 떠안고 있다는 신호일 수 있습니다. variant가 제품의 실제 의미를 표현하는지, 아니면 호출자가 내부 구현을 조종하기 위한 스위치인지 구분해야 합니다.

BOUNDARY CHECK

컴포넌트를 나눌 때 "파일 수를 줄이는가"가 아니라 "변경 이유를 분리하는가"를 봅니다. 디자인만 바뀌는 컴포넌트와 데이터 계약이 바뀌는 컴포넌트는 다른 속도로 변합니다. 같은 파일에 묶이면 작은 디자인 변경이 데이터 로직 회귀로 이어집니다.

FE-05

상태 관리와 서버 상태

프론트엔드 상태 문제의 절반은 "상태의 종류"를 구분하지 않아서 생깁니다. 모든 것을 전역 store에 넣는 것은 단순화가 아니라 책임 회피일 때가 많습니다.

상태 종류예좋은 위치
로컬 UI 상태모달 열림, 탭 선택, 입력 중 값useState, 컴포넌트 내부
공유 UI 상태사이드바 접힘, 선택된 조직가까운 공통 부모 또는 context
URL 상태검색어, 필터, 페이지 번호query string, route params
서버 상태사용자 목록, 상세 데이터, 통계React Query, SWR 같은 서버 상태 도구
클라이언트 도메인 상태편집 캔버스, 복잡한 builder 상태Zustand, reducer, state machine

서버 상태는 클라이언트가 소유하지 않습니다. 캐시된 복사본일 뿐입니다. 그래서 stale time, refetch, optimistic update, invalidation 같은 개념이 필요합니다. "전역 store에 API 결과 저장"은 캐시 무효화, 중복 요청, stale data 문제를 직접 떠안겠다는 뜻입니다.

상태 설계 실패 모드

증상대개 원인리뷰 질문개선 방향
뒤로 가기 후 필터가 사라짐URL 상태를 컴포넌트 내부에만 둠공유·복원되어야 하는 상태인가?검색어, 정렬, 페이지는 query string으로 승격
저장 직후 이전 값이 다시 보임서버 상태 무효화 누락mutation 성공 후 어떤 query가 낡아지는가?관련 query invalidate 또는 응답으로 cache 갱신
모달을 다시 열면 입력값이 남음draft 상태 생명주기 불명확닫기, 취소, 제출 실패 때 draft를 보존할 것인가?컴포넌트 key, explicit reset, 저장 실패 보존 정책 분리
두 탭에서 수정하면 값이 덮임버전·충돌 처리 기준 없음충돌을 자동 덮어쓸 것인가, 사용자에게 보여줄 것인가?version, updatedAt, 409 처리 계약 추가
RULE OF THUMB

새 상태를 추가할 때 먼저 묻습니다. 새로고침 후에도 유지되어야 하는가? URL로 공유되어야 하는가? 서버가 진실의 원천인가? 두 컴포넌트 이상이 동시에 써야 하는가? 이 질문에 답하면 store부터 만들 필요가 줄어듭니다.

FE-06

API 연동과 에러·로딩 UX

API 연동은 fetch 한 줄이 아니라 사용자 흐름 설계입니다. 느림, 실패, 권한 없음, 부분 데이터, 재시도, 중복 제출을 모두 다뤄야 합니다.

상태UI 책임실무 기준
loading기다릴 이유와 범위를 보여줌전체 spinner보다 skeleton, 버튼 단위 pending 선호
empty데이터가 없는 상태를 정상 상태로 설명다음 행동이 있으면 제공, 없으면 조용히 안내
error원인과 회복 방법을 제시재시도 가능/불가능, 권한 문제, 입력 문제를 구분
success변경 결과를 사용자가 확인 가능하게 함낙관 업데이트 시 실패 rollback 설계

백엔드 API는 완벽히 안정적이지 않습니다. 프론트엔드는 실패를 감추는 층이 아니라 실패를 사용자가 이해 가능한 행동으로 바꾸는 층입니다. 같은 500이라도 저장 실패, 조회 실패, 결제 실패의 UI는 다릅니다.

에러별 UI 책임

상태사용자 의미UI 처리재시도
400요청 형식 오류일반 toast보다 계약 오류로 기록하고 원인 추적자동 재시도 금지
401인증 만료세션 갱신 또는 로그인 흐름으로 이동토큰 갱신 1회

DOUBLE SUBMIT

저장 버튼을 누른 뒤 pending 상태를 잠그지 않으면 같은 mutation이 두 번 나갈 수 있습니다. 프론트에서 버튼을 비활성화하고, 백엔드에서는 멱등성 키나 중복 제약을 둡니다. UI 방어와 서버 방어는 서로 대체가 아니라 겹겹이 쌓는 안전장치입니다.

FE-07

폼·검증·사용자 입력

폼은 프론트엔드에서 가장 복잡한 상태 기계 중 하나입니다. 입력값, touched, dirty, validation, async validation, submit 상태가 함께 움직입니다.

검증 위치역할예
브라우저기본 입력 제약type="email", required, min
클라이언트즉각 피드백과 UX형식, 길이, 필수값, UI 전용 규칙
서버최종 진실과 보안권한, 중복, 비즈니스 규칙, 데이터 무결성

클라이언트 검증은 사용자 경험을 위한 것이고, 서버 검증은 시스템 무결성을 위한 것입니다. 클라이언트에서 막았다고 서버가 믿으면 안 됩니다. 반대로 서버만 믿고 클라이언트 피드백을 없애면 사용자는 실패 이유를 너무 늦게 알게 됩니다.

FORM STATE MODEL value 입력값
touched 사용자가 지나간 필드인가

validity 현재 규칙을 통과하는가

serverError 서버가 거절한 이유
FE-08

성능: 번들·렌더링·메모이제이션

프론트엔드 성능은 초기 로드, 상호작용 반응성, 화면 안정성으로 나뉩니다. "빠르다"는 하나의 숫자가 아니라 사용자 여정의 여러 지점입니다.

영역주요 병목대응
초기 로드JS 번들, CSS, 폰트, 이미지code splitting, lazy loading, preload 기준 정리
렌더링불필요한 re-render, 큰 리스트, layout thrash컴포넌트 경계, virtualization, 측정 후 memo
상호작용긴 main thread 작업작업 분할, worker, transition, debounce
화면 안정성이미지·폰트·동적 콘텐츠 크기 변화크기 예약, skeleton, font-display 전략

memo, useMemo, useCallback은 성능 마법이 아닙니다. 비교 비용과 복잡도를 추가합니다. 문제가 "같은 props인데 자주 다시 그려진다"일 때 유효하고, 네트워크나 큰 번들이 병목일 때는 효과가 없습니다. React.memo는 props 얕은 비교에만 관여하므로 자신의 state나 context 변경에는 다시 렌더링되고, 객체·배열·함수 props의 identity가 매번 바뀌면 효과가 줄어듭니다.

useTransition은 상태 업데이트를 non-urgent로 표시해 입력 같은 urgent update가 먼저 반영될 수 있게 하는 도구입니다. 무거운 계산을 worker처럼 다른 thread로 옮기는 기능은 아니므로 작업 분할, virtualization, worker 이동과 함께 판단합니다.

PERFORMANCE SMELL

초기 번들에 관리자 화면, 차트 라이브러리, markdown parser, 에디터가 모두 들어 있으면 첫 화면이 느려집니다. 사용자가 지금 당장 쓰지 않는 기능은 route 단위나 기능 단위로 늦게 불러옵니다. 단, code splitting 후 route 진입 시 JS/CSS chunk와 데이터 요청 waterfall이 생길 수 있으므로 treemap과 Network waterfall로 실제 효과를 확인합니다.

FE-09

디자인 시스템과 UI 품질

디자인 시스템은 버튼 모음이 아니라 제품 UI의 의사결정 체계입니다. 색, 간격, 타이포그래피, 상태, 접근성, 컴포넌트 API가 함께 맞아야 합니다.

층내용실패 신호
토큰color, spacing, radius, typography컴포넌트마다 임의 hex와 px 사용
기초 컴포넌트Button, Input, Modal, Tabs, Menu비슷한 버튼이 여러 구현으로 분기
패턴폼 레이아웃, 빈 상태, 테이블 액션, 필터화면마다 다른 UX 규칙
품질 기준접근성, 반응형, 로딩·에러 상태디자인 QA가 매번 같은 문제를 지적

UI 품질은 디테일의 합입니다. 텍스트가 버튼 밖으로 밀리지 않는지, focus ring이 보이는지, hover만으로 기능을 숨기지 않는지, 작은 화면에서 정보 우선순위가 유지되는지를 봅니다.

DESIGN SYSTEM API

디자인 시스템 컴포넌트의 props는 내부 CSS를 조종하는 스위치가 아니라 제품 의미를 표현해야 합니다. variant="danger"는 의미가 있지만 red shadow rounded large 조합은 호출자가 시스템 내부를 조립하게 만듭니다.

FE-10

테스트 전략

프론트엔드 테스트는 DOM 구조를 얼리는 것이 아니라 사용자 행동을 검증해야 합니다. 좋은 테스트는 refactor를 방해하지 않으면서 회귀를 잡습니다.

테스트검증 대상좋은 예
단위순수 함수, formatter, reducer가격 포맷, 권한 계산, validation schema
컴포넌트사용자 상호작용과 DOM 결과필터 입력 후 목록이 줄어듦, 저장 실패 메시지 표시
E2E핵심 사용자 여정로그인, 생성, 결제, 권한별 접근
시각 회귀디자인 깨짐공통 컴포넌트와 핵심 화면 screenshot diff

테스트는 구현 세부가 아니라 계약을 잡아야 합니다. "이 hook이 두 번 호출된다"보다 "사용자가 저장하면 성공 메시지와 새 데이터가 보인다"가 더 오래 갑니다.

TEST DECISION PACKET 1. 비즈니스 규칙은 단위 테스트로 빠르게 고정한다.
2. 사용자 플로우는 통합 테스트에서 네트워크 성공·실패를 모두 검증한다.
3. 결제, 권한, 데이터 손실 가능성이 있는 흐름은 E2E로 행복 경로와 실패 경로를 둔다.
4. 접근성 회귀는 키보드 이동, accessible name, 오류 연결을 PR 체크리스트에 포함한다.
TESTING SMELL

snapshot이 수백 줄이고 실패할 때 아무도 읽지 않는다면 보호막이 아닙니다. 테스트 실패가 의사결정을 도와야 합니다. 무엇이 깨졌고 사용자가 어떤 영향을 받는지 드러나는 테스트가 좋은 테스트입니다.

FE-11

배포와 관측가능성

프론트엔드는 배포 후에도 살아 있는 시스템입니다. 빌드 산출물, CDN 캐시, 환경변수, 오류 수집, 성능 지표가 운영 품질을 결정합니다.

영역확인할 것위험
빌드환경변수 주입, sourcemap 정책, chunk 크기secret이 번들에 포함됨, sourcemap 공개
캐시assets는 길게, index.html은 짧게새 배포 후 사용자가 옛 chunk를 요청
오류 수집JS error, unhandled rejection, route, user context사용자만 깨진 화면을 보고 팀은 모름
성능 지표Web Vitals, API latency, long task로컬은 빠른데 실제 사용자 환경은 느림
FRONTEND OBSERVABILITY

운영 프론트엔드는 "에러가 발생했다"만으로 부족합니다. 어느 route, 어떤 브라우저, 어떤 release, 어떤 API 응답 후에 깨졌는지 연결해야 합니다. release id를 번들에 넣고 오류 수집과 배포 기록을 연결하면 rollback 판단이 빨라집니다.

FE-12

보안: XSS·CSRF·토큰 저장

프론트엔드 보안의 시작은 브라우저가 적대적 입력을 어떻게 실행 가능한 코드로 바꾸는지 이해하는 것입니다. DOM에 넣는 순간, URL로 이동하는 순간, storage에 저장하는 순간 위험이 달라집니다.

위험원인대응
XSS사용자 입력을 HTML/JS로 실행escape 기본값 유지, sanitize, CSP, dangerouslySetInnerHTML 최소화
CSRF쿠키가 cross-site 요청에 자동 첨부SameSite, CSRF token, 중요 mutation 재인증
토큰 탈취localStorage token을 XSS가 읽음짧은 access token, HttpOnly cookie 검토, refresh rotation
open redirect검증 안 된 redirect URL 사용상대 경로만 허용, allowlist 검증

React는 기본적으로 문자열을 escape하지만, 그 안전장치를 직접 끄는 순간 책임이 개발자에게 넘어옵니다. 외부 HTML, markdown, CMS 콘텐츠를 렌더링할 때는 sanitize가 필수입니다.

TOKEN STORAGE TRADEOFF

localStorage는 다루기 쉽지만 XSS에 취약합니다. HttpOnly 쿠키는 JS가 읽을 수 없지만 CSRF 방어가 필요합니다. 정답은 서비스의 위협 모델에 따라 다릅니다. 중요한 것은 "어디가 위험한지"를 알고 access token 수명, refresh token 회전, SameSite, CSP를 함께 설계하는 것입니다.

FE-13

대규모 제품 운영

프론트엔드가 커질수록 어려운 문제는 컴포넌트 작성이 아니라 제품 전체의 변경 비용을 통제하는 일입니다. 성능, 접근성, 디자인 시스템, API 계약, 배포 전략이 하나의 운영 체계로 묶여야 합니다.

영역고급 판단 기준실패 신호
모듈 경계도메인, route, 권한, 데이터 소유권 기준으로 분리공통 폴더가 모든 것을 알고 순환 의존이 생김
디자인 시스템토큰, 컴포넌트 API, 접근성, release note를 함께 관리Figma와 코드가 어긋나고 화면마다 예외 스타일 증가
성능 예산route별 JS/CSS/image budget과 Web Vitals 목표를 둠차트, 에디터, 관리자 기능이 초기 번들에 섞임
실사용자 관측RUM으로 route, browser, release, API latency를 연결Lighthouse는 통과하지만 실제 모바일 사용자는 느림
API 계약breaking change, fallback, partial data, error code를 문서화백엔드 문자열 메시지에 프론트 분기가 의존
실험과 분석이벤트 이름, 속성, 노출 기준, 개인정보 범위를 표준화AB 테스트 결과를 신뢰할 수 없거나 재현 불가
Case study

관리자 테이블이 느릴 때 먼저 React memo를 붙이는 것은 순서가 아닙니다. Network waterfall로 API 지연과 payload 크기를 보고, Performance 탭으로 main thread long task를 확인하고, React Profiler로 실제 re-render 원인을 봅니다. 그 다음 서버 페이지네이션, column virtualization, row memoization, 필터 debounce, URL state 정리를 선택합니다.

Micro-frontend 판단

마이크로 프론트엔드는 기술 유행이 아니라 조직 경계가 강하고 배포 독립성이 실제 병목일 때 검토합니다. 장점은 독립 배포와 팀 자율성이지만, 단점은 shared dependency, 디자인 일관성, routing, 인증, observability, 성능 예산 관리가 어려워지는 것입니다. 단일 팀·단일 제품에서는 모듈 경계와 monorepo discipline만으로 충분한 경우가 많습니다.

브라우저 동작 원리

BR-01

브라우저는 앱 런타임이다

브라우저는 HTML 뷰어가 아니라 네트워크, 파서, JS 엔진, 렌더링 엔진, 저장소, 보안 모델을 가진 애플리케이션 런타임입니다.

영역하는 일프론트엔드 판단
파싱HTML/CSS/JS를 읽어 구조화blocking script와 critical CSS 관리
JS 엔진코드 실행, microtask 처리긴 작업을 쪼개 input delay를 줄임
렌더링 엔진layout, paint, composite비싼 스타일 변경과 layout thrash 회피
네트워크요청 우선순위, cache, connection 관리preload, lazy loading, cache header 이해
핵심 관점

브라우저는 한 번에 모든 일을 하지 않습니다. JS가 main thread를 오래 점유하면 클릭, 스크롤, 렌더링이 밀립니다. UI 성능은 "코드가 빠른가"보다 브라우저에게 렌더링할 기회를 주는가에 가깝습니다.

BR-02

화면에 보이는 것과 DOM이 말하는 의미가 다르면 UI는 거짓말을 합니다. 접근성의 출발점은 시각 표현이 아니라 의미 구조입니다.

품질나쁜 기준좋은 기준

PRODUCT QUALITY

상황구현 기준놓치기 쉬운 점
필수 입력label 또는 설명에 필수 여부를 텍스트로 제공별표만 표시하면 의미가 전달되지 않음
필드 오류aria-invalid와 aria-describedby로 오류 문구 연결오류 문구 id가 매 렌더마다 바뀌면 연결이 깨짐
서버 오류필드 오류와 전역 오류를 분리모든 오류를 상단 toast로만 보여주면 수정 위치를 잃음
제출 중버튼 pending 상태와 중복 제출 방지disabled만 두면 이유를 모를 수 있어 상태 텍스트도 고려
제출 실패첫 오류 필드 또는 오류 요약으로 focus 이동화면 위쪽 오류를 보지 못한 채 같은 입력을 반복
ERROR UX

오류는 "틀렸습니다"가 아니라 "무엇을 어떻게 고쳐야 하는지"를 알려야 합니다. 서버 오류와 클라이언트 검증 오류도 구분합니다.

A11Y-06

컴포넌트별 접근성 패턴

접근성 품질은 페이지 단위보다 컴포넌트 단위에서 반복적으로 무너집니다. 공통 컴포넌트는 의미, 키보드, focus, 상태 전달을 API 계약으로 가져야 합니다.

컴포넌트필수 계약회귀 신호
Dialog이름 제공, 배경 inert, focus trap, Esc 닫기, 닫힌 뒤 트리거로 focus 복귀Tab이 배경으로 빠지거나 닫힌 뒤 focus가 body로 감
Dropdown/Menu열림 상태 전달, Arrow 이동, Enter/Space 선택, 외부 클릭·Esc 닫기마우스로만 선택 가능하거나 menu role을 내비게이션에 남용
Tabstablist/tab/tabpanel 관계, 선택 상태, Arrow 이동, panel 이름 연결선택된 탭과 패널이 보조기술에서 연결되지 않음
Combobox입력값, 목록 열림, active option, 선택 결과를 명확히 전달검색 결과 개수와 현재 항목을 알 수 없음
Toast작업을 가로막지 않고 중요한 상태는 live region으로 전달오류가 사라져 사용자가 복구 방법을 잃음
Data grid표 의미, 헤더 연결, 정렬 상태, 행 선택 상태, keyboard strategy 정의시각적으로 표지만 DOM은 div 목록이고 셀 맥락을 잃음
Tooltip보조 설명으로만 사용, hover와 focus 모두 지원필수 정보를 tooltip 안에만 숨김
HEADLESS COMPONENT RISK

headless 컴포넌트는 스타일 자유도가 높지만 접근성 책임이 사라지는 것은 아닙니다. 호출자가 label, description, focus target, disabled reason을 빠뜨릴 수 없게 API를 설계해야 합니다.

A11Y-07

검증 루틴

접근성은 자동 검사만으로 충분하지 않습니다. 패턴을 설계한 뒤 자동화와 수동 키보드 검사를 함께 둡니다.

도구잡는 것못 잡는 것
Lighthouse기본 label, contrast, landmark 일부실제 키보드 흐름의 자연스러움
axe규칙 기반 접근성 위반제품 맥락의 설명 품질
키보드 테스트Tab 순서, focus trap, hidden focus스크린 리더 발화 품질 일부
스크린 리더실제 보조기술 경험모든 브라우저 조합 자동 커버
수동 검증기준
Keyboard only핵심 플로우를 Tab/Shift+Tab/Enter/Space/Esc/Arrow로 완료
Screen readerSafari + VoiceOver, Chrome/Firefox + NVDA 조합 중 최소 주요 플로우 확인
Mobile assistive techiOS VoiceOver 또는 Android TalkBack으로 주요 화면 탐색
Display settingszoom 200%, reduced motion, 고대비 모드, text spacing에서 레이아웃 유지
Touch target작은 화면에서 주요 터치 대상 크기와 간격 확인
PR CHECK

새 상호작용 컴포넌트는 keyboard-only 플로우, focus visible, label/name, 오류 메시지 연결, 모바일 화면 겹침 여부를 PR 체크리스트에 포함합니다. 자동 axe/Lighthouse 통과만으로 완료 처리하지 않습니다.

Accessible Name이름
보조기술이 요소를 부를 때 사용하는 이름입니다. 버튼 텍스트나 aria-label에서 옵니다.
Live Region상태 알림
동적으로 바뀐 메시지를 스크린 리더에게 알리는 영역입니다.
A11Y-08

접근성 회귀 방지

접근성은 한 번 감사해서 끝나는 품질이 아닙니다. 공통 컴포넌트와 핵심 플로우에 자동 검사, 수동 검사, 리뷰 질문을 반복 가능하게 넣어야 합니다.

단계확인 항목증거
설계상호작용 컴포넌트의 role, keyboard, focus restore 정의컴포넌트 명세 또는 Storybook notes
구현label, accessible name, describedby, live region 연결DOM 캡처, 테스트 쿼리 이름
테스트keyboard-only 플로우, 오류 상태, zoom 200%, reduced motion 확인체크리스트와 실패 재현 링크
릴리즈새 공통 컴포넌트가 기존 화면의 focus 순서를 깨지 않는지 확인시각 회귀와 접근성 회귀 결과
ACCESSIBILITY PR QUESTIONS1. 이 화면의 주요 작업은 마우스 없이 끝나는가?
2. 보이는 label과 보조기술의 이름이 같은 의미인가?
3. 오류, 저장 성공, 로딩 상태가 시각 정보 외 경로로도 전달되는가?
4. 모달·메뉴·페이지 전환 후 focus 위치가 사용자의 맥락을 보존하는가?
5. 자동 검사 통과 외에 수동으로 확인한 핵심 플로우가 있는가?
