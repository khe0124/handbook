BD-01

Q. 좋은 REST API 설계 원칙은 무엇인가요?

핵심 개념

좋은 API는 예쁜 URL이 아니라 일관된 계약입니다. 리소스 중심 URL, HTTP 메서드 의미, 상태 코드, 에러 포맷, pagination, filtering, versioning, correlation ID까지 포함해야 운영 가능한 API가 됩니다.

개념 해부

REST는 URL 작명 규칙보다 client와 server가 리소스 상태를 어떻게 주고받을지 정하는 제약 집합입니다. 메서드 의미, 상태 코드, 캐시 가능성, 멱등성, 에러 표현이 맞아야 client가 실패와 재시도를 안전하게 처리할 수 있습니다. 좋은 API는 성공 응답뿐 아니라 빈 결과, 권한 거부, 검증 실패, 부분 실패까지 계약으로 다룹니다.

30초 답변

REST API는 리소스 중심 URL, 메서드 의미, 상태 코드, 일관된 에러 포맷이 기본입니다. 실무에서는 pagination, filtering, versioning, backward compatibility, correlation ID까지 같이 설계해야 깨졌을 때도 진단 가능한 계약이 됩니다.

90초 답변

API를 함수 호출처럼 보면 화면 요구에 맞춘 임시 endpoint가 늘어납니다. 저는 API를 계약으로 보고 request/response schema, nullable, enum, error code, 권한 조건, pagination 방식을 문서화해야 한다고 봅니다. 특히 프론트는 raw response에 직접 묶이기보다 adapter에서 view model로 바꿔 API 변경의 충격을 줄이는 편이 안정적입니다.

BD-02

Q. API 스펙이 자주 바뀔 때 어떻게 대응하나요?

핵심 개념

API 변경 대응은 endpoint 교체가 아니라 계약 변경 관리입니다. response 구조, status 값, error format, loading/empty/error UI, backward compatibility를 함께 봐야 합니다.

개념 해부

API 변경은 타입 오류보다 의미 오류가 더 위험합니다. 필드 이름이 같아도 nullable 의미, enum 값, 정렬 기준, 오류 코드가 바뀌면 화면의 판단 조건이 틀어질 수 있습니다. adapter layer는 raw response와 화면 모델 사이에 완충 지대를 만들어 변경 범위와 테스트 지점을 명확히 합니다.

30초 답변

먼저 변경되는 필드, nullable, enum, error response, pagination 방식을 명확히 질문합니다. 화면은 raw response에 직접 의존하지 않게 service/adapter layer에서 view model로 변환하고, contract test나 mock으로 변경 영향을 확인합니다.

90초 답변

스펙이 불완전하면 추측으로 컴포넌트를 먼저 묶지 않습니다. 질문 목록을 만들고 최소 계약을 합의한 뒤, 확정 전에는 mock으로 화면 흐름을 만들 수 있습니다. 실제 연결 시점에는 adapter에서 raw response를 화면 모델로 변환하고, error/empty/loading 상태까지 함께 테스트해야 합니다.

BD-03

Q. Validation은 프론트와 백엔드 중 어디에서 처리하나요?

핵심 개념

프론트 validation은 사용성을 위한 빠른 피드백이고, 백엔드 validation은 보안과 비즈니스 규칙의 최종 방어선입니다. DB 제약은 마지막 정합성 방어선입니다.

개념 해부

Q. 트랜잭션 격리 수준은 어떻게 판단하나요?

핵심 개념

개념 해부

격리 수준은 동시 트랜잭션이 서로를 얼마나 보게 할지 정하는 정책입니다. 높은 격리는 이상 현상을 줄이지만 lock 대기와 deadlock, 처리량 저하가 생길 수 있습니다. 도메인이 허용하지 않는 불일치를 먼저 정의하고, isolation, constraint, lock, retry 중 어떤 조합으로 막을지 선택해야 합니다.

30초 답변

Q. JPA N+1 문제는 왜 생기고 어떻게 해결하나요?

핵심 개념

N+1은 처음 조회한 N개 entity의 연관 데이터를 접근할 때 추가 쿼리가 N번 발생하는 문제입니다. ORM의 lazy loading과 조회 패턴을 이해해야 합니다.

개념 해부

ORM은 객체 그래프 탐색을 SQL로 바꿔주지만, 연관 객체 접근 시점마다 쿼리가 나가는 사실을 없애지는 않습니다. N+1은 목록의 각 row가 lazy association을 건드리며 추가 쿼리를 만드는 패턴입니다. 해결책은 fetch join 하나가 아니라 화면이 필요한 데이터 모양에 맞춰 projection, batch, entity graph를 고르는 것입니다.

30초 답변

N+1은 목록을 한 번 조회한 뒤 각 row의 연관 데이터를 접근하면서 추가 쿼리가 반복되는 문제입니다. 해결은 fetch join, entity graph, batch size, DTO projection 등을 조회 목적에 맞게 선택하고, 실제 SQL 로그와 실행 계획으로 확인합니다.

90초 답변

fetch join은 한 번에 연관 데이터를 가져와 쿼리 수를 줄이지만 pagination이나 중복 row 문제가 생길 수 있습니다. DTO projection은 화면에 필요한 컬럼만 가져와 효율적일 수 있지만 entity 변경 추적과는 성격이 다릅니다. 핵심은 ORM 설정을 외우는 것이 아니라 목록 화면, 상세 화면, 통계 화면의 조회 목적에 맞는 전략을 고르는 것입니다.

꼬리질문

질문답변 방향
fetch join이 항상 답인가요?아닙니다. 다대다/컬렉션 pagination에서 문제가 생길 수 있습니다.
LAZY와 EAGER 기준은?기본은 LAZY로 두고 use case별 fetch 전략을 query에서 명시하는 것이 안전합니다.
어떻게 발견하나요?SQL 로그, query count, APM, slow query, 테스트에서 쿼리 수 검증으로 발견합니다.

오답/주의점

주의

직접 운영 경험이 제한적이면 “학습/적용 범위”를 분리하고, SQL 로그와 조회 목적 기준으로 설명하세요.

BD-09

Q. JWT와 Session은 어떻게 비교하나요?

핵심 개념

JWT와 Session은 인증 상태를 어디에 두고 어떻게 검증, 만료, 철회할지에 대한 선택입니다.

개념 해부

JWT와 Session은 인증 상태를 client에 검증 가능한 token으로 둘지, 서버 저장소에 둘지의 trade-off입니다. JWT는 확장과 stateless 검증에 유리하지만 탈취 후 만료 전 제어가 어렵고, session은 중앙 revoke가 쉽지만 저장소와 확장 전략이 필요합니다. 보안 판단은 저장 위치, 만료, 갱신, 철회, CSRF/XSS 위협을 함께 봐야 합니다.

30초 답변

JWT는 서명된 token을 stateless하게 검증할 수 있어 확장에 유리하지만 탈취, 만료 전 철회, 권한 변경 반영이 어렵습니다. Session은 서버 저장소가 필요하지만 revoke와 중앙 제어가 쉽습니다.

90초 답변

JWT를 쓰더라도 access token은 짧게 두고 refresh token rotation, 재사용 탐지, 저장 위치를 고민해야 합니다. Cookie 기반이면 CSRF 방어가 필요하고, localStorage는 XSS에 취약합니다. Session은 sticky session, shared store, Redis 장애, TTL 정책을 봐야 합니다.

꼬리질문

질문답변 방향
JWT revoke는?짧은 만료, blocklist, token version, refresh rotation을 조합합니다.
cookie와 localStorage?cookie는 httpOnly/SameSite 가능하지만 CSRF를 봐야 하고, localStorage는 XSS 탈취가 쉽습니다.
권한 변경 반영은?token claim 재발급, 서버 조회, version 검증을 고려합니다.

면접에서 말할 증거

만료/갱신 sequence, 401 처리, refresh 실패 UX, token 탈취 가정의 대응 방식을 설명합니다.

오답/주의점

주의

JWT가 항상 더 현대적이거나 안전하다는 식으로 말하지 말고 제어권과 운영 복잡도를 비교해야 합니다.

BD-10

Q. RBAC와 ABAC는 어떻게 다르나요?

핵심 개념

RBAC는 역할 기반, ABAC는 사용자·리소스·환경 속성 기반 인가 모델입니다.

개념 해부

RBAC는 역할이 권한 묶음이 되는 모델이고, ABAC는 사용자·리소스·환경 속성으로 정책을 계산하는 모델입니다. B2B SaaS에서는 role만으로는 tenant, 소유자, 상태 조건을 표현하기 어려운 경우가 많습니다. 정책이 세밀해질수록 권한 matrix, 테스트, 감사 로그가 없으면 운영자가 정책을 신뢰하기 어렵습니다.

30초 답변

RBAC는 관리자, 매니저, 사용자처럼 역할별 권한을 관리해 단순하고 운영하기 쉽습니다. ABAC는 부서, 소유자, 리소스 상태, 시간 같은 조건을 반영할 수 있어 유연하지만 정책 복잡도와 테스트 비용이 커집니다.

90초 답변

B2B 서비스는 역할만으로 부족한 경우가 많습니다. 예를 들어 같은 매니저라도 자기 조직의 리소스만 수정할 수 있다면 role뿐 아니라 resource owner, tenant, status가 필요합니다. 정책이 복잡해질수록 audit log, 정책 테스트, 권한 matrix 문서가 중요해집니다.

꼬리질문

질문답변 방향
권한은 어디서 검증?서버가 최종 검증하고 프론트는 UX guard만 담당합니다.
테넌트 격리는?tenant id를 모든 접근 조건에 포함하고 테스트로 누락을 잡습니다.
정책 변경은?권한 matrix, migration, audit log로 추적합니다.

면접에서 말할 증거

권한 matrix, 403 테스트, audit event, tenant boundary 테스트를 말합니다.

오답/주의점

주의

역할 이름만 나열하면 인가 설계가 아니라 UI 메뉴 설계로 들립니다.

BD-11

Q. 정규화와 비정규화는 어떻게 판단하나요?

핵심 개념

정규화는 중복을 줄이고 정합성을 높이며, 비정규화는 조회 성능과 단순성을 위해 중복을 허용하는 선택입니다.

개념 해부

정규화는 중복을 줄여 원본을 하나로 만들고, 비정규화는 읽기 경로를 줄이기 위해 중복을 의도적으로 받아들입니다. 비정규화된 값은 원본이 아니므로 갱신 실패와 재계산 전략이 필요합니다. 성능 병목이 측정되기 전 비정규화를 먼저 하면 데이터 불일치와 마이그레이션 비용을 앞당길 수 있습니다.

30초 답변

기본은 도메인 정합성을 지키기 위해 정규화로 시작하고, 실제 조회 패턴과 성능 병목이 확인되면 projection, summary table, denormalized column을 검토합니다. 비정규화에는 동기화 실패 비용이 따라옵니다.

90초 답변

주문 금액, 통계, 검색 화면처럼 매번 join과 집계가 부담되는 경우 비정규화가 유리할 수 있습니다. 하지만 원본이 바뀔 때 파생 데이터 갱신이 실패하면 사용자에게 틀린 값을 보여줍니다. 그래서 이벤트 기반 갱신, batch 재계산, 검증 job 같은 보정 전략이 필요합니다.

꼬리질문

질문답변 방향
언제 비정규화?측정된 조회 병목, 명확한 읽기 패턴, 허용 가능한 정합성 지연이 있을 때입니다.
동기화 실패는?재계산 job, reconciliation, alert로 보정합니다.
view와 table?실시간성, 성능, 갱신 비용에 따라 view/materialized view/table을 고릅니다.

면접에서 말할 증거

조회 쿼리 빈도, 실행 계획, summary table 갱신 전략, reconciliation 로그를 제시합니다.

오답/주의점

주의

정규화가 항상 좋은 것도, 비정규화가 무조건 성능 최적화인 것도 아닙니다.

BD-12

Q. 낙관적 락과 비관적 락은 언제 쓰나요?

핵심 개념

낙관적 락은 충돌이 드물다고 보고 저장 시 version으로 감지하고, 비관적 락은 충돌을 막기 위해 먼저 잠급니다.

개념 해부

낙관적 락은 충돌을 사후 감지하고, 비관적 락은 충돌을 사전 차단합니다. 충돌이 드문 문서 편집은 version 기반 감지가 적합하고, 재고 차감처럼 동시에 통과하면 안 되는 경우는 row lock이나 조건부 update가 필요할 수 있습니다. 어떤 방식이든 사용자 충돌 UX와 retry 정책이 없으면 운영에서 실패합니다.

30초 답변

충돌이 드물고 재시도나 사용자 충돌 해결이 가능하면 낙관적 락이 적합합니다. 재고 차감처럼 동시에 수정되면 안 되고 충돌 가능성이 높으면 비관적 락이나 원자적 update를 검토합니다.

90초 답변

낙관적 락은 처리량에 유리하지만 충돌 시 UX와 retry 전략이 필요합니다. 비관적 락은 안전하지만 lock wait, deadlock, timeout 비용이 생깁니다. 어떤 락을 쓰든 트랜잭션 범위를 짧게 하고 lock 순서를 통일해야 합니다.

꼬리질문

질문답변 방향
version 충돌 UX는?최신 값 표시, 사용자 선택, 재시도 안내를 제공합니다.
재고 차감은?조건부 update, row lock, unique constraint를 조합합니다.
deadlock은?락 순서 통일, timeout, retry, 모니터링으로 대응합니다.

면접에서 말할 증거

동시 요청 테스트, lock wait 로그, version conflict 처리 화면, retry 정책을 말합니다.

오답/주의점

주의

락을 세게 걸면 안전하다는 답은 처리량과 장애 전파 비용을 놓친 것입니다.

BD-13

Q. idempotency key는 왜 필요한가요?

핵심 개념

네트워크 timeout이나 retry 때문에 같은 요청이 여러 번 도착해도 부작용이 한 번만 일어나게 하는 장치입니다.

개념 해부

idempotency key는 client 재시도와 네트워크 불확실성을 서버가 견디게 하는 계약입니다. 같은 key와 같은 payload는 같은 결과를 돌려주고, 다른 payload가 같은 key를 쓰면 오류로 보아야 합니다. key 기록과 실제 부작용은 같은 transaction 또는 원자적 경계에 있어야 중복 실행을 막을 수 있습니다.

30초 답변

결제, 주문 생성, 포인트 차감처럼 중복 실행되면 안 되는 요청은 client가 idempotency key를 보내고 서버가 key별 처리 결과를 저장합니다. 같은 key가 다시 오면 새로 처리하지 않고 기존 결과를 반환합니다.

90초 답변

중요한 점은 key 저장과 실제 비즈니스 변경이 원자적으로 연결되어야 한다는 것입니다. 처리 중 상태, 성공 결과, 실패 결과, key TTL, payload mismatch 검증을 설계해야 합니다. 단순히 프론트 버튼을 disable하는 것은 네트워크 retry나 탭 중복 요청을 막지 못합니다.

꼬리질문

질문답변 방향
payload가 다르면?같은 key에 다른 payload면 오류로 보고 재사용을 막습니다.
처리 중 재요청은?processing 상태를 반환하거나 잠시 후 재시도하게 합니다.
TTL은?업무 특성과 재시도 가능 시간에 맞춰 정합니다.

면접에서 말할 증거

unique key 제약, 처리 상태 테이블, timeout 재시도 테스트, 중복 결제 방지 케이스를 말합니다.

오답/주의점

주의

HTTP 메서드가 멱등하다는 의미와 비즈니스 부작용이 안전하다는 의미를 혼동하지 마세요.

BD-14

Q. API versioning과 backward compatibility는 어떻게 관리하나요?

핵심 개념

API 변경 관리는 여러 client가 다른 속도로 배포되는 현실에서 계약을 깨지 않도록 운영하는 일입니다.

개념 해부

API 하위 호환성은 모든 client가 동시에 배포되지 않는다는 현실에서 출발합니다. 필드 추가는 보통 안전하지만 필드 삭제, 의미 변경, enum 추가, error format 변경은 client를 깨뜨릴 수 있습니다. versioning보다 더 중요한 것은 deprecation 기간, 사용량 관측, contract test, 단계적 제거 절차입니다.

30초 답변

가능하면 additive change로 새 필드를 추가하고 기존 필드 의미를 깨지 않습니다. breaking change가 필요하면 versioning, deprecation 기간, migration guide, contract test로 client 영향을 관리합니다.

90초 답변

nullable 변경, enum 추가, status code 변경, pagination 방식 변경은 화면에 큰 영향을 줍니다. 서버는 구버전 client 지원 기간을 정하고, 프론트는 adapter layer와 contract test로 변경 충격을 줄입니다. 배포 순서도 서버 먼저 하위 호환 변경을 내고 client 전환 후 제거하는 식으로 나눕니다.

꼬리질문

질문답변 방향

enum 추가는 breaking?client가 exhaustive 처리하면 깨질 수 있어 주의합니다.
제거는 언제?사용량 관측, 공지, migration 완료 후 제거합니다.

면접에서 말할 증거

deprecation 공지, API usage metric, contract test, adapter mapping 표를 말합니다.

서버와 프론트를 동시에 배포하면 된다는 답은 모바일/외부 client를 고려하지 못한 답입니다.

BD-15

Q. DB 제약과 애플리케이션 검증은 어떻게 나누나요?

핵심 개념

애플리케이션 검증은 사용자에게 의미 있는 오류와 비즈니스 흐름을 제공하고, DB 제약은 동시성과 버그 상황에서도 마지막 정합성을 지킵니다.

개념 해부

애플리케이션 검증은 오류를 사람이 이해하고 고칠 수 있게 만들고, DB 제약은 프로그램 실수와 동시성 경쟁에서도 데이터 불변식을 지킵니다. 예를 들어 중복 이메일은 서버가 친절히 알려줄 수 있지만, 동시에 들어온 요청은 unique constraint가 최종 방어선입니다. DB 오류도 그대로 500으로 내보내지 말고 도메인 오류로 변환해야 합니다.

30초 답변

서버 validation은 권한, 상태 전이, 비즈니스 규칙을 판단하고 DB unique, foreign key, not null, check constraint는 최종 방어선으로 둡니다. 같은 규칙이 중복되어 보일 수 있지만 목적이 다릅니다.

90초 답변

Q. slow query는 어떤 순서로 대응하나요?

핵심 개념

slow query 대응은 인덱스 추가부터가 아니라 재현, 실행 계획, 데이터 분포, lock, 애플리케이션 호출 패턴을 함께 보는 일입니다.

개념 해부

slow query는 SQL 한 줄만의 문제가 아닐 수 있습니다. 실행 계획이 좋아도 호출 횟수가 N+1이면 느리고, 쿼리는 빨라도 connection pool 대기나 lock wait이 있으면 endpoint는 느립니다. 그래서 재현 조건, trace, 실행 계획, 데이터 분포, lock, 호출 패턴을 같은 타임라인에서 봐야 합니다.

JSJ-01

Q. JVM, heap/stack, GC를 설명해보세요.

핵심 개념

JVM은 Java bytecode를 실행하는 런타임입니다. heap은 객체가 저장되는 공유 메모리이고, stack은 thread마다 method call frame과 지역 변수를 저장합니다. GC는 heap에서 더 이상 참조되지 않는 객체를 회수합니다.

개념 해부

JVM 메모리를 heap과 stack으로 나누는 이유는 객체 생명주기와 호출 생명주기가 다르기 때문입니다. stack frame은 method 호출이 끝나면 사라지지만, heap 객체는 더 이상 도달 가능한 참조가 없을 때 GC 대상이 됩니다. GC 문제는 단순히 메모리 부족이 아니라 allocation rate, 오래 살아남는 객체, stop-the-world pause가 응답 지연으로 드러나는 문제입니다.

30초 답변

Java 객체는 주로 heap에 저장되고 각 thread는 자기 stack을 가집니다. GC는 reachable object를 기준으로 사용하지 않는 heap 객체를 회수합니다. 실무에서는 stop-the-world, allocation rate, memory leak, GC log를 성능 문제와 연결해 봐야 합니다.

90초 답변

GC는 메모리를 자동으로 회수하지만 비용이 0은 아닙니다. 객체 생성이 과도하거나 큰 컬렉션을 오래 참조하면 GC 압박이 커집니다. memory leak은 객체가 진짜 필요 없는데도 collection, static field, ThreadLocal, cache 등이 계속 참조해 GC가 회수하지 못하는 상태입니다.

JSJ-02

Q. equals/hashCode는 왜 같이 구현해야 하나요?

핵심 개념

`equals`는 논리적 동등성, `hashCode`는 hash 기반 collection에서 bucket을 찾는 값입니다. equals가 true인 객체는 같은 hashCode를 가져야 합니다.

개념 해부

hash 기반 collection은 먼저 hash로 후보 위치를 좁히고, 그 안에서 equals로 실제 동등성을 확인합니다. 따라서 equals가 true인 두 객체의 hashCode가 다르면 같은 bucket 후보에 들어가지 못해 조회가 실패할 수 있습니다. entity에서는 persist 전 id, proxy class, mutable field 때문에 동등성 기준을 더 보수적으로 잡아야 합니다.

30초 답변

HashMap, HashSet은 hashCode로 후보 bucket을 찾고 equals로 최종 비교합니다. equals만 바꾸고 hashCode를 바꾸지 않으면 같은 값으로 여기는 객체가 set에서 중복되거나 map 조회가 실패할 수 있습니다.

90초 답변

JPA entity에서는 더 조심해야 합니다. DB id가 persist 전에는 null일 수 있고, proxy class와 equals 비교 문제가 생길 수 있습니다. mutable field를 hashCode에 넣으면 collection 안에서 값이 바뀐 뒤 찾지 못할 수 있습니다.

JSJ-03

Q. Spring DI와 Bean lifecycle을 설명해보세요.

핵심 개념

DI는 객체가 필요한 의존성을 직접 만들지 않고 외부 컨테이너가 주입하는 방식입니다. Spring Bean은 생성, 의존성 주입, 초기화 callback, 사용, 소멸 lifecycle을 가집니다.

개념 해부

DI의 본질은 객체 생성 책임과 사용 책임을 분리해 변경 가능한 경계를 만드는 것입니다. Spring container는 bean definition을 보고 객체를 만들고 의존성을 연결한 뒤 초기화 callback과 proxy 후처리를 적용합니다. 생성자 주입은 필수 의존성을 객체 생성 시점에 드러내므로 null 상태와 테스트 어려움을 줄입니다.

30초 답변

Spring은 Bean을 생성하고 의존성을 주입해 객체 간 결합을 낮춥니다. 생성자 주입은 필수 의존성을 명확히 하고 테스트하기 쉽습니다. Bean lifecycle에는 생성, 주입, 초기화, 소멸 단계가 있고 scope에 따라 생명주기가 달라집니다.

JSJ-04

Q. AOP, Filter vs Interceptor는 어떻게 다르나요?

핵심 개념

AOP는 횡단 관심사를 proxy 기반으로 분리합니다. Filter는 Servlet container 레벨, Interceptor는 Spring MVC handler 전후, AOP는 method call 전후에 주로 개입합니다.

개념 해부

Filter, Interceptor, AOP는 모두 중간에 끼어드는 구조지만 위치가 다릅니다. Filter는 Servlet container 경계에서 HTTP 요청을 다루고, Interceptor는 Spring MVC handler 전후를 다루며, AOP는 Spring bean method 호출을 proxy로 감쌉니다. proxy 기반 AOP는 proxy를 통과한 호출에만 적용되므로 같은 객체 내부 호출은 advice가 빠질 수 있습니다.

30초 답변

Filter는 Spring MVC에 들어오기 전 HTTP 요청/응답을 처리하고, Interceptor는 controller handler 전후에 동작합니다. AOP는 service method 같은 Bean method 호출에 proxy로 개입해 logging, transaction, security 같은 횡단 관심사를 처리합니다.

90초 답변

인증 토큰 파싱처럼 모든 요청에 필요한 처리는 Filter가 자연스럽고, handler 정보가 필요한 권한/로깅은 Interceptor가 맞을 수 있습니다. transaction은 보통 AOP proxy로 처리됩니다. proxy 기반이므로 같은 객체 내부에서 자기 메서드를 호출하는 self-invocation은 AOP가 적용되지 않는 문제가 생깁니다.

꼬리질문

질문답변 방향
proxy 종류?interface 기반 JDK dynamic proxy와 class 기반 CGLIB proxy가 있습니다.
self-invocation?proxy를 거치지 않아 AOP advice가 적용되지 않습니다.
순서?Filter → DispatcherServlet → Interceptor → Controller 흐름으로 설명합니다.
JSJ-05

Q. @Transactional propagation, rollback, self-invocation 문제를 설명해보세요.

핵심 개념

`@Transactional`은 Spring AOP proxy를 통해 transaction 경계를 열고 commit/rollback을 관리합니다. propagation은 기존 transaction이 있을 때 새 method가 어떻게 참여할지 결정합니다.

개념 해부

transaction은 SQL 묶음이 아니라 일관된 변경 단위와 실패 시 되돌림 경계입니다. Spring의 @Transactional은 proxy가 method 호출 전후로 transaction을 열고 닫는 모델이라 호출 경로와 예외 타입이 결과에 영향을 줍니다. 외부 API 호출이나 긴 계산을 transaction 안에 두면 DB connection과 lock을 오래 잡아 장애 전파 지점이 됩니다.

30초 답변

@Transactional은 proxy를 거쳐 호출될 때 적용됩니다. 기본 propagation은 REQUIRED라 기존 transaction이 있으면 참여하고 없으면 새로 엽니다. 기본 rollback은 unchecked exception 중심이므로 checked exception rollback은 별도 설정이 필요합니다.

90초 답변

self-invocation은 같은 클래스 안에서 transactional method를 직접 호출해 proxy를 거치지 않는 문제입니다. 또 transaction 안에서 외부 API를 호출하면 connection과 lock을 오래 잡아 장애를 키울 수 있습니다. readOnly는 쓰기 방지 보안장치라기보다 flush/최적화 힌트로 봐야 합니다.

꼬리질문

질문답변 방향
REQUIRES_NEW?기존 transaction을 보류하고 새 transaction을 엽니다.
rollbackOnly?내부에서 rollback-only가 표시되면 외부 commit 시 예외가 날 수 있습니다.
긴 transaction?lock, connection pool 고갈, 장애 전파 위험이 있습니다.
JSJ-06

Q. JPA persistence context, dirty checking, flush를 설명해보세요.

핵심 개념

persistence context는 entity를 관리하는 1차 cache이자 변경 감지 단위입니다. dirty checking은 관리 중인 entity의 변경을 transaction commit/flush 시점에 SQL로 반영하는 기능입니다.

개념 해부

persistence context는 entity identity map, 1차 cache, 변경 감지 단위가 합쳐진 작업 공간입니다. 같은 transaction 안에서 같은 id를 다시 조회하면 동일 객체를 돌려주고, commit/flush 시점에 snapshot과 현재 값을 비교해 SQL을 만듭니다. flush는 DB에 SQL을 보내는 동기화이고 commit은 transaction 확정이므로 둘을 구분해야 합니다.

30초 답변

JPA는 transaction 안에서 조회한 entity를 persistence context에 올리고 동일 id 조회는 같은 객체를 반환합니다. entity field가 바뀌면 dirty checking으로 변경을 감지하고 flush 시 SQL을 DB에 보냅니다.

90초 답변

flush는 DB에 SQL을 보내는 것이고 commit은 transaction을 확정하는 것입니다. flush가 됐다고 commit된 것은 아닙니다. persistence context가 너무 커지면 memory와 dirty checking 비용이 커질 수 있어 batch 처리에서는 flush/clear 전략이 필요합니다.

꼬리질문

질문답변 방향
detached entity?persistence context가 더 이상 관리하지 않는 entity입니다.
merge 주의?detached state를 복사해 managed entity를 반환하므로 반환값 사용에 주의합니다.
N+1과 관계?연관 loading 전략과 persistence context 동작을 이해해야 합니다.
JSJ-07

Q. lazy loading, Entity vs DTO, OSIV를 설명해보세요.

핵심 개념

lazy loading은 연관 객체 접근 시점에 query를 실행합니다. DTO는 API 응답이나 화면 요구에 맞춘 데이터 구조이고 Entity는 도메인/영속성 모델입니다. OSIV는 view rendering까지 persistence context를 열어두는 설정입니다.

개념 해부

lazy loading은 연관 데이터를 늦게 가져오는 최적화이지만 접근 시점에 SQL이 나간다는 사실을 숨깁니다. OSIV를 켜면 view/controller에서도 lazy loading이 가능해 편하지만 query가 service 경계 밖에서 발생하고 connection 보유 시간이 길어질 수 있습니다. Entity는 저장 모델이고 DTO는 API 계약이므로 응답 요구에 맞는 projection을 분리하는 편이 안전합니다.

30초 답변

lazy loading은 필요할 때 연관 데이터를 가져와 초기 조회를 줄이지만, transaction 밖에서 접근하면 LazyInitializationException이 나거나 N+1이 생길 수 있습니다. API 응답은 Entity를 그대로 노출하기보다 DTO로 필요한 필드만 명시하는 편이 안전합니다.

90초 답변

OSIV를 켜면 controller/view에서 lazy loading이 가능해 편하지만, DB connection이 오래 유지되고 query가 service boundary 밖에서 발생해 성능 추적이 어려워질 수 있습니다. 실무에서는 조회 use case에 맞게 fetch join, entity graph, DTO projection을 선택합니다.

꼬리질문

질문답변 방향

JSJ-13

Spring·JPA 개념 인과관계

Spring/JPA 답변이 얕아지는 이유는 어노테이션을 기능 스위치처럼 말하기 때문입니다. 실제로는 proxy, transaction boundary, persistence context, SQL 실행 시점이 연결되어 하나의 요청 경로를 만듭니다.

SPRING JPA CAUSAL MODEL@Transactional은 method에 마법을 거는 것이 아니라 proxy가 호출 전후에 transaction을 여닫는 구조입니다. 그래서 self-invocation은 proxy를 우회하고, checked exception은 기본 rollback 대상이 아니며, transaction 안에서 외부 API를 오래 기다리면 DB connection과 lock을 오래 점유합니다. JPA도 SQL을 없애는 것이 아니라 persistence context 안의 entity 변경을 flush 시점에 SQL로 번역합니다. 이 때문에 lazy loading, N+1, dirty checking 비용, bulk update 후 stale entity 문제가 발생합니다.

JSJ-14

실전 장애 디버깅: Spring 요청 하나를 끝까지 추적하기

전문가 수준의 Java/Spring/JPA 답변은 어노테이션 동작을 설명하는 데서 끝나지 않습니다. 느린 요청 하나를 HTTP thread, transaction proxy, Hibernate SQL, Hikari pool, DB lock, GC까지 같은 타임라인으로 추적해야 합니다.

EXPERT SPRING DEBUGGING CASE시나리오는 “주문 목록 API의 p99가 400ms에서 6초로 상승했고, CPU는 낮고 DB CPU도 아주 높지 않은데 사용자 timeout이 늘었다”입니다. 초보 답변은 “쿼리가 느린 것 같습니다”에서 멈춥니다. 전문가 답변은 요청 trace를 먼저 나눕니다. controller 진입 전 대기인지, service 내부 transaction이 긴지, SQL 실행이 느린지, connection pool에서 기다리는지, downstream API나 serialization이 느린지 확인합니다.

1. 요청 타임라인을 먼저 나눈다

APM trace나 structured log에서 request id를 기준으로 시간을 나눕니다. servlet thread가 요청을 받았는지, filter/security chain에서 지연이 있는지, controller 진입 후 service 호출까지 얼마인지, transaction 시작과 commit 사이가 얼마인지, SQL 개수와 각 SQL 시간이 얼마인지 봅니다. SQL 하나가 5초인 경우와 SQL 100개가 각각 50ms인 경우는 해결책이 다릅니다. 전자는 실행 계획과 lock을 보고, 후자는 N+1이나 반복 호출을 의심합니다.

2. HikariCP metric을 읽는다

Hikari의 active, idle, pending, max, connection timeout을 함께 봅니다. pending thread가 늘고 active가 max에 붙어 있으면 애플리케이션 thread가 DB connection을 기다리는 상태입니다. 이때 connection pool size를 무작정 늘리면 DB가 더 많은 동시 쿼리를 처리해야 해서 상황이 악화될 수 있습니다. 먼저 transaction duration, OSIV, slow query, 외부 API를 transaction 안에서 호출하는지 확인합니다. pool wait가 길고 SQL 시간은 짧다면 connection을 빌린 뒤 반납하지 않거나 transaction 범위가 과도하게 긴 구조일 수 있습니다.

3. Transaction proxy와 rollback 경계를 확인한다

@Transactional이 붙어 있어도 proxy를 통과하지 않는 self-invocation이면 transaction이 열리지 않을 수 있습니다. 반대로 transaction이 너무 넓으면 lock과 connection을 오래 잡습니다. checked exception은 기본 rollback 대상이 아니므로 비즈니스 실패가 commit되는 경우도 있습니다. 디버깅에서는 transaction active log, propagation, rollbackOnly, exception type을 봅니다. “transaction을 걸면 안전하다”가 아니라 “어떤 변경을 하나의 실패 단위로 묶고, 어떤 외부 부작용은 transaction 밖으로 뺄지”가 핵심입니다.

4. Hibernate SQL과 persistence context를 확인한다

Hibernate SQL log에서 같은 패턴의 select가 반복되면 N+1을 의심합니다. 다만 fetch join이 항상 정답은 아닙니다. 컬렉션 fetch join은 row 중복과 pagination 문제를 만들 수 있고, 여러 bag을 동시에 fetch하면 MultipleBagFetchException이 발생할 수 있습니다. 목록 API에서는 DTO projection, batch size, entity graph, 2단계 id 조회를 비교합니다. 대량 update에서는 persistence context가 커져 dirty checking 비용이 커지므로 batch 단위 flush/clear가 필요합니다. JPQL bulk update는 persistence context를 우회하므로 stale entity가 남을 수 있습니다.

5. Thread dump와 GC를 함께 본다

CPU가 낮은데 지연이 높으면 thread가 RUNNABLE이 아니라 WAITING/BLOCKED에 있을 수 있습니다. thread dump에서 Hikari connection borrow, synchronized lock, remote call, file I/O, blocking queue 대기를 확인합니다. GC pause가 길면 모든 thread가 멈춘 것처럼 보일 수 있으므로 GC log와 allocation rate도 확인합니다. 단, GC가 보인다고 바로 heap을 늘리는 것은 위험합니다. cache, ThreadLocal, 큰 DTO list, response buffering처럼 객체 생명주기와 retained path를 먼저 봅니다.

면접에서의 90초 답변

“저라면 먼저 trace id 기준으로 요청 시간을 filter, controller, service transaction, SQL, serialization, downstream으로 나눕니다. Hikari pending이 증가하면 connection pool wait와 transaction duration을 보고, SQL 개수와 실행 시간을 분리해 N+1인지 slow query인지 판단합니다. transaction 안에서 외부 API를 기다리거나 OSIV 때문에 connection이 오래 잡히면 경계를 줄입니다. Hibernate 쪽은 fetch join, DTO projection, batch size를 데이터 규모와 pagination 요구에 따라 고르고, 변경 후에는 p95/p99, query count, pool wait, DB plan을 비교합니다.”
