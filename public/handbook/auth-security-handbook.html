인증과 보안
핸드북

보안은 나중에 붙이는 기능이 아니라 기본값입니다 — 제1원칙은 "모든 사용자 입력은 적대적이라고 가정한다". 이 핸드북은 패턴 암기가 아니라 왜 그렇게 하는가, 무엇을 포기하는가, 어떻게 실패하는가에 초점을 둡니다. 프론트엔드에서 풀스택으로 넘어오며 마주칠 인증·인가·취약점의 사고 모델을 잡습니다.

SEC-01

인증 vs 인가

보안 사고의 큰 갈래 하나는 이 둘을 뭉뚱그리는 데서 시작합니다. authentication은 "너 누구냐"를 확인하는 것, authorization은 "이걸 할 권한이 있냐"를 판정하는 것 — 순서도 책임도 다릅니다.

로그인에 성공했다고 해서 모든 자원에 접근할 수 있는 게 아닙니다. authentication은 신원(identity)을 세우고, authorization은 그 신원이 특정 자원·작업에 대해 권한이 있는지를 매번 따집니다. 둘은 별개의 단계이며, 인증이 끝났다고 인가를 건너뛰면 그게 바로 사고입니다. 흔한 약어로 authentication은 authN, authorization은 authZ로 적습니다.

구분authN (인증)authZ (인가)
질문"너 누구냐?""이걸 할 권한 있냐?"
판정 대상신원(identity) — 자격증명 검증권한(permission) — 자원·작업별
시점보통 한 번 (로그인)요청마다 반복
실패 시 응답401 Unauthorized403 Forbidden
예비밀번호·토큰·OTP 확인"이 주문은 네 것이냐" 검사
HTTP 상태코드의 함정

이름이 헷갈립니다 — 401 Unauthorized는 사실 인증 실패(누군지 모름·자격증명 없음)이고, 403 Forbidden이 인가 실패(누군진 알지만 권한 없음)입니다. 약어 표기로는 401=authN 실패, 403=authZ 실패로 기억하세요. 신원은 확인됐는데 권한이 없으면 401이 아니라 403을 돌려줘야 클라이언트가 "다시 로그인"이 아니라 "권한 없음"으로 옳게 반응합니다(→ 백엔드 개요).

SEC-02

세션 기반 인증

전통적이고 여전히 견고한 방식입니다. 서버가 상태(session)를 쥐고, 브라우저엔 그 세션을 가리키는 식별자만 쿠키로 건넵니다. 핵심 가치는 "서버가 통제권을 쥔다"는 점.

로그인에 성공하면 서버는 세션 객체(사용자 id, 만료 시각 등)를 세션 저장소에 만들고, 임의의 추측 불가능한 session ID를 발급해 Set-Cookie로 내려보냅니다. 이후 브라우저는 같은 도메인 요청마다 그 쿠키를 자동으로 실어 보내고, 서버는 ID로 저장소를 조회해 누군지 압니다. 민감한 데이터는 전부 서버에 남고, 클라이언트는 의미 없는 식별자만 들고 있습니다.

SESSION FLOW [로그인] client ──▶ server : id/pw 검증
server ──▶ store : 세션 생성(sid → user)
server ──▶ client : Set-Cookie: sid=...; HttpOnly; Secure; SameSite
[이후 요청] client ──▶ server : Cookie: sid=... (자동 첨부)
server ──▶ store : sid 조회 → 신원 복원

가장 큰 장점은 즉시 무효화입니다. "로그아웃", "모든 기기에서 로그아웃", "탈취된 세션 강제 종료"가 저장소에서 레코드 하나 지우는 것으로 끝납니다. 단점은 확장성 — 서버가 상태를 쥐므로, 인스턴스가 여러 대면 어느 서버로 요청이 가든 같은 세션을 봐야 합니다. 그래서 세션을 메모리가 아니라 Redis 같은 공유 저장소에 두는 게 정석입니다.

쿠키 속성역할 / 왜 필요한가
HttpOnlyJavaScript에서 쿠키 접근 차단 → XSS로 세션 탈취 방어(→ SEC-09)
SecureHTTPS에서만 전송 → 평문 네트워크 노출 방지
SameSite크로스 사이트 요청에 쿠키 첨부 제한 → CSRF 완화(Lax/Strict)
Max-Age / Expires만료. 없으면 세션 쿠키(브라우저 종료 시 소멸)
세션과 CSRF의 관계

세션 쿠키는 브라우저가 자동으로 첨부하기 때문에, 공격자가 사용자를 속여 악성 폼을 제출하게 하면 쿠키가 딸려 가 정상 요청처럼 보입니다 — 이게 CSRF입니다(→ SEC-09). 그래서 세션 방식은 거의 항상 CSRF 방어(토큰 또는 SameSite 쿠키)와 함께 가야 합니다. 반대로 토큰을 Authorization 헤더로 직접 실어 보내는 방식(→ SEC-03)은 자동 첨부가 없어 CSRF에 구조적으로 덜 취약합니다.

SEC-03

토큰 기반 — JWT

JWT(JSON Web Token)는 서명된 자기 완결적 토큰입니다. 서버에 세션을 저장하지 않고, 토큰 자체에 신원 정보를 담아 서명으로 위변조를 막습니다. 핵심 트레이드오프 한 줄: stateless를 얻는 대신, 만료 전 무효화를 잃는다.

JWT는 점(.)으로 구분된 세 부분 header.payload.signature입니다. header와 payload는 단순 Base64URL 인코딩일 뿐 — 암호화가 아닙니다. 누구나 디코드해 내용을 읽을 수 있으므로 비밀번호 같은 민감 정보를 payload(claims)에 담으면 안 됩니다. 마지막 signature는 서버의 비밀키로 header+payload를 서명한 값이고, 서버는 받은 토큰을 같은 키로 재서명해 일치하는지로 위변조 여부를 검증합니다.

JWT STRUCTURE header.payload.signature
{alg, typ} | {sub, exp, role...} | HMAC/RSA(...)

VERIFY sig == sign(header + payload, secret) ? 신뢰 : 거부
※ payload는 Base64일 뿐 — 누구나 읽을 수 있음(암호화 아님)

stateless라서 서버는 토큰만 검증하면 되고, 저장소 조회가 없어 수평 확장에 유리합니다. 그런데 바로 그 점이 약점입니다 — 서버가 토큰을 기억하지 않으니, 만료 전에 특정 토큰을 무효화하기가 어렵습니다. 탈취된 토큰은 만료될 때까지 유효합니다. 그래서 실무 정석은 두 종류로 나눕니다.

토큰수명역할 / 저장 위치
access token짧게 (분 단위)API 호출에 첨부. 짧아서 탈취돼도 피해 창이 작음
refresh token길게 (일~주)access 재발급용. 서버가 저장·추적 → 무효화 가능

access는 짧게 만들어 탈취 위험을 줄이고, 만료되면 refresh token으로 새 access를 받습니다. refresh token은 보통 서버가 저장소에 보관해 폐기(rotation·revocation)할 수 있게 합니다 — 결국 무효화가 필요한 부분만 stateful로 되돌리는 절충입니다. 저장 위치도 트레이드오프입니다.

저장 위치장점위험
localStorage다루기 쉬움, 헤더로 전송XSS로 읽힘 → 탈취(→ SEC-09)
HttpOnly 쿠키JS 접근 불가 → XSS에 강함자동 첨부 → CSRF 대비 필요
치명적 실수 — alg:none과 키 혼동

JWT 검증에서 반복되는 사고: ① alg: none을 받아들이는 라이브러리 설정 — 서명 없는 토큰을 통과시켜 누구나 위조 가능. ② alg를 토큰이 시키는 대로 따르다 RS256(공개키 검증)을 HS256(대칭키)로 바꿔치기당해, 공개된 키를 비밀키로 오용하는 공격. 검증 알고리즘은 서버가 고정하고 토큰의 alg를 신뢰하지 마세요. 그리고 payload는 평문이니 거기에 비밀을 담지 마세요.

SEC-04

OAuth2 / OIDC

"구글로 로그인"의 뒤편입니다. 둘은 다른 일을 합니다 — OAuth2는 인가 위임(delegated authorization) 프로토콜이고, OIDC는 그 위에 인증(authentication) 층을 얹은 것입니다. 이걸 섞으면 설계가 어긋납니다.

OAuth2의 본래 목적은 "비밀번호를 넘기지 않고 제3자 앱에게 내 자원의 일부 접근을 허용"하는 것입니다 — 즉 인가 위임이지 "누구인지 알려주기"가 아닙니다. 그래서 "OAuth2로 로그인"은 엄밀히는 오용이고, 로그인(=신원 확인)이 필요하면 OAuth2 위에 표준 신원 계층을 더한 OIDC(OpenID Connect)를 씁니다. OIDC는 사용자의 신원을 담은 ID Token(JWT 형식)을 추가로 발급합니다.

역할(role)의미
Resource Owner자원의 주인 = 보통 사용자 본인
Client접근을 원하는 앱 (우리 서비스)
Authorization Server동의를 받고 토큰을 발급 (구글·카카오 등)
Resource Server실제 자원을 가진 API 서버

가장 권장되는 플로우는 Authorization Code + PKCE입니다. 사용자를 authorization server로 보내 로그인·동의를 받으면, 서버는 짧은 일회성 authorization code를 redirect로 돌려줍니다. 클라이언트는 그 code를 다시 서버 백채널로 보내 access token(과 OIDC면 ID token)으로 교환합니다. 토큰이 브라우저 URL에 직접 노출되지 않는 게 핵심입니다.

AUTHORIZATION CODE + PKCE 1. client → auth server : code_challenge 동봉 + 로그인 요청
2. user : 로그인·scope 동의
3. auth server → client : authorization code (redirect)
4. client → auth server : code + code_verifier (백채널 교환)
5. auth server → client : access token (+ id token)
※ PKCE: code 가로채기 공격 방어 — verifier를 아는 자만 교환 성공
개념의미
scope위임 범위 — "이메일·프로필 읽기"처럼 권한의 한정. 최소만 요청
access tokenresource server 호출용 (OAuth2)
ID token사용자 신원 정보 (OIDC, JWT) — 로그인 판정에 사용
PKCEcode 교환에 verifier를 묶어 가로채기 방어. SPA·모바일 필수
자주 하는 혼동

access token으로 "로그인 됐다"고 판단하지 마세요 — access token은 인가용이라 누구인지를 보장하지 않습니다. 로그인 여부는 OIDC의 ID token으로 판정합니다. 또 redirect URI는 화이트리스트로 정확히 등록·검증해야 합니다 — 느슨하면 토큰을 공격자 주소로 빼돌리는 open redirect 사고가 납니다. SSO(한 번 로그인으로 여러 서비스)는 이 OIDC 위에서 구성됩니다.

SEC-05

비밀번호와 해싱

비밀번호 저장의 제1원칙: "원문을 복원할 수 없게 만든다." 그래서 암호화가 아니라 해싱을 씁니다. 둘을 혼동하는 것이 평문 저장 다음으로 흔한 사고입니다.

암호화(encryption)는 키로 되돌릴 수 있는 양방향 변환입니다. 해싱(hashing)은 단방향 — 같은 입력은 같은 출력을 내지만, 출력에서 입력을 복원할 수 없습니다. 비밀번호는 우리조차 원문을 알 필요가 없으니(로그인 시 입력값을 같은 방식으로 해싱해 비교만 하면 됨) 해싱이 맞습니다. DB가 털려도 원문이 드러나지 않게 하는 게 목적입니다.

그런데 아무 해시나 되는 게 아닙니다. MD5·SHA-1·SHA-256은 빠르도록 설계된 함수라 비밀번호용으로는 부적합합니다 — 빠르다는 건 공격자가 초당 수십억 개를 대입(brute force)할 수 있다는 뜻이기 때문입니다. 비밀번호엔 일부러 느린 전용 함수를 씁니다.

요소역할 / 왜
salt사용자마다 다른 랜덤값을 비밀번호에 더해 해싱 → 같은 비밀번호도 다른 해시. rainbow table·동일 비번 노출 방어
work factor해싱 반복 비용(cost). 하드웨어가 빨라지면 올려서 brute force를 계속 비싸게 유지
bcrypt검증된 표준. salt·work factor 내장. 무난한 기본 선택
scrypt메모리도 많이 쓰게 해 GPU 병렬 공격에 강함
argon2최신 권장(argon2id). 시간·메모리·병렬도 조절 가능
절대 하지 말 것

① 평문 저장 — 논외. ② MD5/SHA 단독 해싱 — 빠른 해시라 무용지물. ③ salt 없이 해싱 — 같은 비밀번호가 같은 해시가 되어 rainbow table에 뚫림. ④ 직접 구현 — 암호 함수를 손으로 짜지 말고 검증된 라이브러리(bcrypt/argon2)를 쓰세요. ⑤ 로그에 비밀번호를 남기지 마세요(→ SEC-10). Spring Security라면 PasswordEncoder(예: BCryptPasswordEncoder)를 표준으로 씁니다.

SEC-06

인가 모델

인증을 통과한 다음의 세계입니다. "이 사용자가 이 자원에 이 작업을 해도 되는가"를 일관되게 판정하는 구조 — 그리고 그 판정을 빠뜨리는 것이 가장 흔한 인가 사고입니다.

가장 널리 쓰이는 두 모델입니다. RBAC(Role-Based)는 사용자에게 역할(role)을 주고 역할에 권한을 묶습니다 — "admin은 삭제 가능, member는 읽기만". 단순하고 관리가 쉽습니다. ABAC(Attribute-Based)는 사용자·자원·환경의 속성으로 규칙을 만듭니다 — "같은 부서이고 근무시간 내일 때만 접근". 더 세밀하지만 복잡합니다.

모델판정 기준장점 / 한계
RBAC역할(role)단순·관리 용이 / 세밀한 조건 표현엔 한계
ABAC속성(attribute)세밀·유연 / 규칙이 복잡해지고 디버깅 어려움
ownership 검사자원의 소유자 == 요청자?RBAC/ABAC와 함께 — 역할만으론 못 막는 사고 방어

관통하는 원칙은 최소 권한(least privilege)입니다 — 기본은 거부, 필요한 만큼만 허용. 그리고 역할 검사만으로 끝내면 안 됩니다. "member 역할이면 주문 조회 가능"까지만 보고 그 주문이 이 사용자의 것인지(자원 소유권)를 확인하지 않으면, 남의 주문 id로 바꿔치기해 조회·수정할 수 있습니다.

IDOR — 인가 누락의 대표 사고

IDOR(Insecure Direct Object Reference)는 /orders/1024의 id를 1025로 바꾸기만 해도 남의 자원이 열리는 취약점입니다. 원인은 단 하나 — 요청한 자원이 이 사용자 소유인지 확인하는 인가 검사를 빠뜨린 것. 인증됐다고, 역할이 맞다고 끝이 아닙니다. 모든 자원 접근에서 소유권·권한을 명시적으로 검사하세요. "id는 추측 못 할 거야"(추측 불가 식별자)는 방어가 아니라 가림막일 뿐입니다(→ SEC-07 Broken Access Control).

SEC-08

Injection과 입력 검증

Injection의 본질은 한 문장입니다 — "데이터로 와야 할 것이 코드로 실행된다." 사용자 입력을 코드·쿼리 문자열에 그대로 이어붙이는 순간, 입력이 명령이 됩니다.

대표는 SQL Injection입니다. 입력을 쿼리 문자열에 연결하면, ' OR '1'='1 같은 입력이 쿼리의 구조를 바꿔 인증을 우회하거나 테이블을 통째로 빼냅니다(DROP TABLE까지). 근본 해법은 입력을 거르는 게 아니라 데이터와 코드를 애초에 분리하는 것 — parameterized query / prepared statement입니다. 자리표시자(? 또는 named param)에 값을 바인딩하면, DB가 그 값을 항상 데이터로만 다루고 절대 구문으로 해석하지 않습니다.

왜 parameterized query인가 // 취약 — 입력이 구문이 됨
"SELECT * FROM users WHERE id = '" + input + "'"
input = " ' OR '1'='1 " → 항상 참, 전체 노출

// 안전 — 값은 끝까지 데이터
"SELECT * FROM users WHERE id = ?" + bind(input)
JPA/Hibernate는 기본이 바인딩 — 단, 직접 문자열 조립 시 주의
방어의미
parameterized query값을 바인딩해 데이터/코드 분리. SQLi의 정석 해법
화이트리스트 검증"허용되는 것만 통과" — 블랙리스트("나쁜 것만 차단")보다 안전
최소 권한 DB 계정앱 계정에 DROP·관리 권한 안 줌 → 피해 한정(→ DB 핸드북)
출력 이스케이프명령·HTML로 나갈 때 컨텍스트에 맞게 이스케이프(→ SEC-09)
JPA를 쓴다고 안전한 게 아니다

JPA/Hibernate의 @Query와 메서드 쿼리는 파라미터를 바인딩하므로 기본적으로 SQLi에 안전합니다. 하지만 JPQL·native query를 문자열로 직접 조립하면("... where name = '" + name + "'") 그대로 뚫립니다. 항상 :param 바인딩이나 setParameter()를 쓰세요. SQL만이 아니라 OS 명령 실행에 입력을 넘기면 command injection도 같은 원리로 발생합니다 — 가능하면 셸을 거치지 말고 인자 배열로 직접 호출하세요.

SEC-09

XSS와 CSRF

이름이 비슷해 자주 헷갈리지만 정반대 방향의 공격입니다. XSS는 "악성 스크립트를 피해자 브라우저에서 실행"시키고, CSRF는 "피해자의 인증을 빌려 원치 않는 요청을 보내게" 합니다. 방어법도 다릅니다.

9.1XSS — Cross-Site Scripting

공격자가 주입한 JavaScript가 다른 사용자의 브라우저에서 실행되는 것입니다. 실행되면 세션 쿠키 탈취(→ HttpOnly로 완화), 화면 변조, 키 입력 가로채기까지 가능합니다. 근본 원인은 신뢰할 수 없는 데이터를 HTML/JS 컨텍스트에 이스케이프 없이 출력한 것입니다.

유형경로
Stored(저장형)악성 스크립트가 DB에 저장돼 다른 사용자에게 매번 전달 (댓글 등). 영향 큼
Reflected(반사형)요청 파라미터가 응답에 즉시 반사 (검색어·에러 메시지). 악성 링크 유도
DOM-based서버 거치지 않고 클라이언트 JS가 입력을 위험하게 DOM에 삽입

방어의 핵심은 출력 이스케이프(context-aware escaping)입니다 — 데이터가 HTML 본문/속성/JS/URL 중 어디로 나가는지에 맞춰 인코딩합니다. React·Vue 같은 프레임워크는 기본적으로 출력을 이스케이프하므로 안전하지만, dangerouslySetInnerHTML(React)이나 v-html(Vue)로 우회하면 그 보호가 사라집니다. 보조 방어로 CSP(Content-Security-Policy) 헤더로 실행 가능한 스크립트 출처를 제한합니다.

9.2CSRF — Cross-Site Request Forgery

CSRF는 브라우저가 쿠키를 자동 첨부한다는 신뢰를 악용합니다. 로그인된 사용자가 공격자의 페이지를 열면, 그 페이지가 우리 사이트로 요청을 보내고 브라우저는 세션 쿠키를 딸려 보냅니다 — 서버는 정상 요청으로 착각합니다(→ SEC-02). 사용자는 아무것도 모릅니다.

방어의미
CSRF 토큰폼·요청마다 예측 불가 토큰을 심고 서버가 검증. 공격 페이지는 토큰을 모름
SameSite 쿠키크로스 사이트 요청에 쿠키 첨부 제한(Lax/Strict). 현대 브라우저 기본 방어
헤더 토큰 방식Authorization 헤더 토큰은 자동 첨부 안 됨 → CSRF에 구조적으로 강함
한 줄 비교 — 방향이 반대다

XSS: 공격자 코드가 피해자 브라우저에서 실행 → 방어는 출력 이스케이프 + CSP. CSRF: 피해자의 인증으로 원치 않는 요청 전송 → 방어는 CSRF 토큰 + SameSite. 주의: XSS가 뚫리면 CSRF 방어는 무력화됩니다(스크립트가 토큰을 읽어버림). 그래서 XSS 방어가 더 근본적입니다 — 입력 신뢰 금지의 원칙으로 되돌아옵니다.

SEC-10

전송·저장 보안과 secret 관리

데이터는 두 상태에서 보호받아야 합니다 — 이동 중(in transit)과 저장 중(at rest). 그리고 그 보호의 열쇠인 secret 자체를 어떻게 다루느냐가 전체 보안의 급소입니다.

전송 중: TLS(HTTPS)로 클라이언트-서버 구간을 암호화해 도청·변조(man-in-the-middle)를 막습니다. 평문 HTTP로 자격증명·토큰을 보내는 건 그대로 노출입니다. HSTS 헤더로 브라우저가 항상 HTTPS만 쓰도록 강제할 수 있습니다(→ DB 핸드북·백엔드 개요의 TLS 핸드셰이크 참고). 저장 중: 민감 데이터(개인정보 등)는 디스크에서도 암호화(encryption at rest)하고, 비밀번호는 암호화가 아니라 해싱입니다(→ SEC-05).

보안 헤더역할
Strict-Transport-SecurityHTTPS 강제(HSTS) — 다운그레이드 공격 차단
Content-Security-Policy스크립트·리소스 출처 제한 → XSS 완화(→ SEC-09)
X-Content-Type-Options: nosniffMIME 타입 추측 차단
X-Frame-Options / frame-ancestorsiframe 삽입 제한 → clickjacking 방어

secret 관리가 가장 자주 사고나는 지점입니다. API 키·DB 비밀번호·서명 키를 코드에 하드코딩하거나 git에 커밋하면, 저장소 접근 권한이 곧 모든 비밀의 노출입니다. git 히스토리는 영구적이라 한 번 커밋된 secret은 나중에 지워도 과거 커밋에 남습니다 — 노출 즉시 키 폐기·재발급(rotation)이 정답입니다.

secret은 코드·git·로그에 절대 두지 않는다

① secret은 환경변수 또는 secret manager(Vault·AWS Secrets Manager 등)로 주입하고, .env·credentials/는 .gitignore에 넣어 절대 커밋하지 마세요. ② 로그에 PII·토큰·비밀번호를 흘리지 마세요 — 디버깅용 console.log(req.body) 하나가 자격증명을 평문 로그로 남깁니다. ③ 에러 메시지에 스택트레이스·내부 경로·쿼리를 사용자에게 노출하지 마세요(정보 누출). 이미 노출됐다면 지우는 것보다 해당 secret을 무효화하는 게 먼저입니다.

SEC-12

Case study

관리자가 사용자 권한을 낮췄는데 기존 access token이 계속 유효하면 몇 분 동안 초과 권한이 남습니다. 짧은 access token, refresh token rotation, 권한 버전 claim, 중요 작업의 서버 측 재검증을 조합해야 권한 변경과 세션 경험의 균형을 맞출 수 있습니다.

SEC-13

Security Threat Modeling

위협 모델링은 "무서운 것을 상상하는 회의"가 아니라 설계가 어떤 신뢰 가정 위에 서 있는지 드러내는 작업입니다. 기능을 만들기 전에 데이터 흐름, 신뢰 경계, 권한 경계, 감사 가능성을 놓고 "어디서 공격자가 개입할 수 있는가"를 구체적으로 묻습니다.

좋은 위협 모델은 문서가 길지 않습니다. 대신 자산(asset), 행위자(actor), 진입점(entry point), 신뢰 경계(trust boundary), 실패 시 영향이 분명합니다. 로그인, 결제, 파일 업로드, 웹훅, 관리자 기능, 테넌트 데이터 접근처럼 실패 비용이 큰 기능은 티켓이 아니라 설계 단계에서 먼저 모델링해야 합니다.

13.1STRIDE로 빠르게 훑기

STRIDE는 위협을 여섯 질문으로 나누는 실무 프레임입니다. 모든 항목에 대단한 답을 쓰는 것이 목적이 아니라, "이 기능에서 빠뜨리면 사고가 되는 질문"을 빠르게 누락 없이 통과하는 것이 목적입니다.

STRIDE질문예시 방어
Spoofing공격자가 다른 사용자·서비스인 척할 수 있는가?MFA, 세션 재발급, JWT 서명 검증, webhook 서명 검증
Tampering요청·파일·이벤트·권한값을 바꿔도 서버가 믿는가?서버 측 재계산, 무결성 검증, allowlist, optimistic locking
Repudiation나중에 "내가 한 게 아니다"를 반박할 기록이 있는가?감사 로그, 요청 id, actor/ip/user-agent, 변경 전후 diff
Information Disclosure다른 사용자·테넌트·내부 정보가 새는가?객체 단위 인가, 필드 마스킹, 에러 메시지 축소, 로그 redaction
Denial of Service작은 비용의 요청으로 큰 서버 비용을 만들 수 있는가?rate limit, payload 제한, 큐 격리, timeout, backpressure
Elevation of Privilege일반 사용자가 관리자·소유자 권한을 얻는 경로가 있는가?권한 중앙화, step-up auth, dual control, 정책 테스트
THREAT MODEL MINI LOOP 1. 무엇을 보호하나? 자산: 계정, 결제수단, 개인정보, 파일, API key
2. 누가 접근하나? anonymous, user, tenant admin, internal admin, partner service
3. 어디서 경계가 바뀌나? browser → API → worker → DB → external provider
4. 무엇을 신뢰하지 않나? client input, URL id, headers, file metadata, webhook payload
5. 실패하면 어떻게 알 수 있나? audit log, alert, anomaly query, replayable request id

13.2Trust Boundary는 선으로 그린다

신뢰 경계는 데이터가 다른 신뢰 수준으로 넘어가는 지점입니다. 브라우저에서 온 값, 모바일 앱에서 온 값, 파트너가 보낸 웹훅, 내부 큐 메시지, 관리자 콘솔 입력은 모두 별도 경계입니다. "우리 프론트엔드가 보내니까 안전하다"는 말은 경계 밖의 입력을 신뢰한다는 뜻입니다.

경계위험 질문필수 통제
Browser → API사용자가 hidden field, price, role, ownerId를 바꾸면?서버 측 권한 판정, 값 재계산, schema validation
API → DB쿼리 조건에 tenantId/ownerId가 항상 붙는가?repository 정책, RLS 또는 mandatory predicate, 테스트
API → Worker/Queue큐 메시지를 내부 신뢰로 처리해도 되는가?message schema, idempotency key, actor/context 포함
Service → External외부 응답·callback을 그대로 상태 변경에 쓰는가?서명 검증, allowlist, timeout, replay 방어
Admin Console → Core Data관리자 실수·악용이 고객 데이터에 즉시 반영되는가?승인 단계, reason code, audit log, 민감 작업 재인증
경계 밖 데이터는 다시 검증한다

한 번 검증한 데이터라도 다른 경계를 넘으면 다시 검증합니다. 예를 들어 API에서 검증한 업로드 작업도 worker가 파일을 열기 전 크기·MIME·확장자·바이러스 스캔 상태를 다시 확인해야 합니다. 큐, 캐시, 웹훅은 "내부"처럼 보이지만 공격자가 간접적으로 값을 넣을 수 있으면 신뢰 경계 밖입니다.

13.3Tenant Isolation은 필터가 아니라 불변식

멀티테넌트 서비스의 보안 목표는 단순합니다. 한 테넌트의 사용자가 다른 테넌트의 데이터를 읽거나, 바꾸거나, 존재 여부를 추론할 수 없어야 합니다. 이를 컨트롤러마다 where tenant_id = ?를 기억하는 방식으로만 운영하면 언젠가 빠집니다.

계층설계 기준리뷰 질문
식별자모든 테넌트 소유 리소스에 tenant_id를 필수 보유테넌트 없는 row가 생성될 수 있는가?
쿼리조회·수정·삭제 조건에 tenant predicate를 강제id만으로 findById 하는 경로가 있는가?
권한role은 tenant scope 안에서만 의미를 가짐tenant A admin이 tenant B에 admin인가?
캐시캐시 키에 tenant scope 포함user:123 같은 전역 키가 섞이지 않는가?
검색·통계index, export, analytics에도 tenant filter 적용운영성 기능이 isolation을 우회하지 않는가?
DB 방어가능하면 RLS, view, repository guard 등 2차 방어애플리케이션 버그 하나로 전 테넌트가 노출되는가?

13.4IDOR 테스트 케이스

IDOR는 "id를 바꿔본다"에서 시작하지만, 실무 테스트는 더 넓어야 합니다. 단건 조회뿐 아니라 목록, 검색, export, bulk update, soft delete, 첨부파일 다운로드, websocket subscription까지 객체 참조가 들어가는 모든 표면을 대상으로 봅니다.

케이스테스트 방법기대 결과
단건 조회user A 토큰으로 user B의 /orders/{id} 요청403 또는 존재 숨김이 필요한 경우 404
수정·삭제본문의 ownerId, tenantId, role을 다른 값으로 변경서버가 본문 값을 무시하고 권한 컨텍스트로 판정
목록·검색filter에 다른 tenant/user id를 직접 지정허용된 scope 밖 row가 0건
파일 다운로드첨부파일 id 또는 signed URL을 다른 계정에서 재사용객체 권한 재검증, URL 만료·scope 확인
Bulk API요청 배열에 내 리소스와 남의 리소스를 섞음부분 성공 정책 명확, unauthorized 항목 차단·기록
Export/Report비동기 리포트 생성 요청에 다른 tenant id 삽입worker 단계에서도 tenant scope 재검증
테스트가 통과해야 하는 기준

컨트롤러에서 403을 한 번 돌려주는 것만으로 충분하지 않습니다. repository, service, worker, export job처럼 같은 데이터에 접근하는 다른 경로에서도 같은 정책이 적용되어야 합니다. 인가 테스트는 "성공 케이스"보다 "다른 사용자의 자원을 섞었을 때 실패하는 케이스"가 더 중요합니다.

13.5Audit Log Design

감사 로그는 디버그 로그가 아닙니다. 침해 대응, 내부자 악용 조사, 고객 문의, 규제 대응에서 행위의 증거가 되는 별도 데이터 모델입니다. 그래서 읽기 좋게 남기는 것보다 변조 어렵고, 검색 가능하고, 민감정보를 과하게 담지 않는 것이 중요합니다.

필드기준주의
actoruser id, admin id, service account, impersonation 여부이메일만 남기면 변경·재사용 추적이 약함
actionuser.role.changed, invoice.refunded처럼 안정적 이벤트명자유문장 로그는 집계·alert가 어려움
resourcetype, id, tenant id, owner id테넌트 scope 누락 시 조사 불가
before/after민감 필드는 마스킹한 변경 diff비밀번호, token, secret, 전체 PII를 남기지 않음
contextrequest id, ip, user-agent, auth method, reason code프록시 환경의 실제 client ip 해석 기준 필요
integrityappend-only 저장, 보존 기간, 관리자 삭제 제한관리자가 흔적을 지울 수 있으면 감사가 아님
AUDIT EVENT SHAPE {
  "event": "tenant.member.role_changed",
  "actor": {"type":"user","id":"u_123","tenant":"t_1"},
  "resource": {"type":"member","id":"m_456","tenant":"t_1"},
  "change": {"role":{"from":"viewer","to":"admin"}},
  "context": {"request_id":"req_...","ip":"203.0.113.10","reason":"support_ticket"}
}
※ secret, raw token, password, full card number는 감사 로그에도 저장하지 않는다

13.6File Upload와 Webhook 위험

파일 업로드와 웹훅은 공격자가 서버 안쪽으로 데이터를 밀어 넣는 대표 경로입니다. 둘 다 "사용자가 보낸 입력"보다 위험할 수 있습니다. 파일은 저장소·백신·썸네일러·다운로드 경로를 지나고, 웹훅은 외부 서비스 이름을 달고 내부 상태 변경을 트리거하기 때문입니다.

표면주요 위험실무 방어
파일 업로드웹셸, polyglot 파일, MIME 속임, zip bomb, 악성 이미지 parser exploit크기 제한, 확장자·MIME allowlist, magic byte 확인, AV scan, 이미지 재인코딩
파일 저장경로 조작, 공개 버킷, 원본 파일명 XSS, tenant 간 파일 섞임서버 생성 파일명, private bucket, metadata escape, tenant prefix 분리
파일 다운로드signed URL 재사용, 권한 변경 후 접근 유지, content sniffing짧은 만료, 다운로드 직전 인가, Content-Disposition, nosniff
Webhook 수신서명 없는 위조 이벤트, replay, 이벤트 순서 뒤집힘, SSRF callbackHMAC 서명 검증, timestamp window, idempotency, event id 저장, allowlist
Webhook 처리외부 payload를 내부 권한으로 상태 변경상태 전이 검증, 서버 측 조회로 사실 확인, 실패 큐·재시도 제한
Webhook은 인증된 API처럼 다룬다

웹훅 endpoint가 공개 URL이라는 점을 잊지 마세요. provider 이름이 path에 들어있어도 인증이 아닙니다. 서명 검증, timestamp 검증, replay 방지, idempotency를 통과한 뒤에도 "이 이벤트가 현재 리소스 상태에서 가능한 전이인가"를 서버가 다시 판단해야 합니다.

13.7Admin Abuse Cases

관리자 기능은 "권한이 높으니 괜찮다"가 아니라 권한이 높아서 더 엄격해야 하는 표면입니다. 외부 공격자만 위협 행위자가 아닙니다. 실수한 운영자, 과도한 권한을 가진 CS 담당자, 탈취된 관리자 계정, 악의적 내부자도 모델에 포함해야 합니다.

남용 케이스위험통제
사용자 impersonation고객 계정으로 임의 작업, 부인 가능성읽기 전용 기본값, 명시적 사유, 배너 표시, 모든 행위 audit
권한 상승·역할 변경동료 또는 본인에게 과도 권한 부여self-grant 금지, dual approval, step-up auth, break-glass 절차
PII 조회·export대량 개인정보 유출필드 마스킹, 목적 제한, export 승인, anomaly alert
결제 취소·환불금전 손실, 고객 분쟁금액 threshold, 2인 승인, ledger 불변 기록
감사 로그 삭제침해 흔적 제거append-only 저장, 별도 권한, 보존 정책, 삭제 대신 redaction workflow

13.8Security Review Rubric

리뷰는 "보안 문제 없어 보임"이 아니라 구체적인 질문에 답하는 과정이어야 합니다. 아래 rubric은 PR 설명이나 설계 문서에 붙여도 됩니다. 답이 "해당 없음"이면 왜 해당 없는지까지 짧게 남기는 것이 좋습니다.

영역리뷰 질문통과 기준
AuthN/AuthZ새 endpoint/job/subscription이 인증과 객체 단위 인가를 모두 수행하는가?권한 정책 위치와 negative test가 명확함
Tenant Isolation모든 query, cache key, export, background job에 tenant scope가 있는가?id-only 접근 경로가 없음
Input/Output외부 입력이 schema validation을 통과하고 출력 컨텍스트에 맞게 escape되는가?allowlist, binding, escaping이 코드로 확인됨
Secrets/PIIsecret, token, PII가 코드·로그·URL·analytics에 남지 않는가?redaction과 보관 기간이 정의됨
State Change중요 변경에 CSRF/replay/idempotency/동시성 방어가 있는가?중복 요청과 순서 뒤집힘이 안전함
Files/Webhooks업로드·웹훅이 서명, 크기, 타입, 재시도, replay를 다루는가?비동기 처리 단계에서도 재검증함
Admin관리자 기능에 최소 권한, 사유, 재인증, 감사 로그가 있는가?내부자·계정 탈취 시 피해가 제한됨
Observability침해나 남용을 나중에 발견할 수 있는 이벤트가 남는가?audit event와 alert query가 설계됨
리뷰의 결론은 액션이어야 한다

위협 모델링의 산출물은 멋진 다이어그램이 아니라 코드와 운영 통제입니다. 예를 들어 "tenant isolation 위험"을 찾았다면 repository guard, RLS, negative test, audit event 중 무엇을 이번 변경에 넣을지 결정해야 합니다. 남은 위험은 "known risk"로 기록하고 owner와 만료일을 둡니다. 보안 리뷰가 결정으로 이어지지 않으면 회의록일 뿐입니다.
