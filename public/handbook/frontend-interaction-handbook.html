프론트엔드 인터랙션
애니메이션과 3D

인터랙션은 화면에 움직임을 더하는 장식이 아니라 사용자 의도, 시스템 상태, 입력 장치, 성능 제약, 접근성을 하나의 계약으로 묶는 엔지니어링입니다. 좋은 인터랙션은 사용자가 무엇을 할 수 있고, 무엇이 진행 중이며, 실패하면 어떻게 회복할 수 있는지 즉시 알게 만듭니다.

AUDIENCE : 프론트엔드 개발자 · 제품 디자이너 · QA · REVIEWERS
STANCE : 실무 판단 · 실패 모드 · 체크리스트 · PR 게이트 중심 · UPDATED 2026-06
INT-01

인터랙션 설계의 역할

프론트엔드 인터랙션의 목표는 “멋있게 움직인다”가 아닙니다. 사용자의 입력이 시스템에 도착했고, 시스템이 어떤 상태이며, 다음 행동이 무엇인지 사용자가 확신하게 만드는 것입니다. 애니메이션과 3D는 이 목적을 도울 때만 제품 품질이 됩니다.

관점나쁜 기준실무 기준리뷰 질문
의도클릭하면 뭔가 바뀜사용자가 시작한 작업과 결과 범위가 명확함누르기 전에 영향 범위를 알 수 있는가?
상태로딩 spinner 하나로 처리idle, pending, success, error, disabled reason을 구분현재 시스템 상태가 텍스트와 구조로도 전달되는가?
시간모든 전환에 동일한 duration작업 위험도와 정보량에 맞춰 즉시성, 지연, 확인을 조정기다림이 필요한 이유와 남은 행동이 보이는가?
입력마우스 기준으로만 설계포인터, 터치, 키보드, 보조기술, reduced motion을 함께 설계마우스 없이 핵심 플로우가 끝나는가?
성능프레임이 조금 끊겨도 시각적으로 화려함입력 반응성, layout shift, main thread 점유를 예산으로 관리이 효과가 INP, CLS, 배터리, 저사양 기기에 어떤 비용을 내는가?
INTERACTION CONTRACT

인터랙션은 사용자와 시스템 사이의 계약입니다. 버튼을 눌렀다면 눌림, 처리 중, 완료, 실패, 재시도 가능 여부가 분리되어야 합니다. 애니메이션은 이 계약을 더 읽기 쉽게 만드는 수단이지 계약을 대신할 수 없습니다.

인터랙션 우선순위

우선순위먼저 보장할 것뒤로 미룰 것
1기능이 키보드와 터치로 동작하고 상태가 명확함장식 모션, hover-only affordance
2오류와 중복 실행을 막고 입력을 보존함복잡한 transition choreography
3레이아웃 안정성과 성능 예산을 지킴스크롤 기반 대형 연출
4핵심 흐름의 모션 언어를 일관화함컴포넌트별 임의 easing과 duration
INT-02

상태와 입력 피드백

입력 피드백은 사용자가 같은 행동을 반복하지 않게 만드는 방어선입니다. 눌림, 포커스, 비활성, 진행 중, 성공, 실패가 구분되지 않으면 사용자는 중복 제출, 강제 새로고침, 이탈로 대응합니다.

상태사용자 의미UI 책임실패 모드
hover상호작용 가능해 보임포인터 환경에서 보조 단서 제공hover에만 중요한 정보를 숨김
focus키보드 현재 위치명확한 focus ring과 논리적 이동 순서outline 제거, 숨은 요소에 focus 이동
active/pressed입력이 접수됨짧고 즉각적인 눌림 피드백클릭 후 아무 변화가 없어 중복 클릭 발생
pending작업 처리 중트리거 잠금, 취소 가능성, 진행 범위 표시버튼은 다시 눌리는데 요청은 이미 진행 중
disabled현재 실행 불가이유와 해제 조건을 가까운 곳에 제공회색 버튼만 있고 무엇을 해야 하는지 모름
success작업 완료결과 위치, 저장 시각, 다음 행동을 제공toast만 사라지고 실제 반영 여부를 확인할 수 없음
error작업 실패원인, 보존된 입력, 재시도 또는 수정 경로 표시입력값이 사라지거나 같은 실패를 반복하게 함
INPUT FEEDBACK PACKETtrigger: 어떤 요소가 작업을 시작했는가
scope: 버튼 하나, 카드 하나, 페이지 전체 중 어디가 pending인가
lock: 중복 실행을 막는가, 취소는 가능한가
message: 처리 중/성공/실패가 같은 위치에서 이어지는가
recovery: 실패 후 입력과 focus가 보존되는가

DOUBLE ACTION

중복 실행 방지는 CSS disabled만으로 끝나지 않습니다. 버튼 pending, optimistic update rollback, mutation idempotency, 서버 중복 제약이 함께 필요합니다. 프론트 피드백은 사용자를 돕고, 서버 방어는 시스템을 지킵니다.

INT-03

포인터·터치·키보드

하나의 UI는 여러 입력 장치에서 같은 의미를 가져야 합니다. hover, drag, pinch, keyboard shortcut, screen reader 동작은 서로 충돌하지 않게 설계해야 하며, 포인터 중심 기능에는 반드시 대체 경로가 있어야 합니다.

입력설계 기준실패 신호PR 확인
마우스·트랙패드hover는 보조 단서, click target은 충분한 크기hover해야만 버튼이 나타나 핵심 작업을 발견 못함hover 없이도 작업을 찾을 수 있는가?
터치touch target과 간격 확보, long press 의존 최소화작은 icon button이 붙어 있어 오탭 발생모바일 viewport에서 엄지 조작이 가능한가?
키보드Tab order, Enter/Space, Arrow, Esc 의미를 컴포넌트별로 정의Tab이 visual order와 다르거나 trap에서 빠져나가지 못함키보드만으로 핵심 시나리오를 완료했는가?
스타일러스·정밀 포인터drawing, drag, resize에는 취소와 snap 기준 제공작은 실수가 곧바로 저장됨입력 취소, undo, precision mode가 있는가?
보조기술role, accessible name, live status, focus restore 제공상태 변화가 시각적으로만 전달됨이름과 상태가 DOM에서 읽히는가?

Drag와 gesture 판단표

패턴쓸 때필수 대체 경로주의

Swipe action모바일 목록에서 보조 작업을 빠르게 노출할 때명시적 버튼 또는 overflow menu삭제 같은 위험 작업을 swipe 단독으로 실행하지 않음
Pinch/zoom지도, 캔버스, 이미지 검토처럼 공간 확대가 본질일 때zoom in/out 버튼, keyboard shortcut브라우저 기본 zoom과 충돌하지 않게 설계

5. 실패 메시지와 저장 성공이 focus 또는 live status로 전달되는지 확인한다
HOVER ONLY

hover-only UI는 터치와 키보드에서 사라집니다. 행에 마우스를 올려야만 삭제 버튼이 보이는 패턴은 시각적으로 깔끔하지만 발견성과 접근성 회귀를 만듭니다. 주요 행동은 항상 보이거나 focus, menu, bulk action 같은 대체 경로가 있어야 합니다.

INT-04

모션 시스템과 안전성

모션 시스템은 duration 숫자 모음이 아니라 상태 전환의 언어입니다. 어떤 변화가 즉시 일어나고, 어떤 변화가 공간 이동으로 설명되며, 어떤 변화는 줄이거나 꺼야 하는지 팀 기준을 가져야 합니다.

목적좋은 사용피해야 할 사용성능 기준
원인과 결과 연결버튼에서 panel이 열리는 방향을 짧게 암시관련 없는 요소가 동시에 튀거나 흔들림transform, opacity 중심
공간 변화 안내삽입, 삭제, reorder 위치 변화를 안정적으로 표시layout shift를 애니메이션으로 덮음레이아웃 예약 후 전환
상태 전환loading에서 content로 부드럽게 교체끝나지 않는 shimmer만 반복long task 없이 60fps 근접
주의 환기충돌, 위험 변경, 새 오류 위치를 짧게 강조계속 흔들리거나 색과 모션만으로 의미 전달반복 제한, reduced motion 대응
브랜드 감각핵심 제품 순간에 제한적으로 사용업무 화면 전체에 장식 모션 남발초기 로드와 입력 반응성 영향 측정

duration·easing 판단

전환권장 감각예검토 포인트
즉각 피드백매우 짧고 반응적pressed, toggle, checkbox사용자 입력이 늦게 접수된 것처럼 보이지 않는가?
소형 UI 전환짧고 명확함popover, tooltip, dropdown빠른 반복 조작을 방해하지 않는가?

장시간 처리시간 포장보다 진행 정보upload, export, AI generationprogress, 단계, 취소, 백그라운드 전환이 있는가?
MOTION TOKEN SETduration.fast: 눌림, 작은 상태 변화
duration.base: popover, panel, small list change

easing.standard: 대부분의 상태 전환

REDUCED MOTION

prefers-reduced-motion은 “애니메이션을 모두 없애라”가 아니라 불필요한 이동, 확대, parallax, 반복 모션을 줄이라는 신호입니다. 상태 변화 자체는 텍스트, 위치, opacity, focus로 계속 전달해야 합니다.

INT-05

Scroll interaction

스크롤 인터랙션은 사용자의 기본 탐색 행위를 빌려 쓰는 기능입니다. 스크롤을 조종하거나 과하게 연출하면 정보 탐색, 접근성, 성능이 동시에 나빠질 수 있습니다. 스크롤 기반 효과는 콘텐츠 이해를 돕는 경우에만 사용합니다.

패턴쓸 때실패 모드안전장치
Sticky header긴 목록에서 현재 맥락과 주요 액션을 유지할 때viewport를 과도하게 차지해 본문을 가림높이 제한, anchor offset, focus 이동 테스트
Scroll spy긴 문서의 현재 섹션을 표시할 때짧은 섹션에서 active 상태가 흔들림IntersectionObserver threshold 조정
Infinite scroll탐색형 feed, 끝이 덜 중요한 콘텐츠footer 접근 불가, 위치 복구 실패, SEO/공유성 약화명시적 pagination 대안, load more, scroll restoration
Virtualized list행 수가 많아 DOM 수가 병목일 때브라우저 find, screen reader, 인쇄, 동적 높이 문제검색·선택·focus·접근성 요구 별도 설계
Parallax스토리텔링 화면에서 공간감을 줄 때멀미, scroll jank, 핵심 정보 지연reduced motion에서 제거, transform 기반 구현

INT-06

3D 인터랙션 판단

3D는 제품 이해를 높일 수도 있지만, 비용이 큽니다. 로딩, GPU, 입력 장치, fallback, 접근성, QA 복잡도가 모두 증가합니다. 3D는 “있으면 멋있다”가 아니라 2D로는 해결하기 어려운 공간 이해나 조작 문제가 있을 때 선택합니다.

사용 후보3D가 유효한 이유2D 대안게이트

INT-09

인터랙션 아키텍처

수준급 인터랙티브 개발자는 “클릭하면 상태를 바꾼다”가 아니라 입력, 상태 전이, 비동기 작업, 취소, 실패 복구, 관측을 하나의 작은 시스템으로 봅니다. 복잡한 UI일수록 컴포넌트 내부 boolean 여러 개보다 명시적인 상태 기계와 이벤트 계약이 안전합니다.

축초급 구현수준급 구현검토 질문
상태 모델isOpen, isLoading, hasError를 따로 둠idle → armed → pending → success/error → recovering처럼 가능한 전이를 제한동시에 참이면 안 되는 상태가 boolean 조합으로 가능하지 않은가?
이벤트onClick마다 즉시 side effect 실행user event, system event, async result, timeout, cancel을 구분실패와 취소가 정상 이벤트로 모델링되어 있는가?
비동기요청 시작 후 버튼만 disabledabort, idempotency key, optimistic rollback, stale response 무시를 함께 설계늦게 도착한 응답이 최신 UI 상태를 덮어쓰지 않는가?
동시성빠른 클릭은 QA에서만 잡음rapid click, double submit, route leave, tab hidden을 설계 입력으로 취급사용자가 animation 중 반대 행동을 하면 최종 상태가 하나로 수렴하는가?
관측클릭 수만 남김start/result/cancel/retry와 duration, error class, input type을 남김사용자가 어디서 실패했고 회복했는지 알 수 있는가?

STATE BUG

인터랙션 버그의 상당수는 CSS 문제가 아니라 불가능해야 할 상태가 가능해서 생깁니다. 예를 들어 modal이 닫혔는데 backdrop은 남고, focus는 내부에 있으며, API 요청은 pending인 상태입니다. 복잡한 인터랙션은 상태 전이표를 먼저 쓰는 편이 코드보다 빠릅니다.

INT-10

usePointerSession 예시function usePointerSession({ onMove, onCommit, onCancel }) {
  const sessionRef = useRef(null);
  const start = (event) => {
    event.currentTarget.setPointerCapture(event.pointerId);
    sessionRef.current = { pointerId: event.pointerId, startX: event.clientX, startY: event.clientY };
  };
  const move = (event) => {
    const session = sessionRef.current;
    if (!session || session.pointerId !== event.pointerId) return;
    onMove({ dx: event.clientX - session.startX, dy: event.clientY - session.startY });
  };
  const end = (event) => {
    event.currentTarget.releasePointerCapture(event.pointerId);
    onCommit(sessionRef.current);
    sessionRef.current = null;
  };
  return { onPointerDown: start, onPointerMove: move, onPointerUp: end, onPointerCancel: onCancel };
}

INT-14

근거 기반 판단 기준

인터랙션 품질은 “부드러워 보인다”가 아니라 사용자가 입력한 뒤 어느 시점에 시스템 반응을 확인하고, 실패했을 때 어떤 부작용이 생기는지로 판단합니다. 아래 숫자는 절대 법칙이 아니라 리뷰와 계측을 시작하기 위한 기준선입니다. 실제 승인 기준은 제품의 핵심 행동별 p75, p95 현장 데이터로 보정해야 합니다.

판단 항목실질 숫자근거와 의미부수효과수집할 데이터
전체 입력 반응성INP ≤ 200ms good, 200-500ms needs improvement, > 500ms poorCore Web Vitals의 INP 기준. click, tap, keyboard interaction이 다음 paint까지 걸린 시간을 본다200ms를 넘기면 사용자는 입력이 누락됐다고 느끼기 시작하고, 500ms 이상은 중복 클릭·이탈·불신으로 이어지기 쉽다route, component, action, device tier별 INP p75/p95와 worst interaction name
초기 로드 이후의 반응성Chrome 사용 데이터 기준 페이지 체류 시간의 약 90%는 load 이후초기 렌더링만 빠른 앱보다 세션 중 입력 반응성이 유지되는 앱이 실제 사용 품질에 더 직접적이다대시보드, 폼, 에디터처럼 오래 머무는 화면에서 “처음만 빠른” 최적화가 품질을 속인다session age, route dwell time, interaction count, long task after load

Drag·resize 세션Pointer capture는 pointerup/release 전까지 이후 pointer event를 지정 element로 보냄drag 중 포인터가 요소 밖으로 나가도 세션이 끊기지 않게 하는 표준 API다release 누락 시 다른 UI가 입력을 받지 못하거나, unmount 뒤 stale session이 남는다pointerdown/up/cancel, release success, unmount cleanup, lost pointer session rate
모션 민감도prefers-reduced-motion: reduce는 비필수 움직임 축소 선호를 뜻함사용자가 OS나 브라우저에서 움직임 최소화를 요청한 상태를 CSS/JS에서 감지할 수 있다상태 전달을 이동·확대·parallax에만 의존하면 일부 사용자에게 불편하거나 접근 불가능한 경험이 된다reduced motion 사용률, alternate path 적용 여부, motion-disabled screenshot

근거 출처는 web.dev의 INP 문서, MDN의 setPointerCapture, prefers-reduced-motion 문서를 기준으로 삼고, 제품별 기준은 반드시 현장 RUM 데이터로 보정합니다.

EVIDENCE PACKET1. 측정 대상: route + component + user action 이름을 먼저 고정한다
2. 숫자: p75와 p95를 device tier별로 나누고, 평균값 단독 보고를 금지한다
3. 부수효과: 중복 클릭, 취소 실패, focus 손실, scroll jump, rage click을 함께 본다
4. 재현 조건: CPU slowdown, mobile viewport, touch input, DPR 2, reduced motion을 명시한다
5. 결정: 유지, 단순화, lazy load, optimistic feedback, fallback 중 무엇을 선택했는지 기록한다
PRACTICAL STANDARD

인터랙션 리뷰에서 가장 강한 근거는 “이 효과가 좋다”가 아니라 “이 행동의 p95 입력 지연이 420ms였고, optimistic feedback을 추가한 뒤 170ms로 내려갔으며, 중복 submit이 38% 줄었다” 같은 문장입니다. 수치와 부수효과가 함께 있어야 디자인, 프론트엔드, 백엔드가 같은 문제를 보고 같은 결정을 내릴 수 있습니다.

INT-15

자주 봉착하는 문제와 해결

인터랙션 문제는 대부분 “이벤트가 안 먹는다”가 아니라 입력 세션, 비동기 상태, focus, scroll, 접근성 경로가 서로 다른 수명으로 움직일 때 생깁니다. 문제를 고칠 때는 시각 효과만 만지지 말고 상태 전이, cleanup, 관측 이벤트를 함께 고쳐야 재발하지 않습니다.

문제대표 증상주요 원인해결법검증
중복 클릭·중복 제출주문, 저장, 배포가 두 번 실행됨pending lock이 UI에만 있고 mutation idempotency와 request key가 없음action 단위 requestId, 서버 idempotency key, optimistic state와 rollback, pending 중 재입력 정책을 명시double click, Enter 연타, slow network에서 mutation 1회만 발생
Drag가 중간에 끊김요소 밖으로 나가면 resize/drag가 멈춤pointer capture 미사용, pointerId 추적 없음, scroll container와 이벤트 충돌pointerdown에서 capture, pointerId별 session 관리, pointercancel과 unmount cleanup, scroll lock 여부 결정요소 밖 pointerup, 터치 입력, route 이동 중 cleanup 테스트
Dialog 닫은 뒤 focus 손실Tab이 문서 처음으로 튀거나 숨은 요소에 머묾trigger ref 보존 실패, exit animation과 focus restore 수명 불일치open 시 trigger 저장, close 완료 후 restore, trigger 제거 시 fallback target 지정Escape, backdrop click, route leave, trigger unmount 경로 확인
Scroll jump·위치 복구 실패필터 변경, 탭 이동, infinite scroll 후 위치가 튐DOM height 변동, async content 삽입, scroll restoration 정책 부재list key 안정화, skeleton 높이 예약, route별 scroll state 저장, anchor 기반 복구뒤로가기, 필터 토글, lazy image load 후 viewport 위치 비교
Optimistic UI가 실제 상태와 어긋남성공처럼 보였는데 새로고침하면 사라짐낙관 업데이트와 서버 응답 병합 규칙, 실패 rollback, stale response ignore가 없음operation log, request version, 실패 시 원래 값 복원, 성공 응답으로 canonical state 재동기화느린 응답, 실패 응답, 순서가 뒤바뀐 응답, retry 테스트
Canvas/drag 기능이 접근 불가능마우스로만 조작 가능하고 키보드·보조기술 경로가 없음시각 객체를 DOM control이나 command model로 노출하지 않음object list, keyboard commands, live announcement, hidden이 아닌 대체 control 제공마우스 없이 선택·이동·삭제·취소 완료
INTERACTION DEBUG ORDER1. 문제가 발생한 user action 이름과 입력 장치를 고정한다
2. idle, pending, success, error, cancelled 중 어느 상태에서 깨졌는지 찾는다
3. event lifecycle(pointerdown/move/up/cancel, keydown, blur)을 로그로 확인한다
4. async response가 늦게 오거나 순서가 바뀌는 경우를 재현한다
5. focus, scroll, announcement, cleanup까지 같은 테스트에 넣는다
ANTI-PATCH

인터랙션 버그를 setTimeout, 전역 flag, CSS pointer-events: none만으로 막으면 다음 입력 장치나 느린 네트워크에서 다시 터집니다. 해결 단위는 “한 이벤트 핸들러”가 아니라 “입력 세션과 상태 전이 전체”여야 합니다.
