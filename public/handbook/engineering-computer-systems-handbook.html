컴퓨터 시스템: 코드 아래에서 일어나는 일

시스템 지식은 운영 사고를 줄이는 도구입니다. CPU 캐시, 가상 메모리, 스케줄러, 파일 디스크립터, 소켓, resource limit을 알면 "왜 로컬에서는 빠른데 운영에서는 멈추는가"를 더 빨리 좁힐 수 있습니다.

AUDIENCE : 애플리케이션 엔지니어 · Linux/Cloud runtime 기준 · UPDATED 2026-06
SYS-01

COMPUTER SYSTEMS SPINE

애플리케이션 요청은 CPU 명령, 메모리 접근, 커널 syscall, 파일/소켓 I/O, 스케줄링 대기, 네트워크 왕복으로 분해됩니다. 시스템을 이해한다는 것은 이 비용의 위치를 구분하는 능력입니다.

COMPUTER SYSTEMS SPINErequest → user code → CPU/cache → heap/stack/virtual memory
→ syscall boundary → file/socket descriptor → kernel buffer
→ scheduler → NIC/disk → response

SYS-02

CPU CACHE LOCALITY

CPU는 메모리보다 훨씬 빠릅니다. 그래서 성능은 종종 연산 횟수보다 데이터를 얼마나 캐시에 잘 올려 쓰는지로 갈립니다.

개념의미개발 영향
Temporal locality방금 쓴 데이터를 곧 다시 씀작은 working set, hot data cache
Spatial locality가까운 주소의 데이터를 함께 씀배열 순회가 포인터 구조보다 유리
Cache lineCPU가 메모리를 줄 단위로 가져옴연속 배열, false sharing 주의
Branch prediction분기 방향 예측데이터 분포가 random이면 비용 증가
CPU CACHE LOCALITYbad: linked nodes → pointer chase → cache miss per node
good: packed array → sequential scan → prefetch friendly
risk: multiple threads update nearby counters → false sharing
실무 감각

Hash map이 이론상 O(1)이어도 큰 메모리와 랜덤 접근을 만들면 캐시 미스로 느려질 수 있습니다. 반대로 작은 배열 선형 탐색은 O(n)이어도 n이 작고 연속 메모리라 더 빠를 수 있습니다.

SYS-03

메모리와 가상 메모리

프로세스는 자신만의 연속된 주소 공간을 가진 것처럼 보지만 실제 물리 메모리는 OS가 page 단위로 매핑합니다. 이 추상화가 보호와 격리를 만들고, 동시에 page fault와 OOM의 원인이 됩니다.

영역담는 것장애 신호
Stack함수 호출 프레임, 지역 변수깊은 재귀로 stack overflow
Heap동적 할당 객체GC pressure, memory leak
Mapped regionshared library, mmap file파일-backed page fault
Kernel memorysocket/file buffer, page cachecontainer memory와 함께 압박
OOM 해석

컨테이너에서 OOM은 애플리케이션 heap만의 문제가 아닐 수 있습니다. native memory, thread stack, direct buffer, mmap, page cache까지 cgroup limit에 포함될 수 있으므로 런타임 heap limit과 컨테이너 memory limit을 함께 맞춰야 합니다.

  • heap max, container memory, thread count, direct memory 설정을 함께 기록한다.
  • major page fault와 swap 사용 여부를 확인한다.
  • 대용량 파일은 전체 로드보다 streaming 또는 mmap 비용을 비교한다.
SYS-04

PROCESS THREAD SCHEDULING

프로세스는 자원 격리 단위이고, 스레드는 실행 스케줄링 단위입니다. 동시성 문제는 CPU를 더 쓰는 문제가 아니라 대기, 경합, context switch를 관리하는 문제입니다.

개념역할실무 판단
Process주소 공간과 자원 소유장애 격리, 배포 단위
Thread같은 주소 공간에서 실행공유 상태 lock과 stack memory 비용
Context switch실행 대상을 바꾸는 비용스레드 과다 생성 시 throughput 하락
SchedulerCPU 시간을 나눠줌run queue, priority, cgroup quota 영향
Lock contention공유 자원 대기CPU가 남아도 latency가 증가
PROCESS THREAD SCHEDULINGrunnable threads > available cores → run queue grows
blocked threads wait on I/O/lock → throughput flat, latency rises
CPU quota throttling → app appears slow even with low host CPU
스레드 풀 기준

CPU-bound 작업은 코어 수 근처, I/O-bound 작업은 대기 시간이 길수록 더 많은 스레드가 필요할 수 있습니다. 하지만 무작정 늘리면 stack memory와 context switch가 늘고 하류 DB/API를 압박합니다. 스레드 풀 크기는 하류 커넥션 풀, timeout, queue limit과 함께 정합니다.

SYS-05

FILE SOCKET SYSCALL TRACE

파일과 소켓은 애플리케이션에서 객체처럼 보이지만 커널에서는 file descriptor입니다. 읽기, 쓰기, 연결, accept는 user mode에서 kernel mode로 넘어가는 syscall입니다.

FILE SOCKET SYSCALL TRACEopen/read/write/close for files
socket/bind/listen/accept/read/write/close for servers
connect/send/recv/close for clients
epoll_wait/kqueue for readiness based I/O
syscall하는 일문제 신호
open파일을 fd로 연결권한 오류, fd 누수
read/write커널 버퍼와 데이터 이동partial read/write, blocking
accept새 TCP 연결 수락backlog overflow
connect원격 주소 연결DNS, SYN timeout, refused
epoll_wait준비된 fd 대기event loop block, readiness starvation
fd는 자원이다

파일, socket, pipe는 모두 fd limit을 소비합니다. HTTP client connection leak, 로그 파일 핸들 누수, 너무 많은 동시 업로드는 같은 에러인 EMFILE로 나타날 수 있습니다.

SYS-06

RESOURCE LIMIT PLAYBOOK

운영 장애는 무제한 큐, 무제한 스레드, 무제한 파일, 무제한 메모리에서 시작합니다. limit은 성능 저하가 아니라 장애 반경을 줄이는 안전장치입니다.

자원확인값장애 증상대응

SYS-07

네트워크 기본

네트워크 문제는 애플리케이션 에러처럼 보입니다. DNS, TCP handshake, TLS, HTTP, load balancer, socket pool 중 어느 단계인지 나눠야 합니다.

단계확인 질문대표 오류
DNS이름이 어떤 IP로 풀리는가NXDOMAIN, stale record
TCP3-way handshake가 완료되는가timeout, refused, reset
TLS인증서와 프로토콜이 맞는가expired cert, SNI mismatch
HTTP상태코드와 헤더가 맞는가4xx/5xx, bad gateway
Pooling연결 재사용과 idle timeout이 맞는가stale connection, pool starvation
진단 순서

connect timeout은 서버 코드 실행 전 문제일 수 있고, read timeout은 서버가 요청을 받은 뒤 응답이 늦은 문제일 수 있습니다. timeout을 하나로 뭉개지 말고 DNS, connect, TLS, request, response timeout을 분리해서 로깅합니다.
