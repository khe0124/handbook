JavaScript·TypeScript 면접

언어 면접은 문법 암기가 아니라 런타임이 코드를 어떻게 실행하는지, 타입 시스템으로 어떤 버그를 줄이는지를 설명하는 자리입니다.

JT-01

Q. execution context와 lexical environment를 설명해보세요.

핵심 개념

execution context는 JavaScript 코드가 실행될 때 필요한 환경입니다. 변수 환경, lexical environment, outer reference, this binding이 포함됩니다. lexical environment는 식별자와 값의 매핑, 그리고 바깥 스코프 참조를 가진 구조입니다.

개념 해부

execution context는 코드가 실행되는 순간의 작업대이고, lexical environment는 그 작업대에서 식별자를 찾는 사전입니다. 함수 호출이 call stack에 쌓이면 각 호출은 자기 environment를 가지며, 식별자를 찾을 때 현재 환경에서 바깥 환경으로 올라갑니다. hoisting은 선언이 이 환경에 등록되는 방식의 결과이고, TDZ는 등록은 되었지만 초기화 전 접근을 막는 구간입니다.

30초 답변

JavaScript는 함수가 호출될 때 execution context를 만들고 call stack에 쌓습니다. 이 안의 lexical environment가 현재 스코프의 변수와 바깥 스코프 참조를 들고 있어서 scope chain과 closure가 동작합니다.

90초 답변

면접에서 중요한 점은 “var/let/const가 왜 다르게 보이는가”, “함수 내부에서 바깥 변수를 어떻게 기억하는가”입니다. `let`과 `const`는 lexical environment에 바인딩되지만 초기화 전 TDZ에 있으므로 접근하면 에러가 납니다. closure는 함수가 생성될 때 외부 lexical environment를 참조할 수 있게 유지되는 현상입니다.

TDZ는?let/const 선언 전 접근이 허용되지 않는 구간입니다.

JT-02

Q. closure는 무엇이고 언제 문제가 되나요?

핵심 개념

closure는 함수가 선언된 lexical environment를 기억해, 외부 함수 실행이 끝난 뒤에도 그 변수에 접근할 수 있는 구조입니다.

개념 해부

closure는 함수가 값 자체를 복사해 들고 있는 것이 아니라 자신이 만들어진 lexical environment에 대한 참조를 유지하는 구조입니다. 그래서 외부 함수 호출이 끝나도 내부 함수가 해당 변수를 읽을 수 있습니다. React의 stale closure는 렌더 시점의 값이 callback에 남아 최신 상태와 어긋나는 문제이므로 dependency, functional update, ref 중 어떤 모델이 맞는지 판단해야 합니다.

30초 답변

closure는 내부 함수가 외부 스코프 변수를 계속 참조할 수 있는 현상입니다. 상태 은닉, callback, React hook에서 자주 쓰이지만 오래된 값을 캡처하는 stale closure나 불필요한 메모리 유지 문제가 생길 수 있습니다.

90초 답변

React에서 stale closure는 흔한 꼬리질문입니다. effect나 callback이 렌더 당시 값을 캡처했는데 dependency가 빠져 최신 값을 보지 못하는 경우입니다. 해결은 dependency를 정확히 두거나 functional update, ref, state 구조 조정으로 의도를 명확히 하는 것입니다.

꼬리질문

질문답변 방향
메모리 누수와 관계?closure가 큰 객체를 계속 참조하면 GC가 회수하지 못할 수 있습니다.
for loop 문제는?var는 함수 스코프라 같은 바인딩을 공유하고 let은 블록 스코프로 매 반복 새 바인딩을 만듭니다.
React에서?event handler와 effect가 렌더 시점의 값을 캡처한다는 점을 말합니다.
JT-03

Q. prototype, class, this, arrow function을 어떻게 설명하나요?

핵심 개념

JavaScript 객체는 prototype chain을 통해 속성을 찾습니다. `class`는 prototype 기반 상속을 더 익숙한 문법으로 감싼 형태입니다. `this`는 호출 방식에 따라 결정되고 arrow function은 lexical this를 가집니다.

개념 해부

prototype chain은 객체의 속성 조회 규칙입니다. 객체 자신에게 property가 없으면 prototype을 따라 올라가며, class 문법도 이 모델 위에 method를 prototype에 배치하는 문법적 장치입니다. this는 선언 위치가 아니라 호출 형태로 결정되고, arrow function은 자기 this를 만들지 않아 바깥 lexical this를 사용합니다.

30초 답변

객체에서 속성을 찾을 때 자기 자신에 없으면 prototype chain을 따라 올라갑니다. class method는 prototype에 올라가고, arrow function은 자기 this를 만들지 않아 선언된 바깥 this를 사용합니다.

90초 답변

`this`는 함수가 어디서 선언됐는지가 아니라 어떻게 호출됐는지에 따라 달라집니다. `obj.method()`면 `obj`, 그냥 함수로 호출되면 strict mode에서는 undefined가 됩니다. arrow function은 호출 방식으로 this가 바뀌지 않기 때문에 callback에서 유용하지만 method로 남용하면 prototype 공유 이점이 줄 수 있습니다.

꼬리질문

질문답변 방향
class는 hoisting?class 선언도 lexical binding이지만 TDZ가 있어 선언 전 접근할 수 없습니다.
bind/call/apply?this를 명시적으로 지정해 호출하는 방법입니다.
prototype 오염?외부 입력으로 prototype이 변조되면 보안 문제가 됩니다.
JT-04

Q. event loop, microtask, macrotask를 설명해보세요.

핵심 개념

JavaScript는 call stack이 비면 event loop가 task queue에서 작업을 가져옵니다. Promise callback은 microtask queue에 들어가고, timer나 IO callback은 macrotask 계열로 다뤄집니다.

개념 해부

event loop는 비동기 작업을 병렬로 실행하는 마법이 아니라 call stack이 빈 시점에 예약된 callback을 실행하는 조정 장치입니다. microtask는 현재 task가 끝난 뒤 렌더링이나 다음 task로 넘어가기 전에 비워지는 경향이 있어 Promise callback이 timer보다 먼저 관찰될 수 있습니다. 긴 동기 JavaScript는 queue 순서와 무관하게 입력 처리와 렌더링 자체를 막습니다.

30초 답변

동기 코드는 call stack에서 먼저 실행되고, Promise의 then/await 이후 작업은 microtask로 예약됩니다. microtask는 보통 다음 macrotask보다 먼저 비워지므로 timer보다 Promise callback이 먼저 실행되는 경우가 많습니다.

90초 답변

실무에서는 long task가 event loop를 막아 클릭 반응과 렌더링을 지연시키는 문제가 중요합니다. 무거운 계산은 chunking, Web Worker, debounce, requestIdleCallback 같은 전략으로 나눌 수 있습니다. Node.js와 브라우저 event loop는 세부 phase가 다르므로 환경도 구분해야 합니다.

꼬리질문

질문답변 방향
async/await는?await 이후 코드는 Promise continuation으로 microtask에 예약됩니다.
rendering과 관계?긴 JS task가 브라우저 렌더링과 입력 처리를 막을 수 있습니다.
queueMicrotask?명시적으로 microtask를 예약하는 API입니다.
JT-05

Q. Promise와 async/await 오류 처리는 어떻게 하나요?

핵심 개념

Promise는 비동기 작업의 eventual result를 표현합니다. async 함수는 항상 Promise를 반환하고, throw는 rejected Promise가 됩니다.

개념 해부

Promise는 지금 값이 아니라 나중에 성공 또는 실패로 확정될 결과를 표현합니다. async 함수의 반환값은 항상 Promise로 감싸지고, 함수 안의 throw는 rejected Promise가 됩니다. 순차 await는 읽기 쉽지만 병렬 가능한 작업까지 직렬화할 수 있으므로 의존 관계가 있는지 먼저 봐야 합니다.

30초 답변

async/await는 Promise를 동기 코드처럼 읽게 해주지만 오류는 반드시 try/catch나 상위 caller에서 처리해야 합니다. 병렬 가능한 작업은 순차 await하지 않고 `Promise.all`을 쓰되, 하나 실패하면 전체가 reject되는 성질도 고려합니다.

90초 답변

`Promise.allSettled`는 일부 실패를 수집해야 할 때 유용하고, `Promise.race`나 `AbortController`는 timeout/cancel 흐름에서 씁니다. 실무에서는 unhandled rejection, 중복 요청, 오래된 응답이 최신 상태를 덮는 문제가 자주 생기므로 요청 id나 abort, latest-only guard를 둡니다.

꼬리질문

질문답변 방향
Promise.all 실패?하나라도 reject되면 전체가 reject됩니다.
finally는?성공/실패와 무관하게 정리 로직에 쓰지만 반환/throw에 주의합니다.
에러 경계?비동기 에러는 React Error Boundary가 자동으로 잡지 못하는 경우가 많습니다.
JT-06

Q. TypeScript structural typing과 unknown vs any를 설명해보세요.

핵심 개념

TypeScript는 이름이 아니라 구조가 맞으면 호환되는 structural typing을 사용합니다. `any`는 타입 검사를 끄고, `unknown`은 사용 전에 타입 좁히기를 요구합니다.

개념 해부

TypeScript의 타입은 런타임 객체에 붙는 태그가 아니라 컴파일 타임에 값의 사용 가능성을 검증하는 모델입니다. structural typing은 이름이 달라도 필요한 구조가 맞으면 호환된다는 뜻이라 유연하지만 의도치 않은 호환도 생길 수 있습니다. unknown은 외부 입력의 신뢰 경계를 타입으로 표현하고, any는 그 경계를 컴파일러가 더 이상 확인하지 못하게 합니다.

30초 답변

TypeScript는 구조적 타입 시스템이라 필요한 property 구조가 맞으면 호환됩니다. `any`는 타입 안전성을 포기하는 탈출구이고, `unknown`은 어떤 값인지 모른다는 뜻이라 사용 전에 type guard로 좁혀야 합니다.

90초 답변

API 응답, 외부 JSON, catch error처럼 신뢰할 수 없는 값은 `unknown`으로 받고 runtime validation이나 type guard로 좁히는 편이 안전합니다. `any`를 쓰면 잘못된 property 접근이 컴파일에서 드러나지 않아 타입 시스템의 이점을 잃습니다. `never`는 도달 불가능한 상태나 exhaustive check에 사용됩니다.

꼬리질문

질문답변 방향
structural typing 단점?의도와 달리 구조만 맞아 호환될 수 있어 branded type이 필요할 수 있습니다.
type assertion은?컴파일러에게 믿으라고 말하는 것이지 runtime 검증이 아닙니다.
never는?불가능한 타입으로 exhaustive switch 검증에 유용합니다.
JT-07

Q. generic, union/intersection, type guard, conditional type은 언제 쓰나요?

핵심 개념

generic은 타입을 매개변수화해 재사용하면서 정보 손실을 줄입니다. union은 여러 가능성 중 하나, intersection은 여러 타입의 결합입니다. conditional type은 타입 수준 조건 분기입니다.

개념 해부

generic은 타입을 추상화하는 장치가 아니라 입력과 출력 사이의 타입 관계를 보존하는 장치입니다. union은 가능한 상태를 닫힌 집합으로 표현하고, discriminant를 두면 분기마다 안전하게 좁힐 수 있습니다. conditional type과 infer는 라이브러리 수준 타입 계산에 강하지만, 팀이 읽을 수 없는 타입은 유지보수 비용이 됩니다.

30초 답변

generic은 입력과 출력의 타입 관계를 보존할 때 씁니다. union은 상태나 응답처럼 여러 케이스가 있을 때 쓰고, type guard로 안전하게 좁힙니다. conditional type은 라이브러리나 공통 타입 유틸에서 타입 관계를 계산할 때 유용합니다.

90초 답변

예를 들어 API result를 `{ status: "success"; data: T } | { status: "error"; error: ApiError }` 같은 discriminated union으로 만들면 UI가 성공/실패 상태를 안전하게 처리할 수 있습니다. intersection은 합성에 유용하지만 충돌 property가 생기면 의도가 흐려질 수 있습니다. conditional type은 강력하지만 과하면 팀 이해 비용이 커집니다.

꼬리질문

질문답변 방향
generic 남용?타입 관계가 없는데 generic을 쓰면 읽기만 어려워집니다.
type guard 기준?runtime에서 실제로 확인 가능한 property나 predicate를 사용합니다.
infer는?conditional type 안에서 타입 일부를 추론할 때 씁니다.

JT-10

TypeScript 타입 안전 경계

TypeScript 면접의 핵심은 타입 문법을 많이 아는 것이 아니라, 컴파일 타임 타입과 런타임 데이터의 경계를 정확히 구분하는 것입니다.

TS TYPE SAFETY BOUNDARY TypeScript 타입은 빌드 후 사라지므로 외부 API 응답, localStorage, URL query, form input, postMessage, feature flag, 서버 에러 payload는 런타임 검증이 필요합니다. 타입 선언은 “그럴 것”이라는 약속이고, validation은 “실제로 그렇다”는 확인입니다.
경계위험전문적인 답변
API responsenullable, enum 추가, field renameunknown으로 받고 schema validation 또는 adapter에서 view model로 변환합니다.
form input문자열 기반 입력, 숫자/날짜 파싱 오류입력 타입, 파서, domain validator를 분리하고 오류 path를 유지합니다.
URL query없는 값, 배열/단일 값 혼재, 잘못된 enumdecode layer에서 기본값과 허용값을 명시합니다.
localStorage오래된 schema, 사용자가 수정한 값versioning, migration, validation 실패 시 fallback을 둡니다.
library type너무 넓은 type, 잘못된 assertionwrapper와 branded type으로 도메인 불변식을 보강합니다.

면접에서 말할 증거

API adapter, discriminated union 상태 모델, exhaustive switch, type guard, runtime schema, unknown 사용 기준, any 격리 위치를 말합니다.

흔한 오답

“TypeScript라 런타임 오류가 줄어듭니다”에서 멈추면 얕습니다. TypeScript는 신뢰할 수 없는 입력을 자동 검증하지 않으므로, 외부 경계마다 validation과 fallback을 설계해야 합니다.

JT-D1

TYPESCRIPT ADVANCED BOUNDARY

고급 TypeScript 면접은 타입 퍼즐을 푸는 자리가 아니라, 컴파일러가 보장하는 영역과 런타임 검증이 필요한 영역을 정확히 나누는지 확인하는 자리입니다.

30초 satisfies는 값의 구체 타입을 유지하면서 계약을 검사하고, as const는 literal과 readonly 추론을 고정합니다. excess property check는 객체 literal에서 오타를 잡아주지만 변수에 담기는 순간 약해질 수 있고, variance와 branded type은 구조적 타입 시스템의 의도치 않은 호환을 줄이는 도구입니다. strict flags와 runtime schema validation은 외부 입력 경계에서 타입 안전성을 실제 안정성으로 연결합니다.
키워드전문적인 답변실무 경계
satisfies객체가 특정 계약을 만족하는지 검사하되, 값 자체의 literal 타입 정보는 잃지 않습니다.설정 객체, route map, i18n message map에서 키 오타와 값 추론을 함께 잡습니다.
as constliteral widening을 막고 readonly tuple/object로 추론하게 합니다.상수 enum 대체, discriminated union source, 허용값 목록에 유용합니다.
excess property check객체 literal을 직접 대입할 때 여분 속성을 잡지만, 중간 변수나 assertion으로 우회될 수 있습니다.API payload builder와 config 생성 함수에서 오타를 잡는 보조 장치로 봅니다.
variance타입 매개변수가 입력/출력 위치에서 어떻게 호환되는지 설명합니다.callback, event handler, generic container, readonly collection API 설계에서 중요합니다.
branded type구조가 같은 string/number라도 도메인 의미가 다르면 브랜드를 붙여 섞이지 않게 합니다.UserId, OrderId, ISODate, SanitizedHtml처럼 값의 출처나 검증 상태를 표현합니다.
strict flagsstrictNullChecks, noUncheckedIndexedAccess, exactOptionalPropertyTypes 같은 옵션으로 암묵적 빈틈을 줄입니다.마이그레이션은 한 번에 켜기보다 hot path와 adapter부터 단계적으로 좁힙니다.
runtime schema validationTypeScript 타입은 런타임에 사라지므로 외부 JSON은 Zod, Valibot, io-ts 같은 schema 또는 직접 validator로 확인합니다.API, URL query, localStorage, postMessage, feature flag, server error payload가 대상입니다.

구체 질문

질문답변 방향
satisfies와 type annotation은 설정 객체의 추론 결과에서 무엇이 달라지나요?계약 검사는 하되 literal key/value 정보가 유지되는 점을 예로 듭니다.
as const를 남용하면 어떤 유지보수 문제가 생기나요?readonly 확산, 지나치게 좁은 타입, 불필요한 assertion을 구분합니다.
excess property check가 중간 변수에서 약해지는 이유를 어떻게 설명하나요?객체 literal freshness와 structural typing의 차이를 말합니다.
함수 parameter 타입에서 variance가 왜 callback 안정성과 연결되나요?더 좁은 입력만 처리하는 함수를 넓은 자리로 넘길 때의 위험을 설명합니다.
branded type은 런타임 검증을 대체하나요?대체하지 않으며 검증 후 값에 타입 의미를 부여하는 방식이라고 답합니다.
noUncheckedIndexedAccess를 켜면 어떤 코드가 드러나고 어떻게 고치나요?배열/record 접근 결과가 undefined일 수 있음을 처리합니다.
API 응답 타입을 생성했는데도 runtime schema validation이 필요한 이유는 무엇인가요?서버 배포 차이, nullable, enum 추가, 악성/오염 데이터 가능성을 듭니다.
strict flag를 기존 프로젝트에 도입할 때 어떤 순서로 위험을 줄이나요?adapter, shared utility, domain model부터 좁히고 any 격리 목록을 관리합니다.
외부 입력을 unknown으로 받은 뒤 view model로 변환하는 adapter의 장점은 무엇인가요?검증, 기본값, 오류 path, UI 타입 분리를 한 곳에 모을 수 있습니다.
타입 assertion이 꼭 필요한 경우와 위험한 경우를 어떻게 구분하나요?런타임 근거가 있는 좁히기인지, 컴파일러 경고를 숨기는지로 구분합니다.
흔한 오답

“TypeScript가 있으니 API 응답은 안전합니다”라고 말하면 위험합니다. 타입은 신뢰 경계 안쪽의 개발자 계약이고, 네트워크·저장소·브라우저 메시지 경계에서는 runtime schema validation으로 실제 값을 확인해야 합니다.
