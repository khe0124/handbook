DOCKER-01

Spring Boot Dockerfile

언제·왜: jar 하나를 이미지로 굽는 가장 표준적인 방법. 멀티스테이지로 빌드 도구(Gradle·JDK)는 빌드 스테이지에만 두고, 런타임 이미지엔 JRE와 jar만 남깁니다. 이미지가 수백 MB 작아지고 공격 표면도 줄어듭니다.

Dockerfile — 멀티스테이지 (build → runtime) # ---- build stage ----
FROM gradle:8.7-jdk21 AS build
WORKDIR /app
# 의존성 먼저 복사 → 레이어 캐시 (DOCKER-02 참조)
COPY build.gradle settings.gradle ./
COPY gradle ./gradle
RUN gradle dependencies --no-daemon || true
COPY src ./src
RUN gradle bootJar --no-daemon

# ---- runtime stage ----
FROM eclipse-temurin:21-jre
WORKDIR /app
# 비root 사용자 생성 후 전환
RUN useradd --system --no-create-home spring
USER spring
# build 스테이지에서 jar만 가져옴
COPY --from=build /app/build/libs/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
키워드이 Dockerfile에서의 역할
FROM ... AS build빌드 전용 스테이지에 이름 부여. 런타임에서 --from=build로 참조
COPY --from=buildbuild 스테이지의 산출물(jar)만 런타임 이미지로 복사 — Gradle·소스는 안 따라옴
USER spring이후 명령·컨테이너 프로세스를 비root로 실행. 권한 최소화
ENTRYPOINT컨테이너의 고정 실행 명령. CMD와 달리 인자로 잘 안 덮임 — 메인 프로세스에 적합
ENTRYPOINT vs CMD

ENTRYPOINT는 "이 컨테이너는 무엇인가"(여기선 자바 앱), CMD는 "기본 인자"입니다. exec 형식(["java","-jar",...], 대괄호+JSON)을 쓰세요 — shell 형식(java -jar ...)은 PID 1이 셸이 되어 SIGTERM이 자바 프로세스에 전달되지 않아 graceful shutdown이 깨집니다.

DOCKER-02

.dockerignore와 이미지 최적화

언제·왜: docker build는 먼저 빌드 컨텍스트(현재 디렉터리 전체)를 데몬으로 전송합니다. .gradle·build·node_modules가 딸려가면 느리고, 캐시도 무의미하게 깨집니다. 컨텍스트를 줄이고, 잘 안 바뀌는 것부터 COPY가 캐시의 핵심.

.dockerignore — Dockerfile과 같은 위치 .git
.gradle
build
*.md
.env
.idea
node_modules
Dockerfile
compose.yaml

레이어 캐시는 위에서부터 한 줄씩 검증됩니다. 어떤 줄의 입력이 바뀌면 그 줄부터 아래 레이어가 전부 다시 빌드됩니다. 그래서 거의 안 바뀌는 의존성 정의를 먼저 COPY·다운로드하고, 자주 바뀌는 소스는 맨 나중에 복사합니다 (DOCKER-01의 순서가 그 이유). 소스만 고친 빌드에서 의존성 다운로드 단계가 캐시로 건너뛰어집니다.

캐시 친화적 순서 (요지) COPY build.gradle settings.gradle ./ # 1. 의존성 정의 (드물게 변경)
RUN gradle dependencies --no-daemon # 2. 의존성 받기 → 캐시됨
COPY src ./src # 3. 소스 (자주 변경)
RUN gradle bootJar --no-daemon # 4. 빌드
실전 팁

이미지 크기를 더 줄이려면 런타임 베이스를 eclipse-temurin:21-jre처럼 JRE 변형으로(JDK 아님), 또는 -alpine/-jammy 슬림 태그로. 단 alpine은 glibc가 아니라 musl이라 일부 네이티브 라이브러리에서 문제가 날 수 있으니, 검증 없이 alpine으로 바꾸지 마세요.

DOCKER-03

compose — 이 스택 한 파일

언제·왜: app(Spring Boot) + postgres + redis + nginx를 한 명령으로 띄우고 내립니다. 아래가 네 서비스를 다 담은 compose.yaml 골격 — 이후 챕터(환경변수·볼륨·네트워크·헬스체크)가 이 파일을 한 조각씩 채웁니다.

compose.yaml — 전체 골격 services:
  app:
    build: .
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/carbon
      SPRING_DATASOURCE_USERNAME: carbon
      SPRING_DATASOURCE_PASSWORD: ${POSTGRES_PASSWORD}
      SPRING_DATA_REDIS_HOST: redis
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy

  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: carbon
      POSTGRES_USER: carbon
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U carbon"]
      interval: 10s
      retries: 5

  redis:
    image: redis:7
    command: redis-server --appendonly yes
    volumes:
      - redisdata:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      retries: 5

  nginx:
    image: nginx:1.27
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - app

volumes:
  pgdata:
  redisdata:
포트 노출 범위

외부로 여는 포트는 nginx의 80:80 하나뿐입니다. app·postgres·redis는 ports를 두지 않아 호스트로 노출되지 않고, compose 내부 네트워크에서만 서로 통신합니다(DOCKER-06). DB 포트를 호스트로 굳이 열지 마세요 — 로컬 디버깅이 꼭 필요할 때만 "5432:5432"를 한시적으로 추가하세요.

DOCKER-04

환경변수·설정 주입

언제·왜: 같은 이미지를 환경(local·staging·prod)별로 다르게 띄우려면 설정을 코드가 아니라 환경변수로 주입해야 합니다. Spring Boot는 SPRING_ 접두 환경변수를 properties로 자동 매핑합니다.

환경변수매핑되는 Spring 설정 / 용도
SPRING_PROFILES_ACTIVE활성 프로파일 (local/prod). application-prod.yml 선택
SPRING_DATASOURCE_URLJDBC URL — jdbc:postgresql://postgres:5432/carbon
SPRING_DATASOURCE_USERNAMEDB 사용자
SPRING_DATASOURCE_PASSWORDDB 비밀번호 — .env·secret으로 주입
SPRING_DATA_REDIS_HOSTRedis 호스트 — 서비스명 redis

compose는 같은 디렉터리의 .env 파일을 자동으로 읽어 ${VAR} 치환에 씁니다. 비밀값은 yaml에 직접 박지 말고 .env로 분리하고, .env는 git에서 제외하세요(.gitignore·.dockerignore 둘 다).

.env — compose.yaml과 같은 위치 (git 제외) POSTGRES_PASSWORD=change-me-in-secret-manager
SPRING_PROFILES_ACTIVE=prod
app 서비스 — .env 값 참조 + 파일째 주입   app:
    build: .
    env_file:
      - .env
    environment:
      SPRING_PROFILES_ACTIVE: ${SPRING_PROFILES_ACTIVE}
      SPRING_DATASOURCE_PASSWORD: ${POSTGRES_PASSWORD}
secret을 이미지에 굽지 마라

비밀번호·토큰을 Dockerfile의 ENV나 빌드 인자로 넣으면 이미지 레이어에 영구히 남아 docker history로 노출됩니다. 비밀은 런타임에 환경변수·secret으로 주입하고, 운영에서는 .env가 아니라 secret manager를 쓰세요. .env는 반드시 .dockerignore·.gitignore에 둘 것.

DOCKER-05

데이터 영속 — 볼륨

언제·왜: 컨테이너는 지워지면 안의 데이터도 사라집니다. DB 데이터가 컨테이너 수명과 함께 증발하면 안 되니, named volume에 데이터를 두어 컨테이너를 재생성해도 살아남게 합니다.

postgres·redis 볼륨 매핑   postgres:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data # PG 데이터 디렉터리

  redis:
    image: redis:7
    command: redis-server --appendonly yes # AOF 영속 켜기
    volumes:
      - redisdata:/data # AOF/RDB 저장 위치

# 파일 맨 아래 — named volume 선언
volumes:
  pgdata:
  redisdata:
구분형태 / 설명
named volumepgdata:/var/lib/postgresql/data — Docker가 관리. DB 데이터에 권장
bind mount./nginx.conf:/etc/...:ro — 호스트 파일 직결. 설정 파일·소스에 적합. :ro는 읽기전용
Redis AOF--appendonly yes — 모든 쓰기를 append-only 로그로 기록해 재기동 시 복원
데이터를 지우는 명령 구분

docker compose down은 컨테이너·네트워크만 지우고 named volume은 남깁니다. docker compose down -v는 볼륨까지 삭제 — DB 데이터가 통째로 날아갑니다. 캐시(redis)는 날려도 무방하지만, postgres 볼륨에 -v를 거는 건 운영에서 사고의 지름길입니다. 캐시 용도면 redis는 영속을 끄는 선택도 있습니다.

DOCKER-06

서비스 간 네트워크

언제·왜: compose는 프로젝트마다 기본 네트워크를 만들고, 모든 서비스를 거기에 붙입니다. 같은 네트워크 안에서는 서비스명이 곧 호스트명(내부 DNS)이라, localhost가 아니라 서비스명으로 서로를 찾습니다.

출발 → 도착연결 주소 (서비스명 = 호스트)
app → postgresjdbc:postgresql://postgres:5432/carbon
app → redishost=redis, port=6379
nginx → appproxy_pass http://app:8080;
호스트(브라우저) → nginxhttp://localhost (nginx의 80:80만 외부 노출)
nginx.conf — app으로 리버스 프록시 server {
  listen 80;
  location / {
    # 서비스명 app = compose 내부 DNS
    proxy_pass http://app:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }
}
localhost 함정

컨테이너 안에서 localhost는 그 컨테이너 자신이지 호스트나 다른 서비스가 아닙니다. app의 DB URL을 jdbc:postgresql://localhost:5432로 두면 app 컨테이너 안에서 자기 자신을 찾다가 실패합니다 — 반드시 서비스명 postgres를 쓰세요. 별도 networks: 선언 없이도 같은 compose 파일의 서비스끼리는 기본 네트워크로 자동 연결됩니다.

DOCKER-07

헬스체크·기동 순서

언제·왜: depends_on만으로는 "postgres 컨테이너가 시작됐다"까지만 보장하지, "DB가 접속 받을 준비가 됐다"는 보장하지 않습니다. app이 DB보다 먼저 붙어 죽는 경쟁을 막으려면 healthcheck + condition: service_healthy가 필요합니다.

컨테이너 start→healthcheck 통과→service_healthy→app 기동
healthcheck + depends_on (조건부)   postgres:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U carbon"]
      interval: 10s # 검사 주기
      timeout: 5s # 한 번 검사 제한시간
      retries: 5 # 실패 N회면 unhealthy
      start_period: 30s # 초기 유예 (실패로 안 셈)

  app:
    build: .
    depends_on:
      postgres:
        condition: service_healthy # healthy 될 때까지 대기
      redis:
        condition: service_healthy
condition 값의미
service_started컨테이너가 시작되기만 하면 (기본값, 약한 보장)
service_healthyhealthcheck가 통과해 healthy 상태일 때까지 대기
service_completed_successfully해당 서비스가 정상 종료(exit 0)했을 때 — 마이그레이션 잡 등
실전 팁

그래도 앱은 DB 연결 실패에 재시도로 대응하도록 짜는 게 안전합니다 — healthcheck는 경쟁을 줄여줄 뿐 모든 일시 장애를 막진 못합니다. pg_isready는 postgres 이미지에 기본 포함, redis-cli ping은 redis 이미지에 기본 포함이라 추가 설치가 필요 없습니다.

DOCKER-08

자주 쓰는 명령

언제·왜: 일상의 띄우기·내리기·로그·디버깅·정리. compose 명령과 단일 컨테이너 명령을 나눠 손에 붙입니다.

compose 명령의미 / 형태
docker compose up -d전체 스택을 백그라운드(-d)로 기동. --build로 이미지 재빌드 동반
docker compose down컨테이너·네트워크 정리(볼륨 유지). -v면 볼륨까지 삭제
docker compose logs -f appapp 서비스 로그 실시간 추적(-f). 서비스명 빼면 전체
docker compose ps이 프로젝트 서비스 상태(포트·health) 확인
docker compose exec app sh실행 중인 app 컨테이너에 셸로 진입(디버깅)
docker compose restart appapp만 재기동
단일 컨테이너·이미지의미 / 형태
docker build -t carbon-app .현재 Dockerfile로 이미지 빌드 + 태그(-t)
docker run --rm -p 8080:8080 carbon-app이미지를 단독 실행. --rm 종료 시 자동 삭제
docker ps / docker ps -a실행 중 / 멈춘 것 포함 전체 컨테이너
docker images로컬 이미지 목록·크기
docker exec -it <id> sh임의 컨테이너에 인터랙티브 셸 진입
docker logs -f <id>임의 컨테이너 로그 추적
정리(prune)는 무엇을 지우는지 보고

docker system prune은 멈춘 컨테이너·미사용 네트워크·dangling 이미지를 지웁니다. -a를 붙이면 사용 중이 아닌 모든 이미지까지, --volumes면 미사용 볼륨까지 삭제됩니다 — DB 볼륨이 미사용으로 잡혀 날아갈 수 있으니 운영 머신에서 --volumes는 신중히. 디스크가 부족할 때 docker system df로 먼저 무엇이 용량을 먹는지 확인하세요.

DOCKER-09

치트시트

Dockerfile 키워드·compose 구조·자주 쓰는 명령을 한 곳에. 외우기보다 "이런 게 있다"를 알고 필요할 때 찾으세요.

FROMbase image
베이스 이미지 지정. FROM img AS name으로 멀티스테이지 스테이지 이름 부여
WORKDIR작업 디렉터리
이후 명령의 기준 경로 설정·생성. cd 대신 이걸로
COPY파일 복사
빌드 컨텍스트→이미지. COPY --from=build /src dst로 다른 스테이지에서 가져오기
RUN빌드 시 실행
이미지 빌드 중 명령 실행(의존성 설치·빌드). 결과가 레이어로 남음
USER실행 사용자
이후 명령·런타임 프로세스를 지정 사용자로. 비root 권장
ENTRYPOINT / CMD실행 명령
ENTRYPOINT=고정 실행, CMD=기본 인자. exec 형식 ["java","-jar",..] 권장
멀티스테이지build → runtime
빌드 도구는 빌드 스테이지에만, 런타임 이미지엔 산출물만. 이미지 슬림화
servicescompose 최상위
각 컨테이너를 정의. build(직접 빌드) 또는 image(공개 이미지)
environment / env_file설정 주입
환경변수 직접 지정 / .env 파일째 주입. ${VAR} 치환은 .env에서
ports호스트:컨테이너
"80:80" 호스트 포트를 컨테이너에 매핑. 안 쓰면 외부 비노출
volumes데이터 영속
named volume(pgdata:/path) / bind mount(./f:/path:ro). 파일 하단에 named volume 선언
depends_on기동 순서
condition: service_healthy로 healthcheck 통과까지 대기
