AX 실전 적용 사례

실무 AX는 멋진 프롬프트보다 문제, 하네스, 검증, 지표, 실패 모드가 함께 정의된 사례에서 배워야 합니다. 아래 사례는 특정 회사 내부 자료가 아니라 공개 연구와 현업 패턴을 바탕으로 재구성한 적용 모델입니다.

CASE-01

사례 읽는 법

AX 사례는 “무엇을 AI로 했는가”보다 “어떤 업무를 어떤 위험 경계 안에서 자동화했는가”로 읽습니다.

항목질문
Problem반복되는 병목은 무엇인가
Workflow업무가 어떤 단계와 승인 지점으로 구성되는가
HarnessAI가 어떤 컨텍스트와 도구를 어떤 권한으로 사용하는가
Verification결과를 어떻게 자동/수동으로 검증하는가
Metrics효과와 부작용을 어떤 지표로 보는가
Failure Mode가장 위험한 오작동은 무엇이고 어디서 멈추는가
CASE-02

대규모 조직

Case A · 전사 PR Review Assistant

대규모 조직에서 PR 수가 늘며 리뷰 병목이 커지는 상황입니다. AI가 merge를 결정하지 않고, risk map과 test gap 후보를 먼저 생성합니다. 보안, 데이터 마이그레이션, public API 변경, 성능 위험은 별도 reviewer에게 라우팅합니다.

요소설계
HarnessPR diff, 관련 테스트, ownership, 최근 장애, coding standard를 context package로 제공
Permissionread-only, comment draft만 허용. approve/merge 금지
VerificationAI finding의 path/line/evidence/repro를 사람이 triage
Metricsreview latency, false positive rate, missed defect, reviewer load
Failure Mode사소한 스타일 지적 증가, 중요한 위험 누락, reviewer 피로
Case B · 엔터프라이즈 문서/런북 동기화

서비스가 많아 API 변경, 배포 절차, 장애 런북이 오래되는 문제를 다룹니다. AI는 code/API diff를 읽고 영향을 받는 문서를 찾은 뒤 patch를 제안합니다.

요소설계
HarnessOpenAPI spec, route diff, ADR, runbook, changelog를 검색
Permission문서 PR 생성만 허용, 운영 runbook 자동 반영은 owner 승인
Verification링크 체크, 예제 요청 검증, owner review
Metricsstale doc count, doc update lead time, incident runbook accuracy
Failure Mode코드는 맞지만 운영 절차가 빠진 문서, 출처 없는 요약
CASE-03

소규모 셀

Case C · 프론트엔드 셀의 접근성·성능 점검

작은 제품 셀에서 매 PR마다 접근성, Lighthouse, 핵심 사용자 흐름 점검이 누락되는 문제입니다. AI는 화면 변경 diff와 스토리북/테스트 결과를 바탕으로 위험 체크리스트를 만들고, 개발자는 우선순위를 결정합니다.

요소설계
Harnesschanged components, route, screenshots, axe result, Lighthouse result
Permission테스트 제안과 간단한 aria label 수정만 허용
VerificationPlaywright, axe, Lighthouse threshold, 수동 키보드 탐색
Metricsa11y issue count, performance regression, test gap closure
Failure Mode시각적으로만 맞고 키보드/스크린리더 흐름을 놓침
Case D · 백엔드 셀의 장애 회고 자동화

장애 후 회고가 늦고 원인 분석 품질이 일정하지 않은 문제입니다. AI는 로그, 메트릭, 배포 기록, PR diff를 모아 timeline 초안을 만들고, 사람은 원인과 액션 아이템을 확정합니다.

요소설계
Harnessincident ticket, deploy history, logs, traces, dashboards, ownership map
Permissionread-only 분석. mitigation 실행 금지
Verificationtimeline source link, metric screenshot, owner confirmation
MetricsMTTR, postmortem lead time, action item completion, repeat incident rate
Failure Mode상관관계를 원인으로 오판, 최근 배포에 과도하게 귀인

CASE-05

실패 사례

실패증상근본 원인수정
AI 리뷰가 병목을 늘림finding은 많지만 reviewer 시간이 증가false positive와 severity 기준 부재risk classifier와 triage rubric 도입
생성 코드가 rework를 늘림초기 구현은 빠르지만 수정과 회귀가 증가컨텍스트 부족, 테스트 부재, 큰 diff작은 task, 실패 테스트, acceptance rate/rework 추적
조직 도입이 도구 배포로 끝남사용률은 높은데 delivery 성능이 개선되지 않음업무 workflow와 지표 미정의workflow별 golden path와 DORA/quality metric 연결
셀 단위 실험이 공유되지 않음팀마다 같은 prompt와 실패를 반복episode 저장소와 커뮤니티 부재주간 episode review와 reusable template 운영
자동화가 지식 형성을 막음팀이 결과는 쓰지만 원리를 설명하지 못함완전 위임과 검증 생략explain-before-apply, review notes, learning retention 확인
