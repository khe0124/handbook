ACG·방화벽·NACL

라우팅이 패킷의 길을 정한다면, 방화벽은 그 길을 지나갈 자격을 검사합니다. VPN 터널이 UP이어도 방화벽이 막으면 서비스 통신은 실패합니다.

GOAL : 최소 권한 규칙을 설계하고, 허용 누락을 판독하기
FW-01

ACG의 역할

ACG는 NCP의 VM 단위 방화벽입니다. AWS의 Security Group과 비슷하게 허용 규칙 중심으로 동작합니다.

방향프로토콜포트대상의미
InboundTCP2210.0.1.0/24bastion에서만 SSH
InboundTCP8080172.16.0.0/16고객사에서 API 호출
OutboundTCP543210.0.3.0/24DB 서브넷 접근
OutboundTCP4430.0.0.0/0외부 API 호출
실무 감각

방화벽 규칙은 “서버 기준”으로 읽습니다. API 서버 ACG의 inbound는 API 서버로 들어오는 트래픽이고, outbound는 API 서버에서 나가는 트래픽입니다. 고객사 방화벽은 반대로 보일 수 있으므로 요청 방향을 정확히 맞춰야 합니다.

Stateless 주의

NACL은 요청을 허용해도 응답 포트를 별도로 허용해야 할 수 있습니다. TCP client는 ephemeral port를 쓰므로, NACL에서 return path를 막으면 SYN은 가지만 SYN-ACK가 돌아오지 않는 것처럼 보입니다.

FW-03

규칙 설계

방화벽 설계의 기본 원칙은 최소 권한입니다. 모든 대역과 모든 포트를 열어두면 장애는 줄어들지만 보안 사고의 폭이 커집니다.

나쁜 규칙좋은 규칙
Inbound TCP 0-65535 from 0.0.0.0/0Inbound TCP 8080 from 172.16.1.0/24
SSH from anywhereSSH from bastion subnet or admin /32
DB from app VPC 전체DB from app server ACG 또는 private app subnet
운영 기준

방화벽 요청서는 출발지, 목적지, 프로토콜, 포트, 용도, 만료일을 포함해야 합니다. 임시 테스트 규칙은 만료일을 두고 제거하지 않으면 운영 보안 부채가 됩니다.

FW-05

고급 포인트

Ephemeral port응답 포트
클라이언트가 임시 포트로 연결을 시작합니다. stateless 장비에서는 이 범위를 고려해야 합니다.
Deny rule거부 규칙
NACL이나 고객사 방화벽에는 명시 deny가 있을 수 있습니다. allow를 추가해도 더 높은 우선순위 deny가 이길 수 있습니다.
Security group reference그룹 참조
가능한 플랫폼에서는 IP CIDR보다 보안 그룹 참조가 변경에 강합니다. 단, VPN 너머 고객사 대역에는 보통 CIDR을 씁니다.
Audit trail변경 추적
누가 어떤 규칙을 왜 열었는지 남겨야 장애와 보안 검토가 가능합니다.
