방화벽·Security Group·NACL

라우팅이 “갈 수 있는 길”을 정한다면 방화벽은 “가도 되는 트래픽”을 정합니다. 네트워크 장애를 볼 때 route와 firewall을 분리해서 봐야 합니다.

FW-01

방화벽의 역할

정책 요소질문예시
출발지누가 접근하는가10.0.1.0/24, 172.16.0.0/16
목적지어느 서버 또는 subnet인가API 서버, DB subnet
프로토콜TCP, UDP, ICMP 중 무엇인가TCP 443, TCP 5432, ICMP
방향inbound인가 outbound인가고객사 → API, API → DB
FW-02

stateful vs stateless

구분Security Group/ACGNACL
적용 단위서버/ENI 단위Subnet 단위
상태statefulstateless
응답 트래픽요청 허용 시 응답 자동 허용응답 방향 규칙도 필요
규칙 성격허용 규칙 중심허용/거부와 우선순위
핵심 차이

stateful 방화벽은 연결 상태를 기억합니다. stateless NACL은 기억하지 않으므로 inbound를 열어도 outbound 응답 포트가 닫혀 있으면 통신이 실패할 수 있습니다.

FW-03

나쁜 규칙좋은 규칙이유
TCP 0-65535, 0.0.0.0/0TCP 443, LB Security Group외부 공격면 최소화

FW-04

VPN 허용 정책

VPN은 터널만 열어줄 뿐, 서버 포트를 자동으로 열어주지 않습니다.

방향예시 규칙확인 주체
고객사 → 우리 APIinbound TCP 8080 from 172.16.0.0/16우리 ACG/Security Group
우리 API → 고객 DBoutbound TCP 5432 to 172.16.10.10/32우리 outbound와 고객사 inbound
ICMP 테스트ICMP allow 양쪽 협의보안 정책에 따라 차단 가능
FW-05

운영 기준

Case study

route table이 맞고 VPN 터널도 UP인데 PostgreSQL 접속이 실패했습니다. 원인은 고객사 방화벽이 우리 VPC CIDR이 아니라 NAT 공인 IP만 허용한 것이었습니다. VPN 트래픽은 NAT를 거치지 않으므로 고객사 방화벽에는 실제 source CIDR을 허용해야 합니다.

최소 권한least privilege
필요한 출발지에서 필요한 목적지의 필요한 포트만 허용합니다.
변경 기록audit
누가 왜 언제 어떤 포트를 열었는지 남겨야 방치 규칙을 줄일 수 있습니다.
