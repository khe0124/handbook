프로젝트 심층 면접

프로젝트 질문은 기술 목록이 아니라 문제 정의, 제약, 선택, 결과, 본인 역할, 배운 점을 확인합니다. 답변은 구체적이어야 하지만 과장 없이 경험의 경계를 지켜야 합니다.

PJ-01

Q. 가장 복잡했던 프로젝트를 설명해보세요.

핵심 개념

복잡한 프로젝트 답변은 “큰 프로젝트였습니다”가 아니라 왜 복잡했는지 구조적으로 설명해야 합니다. 도메인 규칙, 데이터 흐름, 상태 전이, API 계약, 검증, 운영 이슈 중 무엇이 어려웠는지 분리합니다.

30초 답변

가장 복잡했던 프로젝트는 단순 화면 구현보다 도메인 규칙이 여러 입력 방식, 검증 단계, API 계약, 결과 화면에 동시에 영향을 준 프로젝트였습니다. 저는 사용자 흐름과 데이터 상태를 먼저 나누고, 입력은 다양해도 내부 모델은 하나로 정규화하며, validation과 저장 실패를 사용자가 회복할 수 있게 설계했습니다.

90초 답변

답변은 문제, 제약, 구조, 결과 순서로 말합니다. 예를 들어 “Excel 업로드, key-in, grid edit가 서로 다른 입력 채널이지만 최종적으로는 같은 domain row로 저장되어야 했다”처럼 복잡성의 원인을 구체화합니다. 이어서 어떤 상태 구조를 잡았고, 어떤 validation 단계를 나눴고, 어떤 API 변경이나 운영 이슈를 처리했는지 말합니다.

꼬리질문

질문답변 방향
본인 역할은 정확히 무엇인가요?설계, 구현, 리뷰, 운영 대응 중 직접 담당한 범위를 분리합니다.
어떤 결과가 있었나요?정량 수치가 없으면 재사용성, QA 포인트 감소, 오류 위치 명확화 같은 증거를 말합니다.
다시 한다면?초기 모델링, 테스트, 관측, 문서화 중 개선할 지점을 말합니다.

오답/주의점

주의

도메인 이름과 기능 목록만 나열하지 마세요. 면접관은 “왜 어려웠고 어떻게 구조화했는가”를 듣고 싶어합니다.

PJ-02

Q. Excel, Key-in, Data Grid 입력을 단일 상태로 통합한다는 것은 무엇인가요?

핵심 개념

입력 채널은 다르지만 저장과 검증의 대상이 되는 도메인 데이터는 하나여야 합니다. channel-specific parsing과 common domain state를 분리합니다.

30초 답변

Excel 업로드, 직접 입력, grid cell edit는 사용자 입장에서는 다른 입력 방식이지만 저장 단계에서는 같은 domain row로 다뤄져야 합니다. 그래서 입력 source는 구분하되 내부에서는 parse, normalize, map을 거쳐 common editable state로 변환하고 validation과 submit은 그 공통 모델 위에서 처리합니다.

90초 답변

Data Grid column definition은 단순 표시 설정이 아니라 formatter, parser, editor, validator를 가진 계약으로 봅니다. validation은 cell 단위 타입/범위, row 단위 cross-field, submit 단위 중복/서버 의존 규칙으로 나눕니다. 오류는 상단 summary만이 아니라 사용자가 수정할 cell, row, page 위치로 연결되어야 합니다.

꼬리질문

질문답변 방향
대용량 grid 성능은?virtualization, 변경분 중심 상태, row memoization, debounced validation을 봅니다.
Excel parsing 실패는?template mismatch, header mapping, type 변환, validation 실패를 구분해 보여줍니다.
dirty check는?server snapshot과 draft를 비교하거나 dirty row/cell을 추적합니다.

오답/주의점

주의

“상태를 Redux에 넣었습니다”로 답하지 마세요. 어떤 상태를 왜 분리했는지가 핵심입니다.

PJ-03

Q. Flow Editor의 node/edge 상태는 어떻게 관리했나요?

핵심 개념

Flow Editor는 화면상의 graph state와 저장 가능한 domain graph를 분리해야 합니다. 사용자의 드래그와 연결선은 UI state이지만 서버에는 도메인 식별자와 관계가 저장됩니다.

30초 답변

핵심은 domain model과 flow view model을 분리하는 것입니다. 화면에서는 node id, position, selected state, edge source/target이 필요하고, 저장할 때는 도메인 entity id, 관계, 순서, 좌표처럼 복원 가능한 데이터로 변환합니다.

90초 답변

domain → flow 변환기는 서버 데이터를 화면 노드와 엣지로 만들고, flow → domain 변환기는 사용자가 편집한 그래프를 저장 가능한 payload로 바꿉니다. 성능은 custom node props 안정화, node component memoization, 전체 nodes/edges 배열 구독 최소화, drag 중 업데이트 비용으로 봅니다. 잘못된 연결을 막는 validation도 필요합니다.

꼬리질문

질문답변 방향
React Flow 성능은?custom node 렌더링, props 안정성, selector 분리, viewport rendering을 봅니다.
저장/불러오기는?node id, type, position, edge source/target, 도메인 id를 손실 없이 저장합니다.
Canvas/SVG/DOM 차이는?DOM은 접근성과 구성 쉬움, Canvas는 대량 렌더링, SVG는 벡터와 shape 표현에 유리합니다.

오답/주의점

주의

라이브러리 사용 여부보다 저장 가능한 구조와 도메인 변환을 설명해야 합니다.

PJ-04

Q. API 전환 프로젝트에서 무엇이 어려웠나요?

핵심 개념

API 전환은 endpoint만 바꾸는 작업이 아니라 화면이 의존하던 데이터 계약, error handling, loading/empty state, status mapping을 함께 바꾸는 작업입니다.

30초 답변

어려운 점은 response 구조와 status 값이 바뀌면 화면 곳곳의 조건, 표시 문구, action 가능 여부가 함께 바뀐다는 점이었습니다. 그래서 raw response를 직접 쓰지 않고 service/adapter layer에서 view model로 변환해 수정 범위를 줄이는 방향으로 접근했습니다.

90초 답변

전환할 때는 기존 API와 신규 API의 field mapping, nullable, enum, error format, pagination, 권한 조건을 비교합니다. 화면은 loading, empty, error, permission denied 상태를 모두 가져야 하고, 실패하면 어느 계층 문제인지 Network, service adapter, state update, render layer로 분리해 봅니다.

꼬리질문

질문답변 방향
스펙이 불완전하면?nullable, enum, error, pagination 질문 목록을 만들고 mock contract로 진행합니다.
adapter가 과한가요?변경 빈도와 화면 모델 차이가 클 때 유용하고, 단순 CRUD에서는 과할 수 있습니다.
테스트는?adapter unit test, API mock integration test, 핵심 화면 E2E를 나눕니다.

오답/주의점

주의

“API 주소만 바꿨습니다”로 들리면 역할이 작아 보입니다. 계약 변경과 화면 영향 분석을 말하세요.

PJ-05

Q. 중복 요청이나 무한 렌더링은 어떻게 디버깅했나요?

핵심 개념

운영 이슈 답변은 “고쳤다”보다 재현 조건을 좁히고 원인을 계층별로 분리한 과정을 보여줘야 합니다.

30초 답변

먼저 재현 조건을 고정하고 Network, Console, React DevTools, state change를 확인합니다. 중복 요청은 Strict Mode, effect dependency, component remount, 버튼 중복 클릭, query key 중복을 보고, 무한 렌더링은 effect가 dependency를 다시 바꾸는지와 매 렌더 새 객체를 만드는지 확인합니다.

90초 답변

중복 요청은 원인에 따라 대응이 다릅니다. 사용자 중복 클릭이면 in-flight guard나 disabled 처리가 맞고, 검색 요청이면 debounce와 AbortController가 필요할 수 있습니다. 같은 서버 데이터가 여러 컴포넌트에서 호출되면 query key 기반 cache를 봅니다. 무한 렌더링은 dependency를 억지로 제거하기보다 effect의 책임과 state 소유권이 맞는지 봐야 합니다.

꼬리질문

질문답변 방향
Strict Mode 중복은?개발 환경에서 side effect 발견을 위해 두 번 실행될 수 있어 idempotent effect가 필요합니다.
AbortController는 언제?검색/필터처럼 이전 요청 결과가 최신 요청을 덮으면 안 될 때 사용합니다.
증거는?Network waterfall, profiler 캡처, 재현 steps, 수정 전후 로그를 남깁니다.

오답/주의점

주의

“dependency array를 비웠습니다”는 위험합니다. 왜 그 effect가 필요한지부터 설명해야 합니다.

PJ-06

Q. 복잡한 B2B 도메인을 어떻게 학습했나요?

핵심 개념

도메인 학습 답변은 열심히 공부했다가 아니라 용어, 규칙, 예외, 사용자 업무 흐름을 어떻게 구조화했는지 보여줘야 합니다.

30초 답변

먼저 화면과 API 용어를 사용자 업무 용어로 다시 정리하고, 입력-검증-승인-저장-조회 흐름을 그렸습니다. 모호한 규칙은 예시 데이터를 기준으로 질문하고, 결정된 내용은 validation rule과 상태 전이로 문서화했습니다.

90초 답변

B2B 도메인은 예외가 많기 때문에 단순 요구사항 목록보다 decision log가 중요했습니다. 어떤 값이 필수인지, 어떤 상태에서 수정 가능한지, 권한별 action이 어떻게 다른지 표로 만들고, 화면 상태와 API contract에 반영했습니다. 모르는 부분은 임의 구현하지 않고 질문 목록으로 묶어 빠르게 합의했습니다.

꼬리질문

질문답변 방향
도메인 전문가와 어떻게 소통?예시 데이터와 사용자 시나리오로 질문했습니다.
문서화는?용어집, 상태 전이, validation rule, 예외 케이스를 남겼습니다.
실수는?초기 가정을 작게 만들고 빠르게 검증하는 방식으로 줄였습니다.

면접에서 말할 증거

용어집, rule table, 상태 전이도, 질문 목록, decision log를 준비합니다.

오답/주의점

주의

도메인을 몰랐다는 방어보다 어떻게 학습 시스템을 만들었는지가 중요합니다.

PJ-07

Q. Data Grid column definition을 어떻게 관리했나요?

핵심 개념

column definition은 표시 설정이 아니라 입력, 변환, 검증, 권한, 저장 payload를 잇는 계약입니다.

30초 답변

각 column에 label, accessor, formatter, parser, editor, validator, editable 조건을 정의했습니다. 표시용 값과 저장용 값을 분리하고, Excel import와 key-in 입력도 같은 column contract를 타도록 맞췄습니다.

90초 답변

column 설정이 화면 곳곳에 흩어지면 validation과 저장 로직이 중복됩니다. 그래서 column definition을 단일 진입점으로 두고 cell edit, Excel mapping, error display가 같은 규칙을 참조하게 했습니다. 다만 너무 범용화하면 복잡해지므로 도메인별 예외는 명시적으로 분리했습니다.

꼬리질문

질문답변 방향
성능 이슈는?column definition 참조 안정성과 cell renderer memoization을 봅니다.
권한별 편집은?editable 조건과 서버 인가를 분리합니다.
과한 추상화는?반복 규칙만 공통화하고 예외는 드러냅니다.

면접에서 말할 증거

column schema 예시, parser/validator 테스트, Excel mapping 재사용 사례를 말합니다.

오답/주의점

주의

라이브러리 설정을 잘했다는 답보다 도메인 규칙을 어디에 모았는지가 중요합니다.

PJ-08

Q. validation error를 cell, row, page 단위로 어떻게 보여줬나요?

핵심 개념

validation UX는 오류를 알려주는 것이 아니라 사용자가 오류 위치로 이동해 수정하고 다시 저장할 수 있게 만드는 흐름입니다.

30초 답변

오류를 field/cell, row, page summary로 나눴습니다. cell 오류는 즉시 수정 위치에 표시하고, row 오류는 cross-field 규칙을 보여주며, page summary는 전체 오류 수와 첫 오류 이동을 제공했습니다.

90초 답변

서버 오류도 UI가 이해할 수 있는 path로 매핑해야 했습니다. 예를 들어 row id, column key, error code를 받아 grid cell에 연결하고, Excel import 오류도 같은 구조로 표현했습니다. 오류 summary만 보여주면 사용자는 어디를 고쳐야 하는지 몰라 작업 시간이 길어집니다.

꼬리질문

질문답변 방향
서버 오류 path가 없으면?백엔드와 error contract를 합의하거나 fallback summary를 둡니다.
대량 오류는?필터, 첫 오류 이동, export report를 고려합니다.
접근성은?aria-describedby, focus 이동, 오류 요약을 제공합니다.

면접에서 말할 증거

error object 구조, cell highlight, summary 이동, 서버 error contract를 말합니다.

오답/주의점

주의

alert로 오류를 한 번 보여주는 것은 복잡한 입력 UI의 복구 흐름이 아닙니다.

PJ-09

Q. React Flow 성능 문제는 어떻게 봐야 하나요?

핵심 개념

Flow editor 성능은 노드 수, 커스텀 노드 렌더링, props 안정성, store 구독 범위, drag 중 업데이트 비용으로 나눠 봅니다.

30초 답변

먼저 Profiler로 어떤 node가 왜 다시 렌더링되는지 확인합니다. custom node props를 안정화하고, 전체 nodes/edges를 넓게 구독하지 않게 하며, drag 중 비싼 계산이나 저장 payload 변환을 분리합니다.

90초 답변

Flow 화면은 domain graph와 view graph를 분리해야 합니다. 서버 저장용 변환이 매 렌더마다 돌면 비용이 커집니다. node component는 memoization하고, callback과 style 객체를 안정화하며, 선택 상태나 hover 상태가 전체 graph 렌더링을 유발하지 않게 합니다.

꼬리질문

질문답변 방향
저장은 언제?사용자 명시 저장 또는 debounce autosave로 렌더와 분리합니다.
validation은?연결 시점의 빠른 검증과 submit 검증을 나눕니다.
큰 그래프는?viewport rendering, clustering, lazy detail을 봅니다.

면접에서 말할 증거

Profiler 캡처, 렌더링 횟수, node/edge 변환 함수, drag 성능 개선 전후를 말합니다.

오답/주의점

주의

React Flow가 느리다고 결론내리기 전에 내 상태 구독과 props 안정성을 확인해야 합니다.

PJ-10

Q. 파일 업로드/다운로드 이슈는 어떻게 해결했나요?

핵심 개념

파일 기능은 UI 이벤트가 아니라 크기, 타입, 인코딩, 보안, 진행률, 실패 복구, 서버 저장 정책을 포함합니다.

30초 답변

업로드는 파일 크기와 타입 검증, 진행률, 취소, 실패 재시도를 고려했습니다. 다운로드는 filename encoding, content-type, 권한, 대용량 stream, 브라우저 호환성을 확인했습니다.

90초 답변

Excel이나 CSV는 인코딩과 formula injection, header mismatch 문제가 생길 수 있습니다. 업로드 후 바로 저장하지 않고 preview와 validation 단계를 둬 사용자가 오류를 확인하게 했습니다. 다운로드는 서버가 내려주는 header와 브라우저 처리 차이를 확인해야 합니다.

꼬리질문

질문답변 방향
대용량 업로드는?chunk, presigned URL, progress, resume을 봅니다.
보안은?파일 타입 검증, 악성 수식, 권한, 저장 경로를 봅니다.
파일명 깨짐은?Content-Disposition encoding과 브라우저 호환성을 확인합니다.

면접에서 말할 증거

Network header, 파일 검증 규칙, 진행률 UI, 실패 재시도 로그를 말합니다.

오답/주의점

주의

파일 input만 구현했다고 말하면 운영과 보안 고려가 빠져 보입니다.

PJ-11

Q. i18n key와 Storybook은 어떻게 운영했나요?

핵심 개념

i18n과 Storybook은 문서 도구가 아니라 UI 품질과 변경 관리를 돕는 운영 자산입니다.

30초 답변

i18n key는 도메인 namespace로 관리하고, 동적 문장 조합을 줄여 번역 품질을 지켰습니다. Storybook은 컴포넌트 상태 조합, 오류/로딩/빈 상태, 접근성 확인을 위해 사용했습니다.

90초 답변

번역 key가 화면 문자열과 1:1로만 흩어지면 재사용과 변경 추적이 어렵습니다. Storybook은 디자이너와 QA가 상태를 확인하는 기준점이 되고, interaction test나 accessibility addon을 붙이면 회귀를 빨리 찾을 수 있습니다. 다만 모든 화면을 Storybook화하기보다 재사용 컴포넌트와 복잡한 상태부터 잡는 것이 현실적입니다.

꼬리질문

질문답변 방향
missing key는?CI check나 runtime report로 추적합니다.
동적 문장 문제는?언어별 어순을 고려해 문장 단위 key를 씁니다.
Storybook 비용은?복잡한 상태와 공통 컴포넌트부터 우선순위를 둡니다.

면접에서 말할 증거

namespace 구조, missing key 리포트, Storybook 상태 matrix, 접근성 addon 결과를 말합니다.

오답/주의점

주의

Storybook을 예쁜 문서로만 보면 품질 검증 도구로서의 가치가 약해집니다.

PJ-12

Q. 프론트엔드 중심 지원자가 백엔드/API 질문을 받으면 어떻게 답하나요?

핵심 개념

경험의 경계를 지키면서도 API 계약, 데이터 흐름, validation, 장애 진단을 이해한다는 것을 보여줘야 합니다.

30초 답변

깊은 백엔드 운영 경험은 제한적일 수 있지만, 프론트 구현에서 API 계약, error format, pagination, 권한, validation 결과를 다뤘고 그 영향이 사용자 흐름에 어떻게 나타나는지 이해하고 있다고 답합니다.

90초 답변

예를 들어 API response를 그대로 화면에 묶지 않고 adapter로 view model을 만들었고, 서버 validation error를 field/cell 단위로 매핑했으며, 스펙 변경 시 nullable, enum, status code, pagination을 확인했다고 말할 수 있습니다. 모르는 운영 영역은 과장하지 말고 “설계한다면 어떤 지표와 로그로 확인하겠다”까지 답합니다.

꼬리질문

질문답변 방향
직접 DB 튜닝했나요?직접 범위를 말하고 실행 계획/인덱스 개념 이해를 분리합니다.
백엔드 설계 가능한가요?API contract와 데이터 흐름 중심으로 기여 가능 범위를 말합니다.
운영 경험은?로그, trace, 재현 조건, 장애 영향 파악 경험을 말합니다.

면접에서 말할 증거

API mapping 표, error contract, Network trace, 백엔드와 합의한 질문 목록을 준비합니다.

오답/주의점

주의

경험을 부풀리기보다 프론트와 서버 경계를 정확히 이해했다는 신뢰를 주는 편이 강합니다.
