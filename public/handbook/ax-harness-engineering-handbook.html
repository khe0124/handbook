AI Harness Engineering

AI Harness Engineering은 모델을 더 똑똑하게 만드는 일이 아니라, 모델이 실무 작업을 안정적으로 수행하도록 작업 환경을 설계하는 일입니다.

HAR-01

하네스의 역할

하네스는 모델과 실제 개발 환경 사이의 런타임 계층입니다. 이 계층은 무엇을 보게 할지, 어떤 도구를 쓰게 할지, 어디까지 수정하게 할지, 무엇을 완료 증거로 인정할지를 통제합니다.

WITHOUT HARNESS prompt → model → plausible output

WITH HARNESS task spec → context package → tool policy → model
→ verification → failure attribution → auditable episode

HAR-04

운영 패턴

좋은 하네스는 작업 전, 작업 중, 작업 후의 책임이 분리됩니다.

단계하네스가 해야 할 일산출물
작업 전명세를 좁히고 권한·검증 기준을 세운다task spec, permission policy
작업 중읽은 파일, 가설, 시도, 실패를 기록한다task state, tool trace
작업 후검증 결과와 실패 귀속을 남긴다verification report, episode package
Case study

결제 완료 후 주문 상태가 바뀌지 않는 문제를 AI에게 맡길 때, 하네스는 결제 webhook spec, 상태 전이 규칙, 관련 테스트, DB migration 금지, 중복 콜백 검증을 함께 제공합니다. 이것이 없으면 모델은 컨트롤러만 고치고 트랜잭션 경계나 멱등성 문제를 놓칠 수 있습니다.

HAR-06

자동화 하네스

업무 자동화를 위한 하네스는 하나의 agent가 아니라 작업 큐, 컨텍스트 빌더, 도구 권한, 검증 게이트, 사람 승인, 감사 로그가 연결된 실행 시스템입니다.

AUTOMATION HARNESS ARCHITECTURE workflow trigger
→ task classifier
→ context builder
→ tool permission policy
→ agent execution
→ verification gate
→ human approval if risk high
→ episode package + metrics
구성요소대규모 조직소규모 셀
TriggerPR, incident, deployment, ticket event 기반 자동 큐스프린트 중 선택한 반복 업무를 수동 실행
Context builder내부 검색, ownership, policy, runbook, service catalog 연결README, ADR, spec, 최근 diff를 작은 context package로 유지
Permission중앙 policy, 모델 게이트웨이, tool auditread-only 기본, write는 작은 파일과 테스트 범위로 제한
Verification업무별 eval set, CI gate, security/policy check테스트·빌드·수동 체크리스트와 사람 리뷰
Metrics팀별 lead time, review burden, rework, defect escape, cost주간 episode 5~10개로 time saved와 실패 원인 리뷰
설계 주의

하네스가 없으면 AI 자동화는 빠른 생성 도구가 됩니다. 하네스가 과하면 작은 업무에도 플랫폼 비용이 커집니다. 업무 위험도와 반복성에 맞춰 가벼운 하네스에서 시작하고, 증거가 쌓이면 공통 플랫폼으로 올립니다.
