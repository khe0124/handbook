DB 심화 면접

DB 심화 면접은 SQL 문법보다 정합성, 동시성, 실행 계획, 장애 복구, 변경 운영을 현실적으로 이해하는지 확인합니다.

DBD-01

Q. DB 심화 질문은 어떤 프레임으로 답하나요?

핵심 개념

DB 답변은 기능 구현보다 데이터 정합성 요구, 읽기/쓰기 패턴, 동시성, 복구, 운영 변경 위험을 함께 설명해야 합니다.

개념 해부

DB 심화 답변은 기능명보다 정합성, 성능, 운영 위험을 동시에 보는 프레임이 중요합니다. 같은 인덱스도 읽기 성능, 쓰기 비용, 락, 배포 위험이 함께 있습니다. 면접에서는 쿼리 하나를 고치는 기술보다 장애와 변경을 견디는 판단 기준을 보여줘야 합니다.

30초 답변

먼저 데이터 정합성 요구와 읽기/쓰기 패턴을 확인합니다. 그 다음 transaction 경계, isolation level, lock, index, query plan을 근거로 설계하고, 마지막으로 백업·복제·마이그레이션·복구 훈련 같은 운영 리스크를 봅니다.

90초 답변

예를 들어 주문 상태 변경은 단순 update가 아니라 중복 요청, lost update, 외부 결제 API timeout, 상태 전이, audit log까지 봐야 합니다. 검색 API는 index와 pagination만이 아니라 cardinality, selectivity, sort 비용, stale read 허용 여부를 봅니다. 면접에서는 “트랜잭션을 겁니다”보다 어떤 anomaly를 막고 어떤 비용을 감수하는지 말해야 합니다.

꼬리질문

질문답변 방향
DB에서 가장 중요한 품질은?정확성, 일관성, 복구 가능성, 성능, 운영 가능성의 균형입니다.
성능과 정합성이 충돌하면?도메인 요구와 실패 비용을 기준으로 선택합니다.
실무 증거는?EXPLAIN, slow query log, migration dry-run, backup restore drill을 말합니다.

오답/주의점

주의

DB 질문에서 도구나 ORM API만 말하면 얕습니다. 데이터가 깨지는 조건을 먼저 봐야 합니다.

DBD-02

Q. MVCC는 무엇이고 왜 필요한가요?

핵심 개념

MVCC는 row version을 관리해 reader와 writer가 서로를 덜 막으면서 일관된 snapshot을 보게 하는 동시성 제어 방식입니다.

개념 해부

MVCC는 독자가 쓰기를 막고 쓰기가 독자를 막는 상황을 줄이기 위해 row version을 유지하는 모델입니다. transaction은 자기 snapshot을 보고, DB는 오래된 version을 일정 기간 보존합니다. 오래 열린 transaction은 vacuum이나 undo 정리를 막아 bloat와 성능 문제를 만들 수 있습니다.

30초 답변

MVCC는 update가 기존 row를 덮어쓰는 대신 새 version을 만들고, transaction은 자신의 snapshot에서 보이는 version을 읽습니다. 그래서 읽기가 쓰기를 항상 막지 않아 동시성이 좋아집니다. 대신 오래 열린 transaction이 있으면 오래된 version 정리가 늦어져 bloat나 vacuum 문제가 생길 수 있습니다.

90초 답변

PostgreSQL 같은 MVCC DB에서는 각 transaction이 볼 수 있는 row version이 정해져 있습니다. Read Committed에서는 statement마다 snapshot이 바뀌고, Repeatable Read에서는 transaction 시작 시점 snapshot이 유지됩니다. MVCC는 lock을 없애는 것이 아니라 읽기/쓰기 충돌을 줄이는 방식이며, 쓰기-쓰기 충돌이나 unique constraint, row lock은 여전히 중요합니다.

꼬리질문

질문답변 방향
vacuum은 왜 필요한가요?더 이상 보이지 않는 오래된 row version을 정리해 bloat를 줄입니다.
long transaction 문제는?old version 정리를 막아 bloat, xid wraparound, replication lag 위험을 키웁니다.
MVCC면 lock이 필요 없나요?아닙니다. 쓰기 충돌, DDL, FK, unique check에는 lock이 필요합니다.

면접에서 말할 증거

autovacuum, dead tuple, long running transaction, lock wait를 실제 지표로 언급하면 깊이가 생깁니다.

DBD-03

Q. 트랜잭션 격리 수준은 어떻게 선택하나요?

핵심 개념

격리 수준은 transaction들이 서로의 중간 상태나 변경을 어느 정도 볼 수 있는지 정하는 정책입니다. 높을수록 anomaly는 줄지만 충돌과 retry 비용이 커질 수 있습니다.

개념 해부

격리 수준은 anomaly를 얼마나 허용할지의 선택입니다. dirty read, non-repeatable read, phantom, write skew 같은 현상 중 도메인이 허용하지 않는 것이 무엇인지 먼저 정해야 합니다. DB마다 같은 이름의 isolation도 구현이 다를 수 있으므로 PostgreSQL, MySQL 차이를 분리해 말하는 편이 정확합니다.

30초 답변

Read Committed는 statement마다 commit된 최신 데이터를 보고, Repeatable Read는 transaction 동안 같은 snapshot을 유지합니다. Serializable은 직렬 실행과 같은 결과를 목표로 하지만 serialization failure에 대한 retry 설계가 필요합니다. 선택 기준은 막아야 하는 anomaly와 성능 비용입니다.

90초 답변

실무에서는 isolation level만 올리는 것이 답이 아닐 수 있습니다. 재고 차감은 조건부 update나 row lock이 더 명확할 수 있고, 편집 충돌은 optimistic locking이 사용자 경험에 맞을 수 있습니다. Serializable을 쓰면 더 안전해 보이지만 충돌 시 재시도 로직과 idempotency가 없으면 장애로 이어질 수 있습니다.

꼬리질문

질문답변 방향
non-repeatable read는?같은 transaction에서 같은 row를 다시 읽었을 때 값이 달라지는 현상입니다.
phantom read는?조건에 맞는 row 집합이 transaction 중 달라지는 현상입니다.
Serializable 실패는?DB가 직렬화 불가능한 실행을 감지하면 transaction retry가 필요합니다.

오답/주의점

주의

“중요하면 Serializable을 씁니다”는 반쪽 답변입니다. retry와 성능 비용까지 말해야 합니다.

DBD-04

Q. 락과 deadlock은 어떻게 다루나요?

핵심 개념

락은 동시 변경의 정합성을 지키기 위한 제어이고, deadlock은 transaction들이 서로가 가진 락을 기다리며 진행하지 못하는 상태입니다.

개념 해부

락은 데이터 정합성을 지키지만 대기와 deadlock이라는 비용을 만듭니다. row lock, gap lock, table lock, metadata lock은 발생 조건과 영향 범위가 다릅니다. deadlock은 단순 오류가 아니라 DB가 교착을 감지해 한쪽 transaction을 희생시킨 것이므로 retry와 lock 순서 통일이 필요합니다.

30초 답변

락 문제는 긴 transaction, 누락 인덱스, 일관되지 않은 락 획득 순서, transaction 안의 외부 API 호출에서 자주 생깁니다. deadlock을 줄이려면 같은 순서로 row를 잠그고, transaction을 짧게 유지하고, lock timeout과 retry를 설계합니다.

90초 답변

lock wait가 길면 먼저 어떤 query가 어떤 lock을 잡고 있는지 봅니다. 인덱스가 없어 더 많은 row를 스캔하며 잠그는 경우, 사용자 입력을 기다리거나 외부 API를 호출하는 동안 transaction이 열려 있는 경우가 흔합니다. deadlock은 DB가 하나를 abort할 수 있으므로 애플리케이션은 안전한 retry와 idempotency를 갖춰야 합니다.

꼬리질문

질문답변 방향
SELECT FOR UPDATE는 언제?읽은 row를 이후 update할 것이고 경쟁 변경을 막아야 할 때 씁니다.
lock timeout은 왜?무한 대기를 막고 장애를 빨리 감지하기 위해 둡니다.
외부 API를 transaction 안에서 호출하면?transaction 시간이 길어져 lock wait와 deadlock 위험이 커집니다.

면접에서 말할 증거

lock table, slow query, transaction duration, deadlock log를 확인한 경험을 말합니다.

DBD-05

Q. lost update는 왜 생기고 어떻게 막나요?

핵심 개념

lost update는 두 transaction이 같은 값을 읽고 각각 수정해 한쪽 변경이 사라지는 동시성 문제입니다.

개념 해부

lost update는 두 transaction이 같은 값을 읽고 각각 수정했는데 나중 commit이 앞선 변경을 덮어쓰는 문제입니다. version column, 조건부 update, row lock, serializable isolation 등으로 막을 수 있습니다. 어떤 방법이든 충돌 감지 후 사용자나 시스템이 어떻게 재시도할지 정해야 합니다.

30초 답변

lost update는 사용자가 같은 데이터를 동시에 수정할 때 자주 생깁니다. 해결은 version column 기반 optimistic locking, row lock 기반 pessimistic locking, 조건부 atomic update, isolation level 조정 중 요구사항에 맞게 선택합니다.

90초 답변

문서 편집처럼 충돌이 드문 작업은 optimistic locking이 적합합니다. update where id=? and version=?로 갱신하고 영향 row가 0이면 충돌을 알려 최신 데이터를 다시 보여줍니다. 재고 차감처럼 경쟁이 높은 작업은 update stock = stock - 1 where stock > 0 같은 조건부 update나 row lock을 검토합니다.

꼬리질문

질문답변 방향
optimistic lock의 UX는?충돌 시 최신 데이터와 내 변경을 비교해 재시도하게 합니다.
pessimistic lock 단점은?대기와 deadlock, throughput 감소가 생길 수 있습니다.
재고 차감은?조건부 update, unique constraint, idempotency를 함께 봅니다.

오답/주의점

주의

“트랜잭션이면 됩니다”로는 부족합니다. 어떤 격리 수준에서 어떤 update가 어떻게 충돌하는지 말해야 합니다.

DBD-06

Q. 느린 쿼리는 어떤 순서로 분석하나요?

핵심 개념

느린 쿼리 대응은 감으로 인덱스를 추가하는 일이 아니라 실제 실행 계획, row 수 추정, I/O, sort, join 비용을 확인하는 과정입니다.

개념 해부

느린 쿼리 분석은 실행 계획의 한 줄을 읽는 일이 아니라 planner의 예상과 실제 실행의 차이를 보는 일입니다. estimated rows와 actual rows가 크게 다르면 통계나 데이터 분포 문제가 있을 수 있습니다. sort, hash, nested loop, buffer read, lock wait를 함께 봐야 원인을 정확히 좁힙니다.

30초 답변

먼저 재현 가능한 쿼리와 파라미터를 확보하고 EXPLAIN ANALYZE로 실제 실행 계획을 봅니다. estimated rows와 actual rows 차이, seq scan이 적절한지, join 방식, sort/hash 메모리, index 사용 여부를 확인합니다. 그 다음 인덱스, 쿼리 구조, 통계, pagination 방식을 조정합니다.

90초 답변

Seq Scan은 항상 나쁜 것이 아닙니다. 작은 테이블이나 selectivity가 낮은 조건에서는 합리적일 수 있습니다. 오히려 planner가 row 수를 크게 잘못 예측하면 잘못된 join 순서나 nested loop 폭증이 생깁니다. offset pagination은 뒤 페이지로 갈수록 비용이 커질 수 있어 keyset pagination을 검토합니다.

꼬리질문

질문답변 방향
estimated와 actual 차이가 크면?통계 갱신, 컬럼 상관관계, skew, extended statistics를 봅니다.
index를 추가하면 항상 빨라지나요?읽기는 빨라질 수 있지만 write 비용, 저장 공간, planner 선택 문제가 생깁니다.
ORDER BY가 느리면?정렬 메모리, 인덱스 순서, limit, keyset pagination을 봅니다.

면접에서 말할 증거

실제 plan의 rows, loops, buffers, sort method를 보고 판단했다는 식으로 말하면 신뢰도가 높습니다.

DBD-07

Q. 인덱스는 어떻게 설계하나요?

핵심 개념

인덱스는 자주 쓰는 WHERE, JOIN, ORDER BY 패턴을 빠르게 하지만 write 비용과 저장 공간을 늘립니다.

개념 해부

인덱스 설계는 where 절 컬럼을 모두 넣는 작업이 아닙니다. equality, range, order by, covering 여부, selectivity, update 빈도를 고려해 컬럼 순서와 포함 범위를 정합니다. partial index나 expression index는 강력하지만 쿼리 조건이 정확히 맞아야 효과가 납니다.

30초 답변

인덱스는 쿼리 패턴과 cardinality를 보고 설계합니다. equality 조건, range 조건, sort 조건의 순서를 보고 composite index 컬럼 순서를 정하고, active row만 자주 조회하면 partial index, projection이 제한되면 covering index를 검토합니다.

90초 답변

복합 인덱스는 왼쪽 prefix와 range 조건 이후 활용도를 이해해야 합니다. 예를 들어 tenant_id, status, created_at 조건이 반복되면 tenant_id와 status 같은 equality 조건을 앞에 두고 created_at range/order를 뒤에 둘 수 있습니다. 하지만 모든 조합에 인덱스를 만들면 write와 vacuum 비용이 커집니다. 실제 workload와 slow query 기준으로 관리해야 합니다.

꼬리질문

질문답변 방향
covering index는?필요 컬럼을 인덱스에서 충족해 table lookup을 줄이는 전략입니다.
partial index는?deleted_at is null 같은 자주 쓰는 subset에만 인덱스를 만들어 크기를 줄입니다.
인덱스가 안 타면?조건식, 타입 캐스팅, 함수 사용, selectivity, 통계, predicate 일치를 확인합니다.

오답/주의점

주의

“자주 검색하는 컬럼에 인덱스”는 너무 얕습니다. 복합 조건, 정렬, 카디널리티, write 비용까지 봐야 합니다.

DBD-08

Q. 트랜잭션 경계는 어떻게 잡나요?

핵심 개념

트랜잭션 경계는 함께 성공하거나 함께 실패해야 하는 비즈니스 정합성 단위입니다.

개념 해부

트랜잭션 경계는 비즈니스 불변식을 지키는 최소 단위여야 합니다. 너무 짧으면 중간 상태가 노출되고, 너무 길면 lock과 connection을 오래 잡습니다. 외부 API 호출, 파일 처리, 사용자 입력 대기 같은 느린 작업을 transaction 안에 넣으면 장애가 DB로 전파됩니다.

30초 답변

트랜잭션은 DB 작업을 많이 묶는 것이 아니라 정합성 단위로 묶어야 합니다. 외부 API 호출, 사용자 입력 대기, 긴 batch 처리는 transaction 밖으로 빼거나 outbox, saga, idempotency 같은 패턴을 검토합니다.

90초 답변

예를 들어 주문 생성과 결제 요청은 하나의 DB transaction으로 외부 API까지 묶기 어렵습니다. DB에는 주문 pending과 outbox event를 같은 transaction으로 저장하고, 별도 worker가 결제 요청을 처리한 뒤 상태를 전이할 수 있습니다. 이렇게 하면 DB 정합성과 외부 시스템 실패를 분리할 수 있습니다.

꼬리질문

질문답변 방향
긴 transaction 문제는?lock 유지, MVCC bloat, replication lag, deadlock 위험이 커집니다.
외부 API와 DB를 원자적으로 묶을 수 있나요?일반적으로 어렵고 outbox/saga/idempotency로 보완합니다.
rollback은 언제 되나요?DB 작업은 되돌릴 수 있지만 외부 side effect는 별도 보상이나 상태 관리가 필요합니다.

면접에서 말할 증거

transaction 안에서 외부 호출을 제거하거나 outbox로 분리한 설계를 말하면 강합니다.

DBD-09

Q. Connection Pool은 왜 필요하고 어떤 문제가 생기나요?

핵심 개념

Connection Pool은 DB connection 생성 비용을 줄이고 동시 접근을 제어하지만, 설정을 잘못하면 DB를 과부하시키거나 app request를 대기시킵니다.

개념 해부

Connection Pool은 DB connection 생성 비용을 줄이고 동시 접근 수를 제한하는 backpressure 장치입니다. pool이 너무 작으면 대기가 늘고, 너무 크면 DB가 context switching과 lock 경합으로 무너질 수 있습니다. leak, slow query, 긴 transaction은 pool 고갈로 먼저 드러나는 경우가 많습니다.

30초 답변

DB connection은 생성 비용이 크고 DB가 처리할 수 있는 수가 제한적이므로 pool로 재사용합니다. 하지만 app instance가 늘 때 pool size도 곱해져 DB max connection을 초과할 수 있습니다. pool size, timeout, transaction duration, slow query를 함께 봐야 합니다.

90초 답변

connection pool 문제는 트래픽 증가, autoscaling, slow query와 같이 나타납니다. DB connection이 부족하면 app은 pool 대기 시간이 늘고, timeout이 발생하며, retry가 더 큰 부하를 만들 수 있습니다. 해결은 pool size를 무작정 키우는 것이 아니라 query 최적화, transaction 단축, read/write 분리, pooler 사용, backpressure를 함께 검토하는 것입니다.

꼬리질문

질문답변 방향
pool size는 어떻게 정하나요?DB max connection, app instance 수, query duration, CPU/I/O 병목을 기준으로 조정합니다.
connection leak은?반납 누락, transaction 미종료, 예외 경로를 확인합니다.
retry 폭주는?timeout, exponential backoff, circuit breaker, request shedding을 봅니다.

오답/주의점

주의

DB가 느리다고 pool만 키우면 병목을 DB로 밀어 넣어 장애를 키울 수 있습니다.

DBD-10

Q. WAL과 replication은 어떤 역할을 하나요?

핵심 개념

WAL은 변경 내용을 데이터 파일보다 먼저 로그에 기록해 crash recovery와 replication의 기반이 됩니다. replication은 가용성과 읽기 확장을 돕지만 복구와 백업을 대체하지 않습니다.

개념 해부

WAL은 변경을 먼저 순차 로그로 남겨 crash recovery와 replication의 기반을 제공합니다. replication은 읽기 확장과 장애 대응에 유리하지만 lag가 있으면 stale read와 read-your-writes 문제가 생깁니다. 복제본은 백업이 아니며, 잘못 삭제한 데이터도 빠르게 복제될 수 있습니다.

30초 답변

WAL은 DB 변경을 먼저 로그로 남겨 장애 후 복구할 수 있게 합니다. replica는 WAL을 받아 primary 변경을 따라가며 읽기 부하 분산이나 failover에 쓰입니다. 하지만 replication lag가 있으면 replica read가 stale할 수 있고, 논리적 삭제나 잘못된 update는 replica에도 전파됩니다.

90초 답변

replication은 고가용성 도구이지 과거 시점 복구 도구가 아닙니다. primary에서 잘못 삭제한 데이터는 replica에도 반영될 수 있으므로 backup과 PITR이 필요합니다. read replica를 사용할 때는 read-after-write 요구가 있는 기능을 primary에서 읽거나 session consistency를 보장해야 합니다.

꼬리질문

질문답변 방향
replication lag 원인은?write 폭증, long query, network, replica 성능, WAL apply 지연을 봅니다.
read replica를 언제 쓰나요?읽기 부하 분산에 쓰지만 stale data 허용 여부를 확인합니다.
failover 후 주의점은?DNS/endpoint, connection 재시도, split brain, 데이터 손실 가능성을 봅니다.

면접에서 말할 증거

replica lag metric, PITR, failover drill, read-after-write 이슈를 설명할 수 있으면 좋습니다.

DBD-11

Q. 백업과 복구는 어떻게 검증하나요?

핵심 개념

백업은 존재하는 것만으로 충분하지 않습니다. 실제 복구에 걸리는 시간, 복구된 데이터의 정합성, 운영 전환 절차를 검증해야 합니다.

개념 해부

백업은 존재한다는 사실보다 복구해봤다는 증거가 중요합니다. RPO/RTO 목표에 맞게 full/incremental backup, PITR, restore drill을 설계해야 합니다. 백업 파일이 있어도 암호화 키, 권한, 절차, 복구 시간 검증이 없으면 실제 사고에서 쓸 수 없습니다.

30초 답변

backup은 과거 시점 복구와 실수 복구를 위해 필요하고, replication은 가용성을 위한 것입니다. RPO와 RTO를 정하고, 정기적으로 restore drill을 수행해 백업 파일이 실제로 복구되는지, 권한과 secret, app config까지 맞는지 확인해야 합니다.

90초 답변

복구 훈련에서는 DB만 띄우는 것으로 끝나면 안 됩니다. 특정 시점으로 복구한 뒤 schema version, application compatibility, user count, checksum, 핵심 쿼리, 권한, connection string 전환까지 확인해야 합니다. 개인정보와 보안 요구가 있다면 백업 암호화, 접근 통제, 보관 기간, 삭제 정책도 필요합니다.

꼬리질문

질문답변 방향

백업이 실패했는지 어떻게 아나요?백업 job success뿐 아니라 restore drill과 alert가 필요합니다.
테스트 환경 복구는?민감정보 마스킹과 접근 제한이 필요합니다.

오답/주의점

주의

“자동 백업이 켜져 있습니다”는 충분한 답변이 아닙니다. 복구 검증이 핵심입니다.

DBD-12

Q. 무중단 DB migration은 어떻게 설계하나요?

핵심 개념

무중단 migration은 구버전과 신버전 애플리케이션이 동시에 동작할 수 있도록 schema 변경을 backward compatible하게 나누는 방식입니다.

개념 해부

무중단 migration은 schema 변경과 애플리케이션 배포가 동시에 깨지지 않도록 시간을 나누는 작업입니다. expand 단계에서 새 구조를 추가하고 양쪽 코드가 호환되게 만든 뒤, backfill과 dual read/write를 거쳐 contract 단계에서 오래된 구조를 제거합니다. lock을 유발하는 DDL과 backfill 부하를 별도로 관리해야 합니다.

30초 답변

expand-contract 방식으로 진행합니다. 먼저 nullable column이나 새 table처럼 호환되는 schema를 추가하고, 앱이 양쪽 schema를 읽고 쓸 수 있게 배포합니다. 그 다음 backfill과 검증을 거쳐 not null/constraint 적용, old column 제거를 마지막에 합니다. 큰 index는 가능하면 concurrently 생성합니다.

90초 답변

위험한 변경은 column rename, type 변경, not null 추가, default backfill, 대형 table index 생성, FK validation입니다. 한 번에 바꾸면 lock과 구버전 앱 호환성 문제가 생깁니다. 따라서 migration은 배포 순서와 함께 설계해야 합니다. backfill은 batch로 나누고, 진행률과 lock wait, replication lag를 관찰합니다.

꼬리질문

질문답변 방향
NOT NULL 추가는?nullable 추가, backfill, check constraint validate, set not null 순서로 나눕니다.
column rename은?새 컬럼 추가, dual write, read fallback, 전환, old column 제거로 나눕니다.
migration rollback은?파괴적 변경은 rollback이 어려우므로 호환 단계와 feature flag가 중요합니다.

면접에서 말할 증거

dry-run, lock timeout, batch backfill, migration version, rollback plan을 말하면 운영 경험이 드러납니다.

DBD-14

MVCC INDEX PLAN DEEP DIVE

DB 심화 면접에서 강한 답변은 MVCC 내부, 인덱스 접근 조건, 실행 계획의 실제 수치를 한 문장 안에서 연결합니다.

MVCC INDEX PLAN DEEP DIVE PostgreSQL은 row version마다 xmin/xmax로 생성·삭제 transaction 정보를 기록하고 snapshot이 어떤 version을 볼 수 있는지 판단합니다. InnoDB는 read view와 undo log를 통해 consistent read를 제공하며, 오래 열린 transaction은 undo 정리를 지연시킵니다. 성능 분석에서는 btree leftmost prefix, index-only scan 가능성, EXPLAIN ANALYZE의 rows/loops/buffers를 함께 확인해야 합니다.

MVCC 내부 비교

항목PostgreSQLInnoDB면접 포인트
버전 식별tuple의 xmin/xmax가 생성·삭제 transaction을 나타냅니다.record와 undo log, transaction id를 기반으로 과거 버전을 재구성합니다.보이는 row는 “최신 row”가 아니라 snapshot 기준으로 visible한 version입니다.
Snapshot활성 transaction 목록과 xmin 경계를 기준으로 visibility를 판단합니다.read view가 low/high watermark와 active transaction을 기준으로 판단합니다.격리 수준에 따라 snapshot 생성 시점이 달라집니다.
정리 지연long transaction이 vacuum과 dead tuple 정리를 막습니다.long transaction이 undo log purge를 지연시킵니다.오래 열린 읽기 transaction도 운영 비용을 만들 수 있습니다.
Lock 관계MVCC가 reader/writer 충돌을 줄이지만 row lock과 predicate 문제는 남습니다.consistent read와 locking read가 다르고 gap/next-key lock이 중요합니다.“MVCC면 lock이 없다”는 오답을 피해야 합니다.

인덱스와 실행 계획 해석

B-tree 복합 인덱스는 leftmost prefix가 중요합니다. (tenant_id, status, created_at) 인덱스는 tenant_id 단독, tenant_id+status, tenant_id+status+created_at 조건에는 잘 맞지만 status만 있는 조건에는 일반적으로 효율이 낮습니다. range 조건이 등장한 뒤에는 뒤 컬럼의 filtering/order 활용도가 제한될 수 있으므로 equality 조건, range 조건, ORDER BY를 순서대로 설명해야 합니다.

index-only scan은 인덱스만 읽는다는 뜻처럼 보이지만, PostgreSQL에서는 visibility map이 충분히 표시되어 있어야 heap visibility 확인을 줄일 수 있습니다. 필요한 컬럼이 모두 인덱스에 있어도 최근 변경이 많거나 visibility map이 깨져 있으면 heap fetch가 생길 수 있습니다. 그래서 covering index 설계와 vacuum 상태를 함께 봐야 합니다.

EXPLAIN ANALYZE 항목읽는 방법위험 신호
rowsplanner 예상 rows와 actual rows를 비교합니다.차이가 크면 통계, skew, 조건 상관관계 문제가 있을 수 있습니다.
loopsplan node가 반복 실행된 횟수입니다.inner node loops가 크면 nested loop가 실제로 폭증했는지 봅니다.
buffersshared hit/read/dirtied/written으로 cache와 I/O를 봅니다.read가 크면 disk I/O, hit만 크면 CPU/반복 scan 병목일 수 있습니다.
actual time각 node의 시작/종료 실제 시간을 봅니다.한 node보다 반복 누적 시간이 병목일 수 있어 rows*loops로 봅니다.
답변 주의

“인덱스를 타서 빠릅니다”는 충분하지 않습니다. 어떤 prefix로 접근했고, index-only scan이 실제 heap fetch를 줄였는지, rows/loops/buffers가 예측과 맞는지를 함께 말해야 합니다.

DBD-15

DB 개념 인과관계

DB 심화 답변은 MVCC, index, isolation을 각각 외우는 것이 아니라 동시 읽기와 쓰기를 어떻게 동시에 허용하면서도 어떤 이상 현상이 남는가를 설명해야 합니다.

DB CAUSAL MODELMVCC는 쓰기 중인 row를 읽기 위해 기다리게 만들지 않고, transaction snapshot이 볼 수 있는 version을 선택하게 합니다. 그래서 읽기 동시성은 좋아지지만 오래 열린 transaction은 vacuum을 막고 dead tuple을 쌓아 bloat를 만들 수 있습니다. 인덱스는 탐색 범위를 줄이지만 쓰기마다 인덱스도 갱신해야 하므로 write amplification을 만듭니다. isolation level은 동시성 이상 현상을 줄이지만 높일수록 abort, lock, retry 비용이 커집니다. 따라서 “Serializable이 가장 안전합니다”가 아니라 어떤 불변식을 보호해야 하는지부터 말해야 합니다.

DBD-16

실행계획 실전 해석: MVCC, 인덱스, 락을 함께 읽기

DB 심화 면접에서 전문가처럼 보이려면 “인덱스를 탔습니다”가 아니라 실행 계획의 rows, loops, buffers, lock wait, MVCC visibility를 같은 현상으로 읽어야 합니다.

EXPERT DB PLAN CASE시나리오는 tenant_id, status, created_at 조건으로 주문 목록을 조회하는 API가 뒤 페이지에서 느려지는 경우입니다. 초보 답변은 “복합 인덱스를 추가합니다”입니다. 전문가 답변은 먼저 쿼리의 목적을 나눕니다. tenant 범위가 강한지, status 선택도가 높은지, date range가 넓은지, ORDER BY가 created_at desc인지, offset pagination인지, 필요한 컬럼이 인덱스에 포함되는지 확인합니다. 그 뒤 실행계획에서 planner 예상 rows와 actual rows가 얼마나 다른지, sort가 발생하는지, heap fetch가 많은지, buffers read가 큰지 봅니다.

1. estimated rows와 actual rows

planner는 통계를 기반으로 어떤 plan이 싸다고 추정합니다. 하지만 tenant별 데이터 분포가 심하게 다르거나 status와 created_at이 강하게 상관되어 있으면 예상 rows가 틀릴 수 있습니다. 예상은 1,000 row인데 실제는 500,000 row라면 nested loop가 폭발하거나 잘못된 index가 선택될 수 있습니다. 이때 해결은 단순히 인덱스를 더 만드는 것이 아니라 ANALYZE, extended statistics, query rewrite, partial index, tenant별 partitioning 가능성까지 검토하는 것입니다.

2. Index scan이 항상 좋은 것은 아니다

Index scan은 필요한 row가 적을 때 강합니다. 하지만 선택도가 낮아 테이블 대부분을 읽어야 한다면 index를 따라 random access하는 비용이 sequential scan보다 클 수 있습니다. PostgreSQL의 index-only scan도 visibility map이 충분히 설정되어 있어야 heap fetch를 줄입니다. 최근 update가 많아 visibility 확인이 필요하면 인덱스에 컬럼이 다 있어도 heap을 읽게 됩니다. 따라서 “covering index라 빠르다”가 아니라 heap fetch와 buffers를 함께 봐야 합니다.

3. Offset pagination의 비용

OFFSET 100000 LIMIT 50은 앞의 100,000개를 건너뛰어야 합니다. 사용자는 50개만 받지만 DB는 정렬된 후보를 많이 처리합니다. cursor/keyset pagination은 마지막으로 본 created_at, id를 기준으로 다음 범위를 좁히므로 뒤 페이지 비용이 안정적입니다. 다만 임의 페이지 이동이 어렵고 정렬 기준이 안정적이어야 합니다. 관리자 화면에서 “페이지 번호 이동”이 꼭 필요한지, 아니면 필터와 cursor 탐색으로 충분한지 제품 요구와 같이 판단해야 합니다.

4. MVCC와 오래 열린 transaction

MVCC는 reader와 writer 충돌을 줄이지만, 오래 열린 transaction은 오래된 row version 정리를 막습니다. PostgreSQL에서는 dead tuple이 쌓이고 vacuum이 지연되면 table bloat와 index bloat가 생깁니다. 이 상태에서는 같은 쿼리도 더 많은 page를 읽게 되어 buffers가 늘고 cache 효율이 떨어집니다. “쿼리는 그대로인데 점점 느려졌다”면 데이터 증가뿐 아니라 long transaction, autovacuum 지연, bloat를 의심해야 합니다.

5. 락과 격리 수준

실행 시간이 길어 보이는 쿼리가 실제로는 실행 중이 아니라 lock을 기다리고 있을 수 있습니다. row lock, relation lock, migration DDL lock, foreign key check가 대기 원인이 됩니다. Read Committed에서는 statement마다 snapshot이 달라질 수 있고, Repeatable Read에서는 transaction 동안 snapshot이 유지됩니다. Serializable은 더 강한 보장을 목표로 하지만 serialization failure가 발생할 수 있어 retry가 설계 일부가 됩니다. 격리 수준을 올리는 답변에는 반드시 retry, 사용자 안내, idempotency가 따라와야 합니다.

면접에서의 90초 답변

“저는 먼저 EXPLAIN ANALYZE에서 estimated rows와 actual rows 차이, loops, sort, buffers, heap fetch를 봅니다. index scan이 있어도 선택도가 낮거나 heap fetch가 많으면 느릴 수 있습니다. offset pagination이면 뒤 페이지 비용이 커지므로 keyset pagination을 검토하고, 통계가 틀리면 ANALYZE나 extended statistics를 봅니다. 동시에 lock wait와 long transaction, autovacuum 지연, bloat도 확인합니다. 격리 수준이나 락을 바꾸는 선택은 정합성 이상 현상을 막는 대신 abort와 retry 비용이 생기므로 도메인 불변식 기준으로 결정합니다.”

전문가 기준
