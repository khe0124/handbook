Gradle 의존성·빌드
실무 가이드

라이브러리를 어떻게 추가·고정·갱신하는가 — npm/package.json 하던 감각을 Gradle로 옮기는 실무 가이드. 개념 설명보다 "이걸 실제로 어떻게 치나"에 답합니다. Gradle 8.x · Spring Boot 3.x · Java 기준.

NOTE : 빌드 스크립트는 Groovy DSL(build.gradle) 기준 표기 — Kotlin DSL(.gradle.kts)은 문법이 조금 다름

GRADLE-01

Gradle이 하는 일·구조

Gradle은 JVM 세계의 빌드 도구입니다 — npm이 의존성 설치·스크립트 실행을 맡듯, Gradle은 의존성 해석 + 컴파일 + 테스트 + 패키징을 한 번에 책임집니다. 프론트엔드 감각으로 매핑부터 잡으면 빠릅니다.

npm 세계Gradle 세계역할
package.jsonbuild.gradle의존성·플러그인·빌드 설정 선언
workspaces 루트 설정settings.gradle프로젝트 이름·포함할 모듈 정의
node_modules~/.gradle/caches받은 라이브러리 저장(프로젝트별 아님, 전역 캐시)
package-lock.jsonresolved 버전(+version catalog)실제 해석된 버전 고정
npx / npm run./gradlew <task>빌드·실행 task 호출
프로젝트 루트 구조 my-app/
├─ build.gradle ← 의존성·플러그인 (≈ package.json)
├─ settings.gradle ← 프로젝트/모듈 정의
├─ gradlew · gradlew.bat ← wrapper 실행 스크립트
├─ gradle/wrapper/gradle-wrapper.properties ← 고정된 Gradle 버전
└─ src/main/java · src/test/java ← 소스

1.1gradlew(wrapper)를 왜 쓰나

gradlew는 Gradle Wrapper — 프로젝트에 고정된 Gradle 버전을 자동으로 내려받아 실행하는 스크립트입니다. 팀원마다 로컬에 설치된 Gradle 버전이 달라 빌드가 갈리는 일을 막습니다. npm으로 치면 .nvmrc로 Node 버전을 고정하는 것과 같은 동기. 로컬에 설치한 gradle 명령 대신 항상 ./gradlew를 쓰는 게 원칙입니다.

실무 습관

레포를 받으면 첫 명령은 ./gradlew build. wrapper가 알아서 맞는 Gradle 버전을 받고, 의존성을 해석하고, 컴파일·테스트까지 돕니다. Windows는 gradlew.bat, macOS/Linux는 ./gradlew. 실행 권한이 없다고 나오면 chmod +x gradlew.

GRADLE-02

의존성 추가

라이브러리 추가는 dependencies { } 블록에 한 줄을 적는 일입니다. npm처럼 CLI로 추가하는 게 아니라 파일을 직접 편집한다는 게 가장 큰 차이입니다. 적고 나면 IDE가 새로고침하거나 빌드 시 자동으로 받습니다.

build.gradle — dependencies 블록 dependencies {
  implementation 'org.springframework.boot:spring-boot-starter-web:3.3.1'
  implementation 'com.google.guava:guava:33.2.1-jre'
  testImplementation 'org.springframework.boot:spring-boot-starter-test'
}

2.1좌표 — group:name:version

모든 의존성은 좌표(coordinates)로 식별됩니다 — group:name:version 세 부분. npm의 @scope/package@version과 같은 개념이지만 group이 보통 역도메인(org.springframework.boot)입니다.

부분예시npm 대응
grouporg.springframework.bootscope(@angular)에 가까움 — 조직/네임스페이스
namespring-boot-starter-web패키지 이름
version3.3.1버전(생략 가능 — ch4)

2.2Maven Central에서 좌표 찾기

라이브러리 좌표는 search.maven.org 또는 mvnrepository.com에서 검색합니다. 라이브러리를 고르면 빌드 도구별 스니펫(Gradle Groovy / Kotlin / Maven)을 그대로 복사할 수 있습니다 — npm 페이지에서 npm install ... 줄을 복사하는 것과 동선이 같습니다.

의존성 저장소(repository)는 어디서?

의존성을 어디서 받을지는 build.gradle(또는 settings.gradle의 dependencyResolutionManagement)의 repositories { mavenCentral() }가 정합니다. npm registry에 해당하는 게 Maven Central. 사내 라이브러리를 쓴다면 회사 Nexus/Artifactory 주소를 maven { url '...' }로 추가합니다.

GRADLE-03

의존성 스코프

npm의 dependencies vs devDependencies가 둘로 갈리는 데 비해, Gradle은 더 세밀합니다 — 이 라이브러리를 언제(컴파일/런타임/테스트) 쓰고, 다른 모듈에 노출할지까지 구분합니다. 이 구분이 빌드 속도와 모듈 캡슐화를 좌우합니다.

configuration언제 쓰나npm 대응
implementation컴파일·런타임 모두 필요. 기본값 — 대부분 이걸 씀dependencies
api컴파일·런타임 필요 + 이 모듈을 쓰는 쪽에도 전파(라이브러리 모듈에서만)(직접 대응 없음)
compileOnly컴파일에만 필요, 런타임 산출물엔 미포함(예: lombok, 서블릿 API)devDependencies 일부
runtimeOnly런타임에만 필요, 컴파일 땐 불필요(예: DB 드라이버 runtimeOnly 'org.postgresql:postgresql')(런타임 전용)
testImplementation테스트 코드 컴파일·실행에만devDependencies(테스트)
annotationProcessor컴파일 시 코드 생성(예: lombok, MapStruct, Spring 설정 메타)(빌드타임 코드젠)

3.1implementation vs api — 핵심 차이

둘 다 컴파일·런타임에 쓰지만 전파(transitivity) 노출이 다릅니다. 멀티모듈일 때만 의미가 있습니다.

A 모듈이 guava를 쓰고, B가 A를 의존할 때 implementation 'guava' → B는 guava를 직접 못 봄 (A 내부에 캡슐화)
api 'guava'        → B도 guava를 컴파일 시 직접 사용 가능 (공개 전파)

기본은 항상 implementation입니다 — 캡슐화가 잘 되고, 의존성이 바뀌어도 그 모듈을 쓰는 쪽을 재컴파일할 필요가 줄어 빌드가 빠릅니다. api는 "내 public API의 타입이 이 라이브러리 타입을 노출한다"(예: 메서드 반환 타입이 guava의 ImmutableList)일 때만 씁니다.

잘못 고르면

전부 api로 두면 모든 모듈이 서로의 의존성을 보게 돼 캡슐화가 무너지고 빌드가 느려집니다. 반대로 노출해야 할 타입을 implementation에 두면 그 모듈을 쓰는 쪽에서 "타입을 찾을 수 없다" 컴파일 에러가 납니다. 의심되면 implementation, 컴파일 에러가 나면 그때 api로 승격.

GRADLE-04

버전 관리

버전을 어디에 어떻게 적느냐가 유지보수의 질을 가릅니다. Spring Boot는 버전을 대신 정해주고, version catalog는 버전을 한 곳에 모읍니다. 둘 다 "버전 숫자가 코드 여기저기 흩어지는" 문제를 푸는 도구입니다.

4.1Spring Boot 의존성 관리 — 버전 생략

io.spring.dependency-management 플러그인(또는 Spring Boot 플러그인)이 적용되면, Spring이 검증한 라이브러리들은 버전을 적지 않아도 됩니다. Spring Boot의 BOM(Bill of Materials)이 서로 호환되는 버전 세트를 보장합니다.

버전을 생략한 의존성 dependencies {
  implementation 'org.springframework.boot:spring-boot-starter-web' // 버전 X
  implementation 'org.springframework.boot:spring-boot-starter-data-jpa'
  runtimeOnly 'org.postgresql:postgresql' // 버전도 BOM이 관리
}

버전이 보이지 않아 처음엔 당황스럽지만, Spring Boot 버전 하나만 올리면 수십 개 라이브러리가 호환 세트로 함께 올라간다는 게 핵심 이점입니다. BOM에 없는 라이브러리만 명시적으로 버전을 적으면 됩니다.

4.2version catalog — libs.versions.toml

여러 모듈에서 같은 라이브러리를 쓸 때 버전을 한 곳에서 관리하는 표준 방식이 version catalog입니다. gradle/libs.versions.toml 파일에 버전·라이브러리를 정의하고, 빌드 스크립트에서 libs.xxx로 참조합니다. npm의 lockfile + 중앙 관리 느낌.

gradle/libs.versions.toml [versions]
  guava = "33.2.1-jre"

[libraries]
  guava = { module = "com.google.guava:guava", version.ref = "guava" }

build.gradle — 참조 dependencies {
  implementation libs.guava
}
언제 무엇

단일 모듈 + Spring Boot라면 BOM만으로 충분 — version catalog는 과합니다. 멀티모듈에서 여러 모듈이 같은 라이브러리 버전을 공유해야 할 때 catalog가 진가를 냅니다. 버전 업그레이드가 .toml 한 줄 수정으로 끝납니다.

GRADLE-05

플러그인

플러그인은 Gradle에 능력을 더합니다 — Java 컴파일, Spring Boot 패키징, 의존성 버전 관리 같은 기능이 플러그인으로 들어옵니다. plugins { } 블록에 선언하며, 보통 파일 맨 위에 둡니다.

build.gradle — plugins 블록 plugins {
  id 'java'
  id 'org.springframework.boot' version '3.3.1'
  id 'io.spring.dependency-management' version '1.1.5'
}
플러그인하는 일
javaJava 컴파일·테스트·JAR 생성 task 추가. compileJava, test, jar 등이 생김(코어 플러그인이라 version 불필요)
org.springframework.boot실행 가능한 fat JAR 패키징(bootJar), 앱 실행(bootRun), Spring Boot BOM 제공
io.spring.dependency-managementBOM 기반 버전 관리 활성화 — 이게 있어야 ch4의 "버전 생략"이 동작
코어 vs 커뮤니티 플러그인

java처럼 Gradle에 내장된 코어 플러그인은 version이 필요 없습니다. 외부에서 받는 플러그인(org.springframework.boot 등)은 version을 명시해야 Gradle Plugin Portal에서 받아옵니다. Spring Initializr로 프로젝트를 생성하면 이 세 줄이 기본으로 들어가 있습니다.

GRADLE-06

의존성 확인·트러블슈팅

"이 라이브러리가 왜 들어왔지", "버전이 왜 이게 됐지"를 추적하는 도구들. transitive dependency(간접 의존성)가 얽히면 눈으로는 못 풉니다 — Gradle이 답을 알려줍니다.

명령의미 / 형태
./gradlew dependencies전체 의존성 트리 출력. --configuration runtimeClasspath로 특정 스코프만
./gradlew dependencyInsight --dependency guava특정 라이브러리가 왜·어떤 버전으로 들어왔는지 역추적. 충돌 진단의 핵심
./gradlew :app:dependencies멀티모듈에서 특정 모듈(app)의 트리만
./gradlew build --refresh-dependencies캐시 무시하고 의존성을 다시 받음(레포 갱신 안 잡힐 때)

6.1버전 충돌과 강제

서로 다른 라이브러리가 같은 의존성의 다른 버전을 요구하면 충돌합니다. Gradle의 기본 전략은 가장 높은 버전 선택(npm의 중첩 설치와 달리 JVM은 클래스패스에 한 버전만 존재 가능). 특정 버전으로 고정해야 하면 강제합니다.

build.gradle — 버전 강제 dependencies {
  implementation('com.google.guava:guava') {
    version { strictly '33.2.1-jre' } // 이 버전으로 고정
  }
}

// 또는 전역 resolutionStrategy
configurations.all {
  resolutionStrategy.force 'com.google.guava:guava:33.2.1-jre'
}
디버깅 순서

"클래스를 못 찾는다 / 메서드가 없다"(NoSuchMethodError)는 거의 버전 충돌입니다. ① dependencyInsight --dependency 문제라이브러리로 어떤 버전이 선택됐고 누가 끌고 왔는지 확인 → ② 원치 않는 transitive를 exclude group: '...', module: '...'로 빼거나 strictly로 고정. 추측으로 버전을 바꾸지 말고 트리를 먼저 보세요.

GRADLE-07

자주 쓰는 task

Gradle은 "task" 단위로 동작합니다 — npm run의 스크립트에 해당. 핵심 몇 개만 손에 붙이면 일상은 충분합니다. 항상 ./gradlew로 호출하세요.

task하는 일 / npm 대응
./gradlew build컴파일 + 테스트 + 패키징 전체. (≈ npm run build + test)
./gradlew cleanbuild/ 산출물 삭제. 꼬였을 때 clean build로 처음부터
./gradlew test테스트만 실행. --tests "*.UserServiceTest"로 특정 테스트만
./gradlew bootRunSpring Boot 앱을 실행(개발 중 띄우기). (≈ npm start)
./gradlew bootJar실행 가능한 fat JAR 생성 → java -jar build/libs/*.jar로 배포
./gradlew tasks사용 가능한 task 목록(설명 포함). 모를 땐 여기부터
./gradlew --stop백그라운드 Gradle 데몬 종료(메모리·캐시 꼬임 해소)
유용한 플래그

--info/--debug(상세 로그), --stacktrace(에러 스택), -x test(테스트 스킵 — 예: build -x test), --offline(네트워크 없이 캐시만). 빌드가 이상하게 빠르면 Gradle이 캐시한 결과를 재사용한 것 — 정상이지만 강제로 다시 돌리려면 clean 후 빌드.

GRADLE-08

멀티모듈 기초

앱이 커지면 하나의 빌드를 여러 모듈로 쪼갭니다 — core(도메인), api(웹), batch(배치) 식으로. npm의 monorepo/workspaces와 같은 동기. settings.gradle이 모듈 구성을, 각 모듈의 build.gradle이 자기 의존성을 책임집니다.

settings.gradle — 모듈 포함 rootProject.name = 'my-app'
include 'core', 'api', 'batch'
api/build.gradle — 다른 모듈 의존 dependencies {
  implementation project(':core') // core 모듈을 의존
  implementation 'org.springframework.boot:spring-boot-starter-web'
}

외부 라이브러리는 좌표(group:name:version)로, 같은 빌드 안의 내부 모듈은 project(':core')로 참조합니다. 콜론(:)은 프로젝트 경로 구분자 — 루트 기준 경로입니다. 여기서 ch3의 api vs implementation 구분이 실제로 의미를 갖습니다.

순환 의존 금지

api → core는 되지만 동시에 core → api를 만들면 순환 의존(circular dependency)으로 빌드가 실패합니다. 모듈 의존은 한 방향으로 흐르게 설계하세요 — 보통 core는 아무도 의존하지 않는 가장 안쪽에 둡니다. 공통 코드는 별도 모듈로 빼서 양쪽이 그것을 의존하게 합니다.

GRADLE-09

치트시트 — 자주 쓰는 한 줄

실전에서 반복되는 조합들. 외우기보다 "이런 게 가능하다"를 알아두고 필요할 때 찾으세요.

의존성 추가dependencies
implementation 'group:name:version' 를 dependencies {} 안에. Spring Boot면 version 생략 가능
기본 스코프implementation
대부분 implementation. 컴파일만=compileOnly, 런타임만=runtimeOnly, 테스트=testImplementation, 코드젠=annotationProcessor
api vs implementation전파 여부
api는 의존하는 쪽에도 노출, implementation은 캡슐화. 기본은 implementation, public 타입 노출 시에만 api
DB 드라이버runtimeOnly
runtimeOnly 'org.postgresql:postgresql' — 컴파일엔 불필요, 실행 시에만 필요
version cataloglibs.versions.toml
gradle/libs.versions.toml에 정의 → implementation libs.guava로 참조. 멀티모듈 버전 중앙 관리
의존성 트리dependencies
./gradlew dependencies — 전체 트리. --configuration runtimeClasspath로 스코프 한정
왜 이 버전?dependencyInsight
./gradlew dependencyInsight --dependency guava — 누가 끌고 왔고 어떤 버전이 선택됐는지
버전 강제resolutionStrategy

캐시 갱신--refresh-dependencies
./gradlew build --refresh-dependencies — 레포 변경이 안 잡힐 때 강제 재다운로드
앱 실행/패키징bootRun / bootJar

./gradlew clean build — 산출물 지우고 새로. 테스트 스킵은 -x test
데몬 종료--stop
./gradlew --stop — 빌드가 이상하게 꼬이거나 캐시 문제 의심 시 데몬 재시작
내부 모듈 의존project(':core')
implementation project(':core') — 같은 빌드의 다른 모듈. settings.gradle에 include 'core' 필요
wrapper 우선./gradlew
로컬 gradle 대신 항상 ./gradlew — 팀과 같은 Gradle 버전 보장. task 목록은 ./gradlew tasks
