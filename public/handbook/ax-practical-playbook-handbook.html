AX 실무 플레이북

이 문서는 AI를 매일 개발 업무에 넣을 때 쓰는 실행 절차와 교육 과정입니다. 목표는 빠른 생성이 아니라 제한된 시간과 비용 안에서 검증 가능한 변경을 만드는 것입니다.

PLAY-01

AI Coding Workflow

Scope→ Context→ Plan→ Test→ Implement→ Verify→ Review→ Reflect→ Ship
단계좋은 기준
Scope목표, 비목표, 성공 조건, 중단 조건을 쓴다
Context관련 파일, 실패 로그, 설정, 최근 diff를 좁힌다
Plan원인 가설, 추가할 테스트, 검증 명령, 리스크를 적는다
Test가능하면 실패 테스트를 먼저 만든다
Implement한 번에 하나의 원인 가설만 수정한다
Verify테스트, 빌드, 수동 시나리오, 보안 속성을 확인한다
Review고위험 변경은 독립 리뷰를 거친다
Reflect틀린 가설과 효과적 검증을 기록한다
Ship변경 이유, 검증 결과, 남은 위험을 짧게 설명한다
PLAY-02

리뷰 루프

  1. 변경 목적을 한 문장으로 고정합니다.
  2. diff만 주지 말고 관련 요구사항과 금지사항을 함께 줍니다.
  3. 먼저 risk map을 요구합니다: authZ, data migration, user-visible behavior, concurrency, external API.
  4. findings는 severity, evidence, affected path, reproduction으로 받습니다.
  5. AI의 finding을 그대로 믿지 말고 코드 위치와 실행 가능성을 확인합니다.
  6. accepted, rejected, needs follow-up으로 triage합니다.
  7. episode package로 남깁니다.
PLAY-03

테스트·마이그레이션 루프

시나리오실행 흐름
버그 수정재현 테스트 먼저 → agent 수정 → 독립 테스트 → regression episode 저장
리팩터링불변조건 선언 → 작은 diff → behavior test → public API 확인
테스트 작성기존 failure mode 수집 → edge case 생성 → adversarial 관점 추가
마이그레이션expand-contract 확인 → rollback/forward plan → staging dry-run → 승인
보안 점검source/sink 추적 → exploitability 판단 → false positive 근거 기록
운영 런북증상 → 진단 → 완화 → 복구 확인 → audit trail
PLAY-04

프롬프트 템플릿

TASK PROMPT TEMPLATE 목표: [구체적 목표]
범위: [관련 파일/모듈], [수정하지 말아야 할 부분]
성공 조건: [테스트, 동작, 성능, 보안 조건]
작업 방식:
1. 먼저 관련 컨텍스트를 요약해라.
2. 원인 가설을 3개 이하로 제시해라.
3. 구현 전 계획을 작성해라.
4. 가능한 경우 실패 테스트를 먼저 추가해라.
5. 검증 결과 없이는 완료라고 말하지 마라.
중단 조건: [반복 횟수, 실패 조건, 비용 한계]
REVIEW PROMPT TEMPLATE 너는 독립 코드 리뷰어다. 이 변경을 작성하지 않았다고 가정해라.
요약보다 결함 탐지를 우선해라.
검토 기준: 실제 버그, 회귀 가능성, 보안/권한 문제, 누락 테스트, 과도한 변경, 요구사항 불일치
출력: Findings, Required Fixes, Test Gaps, Residual Risk

PLAY-06

업무 자동화 운영

실무에서 AX 자동화는 한 번에 완성하지 않습니다. 반복 업무를 수집하고, 작은 하네스로 실험하고, 검증 지표를 보고, 팀 표준으로 남길 것만 확장합니다.

단계소규모 셀대규모 조직
업무 수집스프린트마다 반복 업무 1개 선정여러 팀의 toil, review latency, incident pattern 수집
하네스 작성task template, context checklist, test command로 시작공통 context builder, policy gate, model gateway 제공
실행 범위read-only 분석과 작은 diff부터업무 위험도별 Autonomy ladder 운영
검증테스트, 빌드, 수동 시나리오, 팀 리뷰CI, eval set, audit log, risk-based approval
확산잘 된 template만 팀 문서에 저장golden workflow로 제품군/부서에 확장
2-WEEK AX AUTOMATION SPRINT Day 1: 반복 업무 1개 선정, baseline 측정
Day 2: task spec, context package, permission policy 작성
Day 3-5: read-only 분석 하네스 실행, 실패 원인 기록
Day 6-8: 작은 write 권한 또는 PR draft 허용
Day 9: verification gate와 review rubric 보강
Day 10: lead time, review latency, rework, cost per episode로 지속 여부 결정
4-WEEK AX ENGINEER TRAINING Week 1 교육 과정: developer workshop과 reviewer workshop으로 scope, context, review loop 실습
Week 2: security workshop에서 Prompt Injection 방어와 permission boundary 실습
Week 3: manager workshop과 platform owner workshop으로 metric, governance, rollout 운영 설계
Week 4 Capstone: Trace-based Eval Harness, AI Code Review Pipeline, Autonomy Ladder, Failure Taxonomy 중 하나를 실제 팀 workflow에 적용
TEMPLATES INDEX developer workshop: task prompt, context checklist, verification note

중단 기준

AI 자동화가 리뷰 시간을 늘리거나, rework를 만들거나, 팀이 결과를 설명하지 못하면 확장하지 않습니다. 이 경우 모델을 바꾸기 전에 업무 분해, 컨텍스트, 검증 기준을 먼저 고칩니다.
