DB 스키마·로컬 시딩
실무 가이드

스키마를 안전하게 관리하고 로컬 데이터를 자동 복구하는 실무 가이드. "실제로 어떻게 하나"에 초점을 둡니다 — 누가 스키마를 소유하는지, 마이그레이션을 어떻게 쌓는지, 로컬 DB를 날려도 한 번의 재구동으로 복구되게 만드는 법까지. Spring Boot · JPA · Flyway · PostgreSQL 기준.

STACK : Spring Boot · JPA(Hibernate) · Flyway · PostgreSQL
COMPILED : 2026-06
SEED-01

스키마는 누가 관리하나

먼저 결정할 한 가지: 테이블 구조의 단일 진실원천(source of truth)은 무엇인가. JPA 엔티티가 자동으로 테이블을 만들게 둘 것인가, 아니면 사람이 쓴 SQL 마이그레이션이 정본인가. 정답은 후자입니다. 이유부터 봅니다.

Spring Boot에서 Hibernate의 spring.jpa.hibernate.ddl-auto 설정은 애플리케이션이 뜰 때 엔티티를 보고 DDL을 어떻게 처리할지 결정합니다. 편해 보이지만 운영에서는 위험합니다 — 엔티티 변경이 곧 예측 불가능한 ALTER로 번역되기 때문입니다.

ddl-auto동작위험성 / 용도
none아무것도 안 함가장 안전. 스키마는 Flyway가 전담. 운영 권장
validate엔티티와 실제 테이블이 맞는지 검증만, 불일치면 기동 실패변경은 안 하되 어긋남을 잡아줌. 운영 권장
update차이를 감지해 ALTER로 맞춤컬럼 삭제·타입 변경을 못 하고, 적용 순서·결과가 불투명. 운영 절대 금지
create기동 때 테이블을 DROP 후 재생성기존 데이터 전부 소실. 일회성 실험용
create-drop기동 때 생성, 종료 때 DROP데이터 휘발. 단발 테스트 외 금지
RECOMMENDED — application.yml (운영/공통) spring:
  jpa:
    hibernate:
      ddl-auto: validate  # 엔티티 ↔ 실제 스키마 불일치를 기동 시점에 잡음
  flyway:
    enabled: true  # 스키마의 단일 진실원천은 Flyway 마이그레이션
운영에서 update 금지

update는 컬럼 삭제, 타입 축소, 제약 변경을 수행하지 못합니다 — 더하기만 가능하고, 그나마도 DB 벤더·Hibernate 버전에 따라 결과가 달라집니다. "로컬에선 잘 되던 게 운영에서 ALTER가 다르게 나가는" 사고의 단골입니다. 스키마 변경은 버전 매겨진 SQL 마이그레이션으로만. 엔티티는 validate로 "내가 아는 구조와 실제가 같은가"만 확인하게 두세요.

정리하면 스키마의 주인은 마이그레이션 도구이고, JPA는 그 결과를 읽어 매핑하는 소비자입니다. 다음 장에서 그 도구를 봅니다.

SEED-02

마이그레이션 도구 — Flyway

Flyway는 버전 매겨진 SQL 파일을 순서대로, 한 번씩만 적용하고 그 이력을 DB에 기록합니다. "지금 이 DB가 몇 번 버전까지 와 있는가"를 도구가 알기 때문에, 어떤 환경에서 시작하든 같은 최종 스키마에 도달합니다.

2.1의존성

Spring Boot는 클래스패스에 Flyway가 있으면 기동 때 자동으로 마이그레이션을 돌립니다(spring.flyway.enabled=true가 기본). PostgreSQL은 별도 모듈이 필요합니다.

build.gradle (Gradle · Kotlin DSL이면 괄호 표기로) implementation 'org.flywaydb:flyway-core'
implementation 'org.flywaydb:flyway-database-postgresql'  # Flyway 10+ PostgreSQL 지원 모듈
# 버전은 Spring Boot BOM이 관리 → 명시 불필요

2.2위치와 네이밍 규칙

마이그레이션 파일은 src/main/resources/db/migration에 둡니다. 파일명이 곧 메타데이터라 규칙이 엄격합니다.

FILE NAMING — V<버전>__<설명>.sql V1__init_emission_schema.sql  # V + 버전 + 언더스코어 2개 + 설명
V2__add_credit_status_column.sql
V2.1__backfill_credit_status.sql  # 점 표기로 세부 버전도 가능
R__seed_reference_data.sql  # R = repeatable, 체크섬 바뀌면 매번 재적용(→ SEED-04)
요소규칙
접두 Vversioned 마이그레이션. 한 번만 적용되고 순서대로 진행
버전 번호1, 2, 2.1 등. 정렬 순서가 곧 적용 순서
구분자 __언더스코어 두 개 — 버전과 설명을 가름. 하나면 인식 안 됨
설명단어는 _로 연결. 사람이 읽는 용도

2.3적용 순서와 이력 테이블

Flyway는 처음 돌 때 flyway_schema_history 테이블을 만들고, 적용한 스크립트마다 버전·체크섬·실행시각·성공여부를 한 행씩 기록합니다. 다음 기동 때는 이 표를 읽어 아직 적용 안 된 버전만 순서대로 실행합니다.

flyway_schema_history (개념 — 일부 컬럼) installed_rank | version | description | checksum | success
        1    |   1   | init_emission | -1740… |  t
        2    |   2   | add_credit_… |  913… |  t
불변 원칙 — 적용된 스크립트는 고치지 않는다

한 번 적용돼 이력에 기록된 V 파일은 절대 내용을 바꾸지 않습니다. Flyway는 기동 때 파일 내용의 체크섬을 다시 계산해 이력의 값과 대조하는데, 다르면 Migration checksum mismatch로 기동을 막습니다. 이미 적용된 마이그레이션을 수정하면 다른 환경(동료 로컬·운영)에서 충돌이 터집니다. 변경이 필요하면 기존 파일을 고치지 말고 새 V 파일을 추가하세요. 잘못 머지된 파일을 정리할 때만 flyway repair로 체크섬을 재기록합니다 — 평소엔 쓸 일이 없어야 정상입니다.

SEED-03

마이그레이션 작성 패턴

마이그레이션은 결국 SQL입니다. 자주 쓰는 형태 몇 개와, 운영에서 다운타임 없이 바꾸는 사고법(expand-contract)을 봅니다.

3.1기본 DDL

V2__add_credit_status_column.sql -- 컬럼 추가: 운영 데이터가 있으면 NOT NULL은 기본값과 함께
ALTER TABLE credit
  ADD COLUMN status VARCHAR(20) NOT NULL DEFAULT 'PENDING';

-- 조회 패턴에 맞는 인덱스. CONCURRENTLY는 락을 덜 잡음(아래 주의)
CREATE INDEX idx_credit_status ON credit (status);

-- 제약 추가: FK로 무결성 보장
ALTER TABLE credit
  ADD CONSTRAINT fk_credit_project
  FOREIGN KEY (project_id) REFERENCES project (id);
변경핵심 / 왜
컬럼 추가운영 테이블에 NOT NULL을 그냥 붙이면 기존 행 때문에 실패 → DEFAULT를 함께
인덱스WHERE·JOIN·ORDER BY에 쓰이는 컬럼에. 없으면 풀스캔, 과하면 쓰기 비용
제약(FK·UNIQUE)데이터 무결성을 DB가 강제 — 애플리케이션 버그가 더러운 데이터로 굳는 걸 막음
PostgreSQL — 인덱스와 락

큰 운영 테이블에 그냥 CREATE INDEX를 걸면 그동안 쓰기가 막힙니다. CREATE INDEX CONCURRENTLY는 락을 거의 안 잡지만 트랜잭션 안에서 실행할 수 없습니다. Flyway는 마이그레이션을 기본적으로 트랜잭션으로 감싸므로, 해당 스크립트 상단에 -- flyway:executeInTransaction=false를 두거나 그 마이그레이션을 분리해야 합니다.

3.2무중단 변경 — expand-contract

컬럼 이름 변경, 타입 변경처럼 "구버전 앱과 신버전 앱이 잠깐 함께 도는" 배포에서 한 방에 바꾸면 한쪽이 깨집니다. 확장(expand) → 이행 → 수축(contract)으로 단계를 나눕니다.

1 · EXPAND→ 2 · 양쪽 쓰기→ 3 · 백필→ 4 · 읽기 전환→ 5 · CONTRACT
단계하는 일 / 왜
1 EXPAND새 컬럼을 nullable로 추가. 구버전 앱은 모르고도 정상 동작
2 양쪽 쓰기신버전 앱이 옛 컬럼·새 컬럼에 동시에 기록 — 어느 쪽을 읽어도 일관
3 백필기존 행의 새 컬럼을 채우는 마이그레이션 실행
4 읽기 전환새 컬럼을 읽도록 배포. 모든 인스턴스가 신버전이 된 뒤
5 CONTRACT옛 컬럼 삭제·제약 강화(NOT NULL). 되돌아갈 필요가 없어진 뒤
핵심 — 한 마이그레이션 = 안전한 한 걸음

각 단계가 별도 배포·별도 V 파일입니다. "옛 컬럼 삭제"는 새 코드가 완전히 깔리고 롤백 가능성이 사라진 다음에야 합니다. 한 PR에서 컬럼을 추가하고 같은 PR에서 옛 컬럼을 지우면 expand-contract가 아니라 그냥 위험한 한 방입니다.

SEED-04

로컬 시딩 방법 비교

시딩은 "스키마(빈 테이블)에 개발용 데이터를 채우는" 일입니다. 마이그레이션과는 목적이 다릅니다 — 마이그레이션은 구조, 시딩은 샘플 데이터. Spring 생태계에서 흔한 세 방법을 비교합니다.

방법동작장점 / 단점
data.sqlsrc/main/resources/data.sql을 기동 때 실행가장 간단. 단, 멱등성 없음 → 재실행 시 중복·PK 충돌. 로직 불가
시더 컴포넌트
(CommandLineRunner·ApplicationRunner)		기동 후 Java 코드로 데이터 삽입조건 분기·멱등 처리 가능, Repository 재사용. 코드를 직접 써야 함
Flyway R__seed.sqlrepeatable 마이그레이션으로 시드 SQL마이그레이션 파이프라인에 통합, 체크섬 바뀌면 재적용. 멱등은 SQL로 직접 보장
data.sql과 Flyway는 충돌하기 쉽다

Flyway를 쓰는 프로젝트에서 data.sql(Spring SQL 초기화)을 함께 켜면 실행 시점이 어긋납니다 — 기본 설정에서 Hibernate/Spring의 SQL 초기화가 Flyway보다 먼저 돌아 "테이블 없음" 에러가 날 수 있습니다. 굳이 함께 쓰려면 spring.jpa.defer-datasource-initialization=true로 순서를 미뤄야 합니다. 깔끔한 선택은 시드도 Flyway로 일원화(R__)하거나, Java 시더로 분리하는 것입니다. 환경도 가려 받으세요 — 시더에 @Profile("local")을 붙여 운영에서는 절대 안 돌게.

권장 분리 — 환경별 책임 구조(모든 환경)    Flyway V__  <- 마이그레이션
참조 데이터(공통)   Flyway R__  <- 코드값·ENUM 테이블 등 모든 환경 필요분
개발용 샘플(local)  @Profile("local") 시더  <- 로그인 계정·더미 거래 등
SEED-05

멱등 시더 작성

멱등(idempotent)은 몇 번 실행해도 결과가 같은 성질입니다. 시더가 멱등이면 "DB를 날렸다가 재구동" 한 번으로 항상 같은 깨끗한 상태로 돌아옵니다 — 안전한 비상 복구의 전제입니다. 멱등이 아니면 두 번째 실행에서 PK 충돌로 기동이 깨집니다.

5.1존재 확인 후 저장

LocalDataSeeder.java — existsBy 패턴 @Component
@Profile("local")  // 운영 프로파일에선 빈 자체가 안 생김
public class LocalDataSeeder implements CommandLineRunner {

  private final ProjectRepository projects;

  @Override
  public void run(String... args) {
    // 이미 있으면 건너뜀 -> 재실행해도 행이 안 늘어남
    if (projects.existsByCode("GM-001")) return;

    projects.save(new Project("GM-001", "지역 포인트 정산 시범"));
  }
}

5.2고정 ID + upsert

행이 늘지 않는 것만이 아니라 내용까지 항상 같게 맞추려면, 고정 ID로 "있으면 갱신, 없으면 삽입"(upsert)을 합니다. 시드를 고치면 다음 실행 때 갱신이 반영됩니다.

R__seed_reference.sql — Postgres ON CONFLICT -- 고정 ID로 upsert: 재적용해도 행 수·값이 동일
INSERT INTO credit_status (id, code, label)
VALUES
  (1, 'PENDING', '심사대기'),
  (2, 'APPROVED', '승인')
ON CONFLICT (id) DO UPDATE
  SET code = EXCLUDED.code, label = EXCLUDED.label;
전략언제 / 보장 범위
existsByX 후 저장"없을 때만 1회". 코드 분기 필요 시. 내용 변경은 반영 안 됨
upsert(ON CONFLICT / MERGE)행 수 + 내용까지 고정. 시드를 고치면 갱신 전파
고정 ID 사용upsert·FK 참조의 기준점. 자동증가에 맡기면 매번 ID가 바뀜
왜 멱등이어야 하나

비멱등 시더는 첫 실행만 성공하고 두 번째부터 깨지는 지뢰입니다. 멱등이면 시더를 항상 켜둘 수 있고, 누구든 로컬 DB를 날렸다 재구동하면 같은 상태로 복구됩니다. "내 컴퓨터에선 되는데" 문제의 큰 축 하나를 제거합니다 — 모두가 동일한 시드 결과 위에서 일하기 때문입니다.

SEED-06

로컬 리셋 워크플로우

로컬 데이터가 꼬였을 때 가장 빠른 회복은 "깨끗이 지우고 처음부터 다시 세우기"입니다. 멱등 시더(→ SEED-05)와 Flyway(→ SEED-02)가 갖춰져 있으면 이건 명령 몇 줄로 끝납니다.

앱 정상 종료→ DB 초기화→ 재구동→ Flyway 재적용→ 시더 자동 복구
reset-local-db.sh — 로컬 전용 (docker-compose Postgres 예시) #!/usr/bin/env bash
set -euo pipefail

# 1) 앱 종료는 사람이 먼저 (열린 커넥션 정리)
# 2) 스키마 통째로 비우기 — public 스키마를 drop 후 재생성
docker compose exec -T db psql -U postgres -d app_local \
  -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"

# 3) 재구동하면 Flyway가 V__ 전부 재적용, @Profile("local") 시더가 데이터 복구
./gradlew bootRun --args='--spring.profiles.active=local'

볼륨째 날리는 방식(docker compose down -v 후 up)도 같은 결과를 줍니다 — DB 컨테이너의 데이터 볼륨을 삭제하면 다음 기동 때 빈 DB로 시작합니다. 어느 쪽이든 핵심은 재구동 한 번이 Flyway 적용 + 시드 복구를 자동으로 한다는 것입니다.

운영 절대 금지 — 이 스크립트는 로컬 전용

DROP SCHEMA public CASCADE는 모든 테이블과 데이터를 영구 삭제합니다. 운영 DB에서 실행되면 복구 불가입니다. 가드를 거세요 — ① 접속 문자열을 localhost/app_local로 하드코딩하고 운영 호스트면 즉시 종료, ② 스크립트 첫 줄에서 SPRING_PROFILES_ACTIVE != local이면 exit 1, ③ 운영 자격증명은 이 스크립트가 읽는 환경에 아예 두지 않기. 시더에 @Profile("local")을 단 것과 같은 방어선을 셸 레벨에도 둡니다.

실무 플레이북

"로컬 DB가 이상해요"의 표준 대응: ① 앱 종료 → ② reset-local-db.sh → ③ 재구동. 3분이면 끝나고, 결과는 누구 컴퓨터든 동일합니다. 이걸 README가 아니라 한 줄 스크립트로 만들어 두는 게 핵심 — 사람이 절차를 기억하지 않아도 되게. 마이그레이션이 꼬여 충돌이 났을 때도 가장 빠른 복구는 대개 전체 리셋입니다(운영이 아니므로 데이터를 잃어도 됩니다).

SEED-07

테스트 데이터

테스트는 또 다른 환경입니다 — 빠르고, 격리되고, 매번 같은 상태여야 합니다. H2 같은 인메모리 DB는 빠르지만 Postgres와 SQL 방언이 달라 "테스트는 통과하는데 운영에서 깨지는" 함정을 만듭니다. 진짜 DB로 테스트하는 게 정석입니다.

7.1Testcontainers — 진짜 DB로

Testcontainers는 테스트 중에 실제 PostgreSQL을 Docker 컨테이너로 띄웁니다. 운영과 같은 엔진·같은 Flyway 마이그레이션을 쓰므로, 방언 차이로 인한 거짓 통과가 사라집니다.

RepositoryIT.java — Spring Boot + Testcontainers @SpringBootTest
@Testcontainers
class CreditRepositoryIT {

  @Container  @ServiceConnection  // 컨테이너 접속정보를 스프링에 자동 주입
  static PostgreSQLContainer<?> db =
    new PostgreSQLContainer<>("postgres:16-alpine");

  // Flyway가 컨테이너에 V__ 마이그레이션을 그대로 적용
  @Test void findsByStatus() { /* ... */ }
}

7.2테스트별 데이터 — @Sql

각 테스트가 필요한 데이터만 명시적으로 깔면, 테스트끼리 데이터를 공유해 생기는 미묘한 의존이 사라집니다.

@Sql — 테스트 전/후 SQL 실행 @Test
@Sql("/sql/given-approved-credits.sql")  // 이 테스트만의 fixture
void countsApprovedCredits() { /* ... */ }
도구역할 / 왜
Testcontainers운영과 동일한 진짜 DB. 방언·제약·인덱스까지 실제처럼 검증
@Sql테스트별 fixture를 선언적으로. 데이터 의존을 테스트 안에 가둠
@Transactional 테스트각 테스트가 끝나면 자동 롤백 → 다음 테스트는 깨끗한 상태에서 시작
테스트 격리

테스트는 서로 영향을 주면 안 됩니다 — 한 테스트가 남긴 데이터가 다른 테스트를 통과·실패시키면 순서에 따라 결과가 바뀌는 flaky test가 됩니다. Spring의 테스트 메서드는 기본적으로 @Transactional이면 끝에서 롤백되어 격리됩니다. 다만 @Commit이나 별도 트랜잭션을 쓰는 경우는 직접 정리해야 합니다. 격리 원칙: 각 테스트는 자기가 필요한 데이터를 스스로 세우고, 끝나면 흔적을 남기지 않는다.

한 장 요약

스키마는 Flyway가 소유(JPA는 validate로 검증만), 변경은 새 V 파일로만(적용된 건 불변), 시더는 멱등 + @Profile("local"), 로컬은 리셋 한 줄로 자동 복구, 테스트는 Testcontainers로 진짜 DB + 롤백 격리. 이 다섯 줄이 이 문서 전부입니다.
