문서 구조

이 문서는 다른 핸드북과 같은 구조로 읽습니다. 먼저 핵심 개념을 잡고, 표와 체크리스트로 운영 판단 기준을 만든 뒤, 장애 시나리오와 면접 답변 템플릿으로 실전 적용까지 연결합니다.

  • 핵심 개념
  • 검증 증거와 운영 리스크
  • 실무 체크리스트
  • 장애/운영 시나리오
  • 면접 답변 템플릿
REQ-00

기준 경로

대표 경로는 DNS → CDN/WAF → Load Balancer → App → DB다. 장애가 나면 사용자의 증상, DNS 해석, TLS handshake, CDN cache, WAF 차단, Load Balancer target health, 애플리케이션 로그, DB connection pool을 같은 시간축에 올린다.

판단 질문검증 증거운영 리스크
기준 경로이 실제 서비스 경로에서 어떤 책임을 갖는가? 설정 diff, 로그, 메트릭, trace, smoke test 결과 말로는 이해했지만 장애 시 어느 계층을 볼지 정해지지 않음
변경되면 어떤 사용자 영향과 rollback 조건이 생기는가? 변경 티켓, 승인 기록, 배포 후 지표, runbook 링크 복구 절차가 담당자 기억에 의존함
REQ-01

DNS와 TLS

DNS가 잘못되면 애플리케이션 로그가 비어 있을 수 있다. TLS 인증서 만료, SNI 불일치, intermediate chain 누락은 네트워크 연결은 되지만 HTTPS에서 실패하는 전형적인 원인이다. curl -v, openssl s_client, dig 결과를 함께 남긴다.

REQ-02

Load Balancer와 health check

Load Balancer는 단순 분산 장치가 아니라 장애 격리 장치다. health check path가 실제 의존성을 너무 깊게 보거나 너무 얕게 보면 정상 pod가 빠지거나 죽은 앱이 살아 있는 것처럼 보인다. 502는 upstream 연결 실패, 504는 timeout으로 우선 분리한다.

REQ-03

앱과 DB 경계

App → DB 구간은 connection pool, timeout, transaction lock, migration, secret rotation이 얽힌다. 프론트 증상만 보고 API 문제로 단정하지 말고, traceId로 request log, slow query, DB wait event를 이어 본다.

CHECK

실무 체크리스트

작업 전에는 무엇을 바꿀지보다 무엇을 깨뜨릴 수 있는지를 먼저 본다. 체크리스트는 리뷰와 운영 인수에서 그대로 사용할 수 있어야 한다.

  • 사용자 증상, 지역, 시간, 브라우저 조건을 먼저 기록한다.
  • DNS, TLS, CDN/WAF, Load Balancer, App, DB 로그를 같은 시간축에 맞춘다.
  • 502, 503, 504, timeout, TLS error를 분리한다.
  • traceId로 프론트, API, DB 관측 신호를 연결한다.
작업 단계필수 산출물차단 기준
설계목적, 영향 범위, 비목표, owner영향 범위가 사용자·시스템·비용 기준으로 설명되지 않음
변경diff, 승인 기록, 검증 명령production 변경과 staging 검증의 차이가 설명되지 않음
복구rollback 또는 forward fix, 담당자, 예상 시간실패 시 되돌릴 수 없는데 사전 승인과 공지가 없음
CASE

장애/운영 시나리오

특정 시간대에 로그인 API가 간헐적으로 504를 반환한다. 먼저 DNS와 TLS가 정상인지 배제하고, CDN/WAF 차단 로그와 Load Balancer target health를 본다. 그 다음 App의 request latency와 DB connection pool 대기를 확인해 timeout이 어디에서 시작됐는지 좁힌다.

Triage frame

증상 → 영향 범위 → 최근 변경 → 계층별 증거 → 완화책 → 복구 검증 → 재발 방지 순서로 기록한다. 이 순서가 없으면 원인 분석이 사람마다 달라진다.

처음 10분다음 30분종료 조건
사용자 영향과 최근 변경을 확인한다.계층별 증거를 모아 완화책을 실행한다.지표 회복, 재현 불가, owner 확인, 기록 완료
RISK

자주 틀리는 판단

운영 품질은 아는 개념보다 틀린 가정을 빨리 찾는 능력에서 갈린다. 아래 신호가 보이면 설계나 답변을 다시 점검한다.

  • 프론트 화면 오류만 보고 API 코드부터 수정한다.
  • Load Balancer health check가 통과하면 애플리케이션도 정상이라고 단정한다.
  • TLS 인증서 문제를 네트워크 장애로 오해한다.
  • DB lock wait와 API timeout을 별개 사건으로 본다.
Q&A

면접 답변 템플릿

요청 장애는 DNS → CDN/WAF → Load Balancer → App → DB 순서로 좁힌다고 답합니다. 각 hop마다 볼 증거가 다르기 때문에 dig, curl -v, LB target health, 애플리케이션 로그, DB slow query를 같은 시간축에 놓고 판단합니다.

ANSWER SHAPE 1. 한 줄 결론
2. 핵심 개념과 책임 경계
3. 실무에서 깨지는 지점
4. 검증 명령 또는 관측 증거
5. rollback, runbook, 운영 인수 기준
