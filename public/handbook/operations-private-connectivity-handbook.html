VPN-00

연결 방식

Site-to-Site VPN은 빠르게 구축할 수 있지만 인터넷 품질과 IPsec 협상에 의존한다. 전용선, Direct Connect, Interconnect, ExpressRoute, Private Endpoint는 비용과 리드타임이 크지만 예측 가능성과 격리가 좋다.

운영 질문확인할 증거판단 기준
사설 연결 방식이 요구되는 격리와 예측 가능성에 맞는가?VPN/Direct Connect/ExpressRoute/Private Endpoint option, latency, SLA, cost, lead time빠른 구축과 안정적 전용 연결의 trade-off를 명시해야 한다.
연결 방식을 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?connectivity design, circuit/tunnel status, provider SLA, cost estimate연결 방식 선택이 업무 중요도와 복구 목표에 맞아야 한다.
연결 방식 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?VPN 품질 변동으로 배치 연동 지연임시 VPN과 장기 전용선/Private Endpoint 전환 계획을 분리한다.
VPN-01

IPsec 협상

IPsec은 Phase 1에서 IKE SA를 만들고 Phase 2에서 데이터 트래픽용 SA를 만든다. encryption, integrity, DH group, PFS, lifetime, PSK, NAT-T, traffic selector가 양쪽에서 일치해야 한다.

운영 질문확인할 증거판단 기준
IPsec Phase 1/2 협상이 양쪽에서 같은 값으로 합의되었는가?IKE SA, IPsec SA, encryption, integrity, DH group, PFS, lifetime, PSK, NAT-T터널 DOWN은 proposal mismatch와 rekey 문제를 먼저 본다.
IPsec 협상을 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?VPN tunnel status, IKE log, Phase 1/2 proposal sheet, packet counter터널 UP과 데이터 패킷 흐름은 별개로 검증한다.
IPsec 협상 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?rekey 시점마다 터널 flap 발생lifetime/PFS/proposal 값을 맞추고 rekey 로그를 보관한다.
VPN-02

라우팅과 방화벽

터널 UP은 충분하지 않다. local CIDR, peer CIDR, route table, security group, 고객사 방화벽, return path가 맞아야 앱 통신이 된다. CIDR overlap은 설계 단계에서 제거해야 한다.

운영 질문확인할 증거판단 기준
터널 UP 이후 실제 서비스 통신 경로가 열려 있는가?local CIDR, peer CIDR, route table, customer firewall, return path, DNSICMP 성공만으로 API/DB 포트 성공을 판단하지 않는다.
라우팅과 방화벽을 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?route readback, firewall log, nc -vz, curl internal API, tcpdump양쪽 route와 firewall이 같은 src/dst/port를 허용해야 한다.
라우팅과 방화벽 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?터널은 UP이나 ERP API timeouttraffic selector, route, firewall, DNS를 순서대로 좁힌다.
VPN-03

운영 증거

장애 시 터널 상태, rekey 로그, packet counter, tcpdump, nc, curl 결과를 분리해 본다. 고객사에 요청할 때는 source IP, destination IP, port, timestamp, packet evidence를 제공한다.

운영 질문확인할 증거판단 기준
고객사와 공유 가능한 packet evidence가 준비되어 있는가?timestamp, source IP, destination IP, port, tunnel id, packet counter, tcpdump sample상대방이 같은 시간대와 같은 5-tuple로 확인할 수 있어야 한다.
운영 증거를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?tcpdump, nc/curl output, firewall decision, tunnel byte counter증거가 없으면 고객사 방화벽/라우팅 협업이 추측이 된다.
운영 증거 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?양쪽 모두 상대 네트워크 문제라고 주장공유 evidence template과 escalation owner를 runbook에 추가한다.

실무 체크리스트

VPN·Private Connectivity 체크리스트는 연결 방식, IPsec 협상, 라우팅과 방화벽, 운영 증거의 변경 위험을 사전에 드러내기 위한 검토표다. 각 항목은 리뷰, 배포 승인, 운영 인수에서 같은 증거로 다시 확인되어야 한다.

  • local/peer CIDR, tunnel endpoint, Phase 1/2 값을 표로 합의한다.
  • 터널 UP과 애플리케이션 통신 성공을 분리해 검증한다.
  • route table, 방화벽, return path, DNS를 양쪽에서 확인한다.
  • 장애 시 전달할 packet evidence 양식을 미리 정한다.

CASE

장애/운영 시나리오

고객사 ERP와 Site-to-Site VPN을 연결한다. CIDR overlap을 먼저 확인하고, IPsec proposal을 양쪽 문서로 맞춘다. 터널이 올라온 뒤에는 ICMP가 아니라 실제 API port와 DB port를 nc, tcpdump, 애플리케이션 health check로 검증한다.

Triage frame

VPN·Private Connectivity 장애는 증상 → 영향 범위 → 최근 변경 → Tunnel 증거 → 완화책 → 복구 검증 → 재발 방지 순서로 기록한다. 이 흐름이 있어야 담당자가 바뀌어도 같은 결론에 도달한다.

RISK

자주 틀리는 판단

VPN·Private Connectivity에서 자주 틀리는 판단은 연결 방식와 운영 증거의 책임 경계를 흐리는 데서 시작된다. 아래 신호가 보이면 설계, runbook, 답변 구조를 다시 점검한다.

  • VPN 터널 UP을 서비스 연동 완료로 착각한다.
  • 고객사 방화벽 변경 담당자와 시간을 맞추지 않는다.
  • Phase 2 traffic selector가 실제 CIDR과 다르다.
  • DNS forwarder가 없어 IP로만 임시 테스트하고 끝낸다.
Q&A

면접 답변 템플릿

VPN은 암호화 터널, 라우팅, 방화벽, DNS, 애플리케이션 포트 검증이 모두 맞아야 한다고 답합니다. Phase 1/2 협상과 서비스 통신 성공을 구분하는 것이 핵심입니다.

OPERATIONS ANSWER FRAME 1. VPN은 터널, 라우팅, 방화벽, DNS, 애플리케이션 포트가 모두 맞아야 한다.
2. Phase 1/2 협상 성공과 업무 통신 성공을 구분한다.
3. 고객사와 공유 가능한 packet evidence를 남긴다.
