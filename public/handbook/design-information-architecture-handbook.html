정보구조와
내비게이션

정보구조는 사용자가 필요한 정보를 어디에서 찾고, 어떤 경로로 이동하고, 현재 위치를 어떻게 이해하는지 결정합니다. 복잡한 제품일수록 IA가 곧 사용성의 골격입니다.

IA-01

정보구조의 목적

목적설명
Findability사용자가 필요한 항목을 찾을 수 있게 함
Orientation현재 위치와 상위 맥락을 이해하게 함
Scannability정보를 빠르게 훑고 비교하게 함
Predictability다음에 어디로 가야 할지 예측 가능하게 함
IA-02

분류와 위계

메뉴는 조직도가 아니라 사용자 작업 모델을 따라야 합니다. 내부 팀 구조를 그대로 노출하면 사용자는 기능을 찾기 어렵습니다.

분류 기준적합한 상황주의
Task-based업무 수행 중심 SaaS사용자 역할별 task 차이 반영
Object-based문서, 주문, 프로젝트처럼 객체 중심객체 상태와 action 구분
Lifecycle-based신청 → 승인 → 완료 흐름현재 단계와 다음 행동 표시
Role-based관리자/운영자/사용자 화면 분리권한 없음 상태와 전환 고려
IA-03

내비게이션 패턴

패턴언제주의
Top navigation상위 카테고리가 적고 전환이 잦을 때항목이 늘어나면 overflow 처리 필요
Sidebar업무형 앱, 깊은 메뉴 구조접힘 상태와 현재 위치 표시
Tabs같은 객체의 하위 view 전환서로 독립된 페이지 메뉴로 남용 금지
Breadcrumb계층 깊이가 있는 상세 화면현재 위치와 상위 이동 제공
Command palette숙련자 빠른 이동검색 품질과 keyboard 지원 필요
IA-04

검색·필터·정렬

도구역할실무 기준
Search사용자가 찾는 값을 알고 있을 때검색 대상과 matching rule 명시
Filter조건으로 결과 집합을 줄일 때활성 필터와 초기화 제공
Sort결과 순서를 비교할 때기본 정렬의 의도 명확화
Saved view반복 업무 조건 저장업무형 UI에서 생산성 크게 향상
실무 감각

검색과 필터가 많아질수록 “현재 결과가 왜 이렇게 나왔는가”를 설명해야 합니다. 활성 조건, 결과 수, empty state, reset action을 함께 설계합니다.

IA-06

정보구조 검증

정보구조는 회의실에서 정한 메뉴명이 아니라 사용자가 실제로 찾을 수 있는 구조여야 합니다.

방법목적결과 해석
card sorting사용자가 정보를 어떻게 묶는지 파악팀 내부 조직도와 사용자 mental model의 차이를 찾음
tree testing시각 디자인 없이 탐색 구조만 검증목표 항목을 찾는 성공률, 경로, 시간으로 메뉴 문제 확인
search log review사용자가 찾지 못한 용어와 동의어 발견메뉴명, 필터명, 도움말 용어 개선
content inventory중복, 오래된 정보, 소유자 없는 정보를 정리IA 이전에 콘텐츠 거버넌스 문제가 있는지 확인
Case study

사내 어드민에서 “정산” 메뉴 아래에 환불, 세금계산서, 거래내역을 모두 넣었더니 CS 담당자는 환불을 “고객 처리”에서 찾고 재무 담당자는 세금계산서를 “회계”에서 찾았습니다. 해결은 메뉴 하나를 고르는 것이 아니라 역할별 진입점과 공통 상세 페이지를 분리하고, 동일 리소스를 다른 경로에서 접근 가능하게 만드는 것이었습니다.

IA-07

Taxonomy와 객체 모델

고급 IA는 메뉴 배치가 아니라 제품의 객체 모델을 사용자가 이해할 수 있는 언어로 번역하는 일입니다. 고객, 주문, 캠페인, 승인 요청 같은 핵심 객체와 그 관계가 흐리면 검색, 권한, 상세 화면, 감사 로그가 모두 흔들립니다.

모델 요소설계 질문산출물실패 신호
Entity사용자가 실제로 다루는 핵심 객체는 무엇인가domain object map메뉴명이 팀 조직명이나 DB 테이블명과 같음
Relationship객체가 어떤 상하/소유/참조 관계를 갖는가relationship diagram상세 화면마다 같은 정보가 다르게 연결됨
Attribute사용자가 판단에 쓰는 속성은 무엇인가attribute priority list중요 상태가 상세 진입 후에야 보임
Vocabulary고객 언어와 내부 용어가 어디서 충돌하는가controlled vocabulary, synonym map검색 로그에 같은 의미의 다른 단어가 반복됨
Lifecycle객체가 어떤 상태를 거쳐 변하는가state taxonomy목록, 필터, 상세의 상태명이 서로 다름
Taxonomy governance

분류 체계는 한 번 정하면 끝나는 문서가 아닙니다. 새로운 기능이 추가될 때 객체명, 상태명, 필터명, 검색 동의어, 권한 라벨을 누가 승인하고 어디에 반영하는지 운영 규칙이 필요합니다.

IA-08

권한·상태 기반 IA

실무 제품의 IA는 모든 사용자에게 같은 메뉴를 보여주는 문제가 아닙니다. 역할, 권한, 데이터 소유권, 객체 상태에 따라 보이는 정보와 가능한 액션이 달라집니다.

변수IA에 미치는 영향검증 질문
Role메뉴, 대시보드, 기본 진입점이 달라짐관리자/운영자/감사자가 같은 구조를 억지로 쓰고 있지 않은가
Permission액션 노출, disabled 이유, 요청 경로가 필요권한 없음이 empty state처럼 보이지 않는가
Ownership내 항목/팀 항목/전체 항목 경계가 필요scope 변경이 명확히 보이는가
Status목록 필터, badge, 다음 행동이 달라짐상태가 행동 가능성을 설명하는가
Audit이력, 변경자, 사유가 IA 일부가 됨중요 변경의 책임 경로가 추적되는가
IA REVIEW PROMPT Object: 사용자가 다루는 핵심 객체
Role: 이 객체를 보는 사용자 역할
State: 현재 객체 상태와 다음 가능한 상태
Action: 역할과 상태에 따라 가능한 행동
Evidence: 사용자가 판단에 필요한 속성, 이력, 근거
