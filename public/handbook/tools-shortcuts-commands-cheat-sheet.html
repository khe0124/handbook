Tools · Shortcuts
Commands Cheat Sheet

실무 속도를 높이는 툴, 단축키, 명령어를 역할별로 압축했습니다. 외우는 목록이 아니라 장애 진단, 코드 탐색, API 확인, 배포 검증 때 바로 꺼내 쓰는 기준표입니다.

COMPILED : 2026-06 · FORMAT : QUICK REFERENCE
TOOL-CS-01

개요

툴과 명령어는 많이 아는 것보다 “어떤 문제에서 어떤 증거를 빠르게 얻는가”가 중요합니다. 코드 탐색, 실행 재현, 네트워크 확인, 데이터 확인, 배포 검증 순서로 손에 익히면 실무 대응 속도가 크게 올라갑니다.

상황먼저 쓸 도구얻어야 할 증거
코드 위치를 모름VS Code Search, rg, Git history진입점, 호출자, 최근 변경 커밋
화면/API가 깨짐DevTools, curl, Postman/Insomniastatus, request header, response body, console stack
DB 값이 이상함psql, DBeaver/TablePlus, migration history실제 row, index, transaction, migration 적용 여부
배포 후 장애logs, health check, Docker/Kubernetes CLI릴리스 버전, pod/container 상태, 오류 로그
내 로컬만 안 됨env 확인, port/process 확인, clean installNode/Java 버전, env 누락, 충돌 포트, lockfile 차이

TOOL-CS-04

Git·터미널 명령어

목적명령어해석
상태 확인git status --short변경/추적/삭제 파일을 짧게 확인
변경 diffgit diff, git diff --staged작업 중 변경과 stage된 변경 분리 확인
커밋 이력git log --oneline --graph --decorate -20최근 흐름과 branch 위치 확인
파일 이력git log --follow -- path/to/file이름 변경 포함 파일 변경 추적
작성자/변경 시점git blame path/to/file현재 줄이 언제 왜 들어왔는지 추적
문자열 검색rg "keyword"코드베이스 전체에서 빠른 검색
파일 찾기rg --files | rg "name"파일명 일부로 후보 좁히기
포트 사용 확인lsof -i :3000개발 서버 충돌 프로세스 확인
프로세스 확인ps aux | rg node실행 중인 런타임/서버 확인
디스크 확인df -h, du -sh *공간 부족과 큰 디렉터리 확인
주의

git reset --hard, git clean -fd, rm -rf는 복구 비용이 큰 명령입니다. 실행 전 현재 branch, stash 필요 여부, 삭제 범위를 반드시 확인합니다.

TOOL-CS-07

Red Flags

툴과 명령어 사용에서 가장 큰 위험은 속도가 아니라 증거를 잃거나, 공유 환경을 망가뜨리거나, 민감 정보를 노출하는 것입니다.

신호왜 위험한가즉시 확인
운영 DB에서 검증 없이 update/delete 실행데이터 손상이 즉시 사용자 장애로 이어집니다.transaction, where 조건, affected rows, backup/rollback 계획
secret이 command history나 문서에 남음토큰 유출과 재발급 비용이 발생합니다.shell history, 공유 문서, 로그 마스킹 여부
curl 결과 중 header/body를 저장하지 않음장애 재현 증거가 사라집니다.status, request id, response body, timestamp 기록
강제 명령으로 로컬 변경을 날림원인 분석과 복구에 필요한 diff가 사라집니다.git status, stash/commit 필요 여부 확인
툴 결과를 해석하지 않고 캡처만 공유다음 담당자가 다시 같은 조사를 반복합니다.관찰, 해석, 다음 행동을 같이 기록
TOOL-CS-08

Review Gate

상황차단 기준통과 증거
새 개발 환경 문서필수 버전, env, 실행 명령, 실패 복구가 없음clean clone 기준 실행 절차와 troubleshooting이 있습니다.
운영 명령 공유대상 환경, 읽기/쓰기 여부, rollback 조건이 불명확함환경, 명령, 예상 결과, 실패 시 중단 기준이 명시됩니다.
API 재현 케이스인증 방식, payload, expected status가 없음curl 또는 컬렉션으로 재현 가능합니다.
DB 확인 쿼리where 조건과 row 수 제한이 없음read-only 쿼리, limit, transaction, explain 기준이 있습니다.
단축키/툴 표준화개인 취향만 있고 팀 공유 자산으로 남지 않음팀 문서, onboarding checklist, 공통 collection에 반영됩니다.
TOOL-CS-09

Failure Playbook

TOOLING INCIDENT TRIAGE툴 문제처럼 보이는 장애는 version, env, network, credential, cache, permission 순서로 좁히면 빠르게 분리됩니다.
순서질문증거
1나만 실패하는가, 팀/CI/운영도 실패하는가?동료 재현 여부, CI log, 배포 환경 log
2버전 차이인가?node -v, java -version, package manager, lockfile
3환경변수나 credential 누락인가?.env template, secret 이름, 권한, 만료 여부
4네트워크/DNS/프록시 문제인가?curl -v, dig, VPN, firewall, proxy 설정
5캐시나 의존성 오염인가?clean install, lockfile diff, Docker image rebuild
6복구 뒤 재발 방지 문서가 남았는가?원인, 확인 명령, 수정 커밋, runbook 업데이트
