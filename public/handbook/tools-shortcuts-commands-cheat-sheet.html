TL-CS-01

핵심 명령·스니펫

Git — 상태 · 이력 보기

# 변경 파일 짧게 (가장 많이 씀)
git status --short

# 그래프로 전체 브랜치 흐름
git log --oneline --graph --all

# 특정 파일 변경 이력 (이름 변경 추적 포함)
git log --follow -- path/to/file

# 스테이지된 것 vs 작업 중인 것 분리 확인
git diff            # 아직 add 안 한 변경
git diff --staged   # add 한 변경

Git — 히스토리 고치기

# 최근 3개 커밋 정리 (squash/reword/drop)
git rebase -i HEAD~3

# 방금 커밋 취소하되 변경은 스테이지에 남김
git reset --soft HEAD~1

# 마지막 커밋 메시지/내용 수정
git commit --amend

# 다른 브랜치의 특정 커밋만 가져오기
git cherry-pick <sha>

Git — 잠깐 치워두기 · 복구

# 작업 중인 변경 임시 저장 → 되돌리기
git stash
git stash pop

# 유실 복구의 핵심 — 모든 HEAD 이동 기록
git reflog
git reset --hard HEAD@{2}   # reflog에서 찾은 지점으로 복구

Git — 버그 커밋 이분 탐색 (bisect)

# 언제 깨졌는지 자동 이분 탐색
git bisect start
git bisect bad              # 현재(깨진) 상태
git bisect good <sha>       # 정상이던 과거 커밋
# git이 중간 커밋 체크아웃 → 테스트 후 good/bad 반복
git bisect reset           # 끝나면 원위치

Terminal — 검색 · 파싱 · 파이프

# 재귀 검색 + 줄번호 (파일명·API path 추적)
grep -rn 'keyword' src/

# JSON 필드 뽑기
curl -s api.example.com/orders | jq '.data[] | {id, status}'

# 검색 결과를 다음 명령으로 넘기기
grep -rl 'TODO' src/ | xargs wc -l

# 예전에 친 명령 다시 찾기
history | grep rebase

HTTP — API 찌르기

# 헤더까지 보기 — status·응답 헤더 확인 (첫 진단)
curl -i https://api.example.com/health

# POST + JSON
curl -X POST -H 'Content-Type: application/json' \
  -d '{"amount":1000}' https://api.example.com/payments

# 연결 전 구간 프로파일 (2>&1로 stderr 합치기)
curl -v https://api.example.com/orders 2>&1 | head -20

Node · 패키지

# 설치 없이 일회성 실행
pnpm dlx create-next-app my-app
npx tsc --noEmit

# 프로젝트가 제공하는 스크립트 목록 확인
npm run
명령용도주의
git status --short변경 파일 빠른 확인, 첫 진단습관화 1순위
git reset --soft HEAD~1커밋만 취소, 변경 보존--hard와 혼동 금지
git reflog유실 커밋·브랜치 복구 진입점대부분의 "날아갔다"는 여기서 복구
git stash / pop브랜치 안 만들고 잠깐 치우기여러 개 쌓이면 stash list
grep -rn재귀 검색 + 줄번호큰 저장소는 rg가 빠름
curl -i <url>status·헤더 확인-s로 progress 끄기
pnpm dlx / npx전역 설치 없이 CLI 실행버전 고정 필요하면 명시
주의

git reset --hard는 작업 중 변경을 복구 없이 날린다. 실행 전 git stash나 커밋으로 대피시켜라. rebase로 이미 push한 커밋을 고친 뒤 git push --force하면 팀의 히스토리가 깨진다 — 공유 브랜치엔 --force-with-lease를 쓰고, main엔 아예 하지 마라.

TL-CS-02

판단 한 줄

선택언제 쓰나주의
rebase내 로컬 브랜치 히스토리를 선형으로 정리할 때이미 push·공유된 커밋엔 금지 (히스토리 재작성)
merge공유 브랜치 통합, 실제 병합 시점을 남기고 싶을 때graph가 복잡해짐, 하지만 안전
reset --soft커밋만 취소, 변경은 스테이지에 그대로커밋 다시 묶을 때
reset --mixed커밋·스테이지 취소, 작업 변경은 보존 (기본값)add부터 다시
reset --hard커밋·스테이지·작업 변경 전부 폐기복구 어려움 — reflog 외엔 없음
stash잠깐(분~시간) 치웠다 곧 되돌릴 때오래 두면 잊음 → 브랜치가 나음
branch수명이 길거나 남과 공유할 작업일 때이름으로 맥락이 남음
npm표준·호환성 우선, CI 단순함디스크·설치속도 불리
pnpm모노레포·디스크 절약·빠른 설치lockfile·hoisting 동작 팀과 합의
TL-CS-03

자주 터지는 것 · 즉시 확인

신호왜 위험즉시 확인
git reset --hard로 작업이 사라짐스테이지·작업 변경이 되돌릴 수 없이 폐기됨git reflog로 직전 HEAD 찾아 reset --hard HEAD@{n}
rebase 후 push --force로 팀 히스토리가 깨짐남이 받은 커밋이 사라져 pull 충돌·유실공유 브랜치는 --force-with-lease, main은 금지. 이미 깼으면 동료 reflog로 복원
.gitignore 누락으로 secret이 커밋됨토큰 유출 — 히스토리에 남아 삭제해도 추적됨즉시 토큰 재발급, history 정리(filter-repo), .gitignore 추가
detached HEAD 상태에서 커밋함브랜치에 안 붙은 커밋 → 체크아웃 옮기면 유실git switch -c fix-branch로 지금 커밋을 브랜치로 고정
내 로컬만 빌드/실행 실패런타임·env·lockfile 차이로 재현 안 됨버전 확인 → clean install → 팀/CI와 diff (ch4 triage)

TOOLING INCIDENT TRIAGE "내 로컬만 깨짐"은 아래 순서로 좁히면 대부분 분리된다.
1. runtime version — 나와 팀/CI의 런타임 버전이 같은가
2. env diff — 환경변수·credential 누락 또는 값 차이
3. port/process — 충돌 포트·좀비 프로세스가 잡고 있는가
4. clean install — lockfile 기준 재설치로 의존성 오염 제거
5. container — 그래도 재현 안 되면 컨테이너 기준으로 통제
순서질문증거
1런타임 버전이 팀/CI와 같은가?node -v, java -version, lockfile
2env·credential 누락/차이인가?.env template diff, secret 이름·만료
3포트·프로세스 충돌인가?lsof -i :3000, ps aux | grep node
4의존성·캐시 오염인가?rm -rf node_modules + npm ci, lockfile diff
5컨테이너 기준으론 재현되나?docker compose up, image digest
복구/중단 기준

작업 유실이 의심되면 다른 명령을 치기 전에 git reflog부터 — HEAD 이동 기록이 남아 있는 동안이 복구 골든타임이다. secret이 커밋·push됐다면 history 정리보다 토큰 재발급이 먼저다. 히스토리 재작성(rebase/force)은 공유 브랜치에서 중단하고 merge로 우회한다.
