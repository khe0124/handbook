디자인 개요

DOC : DESIGN-OVERVIEW-V1 SCOPE : UI · UX · PRODUCT DESIGN · DESIGN SYSTEM LANG : KO

디자인 개요

디자인은 화면을 예쁘게 꾸미는 일이 아니라 사용자가 목표를 이해하고, 판단하고, 행동할 수 있게 제품 경험을 구조화하는 일입니다. UI는 그 구조가 화면 위에 드러난 결과이고, UX는 사용자가 그 과정을 겪으며 얻는 전체 경험입니다.

TARGET : PM · 디자이너 · 프론트엔드 · 제품팀 협업자
DES-01

디자인의 목적

좋은 디자인은 사용자의 인지 비용을 낮추고, 제품이 원하는 행동을 무리 없이 수행하게 만듭니다.

목적설명실무 질문
이해현재 상태와 선택지를 명확히 보여줌사용자가 지금 어디에 있고 무엇을 할 수 있는가
판단우선순위와 위험을 비교 가능하게 함중요한 정보가 먼저 보이는가
행동다음 행동을 쉽게 수행하게 함primary action이 명확한가
회복오류와 실패에서 빠져나갈 길을 제공실패 후 무엇을 해야 하는지 보이는가
DES-02

UI와 UX

구분관심사예시
UI화면 구성, 컴포넌트, 색, 타이포그래피, 상태버튼, 테이블, 모달, 폼 오류
UX사용자 목표 달성 과정과 감각가입 흐름, 결제 완료, 검색 성공
Product Design문제 정의와 해결 방식어떤 사용자의 어떤 문제를 해결할지
Design System반복 가능한 UI 규칙과 컴포넌트token, variant, component guideline
실무 감각

UI가 깔끔해도 UX가 나쁠 수 있습니다. 예를 들어 버튼은 예쁘지만 사용자가 왜 눌러야 하는지 모르거나, 오류가 난 뒤 복구 방법이 없으면 제품 경험은 실패합니다.

DES-03

좋은 디자인의 기준

기준좋은 상태나쁜 신호
명확성한 화면의 목적이 바로 드러남모든 요소가 같은 중요도로 보임
일관성같은 행동은 같은 패턴을 사용페이지마다 버튼 위치와 용어가 다름
효율성반복 업무가 빠르게 수행됨매번 모달과 확인을 지나야 함
관용성실수를 예방하고 되돌릴 수 있음삭제 후 복구 불가인데 확인이 약함
접근성키보드, 스크린리더, 저시력 사용자가 이용 가능색만으로 상태를 전달
예쁨의 한계

시각적 완성도는 중요하지만 충분조건이 아닙니다. 실무 디자인 리뷰에서는 “예쁜가”보다 “사용자가 더 적은 판단으로 더 안전하게 목표를 달성하는가”를 먼저 봅니다.

Backend데이터 계약, 권한, 상태 전이API, validation, error contract
QA시나리오 검증, 회귀, edge case테스트 케이스, 버그 리포트

DES-06

제품 디자인 의사결정 프레임

심도 있는 디자인은 화면 취향이 아니라 제품 목표, 사용자 행동, 운영 제약을 함께 판단합니다.

프레임질문실무 적용
North Star Metric이 화면이 장기적으로 키워야 하는 핵심 행동은 무엇인가단순 클릭률보다 활성 사용자, 완료율, 재방문 같은 제품 가치와 연결
Jobs To Be Done사용자는 어떤 상황에서 이 제품을 고용하는가“보고서 다운로드”가 아니라 “회의 전에 최신 수치를 신뢰 있게 공유”로 정의
Risk mapping잘못 설계되면 어떤 손실이 생기는가결제, 권한, 삭제, 개인정보 흐름은 더 강한 검증과 마찰을 둠
Operational fit운영팀이 이 경험을 지속적으로 처리할 수 있는가고객 문의, 예외 승인, 수동 처리 비용까지 고려
Case study

B2B 관리자 화면에서 “사용자 초대” 버튼을 크게 만드는 것보다 초대 상태, 만료, 재발송, 권한 변경 이력을 한 화면에서 확인하게 만든 사례가 더 큰 개선을 만들 수 있습니다. 디자인의 핵심은 버튼 강조가 아니라 운영자가 초대 실패를 추적하고 복구하는 시간을 줄이는 것입니다.

UX 사고와 문제 정의

DOC : DESIGN-UX-THINKING-V1 SCOPE : PROBLEM · USER · JTBD · METRICS LANG : KO

UX 사고와
문제 정의

UX 설계의 출발점은 화면이 아니라 문제입니다. 사용자가 무엇을 하려는지, 왜 실패하는지, 제품이 어떤 행동 변화를 만들고 싶은지를 명확히 해야 합니다.

UX-01

문제와 솔루션

“검색 필터를 추가하자”는 솔루션이고, “사용자가 원하는 항목을 찾지 못한다”가 문제입니다.

구분질문예시
문제사용자가 어디서 막히는가결제 전 배송비를 이해하지 못함
원인왜 막히는가배송비 정보가 마지막 단계에만 노출
솔루션무엇을 바꿀 것인가상품 상세와 장바구니에 예상 배송비 표시
검증좋아졌는지 어떻게 알 것인가결제 이탈률, 문의 감소, 테스트 관찰
솔루션 점프

팀이 바로 UI를 그리기 시작하면 문제를 잘못 풀 위험이 커집니다. 먼저 사용자의 실패 상황, 빈도, 영향, 현재 우회 방법을 확인합니다.

UX-02

사용자 이해

도구쓸 때주의
JTBD사용자가 제품을 “고용”하는 목적을 볼 때인구통계보다 상황과 동기를 봄
Persona팀이 사용자를 같은 언어로 이해할 때가상의 취향 묘사로 흐르지 않게 함
Scenario사용 맥락과 제약을 설명할 때happy path만 쓰지 않음
Journey map긴 여정의 감정과 접점을 볼 때실제 데이터와 인터뷰로 보정
실무 감각

업무 시스템에서는 “사용자”가 구매자, 승인자, 실무자, 관리자, 감사자처럼 여러 역할로 나뉩니다. 같은 화면도 역할별 목표와 권한이 다르면 완전히 다른 UX가 됩니다.

UX-03

가설과 지표

UX HYPOTHESIS 만약 [사용자 문제]를 [디자인 변경]으로 줄이면,
[사용자 행동]이 바뀌고,
[제품 지표]가 개선될 것이다.
지표예시주의
행동 지표완료율, 이탈률, 재시도율사용자 목표와 연결
품질 지표오류율, 문의량, 취소율나쁜 경험의 신호
속도 지표task completion time무조건 짧은 것이 좋은 것은 아님
만족 지표CSAT, NPS, 인터뷰 반응정량 행동과 함께 해석
UX-04

리서치 신호

신호강점한계
Analytics많은 사용자의 행동 패턴왜 그런지는 설명하지 못함
Session replay실제 사용 흐름 관찰개인정보 마스킹과 표본 편향 주의
Interview동기와 맥락 파악말과 실제 행동이 다를 수 있음
Support ticket반복 불만과 실패 지점문제 없는 사용자는 드러나지 않음
Usability test설계안의 이해 가능성 검증표본이 작으므로 패턴 중심 해석
UX-05

의사결정 프레임

Impact영향
얼마나 많은 사용자와 중요한 흐름에 영향을 주는가.
Confidence확신
데이터, 리서치, 실험으로 얼마나 뒷받침되는가.
Effort비용
디자인, 개발, QA, 운영 변경 비용이 어느 정도인가.
Risk리스크
잘못됐을 때 전환율, 신뢰, 데이터, 규정에 어떤 손상이 생기는가.
UX-06

문제 정의 심화

좋은 문제 정의는 사용자의 말, 행동 데이터, 비즈니스 손실, 기술 제약을 분리해서 봅니다.

증거 유형강점주의점
정성 인터뷰맥락과 감정, 우회 행동을 발견말한 선호를 그대로 요구사항으로 바꾸지 않음
행동 로그반복 패턴과 이탈 지점을 확인왜 그런 행동을 했는지는 설명하지 못함
고객 문의실제 고통과 언어를 포착목소리가 큰 일부 고객에 치우칠 수 있음
운영 비용수동 처리와 예외 대응 비용을 드러냄내부 효율만 보고 사용자 가치가 빠질 수 있음
Case study

“검색이 불편하다”는 요청을 그대로 검색창 확대로 해결하면 실패할 수 있습니다. 실제 문제를 보면 필터명이 조직 내부 용어와 달라 사용자가 올바른 조건을 못 고르거나, 결과가 0건일 때 다음 행동이 없어 멈추는 경우가 많습니다. 이때 해결책은 검색 UI 확대가 아니라 용어 정리, 추천 필터, empty state, 최근 사용 조건 저장일 수 있습니다.

Problem statement template

[장애물] 때문에 [측정 가능한 손실]이 발생한다.
우리는 [행동 변화]가 일어나면 문제가 개선됐다고 판단한다.

정보구조와
내비게이션

정보구조는 사용자가 필요한 정보를 어디에서 찾고, 어떤 경로로 이동하고, 현재 위치를 어떻게 이해하는지 결정합니다. 복잡한 제품일수록 IA가 곧 사용성의 골격입니다.

IA-01

정보구조의 목적

목적설명
Findability사용자가 필요한 항목을 찾을 수 있게 함
Orientation현재 위치와 상위 맥락을 이해하게 함
Scannability정보를 빠르게 훑고 비교하게 함
Predictability다음에 어디로 가야 할지 예측 가능하게 함
IA-02

분류와 위계

메뉴는 조직도가 아니라 사용자 작업 모델을 따라야 합니다. 내부 팀 구조를 그대로 노출하면 사용자는 기능을 찾기 어렵습니다.

분류 기준적합한 상황주의
Task-based업무 수행 중심 SaaS사용자 역할별 task 차이 반영
Object-based문서, 주문, 프로젝트처럼 객체 중심객체 상태와 action 구분
Lifecycle-based신청 → 승인 → 완료 흐름현재 단계와 다음 행동 표시
Role-based관리자/운영자/사용자 화면 분리권한 없음 상태와 전환 고려
IA-03

내비게이션 패턴

패턴언제주의
Top navigation상위 카테고리가 적고 전환이 잦을 때항목이 늘어나면 overflow 처리 필요
Sidebar업무형 앱, 깊은 메뉴 구조접힘 상태와 현재 위치 표시
Tabs같은 객체의 하위 view 전환서로 독립된 페이지 메뉴로 남용 금지
Breadcrumb계층 깊이가 있는 상세 화면현재 위치와 상위 이동 제공
Command palette숙련자 빠른 이동검색 품질과 keyboard 지원 필요
IA-04

검색·필터·정렬

도구역할실무 기준

Case study

사내 어드민에서 “정산” 메뉴 아래에 환불, 세금계산서, 거래내역을 모두 넣었더니 CS 담당자는 환불을 “고객 처리”에서 찾고 재무 담당자는 세금계산서를 “회계”에서 찾았습니다. 해결은 메뉴 하나를 고르는 것이 아니라 역할별 진입점과 공통 상세 페이지를 분리하고, 동일 리소스를 다른 경로에서 접근 가능하게 만드는 것이었습니다.

사용자 흐름과
태스크 설계

사용자 흐름은 화면 목록이 아니라 목표 달성을 위한 행동의 순서입니다. 좋은 흐름은 다음 행동을 예측 가능하게 하고, 예외 상황에서도 사용자가 길을 잃지 않게 합니다.

FLOW-01

Flow의 종류

구분설명예시
User flow사용자 목표 달성 경로상품 선택 → 장바구니 → 결제
Task flow특정 업무 수행 단계승인 요청 작성 → 검토자 선택 → 제출
System flow시스템 상태 전이pending → approved → completed
Service blueprint사용자 행동과 내부 운영 연결고객 요청과 운영 처리 흐름
FLOW-02

Happy path와 예외

흐름 설계는 정상 경로만 그리면 부족합니다. 실제 제품은 예외와 중단, 재개가 더 중요할 때가 많습니다.

상황설계 질문
입력 오류어디가 왜 잘못됐고 어떻게 고치는가
권한 없음왜 접근 불가이며 누구에게 요청해야 하는가
네트워크 실패재시도, 임시 저장, 데이터 손실 방지가 있는가
중도 이탈나중에 이어서 할 수 있는가
중복 제출idempotency와 버튼 상태가 설계되어 있는가
FLOW-03

상태 설계

상태필요한 UI놓치기 쉬운 점
Empty왜 비었는지, 무엇을 하면 되는지권한 없음과 데이터 없음 구분
Loadingskeleton, progress, disabled action무한 대기 방지
Error원인, 복구 행동, 문의 경로기술 에러 그대로 노출 금지
Success완료 확인과 다음 행동토스트만 띄우고 맥락 잃게 하지 않음
Dirty저장 전 변경 있음 표시이탈 경고와 자동 저장 기준
FLOW-04

마찰과 안전장치

모든 마찰이 나쁜 것은 아닙니다. 위험한 행동에는 의도적인 마찰이 필요하고, 반복 업무에는 마찰을 줄여야 합니다.

행동권장 패턴이유
삭제undo 또는 명확한 confirmation복구 가능성에 따라 선택
결제/제출요약 확인, disabled 중복 방지실수와 중복 처리 방지
반복 승인bulk action, keyboard shortcut업무 효율
위험 설정 변경영향 범위 표시, 감사 로그운영 리스크 관리
FLOW-05

체크리스트

Entry point진입
사용자가 어디서 이 흐름을 시작하는지 명확한가.
Exit point종료
완료, 취소, 실패, 중단 후 상태가 정의되어 있는가.
Recovery복구
오류와 이탈 후 다시 이어갈 수 있는가.
Feedback피드백
사용자 행동 뒤 시스템 반응이 즉시 보이는가.
FLOW-06

실패와 중복을 포함한 흐름

실무 플로우는 happy path보다 재시도, 중복 제출, 권한 변경, 장시간 작업의 복구가 더 중요합니다.

리스크설계 대응개발 계약
중복 제출제출 중 버튼 잠금, 결과 화면 명확화idempotency key, 중복 요청 응답 규칙
권한 변경작업 중 권한 만료 안내와 저장 보호403 응답 메시지, draft 보존
부분 성공성공/실패 항목 분리와 재시도 액션batch API 결과 구조
긴 처리작업 큐 상태, 알림, 다시 방문 경로job id, polling 또는 webhook 상태
Case study

대량 권한 변경 화면에서 “저장” 후 토스트만 보여주면 일부 실패를 놓칩니다. 100명 중 92명 성공, 8명 실패처럼 결과를 분리하고 실패 사유별 재시도 버튼을 제공해야 운영자가 엑셀로 다시 관리하지 않습니다.
