Loop Engineering

Loop Engineering은 에이전트 작업을 단발성 명령이 아니라 입력, 실행, 검증, 반성, 중단 조건을 가진 반복 가능한 시스템으로 설계하는 기술입니다.

LOOP-01

루프의 정의

AI 코딩은 생성보다 루프 운영이 중요합니다. 좋은 루프는 매 반복마다 불확실성을 줄이고, 나쁜 루프는 더 많은 파일을 읽고 더 많은 코드를 바꾸며 더 많은 가정을 만듭니다.

Training goal

팀의 목표는 모델에게 더 긴 지시를 주는 것이 아니라, Plan, Act, Verify, Reflect가 반복될 때마다 상태가 더 명확해지도록 설계하는 것입니다.

LOOP-02

Plan Act Verify Reflect

Plan→ Act→ Verify→ Reflect
단계질문산출물
Plan무엇을 할지, 어떤 파일과 제약을 볼지 정했는가작업 계획, 가설, 검증 명령
Act한 번에 하나의 원인 가설만 수정했는가작은 diff, 실패 테스트, 조사 메모
Verify자동/수동 검증으로 완료를 확인했는가테스트 결과, 빌드 결과, 수동 시나리오
Reflect실패 원인과 다음 루프의 범위를 줄였는가failure note, 남은 위험, 다음 액션
LOOP-03

State Transition

루프는 상태 기계처럼 운영합니다. 각 iteration은 state transition을 만들고, 상태 전이가 없으면 반복을 멈추고 spec이나 context를 보강해야 합니다.

현재 상태전이 조건다음 상태
Unclear요구사항, 비목표, 검증 기준을 확인Ready to Plan
Ready to Plan가설과 읽을 자료, stop condition 정의Ready to Act
Ready to Act작은 변경 또는 조사 결과 생성Ready to Verify
Ready to Verify테스트, 빌드, 수동 확인 수행Ready to Reflect
Ready to Reflect성공 증거 또는 실패 귀속 기록Done 또는 Next Loop
STATE TRANSITION RULE Every loop must produce one state transition:
unclear → clarified
untested → verified
large risk → smaller risk
missing evidence → captured evidence
If no transition occurs, stop and re-plan.
LOOP-04

Stop Condition

stop condition은 실패 인정이 아니라 품질 통제입니다. 루프가 불확실성을 줄이지 못하면 더 돌리는 것이 아니라 범위를 다시 잡아야 합니다.

stop condition신호대응
unclear spec완료 기준이나 비목표가 없음사용자 또는 owner에게 확인하고 Plan으로 복귀
missing tests변경을 검증할 자동/수동 절차가 없음테스트 후보 또는 수동 체크리스트를 먼저 작성
unsafe permissionDB write, 배포, 고객 데이터 접근이 필요함권한 상승을 멈추고 approval gate로 전환
context gap중요한 정책, schema, API version이 없음Context package를 보강한 뒤 재개
repeated failure같은 검증 실패가 3회 반복구현을 멈추고 Failure Attribution 수행
scope expansion수정 범위가 원래 계획의 2배 이상작업을 쪼개고 새 task spec 작성
LOOP-05

Failure Attribution

Failure Attribution은 “모델이 틀렸다”로 끝내지 않습니다. 실패 원인을 spec, context, tool, permission, verification, model reasoning 중 어디에 둘지 나누어야 다음 루프가 나아집니다.

귀속증상개선
Spec failure요구사항 해석이 계속 바뀜목표, 비목표, 예외, 완료 기준 보강
Context failure중요한 파일·정책·API를 놓침source inventory와 relevance rule 개선
Tool failure도구 결과가 불완전하거나 재현 불가tool adapter 출력과 error handling 개선
Permission failure필요 권한이 없거나 위험 권한을 요구approval gate와 dry-run 경계 명확화
Verification failure완료처럼 보였지만 회귀 발생테스트, lint, manual scenario 추가
Reasoning failure근거가 있는데도 잘못된 결론작업 분해, 리뷰어 분리, 작은 loop 적용
LOOP-06

Loop Log

LOOP LOG TEMPLATE Task: 이번 loop의 목표
Plan: 읽을 자료, 가설, 검증 기준
Act: 실행한 변경 또는 조사
Verify: 테스트, 빌드, 수동 확인 결과
Reflect: 배운 점, 실패 귀속, 남은 위험
state transition: 이번 반복에서 바뀐 상태
stop condition: 계속할지 멈출지 판단한 기준
Next: 다음 loop 또는 reviewer handoff
workflow improvement review

주간 workflow improvement review에서는 성공 사례보다 반복 실패를 먼저 봅니다. unclear spec, missing tests, unsafe permission, context gap이 반복되면 프롬프트가 아니라 업무 정의와 하네스를 고칩니다.
