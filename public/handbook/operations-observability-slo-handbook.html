OBS-00

세 신호

로그는 사건의 세부 맥락, 메트릭은 시간에 따른 수치, 트레이스는 요청 경로를 보여준다. 세 신호가 traceId, service, environment, version으로 연결되어야 장애 시 요청 단위로 원인을 좁힐 수 있다.

운영 질문확인할 증거판단 기준
로그, 메트릭, 트레이스가 같은 요청을 설명할 수 있는가?timestamp, service, env, version, trace_id, request id, deployment marker세 신호가 traceId와 version으로 연결되지 않으면 원인 후보를 좁히기 어렵다.
세 신호를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?log query, RED dashboard, trace waterfall, deploy marker comparison동일 요청의 로그와 trace, 같은 시간대의 metric 변화를 연결한다.
세 신호 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?로그는 있으나 traceId가 없어 특정 사용자 요청을 추적할 수 없음schema 보강과 trace propagation 수정을 영구 개선으로 남긴다.
OBS-01

RED와 USE

HTTP 서비스는 RED(rate, error, duration)를 먼저 본다. 인프라 리소스는 USE(utilization, saturation, errors)를 본다. p95/p99 latency, error rate, saturation은 평균보다 사용자 영향을 잘 드러낸다.

운영 질문확인할 증거판단 기준
서비스 지표와 리소스 지표 중 어떤 신호가 사용자 영향을 더 잘 설명하는가?request rate, error rate, duration p95/p99, utilization, saturation, errorsHTTP 서비스는 RED, 인프라 리소스는 USE로 보되 평균보다 tail과 saturation을 우선한다.
RED와 USE를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?RED/USE dashboard, p95/p99 latency, saturation panel, high-cardinality label reviewCPU 평균이 정상이어도 queue, pool, p99가 나쁘면 사용자 영향 후보로 본다.
RED와 USE 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?평균 latency는 정상이나 p99와 queue saturation만 증가tail latency와 saturation을 기준으로 완화 우선순위를 잡는다.
OBS-02

SLO

SLO는 사용자 경험을 기준으로 잡는 내부 신뢰성 목표다. SLA처럼 외부 보상 약속 자체는 아니지만, 제품 약속과 운영 우선순위의 근거가 된다. SLI를 정의하고 error budget을 계산하면 모든 경고를 paging하지 않고 사용자 영향이 큰 실패에 집중할 수 있다.

운영 질문확인할 증거판단 기준
이 SLO가 실제 사용자 경험을 대표하는지 어떻게 확인하는가?SLI definition, request success rate, latency percentile, freshness, durability, business success사용자 여정의 성공·지연·신선도와 연결되지 않으면 paging 기준으로 쓰지 않는다.
SLO를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?SLO dashboard, error budget remaining, release freeze policy, alert ownerSLO 위반 시 사람이 취할 행동과 owner가 없으면 운영 기준으로 부족하다.
SLO 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?checkout success SLO burn이 fast-burn 기준을 넘음원인 확정 전에도 rollback, feature degrade, traffic shed 같은 영향 완화를 검토한다.
OBS-03

SLO burn rate

SLO burn rate는 error budget이 얼마나 빠르게 소진되는지 보는 방식이다. 짧은 창과 긴 창을 함께 사용하면 급격한 장애와 느린 품질 저하를 모두 잡을 수 있다. 알림에는 owner, dashboard, runbook 링크가 있어야 한다.

운영 질문확인할 증거판단 기준
error budget이 얼마나 빠르게 소진되는지 어떤 창으로 판단하는가?fast burn 5m/1h, slow burn 30m/6h, error budget remaining, paging policy짧은 창과 긴 창이 함께 나빠질 때 즉시 paging하고, 느린 소진은 ticket과 release freeze로 관리한다.
SLO burn rate를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?SLO burn dashboard, alert rule, silence policy, incident runbook알림에는 owner, dashboard, runbook, 완화 선택지가 있어야 한다.
SLO burn rate 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?error rate는 낮아 보이지만 checkout_success_rate burn이 급증평균 지표보다 사용자 성공률과 error budget 소진 속도를 우선한다.
예시 신호관찰값해석
Fast burn5m와 1h 창 모두 error budget 소진 속도 급증즉시 paging하고 완화 조치를 검토한다.
Slow burn30m와 6h 창에서 완만한 품질 저하근무 시간 ticket, release freeze, owner 조치로 관리한다.

실무 체크리스트

Observability·SLO 체크리스트는 세 신호, RED와 USE, SLO, SLO burn rate의 변경 위험을 사전에 드러내기 위한 검토표다. 각 항목은 리뷰, 배포 승인, 운영 인수에서 같은 증거로 다시 확인되어야 한다.

  • SLI를 사용자 행동과 연결해 정의한다.
  • RED, USE, p95/p99, error rate, saturation을 구분한다.
  • log, metric, trace를 traceId와 version으로 연결한다.
  • SLO burn rate와 error budget 기준으로 알림 우선순위를 나눈다.

CASE

장애/운영 시나리오

새 릴리스 후 p99 latency만 증가하고 error rate는 안정적이다. 평균 latency가 아니라 tail latency를 보고, trace에서 특정 downstream 호출이 느려졌는지 확인한다. SLO burn rate가 빠르게 증가하면 paging하고, 느린 소진이면 ticket으로 관리한다.

Triage frame

Observability·SLO 장애는 증상 → 영향 범위 → 최근 변경 → SLI 증거 → 완화책 → 복구 검증 → 재발 방지 순서로 기록한다. 이 흐름이 있어야 담당자가 바뀌어도 같은 결론에 도달한다.

자주 틀리는 판단

Observability·SLO에서 자주 틀리는 판단은 세 신호와 SLO burn rate의 책임 경계를 흐리는 데서 시작된다. 아래 신호가 보이면 설계, runbook, 답변 구조를 다시 점검한다.

  • CPU 알림만 많고 사용자 영향 알림이 없다.
  • 로그는 있지만 traceId가 없어 요청 단위 분석이 안 된다.
  • high-cardinality label을 무분별하게 추가해 비용과 성능을 망친다.
  • SLO 없이 모든 경고를 같은 심각도로 보낸다.
Q&A

면접 답변 템플릿

관측 가능성은 로그, 메트릭, 트레이스를 모으는 것이 아니라 장애 판단 시간을 줄이는 설계라고 답합니다. SLO와 error budget을 기준으로 알림을 나누고, burn rate로 빠른 장애와 느린 품질 저하를 구분합니다.

OPERATIONS ANSWER FRAME 1. 관측 가능성은 수집이 아니라 판단 시간 단축이다.
2. SLI/SLO/error budget으로 알림 우선순위를 나눈다.
3. 로그, 메트릭, 트레이스를 traceId와 version으로 연결한다.
