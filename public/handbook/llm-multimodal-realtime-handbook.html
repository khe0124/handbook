멀티모달·파일·음성·Realtime

멀티모달 LLM 앱은 텍스트 prompt에 이미지나 음성을 붙이는 일이 아닙니다. 입력 매체마다 신뢰도, 비용, 지연, 접근성, 평가 기준이 다르므로 요청 계약과 실패 모드를 따로 설계해야 합니다.

MM-01

MULTIMODAL REQUEST ENVELOPE

멀티모달 요청은 actor, media, extraction goal, risk tier, consent, retention을 함께 기록해야 합니다. 같은 이미지라도 OCR, 시각 질의응답, 품질 검사, 안전 판정은 서로 다른 평가와 보안 기준을 갖습니다.

필드설명실패 모드
media_typeimage, pdf, audio, video, screen, mixed모델이 처리할 수 없는 형식을 늦게 발견
task_typeextract, classify, summarize, compare, answer, act요약 모델에 정밀 추출을 기대
consent_retention사용자 동의, 저장 기간, 삭제 정책민감 이미지·음성의 장기 보관
evidence_requiredbbox, page, timestamp, transcript, citation답은 맞아 보여도 근거 위치가 없음

MM-02

vision extraction과 visual grounding

이미지 이해 기능은 “이미지를 설명한다”와 “증거 위치를 가진 값을 추출한다”를 구분해야 합니다. 업무 자동화에서는 자연어 설명보다 bbox, page, crop, confidence, reviewer override가 더 중요합니다.

작업필수 증거검증
문서 OCR 보정page, bbox, raw_text, normalized_text샘플링 수동 검수와 field-level accuracy
영수증·계약 추출금액·날짜·상대방 필드별 근거 위치schema validation + bbox review
제품 이미지 QA비교 기준 이미지, defect label, cropfalse positive/negative matrix
스크린샷 분석UI element, state, visible textDOM 근거와 시각 근거 충돌 확인
VISION RISK

이미지 안의 텍스트도 prompt injection source가 될 수 있습니다. 스크린샷이나 PDF에 “이전 지시를 무시하라”는 문장이 있어도 문서 데이터로만 처리해야 합니다.

MM-04

audio turn-taking과 voice UX

음성 앱은 STT, VAD, partial transcript, interruption, TTS, latency budget이 함께 움직입니다. 텍스트 챗봇보다 오류 수정과 사용자 중단 처리가 훨씬 중요합니다.

요소좋은 설계실패
VAD말 시작·끝 감지와 침묵 timeout 조정사용자 말이 끊기거나 대기 시간이 길어짐

MM-05

realtime session state machine

Realtime은 websocket을 여는 기능이 아니라 세션 상태 머신입니다. 입력 스트림, 모델 응답, 도구 제안, 사용자 중단, 재연결, 비용 예산을 같은 trace 안에서 관리해야 합니다.

realtime session
created -> listening -> transcribing -> thinking -> speaking
-> interrupted -> tool_pending -> approval_required -> completed
-> reconnecting | failed | cancelled
상태서버 책임로그
listeningaudio buffer와 consent 확인session_id, input_ms
tool_pendingtool schema, permission, idempotency 검증proposed_tool, blocked_reason
interrupted출력과 미승인 tool action 취소cancelled_generation_id
reconnecting중복 실행 방지와 transcript 복구resume_token, replay_range
MM-06

multimodal eval

멀티모달 평가는 텍스트 정답만 비교하면 부족합니다. 위치, 시간, 시각 근거, transcript 품질, 사용자 중단 처리까지 케이스로 고정해야 합니다.

MARKER: concept-depth — 매체가 늘어나면 실패 증거도 늘어난다

텍스트 RAG에서는 citation이 핵심 증거지만, 이미지와 음성에서는 citation만으로 부족합니다. 이미지 추출은 bbox와 crop, PDF는 page와 table cell, 음성은 timestamp와 final transcript, realtime은 session event trace가 있어야 리뷰어가 결과를 검증할 수 있습니다. 증거 단위가 없으면 모델이 맞았는지 사람이 다시 원문 전체를 뒤져야 하고, 자동화 가치는 사라집니다.

MARKER: failure-driven — Realtime을 채팅 API처럼 다루는 실패

Realtime 세션에서 사용자가 말을 끊었는데 서버가 이전 generation과 tool proposal을 취소하지 않으면, 사용자가 정정한 뒤에도 오래된 의도로 외부 전송이나 예약 변경이 실행될 수 있습니다. 세션에는 generation id, cancel token, pending tool action, approval state가 있어야 하며, 재연결 후에도 같은 side effect가 두 번 실행되지 않도록 idempotency가 필요합니다.

MARKER: failure-driven — Vision 추출을 텍스트 요약처럼 평가하는 실패

이미지나 PDF에서 금액, 날짜, 수신자 같은 필드를 추출하는 기능을 최종 답변 텍스트만 보고 평가하면 실제 업무 품질을 놓칩니다. 모델이 총액을 맞췄더라도 bbox가 다른 숫자를 가리키거나, page가 누락되거나, 표의 헤더와 셀을 잘못 연결하면 리뷰어는 오류를 재현할 수 없습니다. 멀티모달 추출은 값의 정답 여부와 근거 위치의 정답 여부를 분리해 측정해야 하며, 낮은 confidence나 겹치는 후보가 있으면 자동 승인 대신 human review queue로 보내야 합니다.
