디자인 핸드오프와 QA

핸드오프는 디자인 파일을 넘기는 이벤트가 아니라 제품 의도를 구현 가능한 계약으로 바꾸는 과정입니다. 좋은 QA는 픽셀 차이만 보는 것이 아니라 상태, 접근성, 반응형, 오류 흐름까지 확인합니다.

QA-01

핸드오프의 목적

목적설명누락 시 문제
의도 전달왜 이 구조와 패턴을 선택했는지 설명구현자가 임의 해석
범위 고정이번 릴리스에 포함되는 상태와 예외 정의개발 중 요구사항 증가
기술 제약 확인데이터, 권한, 성능, 브라우저 제약 반영완성 후 구현 불가 발견
품질 기준 합의QA에서 볼 항목을 사전에 공유출시 직전 반복 수정

QA-03

개발 협업

디자인과 개발의 간극은 “완성 후 검수”가 아니라 “구현 전 계약”에서 줄어듭니다.

협업 지점확인 질문
API 계약UI가 필요한 상태와 오류 코드가 API에 있는가
권한숨김, 비활성, 접근 거부 중 어떤 패턴을 쓰는가
성능목록 크기, 이미지, skeleton, pagination 기준이 있는가
컴포넌트기존 시스템으로 구현 가능한가, 새 패턴이 필요한가
분석어떤 사용자 행동을 이벤트로 남길 것인가

QA-05

출시 전 점검

Design QA checklist 
핵심 흐름 happy path와 실패 경로 확인
loading/empty/error/permission 상태 확인
모바일과 긴 데이터에서 레이아웃 깨짐 확인
키보드만으로 주요 액션 수행 확인
시스템 컴포넌트와 토큰 이탈 여부 확인
분석 이벤트와 성공 지표 확인
릴리스 후

출시 후에는 디자인이 끝난 것이 아닙니다. 사용 로그, 문의, 오류율, 전환율, 세션 리플레이, 고객 피드백을 보고 흐름이 실제로 작동하는지 확인해야 합니다.

QA-06

실무 QA와 회귀 관리

디자인 QA는 시안과 픽셀을 맞추는 일을 넘어 제품 계약이 구현됐는지 확인하는 과정입니다.

QA 유형목적실무 방법
visual diff공통 컴포넌트와 핵심 화면의 시각 회귀 탐지Storybook/Playwright screenshot baseline 관리
Scenario QAhappy path와 실패 경로 검증실제 데이터와 권한 조합으로 테스트
Content QA용어, 날짜, 숫자, 오류 문구 검증운영팀과 고객지원팀 용어까지 확인
Analytics QA개선 여부를 측정할 이벤트 확인클릭보다 완료, 실패, 재시도, 이탈 이벤트 중심
Case study

디자인 QA에서 “버튼 위치가 4px 다르다”만 잡고 loading, empty, permission denied, long text 상태를 놓치면 출시 후 더 큰 문제가 생깁니다. 핸드오프에는 정상 화면뿐 아니라 상태별 스크린, 데이터 예시, 권한별 차이, 측정 이벤트가 포함되어야 합니다.

QA-07

릴리스 품질 게이트

디자인 QA는 출시 직전 감상평이 아니라 release gate여야 합니다. gate는 차단 기준, 예외 승인자, 출시 후 관측 기준이 함께 있어야 작동합니다.

Gate차단 기준통과 증거예외 승인
State coverageloading/empty/error/permission/partial success 누락상태별 스크린샷 또는 StorybookPM + Design + FE lead
Responsive핵심 액션이 모바일에서 가려지거나 겹침320/768/1024px 캡처Design lead
Accessibility키보드로 핵심 흐름 완료 불가keyboard walkthrough, axe/Lighthouse접근성 owner
Analytics성공/실패/이탈 이벤트가 없음event spec, QA logPM
Rollback위험 변경의 되돌림 또는 완화책 없음runbook, feature flag, support planRelease owner
QA-08

디자인 부채 운영

디자인 부채는 “나중에 예쁘게”가 아닙니다. 상태 누락, 용어 불일치, 컴포넌트 이탈, 접근성 결함, 추적 불가능한 예외가 쌓이면 제품 변경 속도와 신뢰성이 떨어집니다.

부채 유형증상운영 방식
State debt오류/권한/빈 상태가 화면마다 다름상태 패턴을 컴포넌트와 템플릿으로 흡수
Content debt같은 개념의 용어가 화면마다 다름controlled vocabulary와 content QA 운영
System debt제품팀별 one-off CSS와 variant 증가component adoption audit와 migration backlog
Research debt결정 근거가 사라져 같은 논쟁 반복decision log와 invalidated assumptions 기록
Accessibility debt키보드, focus, contrast 회귀 반복release gate와 automated check에 포함
DESIGN DEBT TRIAGE Impact: 사용자/운영/개발 속도에 미치는 영향
Frequency: 얼마나 자주 반복되는 문제인가
Blast radius: 몇 개 화면과 팀에 퍼져 있는가
System fix: 개별 수정이 아니라 시스템으로 흡수 가능한가
Owner: 누가 언제 제거할 것인가
