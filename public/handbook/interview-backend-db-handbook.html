백엔드·DB 면접

백엔드·DB 질문은 프레임워크 사용 경험만 묻지 않습니다. 계약, 권한, 정합성, 동시성, 운영 진단성을 기준으로 설계할 수 있는지 확인합니다.

BD-01

Q. 좋은 REST API 설계 원칙은 무엇인가요?

핵심 개념

좋은 API는 예쁜 URL이 아니라 일관된 계약입니다. 리소스 중심 URL, HTTP 메서드 의미, 상태 코드, 에러 포맷, pagination, filtering, versioning, correlation ID까지 포함해야 운영 가능한 API가 됩니다.

개념 해부

REST는 URL 작명 규칙보다 client와 server가 리소스 상태를 어떻게 주고받을지 정하는 제약 집합입니다. 메서드 의미, 상태 코드, 캐시 가능성, 멱등성, 에러 표현이 맞아야 client가 실패와 재시도를 안전하게 처리할 수 있습니다. 좋은 API는 성공 응답뿐 아니라 빈 결과, 권한 거부, 검증 실패, 부분 실패까지 계약으로 다룹니다.

30초 답변

REST API는 리소스 중심 URL, 메서드 의미, 상태 코드, 일관된 에러 포맷이 기본입니다. 실무에서는 pagination, filtering, versioning, backward compatibility, correlation ID까지 같이 설계해야 깨졌을 때도 진단 가능한 계약이 됩니다.

90초 답변

API를 함수 호출처럼 보면 화면 요구에 맞춘 임시 endpoint가 늘어납니다. 저는 API를 계약으로 보고 request/response schema, nullable, enum, error code, 권한 조건, pagination 방식을 문서화해야 한다고 봅니다. 특히 프론트는 raw response에 직접 묶이기보다 adapter에서 view model로 바꿔 API 변경의 충격을 줄이는 편이 안정적입니다.

꼬리질문

질문답변 방향
PUT과 PATCH 차이는?PUT은 리소스 전체 교체 성격, PATCH는 부분 수정 성격입니다. 멱등성도 함께 봅니다.
에러 포맷은 왜 중요한가요?UI가 사용자 메시지, 필드 오류, 재시도 가능 여부를 안정적으로 처리하기 위해 필요합니다.
versioning은 언제 하나요?breaking change를 피할 수 없고 여러 client를 동시에 지원해야 할 때 명시적으로 관리합니다.

오답/주의점

주의

“REST는 URL을 명사로 쓰는 것입니다”에서 끝나면 얕습니다. 운영과 하위 호환성을 같이 말해야 합니다.

면접에서 말할 증거

OpenAPI 문서, 에러 코드 표준, pagination contract, correlation ID가 포함된 로그 예시를 말합니다.

BD-02

Q. API 스펙이 자주 바뀔 때 어떻게 대응하나요?

핵심 개념

API 변경 대응은 endpoint 교체가 아니라 계약 변경 관리입니다. response 구조, status 값, error format, loading/empty/error UI, backward compatibility를 함께 봐야 합니다.

개념 해부

API 변경은 타입 오류보다 의미 오류가 더 위험합니다. 필드 이름이 같아도 nullable 의미, enum 값, 정렬 기준, 오류 코드가 바뀌면 화면의 판단 조건이 틀어질 수 있습니다. adapter layer는 raw response와 화면 모델 사이에 완충 지대를 만들어 변경 범위와 테스트 지점을 명확히 합니다.

30초 답변

먼저 변경되는 필드, nullable, enum, error response, pagination 방식을 명확히 질문합니다. 화면은 raw response에 직접 의존하지 않게 service/adapter layer에서 view model로 변환하고, contract test나 mock으로 변경 영향을 확인합니다.

90초 답변

스펙이 불완전하면 추측으로 컴포넌트를 먼저 묶지 않습니다. 질문 목록을 만들고 최소 계약을 합의한 뒤, 확정 전에는 mock으로 화면 흐름을 만들 수 있습니다. 실제 연결 시점에는 adapter에서 raw response를 화면 모델로 변환하고, error/empty/loading 상태까지 함께 테스트해야 합니다.

꼬리질문

질문답변 방향
adapter가 과한 추상화 아닌가요?API와 화면 모델이 거의 같고 변경 가능성이 낮으면 과할 수 있습니다. 변경 빈도와 변환 복잡도로 판단합니다.
백엔드가 늦으면?mock contract로 화면을 진행하되, 실제 API 연결 전에 계약을 재검증합니다.
필드가 사라지면?optional 처리만 하지 말고 UI 의미와 fallback, error state를 정해야 합니다.

오답/주의점

주의

“타입만 고치면 됩니다”는 부족합니다. 데이터 의미와 사용자 상태까지 바뀔 수 있습니다.

면접에서 말할 증거

기존/신규 response field mapping 표, adapter unit test, MSW mock, nullable/enum 변경 체크리스트를 제시합니다.

BD-03

Q. Validation은 프론트와 백엔드 중 어디에서 처리하나요?

핵심 개념

프론트 validation은 사용성을 위한 빠른 피드백이고, 백엔드 validation은 보안과 비즈니스 규칙의 최종 방어선입니다. DB 제약은 마지막 정합성 방어선입니다.

개념 해부

validation은 계층마다 목적이 다릅니다. 프론트는 사용자가 빨리 고칠 수 있게 안내하고, 서버는 우회 불가능한 비즈니스 규칙과 권한을 검증하며, DB는 동시성 상황에서도 마지막 정합성을 지킵니다. 같은 규칙이 여러 번 보이는 것은 중복이라기보다 서로 다른 실패 모드에 대한 방어선일 수 있습니다.

30초 답변

양쪽에서 처리합니다. 프론트는 required, format, 범위처럼 즉시 알려줄 수 있는 오류를 사용자에게 빠르게 보여주고, 백엔드는 권한, 상태 전이, 동시성, 최종 중복 같은 진실의 규칙을 검증합니다. DB unique나 foreign key도 마지막 방어선입니다.

90초 답변

같은 규칙이 여러 계층에 있어도 중복이 아니라 방어 심도일 수 있습니다. 예를 들어 이메일 형식은 프론트에서 즉시 안내하고 서버에서 최종 검증합니다. 재고 차감, 결제 승인, 권한 변경처럼 서버 상태에 의존하는 검증은 클라이언트가 최종 판단할 수 없습니다. 백엔드 validation 결과는 field, row, page 단위로 UI가 복구 가능하게 매핑해야 합니다.

꼬리질문

질문답변 방향
프론트 검증만으로 충분한가요?아닙니다. 클라이언트는 우회 가능하므로 서버가 최종 방어선입니다.
서버와 프론트 결과가 다르면?서버 결과를 최종으로 보고 UI는 사용자가 수정할 수 있는 위치에 매핑합니다.
DB 제약은 왜 필요한가요?동시 요청이나 앱 버그가 있어도 마지막 정합성을 지키기 위해 필요합니다.

오답/주의점

주의

프론트 검증을 보안으로 설명하지 마세요. 프론트 검증의 1차 목적은 사용성입니다.

면접에서 말할 증거

서버 validation error format, field/row/page 오류 매핑, DB unique violation 처리, 동시 요청 테스트를 말합니다.

BD-04

Q. 인증과 인가는 무엇이 다르고 어떻게 설계하나요?

핵심 개념

인증은 사용자가 누구인지 확인하는 것이고, 인가는 그 사용자가 어떤 리소스에 어떤 행동을 할 수 있는지 판단하는 것입니다.

개념 해부

인증은 actor를 확인하는 과정이고, 인가는 actor가 특정 resource에 특정 action을 해도 되는지 판단하는 과정입니다. 그래서 로그인 성공만으로 API 접근이 끝나지 않고, 요청마다 resource ownership, role, tenant, 상태 조건을 다시 확인해야 합니다. 프론트의 메뉴 숨김은 UX 보조이고 보안 경계는 서버입니다.

30초 답변

인증은 identity 확인, 인가는 permission 판단입니다. 로그인 성공이 곧 모든 API 접근 허용을 의미하지 않습니다. 서버는 매 요청마다 actor, resource, action을 기준으로 권한을 확인해야 하고, 프론트의 라우트 가드는 UX 보조 수단입니다.

90초 답변

JWT와 session은 저장 위치, 만료, revoke, 확장성의 trade-off가 있습니다. RBAC는 역할 기반으로 단순하고 운영하기 쉽지만 세밀한 조건에는 약합니다. ABAC는 속성 기반으로 유연하지만 정책 복잡도가 올라갑니다. 권한이 중요한 서비스는 감사 로그에 actor, target, action, result를 남겨야 합니다.

꼬리질문

질문답변 방향
JWT와 Session 차이는?JWT는 stateless 검증에 유리하지만 revoke와 token 탈취 대응이 어렵고, session은 서버 저장소가 필요하지만 제어가 쉽습니다.
프론트 권한 숨김은 충분한가요?아닙니다. UI 숨김은 UX이고 서버 인가가 최종입니다.
RBAC와 ABAC 차이는?RBAC는 역할 기준, ABAC는 사용자/리소스/환경 속성 기준입니다.

오답/주의점

주의

“로그인했으니 인증/인가가 끝났다”는 표현은 위험합니다. 권한은 리소스와 행동별로 판단해야 합니다.

면접에서 말할 증거

actor/resource/action 기반 인가 정책, 401/403 테스트, 감사 로그 필드, 권한 우회 테스트를 준비합니다.

BD-05

Q. 데이터 모델링은 어떤 기준으로 하나요?

핵심 개념

데이터 모델링은 화면 구조를 DB에 그대로 옮기는 것이 아니라 도메인 규칙, 정합성, 조회 패턴, 변경 가능성을 구조화하는 일입니다.

개념 해부

데이터 모델은 현재 화면을 저장하는 모양이 아니라 도메인 규칙을 오래 유지할 구조입니다. entity의 생명주기, 소유권, 불변 조건, 조회 패턴이 테이블 경계와 제약 조건을 결정합니다. 화면 요구가 바뀌어도 지켜야 하는 규칙은 DB 제약과 서버 validation으로 남아야 합니다.

30초 답변

먼저 도메인 entity와 관계, 생명주기, 소유권, 불변 조건을 봅니다. 그 다음 조회 패턴과 쓰기 패턴을 고려해 정규화와 비정규화, 인덱스, 제약 조건을 결정합니다.

90초 답변

정규화는 중복을 줄이고 정합성을 지키는 데 유리하지만 join 비용과 조회 복잡도가 생길 수 있습니다. 비정규화는 읽기 성능에 유리하지만 동기화와 갱신 비용이 생깁니다. 그래서 사용자가 자주 조회하는 projection, 변경 빈도, 정합성 요구 수준을 보고 결정합니다.

꼬리질문

질문답변 방향
nullable은 어떻게 판단하나요?도메인에서 실제로 없을 수 있는 값인지, 생성 단계에서 잠시 비는 값인지 구분합니다.
soft delete는 언제 쓰나요?감사/복구가 필요하면 쓰지만 unique 제약, 조회 조건, 보관 정책 비용이 생깁니다.
화면 view와 DB 모델이 같아야 하나요?아닙니다. 화면은 사용자 흐름, DB는 정합성과 저장 구조를 기준으로 합니다.

오답/주의점

주의

ERD를 그리는 것만 모델링이라고 말하면 부족합니다. 규칙과 변경 비용을 같이 말해야 합니다.

면접에서 말할 증거

ERD뿐 아니라 불변 조건, unique/foreign key, 조회 패턴, migration 계획을 함께 설명합니다.

BD-06

Q. 인덱스는 어떻게 설계하고 실행 계획에서 무엇을 보나요?

핵심 개념

인덱스는 읽기를 빠르게 하지만 쓰기 비용과 저장 공간을 늘립니다. 쿼리 패턴, 조건, 정렬, selectivity, cardinality, 실행 계획을 보고 설계해야 합니다.

개념 해부

인덱스는 별도 자료구조를 유지해 읽기 경로를 줄이는 대신 쓰기마다 갱신 비용을 추가합니다. 복합 인덱스는 왼쪽부터 어떤 조건으로 좁히고 어떤 정렬을 제거할지에 따라 효과가 달라집니다. 실행 계획을 볼 때는 인덱스 사용 여부만 보지 말고 추정 row와 실제 row 차이, sort, join 방식, buffer 접근을 함께 봐야 합니다.

30초 답변

인덱스는 느린 쿼리에 무조건 추가하는 것이 아니라 실제 where, join, order by, pagination 패턴에 맞춰 설계합니다. 복합 인덱스는 컬럼 순서가 중요하고, 실행 계획으로 index scan 여부, rows estimate, filter, sort, join 방식을 확인합니다.

90초 답변

복합 인덱스는 equality 조건, range 조건, 정렬 순서를 고려합니다. selectivity가 낮은 컬럼만 단독 인덱스로 두면 효과가 작을 수 있습니다. 실행 계획에서는 full scan, sort 비용, nested loop/hash join, rows estimate와 실제 rows 차이, buffer hit, lock wait 가능성까지 봅니다.

꼬리질문

질문답변 방향
인덱스가 많으면 왜 위험한가요?쓰기마다 인덱스 갱신 비용이 생기고 저장 공간과 planner 선택 복잡도가 커집니다.
covering index는?쿼리에 필요한 컬럼을 인덱스만으로 만족해 table access를 줄이는 방식입니다.
offset pagination 문제는?뒤 페이지로 갈수록 skip 비용이 커질 수 있어 cursor pagination을 검토합니다.

오답/주의점

주의

“인덱스를 걸면 빨라집니다”는 반쪽 답입니다. 쓰기 비용과 실제 실행 계획 검증을 같이 말해야 합니다.

면접에서 말할 증거

`EXPLAIN ANALYZE`, rows estimate와 actual rows 차이, buffer hit, sort 제거 여부, index 추가 전후 latency를 말합니다.

BD-07

Q. 트랜잭션 격리 수준은 어떻게 판단하나요?

핵심 개념

트랜잭션 격리 수준은 정합성과 처리량의 균형입니다. 강한 격리는 안전하지만 락, 대기, deadlock, 처리량 저하 비용을 만들 수 있습니다.

개념 해부

격리 수준은 동시 트랜잭션이 서로를 얼마나 보게 할지 정하는 정책입니다. 높은 격리는 이상 현상을 줄이지만 lock 대기와 deadlock, 처리량 저하가 생길 수 있습니다. 도메인이 허용하지 않는 불일치를 먼저 정의하고, isolation, constraint, lock, retry 중 어떤 조합으로 막을지 선택해야 합니다.

30초 답변

모든 구간에 가장 강한 격리를 쓰는 것이 답은 아닙니다. 결제, 재고, 정산처럼 틀리면 안 되는 구간은 더 강하게 보호하고, 일반 조회는 성능과 비용을 고려합니다. 격리 수준뿐 아니라 unique 제약, 낙관적 락, 재시도 전략, 트랜잭션 범위를 함께 봅니다.

90초 답변

Read committed, repeatable read, serializable은 dirty read, non-repeatable read, phantom read 같은 현상을 얼마나 막는지 다릅니다. 다만 실제 동작은 PostgreSQL, MySQL InnoDB처럼 DB 구현에 따라 차이가 있으므로 교과서 표만 외우면 위험합니다. 설계에서는 도메인이 어느 정도 불일치를 허용하는지 먼저 보고, 트랜잭션이 길면 lock 보유 시간이 늘어나므로 외부 API 호출을 트랜잭션 안에 넣는 것도 피해야 합니다.

BD-08

Q. JPA N+1 문제는 왜 생기고 어떻게 해결하나요?

핵심 개념

N+1은 처음 조회한 N개 entity의 연관 데이터를 접근할 때 추가 쿼리가 N번 발생하는 문제입니다. ORM의 lazy loading과 조회 패턴을 이해해야 합니다.

개념 해부

ORM은 객체 그래프 탐색을 SQL로 바꿔주지만, 연관 객체 접근 시점마다 쿼리가 나가는 사실을 없애지는 않습니다. N+1은 목록의 각 row가 lazy association을 건드리며 추가 쿼리를 만드는 패턴입니다. 해결책은 fetch join 하나가 아니라 화면이 필요한 데이터 모양에 맞춰 projection, batch, entity graph를 고르는 것입니다.

30초 답변

N+1은 목록을 한 번 조회한 뒤 각 row의 연관 데이터를 접근하면서 추가 쿼리가 반복되는 문제입니다. 해결은 fetch join, entity graph, batch size, DTO projection 등을 조회 목적에 맞게 선택하고, 실제 SQL 로그와 실행 계획으로 확인합니다.

90초 답변

fetch join은 한 번에 연관 데이터를 가져와 쿼리 수를 줄이지만 pagination이나 중복 row 문제가 생길 수 있습니다. DTO projection은 화면에 필요한 컬럼만 가져와 효율적일 수 있지만 entity 변경 추적과는 성격이 다릅니다. 핵심은 ORM 설정을 외우는 것이 아니라 목록 화면, 상세 화면, 통계 화면의 조회 목적에 맞는 전략을 고르는 것입니다.

꼬리질문

질문답변 방향

BD-09

Q. JWT와 Session은 어떻게 비교하나요?

핵심 개념

JWT와 Session은 인증 상태를 어디에 두고 어떻게 검증, 만료, 철회할지에 대한 선택입니다.

개념 해부

JWT와 Session은 인증 상태를 client에 검증 가능한 token으로 둘지, 서버 저장소에 둘지의 trade-off입니다. JWT는 확장과 stateless 검증에 유리하지만 탈취 후 만료 전 제어가 어렵고, session은 중앙 revoke가 쉽지만 저장소와 확장 전략이 필요합니다. 보안 판단은 저장 위치, 만료, 갱신, 철회, CSRF/XSS 위협을 함께 봐야 합니다.

30초 답변

JWT는 서명된 token을 stateless하게 검증할 수 있어 확장에 유리하지만 탈취, 만료 전 철회, 권한 변경 반영이 어렵습니다. Session은 서버 저장소가 필요하지만 revoke와 중앙 제어가 쉽습니다.

90초 답변

JWT를 쓰더라도 access token은 짧게 두고 refresh token rotation, 재사용 탐지, 저장 위치를 고민해야 합니다. Cookie 기반이면 CSRF 방어가 필요하고, localStorage는 XSS에 취약합니다. Session은 sticky session, shared store, Redis 장애, TTL 정책을 봐야 합니다.

Q. RBAC와 ABAC는 어떻게 다르나요?

핵심 개념

개념 해부

RBAC는 역할이 권한 묶음이 되는 모델이고, ABAC는 사용자·리소스·환경 속성으로 정책을 계산하는 모델입니다. B2B SaaS에서는 role만으로는 tenant, 소유자, 상태 조건을 표현하기 어려운 경우가 많습니다. 정책이 세밀해질수록 권한 matrix, 테스트, 감사 로그가 없으면 운영자가 정책을 신뢰하기 어렵습니다.

30초 답변

RBAC는 관리자, 매니저, 사용자처럼 역할별 권한을 관리해 단순하고 운영하기 쉽습니다. ABAC는 부서, 소유자, 리소스 상태, 시간 같은 조건을 반영할 수 있어 유연하지만 정책 복잡도와 테스트 비용이 커집니다.

90초 답변

B2B 서비스는 역할만으로 부족한 경우가 많습니다. 예를 들어 같은 매니저라도 자기 조직의 리소스만 수정할 수 있다면 role뿐 아니라 resource owner, tenant, status가 필요합니다. 정책이 복잡해질수록 audit log, 정책 테스트, 권한 matrix 문서가 중요해집니다.

Q. 정규화와 비정규화는 어떻게 판단하나요?

핵심 개념

정규화는 중복을 줄이고 정합성을 높이며, 비정규화는 조회 성능과 단순성을 위해 중복을 허용하는 선택입니다.

개념 해부

정규화는 중복을 줄여 원본을 하나로 만들고, 비정규화는 읽기 경로를 줄이기 위해 중복을 의도적으로 받아들입니다. 비정규화된 값은 원본이 아니므로 갱신 실패와 재계산 전략이 필요합니다. 성능 병목이 측정되기 전 비정규화를 먼저 하면 데이터 불일치와 마이그레이션 비용을 앞당길 수 있습니다.

30초 답변

기본은 도메인 정합성을 지키기 위해 정규화로 시작하고, 실제 조회 패턴과 성능 병목이 확인되면 projection, summary table, denormalized column을 검토합니다. 비정규화에는 동기화 실패 비용이 따라옵니다.

90초 답변

주문 금액, 통계, 검색 화면처럼 매번 join과 집계가 부담되는 경우 비정규화가 유리할 수 있습니다. 하지만 원본이 바뀔 때 파생 데이터 갱신이 실패하면 사용자에게 틀린 값을 보여줍니다. 그래서 이벤트 기반 갱신, batch 재계산, 검증 job 같은 보정 전략이 필요합니다.

BD-12

Q. 낙관적 락과 비관적 락은 언제 쓰나요?

핵심 개념

낙관적 락은 충돌이 드물다고 보고 저장 시 version으로 감지하고, 비관적 락은 충돌을 막기 위해 먼저 잠급니다.

개념 해부

낙관적 락은 충돌을 사후 감지하고, 비관적 락은 충돌을 사전 차단합니다. 충돌이 드문 문서 편집은 version 기반 감지가 적합하고, 재고 차감처럼 동시에 통과하면 안 되는 경우는 row lock이나 조건부 update가 필요할 수 있습니다. 어떤 방식이든 사용자 충돌 UX와 retry 정책이 없으면 운영에서 실패합니다.

30초 답변

충돌이 드물고 재시도나 사용자 충돌 해결이 가능하면 낙관적 락이 적합합니다. 재고 차감처럼 동시에 수정되면 안 되고 충돌 가능성이 높으면 비관적 락이나 원자적 update를 검토합니다.

90초 답변

낙관적 락은 처리량에 유리하지만 충돌 시 UX와 retry 전략이 필요합니다. 비관적 락은 안전하지만 lock wait, deadlock, timeout 비용이 생깁니다. 어떤 락을 쓰든 트랜잭션 범위를 짧게 하고 lock 순서를 통일해야 합니다.

BD-13

Q. idempotency key는 왜 필요한가요?

핵심 개념

네트워크 timeout이나 retry 때문에 같은 요청이 여러 번 도착해도 부작용이 한 번만 일어나게 하는 장치입니다.

개념 해부

idempotency key는 client 재시도와 네트워크 불확실성을 서버가 견디게 하는 계약입니다. 같은 key와 같은 payload는 같은 결과를 돌려주고, 다른 payload가 같은 key를 쓰면 오류로 보아야 합니다. key 기록과 실제 부작용은 같은 transaction 또는 원자적 경계에 있어야 중복 실행을 막을 수 있습니다.

30초 답변

결제, 주문 생성, 포인트 차감처럼 중복 실행되면 안 되는 요청은 client가 idempotency key를 보내고 서버가 key별 처리 결과를 저장합니다. 같은 key가 다시 오면 새로 처리하지 않고 기존 결과를 반환합니다.

90초 답변

중요한 점은 key 저장과 실제 비즈니스 변경이 원자적으로 연결되어야 한다는 것입니다. 처리 중 상태, 성공 결과, 실패 결과, key TTL, payload mismatch 검증을 설계해야 합니다. 단순히 프론트 버튼을 disable하는 것은 네트워크 retry나 탭 중복 요청을 막지 못합니다.

꼬리질문

질문답변 방향
payload가 다르면?같은 key에 다른 payload면 오류로 보고 재사용을 막습니다.
처리 중 재요청은?processing 상태를 반환하거나 잠시 후 재시도하게 합니다.
TTL은?업무 특성과 재시도 가능 시간에 맞춰 정합니다.

면접에서 말할 증거

unique key 제약, 처리 상태 테이블, timeout 재시도 테스트, 중복 결제 방지 케이스를 말합니다.

오답/주의점

주의

HTTP 메서드가 멱등하다는 의미와 비즈니스 부작용이 안전하다는 의미를 혼동하지 마세요.

BD-14

Q. API versioning과 backward compatibility는 어떻게 관리하나요?

핵심 개념

API 변경 관리는 여러 client가 다른 속도로 배포되는 현실에서 계약을 깨지 않도록 운영하는 일입니다.

개념 해부

API 하위 호환성은 모든 client가 동시에 배포되지 않는다는 현실에서 출발합니다. 필드 추가는 보통 안전하지만 필드 삭제, 의미 변경, enum 추가, error format 변경은 client를 깨뜨릴 수 있습니다. versioning보다 더 중요한 것은 deprecation 기간, 사용량 관측, contract test, 단계적 제거 절차입니다.

30초 답변

가능하면 additive change로 새 필드를 추가하고 기존 필드 의미를 깨지 않습니다. breaking change가 필요하면 versioning, deprecation 기간, migration guide, contract test로 client 영향을 관리합니다.

90초 답변

nullable 변경, enum 추가, status code 변경, pagination 방식 변경은 화면에 큰 영향을 줍니다. 서버는 구버전 client 지원 기간을 정하고, 프론트는 adapter layer와 contract test로 변경 충격을 줄입니다. 배포 순서도 서버 먼저 하위 호환 변경을 내고 client 전환 후 제거하는 식으로 나눕니다.

꼬리질문

질문답변 방향
버전 위치는?URL, header, media type 방식이 있고 조직 운영 방식에 맞춥니다.
enum 추가는 breaking?client가 exhaustive 처리하면 깨질 수 있어 주의합니다.
제거는 언제?사용량 관측, 공지, migration 완료 후 제거합니다.

면접에서 말할 증거

deprecation 공지, API usage metric, contract test, adapter mapping 표를 말합니다.

오답/주의점

주의

서버와 프론트를 동시에 배포하면 된다는 답은 모바일/외부 client를 고려하지 못한 답입니다.

BD-15

Q. DB 제약과 애플리케이션 검증은 어떻게 나누나요?

핵심 개념

애플리케이션 검증은 사용자에게 의미 있는 오류와 비즈니스 흐름을 제공하고, DB 제약은 동시성과 버그 상황에서도 마지막 정합성을 지킵니다.

개념 해부

애플리케이션 검증은 오류를 사람이 이해하고 고칠 수 있게 만들고, DB 제약은 프로그램 실수와 동시성 경쟁에서도 데이터 불변식을 지킵니다. 예를 들어 중복 이메일은 서버가 친절히 알려줄 수 있지만, 동시에 들어온 요청은 unique constraint가 최종 방어선입니다. DB 오류도 그대로 500으로 내보내지 말고 도메인 오류로 변환해야 합니다.

30초 답변

서버 validation은 권한, 상태 전이, 비즈니스 규칙을 판단하고 DB unique, foreign key, not null, check constraint는 최종 방어선으로 둡니다. 같은 규칙이 중복되어 보일 수 있지만 목적이 다릅니다.

90초 답변

예를 들어 이메일 중복은 서버에서 미리 확인해 사용자에게 친절한 메시지를 줄 수 있지만, 동시 가입 요청은 DB unique가 최종적으로 막아야 합니다. 서버는 DB constraint violation도 domain error로 변환해 UI가 복구 가능한 위치에 표시하게 해야 합니다.

꼬리질문

질문답변 방향

constraint violation은?기술 오류를 domain error로 변환합니다.
마이그레이션은?기존 데이터 정리, NOT VALID/validate, lock 위험을 고려합니다.

동시성 테스트, unique violation 매핑, migration 검증 쿼리, 오류 포맷을 설명합니다.

BD-16

Q. slow query는 어떤 순서로 대응하나요?

핵심 개념

slow query 대응은 인덱스 추가부터가 아니라 재현, 실행 계획, 데이터 분포, lock, 애플리케이션 호출 패턴을 함께 보는 일입니다.

개념 해부

slow query는 SQL 한 줄만의 문제가 아닐 수 있습니다. 실행 계획이 좋아도 호출 횟수가 N+1이면 느리고, 쿼리는 빨라도 connection pool 대기나 lock wait이 있으면 endpoint는 느립니다. 그래서 재현 조건, trace, 실행 계획, 데이터 분포, lock, 호출 패턴을 같은 타임라인에서 봐야 합니다.

30초 답변

먼저 어떤 endpoint에서 어떤 조건으로 느린지 재현하고, `EXPLAIN ANALYZE`로 scan, join, sort, rows estimate, buffer를 확인합니다. 이후 인덱스, 쿼리 구조, pagination, projection, 캐시, 배치화를 비용 대비로 선택합니다.

90초 답변

쿼리 하나만 빠르게 해도 endpoint가 빨라지지 않을 수 있습니다. N+1, connection pool 대기, lock wait, 외부 API 대기, 너무 큰 response도 같이 봐야 합니다. 인덱스 추가는 쓰기 비용과 배포 위험이 있으므로 운영 DB에서는 concurrent 생성, rollback, 모니터링을 고려합니다.

꼬리질문

질문답변 방향

인덱스 배포는?lock 위험과 쓰기 비용을 보고 안전한 생성 방식을 씁니다.

면접에서 말할 증거

주의

느리면 인덱스부터 추가한다는 답은 원인 분리가 약해 보입니다.

BD-18

백엔드·DB 개념 인과관계

백엔드 면접에서 개념을 이해한 답변은 요청 하나가 controller, service, transaction, DB lock, index, connection pool을 지나며 어떤 비용을 만드는지 설명합니다.

BACKEND DB CAUSAL MODELAPI가 느려지는 원인은 controller 코드가 많아서가 아니라 대개 대기 시간이 생기기 때문입니다. transaction이 길면 DB connection을 오래 점유하고, 같은 row를 수정하면 lock wait가 생기며, 인덱스가 없으면 scan이 늘어 buffer와 CPU를 씁니다. connection pool이 꽉 차면 새 요청은 DB에 도달하기 전부터 대기하고, retry가 겹치면 이미 느린 DB에 더 많은 요청을 밀어 넣습니다. 따라서 좋은 답변은 “인덱스를 추가한다”가 아니라 “WHERE, JOIN, ORDER BY 패턴 때문에 scan이 발생하고, 선택도가 높은 컬럼과 정렬 조건을 기준으로 복합 인덱스를 검토하되 쓰기 비용과 lock 시간을 같이 봅니다”로 이어져야 합니다.

면접 답변 전환

“트랜잭션은 ACID입니다”에서 멈추면 암기입니다. “트랜잭션은 여러 변경을 하나의 실패 단위로 묶지만, 길어질수록 connection과 lock을 오래 잡아 동시성을 해칩니다. 그래서 외부 API 호출은 transaction 밖으로 빼고, DB 변경과 이벤트 발행 사이 간극은 outbox로 줄입니다”까지 말해야 합니다.
