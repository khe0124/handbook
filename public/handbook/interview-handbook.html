기술면접 개요

이 섹션은 풀스택 개발자 기술면접을 분야별 질문-답변 핸드북으로 연습하기 위한 입구입니다. 목표는 정답 암기가 아니라, 질문을 받았을 때 정확한 개념, 선택 기준, trade-off, 실패 모드, 본인 경험의 경계를 안정적으로 말하는 것입니다.

FORMAT : 질문 → 핵심 개념 → 30초 답변 → 90초 답변 → 꼬리질문 → 오답/주의점
IV-00

사용 목적

상급 면접에서 면접관은 단순 지식량보다 “어떤 기준으로 판단하는가”를 봅니다. 그래서 각 문서는 짧게 말할 답변과 깊게 파고드는 꼬리질문 대응을 함께 제공합니다.

문서용도먼저 볼 상황
프론트엔드 면접브라우저, React, 상태, 렌더링, 성능, 접근성, 보안, 테스트프론트엔드 기술 깊이 질문 대비
백엔드·DB 면접API 계약, 인증/인가, validation, 모델링, 인덱스, 트랜잭션풀스택 확장성과 서버 이해도 검증 대비
인프라·운영 면접요청 흐름, DNS/TLS/CDN, Docker, CI/CD, 로그, 장애 대응운영 감각과 배포 질문 대비
분산 시스템 면접캐시, 큐, retry, consistency, Saga, Outbox, MSA난이도 높은 설계 꼬리질문 대비
시스템 설계 면접대시보드, Data Grid, 실시간, 복잡한 폼, 인증 앱 설계화이트보드형 구조화 질문 대비
프로젝트 심층 면접대표 프로젝트, Data Grid, Excel, validation, Flow Editor, API 전환이력서 기반 deep dive 대비
컬처·압박 면접자기소개, 실패, 갈등, 일정 압박, 경험 검증 질문과장 없이 신뢰도를 지키는 답변 연습
IV-01

답변 레벨

면접에서 가장 위험한 것은 모르는 것이 아니라, 직접 한 일과 이론으로 아는 일을 같은 톤으로 말하는 것입니다.

레벨의미안전한 표현
A직접 구현했고 장애/운영/리뷰 꼬리질문까지 답할 수 있음“직접 구현했고, 이 부분에서는 이런 문제가 있었습니다.”
B일부 경험했고 구조와 판단 기준을 설명할 수 있음“프로젝트에서 일부 다뤘고, 이 기준으로 접근했습니다.”
C직접 production 경험은 제한적이나 개념과 설계 기준을 알고 있음“직접 운영 경험은 제한적이지만, 설계한다면 먼저 이 위험을 봅니다.”
주의

경험이 제한적인 영역을 명확히 말하는 것은 약점이 아니라 신뢰도 관리입니다. 대신 “그래서 어떤 기준으로 검증하겠다”까지 이어가야 합니다.

IV-02

답변 구조

BASE ANSWER FRAME 1. 한 줄 결론
2. 정확한 개념 정의
3. 왜 실무에서 중요한지
4. 선택 기준과 대안
5. trade-off와 실패 모드
6. 직접 경험 또는 설계 지식의 경계
7. 검증 방법, 지표, 회고
길이구성목표
30초정의 + 핵심 기준 1~2개 + 결론면접관이 “이 사람은 핵심을 안다”고 느끼게 한다
90초정의 + 장단점 + 선택 기준 + 짧은 사례개념을 실무 판단으로 연결한다
3분대안 비교 + 운영 리스크 + 실패 경험 + 검증꼬리질문에서 깊이를 열고 신뢰도를 만든다
IV-03

문서별 사용법

처음부터 모든 문서를 읽지 말고, 면접 유형과 약점에 따라 순서를 정합니다.

상황읽는 순서완료 기준
기술 깊이 면접프론트엔드 → 백엔드·DB → 분산 시스템각 문서에서 5개 질문을 30초로 답한다
풀스택 전환 검증백엔드·DB → 인프라·운영 → 프로젝트 심층직접 경험과 설계 지식을 구분해 답한다
시스템 설계 면접시스템 설계 → 분산 시스템 → 인프라·운영요구사항 확인 질문부터 시작한다
이력서 deep dive프로젝트 심층 → 컬처·압박 → 프론트엔드대표 프로젝트 3개를 문제-구조-결과로 말한다
IV-04

연습 루프

면접 준비는 읽은 양보다 말해본 횟수로 결정됩니다. 같은 질문을 짧게 답하고, 꼬리질문에서 깊이를 여는 연습을 반복합니다.

PRACTICE LOOP 1. 오늘의 문서에서 질문 5개를 고른다
2. 각 질문을 30초 답변으로 녹음한다
3. 꼬리질문 2개를 붙여 90초 답변으로 확장한다
4. 과장 표현과 모호한 단어를 지운다
5. 직접 경험, 일부 경험, 설계 지식을 표시한다
6. 다음 날 같은 질문을 더 짧고 정확하게 다시 말한다
