FE-PER-02

대표 답변

Q. 가장 자신 있는 프론트엔드 경험은 무엇인가요?

30초복잡한 업무 UI를 안정적으로 만드는 경험이 가장 강합니다. 대량 데이터 입력, Data Grid, Form validation, Excel/PDF, 권한별 화면, 인증 만료 같은 흐름을 다뤘고, 단순히 화면을 그리는 것보다 사용자가 실패했을 때 어떻게 복구할지와 서버 API 계약을 함께 보며 구현했습니다.

Q. 성능 최적화 경험을 설명해보세요.

90초기업 웹사이트에서 이미지와 번들, 폰트 로딩을 중심으로 성능을 개선했습니다. 전체 이미지를 next/Image로 바꾸고 명시적 크기를 지정해 layout shift를 줄였고, dynamic import와 bundle analyzer로 초기 번들에 들어갈 필요가 없는 의존성을 분리했습니다. 폰트 preload와 blur placeholder도 적용했습니다. 정확한 수치가 남아 있지 않은 부분은 과장하지 않고, 어떤 병목을 어떤 구조로 줄였는지 중심으로 설명합니다.

Q. i18n 경험은 어떤 수준인가요?

90초단순 문자열 치환보다 locale별 날짜, 통화, 메시지 누락, 언어별 리소스 관리까지 경험했습니다. React 서비스에서는 공통 라벨 hook과 번역 사전을 정리했고, Next.js 서비스에서는 Lingui 기반으로 한·영·일 메시지를 운영하며 빈 메시지 검출 스크립트를 빌드 흐름에 연결했습니다.

질문답변 방향

정량 성능 수치가 있나요?없는 것은 없다고 말하고, 대신 병목 가설, 변경 구조, 재현 조건, 앞으로 측정할 지표를 말합니다.

Vue와 jQuery 경험은 오래된 것 아닌가요?오히려 프레임워크 변화 속에서도 DOM, 스타일링, 사용자 흐름을 이해하고 React/Next로 확장한 성장 서사로 연결합니다.

FE-PER-05

대체 증거

성과 수치가 없을 때는 개선율을 만들지 말고, 작업 규모와 복잡도 증거를 제시합니다. 커밋 수는 성과가 아니라 “얼마나 오래, 얼마나 넓게 만졌는지”를 보여주는 보조 증거입니다.

증거어떻게 말할지주의점
커밋 수“해당 프로젝트에서 690 작업 커밋 규모로 초기 구축부터 리뉴얼까지 참여했습니다.”커밋 수를 생산성이나 품질 수치처럼 말하지 않습니다.
티켓 수“116개 Jira 티켓에 걸쳐 권한, grid, upload, editor, bugfix를 다뤘습니다.”티켓 난이도 차이가 있으므로 범위 증거로만 씁니다.
반복 개선 횟수“거래내역 상세와 필터는 60~80회 이상 반복 개선한 핵심 화면이었습니다.”정확한 출처가 있는 경우에만 말합니다.
운영 기간“기업 웹사이트는 3.5년 동안 단독 운영하며 성능, SEO/AEO, i18n, 보안을 순차적으로 개선했습니다.”운영 기간을 성과로만 말하지 말고 개선 흐름과 연결합니다.
FE-PER-06

심화 드릴 연결

심화 기술 질문을 받았을 때는 “알고 있습니다”가 아니라 본인 경험의 증거와 운영 관점으로 연결합니다. 아래 문장은 프론트엔드·JS/TS 원본 문서의 심화 섹션을 개인 답변으로 바꾸는 연결고리입니다.

마커압박 질문개인화 답변 방향
REACT CONCURRENT SSR DRILLReact/Next를 썼다면 Strict Mode, Suspense, streaming SSR 문제도 다뤄봤나요?Strict Mode에서 effect cleanup과 중복 요청을 먼저 의심하고, Suspense/Error Boundary는 화면 전체를 막지 않는 경계로 둔다고 답합니다. Next 경험은 SEO/AEO, dynamic import, hydration mismatch 방지처럼 실제 운영 품질과 연결합니다.
FRONTEND SECURITY CACHE DRILL프론트엔드 보안과 성능 캐시는 어디까지 책임질 수 있나요?최종 인가는 서버 책임이지만 프론트는 XSS sink 제거, token/cookie trade-off 설명, SameSite 한계 인식, 개인화 응답 cache 정책, preload/resource priority 남용 방지를 책임진다고 답합니다.
TYPESCRIPT ADVANCED BOUNDARYTypeScript를 단순 annotation 이상으로 써본 경험이 있나요?외부 API와 form input은 runtime schema validation 또는 adapter에서 검증하고, satisfies, as const, strict flags, branded type은 도메인 계약을 더 선명하게 만드는 도구라고 답합니다.

짧은 답변 예시

90초React/Next 프로젝트에서는 단순히 화면을 구현하는 것보다 서버에서 만든 HTML과 클라이언트 첫 렌더가 어긋나지 않게 하는 점, effect가 재실행되어도 요청과 listener가 중복되지 않게 정리하는 점을 중요하게 봤습니다. 보안은 token 저장 위치를 단정하지 않고 XSS와 CSRF trade-off를 비교하며, 성능은 HTTP cache와 preload가 실제 LCP 후보에 맞는지 확인합니다. TypeScript는 API 응답이 타입과 다를 수 있다는 전제로 adapter와 validation 경계를 둔다고 설명합니다.
