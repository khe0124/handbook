UI 레이아웃과
시각 위계

레이아웃은 요소를 배치하는 일이 아니라 사용자의 시선을 설계하는 일입니다. 좋은 화면은 어떤 정보가 중요하고, 어떤 행동을 먼저 해야 하며, 어떤 것은 나중에 봐도 되는지 구조적으로 알려줍니다.

LAY-01

레이아웃의 역할

역할설명실무 예시
맥락 제공현재 화면의 목적과 위치를 알려줌상단 제목, breadcrumb, 탭
비교 지원정보를 같은 기준으로 나란히 놓음테이블, 카드 리스트, 요약 패널
행동 유도primary action과 보조 행동을 구분저장 버튼, 필터, bulk action
인지 부하 감소관련 정보를 묶고 불필요한 선택을 줄임섹션 분리, progressive disclosure
LAY-02

그리드와 간격

간격은 장식이 아니라 정보의 관계를 표현합니다. 가까운 것은 관련 있고, 멀리 떨어진 것은 다른 그룹으로 인식됩니다.

요소권장 원칙나쁜 신호
Grid페이지 유형별 반복 가능한 열 구조를 둠화면마다 폭과 정렬 기준이 다름
Spacing4 또는 8 기반 scale을 일관되게 사용13px, 17px 같은 임의 값이 증가
Alignment텍스트, 숫자, 액션의 기준선을 맞춤버튼과 입력이 미세하게 흔들림
Density업무 화면은 스캔 가능한 밀도를 유지카드가 과하게 커서 정보 비교가 어려움
실무 기준

운영 도구, 관리자 화면, 대시보드는 넓은 여백보다 빠른 스캔과 비교가 중요합니다. 마케팅 페이지와 같은 hero-scale 구성을 업무 UI에 그대로 쓰면 반복 사용성이 떨어집니다.

LAY-03

시각 위계

도구쓰임주의점
크기화면 목적과 핵심 수치를 강조모든 제목을 크게 만들면 위계가 사라짐
굵기레이블, 값, 액션의 중요도 구분bold 남용은 스캔을 방해
색상태, 위험, 브랜드 액션을 표현색만으로 의미 전달 금지
대비읽기 가능성과 상태 구분 확보비활성 텍스트가 너무 흐려짐
위치사용자가 먼저 보는 영역에 핵심 배치중요 액션이 화면 하단에 숨어 있음
LAY-04

반응형 구조

반응형은 화면을 줄였을 때 요소가 그냥 접히는 것이 아니라 우선순위가 재배치되는 것입니다.

Reflow재배치
열 수, 카드 폭, 사이드바 표시 방식이 viewport에 맞게 바뀝니다.
Priority우선순위
모바일에서는 보조 정보보다 핵심 행동과 상태가 먼저 보여야 합니다.
Overflow넘침
긴 텍스트, 표, 코드, 버튼 라벨이 깨지지 않도록 별도 규칙을 둡니다.
Touch target터치
모바일 액션은 손가락 입력 크기와 인접 요소 간격을 고려합니다.
LAY-05

리뷰 기준

질문확인 방법
이 화면에서 가장 중요한 정보가 3초 안에 보이는가제목, 핵심 수치, primary action만 가리고 다시 보기
관련 정보가 시각적으로 묶여 있는가간격과 배경 없이도 그룹 관계가 읽히는지 확인
같은 종류의 정보가 같은 형식으로 반복되는가테이블, 카드, 필터, 상태 badge 비교
좁은 화면에서 액션과 텍스트가 겹치지 않는가모바일 viewport와 긴 데이터로 확인
LAY-06

인지 법칙과 업무 UI

레이아웃 판단은 감각만으로 하지 않습니다. 사용자의 시선, 선택지 수, 조작 거리, 반복 업무의 피로를 기준으로 봅니다.

원리의미적용
Fitts가까우면서 큰 목표가 더 빠르게 클릭됨자주 쓰는 primary action은 충분한 크기와 예측 가능한 위치에 둠
Hick선택지가 많을수록 판단 시간이 증가필터와 액션을 그룹화하고 progressive disclosure 사용
Gestalt proximity가까운 요소를 관련 있다고 인식레이블, 입력, 오류 메시지를 같은 그룹으로 묶음
Serial position처음과 마지막 항목이 더 기억됨핵심 정보와 위험 액션의 위치를 의도적으로 설계
Case study

운영자가 하루 수백 건의 승인 요청을 처리하는 화면에서 카드형 큰 레이아웃은 보기 좋지만 처리 속도를 떨어뜨릴 수 있습니다. 테이블 밀도를 높이고, 상태 badge와 keyboard shortcut, 우측 상세 drawer를 결합하면 시각적 화려함보다 업무 처리량이 크게 개선됩니다.

인터랙션 디자인 패턴

인터랙션은 사용자의 행동과 시스템의 반응 사이에 신뢰를 만드는 설계입니다. 버튼을 눌렀을 때 무엇이 일어났는지, 기다려야 하는지, 취소할 수 있는지, 실패하면 어떻게 복구하는지가 명확해야 합니다.

INT-01

피드백

피드백사용 시점주의점
Hover/Pressed클릭 가능한 요소임을 알림터치 환경에는 hover 의존 금지
Loading요청이 진행 중임을 알림중복 제출 방지와 함께 설계
Toast흐름을 막지 않는 짧은 결과 알림중요한 오류를 toast에만 의존하지 않음
Inline message입력 오류나 특정 영역 문제오류 위치와 해결 방법을 같이 제공
Empty state데이터 없음, 첫 사용, 필터 결과 없음권한 부족과 데이터 없음 구분
INT-02

선택과 실행

선택 UI는 사용자가 무엇을 고르는지, 실행 UI는 선택 후 어떤 결과가 생기는지 분리해서 설계합니다.

패턴적합한 상황나쁜 사용
Tabs같은 맥락의 보기 전환서로 다른 페이지 이동에 사용
Segmented control작은 옵션 집합 전환옵션이 5개 이상으로 증가
Dropdown공간 절약이 필요한 단일 선택자주 쓰는 핵심 액션을 숨김
Command menu전문가 사용자의 빠른 작업기본 탐색을 대체해서 초보자를 막음
Bulk action여러 항목 반복 처리선택 범위와 영향이 불명확
INT-03

확인과 되돌리기

상황권장 패턴이유
복구 가능한 삭제Undo toast흐름을 막지 않고 실수를 회복
복구 어려운 삭제Confirmation modal영향 범위를 명확히 확인
권한/과금 변경요약 화면 + 명시적 승인조직과 비용 영향이 큼
긴 작업 취소Cancel + 진행 상태사용자가 통제감을 유지
CONFIRMATION FATIGUE

모든 행동에 확인 모달을 붙이면 사용자는 내용을 읽지 않고 확인을 누릅니다. 위험도, 복구 가능성, 빈도에 따라 undo, confirmation, permission gate를 구분해야 합니다.

INT-04

비동기 상태

Idle→ Pending→ Success/ Error→ Retry
상태UI 책임
Pending중복 입력 방지, skeleton 또는 progress 제공
Optimistic즉시 반영하되 실패 시 rollback 기준 제공
Partial success성공한 항목과 실패한 항목을 분리 표시
Timeout작업이 계속되는지 실패인지 사용자가 알 수 있게 함
INT-05

패턴 선택

빈도frequency
자주 하는 행동일수록 마찰을 줄이고 단축 경로를 제공합니다.
위험도risk
복구 어려운 행동일수록 영향 범위와 확인 절차가 필요합니다.
지연latency
느린 작업은 진행 상태, 취소, 재시도 기준을 명확히 둡니다.
숙련도expertise
초보자에게는 명시적 경로, 숙련자에게는 빠른 명령 경로를 제공합니다.
INT-06

고급 인터랙션 판단

인터랙션 패턴은 멋진 효과가 아니라 지연, 실패, 위험, 반복 사용을 다루는 운영 전략입니다.

패턴적합한 상황실패 조건
Optimistic UI실패 가능성이 낮고 즉시 피드백이 중요한 작업돈, 권한, 삭제처럼 실패 영향이 큰 작업
Pessimistic submit서버 확인 후 상태가 바뀌어야 하는 작업대기 시간이 긴데 진행 상태가 없음
Undo복구 가능한 단일 작업법적/재무적으로 즉시 확정되어야 하는 작업
Inline edit짧고 반복적인 수정검증 규칙이 복잡하고 실수 비용이 큰 입력
Case study

이메일 클라이언트의 “보내기 취소”는 실제 전송을 취소한다기보다 몇 초간 전송을 지연시키는 설계입니다. 제품은 사용자가 느끼는 통제감을 높이지만 시스템은 내부적으로 지연 큐와 상태 표시를 가져야 합니다. 인터랙션은 UI와 백엔드 처리 모델이 함께 설계될 때 안전해집니다.

폼과 입력 경험

폼은 사용자에게 일을 시키는 인터페이스입니다. 좋은 폼은 필요한 데이터만 묻고, 입력 형식을 예측 가능하게 만들며, 오류가 생겼을 때 사용자가 빠르게 고칠 수 있게 합니다.

FORM-01

폼의 목적

목적설계 기준예시
데이터 수집업무에 필요한 최소 필드만 요청회원가입, 견적 요청
설정 변경현재 값, 영향 범위, 저장 결과 표시권한, 알림 설정
검색/필터반복 조작이 빠르고 되돌리기 쉬움기간, 상태, 담당자 필터
대량 입력복사, 붙여넣기, 기본값, 행 단위 오류 지원CSV 업로드, bulk edit
FORM-02

입력 설계

요소좋은 기준피해야 할 것
Label항상 보이고 필드 목적을 명확히 표현placeholder를 label처럼 사용
Placeholder예시나 형식 보조에만 사용입력 후 사라져야 할 중요 설명
Help text왜 필요한지, 어떤 형식인지 설명긴 정책 문구를 필드 아래에 반복
Default대부분 사용자에게 안전한 값을 제공위험한 선택을 기본값으로 둠
Input type날짜, 숫자, 이메일 등 목적에 맞는 컨트롤 사용모든 것을 자유 텍스트로 받음
FORM-03

검증과 오류

오류 메시지는 사용자를 탓하는 문구가 아니라 복구 안내입니다.

검증 단계설명UI 책임
Client validation형식, 필수 값, 범위 같은 즉시 확인필드 근처에 빠르게 표시

컴포넌트 패턴

컴포넌트는 반복되는 UI 조각이 아니라 제품 의사결정을 재사용 가능한 단위로 만든 것입니다. 좋은 컴포넌트는 역할, 상태, 접근성, 사용 제한이 명확합니다.

CMP-01

컴포넌트의 역할

역할설명예시
일관성같은 의미의 UI가 같은 모습과 행동을 가짐Primary button, danger action
속도반복 설계와 구현 비용을 줄임Form field, modal, table
품질접근성, 상태, 오류 처리를 기본 포함Focus trap, aria, keyboard

기본 컴포넌트

컴포넌트핵심 질문필수 상태
Button이 행동의 중요도와 위험도는 무엇인가default, hover, active, disabled, loading
Input무엇을 어떤 형식으로 입력하는가empty, focused, filled, error, disabled
Select검색이 필요한 선택인가 단순 선택인가open, selected, empty, no result
Modal흐름을 막을 만큼 중요한가open, submitting, error, close guard
Badge상태인지 분류인지 우선순위인지neutral, success, warning, danger
CMP-03

상태와 변형

variant는 색상 조합이 아니라 의미의 이름이어야 합니다.

좋은 variant나쁜 variant이유

dangerredShadow위험 행동이라는 의미가 중요
quietgrayNoBorder보조 행동인지 스타일인지 구분

COMPONENT CONTRACT

컴포넌트는 디자이너와 개발자가 공유하는 계약입니다. 어떤 prop이나 variant를 제공하는지보다 어떤 사용을 금지하는지가 더 중요할 때가 많습니다.

패턴구성 요소설계 포인트

Search panelquery, filter, empty, recent, suggestion결과 없음과 검색 전 상태 구분

API 관점

Semantic prop의미
스타일 지시보다 제품 의미를 표현하는 속성을 우선합니다.
Composition합성

컴포넌트 품질과 채택

컴포넌트는 만들어지는 것보다 제품팀이 일관되게 채택하는 것이 더 어렵습니다.

bug recurrence같은 접근성/상태 버그 반복 여부컴포넌트 내부로 품질 기준 이동

접근성 심화 운영

접근성은 WCAG 체크리스트를 한 번 통과하는 일이 아니라 컴포넌트, 콘텐츠, QA, 릴리스에 계속 포함되어야 하는 운영 품질입니다.

기준실무 적용주의점
WCAG대비, 키보드, 오류 식별, 이름/역할/값 기준을 제품 품질로 채택점수만 보고 실제 사용 흐름을 놓치지 않음
ARIA Authoring Practicesdialog, tabs, menu, combobox 같은 복잡한 패턴의 keyboard model 참조ARIA role만 붙이고 키보드 동작을 빼먹지 않음
Assistive tech matrix주요 조합을 정해 반복 검증모든 조합을 매번 수동 검증하기는 어려우므로 핵심 흐름 중심
Inclusive content오류, 성별, 이름, 시간 제한, 인증 흐름에서 배제 여부 점검법적 동의와 UX 설명을 혼동하지 않음
Case study

아이콘만 있는 “삭제” 버튼에 tooltip만 붙이면 마우스 사용자에게는 설명되지만 스크린리더와 터치 사용자에게는 이름이 없을 수 있습니다. 버튼에는 accessible name을 제공하고, 위험 행동이면 결과와 복구 가능성까지 명확히 설명해야 합니다.

프로토타입과 사용성 테스트

DOC : DESIGN-PROTOTYPING-TESTING-V1 SCOPE : PROTOTYPE · USABILITY TEST · RESEARCH · DECISION LANG : KO

프로토타입과
사용성 테스트

프로토타입은 완성도를 보여주기 위한 산출물이 아니라 위험한 가정을 싸게 검증하는 도구입니다. 사용성 테스트는 사용자를 평가하는 일이 아니라 설계가 사용자의 목표 달성을 돕는지 확인하는 과정입니다.

TEST-01

프로토타입 목적

목적검증 질문적합한 산출물
개념 검증이 문제가 실제로 중요한가스토리보드, low-fi wireframe
흐름 검증사용자가 목표까지 도달하는가Clickable prototype
상호작용 검증상태와 피드백이 이해되는가Interactive prototype
핸드오프 검증개발 가능한 구조인가Component spec, responsive variant
TEST-02

충실도 선택

충실도좋은 사용주의점
Low fidelity문제 정의, 흐름, 정보구조 검증시각 완성도 논의로 흐르지 않게 함
Mid fidelity레이아웃, 우선순위, 폼 구조 검증실제 데이터 길이를 넣어야 함
High fidelity브랜드, 상호작용, 최종 QA 근접 검증만들기 전에 핵심 가정이 검증되어야 함
Code prototype성능, 복잡한 interaction, 기술 제약 검증제품 코드로 착각하지 않도록 범위 명시
TEST-03

테스트 설계

좋은 사용성 테스트는 “마음에 드세요?”가 아니라 사용자가 실제 과업을 수행하게 합니다.

항목설계 기준예시
Participant실제 사용자 특성과 업무 맥락 반영신규 운영자, 숙련 관리자
Task제품 목표와 연결된 현실적인 과업실패한 결제 3건을 찾아 재처리
Scenario정답 버튼을 암시하지 않는 상황 설명“고객 문의가 들어왔습니다”
Metric완료 여부, 시간, 오류, 확신도 관찰성공률, 도움 요청 횟수
TEST-04

관찰과 분석

관찰 신호해석후속 질문
멈춤다음 행동이 명확하지 않음무엇을 찾고 있었나요
오클릭레이블이나 affordance 혼동왜 그 요소를 선택했나요
뒤로가기 반복정보구조나 탐색 위치 문제어디에 있을 거라고 예상했나요
확신 없음결과 피드백 부족완료됐다고 느꼈나요
실무 팁

사용자가 말한 선호보다 실제 행동을 더 신뢰합니다. “괜찮다”고 말해도 과업 중 헤매거나 되묻는다면 설계 문제로 기록해야 합니다.

TEST-05

의사결정

Usability finding format 
Issue: 사용자가 필터 적용 여부를 인지하지 못함
Evidence: 5명 중 3명이 같은 필터를 반복 클릭
Impact: 검색 결과 신뢰 저하, 처리 시간 증가
Recommendation: 적용된 필터 영역과 reset action 노출
Severity심각도
사용자 목표 달성 실패, 반복 빈도, 비즈니스 영향으로 판단합니다.
Confidence확신도
참여자 수보다 관찰의 일관성과 맥락 적합성을 봅니다.
Next step조치
수정, 추가 검증, 보류 중 하나로 의사결정을 남깁니다.
TEST-06

사용성 테스트 심화

테스트 결과는 “좋다/나쁘다”가 아니라 어떤 가정이 깨졌고 어떤 의사결정을 해야 하는지로 정리해야 합니다.

지표의미주의점
Task success rate과업을 완료한 비율도움을 받아 성공한 경우는 별도 표시
Time on task목표 달성까지 걸린 시간업무 숙련도와 데이터 복잡도를 함께 기록
Error count오클릭, 잘못된 해석, 반복 시도 횟수사용자 실수로 돌리지 말고 설계 신호로 봄
System Usability Scale전반적 사용성 인식 점수소규모 테스트에서는 정성 관찰과 함께 해석
Case study

결제 실패 복구 플로우를 테스트할 때 성공 결제만 보여주면 핵심 위험을 놓칩니다. 카드 승인 실패, 네트워크 timeout, 중복 결제 의심, 영수증 재발급처럼 실제 문의로 이어지는 실패 시나리오를 포함해야 제품 리스크를 줄일 수 있습니다.

디자인 핸드오프와 QA

핸드오프는 디자인 파일을 넘기는 이벤트가 아니라 제품 의도를 구현 가능한 계약으로 바꾸는 과정입니다. 좋은 QA는 픽셀 차이만 보는 것이 아니라 상태, 접근성, 반응형, 오류 흐름까지 확인합니다.

QA-01

핸드오프의 목적

목적설명누락 시 문제
의도 전달왜 이 구조와 패턴을 선택했는지 설명구현자가 임의 해석
범위 고정이번 릴리스에 포함되는 상태와 예외 정의개발 중 요구사항 증가
기술 제약 확인데이터, 권한, 성능, 브라우저 제약 반영완성 후 구현 불가 발견
품질 기준 합의QA에서 볼 항목을 사전에 공유출시 직전 반복 수정
QA-02

스펙 구성

스펙포함 내용예시
Flow진입, 완료, 취소, 실패, 재개초대 발송 → 대기 → 수락/만료
Stateloading, empty, error, disabled, dirty저장 실패 후 재시도
Data실제 길이, null, 권한별 노출긴 회사명, 값 없음
Responsivebreakpoint별 배치와 우선순위테이블 → 카드 또는 horizontal scroll
Interactionkeyboard, focus, modal, toast, undoEsc 닫기, focus 복귀
QA-03

개발 협업

디자인과 개발의 간극은 “완성 후 검수”가 아니라 “구현 전 계약”에서 줄어듭니다.

협업 지점확인 질문
API 계약UI가 필요한 상태와 오류 코드가 API에 있는가
권한숨김, 비활성, 접근 거부 중 어떤 패턴을 쓰는가
성능목록 크기, 이미지, skeleton, pagination 기준이 있는가
컴포넌트기존 시스템으로 구현 가능한가, 새 패턴이 필요한가
분석어떤 사용자 행동을 이벤트로 남길 것인가
QA-04

디자인 QA

범주점검 항목도구/방법
Visual간격, 정렬, 타이포그래피, 색, 상태시안 비교, screenshot review
Responsive모바일, 태블릿, 데스크톱, 긴 텍스트브라우저 viewport 테스트
Interactionhover, focus, loading, error, modal수동 시나리오 테스트
Accessibility키보드, contrast, aria label, focus orderaxe, Lighthouse, 수동 확인
Content문구, 용어, 숫자/날짜 포맷, 빈 상태실제 데이터 기반 리뷰
QA-05

출시 전 점검

Design QA checklist 
핵심 흐름 happy path와 실패 경로 확인
loading/empty/error/permission 상태 확인
모바일과 긴 데이터에서 레이아웃 깨짐 확인
키보드만으로 주요 액션 수행 확인
시스템 컴포넌트와 토큰 이탈 여부 확인
분석 이벤트와 성공 지표 확인
릴리스 후

출시 후에는 디자인이 끝난 것이 아닙니다. 사용 로그, 문의, 오류율, 전환율, 세션 리플레이, 고객 피드백을 보고 흐름이 실제로 작동하는지 확인해야 합니다.

QA-06

실무 QA와 회귀 관리

디자인 QA는 시안과 픽셀을 맞추는 일을 넘어 제품 계약이 구현됐는지 확인하는 과정입니다.

QA 유형목적실무 방법
visual diff공통 컴포넌트와 핵심 화면의 시각 회귀 탐지Storybook/Playwright screenshot baseline 관리
Scenario QAhappy path와 실패 경로 검증실제 데이터와 권한 조합으로 테스트
Content QA용어, 날짜, 숫자, 오류 문구 검증운영팀과 고객지원팀 용어까지 확인
Analytics QA개선 여부를 측정할 이벤트 확인클릭보다 완료, 실패, 재시도, 이탈 이벤트 중심
Case study

디자인 QA에서 “버튼 위치가 4px 다르다”만 잡고 loading, empty, permission denied, long text 상태를 놓치면 출시 후 더 큰 문제가 생깁니다. 핸드오프에는 정상 화면뿐 아니라 상태별 스크린, 데이터 예시, 권한별 차이, 측정 이벤트가 포함되어야 합니다.
