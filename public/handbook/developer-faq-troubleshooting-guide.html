FAQ-01

질문을 문제 정의로 바꾸기

초보자와 숙련자의 차이는 답을 아는 양보다 질문을 재현 가능한 문제로 바꾸는 속도에서 크게 납니다. 질문이 "안 돼요"에 머물면 도와주는 사람도 추측만 하게 됩니다.

나쁜 질문좋은 질문왜 좋은가
로그인이 안 됩니다Chrome에서 로그인 POST는 200인데 다음 API에 쿠키가 안 붙습니다. SameSite, Secure, Domain, CORS credential 설정을 비교했습니다.HTTP 계층과 쿠키 계층으로 범위를 좁힘
빌드가 깨져요로컬 Node 20에서는 통과하지만 CI Node 18에서 optional dependency 로딩이 실패합니다. lockfile과 native package 차이를 의심합니다.환경 차이와 가설이 있음
DB가 느립니다주문 목록 p95가 200ms에서 2.4s로 늘었고, EXPLAIN에서 created_at 정렬 후 filesort가 발생합니다.증상, 지표, 실행 계획이 있음
MRE TEMPLATE — 도움 요청 전 최소 정보 증상: 무엇이 기대와 다르게 보이는가
재현: 어떤 입력/명령/브라우저/환경에서 100% 또는 간헐적으로 나는가
범위: 마지막 정상 시점, 최근 변경, 영향 사용자/기능
증거: 에러 전문, 로그 앞뒤 30줄, 네트워크 요청, 실행 계획, 스크린샷
시도: 이미 해본 것과 배제한 가설
요청: 무엇을 판단해 달라는 것인가
실무 기준

질문은 "정답을 달라"가 아니라 "내 가설이 맞는지 검토해 달라"에 가까워야 합니다. 재현 조건, 마지막 정상 시점, 최근 변경, 관측 증거가 있으면 대부분의 문제는 질문하는 과정에서 절반 이상 좁혀집니다.

FAQ-02

로컬은 되는데 CI·운영은 실패

가장 흔한 질문 중 하나입니다. 이 문제는 거의 항상 환경 차이, 숨은 상태, 외부 의존성, 권한 중 하나입니다.

분류확인할 것대표 증상
런타임 버전Node/JDK/Python/Gradle/npm/pnpm 버전, OS, CPU 아키텍처로컬은 통과, CI에서 native binding 또는 syntax 오류
lockfilelockfile이 커밋됐는지, CI가 frozen install을 쓰는지팀원마다 다른 transitive dependency 설치
환경변수필수 env 누락, build-time env와 runtime env 구분배포 후 undefined, API base URL 오작동
파일시스템대소문자 구분, 상대 경로, working directorymacOS는 되는데 Linux CI에서 import 실패
네트워크·권한private registry, DB, secret store, firewall 접근 권한CI에서만 패키지 install 또는 migration 실패
DIFF COMMANDS — 환경 차이 먼저 고정 node -v && npm -v && java -version && ./gradlew -v
pwd && uname -a && env | sort
npm ci # 또는 pnpm install --frozen-lockfile
git status --short # 먼저 변경 상태 확인
git diff > /tmp/wip.patch # 커밋 전 변경 백업
git clean -xfdn # dry-run: 삭제될 파일만 미리 보기
git clean -xfd # 정말 지워도 되는 대상임을 확인한 뒤에만
파괴적 명령 주의

git clean -xfd는 untracked 파일과 ignored 파일까지 삭제할 수 있습니다. 로컬 DB dump, 임시 문서, 개인 설정, 생성 산출물이 날아갈 수 있으므로 항상 git clean -xfdn으로 대상 목록을 먼저 보고, 필요한 파일을 백업한 뒤 실행합니다. 의존성만 의심된다면 node_modules, build cache처럼 명확한 대상만 제거하는 쪽이 더 안전합니다.

예방 규칙

팀 프로젝트는 "내 PC에서 됨"을 기준으로 삼지 않습니다. 런타임 버전은 .nvmrc, .tool-versions, mise.toml처럼 파일로 고정하고, CI는 lockfile 기반 설치만 허용합니다. 로컬 성공보다 깨끗한 checkout에서 재현되는지가 더 강한 증거입니다.

DECISION TREE어디서 갈라졌는가

질문YES면NO면
깨끗한 checkout에서 재현되는가?프로젝트 설정 또는 코드 문제로 본다로컬 캐시, 전역 도구, 숨은 파일, IDE 설정을 의심한다
같은 런타임 버전인가?lockfile, env, OS 차이로 이동버전 파일과 CI 이미지부터 맞춘다

증거 묶음

이슈에 올릴 때는 실패 로그만 붙이지 말고 성공 환경과 실패 환경의 차이를 표로 붙입니다. 런타임 버전, OS/arch, install 명령, lockfile hash, env key 목록, 최근 배포 SHA, 실패 명령을 같이 보면 리뷰어가 추측을 줄일 수 있습니다.

FAQ-03

의존성 문제는 "패키지 하나가 안 깔림"처럼 보이지만 실제로는 lockfile, transitive dependency, peer dependency, native package, repository 권한 문제가 섞여 있습니다.

Gradle이 갑자기 느려졌습니다daemon 상태, dependency resolution, test task, build cacheclean build와 incremental build를 분리해 측정한다

FAQ-04

환경변수·쿠키·CORS·시크릿

배포 후 로그인, API 호출, redirect가 깨지는 문제는 대부분 설정 경계에서 발생합니다. 특히 build-time env와 runtime env, 브라우저 쿠키 정책, CORS credential 설정이 자주 섞입니다.

증상가능한 원인확인
프론트에서 API base URL이 이상함빌드 시점 env가 번들에 박힘빌드 로그와 배포 artifact의 env 주입 방식 확인
로그인은 200인데 세션 유지 안 됨SameSite, Secure, Domain, path, credential 누락DevTools Application 탭과 Set-Cookie header 비교
CORS 에러가 브라우저에서만 남서버 CORS, preflight, allowed headers/methods 불일치OPTIONS 응답과 실제 요청 header 확인
운영 secret이 undefinedsecret store 키 누락, 배포 대상 namespace 오류CI/CD variable scope와 runtime env dump의 redacted 목록 확인
CORS 오해

CORS는 서버 간 통신 문제가 아니라 브라우저 보안 정책입니다. curl은 되는데 브라우저가 막는다면 네트워크가 아니라 Origin, credential, preflight 응답을 봐야 합니다. 반대로 서버 로그에 요청 자체가 없다면 DNS, route, firewall, TLS를 먼저 봅니다.

시크릿 대응

시크릿이 로그나 git에 노출되면 "삭제"가 아니라 폐기와 재발급이 먼저입니다. 노출된 값은 이미 복사됐다고 가정하고 rotate한 뒤, 히스토리 정리와 secret scanning을 진행합니다.

COOKIE DEBUG세션 쿠키가 안 붙을 때

확인의미흔한 수정
Set-Cookie가 응답에 있는가서버가 쿠키를 발급했는지인증 서버 응답, proxy header, HTTPS 종료 지점 확인
브라우저 Application 탭에 저장됐는가브라우저 정책에 의해 거부됐는지Secure, SameSite=None, Domain/Path 조정
다음 요청에 Cookie header가 붙는가credential 포함 여부fetch/axios credentials, 서버 Access-Control-Allow-Credentials
preflight가 통과하는가실제 요청 전 OPTIONS에서 막히는지allowed origin/header/method를 실제 요청과 일치
proxy가 header를 제거하지 않는가LB, CDN, gateway가 쿠키를 바꾸는지forwarded proto, cookie allowlist, cache policy 확인
FAQ-05

DB 마이그레이션·락·시딩

DB 문제는 로컬에서 쉽게 고치려고 drop/reset을 반복하다가 운영 사고로 이어질 수 있습니다. 로컬 재생성과 운영 변경은 완전히 다른 절차입니다.

스코프

이 장의 명령과 예시는 PostgreSQL 기준입니다. EXPLAIN (ANALYZE, BUFFERS), pg_stat_activity, pg_locks, NOT VALID, concurrent index 같은 표현은 PostgreSQL 기능입니다. MySQL, MariaDB, SQL Server, Oracle에서는 같은 원칙을 적용하되 실행 계획, lock 조회, online DDL 문법을 해당 DB 기준으로 바꿔야 합니다.

질문실무 답변위험 신호
컬럼 하나 추가인데 왜 조심해야 하나요?테이블 크기, default, NOT NULL, index 생성 방식에 따라 lock과 rewrite가 생긴다운영 대용량 테이블에 한 번에 NOT NULL + default + backfill
마이그레이션이 실패했는데 수정해도 되나요?이미 공유/운영에 적용된 migration은 수정하지 말고 새 migration으로 보정한다기존 migration 파일을 고쳐 팀원 DB와 history가 갈라짐
시더는 왜 멱등이어야 하나요?여러 번 실행해도 같은 상태가 되어야 로컬 복구와 테스트가 안정적이다재실행할 때마다 중복 row 또는 다른 ID 생성
느린 쿼리는 인덱스만 추가하면 되나요?실행 계획, cardinality, 정렬, pagination, lock 비용을 같이 본다EXPLAIN 없이 감으로 index 추가
POSTGRES FIRST LOOK EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
SELECT * FROM pg_stat_activity WHERE state <> 'idle';
SELECT * FROM pg_locks WHERE NOT granted;
SELECT relname, n_live_tup, n_dead_tup FROM pg_stat_user_tables;
운영 변경 원칙

운영 DB 변경은 expand-contract로 쪼갭니다. 먼저 nullable 컬럼과 호환 코드, 그 다음 backfill, 검증, 제약 강화, 마지막으로 옛 코드/컬럼 제거 순서입니다. "한 번에 끝내기"보다 "중간 상태도 서비스 가능한가"가 중요합니다.

FAQ-11

동시성·멱등성·중복 처리

개발자가 자주 놓치는 운영 버그는 "한 번만 일어난다"는 가정에서 나옵니다. 실제 환경에서는 사용자가 두 번 누르고, 네트워크가 재시도하고, 메시지가 중복 소비되고, 같은 요청이 동시에 들어옵니다.

질문실무 답변검증
중복 클릭은 프론트에서 막으면 충분한가요?아니다. UI disable은 UX 개선이고, 서버는 idempotency key나 unique constraint로 최종 방어해야 한다같은 요청을 동시에 2번 보내도 결과가 하나인지
retry를 넣으면 안전해지나요?retry는 장애 완화와 중복 부작용을 동시에 만든다. timeout, backoff, idempotency가 세트다외부 API timeout 후 재시도 시 중복 결제/발송 없음
락을 걸면 해결되나요?락은 정합성을 주지만 latency와 deadlock 비용이 있다. DB 제약, optimistic lock, queue 직렬화를 비교한다경쟁 조건 테스트와 lock wait 지표
메시지 큐는 exactly-once인가요?대부분 실무에서는 at-least-once로 보고 consumer를 멱등하게 만든다같은 message id를 재처리해도 결과 동일
CONCURRENCY TEST IDEAS 같은 API를 같은 payload로 동시에 10회 호출
timeout 직후 같은 idempotency key로 재시도
consumer를 처리 중 강제 종료 후 같은 메시지 재전달
unique constraint 위반 시 사용자에게 409와 복구 경로 제공
optimistic lock 실패 시 최신 상태 재조회 후 재시도 안내
핵심 원칙

멱등성은 "실패해도 다시 해볼 수 있는가"의 기반입니다. 결제, 주문, 초대, 알림, 파일 업로드, 배치 작업처럼 외부 부작용이 있는 기능은 요청 ID, 상태 전이표, 중복 처리 정책을 설계 문서에 남겨야 합니다.

dashboard가 증상 확인용과 원인 탐색용으로 나뉘는가
로그 레벨

INFO는 정상적인 상태 전이와 운영자가 알아야 할 사건, WARN은 사용자 영향 전 단계의 위험 신호, ERROR는 요청 실패나 데이터 정합성 위험에 씁니다. 디버깅 편의를 위해 모든 것을 ERROR로 올리면 알림이 무력화됩니다.

FAQ-13

보안·권한·데이터 삭제

보안 질문은 "막았나요?"가 아니라 "어느 신뢰 경계에서 무엇을 검증하나요?"로 바꿔야 합니다. 프론트 가드, API 인가, DB 제약, 감사 로그는 서로 대체재가 아닙니다.

반복 질문실무 기준테스트
프론트에서 버튼 숨기면 권한 처리인가요?아니다. UI는 편의이고 서버가 actor, action, resource를 최종 검증한다직접 API 호출로 403 확인
내 데이터만 보게 하려면?소유권 조건을 query와 authorization policy에 모두 반영한다다른 tenant/resource id로 접근 시 404/403
삭제는 hard delete가 맞나요?복구, 감사, 개인정보 삭제 요구, 참조 무결성을 보고 soft/hard/anonymize를 나눈다삭제 후 목록, 상세, export, audit log 확인
관리자 기능은 안전한가요?관리자일수록 감사 로그, 2인 승인, 변경 전후 diff가 필요하다actor/target/action/result/ip/userAgent 기록
에러 메시지는 자세할수록 좋은가요?사용자에게는 복구 가능한 메시지, 로그에는 내부 원인을 남긴다권한/존재 여부를 과도하게 노출하지 않는지
데이터 삭제의 함정

삭제 버튼 하나는 UI 기능이 아니라 데이터 수명주기 정책입니다. 연결된 파일, 검색 인덱스, 캐시, 이벤트, 백업, 분석 테이블, 외부 시스템까지 어디에 복제됐는지 모르면 "삭제됨"이라고 말할 수 없습니다.

FAQ-14

문서화·온보딩·지식 이전

문서화는 시간이 남을 때 하는 일이 아니라 질문 비용을 줄이는 개발 생산성 장치입니다. 같은 질문이 세 번 나오면 문서가 없거나, 찾기 어렵거나, 최신성이 깨졌다는 신호입니다.

문서포함할 내용최신성 신호
README실행, 테스트, 환경변수, 주요 명령, 흔한 실패새 팀원이 30분 안에 앱을 띄움
Runbook증상, 영향, 확인 명령, 완화, 롤백, 담당자장애 중 그대로 따라 할 수 있음
ADR맥락, 선택지, 결정, 포기한 것, 재검토 조건왜 이렇게 했는지 과거 사람에게 묻지 않아도 됨
FAQ반복 질문, 원인 후보, 증거, 해결, 예방슬랙 답변이 문서 링크로 대체됨
Onboarding checklist계정, 권한, 로컬 실행, 첫 PR, 배포 관찰첫 주마다 같은 질문이 줄어듦
RUNBOOK ENTRY TEMPLATE 증상: 사용자가 어떻게 느끼는가
영향: 기능, 사용자, 데이터, 금액, SLA
확인: 대시보드, 로그 쿼리, DB 쿼리, 외부 상태 페이지
완화: flag off, rollback, scale, queue pause, read-only 전환
복구: 데이터 보정, 재처리, cache purge, 고객 공지
예방: test, alert, guardrail, 문서 변경
문서의 완료 기준

좋은 문서는 길이가 아니라 행동 가능성으로 평가합니다. 새 사람이 문서만 보고 실행할 수 있고, 장애 중인 사람이 문서만 보고 완화할 수 있고, 리뷰어가 문서만 보고 결정 배경을 이해할 수 있으면 충분히 좋은 문서입니다.

FAQ-16

60문항 문제 해결 매트릭스

아래 문항은 팀 FAQ, 온보딩, PR 리뷰, 장애 회고에서 반복적으로 꺼내야 할 질문입니다. 모두 외우는 표가 아니라, 문제를 받았을 때 빠뜨린 축이 없는지 확인하는 checklist입니다.

판정의미차단 기준기본 owner
P0사용자 영향, 데이터 손상, 보안 노출, 운영 중단 가능성merge/deploy 차단. 완화책과 owner 승인 전 진행 금지기능 owner + 운영/보안 owner
P1운영 중 장애나 재작업 가능성이 높음해당 증거 또는 테스트 없으면 merge 차단기능 owner

P3개선 권장, 현재 릴리즈의 직접 위험은 낮음후속 backlog로 관리팀 backlog owner
매트릭스 사용법

각 문항은 "예/아니오"가 아니라 증거가 있는가로 판정합니다. 보안, 권한, 데이터 삭제, migration, rollback, secret, 사용자 영향 알림은 기본 P0/P1로 보고, 증거가 없으면 PR 설명에 위험과 후속 owner를 남깁니다. 단순 문서 누락이나 온보딩 개선은 P2/P3로 낮출 수 있습니다.

FAQ-17

운영 준비도 루브릭

이 문서가 실제로 충분한지 보려면 "내용이 많다"가 아니라 기능·문서·자동화가 아래 루브릭을 통과하는지 봐야 합니다. 각 기능이나 PR을 이 표로 점검하면 실무 가이드가 행동 기준으로 바뀝니다.

수준상태통과 기준
0동작만 함happy path는 되지만 실패, 권한, 관측, 롤백 기준이 없다
1개발 가능로컬 실행, 기본 테스트, 환경변수 문서, 에러 처리 일부가 있다
2리뷰 가능PR 설명, 테스트 증거, 위험 영역, 롤백 가능성이 정리돼 있다
3배포 가능migration/flag/config가 안전하고, health check와 기본 알림이 있다
4운영 가능p95/p99, error rate, traceId, runbook, incident update template이 있다
5조직 자산장애 회고와 반복 질문이 FAQ, test, guardrail, onboarding에 반영된다
SHIP READINESS GATE 계약: API 요청/응답/에러/권한이 문서화됐는가
데이터: migration, backfill, rollback, deletion policy가 있는가
실패: timeout, retry, duplicate, partial failure가 테스트됐는가
관측: dashboard, log field, traceId, alert가 연결됐는가
보안: authz, secret, PII, audit log, abuse case가 검토됐는가
운영: runbook, owner, escalation, status update template이 있는가
AI: AI 생성/수정 코드라면 risk tier와 human verification이 남았는가
충분함의 기준

이 가이드가 충분한지는 문서 자체가 아니라 팀의 반복 질문 수, 리뷰 지연, 장애 복구 시간, flaky test 수, 재작업률이 줄어드는지로 판단합니다. 문서는 운영 지표를 바꾸지 못하면 지식 저장소일 뿐이고, 지표를 바꾸면 개발 시스템입니다.

FAQ-18

참고 기준과 한계
