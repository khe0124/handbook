모션·애니메이션 원리

easing과 duration이 왜 그렇게 느껴지는지, 여러 요소를 어떻게 안무(choreography)하고, transform·opacity로 어떻게 60fps를 지키며, prefers-reduced-motion으로 어떻게 안전을 기본값으로 두는지를 정리합니다. 이 문서는 모션의 조형·지각 원리와 구현을 다룹니다. AI 처리 상태 표현이나 제품 인터랙션 플로우는 01 디자인 실행, AX 인터랙션 문서에서 다루므로 여기서는 타이밍·느낌·성능에 집중합니다.

MAP

모션 지도

모션은 화면에 생기를 넣는 장식이 아니라, 시간을 쓰는 정보 채널입니다. 사용자의 주의(attention)를 잠깐 빌리는 대가로 무언가를 설명해야 합니다. 이 문서가 다루는 네 영역을 먼저 지도로 봅니다.

WHY

모션의 목적

모든 모션은 하나 이상의 목적을 가져야 합니다. 목적이 없는 모션은 그냥 소음입니다. 사용자의 시선을 붙잡고, 처리 속도를 늦추고, 배터리를 쓰는데 아무것도 설명하지 않는다면 빼는 게 낫습니다. 모션의 목적은 대략 다섯 가지로 나뉩니다.

목적무엇을 하는가실무 판단 · 실패 신호
Feedback내 조작이 접수됐다는 반응 확인탭·클릭 즉시 반응해야 한다. 100ms 넘게 무반응이면 눌렸는지 의심한다.
Continuity공간·객체의 연속성 유지같은 객체가 위치를 옮길 때 튀지 않고 이어져야 한다. 갑자기 사라졌다 나타나면 "다른 것"으로 읽힌다.
Orientation어디서 왔고 어디로 가는지 방향 감각서랍이 왼쪽에서 나오면 닫힐 때 왼쪽으로 간다. 나온 방향과 들어가는 방향이 다르면 공간 모델이 깨진다.
Hierarchy / Focus시선을 중요한 곳으로 유도새로 뜬 에러, 방금 추가된 행에 시선을 모은다. 동시에 여러 곳이 움직이면 초점이 사라진다.
Expression / Brand제품의 성격을 드러냄탄탄함·경쾌함 같은 태도를 타이밍으로 표현한다. 공공 B2B 대시보드는 과한 바운스보다 절제된 감속이 신뢰를 준다.
실무 기준

모션 리뷰의 첫 질문은 "예쁜가"가 아니라 "이걸 끄면 사용자가 무엇을 잃는가"입니다. 답이 "아무것도"라면 그 모션은 근거가 없습니다. Feedback·continuity·orientation은 없으면 혼란이 생기므로 정당하고, expression은 앞의 셋을 해치지 않는 선에서만 얹습니다.

보조 노트

운영 분석 대시보드에서 잔여 할당량 숫자가 바뀔 때 값이 카운트업되며 올라가면 그것은 feedback(집계가 갱신됐다) 겸 focus(여기 값이 변했다)입니다. 하지만 페이지의 모든 카드가 매 로드마다 순차로 날아 들어오면 그건 브랜드 표현을 빙자한 소음입니다. 자주 보는 화면일수록 모션은 짧고 조용해야 합니다.

CURVE

easing · duration

easing은 시간에 따라 속도가 어떻게 변하는지를 정하는 곡선입니다. 현실의 물체는 즉시 최고 속도에 도달하지 않고 가속하며, 멈출 때 감속합니다. 선형(linear) 모션이 기계적으로 느껴지는 이유입니다. 핵심 직관 하나: 화면에 들어오는 것은 감속(ease-out)하고, 나가는 것은 가속(ease-in)한다.

곡선의미실무 판단
ease-out빠르게 시작해 감속하며 멈춤등장·확장의 기본값. 사용자가 결과를 빨리 보고 부드럽게 안착하는 느낌. 진입 모션 대부분은 여기서 시작한다.
ease-in느리게 시작해 가속하며 사라짐퇴장 전용. 나가는 요소는 끝을 볼 필요가 없으므로 가속해 빠져나간다. 진입에 ease-in을 쓰면 답답하게 느껴진다.
ease-in-out감속·가속을 대칭으로한 자리에서 다른 자리로 이동하는 요소(위치 이동, 순서 재배치)에 자연스럽다.
linear속도 일정물리적 이동엔 부적합. 로딩 스피너의 회전, opacity crossfade 같은 "가속 개념이 없는" 변화에만 쓴다.
spring질량·강성·감쇠로 정의되는 물리 곡선duration이 아니라 물리 파라미터로 조절된다. 자연스럽고 인터럽트에 강하지만, 과한 bounce는 공공·데이터 UI에서 경박해 보인다.

duration은 짧을수록 반응이 빠르게 느껴지지만, 너무 짧으면 변화를 인지할 틈이 없어 "그냥 바뀌어 있다"가 됩니다. 대략의 원리는 작은 UI 변화는 100~300ms, 이동 거리가 크거나 화면 전체가 바뀌면 더 길게입니다. 절대 규칙이 아니라, 거리·크기·중요도에 따라 조정하는 감각입니다.

상황대략의 감각조정 원리
hover, 작은 상태 변화100~150ms즉각성이 우선. 길면 UI가 굼떠 보인다.
버튼·토글·드롭다운150~250ms변화가 보이되 기다림은 없는 구간.
모달·시트·큰 패널 등장250~400ms이동 거리·면적이 크면 같은 속도감을 위해 시간을 늘린다.
페이지·전면 전환더 길게, 그러나 절제맥락 전환을 알리되 대기처럼 느껴지면 실패.

거리와 시간의 관계가 핵심입니다. 같은 duration이라도 멀리 이동하는 요소는 빠르게, 가까이 움직이는 요소는 느리게 보입니다. 속도감을 일정하게 유지하려면 큰 이동엔 시간을 더 주고, 작은 이동엔 줄입니다.

/* 진입: 빠르게 시작해 감속하며 안착 */
.panel-enter {
  transition: transform 240ms cubic-bezier(0.16, 1, 0.3, 1),
              opacity 240ms cubic-bezier(0.16, 1, 0.3, 1);
}

/* 퇴장: 가속하며 빠져나감 (더 짧게) */
.panel-exit {
  transition: transform 160ms cubic-bezier(0.4, 0, 1, 1),
              opacity 160ms cubic-bezier(0.4, 0, 1, 1);
}

/* cubic-bezier(x1, y1, x2, y2):
   두 제어점으로 곡선을 정의. y가 x보다 앞서면 감속(out),
   뒤처지면 가속(in). 위 진입 곡선은 초반에 y가 빠르게 올라
   ease-out 성격이 강한 "부드러운 안착" 곡선. */
주의

퇴장(exit)을 진입(enter)과 같은 길이로 두는 실수가 흔합니다. 나가는 요소를 오래 보여줄 이유가 없으므로 exit는 enter보다 짧게(대략 60~70%) 두면 전체가 기민하게 느껴집니다. 또, transition을 all로 걸면 의도치 않은 속성까지 애니메이션되어 성능·지터 문제가 생깁니다. 바꿀 속성만 명시합니다.

CHOR

안무

여러 요소가 동시에 움직이면 눈은 어디를 봐야 할지 모릅니다. 안무(choreography)는 요소들의 등장·이동 순서와 타이밍을 조율해 하나의 이야기로 읽히게 하는 일입니다. 원칙은 하나입니다. 동시에 다 움직이지 않는다.

기법무엇을 하는가실무 판단 · 실패 신호
Stagger리스트·그리드 항목을 짧은 시차로 순차 등장시선이 위→아래로 흐르며 개수를 인지하게 한다. 항목이 많으면 총 시간이 길어지므로 시차를 20~40ms로 좁히거나 상한을 둔다.
Shared element transition화면 A의 요소가 화면 B의 같은 요소로 이어짐썸네일→상세의 이미지처럼 "같은 것"임을 보존. 두 화면의 요소가 실제로 대응되지 않으면 오히려 혼란을 준다.
Orchestration여러 애니메이션의 시작·겹침·지연을 설계배경 먼저, 카드 다음, 텍스트 마지막처럼 층을 나눈다. 전부 0ms에 시작하면 층이 사라진다.
인과 관계원인 요소가 먼저, 결과 요소가 뒤따름버튼을 눌러 → 시트가 올라온다. 결과가 원인보다 먼저 움직이면 논리가 뒤집힌다.
부모-자식 관계컨테이너가 먼저 자리잡고 내용이 채워짐카드가 열리며 내부 텍스트가 뒤이어 나타나면 위계가 드러난다. 동시에 나오면 평평해진다.
실무 기준

좋은 안무는 "무엇을 먼저 봐야 하는가"를 시간으로 답합니다. 사용자의 주의는 움직이는 것으로 끌립니다. 그래서 한 순간에 움직이는 것은 하나여야 하고, 중요한 것이 마지막에 혹은 가장 크게 움직이도록 설계합니다. 모든 것이 동시에 튀어나오면 초점이 없어지고, 그건 안무가 아니라 폭죽입니다.

/* stagger: 각 항목에 인덱스 기반 지연 */
.row {
  opacity: 0;
  transform: translateY(8px);
  animation: rise 220ms cubic-bezier(0.16, 1, 0.3, 1) forwards;
}
.row:nth-child(1) { animation-delay: 0ms; }
.row:nth-child(2) { animation-delay: 30ms; }
.row:nth-child(3) { animation-delay: 60ms; }
/* 실무에선 --i 커스텀 프로퍼티로:
   animation-delay: calc(var(--i) * 30ms);
   단, 항목 수 상한을 둬 총 시간이 길어지지 않게 한다. */

@keyframes rise {
  to { opacity: 1; transform: translateY(0); }
}
TRANS

의미 있는 전환

전환(transition)은 두 상태를 잇는 다리입니다. 상태가 갑자기 바뀌면 사용자는 "무엇이 어떻게 달라졌는지"를 스스로 재구성해야 합니다. 좋은 전환은 그 재구성을 대신 해줍니다. 방향성, 상태 변화, 레이아웃 이동 세 가지를 부드럽게 잇는 것이 핵심입니다.

전환 유형원리패턴 · 실패 신호
Enter / Exit 방향성나온 방향과 들어가는 방향이 대응오른쪽에서 들어온 시트는 오른쪽으로 나간다. 방향이 어긋나면 공간 감각이 깨진다.
State change같은 요소의 상태가 이어짐체크박스 on/off, 탭 전환의 밑줄 이동. 새로 그리지 말고 기존 것을 변형한다.
Layout shift요소 위치·크기 변화를 부드럽게항목 삭제로 아래 목록이 위로 당겨질 때 순간 점프하지 않게 한다. FLIP 기법으로 자연스럽게 처리.
Modal / Sheet배경을 눌러 계층을 표현모달 등장 시 backdrop을 어둡게, 콘텐츠는 살짝 위로 안착. backdrop 없이 뜨면 깊이 관계가 안 보인다.
Page 전환맥락이 바뀜을 알림같은 계층 이동은 crossfade, 상하 계층 이동은 방향을 준다(들어가면 앞으로, 뒤로가면 뒤로).
List 전환추가·삭제·재정렬을 추적 가능하게추가는 fade+rise, 삭제는 fade+collapse, 정렬은 위치 이동. 셋을 섞어 쓰면 무엇이 일어났는지 읽힌다.
실무 기준 · FLIP

FLIP은 layout 변화를 값싸게 애니메이션하는 개념입니다. First(변화 전 위치 측정) → Last(변화 후 위치 측정) → Invert(차이만큼 transform으로 되돌려 놓음) → Play(transform을 0으로 풀며 재생). 실제 layout은 이미 최종 상태이고, 눈에는 transform만 움직이므로 리플로우 없이 부드럽게 이동합니다. 목록 재정렬·항목 삭제 후 당겨짐 같은 layout shift에 이 개념을 씁니다.

보조 노트

브라우저의 View Transitions API는 이런 shared element·페이지 전환을 표준으로 다루려는 시도입니다. 다만 지원 범위와 폴백을 확인하고, 없을 때도 기능이 그대로 동작하도록(전환은 향상일 뿐) 설계해야 합니다. 라이브러리 API가 불확실하면 문서를 확인하고 쓰세요.

MICRO

마이크로인터랙션

마이크로인터랙션은 단일 목적의 작은 순간입니다. 토글 하나, 좋아요 하나, 입력 검증 하나. Dan Saffer의 구조를 빌리면 trigger → rules → feedback → loops/modes로 나뉩니다. 무엇이 시작시키고(trigger), 어떤 규칙으로 동작하며(rules), 무엇을 되돌려주고(feedback), 시간·반복에 따라 어떻게 달라지는가(loops/modes)입니다. 여기서는 feedback의 타이밍과 느낌을 원리 관점으로 봅니다.

Trigger→ Rules→ Feedback→ Loops / Modes
대상모션의 역할타이밍 · 느낌
버튼눌림을 즉시 확인press는 즉각(수십 ms), 살짝 눌리고 놓으면 복귀. 지연되면 "먹통" 인상.
토글 / 스위치on↔off 상태를 이어서 보여줌손잡이가 미끄러지듯 이동. 색·위치가 함께 바뀌어 상태를 이중으로 전달.
입력 검증성공·실패를 부드럽게 알림성공은 조용한 체크, 실패는 짧은 shake. shake는 과하면 짜증을 유발하니 1회·소폭.
당겨서 새로고침당김 정도와 임계점을 몸으로 느끼게당길수록 저항이 커지고, 임계 넘으면 놓아도 된다는 신호. 손가락 속도에 즉시 붙어야 자연스럽다.
로딩 / 진행기다림을 견딜 수 있게불확정은 회전, 확정은 진행바. 실제 진행과 어긋나 끝에서 멈춰버리면 신뢰를 잃는다.
실무 기준

마이크로인터랙션의 가치는 "조작이 접수됐다"를 즉시 알리는 데 있습니다. 그래서 feedback의 시작은 사용자 입력과 지각적으로 동시여야 합니다(대략 100ms 안). 애니메이션이 완료되기 전에 다음 조작이 들어올 수 있으므로, 인터럽트 가능하도록 설계합니다. 되돌리기 어려운 긴 애니메이션 중에 UI가 잠기면 그건 피드백이 아니라 방해입니다.

PERF

구현 · 성능

부드러워 보이려면 프레임을 놓치지 않아야 합니다. 60fps는 프레임당 약 16.7ms의 예산을 뜻합니다. 이 예산 안에서 브라우저가 스타일 계산·레이아웃·페인트·합성을 끝내야 합니다. 원리는 단순합니다. 애니메이션은 transform과 opacity만 건드린다. 이 둘은 layout과 paint를 건너뛰고 compositor에서 처리되기 때문입니다.

속성렌더 비용실무 판단
transform, opacitycompositing만 (저렴)이동·확대·회전·페이드는 전부 여기로 표현한다. 애니메이션의 기본 재료.
width, height, top, left, marginlayout(reflow) 유발 (비쌈)매 프레임 레이아웃 재계산. 크기·위치는 transform: scale/translate로 대체한다.
color, background, box-shadowpaint 유발 (중간~비쌈)정적 상태 변화엔 괜찮지만 매 프레임 애니메이션은 피한다. shadow는 특히 무겁다.
will-change레이어 승격 힌트미리 compositor 레이어로 올려 준비시키지만, 남용하면 메모리를 먹고 오히려 느려진다.
고위험

will-change를 모든 요소에 상시로 걸면 GPU 메모리가 폭증하고 전체가 느려집니다. 애니메이션 직전에만 붙이고 끝나면 제거하는 게 원칙입니다. "혹시 몰라서" 넣지 마세요. 측정 없이 최적화 힌트를 뿌리는 것은 최적화가 아니라 부채입니다.

/* 나쁜 예: layout을 매 프레임 재계산 → jank */
.bad {
  transition: left 300ms ease, width 300ms ease;
}
.bad.open { left: 0; width: 320px; }

/* 좋은 예: transform만 → compositor에서 처리, 60fps */
.good {
  transform: translateX(-320px);
  transition: transform 300ms cubic-bezier(0.16, 1, 0.3, 1);
}
.good.open { transform: translateX(0); }
주의

긴 리스트·스크롤 연동 애니메이션은 비용이 누적됩니다. 화면 밖 요소까지 매 프레임 계산하면 스크롤이 끊깁니다(jank). 가상화(화면에 보이는 것만 렌더), scroll 이벤트 대신 IntersectionObserver나 scroll-driven animation 사용, 그리고 "정말 스크롤마다 움직여야 하는가"를 먼저 물으세요. 성능은 측정으로 판단합니다. DevTools의 Performance 패널에서 프레임 드롭과 layout/paint 발생을 확인한 뒤 최적화하세요. 추측으로 고치지 않습니다.

A11Y

접근성 · 모션 안전

모션은 일부 사용자에게 불편이 아니라 신체 증상을 유발합니다. 전정기관(vestibular) 장애가 있는 사람은 큰 화면 이동·parallax·zoom·autoplay에서 어지럼증, 메스꺼움, 방향 상실을 겪을 수 있습니다. 그래서 원칙은 하나입니다. 안전이 기본값이다. 화려함은 옵트인, 절제는 디폴트입니다.

유발 요인왜 위험한가대체 처리
큰 화면 이동 · slide넓은 광학 흐름이 전정 감각과 충돌이동 대신 제자리 fade로 대체한다.
Parallax배경·전경의 속도 차가 깊이 착시·어지럼 유발reduced-motion에선 parallax를 끄고 정적 배경으로.
Zoom / scale 급변화면이 덮치는 느낌확대폭을 줄이거나 opacity로 대체.
Autoplay 영상 · 루프제어 못 하는 지속 모션은 피로·발작 위험정지 컨트롤 제공, 자동재생 지양, 3회 이상 번쩍임 금지.
고위험 · prefers-reduced-motion

OS의 "동작 줄이기" 설정은 사용자가 명시적으로 보낸 신호입니다. 반드시 존중해야 합니다. 이 설정이 켜지면 모션을 전부 없애기보다, 위치 이동·회전·zoom 같은 전정 유발성 모션을 제거하고 opacity crossfade 같은 안전한 전환으로 대체하는 것이 원칙입니다. feedback을 완전히 없애면 상태 변화를 놓치므로, "움직임은 줄이되 정보는 남긴다"가 올바른 방향입니다.

/* 기본: 위치 이동 포함 전환 */
.card {
  transition: transform 260ms cubic-bezier(0.16, 1, 0.3, 1),
              opacity 260ms ease;
}

/* 동작 줄이기: 이동 제거, 안전한 fade만 남김 */
@media (prefers-reduced-motion: reduce) {
  .card {
    transition: opacity 200ms ease;
    transform: none;
  }
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
  }
}
보조 노트

색만으로 상태를 전달하지 않는 원칙(색채 문서)과 같은 논리가 모션에도 적용됩니다. 모션만으로 정보를 전달하면 모션을 끈 사용자는 그 정보를 못 받습니다. 모션은 이미 존재하는 정보를 강화하는 층이어야지, 유일한 전달 수단이면 안 됩니다.

CRIT

리뷰 체크리스트

모션은 코드 리뷰에서 자주 빠집니다. 눈으로는 "괜찮아 보인다"로 넘어가기 쉽기 때문입니다. 아래 네 축으로 점검하면 감각이 아니라 원리로 판단할 수 있습니다.

MOTION REVIEW CHECKLIST — 목적 1. 이 모션을 끄면 사용자가 무엇을 잃는가. 답이 없으면 뺀다.
2. feedback · continuity · orientation · focus · brand 중 어떤 목적인가.
3. 같은 정보를 모션 없이도 전달하는가 (모션이 유일한 채널이 아닌가).
MOTION REVIEW CHECKLIST — 타이밍 1. 진입은 감속(ease-out), 퇴장은 가속(ease-in)인가.
2. 퇴장이 진입보다 짧은가.
3. 이동 거리·면적에 비해 duration이 적절한가 (작은 UI 100~300ms 원리).
4. 동시에 움직이는 것이 하나로 읽히는가, 아니면 초점이 흩어지는가.
MOTION REVIEW CHECKLIST — 성능 1. transform · opacity만 애니메이션하는가 (width/left/height 아님).
2. transition을 all이 아니라 필요한 속성만 지정했는가.
3. will-change를 상시로 남발하지 않았는가.
4. 긴 리스트·스크롤 애니메이션을 DevTools로 60fps 측정했는가.
MOTION REVIEW CHECKLIST — 접근성 1. prefers-reduced-motion을 존중하는가.
2. reduced 상태에서 이동·zoom·parallax를 제거하고 fade로 대체하는가.
3. autoplay·무한 루프·번쩍임을 피하고 정지 컨트롤을 주는가.
4. 모션을 줄여도 상태 정보가 그대로 전달되는가.
GLOSSARY

용어집

Easing이징
시간에 따른 속도 변화 곡선. ease-out은 감속, ease-in은 가속, linear는 일정.
cubic-bezier큐빅 베지어
두 제어점(x1,y1,x2,y2)으로 easing 곡선을 정의하는 CSS 함수. 커스텀 감속·가속을 만든다.
Duration지속 시간
애니메이션이 시작부터 끝까지 걸리는 시간. 거리·크기·중요도에 따라 조정한다.
Spring스프링
질량·강성·감쇠 같은 물리 파라미터로 정의되는 모션. duration 대신 물리로 조절.
Stagger스태거
여러 항목을 짧은 시차로 순차 등장시켜 시선의 흐름과 개수를 인지시키는 기법.
Choreography안무
여러 요소의 등장·이동 순서와 타이밍을 하나의 이야기로 조율하는 설계.
Shared element transition공유 요소 전환
한 화면의 요소가 다음 화면의 같은 요소로 이어지며 동일성을 보존하는 전환.
FLIP플립
First·Last·Invert·Play. layout 변화를 transform으로 되돌렸다 풀어 리플로우 없이 부드럽게 이동시키는 기법.
Compositing합성
layout·paint를 건너뛰고 GPU에서 레이어를 합치는 단계. transform·opacity가 여기서 처리된다.
Jank버벅임
프레임 예산(약 16.7ms)을 넘겨 프레임을 놓쳐 끊겨 보이는 현상.
will-change윌 체인지
요소를 미리 레이어로 승격시키는 힌트. 남용하면 메모리를 먹고 오히려 느려진다.
prefers-reduced-motion동작 줄이기
OS 설정에 연동되는 미디어 쿼리. 사용자가 모션 최소화를 요청한 신호로 반드시 존중한다.
Vestibular disorder전정기관 장애
균형 감각 기관의 이상. 큰 화면 이동·parallax·zoom에 어지럼·메스꺼움을 겪을 수 있다.
