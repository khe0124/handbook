백엔드 엔지니어링 핸드북

백엔드 역량의 핵심은 기능 구현을 운영 가능한 시스템으로 바꾸는 설계 판단입니다. 같은 CRUD라도 트랜잭션 경계·인덱스·실패 처리·관측가능성에서 품질이 갈립니다. 이 핸드북은 "되게 만드는 법"이 아니라 틀려도 안전하고, 바뀌어도 견디고, 터지면 추적 가능한 백엔드를 만드는 판단의 지도입니다.

AUDIENCE : 프론트엔드 기반 풀스택 확장 · JAVA/SPRING/JPA
STANCE : 개념 이해 중심 — 왜 존재하는가 · 무엇과 무엇의 트레이드오프인가 · COMPILED 2026-06
BE-01

무엇이 실무형 백엔드를 만드는가

실무형 백엔드 엔지니어는 더 많은 기능을 짜는 사람이 아니라, 모든 선택에 대해 "이 결정의 비용은 무엇인가"를 묻는 사람입니다. 코드가 도는 것은 시작점일 뿐입니다.

관점기능 구현운영 가능한 시스템지속 가능한 구조
성공 기준기능이 동작한다틀려도 안전하고, 부하·동시성·장애에서 견딘다시스템·팀·로드맵이 견딘다
DB쿼리가 결과를 준다트랜잭션 경계·인덱스·N+1·격리수준을 본다데이터 모델이 도메인 변화를 흡수하게 설계한다
실패해피패스를 만든다타임아웃·재시도·멱등·부분실패를 설계한다장애 모드와 폭발 반경을 사전 격리한다
운영로컬에서 동작을 확인한다로그·메트릭·트레이스로 원인을 짚는다SLO·용량·온콜 기준까지 연결한다

프론트엔드 경험의 자산은 그대로 옮겨옵니다 — 계약(인터페이스) 사고, 상태 관리, 사용자 관점은 백엔드의 API 설계·일관성·에러 응답으로 직결됩니다. 새로 장착할 근육은 지속성(persistence)과 동시성입니다: 프론트는 대개 한 사용자의 한 세션을 다루지만, 백엔드는 같은 데이터를 동시에 만지는 수천 개의 요청과 영구 저장된 상태를 다룹니다. 이 핸드북의 무게중심이 DB(BE-03·04)와 동시성(BE-05)에 쏠린 이유입니다.

CORE STANCE

숙련된 백엔드의 표식은 트레이드오프를 언어화하는 능력입니다. "JWT 쓸게요"가 아니라 "stateless라 확장은 쉽지만 즉시 무효화가 안 되니, 짧은 access + refresh로 가겠습니다"라고 말할 수 있는가. 정답을 외우는 게 아니라 비용과 대안을 같이 제시하는 것 — 면접도 실무도 여기서 갈립니다.

BE-02

HTTP·API 설계 — 계약으로서의 인터페이스

API는 코드가 아니라 계약입니다. 한 번 공개하면 수많은 클라이언트가 의존하고, 함부로 못 바꿉니다. 설계의 무게가 내부 함수와 다른 이유입니다.

2.1메서드 의미와 멱등성

HTTP 메서드는 단순 동사가 아니라 안전성(safe)과 멱등성(idempotent)의 약속입니다. GET은 안전(상태 변경 없음)하고 멱등하며, PUT·DELETE는 멱등(여러 번 호출해도 결과 동일), POST는 비멱등(호출마다 새 자원). 이 약속이 중요한 이유는 재시도 때문입니다 — 네트워크가 끊겨 응답을 못 받았을 때 멱등한 요청만 안전하게 다시 보낼 수 있습니다.

IDEMPOTENCY KEY

결제·주문 같은 비멱등 POST를 재시도 안전하게 만들려면 멱등성 키를 씁니다 — 클라이언트가 요청마다 고유 키를 헤더로 보내고, 서버는 같은 키의 두 번째 요청을 "이미 처리됨"으로 같은 응답을 돌려줍니다. "결제 버튼 두 번 눌러서 두 번 결제됨"을 막는 표준 장치입니다.

2.2상태코드 — 4xx와 5xx의 경계

상태코드는 의미군입니다: 2xx 성공, 3xx 리다이렉트, 4xx 클라이언트 잘못(요청이 틀림 — 재시도해도 같은 결과), 5xx 서버 잘못(서버가 못 함 — 재시도하면 될 수도). 이 경계가 운영을 가릅니다 — 입력 검증 실패를 500으로 던지면 알람이 울리고 온콜이 깨지지만, 실제로는 400이어야 합니다. "누구의 잘못인가"를 상태코드로 정직하게 말하는 것이 API의 기본 예의입니다.

2.3인증과 인가 — authN vs authZ

개념질문수단
인증 (authN)"너 누구냐"세션 쿠키, JWT, OAuth2/OIDC(소셜·SSO)
인가 (authZ)"이걸 할 권한 있냐"RBAC(역할), ABAC(속성), 자원 소유권 검사

둘은 다른 층입니다. 인증을 통과해도 이 자원에 접근할 권한은 따로 확인해야 합니다 — 이걸 빠뜨리면 BE-10의 IDOR(남의 id로 남의 데이터 조회)가 됩니다.

SESSION vs JWT

세션: 서버가 상태를 가짐 → 즉시 무효화 가능, 확장 시 세션 저장소(Redis) 공유 필요. JWT: stateless, 서명만 검증 → 수평 확장 쉽지만 만료 전 강제 로그아웃이 어렵다(토큰이 이미 발급됨). 실무 절충: access token은 짧게(분 단위) + refresh token으로 갱신, 블랙리스트는 refresh에만. "정답"이 아니라 무효화 요구 강도 vs 확장 편의의 트레이드오프입니다.

2.4진화 가능한 API

계약은 깨지지 않게 진화해야 합니다 — 버저닝(URL /v1 또는 헤더), 필드 추가는 OK·삭제/의미변경은 breaking, 페이지네이션은 offset(간단하지만 깊은 페이지·삽입에 취약)보다 대규모에선 cursor(키 기반)가 안정적입니다. 에러 응답도 계약입니다 — {code, message, details}처럼 구조를 표준화(RFC 9457 problem+json)해 클라이언트가 기계적으로 처리하게 하세요.

BE-03

관계형 DB — 트랜잭션·인덱스·격리

백엔드 실무의 진짜 시험장. 성능 문제의 대부분과 정합성 버그의 거의 전부가 여기서 납니다. ORM 뒤에 숨지 말고 직접 봐야 하는 층입니다.

3.1ACID — 트랜잭션의 4가지 약속

  • 원자성(Atomicity) — 전부 되거나 전부 안 된다. 중간 실패 시 롤백.
  • 일관성(Consistency) — 제약(외래키·유니크·체크)을 깨는 상태로 못 간다.
  • 격리성(Isolation) — 동시 트랜잭션이 서로의 중간 상태를 안 본다 (수준은 조절 가능 → 3.3).
  • 지속성(Durability) — 커밋되면 장애가 나도 남는다 (WAL·로그).

3.2인덱스 — B-tree가 빠른 이유

인덱스는 정렬된 사본입니다. B-tree는 균형 트리라 O(log n)에 값을 찾고, 정렬되어 있어 범위 검색(BETWEEN, ORDER BY)과 접두 매칭에도 강합니다. 실무에서 자주 밟는 함정:

규칙이유
복합 인덱스는 선두 컬럼부터(a,b) 인덱스는 a 또는 a,b 검색엔 쓰이지만 b 단독엔 안 쓰인다 (전화번호부 정렬과 같음)
컬럼에 함수·연산을 걸면 인덱스 무력화WHERE DATE(created_at)=...는 인덱스를 못 탄다 → 범위 조건으로 재작성
앞 와일드카드 LIKE '%foo'는 못 탄다정렬의 시작점을 모르므로. 'foo%'는 탄다
커버링 인덱스필요한 컬럼이 인덱스에 다 있으면 테이블을 안 읽는다 (가장 빠른 읽기)
인덱스는 공짜가 아니다쓰기마다 인덱스도 갱신 → 과다 인덱스는 쓰기·저장 비용. 필요한 것만

판단의 도구는 EXPLAIN(실행계획)입니다 — "이 쿼리가 인덱스를 타는가, 풀스캔하는가, 몇 행을 훑는가"를 추측하지 말고 보세요. 측정 없는 인덱스 추가는 BE-12의 "측정 없는 최적화"와 같은 실수입니다.

N+1 — ORM의 대표 함정

목록 100건을 가져온 뒤 각 건의 연관 데이터를 위해 쿼리를 100번 더 날리는 것 = 1+100 쿼리. 단위 테스트·로컬에선 안 보이다가 운영 부하에서 터집니다. 해법은 fetch join·배치 로딩·명시적 IN 조회 — 핵심은 "이 코드가 쿼리를 몇 번 날리는가"를 항상 의식하는 것입니다(→ BE-04).

BE-04

ORM·JPA — 추상화의 비용

ORM은 객체와 테이블 사이의 임피던스 불일치를 가립니다 — 가리지만 없애지 않습니다. 생성되는 SQL을 모르면, ORM은 편의가 아니라 지뢰밭이 됩니다.

4.1영속성 컨텍스트 — JPA의 심장

JPA의 핵심은 영속성 컨텍스트(persistence context)라는 1차 캐시입니다. 트랜잭션 동안 조회한 엔티티를 들고 있다가, 변경을 감지(dirty checking)해 커밋 시점에 자동으로 UPDATE를 날립니다 — 그래서 save()를 명시 안 해도 필드만 바꾸면 반영됩니다. 이게 강력하면서 동시에 혼란의 원천입니다: "내가 안 부른 쿼리가 왜 나가지?"의 답이 대개 dirty checking과 flush 타이밍입니다.

4.2지연 로딩과 트랜잭션 경계

@Transactional의 범위 = 영속성 컨텍스트의 생명주기입니다. 지연 로딩(lazy)된 연관을 트랜잭션 밖에서 접근하면 그 유명한 LazyInitializationException이 납니다 — 컨텍스트가 이미 닫혀서 추가 쿼리를 못 날리는 거죠. 실무 판단:

  • 기본은 lazy — 필요할 때만 로딩. @ManyToOne의 기본이 eager인 함정 주의.
  • 화면에서 함께 쓸 연관은 fetch join으로 한 번에 (N+1 회피).
  • 트랜잭션 경계를 서비스 계층에 명확히 — 컨트롤러·뷰에 영속성 컨텍스트를 끌고 가지 않기(OSIV 논쟁의 핵심).
RULE

ORM을 SQL 없이 쓰지 마라. JPA를 배우는 가장 빠른 길은 SQL 로그를 켜놓고(show_sql, p6spy) "내 코드 한 줄이 어떤 쿼리를 만드는가"를 눈으로 확인하는 것입니다. 복잡한 조회·통계는 ORM을 고집하지 말고 네이티브 쿼리·QueryDSL로 내려가는 게 실무 판단입니다 — 추상화는 도구지 신앙이 아닙니다.

BE-05

동시성·일관성

단일 요청에선 멀쩡하던 코드가 동시 요청에서 무너집니다. "재고 1개 남았는데 두 명이 동시에 주문"은 백엔드의 통과의례입니다.

5.1경쟁 상태와 갱신 손실

두 트랜잭션이 같은 값을 읽고(read), 각자 계산하고, 각자 쓰면(write) — 나중 쓰기가 먼저 쓰기를 덮습니다(lost update). 애플리케이션 코드의 if (재고 > 0) 재고--는 동시성 하에서 거의 항상 틀립니다. 해결은 두 갈래입니다.

전략방식언제
비관적 락
Pessimistic		읽을 때 행을 잠금(SELECT ... FOR UPDATE) — 남이 못 건드림충돌이 잦고, 충돌 비용이 큰 경우(재고·잔액). 락 대기·데드락 위험
낙관적 락
Optimistic		@Version 컬럼 — 쓸 때 버전이 그대로면 성공, 바뀌었으면 실패→재시도충돌이 드문 경우. 락이 없어 처리량 높지만 충돌 시 재시도 로직 필요

5.2멱등성 — 분산의 안전벨트

네트워크·메시지큐·재시도가 있는 한 "정확히 한 번"은 보장되지 않습니다(→ BE-07). 그래서 백엔드는 같은 작업이 두 번 실행돼도 결과가 같도록 설계합니다 — 멱등성 키(BE-02), 유니크 제약으로 중복 삽입 차단, "이미 처리된 주문인가" 상태 검사. 멱등성은 기능이 아니라 분산 환경의 기본 자세입니다.

CORE INSIGHT

동시성 버그의 공통 뿌리는 "읽고-판단하고-쓰는" 사이의 틈입니다. 그 틈에 남이 끼어듭니다. 해법은 항상 둘 중 하나 — 틈을 잠그거나(비관적), 틈에서 바뀌었는지 나중에 확인하거나(낙관적). DB의 원자적 연산(UPDATE ... SET n=n-1 WHERE n>0) 한 줄로 틈 자체를 없애는 것이 가장 단순한 정답일 때도 많습니다.

BE-06

캐싱 — 속도와 정합성의 거래

캐시는 가장 손쉬운 성능 개선이자 가장 흔한 정합성 버그의 원천입니다. "캐시 무효화는 컴퓨터 과학의 두 어려운 문제 중 하나"라는 농담은 진담입니다.

패턴동작특징
Cache-aside (가장 흔함)앱이 캐시 먼저 조회 → 없으면 DB 읽고 캐시에 채움구현 단순·범용. 첫 요청은 느림(cold), 무효화는 앱 책임
Read-through캐시 계층이 DB 읽기를 대행앱 코드 단순. 캐시 라이브러리 의존
Write-through쓸 때 캐시와 DB 동시 갱신일관성↑, 쓰기 지연↑
Write-back캐시에 먼저 쓰고 나중에 DB 반영쓰기 빠름, 장애 시 유실 위험

핵심 결정은 무효화(invalidation)와 TTL입니다. 데이터가 바뀌었는데 캐시가 옛 값을 들고 있으면(stale) 사용자는 틀린 걸 봅니다 — 짧은 TTL은 신선도를, 긴 TTL은 부하 절감을 삽니다. 정답은 데이터 특성에 따라 다릅니다(가격은 짧게, 카테고리 목록은 길게).

CACHE STAMPEDE

인기 키의 캐시가 동시에 만료되면, 수천 요청이 한꺼번에 DB로 몰립니다(thundering herd) — 캐시를 깔았는데 DB가 더 위험해집니다. 방어: 만료 직전 한 요청만 갱신하게 잠금(mutex/single-flight), TTL에 무작위 지터 추가, 백그라운드 사전 갱신. 캐시는 "있으면 빠른" 게 아니라 없을 때 무슨 일이 나는가까지 설계해야 합니다.

BE-07

메시징·비동기

동기 호출은 두 서비스를 시간으로 묶습니다 — 하나가 느리거나 죽으면 다른 하나도 같이 멈춥니다. 큐는 그 결합을 끊어, 시간과 장애를 분리합니다.

도구 유형모델대표강점
메시지 큐작업을 소비하면 사라짐 (work queue)RabbitMQ, AWS SQS작업 분산·부하 평탄화
이벤트 스트림로그에 쌓이고 여러 소비자가 각자 읽음Apache Kafka이벤트 소싱·재처리·다중 구독

전달 보장은 세 단계입니다: at-most-once(유실 가능), at-least-once(중복 가능), exactly-once(이상적). 실무의 현실은 거의 항상 at-least-once + 멱등 소비자입니다 — 메시지가 중복 도착할 수 있다고 가정하고, 소비 측에서 "이미 처리했나"를 검사해 중복을 무해하게 만듭니다(→ BE-05 멱등성).

OUTBOX PATTERN

"DB에 주문을 저장하고 + 메시지큐에 이벤트를 발행한다"를 원자적으로 하려는 순간 문제가 생깁니다 — DB 커밋은 됐는데 발행이 실패하면(또는 반대) 둘이 어긋납니다(이중 쓰기 문제). 해법인 outbox 패턴: 이벤트를 같은 DB 트랜잭션 안의 outbox 테이블에 함께 쓰고, 별도 프로세스가 그 테이블을 읽어 발행합니다. "DB 트랜잭션 하나"라는 원자성에 메시지 발행을 얹는 영리한 우회입니다.

DLQ · 재시도

소비가 계속 실패하는 메시지는 무한 재시도하면 큐를 막습니다 — 지수 백오프 + 지터로 재시도하되, N회 실패하면 DLQ(dead letter queue)로 보내 격리하고 사람이 보게 하세요. 실패를 삼키지도, 무한 반복하지도 않는 중간 길입니다.

BE-08

네트워킹 — 분산의 토대

백엔드는 네트워크 위에 삽니다. 그리고 네트워크의 8가지 오류(fallacy) 중 첫 번째가 "네트워크는 신뢰할 수 있다"입니다 — 네트워크는 느리고, 실패하고, 가끔 거짓말합니다.

8.1TCP·DNS·TLS — 한 번의 요청에 숨은 비용

  • TCP는 연결을 3-way handshake로 맺습니다 — 연결 수립 자체가 왕복 비용입니다. 그래서 매 요청마다 새 연결을 맺지 않고 커넥션 풀로 재사용합니다(DB·HTTP 클라이언트 모두).
  • DNS는 도메인 → IP 변환이며 TTL 동안 캐시됩니다. 배포·장애조치 시 DNS TTL이 길면 트래픽 전환이 늦어집니다.
  • TLS는 추가 handshake로 암호화 채널을 맺고 인증서로 신원을 증명합니다 — HTTPS의 토대. 인증서 만료는 흔하고 치명적인 운영 사고입니다.

8.2외부 호출의 3종 안전장치

타임아웃
Timeout→ 재시도
Retry+백오프→ 서킷브레이커
Circuit Breaker

다른 서비스를 호출할 때 반드시 갖춰야 할 셋입니다. 타임아웃 — 무한 대기는 스레드·커넥션을 고갈시켜 한 서비스의 느림이 전체 장애로 번지게 합니다(연결·읽기 타임아웃 둘 다). 재시도 — 일시적 실패엔 효과적이지만, 즉시·고정 간격 재시도는 죽어가는 서버를 더 때립니다 → 지수 백오프 + 지터, 그리고 멱등한 요청만. 서킷브레이커 — 연속 실패가 임계를 넘으면 회로를 열어 한동안 호출 자체를 차단하고 빠르게 실패시켜, 회복할 시간을 줍니다.

FAILURE MODE — 캐스케이딩 장애

타임아웃 없는 호출 하나가 전체를 무너뜨리는 전형: A가 느려진 B를 무한 대기 → A의 스레드풀 고갈 → A도 응답 불능 → A를 호출하던 C도... 한 군데의 느림이 도미노로 번집니다. 타임아웃과 격벽(bulkhead)은 선택이 아니라 분산 시스템의 기본 위생입니다.

BE-09

관측가능성 — 운영 책임

운영을 책임질 수 있느냐가 백엔드 품질의 분기점입니다. 관측가능성은 "안 돼요"를 "여기가 이래서 안 됩니다"로 바꾸는 능력입니다.

기둥답하는 질문형태
로그 Logs"이 요청에 정확히 무슨 일이구조화 로그(JSON) + 상관관계 ID
메트릭 Metrics"전체가 지금 건강한가"집계 수치(요청율·에러율·지연·자원)
트레이스 Traces"이 요청이 어느 서비스에서 느렸나"분산 트레이싱(OpenTelemetry)

실천의 핵심은 둘입니다. 상관관계 ID — 요청 진입점에서 ID를 발급해 모든 로그·하위 호출에 전파하면, 마이크로서비스 5개를 거친 한 요청의 궤적을 한 줄로 추적할 수 있습니다. RED 메트릭 — 모든 서비스에 대해 Rate(처리량)·Errors(에러율)·Duration(지연)을 보면 "느리다/터진다"를 숫자로 잡습니다. 그리고 p50이 아니라 p95·p99를 보세요 — 평균은 꼬리의 고통을 숨깁니다.

HEALTH CHECK

liveness(살아있나 — 죽었으면 재시작)와 readiness(요청 받을 준비 됐나 — DB 연결 전엔 트래픽 차단)를 구분하세요. 둘을 합치면 "DB가 잠깐 끊겼다고 컨테이너가 통째로 재시작되는" 사고가 납니다. SLI(측정 지표)로 SLO(목표, 예: 가용성 99.9%)를 정하는 것이 운영 성숙의 시작입니다.

보안은 나중에 붙이는 기능이 아니라 기본값입니다. 제1원칙: 모든 사용자 입력은 적대적이라고 가정한다.

MOST COMMON BREACH

화려한 공격보다 인가 누락이 가장 흔한 실제 사고입니다 — GET /orders/12345에서 "이 주문이 이 사용자 것인가"를 검사 안 해, 12346으로 남의 주문을 봅니다. 인증만 통과시키고 인가를 빠뜨리는 실수. 모든 자원 핸들러에서 "이 사람이 이걸 볼/바꿀 권한이 있는가"를 기계적으로 물으세요.

BE-11

테스트의 목적은 커버리지 숫자가 아니라 변경을 두렵지 않게 만드는 것입니다. 리팩토링·배포의 자신감이 테스트에서 나옵니다.

TDD as DESIGN

버그를 고칠 땐 먼저 그 버그를 재현하는 실패 테스트를 쓰세요 — 고친 뒤 그 테스트가 통과하면 "정말 고쳤다"의 증거이자, 재발을 막는 회귀 방어선이 됩니다. 테스트를 먼저 쓰면 자연스럽게 "이 코드를 어떻게 호출할까(인터페이스)"를 먼저 설계하게 됩니다 — TDD의 진짜 가치는 커버리지가 아니라 설계 압력입니다.

BE-12

시스템 설계 사고

백엔드 설계의 최종 시험은 "정답을 아는가"가 아니라 "트레이드오프를 설명하는가"입니다. 시스템 설계엔 정답이 없고, 제약에 맞는 선택과 그 비용의 명시가 있을 뿐입니다.

12.1먼저 묻는 질문 — 비기능 요구

기능("무엇을")보다 비기능 요구("얼마나")가 설계를 결정합니다: 예상 트래픽(읽기:쓰기 비율, peak), 지연 목표(p99), 가용성 목표, 데이터 규모와 증가율, 일관성 요구(강한가 최종이어도 되나). 이걸 안 묻고 그린 설계는 추측입니다 — 당신의 작업 원칙 "요구사항이 모호하면 채우지 말고 질문하라"가 그대로 적용됩니다.

12.2확장과 일관성의 거래

  • 수직 확장(서버 키우기)은 단순하지만 한계·단일 장애점. 수평 확장(서버 늘리기)은 무한하지만 무상태(stateless) 설계와 데이터 분산을 요구합니다.
  • CAP — 네트워크 분단(P)은 피할 수 없으니, 분단 시 일관성(C)과 가용성(A) 중 무엇을 포기할지의 선택입니다. 은행 잔액은 C, 좋아요 수는 A.
  • 일관성 모델 — 강한 일관성(항상 최신)은 비싸고, 최종 일관성(잠깐 어긋나도 결국 수렴)은 싸고 빠릅니다. 도메인이 결정합니다.

12.3장애를 전제로 설계

"무엇이 깨질 수 있는가"를 먼저 그리는 것이 실무 설계입니다 — 단일 장애점(SPOF) 제거, 우아한 성능저하(graceful degradation: 추천이 죽어도 상품은 보이게), 백프레셔(과부하 시 받지 않고 거절), 격벽(한 기능의 자원 고갈이 다른 기능을 안 죽이게).

측정 없는 최적화 금지

"이게 더 빠를 것 같아서"는 근거가 아닙니다 — 병목은 측정한 뒤 잡으세요(프로파일링·메트릭). 명확성과 성능이 충돌하면 명확성을 먼저 택하고, 측정이 정당화할 때 최적화하세요. 예외는 알고리즘 복잡도 — O(n²)가 될 게 명백하면 처음부터 피합니다. 용량 산정은 정밀할 필요 없이 back-of-envelope(자릿수 추정)면 충분합니다: "초당 만 건 × 1KB = 10MB/s, 하루 ~860GB" 수준의 감각이 설계 방향을 가릅니다.

연결 수립 비용을 피하려 미리 맺은 연결을 재사용하는 풀. DB·HTTP 클라이언트의 기본.
서킷브레이커Circuit Breaker
연속 실패 시 호출을 차단해 빠르게 실패시키고 회복 시간을 주는 패턴. 캐스케이딩 장애 방어.
지수 백오프Exponential Backoff + Jitter
재시도 간격을 점점 늘리고(2배씩) 무작위 지터를 더해, 죽어가는 서버에 몰리는 재시도 폭주를 방지.
상관관계 IDCorrelation / Trace ID
요청 진입점에서 발급해 모든 로그·하위 호출에 전파하는 식별자. 분산 추적의 실타래.
RED 메트릭Rate·Errors·Duration
서비스 건강의 3대 지표. 평균이 아니라 p95·p99로 꼬리 지연을 본다.
liveness/readinessHealth Probe
살아있나(죽으면 재시작) vs 요청 받을 준비됐나(아니면 트래픽 차단). 합치면 사고.
IDORInsecure Direct Object Reference
남의 식별자로 남의 자원에 접근하는 인가 누락. 가장 흔한 실제 침해. authN≠authZ.
Parameterized QueryPrepared Statement
값을 코드와 분리해 바인딩하는 쿼리. SQL injection의 근본 방어 — 문자열 결합 금지.
CAP 정리Consistency·Availability·Partition
네트워크 분단 시 일관성과 가용성 중 하나를 포기해야 한다는 분산 시스템의 제약.
최종 일관성Eventual Consistency
잠깐 어긋나도 결국 수렴하는 일관성 모델. 강한 일관성보다 싸고 빠름 — 도메인이 허용하면 선택.
TestcontainersIntegration Test
테스트 시 진짜 DB·브로커를 컨테이너로 띄워 통합 테스트. mock DB의 거짓 안심을 피함.
BE-14

실무 API 설계 심화

백엔드 API는 기능 호출점이 아니라 장기적으로 유지되는 제품 계약입니다. 클라이언트가 늘고 배포 주기가 갈라질수록 backward compatibility가 핵심 품질이 됩니다.

주제실무 기준실패 신호
idempotency key결제, 주문, 포인트 적립처럼 중복 실행이 위험한 POST에 적용네트워크 timeout 후 사용자가 재시도하면 중복 생성됨
paginationoffset은 단순 목록, cursor는 대량·실시간 목록에 사용페이지 이동 중 데이터 삽입으로 누락/중복 발생
error contractcode, message, field, retryable, traceId를 구조화프론트가 문자열 파싱으로 분기
API versioning필드 추가는 호환, 의미 변경과 삭제는 버전 또는 migration 필요모바일 앱 구버전이 갑자기 깨짐
Case study

주문 생성 API가 timeout을 냈을 때 클라이언트는 성공인지 실패인지 알 수 없습니다. 이때 같은 idempotency key로 재요청하면 서버는 이전 처리 결과를 반환해야 합니다. key 저장소에는 요청 fingerprint, 응답, 만료 시간을 함께 저장하고, 같은 key에 다른 payload가 오면 충돌로 거절해야 합니다.

BE-15

대규모 운영 판단

시스템이 커지면 “어떤 기술을 쓸까”보다 “어디까지 단순하게 버티고, 언제 복잡도를 받아들일까”가 중요해집니다. 고급 백엔드 판단은 규모, 위험, 팀 역량, 운영 비용을 함께 놓고 결정합니다.

상황먼저 할 일복잡도를 받아들일 신호
조회가 느림EXPLAIN, slow query, cardinality, index hit ratio 확인인덱스와 쿼리 재작성 후에도 p95 목표를 못 맞춤
쓰기 충돌 증가unique 제약, 낙관적 락, 원자적 UPDATE 검토충돌이 정상 트래픽에서 빈번하고 재시도 비용이 큼
DB 부하 집중캐시, read replica, pagination, batch size 조정읽기/쓰기 분리와 샤딩 없이는 성장률을 못 감당
외부 API 장애 전파timeout, retry budget, circuit breaker, fallback 적용하류 장애가 핵심 사용자 여정을 반복적으로 멈춤
큐 적체consumer lag, 처리 시간, poison message, DLQ 확인생산 속도와 소비 속도의 구조적 차이가 지속됨
릴리스 위험 증가feature flag, expand-contract migration, canary 적용한 번의 배포가 여러 서비스와 DB 계약을 동시에 바꿈
Capacity thinking

용량 산정은 완벽한 예측이 아니라 자릿수 감각입니다. 예를 들어 초당 2,000요청, 평균 응답 20KB면 egress만 약 40MB/s입니다. 요청당 DB 쿼리 5개면 DB는 초당 10,000쿼리를 받습니다. 이 숫자가 connection pool, CPU, index, cache, network 비용을 압박하는지 먼저 계산해야 합니다.

Case study

관리자 검색 화면이 느려졌다고 바로 Elasticsearch를 붙이면 운영 복잡도가 커집니다. 먼저 실제 검색 조건, 정렬 기준, total count 필요 여부, keyset pagination 가능 여부, 부분 인덱스와 covering index를 확인합니다. 검색어 형태가 전문 검색이고 필터 조합이 많으며 DB가 OLTP 부하를 침해할 때 별도 검색 엔진을 검토합니다.
