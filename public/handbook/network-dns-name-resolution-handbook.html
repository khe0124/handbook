DNS와 이름 해석

DNS는 사람이 읽는 이름을 IP 주소로 바꾸는 시스템입니다. 네트워크 경로와 방화벽이 맞아도 DNS가 다른 IP를 주면 전혀 다른 곳으로 접속합니다.

DNS-01

DNS의 역할

질문DNS가 답하는 것예시
이 도메인의 IP는?A/AAAA recordapi.example.com → 203.0.113.10
이 서비스의 별칭은?CNAME recordwww → lb.example.com
메일 서버는?MX recordexample.com → mail provider
검증/정책 값은?TXT recordSPF, DKIM, domain verification
DNS-02

레코드와 TTL

TTL은 DNS 응답을 캐시해도 되는 시간입니다. 장애 전환과 배포 전략에 직접 영향을 줍니다.

항목설명실무 기준
A이름을 IPv4로 매핑고정 IP, LB IP 연결
CNAME이름을 다른 이름으로 매핑서비스 별칭, CDN/LB 연결
TTLresolver가 응답을 캐시하는 시간전환 전 낮추고, 안정화 후 높임
Negative cache없는 이름도 캐시될 수 있음새 레코드 생성 직후 NXDOMAIN 캐시 주의
DNS-03

Public·Private DNS

구분용도예시
Public zone인터넷에서 조회 가능한 도메인api.company.com
Private zoneVPC 내부에서만 조회되는 도메인db.internal.company.local
split-horizon DNS같은 이름이 위치에 따라 다른 IP로 해석사내에서는 private IP, 외부에서는 public IP
주의

같은 도메인이 public과 private zone에 모두 있으면 조회 위치에 따라 IP가 달라집니다. “내 PC에서는 되는데 서버에서는 안 됨”의 흔한 원인입니다.

DNS-04

VPN과 DNS

VPN은 IP 경로를 연결하지만 이름 해석까지 자동으로 해결하지는 않습니다.

상황필요한 설정실패 증상
우리 서버가 고객사 내부 도메인 조회고객사 DNS resolver로 forwardingNXDOMAIN 또는 public IP로 해석
고객사가 우리 private 도메인 조회우리 private DNS 공유 또는 조건부 forwarding서비스 IP를 못 찾음
양쪽에 같은 도메인 사용split-horizon 정책 명확화서로 다른 IP로 접속
DNS-05

진단

DNS debug commands 
dig api.example.com
dig @8.8.8.8 api.example.com
dig @172.16.0.2 db.customer.internal
nslookup db.customer.internal 172.16.0.2
Case study

VPN 연결 후 고객사 DB 도메인으로 접속했는데 public IP로 해석되어 실패하는 경우가 있습니다. 네트워크 경로가 아니라 DNS resolver가 고객사 내부 DNS를 보지 않는 문제입니다. 이때는 조건부 forwarding 또는 고객사 DNS 서버 지정이 필요합니다.
