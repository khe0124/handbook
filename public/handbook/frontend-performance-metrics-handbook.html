성능과 지표
핸드북

성능은 개발자의 느낌이 아니라 사용자가 체감하는 시간입니다. Lighthouse 점수는 출발점이고, 실제 판단은 사용자 환경의 field data와 제품 목표를 함께 봐야 합니다.

TOOLS : Lighthouse · Chrome UX Report · Web Vitals · Performance API
PM-01

성능은 사용자 경험이다

빠른 앱은 단순히 로드가 빠른 앱이 아닙니다. 빨리 보이고, 빨리 반응하고, 화면이 흔들리지 않아야 합니다.

사용자 질문지표대표 원인
뭔가 보이나?FCP, LCPHTML/CSS/이미지/폰트/서버 응답
누르면 반응하나?INP, TBT긴 JS 작업, event handler, hydration, 무거운 render/paint
화면이 밀리지 않나?CLS이미지 크기 미예약, 광고, 폰트 교체
전체적으로 쾌적한가?RUM, error rate, route latency기기/네트워크 차이, API 품질
PM-02

Core Web Vitals

Core Web Vitals는 사용자 체감 성능을 대표하는 세 지표입니다. 제품 성능 대화의 공통 언어로 삼기 좋습니다.

지표의미개선 방향
LCP가장 큰 주요 콘텐츠가 보이는 시간hero 이미지 최적화, 서버 응답, CSS/폰트 blocking 제거
INPclick/tap/keyboard 상호작용부터 다음 paint까지의 지연input delay, event handler, render/commit, style/layout/paint를 나눠 개선
CLS예상치 못한 layout shift이미지/광고 크기 예약, font-display, skeleton 크기 고정

INP는 네트워크 요청 완료나 비동기 UI 변경 전체 시간을 직접 재는 지표가 아닙니다. 사용자가 입력한 시점부터 브라우저가 다음 frame을 paint할 수 있는 시점까지의 상호작용 지연을 봅니다.

PERCENTILE

Core Web Vitals의 공식 평가는 모바일과 데스크톱을 나눠 p75 기준으로 봅니다. p95는 내부 SLO와 꼬리 지연 감시에 별도로 사용하되, CWV 통과 기준과 섞지 않습니다.

PM-03

Lighthouse 읽는 법

Lighthouse는 자동 진단 도구입니다. 점수 자체보다 어떤 병목을 지목하는지 읽어야 합니다.

Lighthouse Performance 점수는 FCP, Speed Index, LCP, TBT, CLS 같은 lab metric의 가중 평균입니다. Opportunities와 Diagnostics는 점수 원인이 아니라 개선 힌트입니다. INP는 field metric이므로 Lighthouse 기본 lab 실행에서는 직접 측정하지 못하고 TBT를 대리 신호로 봅니다.

섹션해석주의
Performance로드와 반응성 진단lab 환경이라 실제 사용자와 다를 수 있음
Accessibility기본 규칙 검사키보드 흐름과 의미 품질은 수동 확인 필요
Best Practices보안/브라우저 기본 품질낮은 점수 원인을 항목별로 확인
SEO검색 엔진 기본 구조SPA는 route별 metadata 전략 필요
LIGHTHOUSE 해석 순서Opportunities를 비용순으로 정렬
Diagnostics에서 main thread와 JS 비용 확인
Treemap으로 큰 dependency 확인
Before/After를 같은 조건에서 비교
점수보다 사용자 여정 지표와 함께 판단
PM-04

Lab vs Field

Lab data는 재현 가능한 테스트이고, field data는 실제 사용자 경험입니다. 둘 중 하나만 보면 판단이 왜곡됩니다.

구분장점한계
Lab재현 가능, PR/CI 자동화 가능실제 기기/네트워크 다양성 반영 부족
Field실제 사용자 기준원인 분석은 별도 도구 필요, 데이터 누적 시간 필요
Synthetic정해진 경로를 주기적으로 측정예상 경로 밖 문제는 놓침
RUM실사용 이벤트와 성능 연결개인정보와 샘플링 설계 필요
PM-05

개선 우선순위

성능 개선은 가장 큰 사용자 영향과 가장 낮은 구현 위험을 함께 봅니다.

문제먼저 볼 것대표 조치
LCP 느림server response, hero resource이미지 압축, preload, critical CSS
INP 나쁨input delay, event handler, React render/commit, presentation delay작업 분할, handler 단순화, state 경계 조정, rendering 비용 감소
CLS 큼shift source크기 예약, dynamic banner 자리 확보
JS 큼bundle treemapdependency 대체, route split
SCORE CHASING

Lighthouse 100점을 목표로 제품 UX를 희생하면 안 됩니다. 성능 목표는 핵심 사용자 경로별로 세우고, 기능 가치와 함께 판단합니다.
