성능과 지표
핸드북

성능은 개발자의 느낌이 아니라 사용자가 체감하는 시간입니다. Lighthouse 점수는 출발점이고, 실제 판단은 사용자 환경의 field data와 제품 목표를 함께 봐야 합니다.

TOOLS : Lighthouse · Chrome UX Report · Web Vitals · Performance API
PM-01

성능은 사용자 경험이다

빠른 앱은 단순히 로드가 빠른 앱이 아닙니다. 빨리 보이고, 빨리 반응하고, 화면이 흔들리지 않아야 합니다. 성능 대화는 평균 점수보다 핵심 사용자 경로에서 몇 퍼센트의 사용자가 어떤 지연을 겪는지로 시작해야 합니다.

사용자 질문대표 지표원인 계층판정 기준
뭔가 보이나?FCP, LCPHTML, CSS, font, hero resource, server response주요 route의 LCP 후보가 의도한 콘텐츠인지 먼저 확인
누르면 반응하나?INP, TBT, long taskinput delay, event handler, React render/commit, style/layout/paint느린 interaction 이름과 target을 알아야 개선 가능
화면이 밀리지 않나?CLS, shift sourcemedia 크기 미예약, dynamic banner, skeleton, font swap사용자 입력으로 발생한 shift와 예상치 못한 shift를 구분
전환이 쾌적한가?route transition p95, cache hit, API latencychunk waterfall, data loading, client render, APISPA 전환은 Navigation Timing만으로 충분하지 않음
전체적으로 안정적인가?RUM, error rate, retry, abandonment기기·네트워크 차이, 배포 회귀, API 품질release, route, device class별 분포로 본다
PERFORMANCE QUESTION이 route의 핵심 사용자는 누구인가
사용자가 기다리는 대상은 콘텐츠, 입력 반응, 저장 완료 중 무엇인가
field p75와 p95 중 어떤 의사결정에 쓸 지표인가
프론트엔드가 소유한 시간과 API·네트워크 시간이 분리되는가
개선 후 제품 행동이나 접근성이 훼손되지 않는가
PM-02

Core Web Vitals

Core Web Vitals는 사용자 체감 성능을 대표하는 세 지표입니다. 제품 성능 대화의 공통 언어로 삼기 좋지만, 지표 이름만 외우면 개선이 불가능합니다. 각 지표의 후보 요소, attribution, percentile, route segment를 같이 봐야 합니다.

지표좋음개선 필요나쁨개선 방향
LCP2.5s 이하2.5s 초과 4.0s 이하4.0s 초과TTFB, render-blocking CSS, hero image priority, font blocking을 분리
INP200ms 이하200ms 초과 500ms 이하500ms 초과input delay, processing duration, presentation delay를 나눠 개선
CLS0.1 이하0.1 초과 0.25 이하0.25 초과이미지/광고 크기 예약, font-display, skeleton 크기 고정, late banner 정책

INP는 네트워크 요청 완료나 비동기 UI 변경 전체 시간을 직접 재는 지표가 아닙니다. 사용자가 입력한 시점부터 브라우저가 다음 frame을 paint할 수 있는 시점까지의 상호작용 지연을 봅니다.

FIELD 판정 예시모바일 route_view 10,000건 중 p75 LCP가 2.8s이면 Core Web Vitals 기준으로는 개선 필요입니다.
p50 LCP가 1.6s여도 p75가 2.8s라면 “대부분 빠르다”가 아니라 “상위 25% 사용자는 느리다”로 해석합니다.
p95 INP가 780ms이면 공식 통과 여부와 별개로 반복 업무 사용자에게 체감 문제가 있을 가능성이 큽니다.
지표attribution에서 볼 것흔한 오판
LCPelement, resource URL, TTFB, load delay, load duration, render delay이미지만 압축하고 실제 병목인 서버 응답이나 CSS blocking을 놓침
INPinteraction target, event type, input delay, processing, presentation delayAPI 응답 시간을 INP로 착각하거나 React render만 보고 paint 지연을 놓침
CLSshift source, impacted nodes, session window, user input 여부사용자 클릭 직후 의도된 확장을 회귀로 잘못 판정
PERCENTILE

Core Web Vitals의 공식 평가는 모바일과 데스크톱을 나눠 p75 기준으로 봅니다. p95는 내부 SLO와 꼬리 지연 감시에 별도로 사용하되, CWV 통과 기준과 섞지 않습니다.

PM-03

Lighthouse 읽는 법

Lighthouse는 자동 진단 도구입니다. 점수 자체보다 어떤 병목을 지목하는지 읽어야 합니다. Lighthouse 결과는 재현 가능한 lab 신호이지 실제 사용자 전체의 판정이 아니므로, PR에서는 같은 조건의 delta를 보고 운영 판단은 field data와 합쳐야 합니다.

Lighthouse Performance 점수는 FCP, Speed Index, LCP, TBT, CLS 같은 lab metric의 가중 평균입니다. Opportunities와 Diagnostics는 점수 원인이 아니라 개선 힌트입니다. INP는 field metric이므로 Lighthouse 기본 lab 실행에서는 직접 측정하지 못하고 TBT를 대리 신호로 봅니다.

Lighthouse 점수색상/판정실무 해석
90-100Goodlab 기준은 양호하지만 field p75/p95와 핵심 interaction은 별도 확인
50-89Needs Improvement병목 후보가 명확히 있을 가능성이 높아 trace와 treemap으로 원인 분해
0-49Poor초기 로드 또는 main thread 비용이 제품 사용을 막는 수준일 수 있어 우선 대응
섹션해석주의
Performance로드, main thread, render-blocking resource의 lab 진단한 번의 점수보다 같은 throttling 조건의 반복 측정과 delta가 중요
Opportunities예상 절감치가 큰 개선 힌트사용자 경로에 영향 없는 항목은 우선순위가 낮을 수 있음
Diagnosticsmain thread, unused JS, DOM size, long task 같은 원인 후보원인을 확정하려면 Performance trace와 bundle analyzer가 필요
Treemap큰 dependency와 unused byte 확인큰 라이브러리가 항상 제거 대상은 아니며 대체 비용도 봐야 함
Accessibility/SEO기본 규칙 검사와 route metadata 점검키보드 흐름, 의미 품질, SPA route별 메타는 수동 확인 필요
LIGHTHOUSE REVIEW ORDER실행 환경, throttling, viewport, 로그인 상태를 기록한다
점수보다 LCP/TBT/CLS 각각의 변화량을 본다
Opportunities를 예상 절감치가 아니라 사용자 경로 영향으로 재정렬한다
Diagnostics에서 main thread와 unused JS 후보를 찾는다
Treemap으로 새 dependency와 중복 vendor를 확인한다
Before/After를 같은 조건에서 3회 이상 비교한다
LAB 비교 숫자 예시PR 전: Performance 82, LCP 3.1s, TBT 420ms, CLS 0.04
PR 후: Performance 76, LCP 3.0s, TBT 680ms, CLS 0.04
해석: 점수 하락의 핵심은 LCP가 아니라 TBT 증가입니다. 새 dependency, hydration, route 초기 실행 비용을 먼저 봅니다.
SCORE TRAP

Lighthouse 100점은 제품 목표가 아닙니다. 점수를 올리려고 skeleton을 없애거나 중요한 기능을 늦게 로드하면 실제 사용성은 나빠질 수 있습니다. 점수 회귀는 조기 경보이고, 최종 판정은 핵심 route의 field 지표와 제품 영향으로 합니다.

PM-04

Lab vs Field

Lab data는 재현 가능한 테스트이고, field data는 실제 사용자 경험입니다. 둘 중 하나만 보면 판단이 왜곡됩니다.

구분장점한계사용 위치
Lab재현 가능, PR/CI 자동화 가능실제 기기/네트워크 다양성 반영 부족회귀 후보를 빠르게 찾고 수정 전후를 비교
Field실제 사용자 기준원인 분석은 별도 도구 필요, 데이터 누적 시간 필요제품 SLO와 release health 판정
Synthetic정해진 경로를 주기적으로 측정예상 경로 밖 문제는 놓침로그인, 검색, 저장 같은 핵심 흐름의 감시
RUM실사용 이벤트와 성능 연결개인정보와 샘플링 설계 필요route, device, release별 분포와 회귀 감지
DATA DECISION RULELab에서 나쁘고 field에서도 나쁘면 즉시 개선 대상
Lab에서 나쁘지만 field 영향이 작으면 예산과 우선순위를 조정
Lab은 좋지만 field가 나쁘면 저사양 기기, 특정 route, 특정 release segment를 의심
Field가 나쁘지만 원인이 불명확하면 synthetic scenario와 trace 재현으로 좁힌다
PM-05

개선 우선순위

성능 개선은 가장 큰 사용자 영향과 가장 낮은 구현 위험을 함께 봅니다. 우선순위는 “점수에 많이 기여하는 항목”이 아니라 “핵심 사용자의 실패율·이탈·반복 작업 지연을 줄이는 항목”입니다.

문제먼저 볼 것대표 조치검증 기준
LCP 느림TTFB, LCP resource load delay, render delay이미지 압축·크기 지정, priority/preload, critical CSS, 서버 캐시LCP p75 개선과 LCP element 안정성
INP 나쁨input delay, processing duration, React commit, presentation delayhandler 단순화, 작업 분할, state 경계 조정, virtualization, worker느린 interaction p75/p95와 long task 수 감소
CLS 큼shift source, session window, user input 여부크기 예약, skeleton 고정, dynamic banner 자리 확보, font 전략unexpected shift source 제거
JS 큼initial JS, unused JS, parse/compile, dependency treemapdependency 대체, route split, lazy boundary 재배치, import 축소initial byte와 main thread blocking time 감소
route 전환 느림chunk/data waterfall, cache hit, render durationroute prefetch, data loader 병렬화, optimistic shell, cache policyroute transition p95와 fallback 노출 시간 감소
SCORE CHASING

Lighthouse 100점을 목표로 제품 UX를 희생하면 안 됩니다. 성능 목표는 핵심 사용자 경로별로 세우고, 기능 가치와 함께 판단합니다.

PM-06

운영 지표

운영 지표는 “우리 서비스가 빠른가”를 배포, route, 사용자 환경별로 답하기 위한 데이터입니다. 평균 하나로는 회귀를 찾을 수 없으므로 분포와 segment를 기본값으로 둡니다.

지표의미필수 차원알림 기준
Web VitalsLCP, INP, CLS의 사용자 체감 성능route, device class, connection, releasep75 목표 초과 또는 release 대비 악화율
Route latencySPA route 전환부터 안정 화면까지 걸린 시간from/to route, cache hit, auth state핵심 route p95 악화
Interaction latency검색, 필터, 저장, 행 편집 같은 제품 동작 지연interaction name, item count, result state업무 반복 동작의 p95 악화
Resource budgetinitial JS/CSS/image/font byte와 chunk 수route, chunk name, dependency ownerPR 예산 초과 또는 특정 dependency 급증
Error correlation성능 저하와 JS/API 오류의 동시 발생release, route, request id, error class성능 악화와 오류율 상승이 같은 release에 묶임
DASHBOARD MINIMUM평균 대신 p50/p75/p95를 기본으로 표시
route와 release를 필수 필터로 둔다
모바일 저사양 또는 low memory segment를 분리한다
Web Vitals와 JS/API error를 같은 화면에서 본다
회귀 PR 또는 배포 버전으로 역추적 가능해야 한다
운영 대시보드 숫자 예시/dashboard route: LCP p50 1.4s · p75 2.3s · p95 5.8s
/grid route: INP p50 120ms · p75 260ms · p95 910ms
/checkout route: CLS p75 0.03 · p95 0.18
해석: dashboard는 공식 p75는 통과하지만 꼬리 로드가 크고, grid는 INP p75가 200ms를 넘어 상호작용 개선 대상입니다. checkout은 p75는 안정적이지만 특정 segment에서 layout shift source를 찾아야 합니다.
PM-07

성능 예산과 SLO

성능 예산은 “빠르게 만들자”를 PR에서 판단 가능한 기준으로 바꾸는 장치입니다. 기능 추가가 사용자 경험 비용을 얼마나 늘리는지 수치로 드러나야 합니다.

PERFORMANCE BUDGET POLICY핵심 route별 예산을 별도로 둔다

예시 숫자

위 예산은 공식 표준이 아니라 팀이 출발점으로 둘 수 있는 실무 예시입니다. 제품의 사용자 기기, 지역, 기능 가치에 맞게 조정하되, 한 번 정한 예산은 PR과 배포 대시보드에서 같은 기준으로 써야 합니다.

SLO

프론트엔드 SLO는 백엔드 API latency와 분리할 수 없습니다. route latency를 화면 단계별로 쪼개고, server time, network time, JS execution, render/paint 중 어디가 owner인지 나눠야 팀 간 책임 공방을 줄일 수 있습니다.

PM-08

RUM 계측 설계

RUM은 실제 사용자의 성능을 수집하지만, 아무 이벤트나 많이 보내면 원인 분석도 어렵고 개인정보 위험도 커집니다. 계측은 질문에서 시작해야 합니다.

계측 항목목적권장 차원주의
Web Vitals attributionLCP element, CLS source, INP target 확인route, device class, connection, releaseDOM text, 입력값, 개인 식별자가 섞이지 않게 정규화
Route timingSPA 전환 성능 파악from route, to route, cache hit, auth state, data size브라우저 navigation timing만으로 SPA 전환은 부족
Interaction timing검색, 필터, 저장 같은 핵심 동작 추적interaction name, result state, item count, validation count클릭 좌표나 입력값 원문 전송 금지
Resource timing느린 asset/API와 route 성능 연결resource type, normalized host, cache state, status classURL query와 header에 민감 정보가 없는지 마스킹
Error correlation성능 저하와 JS/API 오류 연결release, route, error type, request idstack trace와 사용자 데이터 마스킹
RUM EVENT CONTRACTevent_name: route_view | web_vital | interaction_latency | resource_timing
route_id: 실제 URL 대신 정규화된 route 이름
release_id: 배포 버전
device_class: low | mid | high
metric_value: ms 또는 score
metric_phase: load_delay | processing | presentation | render
attribution: 안전하게 정제된 원인 식별자
sample_rate: 이벤트 유형별 샘플링 비율

좋은 RUM 대시보드는 평균이 아니라 분포를 보여줍니다. p50, p75, p95를 route와 release별로 보고, 특정 배포 이후 악화된 지표를 자동으로 표시해야 회귀를 빠르게 찾을 수 있습니다.

PRIVACY

RUM은 디버깅 욕심 때문에 쉽게 과수집됩니다. route는 정규화하고, DOM text와 입력값은 버리며, user id는 직접 보내지 않습니다. 필요한 경우 익명 segment와 request id만 남겨 서버 로그와 제한적으로 연결합니다.

PM-09

자주 봉착하는 성능 문제

프론트엔드 성능 문제는 매번 새롭지 않습니다. 대부분은 비슷한 증상으로 나타나고, 잘못된 첫 가설 때문에 시간이 낭비됩니다. 아래 표는 현장에서 자주 만나는 문제를 증상, 흔한 오진, 확인 증거, 해결 방향으로 정리한 것입니다.

자주 만나는 문제흔한 오진확인 증거해결 방향
첫 화면 LCP가 늦다이미지만 압축하면 해결된다고 봄LCP element, TTFB, resource load delay, render delay서버 응답, CSS/font blocking, hero image priority를 분리해서 개선
검색 input이 버벅인다debounce만 넣으면 된다고 봄INP attribution, handler 시간, React commit, row render 수입력 echo는 즉시 유지하고 결과 목록은 deferred/transition/virtualization으로 분리
대형 table 편집이 느리다React.memo를 전체에 감싸면 된다고 봄Profiler commit, render된 row/cell 수, draft state 범위server data와 draft state를 분리하고 visible row만 렌더링
route 전환이 느리다code splitting을 더 잘게 쪼갬chunk, CSS, data request waterfallprefetch, data loader 병렬화, shared chunk, fallback 크기를 함께 조정
스크롤이 끊긴다JS가 많아서만 그렇다고 봄Frames lane, Layout/Paint 비용, DOM node 수, layer 수DOM 수 축소, read/write batching, paint 영역 축소, passive listener 적용
배포 후 특정 사용자만 느리다개발 환경에서 빠르니 문제없다고 봄RUM segment: device class, connection, region, release저사양 기기와 느린 네트워크 기준의 p75/p95를 별도 SLO로 둔다
