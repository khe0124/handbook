CS 기본기 면접

CS 기본기 질문은 이론 암기가 아니라 코드 선택, 성능 병목, 동시성 버그, 운영 현상을 설명하는 기반입니다.

CS-01

Q. Big-O 시간/공간 복잡도는 어떻게 설명하나요?

핵심 개념

Big-O는 입력 크기가 커질 때 실행 시간이나 메모리 사용량이 어떻게 증가하는지 상한 관점으로 표현합니다.

개념 해부

Big-O는 특정 컴퓨터에서 몇 ms 걸리는지를 맞히는 도구가 아니라 입력이 커질 때 비용이 어떤 모양으로 증가하는지 보는 언어입니다. 낮은 차수 항과 상수를 버리는 이유는 큰 입력에서 지배적인 증가율을 보기 위해서입니다. 다만 실무에서는 입력 크기, 호출 빈도, I/O 비용, 메모리 locality 때문에 이론 복잡도만으로 판단하면 안 됩니다.

30초 답변

Big-O는 입력 n이 커질 때 비용 증가 추세를 보는 표기입니다. O(1), O(log n), O(n), O(n log n), O(n²)처럼 표현하며, 실무에서는 데이터가 커졌을 때 현재 알고리즘이 버틸 수 있는지 판단하는 데 씁니다.

90초 답변

상수 시간도 실제로는 상수가 크면 느릴 수 있고, O(n)도 n이 작으면 충분할 수 있습니다. 면접에서는 “이중 루프라 O(n²)입니다”보다 입력 크기, 호출 빈도, 메모리 trade-off, 캐시/DB 같은 외부 비용을 함께 말하면 좋습니다.

꼬리질문

질문답변 방향
공간 복잡도?추가 자료구조나 recursion stack이 쓰는 메모리 증가량입니다.
O(log n) 예?정렬된 배열 binary search, balanced tree 탐색입니다.
실무 연결?대용량 grid filtering, API pagination, DB scan 비용으로 연결합니다.
CS-02

Q. Array/List/Map/Set/Heap/Queue/Stack/Tree/Graph는 언제 쓰나요?

핵심 개념

자료구조 선택은 접근 패턴, 삽입/삭제 비용, 정렬 필요성, 중복 허용, 우선순위, 관계 표현에 따라 달라집니다.

개념 해부

자료구조는 데이터를 담는 통이 아니라 필요한 연산의 비용표입니다. Array는 index 접근이 빠르지만 중간 삽입/삭제가 비싸고, HashMap은 평균 조회가 빠르지만 hash 품질과 resize 비용이 있습니다. Tree와 Heap처럼 구조적 제약을 둔 자료구조는 정렬, 우선순위, 범위 탐색 같은 특정 요구를 위해 일반성을 포기합니다.

30초 답변

순서 기반 index 접근은 Array/List, key 기반 조회는 Map, 중복 제거와 membership check는 Set, 우선순위 처리는 Heap, FIFO는 Queue, LIFO는 Stack, 계층은 Tree, 복잡한 관계는 Graph를 씁니다.

90초 답변

면접에서는 자료구조 이름보다 비용을 말해야 합니다. HashMap 조회는 평균 O(1)이지만 hash collision과 resize 비용이 있고, TreeMap은 O(log n)이지만 정렬 순서를 유지합니다. Queue는 작업 처리, Stack은 call stack/undo, Heap은 top-k나 scheduler에 연결할 수 있습니다.

꼬리질문

질문답변 방향
Set 중복 기준?equals/hashCode 또는 비교 기준에 따라 달라집니다.
Heap 정렬?최솟값/최댓값을 빠르게 꺼내는 데 유리합니다.
Graph 저장?인접 리스트와 인접 행렬을 규모와 밀도에 따라 선택합니다.
CS-03

Q. Hash collision은 무엇이고 왜 중요한가요?

핵심 개념

Hash collision은 서로 다른 key가 같은 hash bucket에 배치되는 현상입니다. 충돌이 많으면 HashMap의 평균 O(1) 성능이 나빠집니다.

개념 해부

hash collision은 예외 상황이 아니라 hash 공간이 유한하기 때문에 언제든 생길 수 있는 정상 현상입니다. 좋은 hash 함수는 key를 고르게 분산시켜 bucket 길이를 짧게 유지하고, 최종 동등성은 equals가 판단합니다. collision이 많아지면 평균 O(1)이라는 기대가 깨지고, 공격자가 의도적으로 몰리게 만들면 보안 이슈가 될 수 있습니다.

30초 답변

HashMap은 hashCode로 bucket을 찾고 equals로 실제 key를 비교합니다. 서로 다른 key가 같은 bucket에 몰리는 Hash collision이 많으면 조회 성능이 떨어지고, equals/hashCode 계약이 깨지면 조회 자체가 실패할 수 있습니다.

90초 답변

좋은 hash 함수는 key를 bucket에 고르게 분산시켜야 합니다. Java HashMap은 충돌이 심하면 bucket 구조를 tree로 바꾸는 최적화가 있지만, 기본적으로 equals/hashCode를 올바르게 구현해야 합니다. 보안 관점에서는 의도적인 hash flooding 공격도 고려할 수 있습니다.

꼬리질문

질문답변 방향
equals가 true면?hashCode도 반드시 같아야 합니다.
hashCode 같으면 equals true?아닙니다. 충돌 가능성이 있습니다.
mutable key?Map에 넣은 뒤 hash 기준 값이 바뀌면 찾지 못할 수 있습니다.
CS-04

Q. BFS/DFS는 언제 쓰나요?

핵심 개념

BFS는 가까운 노드부터 넓게 탐색하고, DFS는 한 경로를 깊게 탐색합니다. 그래프/트리 탐색의 기본입니다.

개념 해부

BFS와 DFS의 차이는 방문 순서가 아니라 문제의 증거를 어떤 순서로 찾는가입니다. BFS는 같은 거리의 노드를 먼저 모두 확인하므로 무가중치 최단 거리와 단계별 확산에 적합합니다. DFS는 한 경로를 끝까지 파고들기 때문에 조합 탐색, cycle detection, dependency 탐색에서 자연스럽지만 깊이가 크면 stack 위험이 있습니다.

30초 답변

BFS는 queue를 사용해 최단 거리나 단계별 탐색에 유리하고, DFS는 stack 또는 recursion으로 깊은 경로 탐색, cycle detection, backtracking에 자주 씁니다.

90초 답변

무가중치 그래프의 최단 경로는 BFS가 적합합니다. DFS는 모든 조합 탐색이나 dependency graph에서 cycle을 찾는 데 유용합니다. 실무에서는 메뉴 트리, 권한 그래프, 빌드 dependency, 조직도, workflow graph 같은 문제로 연결해 설명할 수 있습니다.

꼬리질문

질문답변 방향
visited가 왜 필요?cycle이 있는 그래프에서 무한 반복을 막기 위해 필요합니다.
DFS recursion 위험?깊이가 크면 stack overflow가 날 수 있어 iterative 방식이 필요할 수 있습니다.
가중치 최단 경로?Dijkstra 같은 알고리즘을 검토합니다.
CS-05

Q. race condition, deadlock, mutex/semaphore를 설명해보세요.

핵심 개념

race condition은 실행 순서에 따라 결과가 달라지는 문제입니다. deadlock은 둘 이상의 작업이 서로가 가진 자원을 기다리며 멈추는 상태입니다. mutex는 상호 배제, semaphore는 제한된 개수의 permit으로 동시 접근을 제어합니다.

개념 해부

동시성 문제는 코드가 틀렸다기보다 가능한 실행 순서가 너무 많아 일부 순서에서 불변식이 깨지는 문제입니다. mutex는 공유 자원에 한 번에 하나만 들어가게 하고, semaphore는 제한된 수만 들어가게 합니다. deadlock은 네 가지 조건이 동시에 맞을 때 생기므로 lock 순서, timeout, 범위 축소로 조건을 깨는 식으로 접근합니다.

30초 답변

공유 상태를 여러 thread가 동시에 수정하면 race condition이 생길 수 있어 lock, atomic operation, transaction으로 보호합니다. deadlock은 lock 순서 불일치가 대표 원인이고, mutex는 하나만 들어가게, semaphore는 정해진 수만 들어가게 제어합니다.

90초 답변

실무에서는 중복 클릭, 중복 결제, 재고 차감, DB row lock, thread pool 고갈 같은 문제로 나타납니다. deadlock 예방은 lock 순서 통일, lock 범위 축소, timeout, retry가 기본입니다. 동시성 문제는 재현이 어렵기 때문에 테스트와 로그, transaction isolation 이해가 필요합니다.

꼬리질문

질문답변 방향
atomic?중간 상태를 다른 작업이 볼 수 없는 단위 연산입니다.
semaphore 예?DB connection pool처럼 동시에 사용할 수 있는 자원 수 제한입니다.
frontend 연결?중복 요청, stale response, optimistic update 충돌로 나타날 수 있습니다.
CS-06

Q. process vs thread, context switching, memory, file descriptor를 설명해보세요.

핵심 개념

process는 독립된 실행 단위와 주소 공간을 가지고, thread는 process 안에서 메모리를 공유하는 실행 흐름입니다. context switching은 CPU가 실행 대상을 바꾸며 상태를 저장/복원하는 비용입니다. file descriptor는 열린 파일, socket 같은 OS resource를 가리키는 핸들입니다.

개념 해부

process는 격리된 주소 공간과 자원을 가진 실행 단위이고, thread는 같은 process 안에서 메모리를 공유하는 실행 흐름입니다. thread는 가볍지만 공유 메모리 때문에 race condition이 생기고, process는 격리가 강하지만 IPC와 전환 비용이 큽니다. file descriptor는 파일뿐 아니라 socket, pipe 같은 OS 자원 핸들이므로 누수되면 네트워크 서버가 새 연결을 받지 못할 수 있습니다.

30초 답변

process는 메모리 공간이 분리되어 안정적이지만 생성/전환 비용이 크고, thread는 같은 process 메모리를 공유해 가볍지만 동시성 버그가 생길 수 있습니다. context switching이 많으면 CPU가 실제 작업보다 전환에 시간을 쓰고, file descriptor가 누수되면 파일이나 socket을 더 열 수 없습니다.

90초 답변

웹 서버에서 thread pool이 고갈되면 요청이 대기하고, DB connection pool이 작거나 누수되면 API가 멈춘 것처럼 보입니다. socket도 file descriptor로 관리되므로 connection leak은 FD 고갈로 이어질 수 있습니다. OS 기본기를 알면 “서버가 느리다”를 CPU, memory, thread, FD, network로 나눠 볼 수 있습니다.

꼬리질문

질문답변 방향
memory leak?더 이상 필요 없는 객체나 resource가 참조/해제되지 않아 계속 남는 상태입니다.
context switching 비용?register/cache 상태 전환 비용과 scheduler overhead가 있습니다.
FD 확인?lsof, ulimit, proc filesystem 등으로 확인할 수 있습니다.

CS-08

Q. CS 지식을 실제 장애·성능 현상과 어떻게 연결하나요?

면접에서 CS 기본기가 깊어 보이는 순간은 정의를 말할 때가 아니라, 운영 현상을 OS·네트워크·런타임 개념으로 분해할 때입니다.

CS SYSTEM TRACE “API가 느립니다”라는 현상을 CPU, memory, thread, event loop, file descriptor, DB connection, network latency로 나누어 봅니다. CPU가 낮아도 thread pool이 꽉 차 있거나, FD가 고갈되었거나, downstream timeout 때문에 요청이 대기할 수 있습니다. 그래서 지표를 한 개만 보지 않고 queue length, active thread, GC pause, socket state, connection pool wait time을 함께 확인합니다.
현상관련 CS 개념전문적인 답변 방향
CPU는 낮은데 응답이 느림I/O wait, blocking call, thread pool saturation실행 시간이 아니라 대기 시간이 병목일 수 있으므로 thread dump, pool wait, downstream latency를 봅니다.
간헐적 timeoutqueueing, tail latency, retry amplification평균 latency보다 p95/p99와 retry 폭증, connection reuse, timeout budget을 확인합니다.
메모리가 계속 증가heap, GC, resource leak, retained reference객체 수명과 참조 경로, cache eviction, listener/socket close 여부를 확인합니다.
동시 요청에서 값이 꼬임race condition, atomicity, visibility공유 상태와 불변식이 어떤 순서에서 깨지는지 추적하고 lock, transaction, idempotency를 검토합니다.
서버가 새 연결을 못 받음file descriptor, socket lifecycle, backlogFD 한도, CLOSE_WAIT, accept queue, keep-alive 설정, connection leak을 확인합니다.
답변 패턴

“정의 → 발생 조건 → 관측 지표 → 해결 선택지 → trade-off” 순서로 말합니다. 예를 들어 context switching은 “CPU가 작업을 바꾸는 비용”에서 끝내지 말고 thread가 과도하게 많을 때 cache miss와 scheduler overhead가 늘고, 해결책은 pool 크기 조정과 blocking 구간 축소이며, 너무 줄이면 처리량이 떨어진다는 trade-off까지 연결합니다.

CS-09

CS 면접 답변 심화 루브릭

암기 답변과 전문 답변은 같은 키워드를 쓰더라도 검증 가능한 근거를 붙이는지에서 갈립니다.

루브릭부실한 답변출판 가능한 답변 기준
입력 규모“O(n)이라 괜찮습니다.”n의 범위, 호출 빈도, 메모리 비용, I/O 비용을 함께 말합니다.
불변식“lock을 걸면 됩니다.”어떤 공유 상태의 어떤 조건이 깨지는지 먼저 설명합니다.
실패 모드“retry하면 됩니다.”중복 실행, 순서 역전, timeout budget, retry storm 가능성을 같이 다룹니다.
관측 근거“느릴 것 같습니다.”CPU, memory, thread, GC, latency percentile, queue length처럼 확인 지표를 제시합니다.
대안 비교“HashMap이 빠릅니다.”HashMap, TreeMap, array scan, DB index처럼 선택지를 비용과 제약으로 비교합니다.
CS DEPTH RUBRIC 좋은 CS 답변은 “용어를 안다”가 아니라 “제약에서 선택을 도출한다”입니다. 면접 중에는 먼저 문제 규모와 불변식을 확인하고, 그다음 자료구조·동시성 제어·OS 자원·네트워크 비용 중 무엇이 핵심 병목인지 좁히는 방식으로 답합니다.
CS-10

네트워크·메모리 압박 질문

CS 기본기 면접은 알고리즘만 묻지 않습니다. 웹 개발자에게는 네트워크, 메모리, 런타임 자원까지 연결해 묻는 경우가 많습니다.

CS NETWORK MEMORY DRILL “브라우저에서 API 요청이 느립니다”라는 질문에는 DNS, TCP handshake, TLS, HTTP keep-alive, server queue, DB query, response serialization, browser parsing을 순서대로 나눕니다. “메모리가 샙니다”라는 질문에는 heap object, native resource, event listener, closure, cache, file descriptor를 나눠 봅니다.
압박 질문얕은 답변좋은 답변
TCP와 HTTP 차이는?TCP는 전송, HTTP는 웹입니다.TCP는 신뢰성 있는 byte stream이고 HTTP는 그 위에서 request/response 의미를 정의하는 application protocol입니다. 지연은 handshake, congestion, keep-alive, server processing으로 나눠 봅니다.
메모리 누수는 GC가 막아주지 않나요?GC가 알아서 치웁니다.GC는 도달 불가능한 객체만 회수합니다. cache, static reference, listener, closure, ThreadLocal이 계속 참조하면 논리적으로 필요 없어도 회수되지 않습니다.
CPU가 낮은데 왜 느릴까요?네트워크 문제 같습니다.I/O wait, lock wait, thread pool saturation, DB connection wait, downstream timeout처럼 CPU를 쓰지 않고 대기하는 병목을 의심합니다.
p99가 중요한 이유는?느린 사용자도 있기 때문입니다.평균은 tail latency를 숨깁니다. queueing과 retry가 겹치면 일부 요청이 전체 사용자 경험과 장애 전파를 결정합니다.

면접에서 말할 증거

waterfall, TCP reuse, TLS handshake time, p95/p99 latency, heap snapshot, retained path, FD count, thread dump, pool wait time을 말합니다.

주의

네트워크와 메모리를 “느리다/샌다” 정도로 말하면 초안 수준입니다. 어느 계층에서 어떤 자원이 대기하거나 유지되는지 쪼개야 합니다.

CS-11

동시성 검증 질문

동시성 답변은 lock 이름을 아는 것보다 불변식이 어떤 interleaving에서 깨지는지 설명하는 것이 핵심입니다.

CONCURRENCY INVARIANT CHECK 동시성 문제를 받으면 공유 상태, 읽기-수정-쓰기 순서, 원자성 경계, visibility, 실패 시 재시도 가능성을 먼저 봅니다. “lock을 겁니다”가 아니라 어떤 invariant를 보호하고 lock 범위를 어디까지 둘지 말해야 합니다.
질문답변 기준놓치기 쉬운 점
재고 차감 중복 요청재고가 0 미만이 되지 않는 invariant를 DB transaction, row lock, optimistic lock, idempotency로 보호합니다.API retry와 사용자 더블클릭이 같은 중복 문제를 만들 수 있습니다.
낙관적 락 충돌version 조건 update 실패를 감지하고 사용자 재시도 또는 merge 정책을 둡니다.무한 retry는 부하를 키울 수 있습니다.
캐시와 DB 동시 갱신source of truth, invalidation 순서, stale read 허용 범위, outbox/event 기반 보정을 말합니다.DB commit 전 캐시를 바꾸면 실패 시 불일치가 생깁니다.
분산 locklease, timeout, fencing token, lock holder 장애를 함께 봅니다.lock 획득 성공만으로 안전하지 않습니다.
thread-safe collection개별 연산 안전과 복합 연산 안전을 구분합니다.contains 후 put 같은 check-then-act는 여전히 race가 날 수 있습니다.
흔한 오답

동시성 문제를 “synchronized를 쓰면 됩니다”로 끝내면 부족합니다. 단일 프로세스 lock인지, DB transaction인지, 분산 환경인지에 따라 보호 범위가 달라집니다.

CS-12

OS NETWORK INTERNALS DRILL

심화 CS 면접은 정의 암기에서 끝나지 않고, 요청 하나가 OS·네트워크·런타임을 지나며 어떤 비용과 실패 모드를 만드는지 시간순으로 설명하게 합니다.

OS NETWORK INTERNALS DRILL 브라우저가 API를 호출하면 DNS 조회 후 TCP 3-way handshake, TLS handshake, HTTP request 전송, 서버 accept, kernel/user mode 전환, application handler, DB/network I/O, response write, TCP ACK 흐름이 이어집니다. keep-alive와 connection pool이 있으면 일부 handshake는 생략되지만, TLS 재협상, packet loss, head-of-line blocking, server queue 대기는 여전히 tail latency를 만들 수 있습니다.

메모리와 커널 경계

virtual memory는 process마다 독립된 주소 공간을 제공하고, page table을 통해 virtual address를 physical frame에 매핑합니다. 접근한 page가 메모리에 없거나 권한이 맞지 않으면 page fault가 발생하며, 정상적인 minor fault는 매핑만 보정하고 major fault는 disk I/O를 동반할 수 있습니다. page cache는 파일과 socket I/O의 반복 접근을 빠르게 하지만, cache pressure가 커지면 reclaim과 writeback이 응답 지연으로 보일 수 있습니다. system call은 user mode에서 kernel mode로 넘어가므로 너무 잦은 read/write, context switching, copy 비용도 병목이 됩니다.

TCP/TLS/HTTP 타임라인

단계확인할 내부 동작면접 답변 포인트
DNScache hit, resolver 지연, TTL첫 요청만 느린지, 특정 region에서만 느린지 분리합니다.
TCPSYN/SYN-ACK/ACK, congestion window, retransmissionpacket loss와 handshake 비용은 p95/p99를 악화시킬 수 있습니다.
TLScertificate 검증, key exchange, session resumption새 연결이 많으면 CPU와 latency가 증가하고 keep-alive가 중요합니다.
HTTPheader/body, keep-alive, HTTP/2 multiplexing, server queueapplication 처리 전 대기 시간이 waterfall에 나타날 수 있습니다.
OS I/Osocket buffer, epoll/kqueue, kernel/user copy, page cacheCPU가 낮아도 I/O wait와 buffer pressure가 병목일 수 있습니다.

동시성 내부 모델

happens-before는 한 작업의 write가 다른 작업의 read에 관측될 수 있음을 보장하는 순서 관계입니다. lock, volatile, channel, thread start/join 같은 동기화 지점이 없으면 CPU cache와 compiler reorder 때문에 값이 보이지 않을 수 있습니다. CAS는 compare-and-swap으로 lock 없이 상태를 바꾸는 원자 연산이지만, 값이 A에서 B로 갔다가 다시 A가 되는 ABA 문제를 놓칠 수 있어 version stamp나 hazard pointer 같은 보완이 필요합니다. thread pool starvation은 pool 안 작업이 같은 pool의 다른 작업 완료를 기다리거나 blocking I/O가 pool을 모두 점유할 때 발생하며, 별도 executor, bounded queue, timeout, backpressure로 완화합니다.

압박 포인트

“CPU가 낮은데 API가 느립니다”라는 질문에는 thread pool starvation, connection pool wait, kernel I/O wait, page fault, TLS handshake, DNS 지연을 같은 타임라인 위에 올려서 좁혀야 합니다.

CS-13

핵심 질문 30선과 구체 답변 훈련

아래 질문은 키워드 암기가 아니라 면접장에서 바로 말할 수 있는 구체 상황형 질문입니다.

번호구체 질문답변 방향
1HashMap 조회가 평균 O(1)인데도 대량 요청에서 느려진다면 어떤 원인을 의심하나요?collision, resize, hash 품질, key mutation, cache locality, 동시성 경합을 분리합니다.
2무가중치 그래프 최단 경로는 왜 DFS보다 BFS가 적합한가요?BFS가 거리 k 노드를 모두 본 뒤 k+1로 넘어가므로 처음 도달이 최단입니다.
3프로세스 격리와 스레드 공유 메모리 차이가 장애 격리에 어떤 영향을 주나요?process는 주소 공간 격리가 강하고 thread는 저렴하지만 공유 상태 오염 위험이 큽니다.
4context switching이 많아졌다는 지표를 보면 어떤 설계 문제를 의심하나요?과도한 thread 수, blocking I/O, lock 경합, 작은 작업의 과도한 분할을 봅니다.
5virtual memory에서 page fault가 항상 장애나 버그는 아닌 이유는 무엇인가요?lazy allocation과 file mapping의 정상 경로일 수 있고, major fault가 많을 때 I/O 병목을 의심합니다.
6page cache 때문에 파일 읽기 benchmark가 왜 오해를 만들 수 있나요?두 번째 실행은 disk가 아니라 memory cache를 읽을 수 있어 cold/warm cache를 구분해야 합니다.
7브라우저 요청의 TCP/TLS/HTTP 타임라인을 어디까지 쪼개 설명할 수 있나요?DNS, TCP handshake, TLS, request queue, server processing, response transfer, reuse 여부를 말합니다.
8kernel/user mode 전환 비용은 어떤 코드에서 문제가 되나요?작은 read/write 반복, logging 폭증, system call 과다, context switch가 많은 서버에서 커집니다.
9happens-before가 없는 공유 변수 읽기는 왜 간헐적으로만 실패하나요?CPU cache, reorder, scheduler interleaving에 따라 visibility가 달라져 재현성이 낮습니다.
10CAS가 lock보다 항상 낫지 않은 이유는 무엇인가요?retry loop가 경쟁에서 CPU를 태우고 ABA, starvation, 복합 invariant 보호 한계가 있습니다.
11ABA 문제를 면접에서 짧게 예로 들면 어떻게 설명하나요?값이 A라서 안전하다고 봤지만 중간에 A-B-A로 바뀌어 참조 의미가 달라진 경우입니다.
12thread pool starvation을 실제 서비스에서 어떻게 탐지하고 완화하나요?active thread, queue length, pool wait, thread dump를 보고 blocking 작업 분리와 timeout을 둡니다.
답변 기준

각 답변은 “정의 → 발생 조건 → 관측 지표 → 완화책 → trade-off” 순서로 말하면 키워드 나열을 넘어섭니다.

CS-14

CS 개념 인과관계

CS 답변은 용어 정의보다 “왜 그 현상이 생기는지”를 설명할 때 깊어집니다. 면접관이 보고 싶은 것은 Big-O, thread, TCP 같은 단어 자체가 아니라 그 개념으로 실제 성능과 장애를 분해하는 사고입니다.

CS CAUSAL MODEL입력 크기가 커지면 알고리즘 비용이 늘고, 비용이 CPU 계산인지 I/O 대기인지에 따라 해결책이 달라집니다. thread가 많아지면 병렬성이 늘 수 있지만 context switching과 lock 경합이 늘어 tail latency를 키울 수 있습니다. TCP 연결이 새로 열리면 handshake와 congestion window 때문에 첫 요청이 느릴 수 있고, keep-alive를 쓰면 그 비용은 줄지만 connection leak과 FD 고갈을 관리해야 합니다. 좋은 답변은 한 개념을 다른 계층의 증상과 연결합니다.
개념원인결과면접에서 말할 문장
Big-O입력 크기 증가에 따라 반복 횟수나 탐색 공간이 커진다작은 데이터에서는 괜찮던 코드가 대량 데이터에서 급격히 느려진다“복잡도는 ms 예측이 아니라 증가율 판단이고, 실제로는 I/O와 호출 빈도를 함께 봅니다.”
context switchingthread가 너무 많거나 blocking 대기가 많아 scheduler가 자주 실행 대상을 바꾼다CPU가 실제 업무보다 전환과 cache miss에 시간을 쓴다“thread를 늘리는 것은 항상 처리량 증가가 아니고 pool wait와 runnable thread 수를 함께 봅니다.”
race condition읽기-수정-쓰기 사이에 다른 실행 흐름이 끼어든다테스트에서는 가끔만 실패하고 운영에서 값이 꼬인다“lock 이름보다 어떤 invariant가 어떤 interleaving에서 깨지는지 먼저 설명합니다.”
TCP/TLS새 연결에는 handshake, 인증서 검증, congestion warm-up이 필요하다첫 요청이나 connection churn이 p95/p99를 악화시킨다“느린 API를 보면 application time뿐 아니라 DNS, TCP, TLS, server queue를 분리합니다.”
암기 답변 교정

“프로세스는 자원 할당 단위, 스레드는 실행 단위”에서 멈추면 암기입니다. “그래서 thread는 메모리를 공유해 빠르지만 race condition이 생기고, process는 격리가 강해 장애 전파는 줄지만 IPC 비용이 듭니다”까지 이어져야 합니다.
