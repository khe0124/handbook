VPN 구축
체크리스트

VPN 구축에서 빠지는 항목은 대부분 정해져 있습니다. IP 대역, Peer IP, IKE 파라미터, route, firewall, 검증 명령을 표로 고정하면 협의와 장애 대응이 빨라집니다.

GOAL : 구축 전·중·후 확인 항목을 하나의 Runbook으로 남기기
CHK-01

사전 협의

체크값완료 기준
우리 사설 대역10.0.0.0/16고객사 대역과 미충돌
고객사 사설 대역172.16.0.0/16우리 VPC, Docker, Kubernetes CIDR과 미충돌
우리 VPN 공인 IP클라우드 Gateway IP고객사에 전달
고객사 VPN 공인 IPPeer IP콘솔에 입력
허용 서비스TCP 443, 5432 등출발지/목적지/포트 표 확정
개인명 제거 원칙

운영 문서에는 담당자 개인 이름보다 역할과 연락 채널을 적습니다. 예: “고객사 네트워크 담당”, “우리 클라우드 운영 채널”, “애플리케이션 온콜”. 사람이 바뀌어도 문서가 유지됩니다.

CHK-02

구성 작업

BUILD ORDER [1] VPC와 서브넷 생성
[2] VPN Gateway 생성
[3] Customer Gateway 또는 Peer 정보 등록
[4] Phase 1/2 proposal과 PSK 입력
[5] Route table에 고객사 대역 → VPN Gateway 추가
[6] ACG/Security Group에 필요한 포트 허용
[7] 고객사 방화벽과 return route 확인
항목실패하면 보이는 증상
PSK 또는 Phase 1 불일치터널 DOWN
Phase 2 selector 불일치터널 DOWN 또는 특정 대역 실패
route table 누락패킷이 VPN으로 가지 않음
ACG 누락터널 UP이지만 포트 연결 실패
고객사 return route 누락요청은 가지만 응답 없음

문서화 항목내용

운영 기준

구축 완료의 기준은 “한 번 연결됐다”가 아니라, 장애가 났을 때 다른 운영자가 같은 정보로 재현·진단·복구할 수 있는 상태입니다.

CHK-05

변경관리

변경 전before
현재 route, firewall, VPN proposal, 터널 상태를 캡처합니다. rollback 기준도 정합니다.
변경 중during
우리 쪽과 고객사 쪽 담당자가 동시에 대기합니다. 한쪽만 바꾸면 장애 시간이 길어집니다.
변경 후after
터널 UP, route, 포트, 앱 health를 모두 검증하고 증거를 남깁니다.
PSK rotationsecret
PSK 교체는 양쪽 동시 변경이 필요합니다. 변경 창, 보관 위치, 폐기 절차를 문서화합니다.
