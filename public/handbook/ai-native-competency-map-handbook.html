AI Native 역량
매트릭스와 진단

AI Native 역량은 "AI를 써봤다"가 아니라 제품 기능, 평가, 보안, 운영, 작업 루프를 증거로 닫는 능력입니다.

OUTPUT : competency scorecard · evidence ledger · next training queue
MAP-00

큰그림 로드맵

중급 개발자가 AI Native로 전환할 때의 목표는 “모델을 잘 쓰는 사람”이 아니라 불확실한 AI 출력을 제품 변경, 평가, 보안, 운영 증거로 통제하는 사람이 되는 것입니다. 시작 단계에서는 아래 6단계를 순서대로 밟습니다.

문제 정의→ 출력 계약→ 근거 공급→ 평가→ 보안·권한→ 운영·증거화
단계핵심 질문훈련 항목완료 산출물
1. 문제 정의AI가 반드시 필요한 사용자 문제인가product brief, non-goal, risk tierproblem brief, user flow
2. 출력 계약모델 출력이 API·UI 계약을 만족하는가prompt/schema regressionprompt contract, response schema
3. 근거 공급모델이 모르는 지식을 어디서 가져오는가RAG retrieval evalretrieval report, citation failure log
4. 평가변경이 개선인지 퇴행인지 반복 측정되는가golden set, baseline/candidate 비교eval dataset, quality report
5. 보안·권한비신뢰 입력과 도구 실행을 통제하는가red-team fixture, approval policysecurity report, approval ledger
6. 운영·증거화비용·지연·품질 실패를 감지하고 설명하는가incident drill, portfolio evidence packdashboard spec, runbook, evidence pack
냉정한 기준

중급 개발자의 시작 목표는 L4가 아닙니다. 1차 목표는 모든 축에서 L2를 통과하고, RAG·평가·보안·운영 중 최소 2개 축에서 L3 증거를 남기는 것입니다. 이 수준을 못 넘으면 “AI Native”가 아니라 “AI 도구 사용자”에 가깝습니다.

MAP-01

판정 기준

역량 진단은 감상이나 자신감이 아니라 제출 가능한 산출물로 판정합니다. 설명할 수 있어도 재현 가능한 증거가 없으면 아직 실무 역량으로 보지 않습니다. 점수는 평균이 아니라 최저 축으로 제한합니다. 평가·보안·운영이 비어 있으면 L3 이상으로 판정하지 않습니다.

수준점수판정 기준대표 증거탈락 조건
L1 입문1개념을 설명하고 작은 예제를 따라 할 수 있다학습 메모, 작은 실습 결과동작 원리와 실패 모드를 구분하지 못함
L2 실무 가능2제한된 기능을 구현하고 기본 검증을 남긴다테스트, prompt contract, API 계약샘플 성공만 있고 회귀 기준이 없음
L3 독립 운영 가능3실패 모드, 보안 경계, 운영 지표까지 책임진다eval report, runbook, incident packet품질·비용·권한 실패를 재현하지 못함
L4 리드 가능4팀 표준, 리뷰 기준, 자동화, 거버넌스를 설계한다rubric, CI gate, approval policy, dashboard개인 프로젝트 증거만 있고 팀 적용 기준이 없음
산정 규칙계량 방식
총점10개 역량 축을 1~4점으로 채점한다. 만점 40점.
현재 등급평균 점수보다 최저 핵심 축을 우선한다. 평가·보안·운영 중 하나라도 L1이면 전체 등급은 최대 L2.
중급 개발자 목표28점 이상, 모든 축 L2 이상, 평가·보안·운영 중 2개 이상 L3.
포트폴리오 제출선32점 이상, RAG 또는 Agent 프로젝트 하나에 eval/security/ops 증거가 모두 존재.

MAP-03

빠른 수준 진단

각 질문에 대해 현재 남길 수 있는 증거가 없다면 해당 수준은 아직 통과하지 않은 것으로 봅니다.

질문L1L2L3L4
LLM 품질을 어떻게 판단하는가샘플 답변을 본다golden set을 만든다회귀 gate를 둔다팀 릴리스 기준으로 운영한다
RAG 오답을 어떻게 고치는가프롬프트를 바꾼다retrieval과 generation을 분리한다chunk/rerank/citation 지표를 본다index 운영과 drift를 관리한다
도구 호출을 어떻게 통제하는가schema를 정의한다approval을 둔다idempotency와 audit을 둔다policy와 replay를 표준화한다
AI 경험을 어떻게 설명하는가사용 도구를 말한다구현 기능을 말한다평가/보안/운영 증거를 말한다팀 표준화 경험을 말한다
현재 상태냉정한 판정다음 훈련
LLM API 호출과 간단한 챗봇 구현 가능L1 후반. 제품 역량으로는 부족함.LAB-01 prompt/schema contract

eval dataset과 보안 fixture가 있음L2 후반. 운영 지표가 붙어야 L3.LAB-05 incident drill
릴리스 gate, red-team, runbook까지 있음L3. 독립 운영 가능 후보.포트폴리오 evidence pack 정리

MAP-05

보강 루프

COMPETENCY HARDENING LOOP 1. 10개 역량 축을 1~4점으로 채점한다
2. 평가·보안·운영 중 가장 낮은 축을 먼저 고른다
3. 해당 축의 AI Native 실습 랩 fixture를 실행한다
4. 실패 원인을 prompt, retrieval, tool, permission, operation 중 하나로 분류한다
5. 개선 diff와 재측정 결과를 남긴다
6. evidence ledger와 포트폴리오 문장을 동시에 업데이트한다
주차훈련 초점통과 산출물
1주차문제 정의와 출력 계약problem brief, prompt contract, schema regression result
2주차RAG 근거와 평가retrieval dataset, eval report, failure analysis
3주차보안과 도구 통제red-team report, approval policy, replay trace
4주차운영과 포트폴리오LLM runbook, incident packet, evidence pack
완료 기준

이 문서를 끝냈다는 뜻은 표를 읽었다는 뜻이 아니라, 현재 자신의 L1-L4 위치, 총점, 다음 실습 랩, 제출할 산출물을 정했다는 뜻입니다. 중급 개발자 기준으로는 4주 안에 최소 28점과 L3 증거 2개를 목표로 둡니다.
