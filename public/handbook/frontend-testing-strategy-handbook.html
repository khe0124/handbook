프론트엔드 테스트 전략
핸드북

좋은 테스트는 코드를 얼리지 않고 사용자 계약을 보호합니다. 구현 세부를 많이 알수록 refactor를 방해하고, 사용자 행동을 검증할수록 오래 갑니다.

TOOLS : Vitest · Testing Library · Playwright · Visual Regression
FTEST-01

테스트의 목적

테스트는 QA 문서가 아니라 변경 안전장치입니다. 실패했을 때 무엇이 깨졌고 사용자에게 어떤 영향이 있는지 알려야 합니다.

나쁜 테스트좋은 테스트
컴포넌트 내부 함수 호출을 검증사용자 입력 후 화면 결과를 검증
거대한 snapshot만 비교의미 있는 텍스트, role, 상태를 확인
mock 구현을 테스트실제 DOM과 사용자 이벤트를 테스트
깨져도 원인을 모름테스트 이름이 시나리오와 기대 결과를 말함
FTEST-02

테스트 피라미드

프론트엔드에서는 피라미드보다 "테스트 트로피" 사고가 더 실용적일 때가 많습니다. 순수 로직은 단위, UI 계약은 컴포넌트, 핵심 여정은 E2E로 보호합니다.

층대상예
Unit순수 함수, reducer, formatter가격 포맷, 권한 계산, validation schema
ComponentDOM과 사용자 상호작용필터 입력 후 목록 갱신, 오류 메시지 표시
Integration라우터, 데이터 fetching, provider 조합상세 페이지 로딩/실패/성공 상태
E2E핵심 사용자 여정로그인 후 생성, 수정, 권한 차단
FTEST-03

컴포넌트 테스트

Testing Library의 관점은 "사용자가 찾는 방식으로 찾는다"입니다. role, label, text를 우선합니다.

COMPONENT TEST 원칙getByRole로 상호작용 요소 찾기
userEvent로 실제 입력에 가깝게 조작
비동기 UI는 findBy/waitFor 사용
내부 state나 private 함수는 직접 검증하지 않음
접근 가능한 이름을 테스트가 자연스럽게 요구하게 만들기
Testing Library 규칙이유
getByRole({ name }) 우선사용자와 보조기술이 인식하는 이름을 검증
queryBy*는 부재 검증에 사용없는 요소를 찾을 때 throw하지 않음
waitFor 안에 여러 assertion 남발 금지실패 원인이 흐려지고 재시도 비용 증가
jest-dom matcher 사용toBeVisible, toBeDisabled, toHaveAccessibleName처럼 의도를 드러냄
fake timer와 userEvent 조합 주의입력 이벤트와 timer 진행이 어긋나 flaky해질 수 있음
ACCESSIBILITY BENEFIT

role과 label로 테스트가 가능하다는 것은 보조기술도 그 UI를 이해할 가능성이 높다는 뜻입니다. 테스트 가능성과 접근성은 자주 같은 방향으로 움직입니다.

FTEST-04

E2E 테스트

E2E는 비싸지만 실제 통합 문제를 잡습니다. 모든 것을 E2E로 덮지 말고, 실패하면 매출/업무 흐름이 깨지는 핵심 경로에 집중합니다.

좋은 후보이유
로그인/로그아웃모든 기능의 진입점
핵심 생성/수정/삭제업무 가치가 직접 연결
결제/신청/제출중복 제출과 실패 처리가 중요
권한별 접근보안 회귀 영향이 큼
Playwright 규칙기준
locatorgetByRole, getByLabel, getByText 우선. data-testid는 안정적인 사용자 식별자가 없을 때만
대기page.waitForTimeout 금지. web-first assertion으로 상태를 기다림
데이터테스트별 독립 데이터 생성/정리. 실행 순서에 의존하지 않음
인증storageState 재사용은 빠르지만 권한/세션 격리 위험을 고려
CI artifact실패 시 trace, video, screenshot을 남겨 재현 비용을 줄임
FLAKY TEST

flaky 테스트는 신뢰를 갉아먹습니다. 임의 sleep 대신 명확한 상태를 기다리고, 테스트 데이터는 격리하며, 외부 서비스 의존은 통제합니다.

FTEST-05

Mock 전략

mock은 빠른 테스트를 만들지만 현실과 멀어질 수 있습니다. 어떤 경계를 mock할지 명확히 정해야 합니다.

방식언제주의
함수 mock작은 단위 테스트구현 세부에 결합되기 쉬움
MSW컴포넌트/통합 테스트에서 API 경계실제 API schema와 동기화 필요
fixture반복 데이터현실적인 edge case 포함
real backend소수의 smoke/E2E데이터 격리와 속도 문제
FTEST-06

시각 회귀

프론트엔드는 로직이 맞아도 UI가 깨질 수 있습니다. 시각 회귀는 디자인 시스템과 핵심 화면에 특히 유용합니다.

대상왜
공통 컴포넌트작은 변경이 여러 화면에 퍼짐
핵심 폼오류/로딩/비활성 상태 깨짐 방지
반응형 레이아웃모바일에서 겹침/잘림 방지
차트/테이블데이터 길이 변화에 취약
Test Contractuser behavior
테스트가 보호하는 사용자 관찰 가능 동작입니다.
Fixturetest data
테스트에서 반복 사용하는 현실적인 예제 데이터입니다.
PR CHECK

새 기능 PR은 어떤 계층의 테스트가 필요한지 명시합니다. 순수 로직은 unit, 사용자 상호작용은 component, 핵심 업무 흐름은 E2E, 공통 UI 변경은 visual regression 후보로 분류합니다.
