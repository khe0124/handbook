PRM-01

PROMPT CONTRACT

프롬프트는 문자열이 아니라 버전 관리되는 API contract입니다. 변경 시 품질, 비용, 지연, 보안 회귀가 생길 수 있으므로 코드처럼 소유자와 검증 기준이 있어야 합니다.

PRM-02

STRUCTURED OUTPUT CONTRACT

JSON을 “잘 써줘”라고 요청하는 것과 schema로 강제하는 것은 다릅니다. 제품 코드는 파싱 가능한 계약에 의존해야 하며, schema evolution 규칙도 필요합니다.

{
  "type": "object",
  "additionalProperties": false,
  "required": ["summary", "key_points", "risks", "confidence", "citations"],
  "properties": {
    "summary": {"type": "string", "maxLength": 600},
    "key_points": {"type": "array", "items": {"type": "string"}},
    "risks": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["level", "reason", "evidence"],
        "properties": {
          "level": {"type": "string", "enum": ["low", "medium", "high"]},
          "reason": {"type": "string"},
          "evidence": {"type": ["string", "null"]}
        }
      }
    },
    "confidence": {"type": "string", "enum": ["low", "medium", "high"]},
    "citations": {"type": "array", "items": {"type": "string"}}
  }
}
Schema 설계좋은 기준실패 예
enumUI 분기와 저장 값을 고정`maybe`, `normal` 같은 임의 값
nullable모르는 값과 빈 문자열을 구분근거가 없는데 `""`로 저장
additionalProperties예상 밖 필드를 거부모델이 만든 새 필드가 downstream을 오염
schema evolution새 필드는 optional로 추가 후 version bump기존 클라이언트가 즉시 깨짐
PRM-03

Validation And Repair Loop

schema validation 실패는 예외 상황이 아니라 정상 운영 경로입니다. 실패할 때마다 같은 요청을 무한 재시도하면 비용만 증가합니다.

검증 계층검사실패 처리
SyntaxJSON parse 가능 여부1회 repair prompt
Schemarequired, enum, type, maxLength잘못된 필드만 수정 요청
Semanticcitation이 실제 chunk를 가리키는가해당 주장 제거 또는 no-answer
Business rule권한, 위험 등급, 저장 가능 여부fail closed, human review
repair loop invalid output → classify failure → targeted repair once → validate again → fallback or fail closed → log schema_failure_rate

PRM-05

streaming event contract

스트리밍은 UX 개선이지만, 저장과 후속 처리는 검증된 최종 structured output만 사용해야 합니다. 부분 토큰을 곧바로 DB 상태로 믿으면 중간 실패가 제품 데이터가 됩니다.

PRM-07

OUTPUT SINK SECURITY

모델 출력은 사용자에게 보이는 텍스트에서 끝나지 않습니다. HTML/Markdown, SQL, shell command, browser action, email, ticket, DB write 같은 sink로 흘러갈 때는 다시 비신뢰 데이터로 검증해야 합니다.

sink위험통제
HTML/MarkdownXSS, 악성 링크, 이미지 추적 pixelsanitize, link allowlist, raw HTML 금지
SQL권한 밖 조회, destructive query, injectionread-only role, query parser, allowlisted template
shell command임의 명령 실행, 파일 삭제, secret 노출승인 게이트, sandbox, command allowlist, dry-run
browser action폼 제출, 결제, 외부 전송preview, user confirmation, idempotency
email/ticket개인정보 유출, 잘못된 수신자recipient validation, PII scan, approval payload
OUTPUT SINK RULE모델이 생성한 것은 파싱 성공 후에도 신뢰 입력이 아니다. sink 직전의 서버 코드가 정책, 권한, sanitizer, audit를 다시 적용한다.
DEPTH

프롬프트를 제품 계약으로 운영하기

MARKER: concept-depth — Prompt contract as API contract

나쁜 프롬프트는 “이 문서를 요약하고 위험을 찾아줘”처럼 모델에게 의도를 맡깁니다. 프로덕션 프롬프트는 API contract처럼 동작해야 합니다. 어떤 입력을 신뢰할 수 있는지, 어떤 근거만 사용할 수 있는지, 어떤 경우에 no-answer로 멈춰야 하는지, 출력 schema가 무엇인지, 실패 시 repair를 몇 번 허용하는지까지 명시해야 합니다. 프롬프트가 바뀌면 API 동작도 바뀌므로 version, owner, eval dataset, rollback criteria가 필요합니다.

이 관점에서 structured output은 예쁜 JSON을 받기 위한 장치가 아닙니다. downstream UI는 enum 값을 기준으로 배지를 그리고, DB는 nullable 필드와 빈 문자열을 다르게 저장하며, analytics는 confidence와 risk level을 집계합니다. 모델이 임의 필드를 추가하거나 citation 없는 risk를 만들면 화면과 데이터 품질이 동시에 무너집니다. schema는 모델 출력과 제품 코드 사이의 방화벽입니다.

MARKER: artifact-ready — Prompt contract YAML

version: doc-analysis@2026-06-30
owner: ai-platform
task: summarize_and_extract_risks
allowed_inputs: [user_text, document_text, document_metadata, retrieved_chunks]
forbidden_behavior: [external_fact_guessing, uncited_claims, hidden_policy_disclosure]
evidence_policy: every_risk_requires_citation
refusal_shape: {reason: string, missing_evidence: string[], next_action: string}
output_schema_ref: document_analysis.v2.json
eval_dataset: golden-doc-analysis-v4
rollback_criteria: groundedness_drop_gt_2pp or schema_failure_rate_gt_1pct
logs: [prompt_version, model_version, input_tokens, output_tokens, retry_count]

MARKER: failure-driven — Repair loop worked example

repair loop는 “다시 JSON으로 답해”가 아닙니다. 먼저 validator가 실패 유형을 분류해야 합니다. 예를 들어 모델이 `confidence: "normal"`을 반환했다면 enum 위반이고, `citations: []`인데 high risk를 반환했다면 semantic validation 실패입니다. 첫 번째는 해당 필드만 고치도록 targeted repair를 보낼 수 있지만, 두 번째는 근거가 없으므로 risk를 제거하거나 no-answer로 전환해야 합니다. 두 실패를 같은 재시도로 처리하면 비용은 늘고 품질은 개선되지 않습니다.

validator_error: {path: "/risks/0/evidence", error: "required citation missing"}
repair_prompt: "Do not add new claims. Either attach an existing citation from chunks [c12,c18] or remove the unsupported risk."
second_failure_policy: fail_closed_and_send_to_review_queue

MARKER: artifact-ready — Tool approval and audit payload

tool_proposal: {tool: "send_email", args: {to: "client@example.com", template: "renewal"}}
approval_payload: {risk_tier: "write-external", preview_required: true, rollback: "not_possible"}
audit_log: {actor_id: "usr_7", tool: "send_email", args_hash: "sha256:...", approval_actor: "usr_7", result_hash: "sha256:..."}

Streaming 역시 상태 머신으로 봐야 합니다. `message.delta`는 사용자에게 보여주는 임시 상태일 뿐이고, DB 저장이나 후속 workflow는 `output.validated` 이후에만 시작해야 합니다. 사용자가 중간에 취소하거나 네트워크가 끊긴 뒤 재연결하면 같은 tool call이 두 번 실행될 수 있으므로 idempotency key가 필요합니다. 스트리밍은 UX 최적화지만, 검증되지 않은 부분 출력을 제품 상태로 승격하면 데이터 정합성 버그가 됩니다.

MARKER: failure-driven — JSON만 강제하고 의미 검증을 하지 않는 실패

구조화 출력은 “파싱 가능한 JSON”을 받는 것으로 끝나지 않습니다. 예를 들어 risk level은 enum에 맞고 citations 배열도 존재하지만, citation이 실제 근거 문장과 연결되지 않거나 confidence가 높은데 missing evidence가 있는 경우가 더 위험합니다. 이 실패는 API 레벨에서는 성공처럼 보이기 때문에 운영에서 늦게 발견됩니다. 따라서 prompt contract는 형식 검증, 의미 검증, repair policy를 함께 포함해야 합니다. schema validation은 첫 번째 관문이고, field 간 일관성, citation coverage, unsupported claim 여부, tool approval 상태까지 검증해야 프로덕션 프롬프팅이라고 부를 수 있습니다.
