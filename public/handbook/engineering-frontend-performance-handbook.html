번들링·렌더링·메모이제이션

DOC : FE-RENDERING-V1SCOPE : BUNDLE · REACT RENDER · MEMO · VIRTUALIZATIONLANG : KO

번들링·렌더링·메모이제이션
핸드북

성능 최적화는 덜 보내고, 덜 실행하고, 덜 다시 그리는 것입니다. 하지만 무엇을 줄일지는 측정 후 결정해야 합니다.

STACK : Vite · React · TypeScript · Browser Performance
RO-01

최적화의 순서

최적화는 기법 목록이 아니라 병목을 찾는 절차입니다. 번들이 큰데 memo를 넣거나, 렌더링이 병목인데 CDN을 바꾸면 효과가 없습니다.

질문확인 도구대표 조치
초기 로드가 느린가?Network, Lighthousecode splitting, 이미지 최적화, font 전략
클릭 후 반응이 느린가?Performance, React Profiler렌더 경계 조정, 긴 작업 분할
스크롤이 버벅이나?Performance FPS, Layersvirtualization, layout thrash 제거
상태 변경마다 전체가 다시 그려지나?React Profilerstate 위치 조정, memo 검토
ORDER

대부분의 제품은 초기에는 번들/이미지/네트워크가, 기능이 복잡해지면 렌더링 경계와 상태 구조가 병목이 됩니다. 최적화 대상도 제품 단계에 따라 달라집니다.

RO-02

번들링과 code splitting

번들러는 여러 모듈을 브라우저가 받을 수 있는 산출물로 묶습니다. Vite는 개발 중 ESM 기반으로 빠르게 제공하고, 빌드 시 Rollup 계열 최적화를 수행합니다.

전략사용 기준주의
route splitting페이지별 코드 분리첫 진입 시 loading 상태 필요
component lazy무거운 차트, 에디터, 모달자주 쓰는 작은 컴포넌트는 분리 이득 작음
vendor split큰 라이브러리 캐시 안정화chunk 수가 너무 늘면 요청 관리 비용 증가
tree shaking안 쓰는 export 제거side effect 있는 모듈은 제거 어려움

code splitting은 JS 크기만 줄이는 작업이 아닙니다. 새 route 진입 시 JS chunk, CSS chunk, 데이터 요청이 waterfall을 만들 수 있으므로 preload/modulepreload, route prefetch, shared chunk 중복, 캐시 무효화 단위를 함께 봅니다. 분리 후에는 treemap과 Network waterfall로 실제 초기 실행 비용과 전환 지연을 비교합니다.

SPLIT 기준첫 화면에 필요 없음
크기가 큼
사용 빈도가 낮음
기능 경계가 명확함
loading fallback이 UX를 해치지 않음
RO-03

React 렌더링 모델

React에서 렌더링은 컴포넌트 함수를 다시 실행해 UI 설명을 새로 만드는 과정입니다. DOM 업데이트와 같은 말이 아닙니다.

개념의미실무 영향
render컴포넌트 함수 실행비싼 계산을 render 안에 두면 반복 비용 증가
reconciliation이전 UI 설명과 새 설명 비교key가 불안정하면 목록 상태가 꼬임
commit실제 DOM 반영layout effect는 paint를 막을 수 있음
state 위치어디에 state를 두는가상위 state 변경은 하위 render 범위를 키움
KEY

목록의 key는 index가 아니라 항목의 안정적인 id여야 합니다. 정렬/삭제/삽입이 있는 목록에서 index key는 상태를 잘못된 행에 붙일 수 있습니다.

RO-04

메모이제이션 판단

memoization은 공짜가 아닙니다. 비교 비용, 메모리, 코드 복잡도를 추가합니다.

도구좋은 사용나쁜 사용
React.memoprops가 같을 때 비싼 child render 차단모든 컴포넌트에 습관적으로 감싸기
useMemo비싼 계산 결과 캐시간단한 map/filter에 과도 적용
useCallbackmemoized child에 안정 함수 전달dependency 관리만 복잡하게 만듦
useTransition긴 UI 업데이트를 낮은 우선순위로네트워크 지연 자체를 해결한다고 착각

React.memo는 부모가 넘긴 props의 얕은 비교에만 관여합니다. memoized 컴포넌트도 자신의 state가 바뀌거나 구독한 context가 바뀌면 다시 렌더링됩니다. 객체·배열·함수 props를 매 렌더마다 새로 만들면 memo 효과가 사라지므로 props 형태와 context 경계를 먼저 줄입니다.

useTransition은 상태 업데이트를 non-urgent로 표시해 입력 같은 urgent update가 먼저 반영될 수 있게 합니다. 무거운 동기 계산이나 큰 DOM 작업을 자동으로 빠르게 만들지는 않으므로 작업 분할, virtualization, worker 이동과 함께 판단합니다.

ANTI-PATTERN

"렌더링이 많다"는 것만으로 문제는 아닙니다. 느린 render가 문제입니다. Profiler에서 실제 시간을 보고, state 위치를 먼저 조정한 뒤 memo를 적용합니다.

RO-05

큰 목록과 무거운 UI

수천 행 테이블, 차트, 에디터는 일반 컴포넌트와 다른 전략이 필요합니다.

문제해법주의
큰 목록 DOMvirtualization동적 높이와 keyboard navigation 고려
무거운 차트lazy load, data downsample초기 dashboard 전체를 막지 않음
입력 중 필터debounce, deferred value입력값 자체를 늦게 반영하지 않기
큰 계산worker, incremental processing직렬화 비용과 취소 처리 고려

성능과 지표

성능과 지표
핸드북

성능은 개발자의 느낌이 아니라 사용자가 체감하는 시간입니다. Lighthouse 점수는 출발점이고, 실제 판단은 사용자 환경의 field data와 제품 목표를 함께 봐야 합니다.

TOOLS : Lighthouse · Chrome UX Report · Web Vitals · Performance API
PM-01

성능은 사용자 경험이다

빠른 앱은 단순히 로드가 빠른 앱이 아닙니다. 빨리 보이고, 빨리 반응하고, 화면이 흔들리지 않아야 합니다.

사용자 질문지표대표 원인
뭔가 보이나?FCP, LCPHTML/CSS/이미지/폰트/서버 응답
누르면 반응하나?INP, TBT긴 JS 작업, event handler, hydration, 무거운 render/paint
화면이 밀리지 않나?CLS이미지 크기 미예약, 광고, 폰트 교체
전체적으로 쾌적한가?RUM, error rate, route latency기기/네트워크 차이, API 품질
PM-02

Core Web Vitals

Core Web Vitals는 사용자 체감 성능을 대표하는 세 지표입니다. 제품 성능 대화의 공통 언어로 삼기 좋습니다.

지표의미개선 방향
LCP가장 큰 주요 콘텐츠가 보이는 시간hero 이미지 최적화, 서버 응답, CSS/폰트 blocking 제거
INPclick/tap/keyboard 상호작용부터 다음 paint까지의 지연input delay, event handler, render/commit, style/layout/paint를 나눠 개선
CLS예상치 못한 layout shift이미지/광고 크기 예약, font-display, skeleton 크기 고정

INP는 네트워크 요청 완료나 비동기 UI 변경 전체 시간을 직접 재는 지표가 아닙니다. 사용자가 입력한 시점부터 브라우저가 다음 frame을 paint할 수 있는 시점까지의 상호작용 지연을 봅니다.

PERCENTILE

Core Web Vitals의 공식 평가는 모바일과 데스크톱을 나눠 p75 기준으로 봅니다. p95는 내부 SLO와 꼬리 지연 감시에 별도로 사용하되, CWV 통과 기준과 섞지 않습니다.

PM-03

Lighthouse 읽는 법

Lighthouse는 자동 진단 도구입니다. 점수 자체보다 어떤 병목을 지목하는지 읽어야 합니다.

Lighthouse Performance 점수는 FCP, Speed Index, LCP, TBT, CLS 같은 lab metric의 가중 평균입니다. Opportunities와 Diagnostics는 점수 원인이 아니라 개선 힌트입니다. INP는 field metric이므로 Lighthouse 기본 lab 실행에서는 직접 측정하지 못하고 TBT를 대리 신호로 봅니다.

섹션해석주의
Performance로드와 반응성 진단lab 환경이라 실제 사용자와 다를 수 있음
Accessibility기본 규칙 검사키보드 흐름과 의미 품질은 수동 확인 필요
Best Practices보안/브라우저 기본 품질낮은 점수 원인을 항목별로 확인
SEO검색 엔진 기본 구조SPA는 route별 metadata 전략 필요
LIGHTHOUSE 해석 순서Opportunities를 비용순으로 정렬
Diagnostics에서 main thread와 JS 비용 확인
Treemap으로 큰 dependency 확인
Before/After를 같은 조건에서 비교
점수보다 사용자 여정 지표와 함께 판단
PM-04

Lab vs Field

Lab data는 재현 가능한 테스트이고, field data는 실제 사용자 경험입니다. 둘 중 하나만 보면 판단이 왜곡됩니다.

구분장점한계
Lab재현 가능, PR/CI 자동화 가능실제 기기/네트워크 다양성 반영 부족
Field실제 사용자 기준원인 분석은 별도 도구 필요, 데이터 누적 시간 필요
Synthetic정해진 경로를 주기적으로 측정예상 경로 밖 문제는 놓침
RUM실사용 이벤트와 성능 연결개인정보와 샘플링 설계 필요
PM-05

개선 우선순위

성능 개선은 가장 큰 사용자 영향과 가장 낮은 구현 위험을 함께 봅니다.

문제먼저 볼 것대표 조치
LCP 느림server response, hero resource이미지 압축, preload, critical CSS
INP 나쁨input delay, event handler, React render/commit, presentation delay작업 분할, handler 단순화, state 경계 조정, rendering 비용 감소
CLS 큼shift source크기 예약, dynamic banner 자리 확보
JS 큼bundle treemapdependency 대체, route split
SCORE CHASING

Lighthouse 100점을 목표로 제품 UX를 희생하면 안 됩니다. 성능 목표는 핵심 사용자 경로별로 세우고, 기능 가치와 함께 판단합니다.

DevTools 사용법
핸드북

DevTools는 문제를 추측에서 측정으로 바꾸는 도구입니다. 좋은 프론트엔드 디버깅은 증상을 재현하고, 레이어를 나누고, 가장 싼 가설부터 제거하는 과정입니다.

TOOLS : Chrome DevTools · React DevTools
DT-01

디버깅 루틴

DevTools를 열기 전에 문제를 한 문장으로 정의합니다. "느리다"가 아니라 "검색어 입력 후 목록 갱신까지 600ms 걸린다"처럼 관찰 가능한 증상이어야 합니다.

DEBUG LOOP재현 조건 고정
증상과 기대 동작 기록
Network로 데이터 문제 배제
Console로 runtime error 확인
Elements로 DOM/CSS 확인
Performance/Profiler로 비용 측정
수정 후 같은 조건에서 비교
HYPOTHESIS

한 번에 여러 것을 고치지 마세요. 하나의 가설을 세우고, DevTools로 근거를 확인하고, 수정 후 같은 조건에서 비교합니다.

DT-02

Elements·Styles

Elements 패널은 DOM과 CSS cascade를 확인하는 곳입니다. UI 깨짐은 대부분 여기서 원인을 찾습니다.

기능사용 상황확인할 것
Computed왜 이 스타일이 적용됐는가최종 계산값, 상속, overridden rule
Box Model간격/크기 문제content, padding, border, margin
Force statehover/focus 상태 확인focus-visible 스타일 누락
Accessibility pane보조기술 이름 확인accessible name, role, ignored 여부
DT-03

Network

Network는 API와 리소스 로딩 문제를 분리하는 첫 관문입니다.

볼 것의미판단
StatusHTTP 결과401/403/404/500을 구분
Waterfall요청 시간 분해DNS, TTFB, download 중 어디가 느린지 확인
Initiator요청을 만든 코드중복 요청의 출처 추적
Disable cache캐시 없는 조건초기 방문 성능 재현
Throttling느린 네트워크 시뮬레이션실사용 환경 검증
CACHE CONFUSION

개발 중 cache 때문에 문제가 사라지거나 생길 수 있습니다. 재현 조건을 명확히 하려면 Disable cache, hard reload, incognito를 구분해서 사용합니다.

DT-04

Performance·Memory

Performance 패널은 main thread의 실제 비용을 보여줍니다. Memory 패널은 누수가 의심될 때 사용합니다.

패널확인대표 문제
Performancelong task, scripting, rendering, painting입력 지연, scroll jank
Performance monitorCPU, JS heap, DOM nodesDOM 증가, heap 증가 추세
Memory snapshot객체 유지 경로unmounted 컴포넌트 참조 유지
Allocation timeline상호작용 중 메모리 할당반복 작업에서 누적되는 객체
성능 진단 루틴기준
CPU throttling고사양 개발 머신 기준을 피하고 중저가 기기 조건으로 재현

Application·Security

Application 패널은 storage, cookie, cache, service worker를 보는 곳입니다. Security 패널은 HTTPS와 인증서 문제를 확인합니다.

영역확인할 것
CookiesHttpOnly, Secure, SameSite, domain/path
Local Storage민감 정보 저장 여부
Service Worker등록 상태, update, cache 영향
Cache Storage오래된 asset이 남아 있는지
Securitymixed content, certificate, TLS 상태
보안/캐시 재현 루틴확인
CSP violationConsole과 Network에서 blocked URI와 directive 확인
Mixed contentSecurity 패널에서 HTTPS 페이지의 HTTP 리소스 차단 확인
Cookie 정책SameSite, HttpOnly, Secure, domain/path가 기대와 맞는지 확인
Service Workerupdate/unregister 후 재현, cache storage 삭제 후 stale cache 여부 확인
StoragelocalStorage/sessionStorage에 token이나 민감 정보가 남는지 확인
