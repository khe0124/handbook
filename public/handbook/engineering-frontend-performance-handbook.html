RO-01

최적화의 순서

최적화는 기법 목록이 아니라 병목을 찾는 절차입니다. 먼저 사용자 증상을 한 문장으로 고정하고, 같은 조건에서 측정한 뒤, 네트워크·JS 실행·React render·DOM commit·layout·paint 중 어디에 시간이 쓰였는지 분리합니다. 번들이 큰데 memo를 넣거나, 렌더링이 병목인데 CDN을 바꾸면 효과가 없습니다.

RO-02

번들링과 code splitting

번들러는 여러 모듈을 브라우저가 받을 수 있는 산출물로 묶습니다. 성능 관점의 핵심은 파일 개수가 아니라 초기 route에 필요한 byte, parse/compile 비용, chunk waterfall, 장기 캐시 안정성입니다. Vite는 개발 중 ESM 기반으로 빠르게 제공하고, 빌드 시 Rollup 계열 최적화를 수행합니다.

전략사용 기준진단 증거주의
route splitting첫 화면에 필요 없는 route 코드를 분리initial JS 감소, route 전환 waterfall전환 시 JS·CSS·data 요청이 직렬화되면 체감이 나빠짐
component lazy차트, 에디터, 지도, 드문 모달처럼 무겁고 사용 빈도가 낮은 UItreemap의 큰 leaf, Coverage unused byte작은 컴포넌트까지 쪼개면 fallback과 네트워크 관리 비용이 커짐
vendor split자주 바뀌지 않는 큰 라이브러리 캐시 안정화chunk hash 변화, duplicate dependencyvendor를 하나로 몰면 첫 로드가 다시 커질 수 있음
tree shaking사용하지 않는 export 제거bundle analyzer, sideEffects 설정namespace import, CJS, side effect 모듈은 제거가 어려움
resource hint곧 필요한 chunk·font·origin의 대기 시간 축소Network priority, request start timepreload 남발은 더 중요한 리소스 우선순위를 밀어냄

code splitting은 JS 크기만 줄이는 작업이 아닙니다. 새 route 진입 시 JS chunk, CSS chunk, 데이터 요청이 waterfall을 만들 수 있으므로 preload/modulepreload, route prefetch, shared chunk 중복, 캐시 무효화 단위를 함께 봅니다. 분리 후에는 treemap과 Network waterfall로 실제 초기 실행 비용과 전환 지연을 비교합니다.

BUNDLE REVIEW CHECKroute별 initial JS와 CSS가 예산 안에 있는가
새 dependency가 같은 기능의 기존 라이브러리와 중복되지 않는가
dynamic import 뒤에 CSS·data waterfall이 생기지 않는가
preload/modulepreload가 실제 LCP 또는 route 전환에 기여하는가
source map과 treemap에서 앱 코드와 vendor 비용을 분리했는가
SPLIT FAILURE

chunk를 쪼갰는데 느려졌다면 대개 네트워크 요청 수가 문제가 아니라 의존 순서가 문제입니다. route component chunk를 받은 뒤 그 안에서 chart chunk와 CSS chunk를 다시 요청하고, 그 다음 data 요청을 시작하면 사용자는 더 오래 기다립니다. 분리 전후의 waterfall을 같이 봐야 합니다.

RO-03

React 렌더링 모델

React에서 렌더링은 컴포넌트 함수를 다시 실행해 다음 UI tree를 계산하는 과정입니다. DOM 업데이트와 같은 말이 아닙니다. 성능 진단은 render phase의 계산 비용, reconciliation의 remount 여부, commit phase의 DOM 변경, 그 뒤 브라우저 layout/paint 비용을 분리해야 정확해집니다.

개념의미진단 질문실무 영향
render phase컴포넌트 함수를 실행해 다음 UI 설명을 계산어떤 props/state/context 변경이 이 컴포넌트를 깨웠는가?비싼 계산, 큰 map/filter, 새 객체 생성이 반복 비용을 만든다
reconciliation이전 tree와 새 tree를 비교해 유지·수정·remount를 결정key가 안정적인가, type이 바뀌어 subtree가 remount되는가?불안정한 key는 state 손실과 불필요한 mount 비용을 만든다
commit phase실제 DOM mutation, ref 연결, layout effect 실행render보다 commit duration이 긴가?DOM 수가 많거나 layout effect가 무거우면 paint를 늦춘다
effect phasepaint 이후 effect 실행과 외부 시스템 동기화effect가 다시 state를 바꿔 render loop를 만드는가?fetch, subscription, timer는 cleanup과 dependency가 품질을 좌우한다
batched update여러 상태 변경을 묶어 한 번의 render로 처리상태 변경 경계가 의도한 사용자 interaction과 맞는가?불필요한 중간 상태 render를 줄이지만, 큰 업데이트 자체를 없애지는 않는다

개발 모드의 Strict Mode는 일부 render와 effect를 재실행해 순수하지 않은 로직을 드러냅니다. 이것을 “React가 두 번 렌더링해서 느리다”로 해석하면 안 됩니다. 원인 추적은 개발 모드에서 편하게 하되, 최종 수치는 production build, 동일한 CPU/network throttling, 같은 데이터 크기에서 다시 재야 합니다.

렌더링 원인흔한 냄새보정 방향
Parent state fan-out검색 input 한 글자에 페이지 전체가 다시 render됨state를 소비 지점으로 내리고, 큰 하위 영역은 경계를 나눈다
Context propagationtheme 외 값까지 하나의 provider value에 묶임context를 목적별로 분리하고, 자주 바뀌는 값은 selector store를 검토한다
Unstable identity매 render마다 새 object, array, function을 props로 전달props 형태를 primitive/id 중심으로 줄이고, 필요한 곳만 memoize한다
Key remount정렬·필터 뒤 입력 focus나 row state가 다른 항목으로 이동index key를 피하고 domain id로 list identity를 고정한다
KEY

목록의 key는 index가 아니라 항목의 안정적인 id여야 합니다. 정렬/삭제/삽입이 있는 목록에서 index key는 상태를 잘못된 행에 붙일 수 있습니다.

RO-04

메모이제이션 판단

memoization은 공짜가 아닙니다. 비교 비용, 메모리, stale closure 위험, dependency 관리 비용을 추가합니다. 좋은 memoization은 “렌더링이 많다”가 아니라 “같은 입력에서 비싼 계산이나 비싼 child render가 반복된다”는 증거가 있을 때 적용합니다.

도구좋은 사용효과가 없는 경우실패 모드
React.memoprops가 안정적인 비싼 child render 차단child가 자체 state나 context 변경으로 다시 render됨custom comparator가 깊은 비교로 더 느려지거나 변경을 놓침
useMemo비싼 sort/group/derive 결과를 같은 입력에서 재사용계산이 싸거나 dependency가 매번 새 객체임dependency 누락으로 stale value가 생김
useCallbackmemoized child나 effect dependency에 안정 함수 전달받는 쪽이 memoized가 아니거나 함수 생성 비용만 걱정하는 경우closure에 오래된 state를 잡거나 dependency 배열이 거짓 안정성을 만듦
useTransition입력 반응성은 urgent로 유지하고 큰 결과 UI를 non-urgent로 낮춤무거운 동기 계산 자체가 main thread를 오래 막는 경우pending UX와 stale 결과 표시가 없으면 사용자가 혼란스러움
useDeferredValue입력값은 즉시 보여주고 큰 목록 필터 결과만 늦춰 반영목록 자체가 너무 크거나 DOM이 과도한 경우이전 결과가 최신처럼 보여 데이터 신뢰를 해침

React.memo는 부모가 넘긴 props의 얕은 비교에만 관여합니다. memoized 컴포넌트도 자신의 state가 바뀌거나 구독한 context가 바뀌면 다시 렌더링됩니다. 객체·배열·함수 props를 매 렌더마다 새로 만들면 memo 효과가 사라지므로 props 형태와 context 경계를 먼저 줄입니다.

useTransition은 상태 업데이트를 non-urgent로 표시해 입력 같은 urgent update가 먼저 반영될 수 있게 합니다. 무거운 동기 계산이나 큰 DOM 작업을 자동으로 빠르게 만들지는 않으므로 작업 분할, virtualization, worker 이동과 함께 판단합니다.

MEMO DECISION GATEProfiler에서 같은 props로 반복 render되는 비싼 컴포넌트인가
props identity를 안정화할 수 있는가
context 변경이 병목이 아닌가
비교 비용보다 render 절감이 큰가
dependency 누락과 stale closure를 테스트로 방어할 수 있는가
적용 후 commit duration 또는 interaction latency가 실제로 줄었는가
MEMO BEFORE/AFTER 예시Before: 검색 input 1회 입력마다 GridPage commit 180ms, Row 1,000개 render, INP p75 340ms
After: draft state를 검색 영역으로 내리고 row renderer만 memo 적용, commit 55ms, visible Row 40개 render, INP p75 170ms
판정: memo 자체보다 state fan-out 축소와 visible render 범위 축소가 핵심 개선입니다.
ANTI-PATTERN

"렌더링이 많다"는 것만으로 문제는 아닙니다. 느린 render가 문제입니다. Profiler에서 실제 시간을 보고, state 위치를 먼저 조정한 뒤 memo를 적용합니다.

RO-05

큰 목록과 무거운 UI

수천 행 테이블, 차트, 에디터는 일반 컴포넌트와 다른 전략이 필요합니다. 큰 UI의 병목은 “React가 느리다”가 아니라 DOM 수, 측정·layout 비용, 데이터 파생 계산, 편집 상태 소유권, 접근성 요구가 동시에 얽힌 문제인 경우가 많습니다.

RO-06

증상별 빠른 판단표

치트시트는 용어장이 아니라 첫 가설을 고르는 도구여야 합니다. 아래 표는 증상에서 바로 확인 도구와 배제 순서로 이어지도록 정리한 빠른 판단표입니다.

PM-02

Core Web Vitals

Core Web Vitals는 사용자 체감 성능을 대표하는 세 지표입니다. 제품 성능 대화의 공통 언어로 삼기 좋지만, 지표 이름만 외우면 개선이 불가능합니다. 각 지표의 후보 요소, attribution, percentile, route segment를 같이 봐야 합니다.

PM-03

Lighthouse는 자동 진단 도구입니다. 점수 자체보다 어떤 병목을 지목하는지 읽어야 합니다. Lighthouse 결과는 재현 가능한 lab 신호이지 실제 사용자 전체의 판정이 아니므로, PR에서는 같은 조건의 delta를 보고 운영 판단은 field data와 합쳐야 합니다.

Lighthouse Performance 점수는 FCP, Speed Index, LCP, TBT, CLS 같은 lab metric의 가중 평균입니다. Opportunities와 Diagnostics는 점수 원인이 아니라 개선 힌트입니다. INP는 field metric이므로 Lighthouse 기본 lab 실행에서는 직접 측정하지 못하고 TBT를 대리 신호로 봅니다.

PRIVACY

RUM은 디버깅 욕심 때문에 쉽게 과수집됩니다. route는 정규화하고, DOM text와 입력값은 버리며, user id는 직접 보내지 않습니다. 필요한 경우 익명 segment와 request id만 남겨 서버 로그와 제한적으로 연결합니다.

PM-09

자주 봉착하는 성능 문제

프론트엔드 성능 문제는 매번 새롭지 않습니다. 대부분은 비슷한 증상으로 나타나고, 잘못된 첫 가설 때문에 시간이 낭비됩니다. 아래 표는 현장에서 자주 만나는 문제를 증상, 흔한 오진, 확인 증거, 해결 방향으로 정리한 것입니다.

자주 만나는 문제흔한 오진확인 증거해결 방향
첫 화면 LCP가 늦다이미지만 압축하면 해결된다고 봄LCP element, TTFB, resource load delay, render delay서버 응답, CSS/font blocking, hero image priority를 분리해서 개선
검색 input이 버벅인다debounce만 넣으면 된다고 봄INP attribution, handler 시간, React commit, row render 수입력 echo는 즉시 유지하고 결과 목록은 deferred/transition/virtualization으로 분리
대형 table 편집이 느리다React.memo를 전체에 감싸면 된다고 봄Profiler commit, render된 row/cell 수, draft state 범위server data와 draft state를 분리하고 visible row만 렌더링
route 전환이 느리다code splitting을 더 잘게 쪼갬chunk, CSS, data request waterfallprefetch, data loader 병렬화, shared chunk, fallback 크기를 함께 조정
스크롤이 끊긴다JS가 많아서만 그렇다고 봄Frames lane, Layout/Paint 비용, DOM node 수, layer 수DOM 수 축소, read/write batching, paint 영역 축소, passive listener 적용
배포 후 특정 사용자만 느리다개발 환경에서 빠르니 문제없다고 봄RUM segment: device class, connection, region, release저사양 기기와 느린 네트워크 기준의 p75/p95를 별도 SLO로 둔다
메모리 사용량이 계속 오른다브라우저 GC가 늦는 것이라고 넘김반복 시나리오 후 heap, DOM nodes, detached node, retaining pathsubscription cleanup, timer 정리, cache 상한, modal/portal unmount 확인

2. “느림”을 load, interaction, scroll, memory, route transition 중 하나로 분류한다

React DevTools는 컴포넌트 트리, props, state, render 비용을 확인하는 도구입니다. Performance 패널이 “main thread에서 오래 걸린 일”을 보여준다면 React Profiler는 “어떤 컴포넌트 commit이 그 비용에 기여했는지”를 좁히는 도구입니다.

기능사용주의판정
Componentsprops/state/context 확인실제 DOM 구조와 다를 수 있음상태 소유권과 provider 범위가 의도와 맞는지 확인
Profiler flamegraphcommit별 render 비용 측정개발 모드 비용은 운영과 다를 수 있음가장 비싼 commit과 컴포넌트 subtree를 찾음
Ranked view비용이 큰 컴포넌트 순서 확인작은 컴포넌트 다수보다 큰 컴포넌트 하나가 더 중요할 수 있음최적화 후보를 비용순으로 좁힘
Highlight updates렌더 범위 시각화많이 렌더된다고 항상 문제는 아님fan-out이 큰지 직관적으로 확인
Why did this renderprops/context 변경 원인 추적도구나 설정 필요할 수 있음새 객체 props, context 변경, hook state 변경을 구분
REACT PROFILER ROUTINE1. 느린 사용자 interaction만 녹화한다
2. 가장 긴 commit을 선택한다
3. render된 컴포넌트 중 비용 상위와 넓게 퍼진 subtree를 나눈다
4. props/state/context/key 중 render 원인을 기록한다
5. state 이동, context 분리, props identity 안정화, virtualization 중 하나만 적용한다
6. 같은 interaction으로 commit duration과 실제 INP 후보를 다시 비교한다
Profiler 숫자좋은 해석위험 해석
commit 8-20ms대부분의 일반 interaction에서 부담이 작음짧은 commit이 아주 자주 반복되면 fan-out 확인
commit 50ms 이상입력 interaction에서는 개선 후보INP p75 200ms 초과와 함께 보이면 우선순위 상승
commit 150ms 이상큰 route 전환이나 초기 hydration이 아니라면 위험state 범위, context, virtualization, heavy child를 즉시 분해
PROFILER LIMIT

React Profiler는 browser layout과 paint 비용을 모두 설명하지 않습니다. commit이 짧은데 INP가 나쁘면 Chrome Performance에서 style recalculation, layout, paint, composite를 다시 봐야 합니다.

DT-07

Performance trace 해석

Performance trace는 “느린 것 같다”를 시간축 위의 증거로 바꿉니다. 핵심은 긴 막대를 찾는 것이 아니라 사용자 상호작용부터 다음 paint까지 어떤 단계가 시간을 썼는지 연결하는 것입니다.

Trace 영역읽는 방법자주 보이는 원인다음 도구
NetworkHTML, JS, CSS, image waterfall과 우선순위 확인render-blocking CSS, 늦은 hero image, chunk waterfallCoverage, Lighthouse, bundle analyzer
Mainscripting, rendering, painting, long task 확인큰 JSON parsing, 무거운 event handler, hydrationReact Profiler, source map
InteractionsINP 후보 interaction과 presentation delay 확인입력 후 render/paint 지연Web Vitals attribution
Framesframe drop과 FPS 저하 확인scroll handler, layout thrash, paint stormLayers, Rendering panel
Experiencelayout shift, long animation frame, responsiveness 신호 확인CLS source, LoAF, slow interactionWeb Vitals overlay, screenshots
Memoryheap과 DOM node 증가 추세 확인subscription 미해제, detached DOMHeap snapshot, allocation timeline
TRACE READING ORDER1. CPU throttling과 network 조건 기록
2. 사용자 action marker 또는 interaction 구간 선택
3. 가장 긴 long task와 다음 paint 지점 확인
4. scripting/rendering/painting 비중을 분리
5. call stack에서 앱 코드와 라이브러리 코드를 구분
6. React Profiler commit과 같은 시간대인지 대조
trace 신호해석다음 질문
Evaluate Script가 김큰 JS parse/compile 또는 초기 실행 비용initial bundle과 third-party script가 예산을 넘는가?
Function Call 아래 앱 코드가 김handler 또는 동기 계산이 main thread를 막음작업을 쪼개거나 worker로 옮길 수 있는가?
Recalculate Style/Layout 반복DOM read/write 섞임 또는 DOM 수 과다batching, containment, DOM 축소가 가능한가?
Paint가 큼넓은 영역 repaint 또는 비싼 시각 효과paint 영역, box-shadow/filter, layer 전략을 줄일 수 있는가?
Composite Layers가 큼layer 수 또는 texture memory 부담will-change나 fixed/sticky 요소를 남발했는가?
TRACE 숫자 예시Interaction: grid cell 입력
Total interaction: 620ms
Input delay: 40ms · handler/script: 180ms · React commit: 210ms · style/layout: 120ms · paint: 70ms
해석: handler만 줄여서는 부족합니다. React commit과 layout 비용이 함께 크므로 state fan-out, DOM 수, layout effect를 같이 줄입니다.
LAYOUT THRASH

반복문 안에서 DOM 읽기와 쓰기를 섞으면 브라우저가 layout을 여러 번 강제 계산합니다. trace에서 Recalculate Style과 Layout이 짧게 여러 번 반복되면 read/write batching, CSS containment, DOM 수 축소를 검토합니다.

DT-08

회귀 재현 패킷

성능 이슈는 “느립니다”로 공유하면 해결이 늦어집니다. 누가 봐도 같은 조건에서 재현하고 비교할 수 있는 패킷으로 남겨야 합니다.

항목포함할 내용이유
환경브라우저, OS, device class, CPU/network throttling, build mode개발자 고사양 환경 착시와 dev/prod 차이 제거
시나리오route, 로그인 상태, 데이터 크기, 정확한 사용자 행동상호작용 비용은 데이터와 권한에 따라 달라짐
증거Performance trace, React Profiler export, Network HAR, screenshot, RUM link원인 후보를 계층별로 검증 가능
비교이전 release, 현재 release, 수정 branch의 같은 조건 수치개선 여부를 감각이 아니라 delta로 판단
판정사용자 영향, 회귀 범위, 임시 완화, rollback 기준기술 이슈를 제품 위험으로 번역
재발 방지bundle budget, RUM alert, synthetic scenario, regression test같은 성능 회귀가 다음 배포에서 반복되지 않게 함
PERFORMANCE REGRESSION PACKETIssue: 어떤 사용자 행동이 얼마나 느려졌는가
Scope: route, device, release, traffic segment
Evidence: trace/HAR/profiler/RUM link
Root cause: network | JS | render | layout | paint | API
Fix options: 빠른 완화와 구조 개선을 분리
Verification: before/after 수치와 재발 방지 예산
CLOSE CRITERIA

성능 이슈는 코드가 merge됐다고 닫지 않습니다. 같은 조건의 before/after trace, field 지표 회복 여부, 남은 예외와 재발 방지 장치가 있어야 닫을 수 있습니다.

DT-09

AI 성능 진단 방법론

AI는 성능 문제를 대신 측정하지 못합니다. 하지만 재현 조건, trace 요약, 코드 경로, 가설 목록, 검증 계획을 구조화하는 데 매우 유용합니다. AI를 잘 쓰는 방법은 “느린데 고쳐줘”가 아니라 “이 증거로 가능한 원인을 좁히고, 가장 작은 실험을 설계해줘”라고 요청하는 것입니다.

AI가 잘하는 일입력으로 줘야 할 것기대 산출물주의
증상 분류사용자 행동, route, 기기 조건, p75/p95 지표load, interaction, render, layout, memory 중 분류숫자 없이 “느림”만 주면 일반론으로 흐름
trace 해석 보조Performance summary, long task 목록, 주요 call stack원인 후보와 추가 확인 질문trace 원본을 직접 본 것처럼 단정하게 두면 안 됨
React 원인 추정Profiler commit, render된 컴포넌트, props/state/context 변화state fan-out, context invalidation, key remount 후보memo를 자동 처방하게 두지 말고 측정 기준을 요구
코드 변경안 작성관련 컴포넌트, 상태 소유권, 데이터 크기, UX 제약작은 diff, 위험, rollback 기준동작 변경과 접근성 회귀를 반드시 검토
회귀 방지 설계before/after 수치, 실패 원인, 배포 경로budget, RUM alert, synthetic scenario, PR checklist테스트 불가능한 “주의하자” 문구로 끝내지 않음
AI PERFORMANCE LOOP1. 재현 패킷을 만든다: route, action, data size, device, network, build mode
2. 숫자를 붙인다: LCP/INP/CLS, commit duration, long task, DOM node, heap
3. AI에게 원인 후보를 계층별로 분류하게 한다: network | JS | React | layout | paint | memory
4. 후보마다 확인할 DevTools 증거를 요구한다
5. 가장 작은 코드 변경 실험 하나만 만든다
6. before/after 측정 명령과 판정 기준을 함께 생성한다
7. 개선이 확인되면 재발 방지 항목을 PR에 남긴다
좋은 프롬프트왜 좋은가
“/grid에서 cell 입력 INP p75가 340ms, p95가 910ms입니다. React Profiler에서 GridPage commit 180ms, Row 1,000개 render가 보입니다. state fan-out, context, key, DOM 수 중 어떤 순서로 확인할지 가설과 DevTools 증거를 표로 정리해줘.”증상, 숫자, 도구 증거, 후보 범위를 함께 제공해 일반론을 줄입니다.
“이 trace 요약에서 handler 180ms, commit 210ms, layout 120ms, paint 70ms입니다. handler만 최적화했을 때 남는 병목과 다음 실험 3개를 작은 변경 순서로 제안해줘.”AI가 한 가지 최적화에 과몰입하지 않고 남는 비용을 계산하게 합니다.
“아래 컴포넌트에서 검색 input 변경이 table 전체 render를 유발합니다. UX는 input echo 100ms 이하, 결과 목록 stale 표시 필요입니다. state 위치 변경안과 검증 기준을 제안해줘.”성능 목표와 UX 제약을 같이 줘서 무리한 debounce 처방을 막습니다.
나쁜 프롬프트문제보정
“이 페이지 느린데 최적화해줘.”재현 조건과 병목 계층이 없어 추측 답변이 나옴route, action, device, p75/p95, trace 요약을 붙임

“Lighthouse 100점 만들 방법 알려줘.”제품 UX보다 점수 추격으로 흐름핵심 route의 LCP/INP/CLS budget과 사용자 영향 기준을 제시

AI 사용 한계

AI가 만든 성능 결론은 측정으로 검증하기 전까지 가설입니다. 특히 browser trace, Profiler export, RUM 데이터는 누락된 맥락이 많기 때문에 AI 답변을 그대로 merge하지 말고 반드시 같은 조건에서 before/after를 비교합니다.
