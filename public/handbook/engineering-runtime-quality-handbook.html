비동기 처리와
메시지 큐 핸드북

동기 호출은 두 서비스를 시간으로 묶는다 — 큐는 그 결합을 끊어 시간과 장애를 분리한다. 이 핸드북은 명령어가 아니라 사고 모델을 다룹니다: 왜 비동기로 가는가, 무엇을 포기하는 대가인가, 그리고 어디서 깨지는가.

MSG-01

동기 vs 비동기

핵심 질문은 "빠르냐 느리냐"가 아니라 "호출자가 결과를 기다리느냐"입니다. 동기 호출은 두 서비스를 같은 시간선에 묶습니다 — 한쪽이 느리거나 죽으면 다른 쪽도 그만큼 멈춥니다. 비동기는 그 시간 결합(temporal coupling)을 끊는 도구입니다.

TEMPORAL COUPLING — 동기 호출의 사슬
[ 주문 서비스 ] ──요청──▶ [ 결제 ] ──요청──▶ [ 알림 ]
◀──────── 응답 대기(blocked) ────────────────┘
└ 알림이 2초 느리면 주문 응답도 2초 느려진다 — 지연·장애가 전파

ASYNC — 큐로 시간선을 끊는다 [ 주문 서비스 ] ──발행──▶ [ QUEUE ] ──▶ [ 결제 ] [ 알림 ]
└ 발행 즉시 응답 반환. 소비자 장애는 큐에 쌓일 뿐 호출자엔 안 보임
관점동기 (sync / request-response)비동기 (async / messaging)
시간 결합강함 — 둘이 동시에 살아 있어야 함약함 — 소비자는 나중에 처리해도 됨
장애 전파하류 장애가 상류로 즉시 전파큐가 완충 — 장애가 격리됨
응답결과를 바로 받음(정합성 즉시)접수만 확인, 결과는 최종 일관성(→ MSG-10)
피크 흡수못 함 — 부하가 그대로 하류로큐가 버퍼로 부하를 평탄화
복잡도낮음 — 흐름이 직선적높음 — 중복·순서·실패를 직접 다뤄야(→ MSG-04~08)
언제 비동기로 분리하나

비동기는 공짜가 아닙니다 — 디버깅이 어려워지고, 중복·순서·최종 일관성을 직접 책임져야 합니다. 호출자가 결과를 즉시 알아야 하면 동기를 유지하세요. 비동기가 정당한 신호: ① 작업이 느리고 호출자가 기다릴 이유가 없다(이메일 발송, 리포트 생성) ② 트래픽 피크를 완충해야 한다 ③ 한 이벤트에 여러 소비자가 독립적으로 반응한다(fan-out) ④ 하류 장애가 상류를 멈추면 안 된다. "MSA니까 다 큐"는 안티패턴입니다.

MSG-02

비동기 실행 모델

메시지 큐로 가기 전에, 한 프로세스 안의 비동기부터 짚습니다. "비동기"는 한 단어지만 구현 모델이 다릅니다 — 스레드를 늘리는 방식(Java 전통)과 스레드를 안 막는 방식(이벤트 루프)은 출발점이 정반대입니다.

2.1블로킹 vs 논블로킹 I/O

블로킹 I/O는 DB·네트워크 응답을 기다리는 동안 스레드가 멈춰 있습니다. 그 스레드는 CPU를 안 쓰면서도 점유돼 있어, 동시 요청 수만큼 스레드가 필요합니다. 스레드는 비싼 자원(스택 메모리 ~1MB, 컨텍스트 스위칭 비용)이라 수천 개를 띄우면 한계가 옵니다. 논블로킹 I/O는 "준비되면 알려줘"라고 등록만 하고 스레드를 반납합니다.

모델동시성 단위강점 / 약점
Thread-per-request (전통 Spring MVC)요청당 OS 스레드 (thread pool)코드가 직관적(순차 읽힘) / 스레드 수가 동시성 상한, I/O 대기에 스레드 낭비
Event loop (Node.js, Netty)단일 루프 + 콜백/Promise적은 스레드로 많은 연결 / CPU 무거운 작업이 루프를 막으면 전체 정지
Reactive (Spring WebFlux)논블로킹 + 백프레셔(→ MSG-09)고동시성·자원 효율 / 학습곡선 가파름, 디버깅·스택트레이스 난해
Virtual threads (Java 21+)경량 스레드 (JVM이 스케줄링)블로킹 코드를 그대로 쓰면서 수백만 동시성 / 비교적 신기능, 일부 라이브러리 호환 주의

2.2Java에서의 비동기 — @Async와 CompletableFuture

Spring의 @Async는 메서드를 별도 스레드 풀에서 실행시켜 호출자를 즉시 반환시킵니다 — 가장 단순한 "발사 후 망각(fire-and-forget)". 결과를 조합해야 하면 CompletableFuture로 여러 비동기 작업을 합성합니다(thenCompose로 연결, thenCombine으로 병합, exceptionally로 실패 처리). 이건 어디까지나 한 JVM 안의 비동기입니다 — 프로세스가 죽으면 진행 중이던 작업도 사라집니다. 프로세스 경계를 넘어 내구성(durability)이 필요한 순간이 메시지 큐가 등장하는 지점입니다.

in-process async ≠ message queue

@Async/CompletableFuture는 같은 프로세스 메모리 안에서만 삽니다. 서버를 재시작하거나 인스턴스가 죽으면 큐에 없던 작업은 증발합니다. "재시도", "여러 인스턴스가 나눠 처리", "장애 후 재개"가 필요하면 in-process로는 부족합니다 — 메시지를 프로세스 밖 내구성 있는 저장소(브로커)에 맡겨야 합니다(→ MSG-03). 가상 스레드(virtual thread)는 동시성 모델을 바꿀 뿐, 이 내구성 문제를 해결하지는 않습니다.

MSG-03

메시지 큐 vs 이벤트 스트림

"카프카랑 래빗엠큐 뭐 써요?"의 진짜 질문은 "메시지가 소비되면 사라지길 원하나, 로그로 남아 여러 번 읽히길 원하나"입니다. 둘은 비슷해 보이지만 데이터 수명 모델이 근본적으로 다릅니다.

WORK QUEUE (RabbitMQ) — 소비하면 사라진다
producer ──▶ [ msg msg msg ] ──▶ consumer A (ack 후 삭제)
└──────────▶ consumer B ← 같은 메시지는 둘 중 하나만 받음(경쟁 소비)

EVENT LOG (Kafka) — 쌓이고, 각자 오프셋으로 읽는다 producer ──▶ [ 0 1 2 3 4 5 6 ... ] (append-only log)
▲offset=3 ▲offset=6
        group A 여기 읽는 중   group B 여기 (서로 독립)
└ 읽어도 안 사라짐. 보존기간 동안 재생(replay) 가능
관점메시지 큐 (RabbitMQ, SQS)이벤트 스트림 (Kafka, Kinesis)
모델work queue — 작업 분배append-only log — 사건 기록
소비 후ack하면 큐에서 제거남아 있음 — 오프셋만 전진
여러 소비자경쟁 소비(한 메시지=한 소비자)consumer group마다 전체를 독립적으로 읽음(fan-out)
재생(replay)기본적으로 불가(소비 후 사라짐)가능 — 오프셋을 되감으면 과거 재처리
순서큐 단위, 보장 제한적파티션 단위로 보장(→ MSG-06)
적합한 일작업 분산, RPC 대체, 태스크 큐이벤트 소싱, 로그 집계, 여러 소비자 fan-out, 스트림 처리
선택 기준

"작업을 누군가 한 번 처리하면 끝"이면 메시지 큐(태스크 분배). "사건이 일어났고, 누가 어떻게 반응할지는 소비자가 각자 결정"이면 이벤트 스트림(fan-out·재생). 정산 플랫폼 맥락이라면: 인증서 발급 PDF 생성 같은 단발 작업은 큐, "정산 검증 완료" 같은 도메인 이벤트를 정산·알림·감사로그가 각자 받아 처리하면 스트림이 맞습니다. 둘을 한 시스템에서 섞어 쓰는 것도 정상입니다(→ 아키텍처 패턴 BE-12).

MSG-04

전달 보장 — delivery semantics

분산 메시징의 가장 중요한 진실: 네트워크는 신뢰할 수 없고, ack는 유실될 수 있습니다. 그래서 "정확히 한 번"은 대부분 환상이고, 현실의 표준은 at-least-once + 멱등 소비자입니다.

보장의미대가 / 실패 모드
at-most-once많아야 한 번 — 받기 전에 ack, 처리 실패해도 재전송 안 함메시지 유실 가능. 잃어도 되는 메트릭·로그에만
at-least-once적어도 한 번 — 처리 후 ack, ack 유실 시 재전송중복 발생. 가장 흔한 현실 선택 — 단, 소비자가 멱등해야(→ MSG-05)
exactly-once정확히 한 번분산 환경에서 일반적으로는 불가능. 특정 조건에서만 근사
왜 exactly-once가 환상인가 — ack가 유실되는 순간
consumer ──처리완료──▶ DB commit OK
consumer ──ack 전송──▶ ✗ 네트워크 단절 (브로커는 ack를 못 받음)
broker ──타임아웃──▶ 같은 메시지 재전송
└ 소비자 입장: 이미 처리한 메시지가 또 옴 → 중복. 막을 방법은 멱등성뿐
exactly-once는 마케팅 용어에 가깝다

Kafka의 "exactly-once semantics(EOS)"는 Kafka 내부에서 읽기→처리→쓰기가 모두 Kafka일 때(트랜잭션 + idempotent producer)만 성립합니다. 메시지를 받아 외부 DB나 외부 API를 건드리는 순간 그 보장은 깨집니다 — 외부 시스템은 Kafka 트랜잭션 밖이니까요. 그래서 실무 설계 원칙은 단순합니다: "at-least-once로 전달되고 중복이 온다"를 가정하고, 소비자를 멱등하게 만들어 effectively-once를 달성한다. 중복을 막으려 하지 말고 중복에 무해하게 만드세요.

MSG-05

멱등 소비와 중복 처리

멱등성(idempotency)은 "같은 메시지를 몇 번 처리해도 결과가 한 번 처리한 것과 같다"입니다. at-least-once 세계에서 중복은 버그가 아니라 일상이라, 소비자는 중복을 견디도록 설계되어야 합니다.

5.1멱등성 키와 dedup

각 메시지에 안정적인 고유 키(메시지 ID, 또는 "주문 12345의 결제"처럼 비즈니스적으로 유일한 식별자)를 부여하고, 소비자는 처리 전에 "이 키를 이미 봤나?"를 검사합니다. 핵심은 키가 재전송돼도 같아야 한다는 것 — 그래야 중복을 인식합니다.

메시지 수신→dedup 테이블에 key 있나?→ 있음skip (이미 처리)
↓ 없음비즈니스 처리 + key INSERT→같은 트랜잭션 commit→ack

5.2두 가지 구현 — dedup 테이블 vs 유니크 제약

방식작동주의
dedup 테이블 + "이미 처리했나" 검사처리 기록을 별도 테이블에 남기고 매번 조회검사와 처리가 같은 트랜잭션이어야 함. 아니면 검사~처리 사이 동시 중복이 통과(race)
유니크 제약(unique constraint)멱등성 키에 DB 유니크 인덱스 → 중복 INSERT는 DB가 거부가장 견고. 동시성도 DB가 막아줌(→ DB BE-03). 제약 위반 예외를 "중복=정상"으로 처리
멱등성은 자연적으로 얻을 수도 있다

모든 처리에 dedup이 필요한 건 아닙니다. 본질적으로 멱등한 연산이면 키 없이도 안전합니다 — "잔액을 100으로 설정"(SET)은 몇 번 해도 100이지만, "잔액을 10 증가"(INCREMENT)는 중복 시 망가집니다. 가능하면 연산 자체를 멱등하게(upsert, set 기반) 설계하는 게 dedup 인프라보다 단순합니다. 그게 안 되는 증분·금액 처리에만 멱등성 키를 도입하세요. PII나 키 값을 로그에 그대로 흘리지 않도록 주의(→ 보안).

MSG-06

순서와 파티셔닝

"메시지가 보낸 순서대로 처리되나?" — 분산 큐에서 전역 순서(total order)는 비싸고, 대부분의 시스템은 "관련된 것끼리만" 순서를 보장하는 타협을 택합니다. Kafka의 파티션이 그 도구입니다.

PARTITION KEY로 순서 범위를 정한다
key="account-7" ─▶ hash ─▶ partition 1 : [ e1 e2 e3 ] ← 같은 키는 항상 같은 파티션 = 순서 보장
key="account-9" ─▶ hash ─▶ partition 2 : [ e4 e5 ]
파티션 1과 2 사이의 상대 순서는 보장 안 됨 — 하지만 한 계정 안에서는 보장됨
순서 범위방법대가
파티션 내 순서같은 엔티티는 같은 파티션 키(예: accountId)저렴 — 키별로 병렬 처리 가능. 대부분 이걸로 충분
전역 순서파티션 1개 + 소비자 1개병렬성 0 — 처리량이 단일 소비자 속도로 묶임. 사실상 확장 포기
"순서가 정말 필요한가"를 먼저 물어라

전역 순서를 요구하면 병렬성을 통째로 포기하게 됩니다. 보통 진짜 필요한 건 "같은 엔티티에 대한 이벤트만 순서대로"입니다 — 계정 A의 입출금은 순서가 중요하지만, 계정 A와 계정 B 사이는 무관합니다. 파티션 키를 엔티티 ID로 잡으면 키별 순서 + 키 간 병렬을 동시에 얻습니다. 또 하나의 함정: 파티션 수를 늘리면 같은 키가 다른 파티션으로 갈 수 있어 그 시점 전후 순서가 흔들립니다 — 파티션 수는 신중히 정하세요. 멱등 소비(→ MSG-05)와 결합하면 "순서가 약간 어긋나도 결과가 같은" 설계가 더 견고합니다.

MSG-07

이중 쓰기 문제와 Outbox 패턴

비동기 설계의 핵심 함정 하나: "DB에 저장하고, 그다음 메시지를 발행한다"는 두 줄의 코드는 원자적이지 않습니다. 둘 사이에서 프로세스가 죽으면 DB는 바뀌었는데 메시지는 안 나갔거나, 그 반대가 됩니다.

DUAL WRITE — 두 시스템에 쓰는데 트랜잭션이 하나가 아니다
db.save(order) → commit OK
✗ 프로세스 crash
broker.publish(event) → 실행 안 됨
결과: 주문은 DB에 있는데 "주문 생성됨" 이벤트는 영영 안 나감 → 하류가 모름

순서를 뒤집어 "발행 먼저, DB 나중"으로 해도 반대 불일치(이벤트는 나갔는데 DB엔 없음)가 생길 뿐, 근본 문제는 같습니다 — 서로 다른 두 시스템에 걸친 원자성은 분산 트랜잭션(2PC) 없이는 보장 안 됩니다. 2PC는 느리고 깨지기 쉬워 대부분 피합니다.

7.1Outbox 패턴 — 한 트랜잭션으로 묶는다

해법은 영리하게 단순합니다: 메시지를 브로커에 직접 보내지 말고, 같은 DB의 outbox 테이블에 비즈니스 데이터와 함께 한 트랜잭션으로 씁니다. DB 트랜잭션은 원자적이니 "주문 저장 + outbox 기록"은 전부 되거나 전부 안 됩니다. 그다음 별도 프로세스가 outbox를 읽어 브로커로 발행합니다.

TX: order INSERT + outbox INSERT→ commit (원자적)relay가 outbox polling / CDC→broker 발행→outbox 행 표시·삭제
Outbox는 at-least-once를 만든다 — 멱등성과 한 세트

relay가 "발행 성공 → outbox 삭제" 사이에서 죽으면, 재시작 후 같은 행을 또 발행합니다. 즉 Outbox는 중복 발행을 허용합니다 — 의도된 트레이드오프입니다(유실보다 중복이 낫다). 그래서 Outbox는 항상 소비자 멱등성(→ MSG-05)과 함께 가야 완성됩니다. 발행 방식은 두 가지: ① outbox 테이블 polling(단순, 약간의 지연) ② CDC(Change Data Capture, 예: Debezium)로 DB의 트랜잭션 로그를 직접 읽어 발행(지연 적음, 인프라 복잡). 처음엔 polling으로 시작하는 게 합리적입니다(→ DB 트랜잭션 BE-03).

MSG-08

실패 처리 — 재시도·백오프·DLQ

소비자는 실패합니다 — 일시적 실패(DB 잠깐 끊김)와 영구적 실패(메시지 자체가 깨짐)는 대응이 달라야 합니다. 무한 재시도는 시스템을 마비시키고, 재시도 없는 포기는 데이터를 잃습니다.

8.1지수 백오프 + 지터

실패를 즉시·반복 재시도하면 이미 힘든 하류를 더 두드려 장애를 키웁니다(retry storm). 지수 백오프(exponential backoff)는 재시도 간격을 1s → 2s → 4s → 8s로 늘려 압박을 줄입니다. 여기에 지터(jitter, 무작위 흔들기)를 더하는 이유는, 동시에 실패한 수천 소비자가 똑같은 타이밍에 재시도하면 또 한꺼번에 몰리는 thundering herd가 생기기 때문입니다 — 간격을 랜덤하게 분산시켜 충돌을 막습니다.

RETRY LADDER — 횟수 제한 후 격리
attempt 1 fail ─▶ wait ~1s ─▶ attempt 2 fail ─▶ wait ~2s ─▶ attempt 3 fail
                                       ↓ N회 초과
                              [ DLQ ] ← 격리. 메인 흐름은 계속 흐른다

8.2DLQ와 독성 메시지

독성 메시지(poison message)는 몇 번을 재시도해도 항상 실패하는 메시지입니다 — 깨진 페이로드, 처리 불가능한 상태. 이걸 큐 맨 앞에 두고 무한 재시도하면 뒤 메시지가 영영 못 나가는 head-of-line blocking이 생깁니다. DLQ(Dead Letter Queue)는 N회 실패한 메시지를 별도 큐로 격리해, 메인 흐름은 계속 흐르게 하고 실패 건은 나중에 사람이 조사하게 합니다.

실패 유형예대응
일시적(transient)DB 커넥션 끊김, 하류 타임아웃, 락 경합지수 백오프 + 지터로 재시도. 대개 곧 성공
영구적(permanent)스키마 안 맞는 페이로드, 존재하지 않는 참조재시도해도 무의미 → 즉시 DLQ로. 재시도는 자원 낭비
중독(poison)처리 중 항상 같은 예외로 죽음N회 후 DLQ 격리 + 알림. head-of-line blocking 방지
DLQ는 쌓아두는 곳이 아니라 알람이다

DLQ에 메시지가 들어갔는데 아무도 안 보면, 그건 조용한 데이터 유실입니다. DLQ 적재는 반드시 알림·모니터링과 연결하세요(→ 관측가능성). 그리고 재시도 횟수 상한을 두지 않으면 독성 메시지 하나가 컨슈머를 영원히 점유합니다. 운영 체크리스트: ① 최대 재시도 횟수 명시 ② DLQ 적재 시 알람 ③ DLQ 재처리(redrive) 절차 준비 ④ 재시도가 멱등한지 확인(→ MSG-05) — 재시도는 정의상 중복 처리를 만듭니다.

MSG-09

백프레셔와 흐름 제어

생산자가 소비자보다 빠르면 어딘가 메시지가 쌓입니다. 백프레셔(backpressure)는 "느리니까 천천히 보내"라는 역방향 신호입니다 — 이게 없으면 큐 적체나 메모리 폭발로 시스템이 무너집니다.

생산 > 소비 — 어디선가 부하가 새어 나간다
producer 1000/s ──▶ [ QUEUE 무한정 적체 ▓▓▓▓▓▓▓▓ ] ──▶ consumer 200/s
결과 ① 큐 지연 폭증(메시지가 오래된 채 처리됨)
결과 ② 메모리 큐면 OOM, 디스크 큐면 디스크 풀

BACKPRESSURE — 소비자가 속도를 되먹인다 consumer ──"천천히"──▶ producer (요청 거절 / rate limit / 버퍼 제한)
전략작동트레이드오프
버퍼링(buffer)일시적 급증을 큐에 흡수피크엔 좋지만 지속적 초과면 적체만 미룸
스로틀링(throttle / rate limit)생산 속도를 소비 가능 범위로 제한안정적이지만 생산 측 지연 발생
거절(drop / reject / load shedding)한계 초과분을 버리거나 429로 거절가용성 보호 ↔ 일부 요청 손실(잃어도 되는 데이터에)
스케일 아웃소비자(consumer group) 인스턴스 증설근본 해결이지만 비용·파티션 수 상한에 묶임(→ MSG-06)
무한 큐는 문제를 미루는 것이지 푸는 게 아니다

"큐가 있으니 다 받아주면 되지"는 위험한 착각입니다. 소비가 생산을 지속적으로 못 따라가면 큐 길이는 무한히 늘고, 메시지는 점점 오래된 채 처리되며(지연), 결국 자원이 터집니다. 큐는 일시적 급증을 흡수하는 완충재이지 영구적 속도 차이를 메우지 못합니다. 반드시 큐 깊이(queue depth)와 컨슈머 랙(consumer lag)을 모니터링하고(→ 관측가능성), 상한과 백프레셔 정책을 정하세요. 동기 시스템은 자연적 백프레셔가 있지만(호출자가 기다림), 비동기는 직접 만들어야 합니다.

MSG-10

사가 패턴 — 분산 트랜잭션의 대체

여러 서비스에 걸친 작업을 하나의 ACID 트랜잭션으로 묶을 수는 없습니다(서비스마다 DB가 다름). 사가(Saga)는 이를 "로컬 트랜잭션의 연쇄 + 실패 시 보상"으로 풀어, 최종 일관성(eventual consistency)을 달성합니다.

핵심 아이디어: 전체를 롤백하는 대신, 이미 커밋된 단계들을 보상 트랜잭션(compensating transaction)으로 의미적으로 되돌립니다. "결제 취소", "예약 해제"처럼 — 진짜 롤백이 아니라 반대 작업입니다.

SAGA — 성공 경로와 보상 경로
주문생성 ─▶ 재고차감 ─▶ 결제 ─▶ ✗ 배송예약 실패
↓ 보상 시작(역방향)
◀── 재고복원 ◀────── 결제취소 ◀──┘
└ 이미 커밋된 단계를 "반대 작업"으로 되돌린다 — rollback이 아니라 compensate
방식조정 주체강점 / 약점
오케스트레이션(orchestration)중앙 조정자(orchestrator)가 각 단계를 명령흐름이 한곳에 보여 추적·디버깅 쉬움 / 조정자가 단일 의존점, 다소 결합
코레오그래피(choreography)각 서비스가 이벤트를 듣고 자율 반응느슨한 결합·확장 용이 / 전체 흐름이 흩어져 추적 어려움, 순환 의존 위험
사가는 격리성을 포기한다 — ACID의 I가 없다

사가의 중간 상태는 다른 트랜잭션에 보일 수 있습니다. "결제는 됐지만 배송 예약 전"인 순간이 외부에 노출되고, 그 사이 다른 작업이 그 데이터를 읽을 수 있습니다(ACID의 Isolation 상실 → DB BE-03). 그래서 사가엔 추가 설계가 필요합니다: ① 보상 가능한 단계와 불가능한 단계 구분(이메일 발송은 못 되돌림 → 마지막에 배치) ② semantic lock 같은 중간상태 표시 ③ 모든 단계와 보상이 멱등(→ MSG-05). 단순한 흐름엔 사가가 과합니다 — 2~3 서비스면 오케스트레이션, 그 이상의 복잡한 장기 흐름이면 전용 워크플로 엔진(예: Temporal)을 고려하세요(→ 아키텍처 패턴 BE-12).

큐는 시간 결합을 끊어 가용성·확장성·장애 격리를 줍니다. 그 대가로 중복·순서 흐트러짐·최종 일관성을 직접 책임지게 됩니다. 이 핸드북의 거의 모든 패턴은 그 대가를 갚는 도구입니다 — at-least-once를 가정하고(MSG-04), 멱등하게 소비하고(MSG-05), 엔티티 단위 순서만 지키고(MSG-06), 이중 쓰기를 Outbox로 막고(MSG-07), 실패를 백오프·DLQ로 다루고(MSG-08), 과부하를 백프레셔로 제어하고(MSG-09), 분산 작업을 사가로 조율한다(MSG-10). "비동기로 바꾸자"는 결정 전에 MSG-01의 질문으로 돌아가세요 — 정말 시간을 분리해야 하는가(→ 백엔드 개요 BE-07 · 아키텍처 패턴 BE-12 · DB 트랜잭션 BE-03).

TEST-01

테스트의 목적

테스트를 쓰는 이유를 잘못 잡으면 방향이 통째로 틀어집니다. 테스트는 "맞다는 증명"이 아니라 변경 앞에서의 안전망입니다 — 코드를 고칠 때 무엇이 깨졌는지 즉시 알려주는 장치죠.

핵심 가치는 세 가지입니다. 첫째, 변경의 자신감. 테스트가 있으면 리팩토링·라이브러리 업그레이드·기능 추가를 하고도 "기존 동작이 살아있다"를 초 단위로 확인합니다. 테스트가 없으면 모든 변경이 도박이 되고, 결국 아무도 코드를 건드리지 않으려 합니다 — 코드가 굳습니다. 둘째, 회귀 방어. 한 번 고친 버그가 다시 돌아오지 않게 막습니다. 셋째, 실행 가능한 문서. 잘 쓴 테스트는 "이 함수는 이런 입력에 이렇게 반응한다"를 거짓말하지 않고 보여줍니다 — 주석과 달리 낡으면 빨갛게 깨지니까요.

테스트의 가치없을 때 벌어지는 일
변경의 자신감리팩토링·배포가 도박이 됨 → 코드를 아무도 안 건드림 → 부패
회귀 방어고친 버그가 다시 등장, 같은 불을 반복해서 끔
실행 가능한 문서의도가 코드 안에만 있어 신규 합류자가 추측으로 읽음
설계 피드백테스트하기 어려운 구조를 늦게 발견(→ TEST-06)
관점

"커버리지 80%를 채워라"는 목표는 수단을 목적으로 착각한 것입니다. 진짜 질문은 "이 테스트가 있어서 내가 더 과감하게 코드를 고칠 수 있는가"입니다. 그 답이 '아니오'라면 그 테스트는 숫자만 올리는 비용입니다(→ TEST-08 커버리지의 함정).

TEST-02

테스트 피라미드

모든 테스트가 같은 비용·속도·신뢰도를 갖지 않습니다. 피라미드는 어떤 종류를 얼마나 쓸지의 비율 가이드입니다 — 빠르고 싼 단위 테스트를 많이, 느리고 비싼 E2E를 적게.

E2E — 적게 ▲ 느림·비쌈·불안정
통합 — 중간 실제 경계(DB·HTTP) 검증
단위 — 많게 ▼ 빠름·쌈·안정

아래로 갈수록 빠르고, 격리돼 있고, 실패 원인이 명확합니다. 위로 갈수록 실제 사용자 경로에 가까워 신뢰도가 높지만, 느리고 깨지기 쉽고 실패해도 "어디가 문제인지" 짚기 어렵습니다. 그래서 단위를 토대로 두껍게 깔고, 위로 갈수록 얇게 가져갑니다.

종류대상속도신뢰도 / 비용
단위(unit)함수·클래스 하나, 순수 로직ms 단위, 매우 빠름국소적 / 매우 쌈
통합(integration)여러 컴포넌트 + 실제 경계(DB·외부)초 단위경계 신뢰 / 중간
E2E실제 배포 형태의 전체 흐름수십 초~분높지만 불안정 / 비쌈
안티패턴 — 아이스크림콘

피라미드를 뒤집으면 아이스크림콘이 됩니다 — E2E·수동 테스트가 위에 잔뜩 쌓이고 단위 테스트는 바닥에 거의 없는 형태. 이러면 CI가 몇십 분씩 걸리고, 자주 깨지고, 깨져도 원인을 못 찾아 결국 "재실행"으로 넘기게 됩니다(→ TEST-09 flaky test). 느린 E2E로 잡으려던 버그 대부분은 사실 단위·통합에서 더 싸게 잡힙니다.

TEST-03

단위 테스트

단위 테스트의 본령은 순수 로직입니다 — 입력을 받아 출력을 내는, 외부 의존(DB·네트워크·시간·랜덤)이 없는 코드. 이 영역을 두껍게 덮으면 피라미드의 바닥이 단단해집니다.

좋은 단위 테스트의 성질을 FIRST로 외웁니다.

FIRST의미
Fast밀리초 단위. 느리면 자주 안 돌리고, 안 돌리면 가치가 사라짐
Isolated다른 테스트·외부 상태에 의존하지 않음. 실행 순서가 결과를 바꾸면 안 됨(→ TEST-09)
Repeatable몇 번을 돌려도, 어느 환경에서도 같은 결과. 시간·랜덤은 주입해서 고정
Self-validating통과/실패가 단언(assert)으로 자동 판정. 로그 눈으로 보는 거 아님
Timely코드와 함께(또는 먼저) 작성. 나중으로 미루면 안 씀(→ TEST-06)

구조는 given-when-then으로 잡습니다 — 상황을 준비(given), 행위를 실행(when), 결과를 단언(then). 그리고 테스트 이름은 "어떤 조건에서 어떤 결과가 나와야 하는지"를 문장으로 읽히게 씁니다. test1이나 testCalculate가 아니라 만료된_쿠폰이면_할인을_적용하지_않는다처럼요. 테스트가 깨졌을 때 이름만 보고 "무엇이 무너졌는지" 알 수 있어야 합니다.

GIVEN — WHEN — THEN
given // 만료일이 지난 쿠폰과 주문 금액 10000원
  Coupon coupon = expiredCoupon();
when // 할인을 적용하면
  int total = applyCoupon(10000, coupon);
then // 할인 없이 원래 금액이 나온다
  assertThat(total).isEqualTo(10000);
실전

단위 테스트가 자꾸 mock 범벅이 된다면, 그건 테스트 문제가 아니라 설계가 순수 로직과 I/O를 안 갈라놨다는 신호입니다. 계산·판단 같은 결정 로직을 부수효과 없는 함수로 떼어내면, mock 없이 입력·출력만으로 테스트됩니다(→ TEST-04 과한 mock의 함정).

TEST-04

테스트 더블

"테스트 더블(test double)"은 진짜 협력 객체 대신 끼워 넣는 대역의 총칭입니다 — 영화의 스턴트 더블에서 온 말이죠. 다섯 종류를 뭉뚱그려 "mock"이라 부르는 경우가 많은데, 구분이 의도를 분명하게 합니다.

종류하는 일검증 대상
dummy자리만 채움. 호출되지 않음(인자 채우기용)없음
stub미리 정한 값을 돌려줌(상태 제공)결과 상태(state)
spy실제로 동작하되 호출 내역을 기록일부 상호작용
mock"이렇게 호출돼야 한다"는 기대를 미리 설정·검증상호작용(interaction)
fake가볍지만 실제로 동작하는 구현(예: in-memory DB)결과 상태

실무에서 가장 중요한 갈림길은 stub/fake(상태 검증) vs mock(상호작용 검증)입니다. stub은 "이 값을 주면 결과가 이래야 한다"를 봅니다. mock은 "이 메서드가 이 인자로 한 번 불렸어야 한다"를 봅니다. 후자는 구현 방식 그 자체를 단언하기 때문에 위험합니다.

과한 mock의 함정

"repository.save()가 정확히 한 번 호출됐는가"를 검증하는 테스트는 내부 구현에 결합됩니다. 저장을 두 번 호출하든 다른 메서드로 바꾸든, 최종 동작(데이터가 저장됨)이 같다면 그건 리팩토링일 뿐인데 테스트는 빨갛게 깨집니다. 이러면 테스트가 변경을 막는 족쇄가 됩니다 — 정확히 테스트가 주려던 자신감의 반대. 가능하면 mock 대신 fake(예: in-memory 구현)로 관찰 가능한 결과를 검증하세요(→ TEST-07 행동 vs 구현).

TEST-05

통합 테스트

단위 테스트는 "내 로직이 맞다"를 보장하지만 경계는 못 봅니다 — SQL이 진짜 도는지, JPA 매핑이 맞는지, 트랜잭션이 의도대로 커밋·롤백되는지는 진짜 DB 앞에서만 드러납니다. 통합 테스트는 이 경계를 검증합니다.

가장 흔한 실수는 DB를 mock으로 대체하는 것입니다. "repository.findById()가 이 객체를 반환한다"고 stub하면, 정작 그 쿼리가 실제로 그 객체를 반환하는지는 영원히 검증되지 않습니다. 컬럼명 오타, N+1, 잘못된 fetch 전략, native query의 문법 오류 — 전부 빠져나갑니다. 이것이 mock DB의 거짓 안심입니다(→ DB 핸드북).

방식잡히는 것 / 놓치는 것
mock repository로직 흐름만 / 실제 SQL·매핑·트랜잭션은 전혀 못 봄
H2 등 in-memory DB빠름 / 운영 DB와 SQL 방언·기능 차이로 거짓 통과·거짓 실패
Testcontainers운영과 같은 엔진의 진짜 DB / 컨테이너 기동 비용(초 단위)

Testcontainers는 테스트 시작 시 Docker로 운영과 동일한 DB(예: PostgreSQL)를 띄우고 끝나면 버립니다. H2 같은 in-memory DB로 대체하면 빠르지만, 운영이 PostgreSQL인데 H2로 테스트하면 방언 차이로 거짓 통과/거짓 실패가 납니다. "진짜 환경에 가까울수록 신뢰도가 높다"는 통합 테스트의 본질을 지키려면 같은 엔진을 쓰는 게 맞습니다.

트레이드오프

통합 테스트는 느립니다. 그래서 모든 케이스를 통합으로 덮으려 하면 피라미드가 뒤집힙니다(→ TEST-02). 비즈니스 분기·경계값은 단위로 빠르게 덮고, "쿼리·매핑·트랜잭션이 진짜 도는가"의 대표 경로만 통합으로 검증하세요. 통합 테스트는 양이 아니라 경계의 신뢰를 사는 것입니다.

TEST-06

TDD

TDD(Test-Driven Development)는 "테스트를 많이 쓰자"가 아니라 테스트를 먼저 쓰는 개발 리듬입니다. 핵심은 검증이 아니라 설계 압력 — 쓰기 전에 "이걸 어떻게 호출할까"를 강제로 먼저 정하게 만듭니다.

RED — 실패하는 테스트 → GREEN — 통과시키는 최소 코드 → REFACTOR — 구조 정리 ↺

사이클은 짧게 반복합니다. Red: 아직 없는 동작에 대한 실패하는 테스트를 먼저 씁니다. 이때 자연스럽게 "이 함수의 시그니처는 뭐지, 뭘 받고 뭘 돌려주지"를 호출자 입장에서 설계하게 됩니다. Green: 테스트를 통과시키는 최소한의 코드만 씁니다(과하게 만들지 않음). Refactor: 테스트가 초록인 상태에서 중복 제거·이름 개선 등 구조를 다듬습니다 — 초록이라 안전합니다.

단계목표흔한 실수
Red의도를 테스트로 먼저 표현, 실패 확인테스트가 처음부터 통과 → 아무것도 검증 안 함
Green가장 단순한 방법으로 통과필요 없는 일반화·추상화를 미리 넣음
Refactor초록 유지하며 구조 개선리팩토링과 기능 추가를 한 단계에 섞음
버그 = 재현 테스트 먼저

버그를 고칠 땐 먼저 그 버그를 재현하는 실패하는 테스트를 씁니다. 그 테스트가 빨갛게 떠야 "버그를 정확히 포착했다"가 증명되고, 고친 뒤 초록이 되면 "고쳤다 + 다시는 안 돌아온다(회귀 방어)"가 동시에 확보됩니다. 테스트 없이 바로 고치면, 진짜 원인을 짚었는지도 불확실하고 같은 버그가 재발해도 모릅니다(→ TEST-01 회귀 방어).

TEST-07

행동 vs 구현 테스트

테스트가 깨지는 데는 두 종류가 있습니다 — 동작이 진짜 망가져서 깨지는 좋은 실패, 그리고 동작은 멀쩡한데 내부 구조를 바꿨다고 깨지는 나쁜 실패. 후자를 만드는 게 구현 테스트입니다.

원칙은 하나입니다: 구현이 아니라 동작을 테스트하라. 호출자가 관찰할 수 있는 것 — 반환값, 던지는 예외, 외부에서 확인 가능한 상태 변화 — 만 단언합니다. "내부 헬퍼 메서드를 몇 번 불렀는지", "private 필드가 무엇인지", "어떤 순서로 호출했는지"는 구현 디테일이라 단언하면 안 됩니다.

구현 테스트(피하기)행동 테스트(지향)
save()가 1회 호출됨을 verify조회하면 저장된 값이 나옴
private 메서드를 직접 호출public API의 결과만 검증
내부 호출 순서를 단언최종 상태·반환값을 단언
리팩토링하면 깨짐(거짓 경보)동작이 같으면 통과(리팩토링 내성)

행동 테스트의 가장 큰 보상은 리팩토링 내성입니다. 내부를 어떻게 뜯어고치든 관찰 가능한 동작이 같으면 테스트는 초록을 유지합니다 — 그래서 테스트가 변경을 지키는 동시에 변경을 허락합니다. 구현에 결합된 테스트는 정반대로, 리팩토링할 때마다 무더기로 깨져서 "코드를 고치면 테스트도 다 고쳐야 한다"는 피로를 만들고 결국 테스트를 지우게 만듭니다.

예외 — 부수효과가 곧 동작일 때

"이메일을 보낸다", "이벤트를 발행한다"처럼 외부로의 호출 자체가 명세된 동작인 경우엔 상호작용 검증(mock verify)이 정당합니다. 판단 기준은 "이 호출이 사용자·외부 시스템에 관찰되는 결과인가, 아니면 그저 내가 택한 내부 구현인가"입니다(→ TEST-04).

TEST-08

커버리지의 함정

커버리지는 "테스트가 실행한 코드 줄의 비율"입니다 — 실행했다는 사실만 말할 뿐, 검증했다는 보장이 아닙니다. 이 차이를 놓치면 숫자는 높은데 안전망은 없는 상태가 됩니다.

가장 흔한 자기기만은 단언 없는 테스트입니다. 함수를 호출만 하고 결과를 단언하지 않으면, 예외만 안 터지면 통과하고 커버리지는 올라갑니다. 하지만 반환값이 틀려도 그 테스트는 아무것도 잡지 못합니다 — 커버리지 100%, 신뢰도 0%.

지표말해주는 것 / 못 말하는 것
line coverage실행된 줄 / 그 줄이 옳게 동작하는지는 모름
branch coverage분기 양쪽을 탔는지 / 경계값·조합은 모름
단언 유무커버리지에 안 잡힘 / 실제 검증의 핵심
의미 있는 경계값0·음수·null·빈 컬렉션·최대치 등 진짜 위험 지점

커버리지를 목표로 삼으면 사람들은 숫자를 채우는 쪽으로 최적화합니다 — getter/setter, 단언 없는 호출, 의미 없는 경로. 반대로 커버리지를 진단 도구로 쓰면 유용합니다: "이 핵심 결제 로직의 에러 분기가 한 번도 안 실행됐네?"를 발견하는 식으로요. 숫자가 낮은 곳은 단서지만, 숫자가 높다고 안전한 건 아닙니다.

무엇을 테스트할지

줄 수가 아니라 위험을 따라가세요. 복잡한 분기, 돈·권한이 걸린 로직, 과거에 버그가 났던 곳, 경계값(0·null·빈 값·오버플로). 단순 위임 코드나 프레임워크가 보장하는 부분에 단언을 욱여넣어 숫자를 채우는 건 비용만 늘립니다(→ TEST-01).

TEST-09

flaky test

flaky test는 코드 변경 없이 어떤 때는 통과하고 어떤 때는 실패하는 테스트입니다. 가장 해로운 종류인데, 신뢰를 갉아먹기 때문입니다 — 빨간불을 보고도 "또 그거겠지" 하며 재실행을 누르는 순간, 테스트 전체가 무의미해집니다.

원인전형적 증상 / 대응 방향
시간 의존now()·타임존·sleep에 의존 → 시간(clock)을 주입해 고정
실행 순서테스트가 공유 상태를 남김 → 각 테스트 격리·정리(isolated)
동시성스레드·비동기 race condition → 결정적 대기·동기화로 제거
외부 의존실제 네트워크·외부 API → 경계를 통제 가능한 더블·컨테이너로
랜덤고정 seed 없이 난수 사용 → seed 고정 또는 값 주입

flaky test에 대한 입장은 단호해야 합니다: 즉시 격리하거나 고쳐라. "가끔 깨지지만 보통 통과하니까 둔다"는 가장 나쁜 선택입니다. flaky 한 개를 용인하면 팀은 빨간불을 무시하는 법을 배우고, 그러면 진짜 회귀가 같은 빨간불에 섞여 들어와도 아무도 눈치채지 못합니다. 신뢰는 한 번 무너지면 복구가 어렵습니다.

flaky 한 개의 비용

flaky test 하나는 그 테스트 하나의 문제로 끝나지 않습니다 — CI 전체의 신호 가치를 떨어뜨립니다. 당장 고칠 수 없다면 일단 격리(quarantine)하고 추적 이슈를 남기세요. 그냥 두는 것보다, 비활성화하고 "왜·언제·누가" 고칠지 적어두는 게 정직합니다. 무시는 답이 아닙니다(→ TEST-01 신뢰).

TEST-10

Spring 테스트

Spring Boot에서 가장 비싼 비용은 애플리케이션 컨텍스트 기동입니다. 테스트가 느려지는 주범이 보통 여기죠. 그래서 "전체 컨텍스트를 띄울지, 필요한 슬라이스만 띄울지"의 선택이 테스트 속도를 좌우합니다.

애너테이션띄우는 범위 / 용도
@SpringBootTest전체 컨텍스트 — 모든 빈 로드. 무겁고 느림. 진짜 통합 흐름에만
@DataJpaTestJPA 슬라이스 — repository·EntityManager만. 기본 인메모리 DB·롤백
@WebMvcTest웹 슬라이스 — controller·MVC만. 서비스 계층은 @MockBean으로 대체
@MockBean컨텍스트의 특정 빈을 mock으로 교체(슬라이스 테스트의 짝)

핵심 전략은 느린 통합 테스트를 줄이고 슬라이스로 대체하는 것입니다. controller의 요청 검증·직렬화를 보려면 전체를 띄울 필요 없이 @WebMvcTest로 웹 계층만, JPA 매핑·쿼리를 보려면 @DataJpaTest로 영속 계층만 띄웁니다. @SpringBootTest는 "여러 계층이 실제로 함께 도는가"를 봐야 하는 대표 경로에만 아껴 씁니다.

@DataJpaTest와 거짓 안심

@DataJpaTest는 기본적으로 인메모리 DB(H2)로 돌아갑니다 — 빠르지만 운영이 PostgreSQL이면 방언 차이로 거짓 통과/실패가 납니다. 진짜 쿼리 신뢰가 필요하면 @AutoConfigureTestDatabase(replace = NONE)로 운영 DB를 쓰거나 Testcontainers와 결합하세요(→ TEST-05). 그리고 @MockBean은 컨텍스트를 재생성시켜 캐시를 깨뜨리니, 같은 mock 구성을 공유해 컨텍스트 재사용을 늘리는 게 속도에 유리합니다.

용어집

핸드북 전체에서 쓴 핵심 용어를 한자리에. 정의는 "무엇이며 왜 중요한가" 중심으로.

한 줄 요약

좋은 테스트 스위트는 커버리지 숫자가 아니라 변경 앞에서의 자신감으로 측정됩니다. 단위를 두껍게, 경계를 통합으로, 동작을 검증하고 구현은 풀어주고, flaky는 즉시 처리하고, 버그는 재현 테스트부터 — 이게 핸드북 전체의 한 문장입니다(→ 백엔드 개요, → DB 핸드북과 함께 보면 좋습니다).

OBS-01

관측가능성이란

관측가능성(observability)은 "대시보드를 많이 띄우는 것"이 아닙니다. 시스템 외부로 나오는 신호만 보고 내부에서 무슨 일이 벌어지는지 추론할 수 있는 정도를 말합니다. 핵심은 "사후에, 미리 예상 못 한 질문에 답할 수 있는가"입니다.

monitoring과 observability는 자주 섞여 쓰이지만 사고방식이 다릅니다. monitoring은 이미 아는 질문에 답합니다 — "CPU 80% 넘으면 알려줘", "5xx 비율이 1% 넘으면 알려줘". 무엇이 고장날지 미리 알고 그 지표를 지켜보는 것입니다. observability는 모르는 질문까지 사후에 탐색할 수 있게 합니다 — "왜 하필 결제 API만, 특정 B2B 어드민 사용자에게만, 지난 화요일 오후에 느렸지?" 이런 질문은 미리 알람으로 걸어둘 수 없습니다. 데이터가 충분히 풍부하게(high-cardinality) 쌓여 있어야 사후에 잘라 볼 수 있습니다.

관측가능성은 보통 세 기둥(three pillars)으로 설명합니다 — 로그(logs), 메트릭(metrics), 트레이스(traces). 셋은 경쟁 관계가 아니라 서로 다른 질문에 답하는 보완 관계입니다.

기둥답하는 질문형태 / 비용 특성
로그 (logs)"이 요청에서 정확히 무슨 일이 있었나"이벤트 단위 텍스트/JSON. 풍부하지만 양이 많고 비쌈
메트릭 (metrics)"전체적으로 얼마나, 얼마나 자주, 얼마나 느린가"시간에 따른 숫자 집계. 싸고 가볍지만 개별 사건은 못 봄
트레이스 (traces)"한 요청이 어느 서비스를 거쳐 어디서 느렸나"요청의 end-to-end 경로. 분산 시스템에서 필수

실전 디버깅은 보통 메트릭 → 트레이스 → 로그 순으로 좁힙니다. 메트릭으로 "p99가 튀었다"를 감지하고 → 트레이스로 "결제 서비스의 DB 호출이 범인"임을 찾고 → 그 trace id로 로그를 뒤져 "어떤 쿼리가, 어떤 파라미터로 락에 걸렸는지"를 봅니다. 셋을 trace id로 연결해두지 않으면(→ OBS-03) 이 사다리가 끊깁니다. 세 기둥을 다 갖추는 것보다, 서로 이어지게 갖추는 것이 핵심입니다.

OBS-02

로깅

로그는 가장 익숙하면서 가장 자주 잘못 쓰이는 도구입니다. System.out.println("여기까지 옴")은 로그가 아닙니다. 좋은 로그는 사람이 읽기 전에 기계가 먼저 검색·집계할 수 있는 구조를 가집니다.

2.1구조화 로그 — 왜 JSON인가

비구조화 로그("User 42 failed to pay order 1007")는 사람 눈엔 읽히지만, "사용자 42의 모든 실패를 세어줘" 같은 질의에 약합니다. 정규식으로 텍스트를 다시 파싱해야 하기 때문입니다. 구조화 로그(structured logging)는 처음부터 키-값 형태(보통 JSON)로 남겨, 로그 수집 시스템이 필드로 인덱싱·필터·집계할 수 있게 합니다.

UNSTRUCTURED vs STRUCTURED
# 나쁨 — 나중에 파싱 지옥
2026-06-23 03:14:07 ERROR User 42 failed to pay order 1007

# 좋음 — 필드로 바로 질의 가능
{"ts":"2026-06-23T03:14:07Z", "level":"ERROR", "event":"payment_failed",
 "user_id":42, "order_id":1007, "reason":"insufficient_credit",
 "trace_id":"a1b2c3d4", "service":"payment-api"}

Java/Spring 쪽이라면 SLF4J 인터페이스 뒤에 Logback이나 Log4j2를 두고, JSON encoder(예: logstash-logback-encoder)로 출력합니다. 메시지 안에 값을 문자열로 박지 말고 구조화된 필드(MDC, structured arguments)로 붙이는 게 요점입니다.

2.2로그 레벨 — 의미를 지켜라

레벨의미 — 언제 쓰나흔한 오용
ERROR사람이 봐야 하는 실패 — 요청이 실패했고 자동 복구 안 됨예상된 검증 실패(400)를 ERROR로 — 알림 노이즈의 주범
WARN지금은 괜찮지만 추세가 나쁘면 문제 — 재시도 성공, deprecated 사용아무도 안 보는 WARN을 남발 → 무의미
INFO정상 흐름의 이정표 — 요청 처리됨, 작업 완료, 상태 전이루프 안에서 INFO → 디스크·비용 폭발
DEBUG개발·심층 디버깅용 상세 — 운영에선 보통 끔운영에 DEBUG 켜둔 채 방치

핵심 구분은 "이 줄이 ERROR면 누군가 새벽에 깨어나야 하는가?"입니다. 사용자가 비밀번호를 틀린 것은 시스템 입장에서 정상 동작이므로 ERROR가 아닙니다(이건 expected failure). NullPointerException으로 요청이 터진 것은 bug이므로 ERROR입니다. 이 구분이 흐려지면 ERROR 로그가 신뢰를 잃고, 진짜 장애가 노이즈에 묻힙니다.

2.3무엇을 남기고 무엇을 절대 안 남기나

남길 것: 무슨 일(event 이름), 누구·무엇에 대해(엔티티 id), 결과(성공/실패·이유), 맥락(trace id·service). 안 남길 것은 더 중요합니다.

절대 로그에 흘리지 말 것 — PII·비밀

비밀번호, 토큰(access/refresh token, API key), 주민번호·카드번호 같은 PII, 인증 헤더(Authorization) 전문, 요청 body 통째 덤프. 로그는 여러 시스템으로 복제되고 장기 보관되며 접근 권한이 코드보다 넓습니다 — 한 번 새면 회수 불가입니다. user_id 같은 식별자는 남기되, 식별자로 다시 PII를 조회할 수 있게만 하고 PII 자체는 남기지 마세요. 객체를 통째로 로깅할 때 toString()에 민감 필드가 딸려오는 사고가 흔하므로 마스킹/직렬화 정책을 둬야 합니다.

OBS-03

상관관계 ID·분산 추적

단일 서버라면 로그를 시간순으로 읽으면 됩니다. 하지만 요청 하나가 게이트웨이 → 인증 → 결제 → 알림 서비스를 거치면, 각 서비스의 로그가 따로 흩어집니다. 이 흩어진 조각을 다시 꿰는 실 하나가 상관관계 ID(correlation/trace id)입니다.

원리는 단순합니다. 요청이 시스템에 처음 들어오는 진입점(보통 API gateway나 첫 서버)에서 고유 id를 한 번 발급하고, 이후 모든 내부 호출과 모든 로그 줄에 그 id를 함께 실어 나릅니다. 그러면 나중에 그 id 하나로 전 서비스의 로그를 모아 한 요청의 전체 여정을 복원할 수 있습니다.

한 trace_id 로 흩어진 로그를 꿰다
[gateway] {"trace_id":"a1b2c3", "span":"http_in", msg:"POST /orders"}
[auth] {"trace_id":"a1b2c3", "span":"verify_token", ok:true}
[payment] {"trace_id":"a1b2c3", "span":"charge", "dur_ms":2140} ← 범인
[notify] {"trace_id":"a1b2c3", "span":"send_email", skipped:true}

검색: trace_id = "a1b2c3" → 요청 하나의 전체 타임라인 복원

여기서 span은 trace 안의 한 작업 구간입니다 — 시작·끝 시각과 부모-자식 관계를 가집니다. trace는 여러 span의 트리이고, 각 span의 소요시간을 보면 어디서 시간을 썼는지가 드러납니다. 위 예에서 payment의 charge span이 2140ms로 튀는 게 보입니다.

이걸 표준화한 것이 OpenTelemetry(OTel)입니다 — 벤더 중립 표준으로, 코드에 한 번 계측(instrument)해두면 백엔드(Jaeger, Tempo, Datadog 등)를 갈아끼울 수 있습니다. Spring 진영은 Micrometer Tracing + OTel 조합이 흔하고, MDC에 trace id를 넣어 로그 패턴에 자동으로 찍히게 합니다. context 전파(propagation)는 보통 W3C traceparent HTTP 헤더로 서비스 간에 넘깁니다.

전파가 끊기는 지점 — 비동기와 경계

trace context는 스레드/요청 컨텍스트에 묶여 전파됩니다. 그래서 새 스레드 풀에 작업을 던지거나(@Async, 수동 ExecutorService), 메시지 큐(Kafka·SQS)를 건너거나, 외부 HTTP를 부를 때 context를 손으로 넘겨주지 않으면 trace가 거기서 끊깁니다. "분명 계측했는데 일부 서비스가 trace에 안 보인다"의 90%는 이 경계 누락입니다. 비동기·메시징 구간은 라이브러리의 context-propagation 지원을 명시적으로 켜야 합니다(→ OBS-01의 세 기둥 연결과 직결).

OBS-04

메트릭

메트릭은 시간에 따라 변하는 숫자입니다 — 싸고, 집계가 빠르고, 추세를 봅니다. 문제는 "무슨 숫자를 재느냐"입니다. 무작정 다 재면 노이즈가 되고 비용만 듭니다. 검증된 두 프레임워크가 길을 줄여줍니다.

프레임워크무엇을 재나대상
REDRate(초당 요청 수) · Errors(실패율) · Duration(응답시간 분포)요청을 받는 서비스(API, 마이크로서비스)
USEUtilization(사용률) · Saturation(대기·포화) · Errors(에러)자원(CPU, 메모리, 디스크, 커넥션 풀)

둘은 보는 층이 다릅니다. RED는 사용자가 겪는 것(요청이 얼마나 들어오고, 얼마나 실패하고, 얼마나 느린가)을, USE는 그 아래 자원이 버티는지를 봅니다. RED로 증상을 보고 USE로 원인을 좁히는 식입니다 — "응답이 느리다(RED-Duration)" → "DB 커넥션 풀이 포화다(USE-Saturation)".

4.1메트릭 타입 — 카운터·게이지·히스토그램

타입성질예
Counter단조 증가만(리셋 전까지 줄지 않음). 변화율을 봄총 요청 수, 총 에러 수
Gauge오르내리는 순간값현재 메모리, 큐 길이, 활성 커넥션 수
Histogram값을 구간(bucket)에 나눠 분포를 기록 — 백분위 계산의 기반응답시간 분포(p50/p95/p99 산출)

"평균 응답시간"을 gauge로 하나만 두면 안 됩니다 — 평균은 거짓말을 합니다(→ OBS-05). 응답시간은 반드시 histogram으로 분포를 잡아야 꼬리(tail)를 볼 수 있습니다.

4.2Prometheus의 pull 모델

전통적 모니터링은 앱이 중앙 서버로 지표를 밀어 보내는(push) 방식이었습니다. Prometheus는 반대로, 각 앱이 /actuator/prometheus 같은 엔드포인트에 현재 지표를 노출만 해두고 Prometheus가 주기적으로 긁어갑니다(pull/scrape). 장점은 "수집 대상이 살아있는지"를 scrape 성공 여부로 알 수 있고, 누가 무엇을 긁는지 중앙에서 통제된다는 것입니다. Spring Boot는 Micrometer가 표준 façade라, RED 지표를 자동/수동으로 등록하면 Prometheus·Datadog 등 어디로든 내보낼 수 있습니다.

cardinality 폭발 — 라벨에 무한 값 금지

메트릭에 라벨(label/tag)을 붙여 차원을 나눕니다(status="500", endpoint="/orders"). 그런데 라벨 값에 user_id, order_id, trace_id처럼 거의 무한한 값을 넣으면 time series 개수가 폭발해 저장소가 무너집니다(high cardinality). 그런 고유 식별자는 메트릭 라벨이 아니라 로그·트레이스에 둡니다. 메트릭의 라벨은 값 종류가 유한하고 적은 것(상태코드, 엔드포인트, 지역)만. 이 경계가 메트릭과 로그를 가르는 실용적 기준입니다.

OBS-05

백분위 지연

"평균 응답시간 120ms"는 안심되는 숫자처럼 보이지만, 종종 거짓말입니다. 평균은 소수의 느린 요청을 다수의 빠른 요청에 희석시켜 숨깁니다. 사용자 경험을 결정하는 건 평균이 아니라 꼬리(tail)입니다.

백분위(percentile)는 "전체 요청을 느린 순으로 줄 세웠을 때 N%가 이 값 이하"를 뜻합니다. p99 = 300ms는 "요청의 99%는 300ms 안에 끝났고, 가장 느린 1%만 그보다 느렸다"는 의미입니다.

평균의 거짓말 — 같은 평균, 다른 경험
요청 100건: 99건이 50ms, 1건이 5000ms
평균(avg) = (99×50 + 5000) / 100 = 99.5ms ← "빠르네"
p50 = 50ms
p99 = 50ms
p99.9 = 5000ms ← 누군가는 5초를 기다렸다
지표읽는 법무엇에 쓰나
p50 (중앙값)절반의 사용자가 이보다 빠름 — "전형적" 경험일반적 체감 속도의 기준선
p9595%가 이 안 — 나쁜 날의 경험SLO 목표로 흔히 씀
p99최악 1%의 경계 — 꼬리 지연의 핵심대규모에선 "1%"도 매우 많은 사람
왜 꼬리가 곧 사용자 경험인가

한 화면을 그리는 데 내부 호출 10개가 필요하면, 그 페이지는 가장 느린 호출 하나에 발목 잡힙니다. 개별 호출의 p99가 1%라도, 10개 중 하나라도 걸릴 확률은 훨씬 높아집니다 — 꼬리 지연은 곱해집니다. 그래서 대규모 시스템일수록 p99·p99.9를 봅니다. 한 가지 함정: 여러 인스턴스의 p99를 평균 내면 안 됩니다. 백분위는 평균으로 합칠 수 없습니다 — 원본 histogram을 합산한 뒤 백분위를 다시 계산해야 합니다(→ OBS-04 histogram이 필요한 이유).

OBS-06

대시보드·시각화

대시보드의 목적은 "예뻐 보이는 것"이 아니라 장애 중에 5초 안에 무엇이 비정상인지 가리키는 것입니다. 그래프가 30개면 아무것도 안 보이는 것과 같습니다. 좋은 대시보드는 적게, 의도적으로 고릅니다.

업계 표준 출발점은 구글 SRE의 네 가지 골든 시그널(four golden signals)입니다 — 사용자를 향한 서비스를 볼 때 이 넷부터 깔면 대부분 커버됩니다.

골든 시그널의미RED/USE 대응
Latency (지연)요청 처리 시간 — 성공/실패를 나눠서 봐야 함RED-Duration (p50/p95/p99)
Traffic (트래픽)수요의 양 — 초당 요청 수RED-Rate
Errors (에러)실패율 — 명시적(5xx)·암묵적(느려서 사실상 실패) 모두RED-Errors
Saturation (포화)자원이 얼마나 꽉 찼나 — 한계에 얼마나 가까운가USE-Saturation

도구로는 Grafana가 사실상 표준입니다 — Prometheus·Loki(로그)·Tempo(트레이스) 등 여러 소스를 한 화면에 모으고, 패널을 trace id로 이어 메트릭에서 로그로 점프할 수 있습니다(→ OBS-01의 세 기둥 연결을 실제로 구현하는 자리).

한 화면 설계 원칙

① 위에서 아래로 추상화가 내려가게 — 맨 위엔 골든 시그널(증상), 아래로 갈수록 자원·내부 지표(원인). ② 에러율은 성공/실패를 분리해 그려야 의미가 있습니다(실패 요청의 지연을 성공에 섞으면 왜곡). ③ 한 화면은 한 질문에 답하게 — "이 서비스가 건강한가" 대시보드와 "이 장애의 원인은" 대시보드는 분리. ④ 임계선(threshold)을 그래프에 그어 "정상 범위"를 눈으로 보이게. 대시보드는 만드는 것보다 장애 때 실제로 쓰이는지로 평가하세요.

OBS-07

헬스체크

오케스트레이터(Kubernetes 등)나 로드밸런서는 앱에게 "너 괜찮아?"를 주기적으로 묻습니다. 이 질문이 사실 두 개의 다른 질문이라는 걸 모르면, 헬스체크가 오히려 장애를 만듭니다.

프로브묻는 질문실패하면
Liveness (생존)"이 프로세스가 살아있나 — 데드락·먹통은 아닌가"컨테이너를 재시작
Readiness (준비)"지금 트래픽을 받을 준비가 됐나"트래픽을 차단(LB에서 제외), 재시작 안 함

차이가 결정적입니다. liveness는 "재시작이 답인가"를 묻습니다 — 프로세스가 회복 불가능하게 멈췄을 때만 true가 되어야 합니다. readiness는 "지금 일을 시켜도 되나"를 묻습니다 — 아직 캐시를 데우는 중이거나, 의존 DB가 잠깐 끊겼을 때 false가 되어 트래픽을 잠시 비껴가게 합니다(재시작은 하지 않음). Spring Boot Actuator는 /actuator/health/liveness와 /readiness를 분리 제공합니다.

둘을 합치면 사고 — cascading restart

흔한 치명적 실수: liveness 체크 안에 DB 연결 확인을 넣는 것. DB가 잠깐 느려지면 → liveness가 실패 → 오케스트레이터가 멀쩡한 앱을 재시작 → 재시작한 앱들이 DB에 동시에 재연결을 시도해 DB를 더 죽임 → 더 많은 재시작. 외부 의존성은 readiness에만, liveness는 오직 "이 프로세스 자신이 먹통인가"만 확인해야 합니다. liveness는 의존성에 무지해야 안전합니다. readiness 실패는 트래픽만 빼고 기다리므로, 일시적 의존성 장애를 재시작 폭풍 없이 흘려보낼 수 있습니다.

OBS-08

SLI·SLO·에러버짓

"우리 서비스는 안정적이어야 한다"는 측정할 수 없습니다. "99.9%의 요청이 300ms 안에 성공한다"는 측정할 수 있습니다. SLI/SLO는 신뢰성을 구호에서 숫자로 바꾸고, 에러버짓은 그 숫자를 의사결정 도구로 만듭니다.

개념정의예
SLI
(Indicator)		신뢰성을 재는 실제 측정 지표"성공 요청 비율", "p99 지연"
SLO
(Objective)		그 SLI에 대한 목표치"30일간 성공률 ≥ 99.9%"
에러버짓
(error budget)		SLO가 허용하는 실패의 총량 = 100% − SLO99.9% → 0.1% = 30일에 약 43분

핵심 통찰은 "100% 신뢰성은 목표가 아니다"입니다. 100%를 추구하면 비용이 무한대로 가고 어떤 변경도 못 합니다. SLO를 99.9%로 정하면 0.1%만큼의 실패할 여유(에러버짓)가 생기고, 이 여유를 어디에 쓸지 정할 수 있습니다 — 새 기능 배포, 위험한 마이그레이션, 인프라 실험. 에러버짓은 "안정성 팀 vs 기능 팀"의 해묵은 싸움을 숫자로 중재합니다.

ERROR BUDGET — 한 달치 실패 예산
SLO 99.9% (30일 기준)
허용 다운타임 = 30일 × 0.1% = 약 43분 / 월

budget 남음 → 배포 속도 올려도 됨, 위험한 실험 OK
budget 소진 → 기능 동결, 안정화에 집중 (새 배포 중단)
budget 초과 → 인시던트 — 무엇이 예산을 태웠는지 회고
SLI는 사용자 관점으로 골라라

SLI를 CPU 사용률 같은 내부 지표로 잡으면 안 됩니다 — 사용자는 CPU에 관심 없습니다. 사용자가 체감하는 것(요청 성공률, 응답 지연, 데이터 신선도)으로 골라야 의미가 있습니다. 그리고 SLO는 한 번 정하고 끝이 아니라, 에러버짓 소진 추세를 보며 현실에 맞게 조정하는 살아있는 합의입니다. 버짓 소진 속도(burn rate)를 알림으로 거는 게 단순 임계치 알림보다 훨씬 신호가 좋습니다(→ OBS-09).

OBS-09

알림

알림(alert)의 유일한 목적은 사람의 행동을 유발하는 것입니다. 행동할 게 없는 알림은 알림이 아니라 노이즈이고, 노이즈는 진짜 신호를 죽입니다. 좋은 알림 설계는 "더 거는 것"이 아니라 "덜 거는 것"에서 나옵니다.

운영을 망치는 첫 번째 적은 alert fatigue(알림 피로)입니다 — 울려도 대개 별일 아니면, 사람은 알림을 무시하게 학습됩니다. 그러다 진짜 장애 알림을 놓칩니다. "양치기 소년"이 시스템 안에서 일어나는 것입니다. 그래서 모든 알림에 던질 질문은 하나입니다: "이게 울리면 내가 지금 뭘 해야 하지? 답이 없으면 끄거나 대시보드로 강등."

구분증상 기반 (symptom)원인 기반 (cause)
무엇에사용자가 겪는 것 — "성공률이 SLO를 깬다"내부 원인 — "CPU 90%", "디스크 80%"
장점실제 영향이 있을 때만 울림 — 노이즈 적음원인을 직접 가리킴
함정(거의 없음 — 기본으로 삼아라)CPU 90%여도 사용자는 멀쩡할 수 있음 → 오탐

원칙은 "증상에 알림을 걸고, 원인은 대시보드에 둔다"입니다. CPU가 높다고 항상 문제는 아닙니다 — 사용자가 멀쩡하면 그건 알릴 일이 아니라 참고 정보입니다. 사용자 영향(SLO 위반, 에러버짓 burn rate)에 알림을 걸고, 알림이 울린 뒤 원인 대시보드로 내려가 CPU·디스크를 봅니다. 이렇게 하면 알림 수가 급감하면서 하나하나가 진짜가 됩니다.

온콜 — 알림은 사람의 잠을 깨운다

알림은 결국 온콜(on-call) 엔지니어를 호출합니다. 그래서 모든 알림에는 ① 명확한 심각도(지금 깨워야 하나, 아침에 봐도 되나) ② 실행 가능한 런북(runbook) 링크가 있어야 합니다 — "이 알림이 울리면 먼저 X를 확인하고 Y를 하라". 새벽에 깬 사람에게 맥락 없는 알림 한 줄만 던지는 건 잔인하고 비효율적입니다. 알림을 추가할 때마다 "이걸로 누군가 새벽에 깰 가치가 있는가"를 먼저 물으세요. 자동 복구가 가능한 건 알림이 아니라 자동화로 처리하는 게 맞습니다.

OBS-10

장애 대응

장애가 터지면 본능은 "왜 이렇게 됐지?"를 파고드는 것입니다. 그건 나중 일입니다. 인시던트 대응의 첫 계명은 원인 분석보다 복구가 먼저입니다 — 피를 멈추는 게 먼저, 부검은 나중에.

감지 detect→ 분류 triage→ 완화·롤백 mitigate→ 복구 recover→ 커뮤니케이션 communicate

단계별 핵심:

  • 감지(detect) — 알림이나 제보로 인지. 인지 시각을 기록해야 나중에 MTTR을 잴 수 있습니다.
  • 분류(triage) — 심각도와 영향 범위를 빠르게 판단. 사용자 몇 %가, 무엇을 못 하나. 인시던트 지휘자(incident commander)를 정합니다.
  • 완화·롤백(mitigate) — 가장 중요한 단계. 원인을 몰라도 영향을 멈출 수 있습니다 — 직전 배포 롤백, 트래픽 차단, 기능 플래그 끄기, 스케일 아웃. "왜?"는 잠시 미룹니다.
  • 복구(recover) — 정상 상태로 되돌리고 안정화 확인.
  • 커뮤니케이션(communicate) — 전 단계에 걸쳐 계속. 이해관계자·사용자에게 상태를 알립니다. 공공 B2B 시스템에서는 특히 중요합니다.
심각도의미 (예시 기준)대응
SEV1전면 장애 — 핵심 기능 불가, 광범위 사용자 영향즉시 전원 소집, 24/7 대응
SEV2중대한 부분 장애 — 일부 기능·일부 사용자온콜 즉시 대응, 우선순위 높음
SEV3경미 — 우회 가능, 영향 제한적업무 시간 내 처리
복구 먼저 — 디버깅의 유혹을 참아라

새 배포 직후 장애가 났다면 원인 분석 전에 일단 롤백하세요. "거의 다 찾았는데"의 유혹이 가장 위험합니다 — 그 5분이 영향을 몇 배로 키웁니다. 롤백으로 사용자를 구한 뒤, 안전한 환경에서 천천히 원인을 봅니다. 또 하나: 대응 중 한 사람만 변경하게 하세요. 여러 명이 동시에 손대면 무엇이 효과였는지 알 수 없고 상황을 더 망칩니다. 모든 조치와 시각을 채널에 기록(timeline)해두면 포스트모템(→ OBS-11)이 쉬워집니다.

OBS-11

포스트모템

장애가 끝났다고 일이 끝난 게 아닙니다. 같은 장애가 다시 안 나게 하는 게 진짜 마무리입니다. 포스트모템(postmortem)은 처벌 문서가 아니라 학습 문서입니다 — 그래서 비난 없이(blameless)가 전제입니다.

왜 blameless인가. 사람을 탓하면 다음부터 모두가 사실을 숨깁니다 — "내가 그 명령을 쳤다"를 말 안 하게 됩니다. 그러면 진짜 원인(왜 그 위험한 명령이 그렇게 쉽게 실행됐는가, 왜 가드가 없었는가)에 도달할 수 없습니다. 사람은 시스템이 허용한 실수를 할 뿐이라는 전제로, 시스템·프로세스의 약점을 찾습니다. 개인의 잘못이 아니라 "이런 실수가 영향을 미칠 수 있었던 구조"가 문제입니다.

포스트모템 구성요소내용
타임라인감지→완화→복구의 시각별 사실 기록 — 추측 아닌 사실
영향사용자 몇 %가, 얼마 동안, 무엇을 — 가능하면 정량화
근본 원인5 whys로 표면 너머까지 — 기술적·조직적 원인 모두
액션 아이템재발 방지책 — 담당자·기한이 붙은 실행 가능한 것만

5 Whys는 "왜"를 다섯 번 정도 반복해 표면 증상에서 구조적 원인으로 내려가는 기법입니다.

5 WHYS — 표면에서 구조로
결제가 5분간 실패했다.
why? → DB 커넥션 풀이 고갈됐다.
why? → 느린 쿼리가 커넥션을 오래 잡았다.
why? → 새 기능이 인덱스 없는 컬럼으로 조회했다.
why? → 리뷰에서 쿼리 플랜을 아무도 안 봤다.
why? → PR 체크리스트에 쿼리 검토 항목이 없다. ← 진짜 액션
액션 아이템이 핵심 — 안 그러면 의식일 뿐

위 예에서 액션은 "그 개발자를 혼낸다"가 아니라 "인덱스 없는 컬럼 조회를 CI에서 막는다 / PR 체크리스트에 쿼리 플랜 검토를 넣는다"입니다 — 같은 부류의 실수를 시스템이 자동으로 잡게 만드는 것. 액션 아이템에 담당자와 기한이 없으면 포스트모템은 한 번 쓰고 잊히는 의식에 그칩니다. 그리고 액션은 추적되어 실제로 완료돼야 합니다 — 같은 원인의 장애가 두 번 나면 그건 첫 포스트모템이 실패했다는 뜻입니다(→ OBS-08 에러버짓을 무엇이 태웠는지와 연결).

OBS-12
