통합 문서

프론트엔드 품질·릴리스는 보안, 테스트, 배포를 따로 점검하는 목록이 아니라 변경을 안전하게 사용자에게 내보내는 운영 루프입니다. 변경 위험을 분류하고, 그 위험에 맞는 테스트와 보안 리뷰를 선택하고, preview와 CI에서 확인한 뒤, release health window 동안 오류·Web Vitals·API 실패율을 관찰하고 rollback 또는 forward fix를 결정합니다.

MODEL

FRONTEND RELEASE QUALITY OPERATING MODEL

프론트엔드 품질은 테스트 개수나 배포 성공 로그가 아니라 사용자의 브라우저에서 새 버전이 안전하게 작동한다는 증거입니다. 이 모델은 변경이 들어온 순간부터 release health window가 끝날 때까지의 판단 순서를 고정합니다.

FRONTEND RELEASE QUALITY OPERATING MODEL change risk classification → security review gate → risk based test matrix
→ preview deployment → CI quality gate → release notes
→ canary or staged rollout → release health window
→ Web Vitals regression / error spike / API failure review
→ rollback, feature flag kill switch, or forward fix decision

핸드북

프론트엔드 보안은 브라우저가 신뢰 경계를 어떻게 다루는지 이해하는 일입니다. 입력, HTML, URL, storage, dependency가 모두 공격 표면입니다.

COMPANION : 백엔드 인증과 보안 핸드북
FSEC-01

브라우저 위협 모델

사용자 입력, URL parameter, 외부 API, CMS 콘텐츠, third-party script는 모두 신뢰할 수 없는 입력입니다.

표면위험대응
DOM 삽입XSSescape 기본값 유지, sanitize
쿠키CSRF, session fixationSameSite, CSRF token, Secure, HttpOnly
StorageXSS 시 token 탈취민감 정보 저장 최소화
URLopen redirect, reflected XSSallowlist, 상대 경로 제한
Dependencymalicious packagelockfile, audit, 최소 의존성
FRONTEND SECURITY RELEASE GATE릴리스 차단 조건통과 증거
HTML 주입 표면dangerouslySetInnerHTML, markdown, CMS, rich text renderer가 바뀌었다허용 tag/attribute 리뷰, sanitizer regression, XSS payload negative test
토큰·세션token storage, cookie option, refresh flow, logout/revoke가 바뀌었다XSS/CSRF 위험 비교, SameSite/HttpOnly/Secure 확인, 세션 폐기 테스트
redirect·URLcallback URL, deep link, 외부 이동, returnTo 파라미터가 바뀌었다origin/domain allowlist, open redirect negative test, URL scheme 검증
third-party scriptanalytics, 채팅, 결제, A/B test script가 추가되었다CSP/SRI 검토, 로드 범위 제한, PII 전송 여부 확인
dependencylockfile 또는 install script가 바뀌었다audit 결과, transitive diff, maintainer/reachability 판단, rollback 가능성
FSEC-02

XSS와 HTML 주입

XSS는 공격자가 브라우저에서 스크립트를 실행하게 만드는 취약점입니다. React는 문자열 escape를 기본 제공하지만, 그 안전장치를 끄면 책임은 개발자에게 넘어갑니다.

패턴위험대응
dangerouslySetInnerHTMLHTML 실행 표면sanitize 후 제한된 곳에서만 사용
markdown 렌더링링크, HTML, script 주입HTML 비활성화 또는 sanitizer
URL 삽입javascript: schemeprotocol allowlist
third-party script공급망 XSSSRI, CSP, 최소 로드
React 실무 지점기준
외부 HTML/CMS/markdown서버에서 왔어도 untrusted로 취급. raw HTML 비활성화 또는 DOMPurify 같은 검증된 sanitizer 사용
링크target="_blank"에는 rel="noopener noreferrer"를 함께 사용
URL/path/style 속성사용자 입력을 직접 넣지 않고 protocol/domain/value allowlist 검증
sanitizer 설정허용 tag/attribute를 리뷰하고, 보안 변경으로 취급
SANITIZE

escape와 sanitize는 다릅니다. escape는 문자열을 HTML로 실행되지 않게 바꾸고, sanitize는 허용된 HTML만 남깁니다. 사용자 생성 HTML이 필요하면 반드시 검증된 sanitizer를 사용합니다.

FSEC-03

CSRF와 쿠키

CSRF는 브라우저가 쿠키를 자동 첨부한다는 성질을 악용합니다. 사용자가 공격자 사이트를 방문했을 뿐인데 우리 서비스에 mutation 요청이 갈 수 있습니다.

방어설명주의
SameSitecross-site 요청에 쿠키 첨부 제한OAuth redirect와 호환성 확인
CSRF token공격자가 모르는 값을 mutation에 요구token 전달/검증 경로 필요
Origin/Referer 검증요청 출처 확인보조 방어로 사용
재인증위험 작업 전 추가 확인UX 비용 있음
FSEC-04

토큰 저장

토큰 저장 위치에는 정답이 없습니다. XSS와 CSRF 중 어느 위험을 어떻게 줄일지의 선택입니다.

위치장점위험
memory새로고침 시 사라져 탈취 창이 작음새로고침 후 재인증/refresh 설계 필요
localStorage구현 쉬움XSS가 읽을 수 있음
HttpOnly cookieJS가 읽을 수 없음CSRF 방어 필요
sessionStorage탭 단위 격리XSS에는 읽힘
권장 의사결정설명
장기 refresh token브라우저 JS 접근 저장소에 두지 않음. HttpOnly + Secure + SameSite 쿠키와 서버 측 rotation/reuse detection 검토
access token짧은 TTL, 메모리 보관 우선. 새로고침 UX와 refresh 흐름을 함께 설계
쿠키 방식JS가 읽을 수 없어도 XSS가 사용자 권한으로 요청을 보낼 수 있음. CSRF, CSP, 입력 처리와 함께 설계
logout/revokerefresh token 폐기, 세션 목록, 모든 기기 로그아웃 요구를 백엔드와 계약
ACCESS TOKEN

access token은 짧게, refresh token은 회전과 폐기를 설계합니다. 프론트만으로 인증 보안을 완성할 수 없고 백엔드 정책과 함께 설계해야 합니다.

FSEC-05

CSP와 보안 헤더

CSP는 브라우저가 어떤 리소스를 실행할 수 있는지 제한하는 정책입니다. XSS 피해를 줄이는 강력한 방어층입니다.

헤더역할
Content-Security-Policyscript/style/image/connect 출처 제한
X-Content-Type-OptionsMIME sniffing 방지
Referrer-Policy외부 이동 시 referrer 노출 제어
Permissions-Policy카메라, 위치 등 브라우저 기능 제한
CSP 예시 골격
Content-Security-Policy:
default-src 'self';
script-src 'self' 'nonce-{REQUEST_NONCE}';
style-src 'self' 'nonce-{REQUEST_NONCE}';
img-src 'self' data: https:;
connect-src 'self' https://api.example.com https://errors.example.com;
object-src 'none';
base-uri 'self';
frame-ancestors 'none'

unsafe-inline은 XSS 방어력을 크게 낮추므로 nonce나 hash 기반으로 inline 실행을 제한합니다. nonce는 요청마다 새로 생성해야 하며, SPA가 호출하는 API, analytics, error monitoring 도메인은 connect-src에 명시합니다.

CSP 적용 순서
Report-Only로 먼저 배포
위반 로그 수집
필요 출처 allowlist 정리
inline script 제거 또는 nonce 적용
enforce 모드로 전환
FSEC-06

의존성과 공급망

프론트엔드는 npm 생태계 위에서 움직입니다. 작은 패키지 하나도 배포 산출물과 사용자 브라우저에 들어갈 수 있습니다.

관리기준
lockfile리뷰하고 커밋, 예기치 않은 버전 변경 방지
dependency audit심각도와 실제 reachability를 함께 판단
package 평가유지보수, 다운로드, 권한, 대체 가능성 확인
bundle 영향기능 대비 크기와 transitive dependency 확인
운영 정책내용
lockfile diff의존성 PR에서 transitive 변경을 리뷰
package manager 고정팀/CI가 같은 install 결과를 얻도록 버전과 설정 고정
install script 주의postinstall 등 실행 script가 있는 패키지는 별도 검토
dependency confusionprivate scope registry 설정과 패키지 이름 충돌 방지
자동 업데이트Renovate/Dependabot 정책으로 보안 패치와 breaking change를 분리
SUPPLY CHAIN RELEASE DECISION확인 질문실패 신호
reachability취약 패키지가 실제 bundle이나 실행 경로에 포함되는가audit severity만 보고 무조건 무시하거나 무조건 배포 중단
install scriptpostinstall/preinstall이 CI나 개발자 머신에서 실행되는가새 패키지가 파일 시스템, network, token에 접근하는지 보지 않음
lockfile drift기능 변경 없이 transitive dependency가 대량으로 바뀌었는가lockfile을 사람이 리뷰하지 않고 자동 병합
third-party runtime브라우저에서 외부 script가 사용자 세션과 DOM에 접근하는가CSP 없이 전역 script를 추가하고 PII 전송 여부를 모름
PR CHECK

dangerouslySetInnerHTML, token storage, CSP, 새 third-party script, 인증 redirect, sanitizer 설정 변경은 보안 리뷰 대상으로 표시합니다. 단순 UI 변경처럼 취급하지 않습니다.

SRISubresource Integrity

NonceCSP

FTEST-01

테스트의 목적

테스트는 QA 문서가 아니라 변경 안전장치입니다. 실패했을 때 무엇이 깨졌고 사용자에게 어떤 영향이 있는지 알려야 합니다.

나쁜 테스트좋은 테스트

FTEST-02

FTEST-03

컴포넌트 테스트

Testing Library의 관점은 "사용자가 찾는 방식으로 찾는다"입니다. role, label, text를 우선합니다.

COMPONENT TEST 원칙

fake timer와 userEvent 조합 주의입력 이벤트와 timer 진행이 어긋나 flaky해질 수 있음
ACCESSIBILITY BENEFIT

role과 label로 테스트가 가능하다는 것은 보조기술도 그 UI를 이해할 가능성이 높다는 뜻입니다. 테스트 가능성과 접근성은 자주 같은 방향으로 움직입니다.

FTEST-04

E2E 테스트

E2E는 비싸지만 실제 통합 문제를 잡습니다. 모든 것을 E2E로 덮지 말고, 실패하면 매출/업무 흐름이 깨지는 핵심 경로에 집중합니다.

좋은 후보이유
로그인/로그아웃모든 기능의 진입점
핵심 생성/수정/삭제업무 가치가 직접 연결
결제/신청/제출중복 제출과 실패 처리가 중요
권한별 접근보안 회귀 영향이 큼
Playwright 규칙기준
locatorgetByRole, getByLabel, getByText 우선. data-testid는 안정적인 사용자 식별자가 없을 때만
대기page.waitForTimeout 금지. web-first assertion으로 상태를 기다림
데이터테스트별 독립 데이터 생성/정리. 실행 순서에 의존하지 않음
인증storageState 재사용은 빠르지만 권한/세션 격리 위험을 고려

FLAKY TEST TRIAGE증상처리

데이터 오염테스트 순서나 병렬 실행 여부에 따라 실패테스트별 독립 계정/tenant/data seed를 만들고 cleanup 또는 unique key를 둔다
네트워크 의존외부 API, analytics, feature flag 응답에 따라 흔들림핵심 경계는 MSW/route mock으로 통제하고 소수 smoke만 real backend로 둔다
시간·locale날짜, timezone, animation, debounce 때문에 결과가 바뀜clock, locale, reduced motion, deterministic fixture를 고정한다
권한 상태storageState 재사용 후 세션 만료·role 변경을 놓침권한별 state를 분리하고 refresh/expired session 시나리오를 별도 테스트한다
FLAKY TEST

flaky 테스트는 신뢰를 갉아먹습니다. 임의 sleep 대신 명확한 상태를 기다리고, 테스트 데이터는 격리하며, 외부 서비스 의존은 통제합니다.

프론트엔드 배포
핸드북

프론트엔드 배포는 파일을 올리는 일이 아니라 사용자의 브라우저와 CDN 캐시가 새 버전을 안전하게 만나게 하는 과정입니다. 빌드 산출물, cache header, release id, 오류 수집이 함께 맞아야 합니다.

STACK : Vite · Static Hosting · CDN · nginx · Error Monitoring
DEP-01

프론트엔드 배포의 본질

백엔드는 서버 프로세스를 교체하지만, 프론트엔드는 이미 사용자 브라우저에 내려간 파일과도 공존해야 합니다.

특성의미위험
정적 파일HTML, JS, CSS, asset을 CDN/서버에서 제공캐시된 옛 파일과 새 파일 불일치
해시 파일명내용이 바뀌면 이름이 바뀜index.html이 옛 chunk를 가리키면 404
브라우저 cache사용자별로 cache 상태가 다름재현 어려운 배포 이슈
API 계약프론트/백엔드가 동시에 바뀌지 않을 수 있음호환성 없는 배포 순서
FRONTEND DEPLOYMENT HEALTH GATE배포 전 확인배포 후 확인
preview deploymentproduction build 산출물로 preview URL을 만들고 핵심 route smoke를 실행한다preview와 production의 env/config 차이가 릴리스 노트에 남아 있는지 본다
artifact integrityrelease id, commit SHA, asset hash, sourcemap upload가 서로 매핑된다오류 이벤트가 minified stack이 아니라 원본 파일/line으로 복원된다
cache policyindex.html은 재검증, hashed asset은 immutable, 이전 asset 보존 기간을 확인한다chunk load error와 stale index 비율을 release별로 본다
quality signaltest, build, bundle budget, a11y smoke, visual regression gate를 통과한다error rate, Web Vitals regression, API failure, user report를 release health window 동안 본다
RULE

assets는 길게 캐시하고, entry HTML은 짧게 캐시합니다. 해시가 있는 JS/CSS는 immutable하게 둘 수 있지만, index.html은 최신 asset 목록을 받아야 하므로 재검증되어야 합니다.

DEP-02

Vite build와 환경변수

Vite의 환경변수는 빌드 시점에 클라이언트 번들로 주입됩니다. 브라우저에 들어가는 값은 secret이 될 수 없습니다.

항목기준
VITE_ prefix클라이언트에 노출할 변수만 사용
API base URL환경별로 빌드하거나 runtime config 제공
secret프론트 번들에 절대 포함하지 않음
modedevelopment/staging/production 설정 분리
BUILD OUTPUT
npm run build
→ dist/index.html
→ dist/assets/index-[hash].js
→ dist/assets/index-[hash].css
→ static assets
DEP-03

정적 호스팅·CDN·캐시

CDN은 빠르게 전달하지만 잘못된 cache header는 배포 사고를 키웁니다.

파일권장 cache이유
/assets/*-[hash].jsmax-age=31536000, immutable내용 변경 시 파일명이 바뀜
/assets/*-[hash].cssmax-age=31536000, immutable내용 변경 시 파일명이 바뀜
/index.htmlno-cache최신 asset 목록을 받아야 함
이미지변경 정책에 따라파일명 해시 여부 확인
SPA FALLBACK

클라이언트 라우팅을 쓰면 서버는 없는 경로를 404로 끝내지 말고 index.html로 fallback해야 합니다. 단, 실제 asset 경로까지 fallback하면 문제를 숨길 수 있으니 라우팅 경로와 asset 경로를 구분합니다.

배포 사고 방어기준
이전 assets 보존오래 열린 탭과 stale index가 참조할 수 있으므로 이전 hashed assets를 일정 기간 유지
CDN purge대상은 주로 index.html. hashed asset은 overwrite하지 않고 새 이름으로 배포
chunk load error감지 후 사용자에게 새로고침 UX 제공. 무한 reload 방지
Service WorkerskipWaiting/clientsClaim 전략과 업데이트 알림 UX를 명시적으로 결정
API 호환성백엔드 API는 최소 한 버전 backward-compatible하게 배포해 프론트/백엔드 순서 차이를 흡수
DEP-04

sourcemap과 오류 수집

운영 오류를 추적하려면 minified stack trace를 원본 코드로 복원할 수 있어야 합니다. sourcemap은 강력하지만 노출 정책을 신중히 정해야 합니다.

전략장점주의
비공개 업로드오류 수집 도구에서만 복원CI에서 release id와 함께 업로드 필요
공개 sourcemap설정 단순소스 노출 위험
sourcemap 없음노출 위험 낮음운영 디버깅이 매우 어려움

권장 운영 방식은 production sourcemap을 생성하되, Sentry 같은 오류 수집 도구에 release id와 함께 비공개 업로드하고 공개 배포 산출물에서는 .map 파일을 제외하는 것입니다. 빌드 전 secret scan을 수행하고, release artifact와 commit SHA를 매핑해야 stack trace를 배포 이력과 연결할 수 있습니다.

RELEASE ID

오류 이벤트에는 release id, commit hash, route, browser, user action context가 함께 있어야 합니다. "Cannot read property"만으로는 원인 분석이 어렵습니다.

DEP-05

rollback과 release

좋은 배포는 실패하지 않는 배포가 아니라 실패해도 빠르게 되돌릴 수 있는 배포입니다.

전략설명
atomic deploy새 dist를 완성한 뒤 한 번에 current 포인터 전환
versioned assets이전 index가 참조하는 asset도 일정 기간 유지
feature flag코드는 배포하되 노출은 점진적으로 제어
canary일부 사용자/트래픽에 먼저 노출
ROLLBACK FORWARD FIX DECISION선택 기준실행
feature flag kill switch새 기능만 문제이고 기존 경로가 안전하다flag off, affected route smoke, 사용자 공지 여부 판단
rollbackchunk load error, auth break, blank screen, 결제/제출 불가처럼 핵심 경로가 깨졌다이전 index와 asset pointer 복구, CDN purge 범위 제한, release health window 재시작
forward fix영향이 작고 원인이 명확하며 rollback이 더 큰 API/cache 불일치를 만든다작은 patch, hotfix test, canary, error rate 비교
CDN purge잘못된 index.html 또는 runtime config가 캐시됐다hashed asset 전체 삭제가 아니라 entry/config만 목표로 purge한다
service worker update오프라인 캐시나 SW가 오래된 asset을 붙잡는다업데이트 알림, skipWaiting 전략, 무한 새로고침 방지 조건을 확인한다
OLD INDEX PROBLEM

사용자가 오래 열린 탭에서 옛 index가 참조하는 chunk를 요청할 수 있습니다. 배포 직후 이전 assets를 바로 삭제하면 chunk load error가 발생합니다. 이전 assets 보존 기간을 둡니다.
