VPN 트러블슈팅

VPN 장애는 “터널이 내려감”과 “터널은 올라왔지만 서비스 통신이 안 됨”을 먼저 분리해야 합니다.

TS-01

분리 원칙

상태의미다음 확인
Phase 1 DOWN상대 인증/관리 채널 협상 실패peer IP, PSK, IKE policy
Phase 2 DOWN데이터 터널 협상 실패local/remote CIDR, PFS, lifetime
Tunnel UP, ping 실패라우팅/방화벽/ICMP 정책 문제 가능route table, return path, ICMP allow
ping 가능, 포트 실패L4 방화벽 또는 서비스 리스닝 문제nc, server listen, ACG
TS-02

터널 DOWN

원인확인
Peer public IP 불일치양쪽 endpoint IP 재확인
PSK 불일치공유 방식과 앞뒤 공백 확인
IKE policy 불일치암호화, hash, DH group, lifetime 비교
Traffic selector 불일치Phase 2 local/remote CIDR 비교
TS-03

터널 UP·통신 실패

우리 route→ 고객 return path→ 우리 firewall→ 고객 firewall

터널 UP은 충분조건이 아닙니다. 양쪽 라우팅과 방화벽이 모두 맞아야 실제 패킷이 왕복합니다.

TS-04

특정 서비스 실패

증상가능 원인확인
DB만 실패5432 방화벽 미허용, DB bind addressnc -zv host 5432
HTTP만 실패앱 오류, proxy, TLS/SNI 문제curl -v, app logs
도메인만 실패DNS resolver/zone 문제dig @resolver name
대량 전송만 실패MTU/MSS 문제 가능fragmentation, PMTUD 확인
TS-05

간헐 장애

Case study

어제는 됐는데 오늘 VPN이 끊겼다면 SA lifetime 만료 후 재협상 실패, 고객사 공인 IP 변경, 장비 재부팅, 방화벽 정책 변경을 의심합니다. “터널 재시작”으로 임시 복구하더라도 Phase 1/2 로그와 변경 이력을 확인해야 같은 장애를 줄일 수 있습니다.

터널 로그evidence
DOWN 시각, 재협상 실패 이유, peer IP 변경 여부를 기록합니다.
변경 이력change
방화벽, route, VPN parameter 변경을 장애 시간대와 대조합니다.
