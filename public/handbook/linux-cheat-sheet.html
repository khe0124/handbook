LX-CS-01

핵심 명령·스니펫

프로세스

# 메모리 많이 먹는 순으로 상위 프로세스
ps aux --sort=-%mem | head

# CPU 많이 먹는 순
ps aux --sort=-%cpu | head

# 실시간 모니터 (htop 있으면 더 편함)
top
htop

# 특정 프로세스 찾기
pgrep -a node
ps aux | grep '[n]ode'

# 정상 종료 요청 (SIGTERM) → 안 죽으면 강제 종료
kill -TERM <pid>
kill -9 <pid>

# 이름으로 종료
pkill -TERM -f 'my-service'

로그

# systemd 서비스 로그 실시간
journalctl -u <svc> -f

# 최근 시간 범위만
journalctl -u <svc> --since '10 min ago'
journalctl --since '2026-07-01 09:00' --until '2026-07-01 10:00'

# 부팅 이후 / 이전 부팅
journalctl -b
journalctl -b -1

# 파일 로그 실시간 추적 (rotation엔 -F)
tail -f app.log
tail -F app.log

# 여러 로그에서 에러 재귀 검색
grep -rn 'err' /var/log
grep -rni 'error\|panic\|fatal' /var/log/myapp/

디스크

# 파일시스템 사용량
df -h
df -i          # inode 고갈은 별개로 확인

# 현재 디렉터리에서 큰 것부터
du -sh * | sort -h
du -xhd1 /var | sort -h

# 삭제됐지만 프로세스가 열고 있어 공간 점유 중인 파일
lsof +L1
주의

rm -rf와 kill -9는 되돌릴 수 없다. 삭제 전 pwd와 대상 경로 확장 결과를 먼저 출력하고, 종료는 kill -TERM으로 시작한다. kill -9는 flush/cleanup을 건너뛴다.

포트·소켓

# 특정 포트를 누가 쓰는지
lsof -i :8080

# LISTEN 중인 TCP 소켓과 프로세스
ss -tlnp

# 모든 연결 상태
ss -tunap

서비스·검색

명령용도주의
systemctl status <svc>서비스 상태·최근 로그·PID 확인Active/exit code/restart 횟수 함께 본다
systemctl restart <svc>서비스 재시작재시작 전 영향 범위·rollback 확인
find / -name '*.log' -mtime -1최근 1일 내 수정된 로그 찾기/ 전체는 느림, 경로 좁히기
free -h메모리·swap 사용량available와 free는 다르다
uptimeload average 확인코어 수 대비로 해석

선택언제 쓰나주의
SIGTERM (kill -15)기본. 프로세스에 정리할 시간을 줌graceful shutdown 로그를 먼저 확인
SIGKILL (kill -9)SIGTERM 실패 시 마지막 수단cleanup hook·flush 없이 즉사, 상태 손상 가능
hard link같은 inode에 이름 여러 개, 원본 지워도 데이터 유지파일시스템 경계·디렉터리 불가
soft link (ln -s)경로를 가리키는 참조, 파일시스템 넘나듦원본 지우면 dangling link
>파일 새로 덮어쓰기기존 내용 즉시 날아감
>>파일 끝에 이어붙이기로그 누적엔 이쪽

cron정해진 시각에 주기 실행환경변수·PATH가 셸과 다름
LX-CS-03

자주 터지는 것 · 즉시 확인

신호왜 위험즉시 확인
디스크 100%인데 df상 용량이 남아 보임삭제됐지만 프로세스가 연 파일이 공간을 점유lsof +L1 → 해당 프로세스 restart
좀비(defunct)·고아 프로세스 누적부모가 wait()로 회수(reaping)하지 않음. 컨테이너에선 PID 1이 init로서 reaping 책임ps aux | grep defunct, 컨테이너면 init(예: tini) 여부
프로세스가 갑자기 죽고 로그에 흔적 없음OOM killer가 메모리 부족으로 강제 종료dmesg | grep -i 'oom\|killed', journalctl -k
chmod 777 남용누구나 쓰기·실행 가능, 권한 문제를 덮기만 함id, stat, namei -l <path>로 실제 원인
로그 로테이션 부재단일 로그 파일이 디스크를 서서히 채움ls -lh로 크기, logrotate 설정 확인
보는 로그 파일과 앱이 쓰는 파일이 다름rotation으로 inode가 바뀜tail -F, journalctl --since
LX-CS-04

장애 증거 1카드

DISK FULL TRIAGE1. df -h · df -i로 block/inode 중 무엇이 찼는지 구분
2. du -xhd1 /path | sort -h로 큰 디렉터리 추적
3. lsof +L1로 삭제됐지만 열린 파일 확인 (핵심 함정)
4. 로그·코어덤프(/var/log, core.*) 크기 확인
5. 백업·owner·서비스 영향 확인 후 안전 삭제
순서질문증거
1block이 찼나 inode가 찼나?df -h, df -i의 Use%
2어느 디렉터리가 큰가?du -xhd1 /path | sort -h
3지웠는데 공간이 안 돌아오나?lsof +L1의 열린 삭제 파일
4로그·덤프가 원인인가?큰 .log, core.* 파일
복구·중단

3번에서 열린 삭제 파일이 잡히면 파일을 더 지우지 말고 해당 프로세스를 restart해 fd를 놓게 한다. 대상 경로를 눈으로 확인하지 못하면 삭제를 중단한다.
