LX-CS-01

개요

운영 조사에서는 명령을 많이 아는 것보다 현재 위치, 대상 파일, 프로세스, 포트, 로그 범위를 안전하게 좁히는 것이 중요합니다.

상황먼저 확인할 것주의할 점
파일 찾기pwd, ls -la, find, rg대상 경로와 숨김 파일 포함 여부 확인
포트 점유lsof -i :3000, ss -tlnplocalhost bind인지 0.0.0.0 bind인지 구분
로그 확인tail -f, less, grep, journalctl시간대, rotation, service unit 확인
리소스 확인df -h, du -sh *, top, free -h디스크 full과 inode full은 별개
권한 오류ls -l, id, stat, namei -l파일뿐 아니라 상위 디렉터리 실행 권한도 확인
LX-CS-02

핵심 명령/개념

명령압축 설명해석 포인트
rg "text"현재 디렉터리 이하에서 빠른 텍스트 검색.gitignore를 따르므로 숨은 대상은 옵션 확인
less app.log큰 파일을 페이지 단위로 탐색/pattern, Shift+F로 검색/추적
tail -f app.log로그를 실시간으로 따라감rotation 환경에서는 -F가 더 적합할 수 있음

du -sh * | sort -rh용량 큰 항목을 정렬현재 디렉터리 기준이라 mount point를 확인

LX-CS-03

판단 기준

선택기준주의점
cat작은 파일 전체를 즉시 볼 때큰 로그에는 사용하지 않음
less큰 파일을 검색하며 볼 때원본을 수정하지 않고 안전하게 탐색
tail -f새로 쌓이는 로그를 실시간 확인할 때과거 원인은 less/journalctl --since가 낫다
kill정상 종료 요청이 가능할 때데이터 flush, connection drain 기회를 줌
kill -9정상 종료가 실패한 마지막 수단cleanup hook이 실행되지 않음
sudo권한 상승이 명확히 필요한 작업원인 확인 없이 권한 문제를 덮지 않음
LX-CS-04

자주 나는 실수

  • rm -rf 전에 대상 경로를 확인하지 않는다. 절대경로, 현재 디렉터리, 변수 확장을 먼저 출력해 본다.
  • 큰 로그를 cat으로 터미널에 모두 쏟는다. 검색과 paging이 필요한 파일은 less를 쓴다.
  • kill -9를 먼저 써서 graceful shutdown 기회를 없앤다. lock file과 임시 파일이 남을 수 있다.
  • 권한 문제를 무조건 sudo로 덮고 원인을 보지 않는다. 소유자, 그룹, 상위 디렉터리 권한을 확인한다.
  • macOS BSD 명령과 GNU/Linux 옵션 차이를 고려하지 않는다. sed, stat, date 옵션이 다를 수 있다.
  • 디스크 여유만 보고 inode 고갈을 놓친다. 작은 파일이 많으면 df -i도 확인한다.
LX-CS-05

면접/리뷰용 한 줄 답변

파이프pipe
셸의 힘은 작은 명령의 표준 출력을 다음 명령의 표준 입력으로 연결해 조합하는 데 있습니다.
권한permission
Unix 권한은 소유자, 그룹, 기타 사용자에게 읽기, 쓰기, 실행 권한을 나눠 부여하는 모델입니다.
프로세스 종료signal
먼저 SIGTERM으로 정리할 시간을 주고, 실패할 때만 SIGKILL을 사용합니다.
로그 조사triage
로그는 최신 줄만 보는 것이 아니라 배포 시각, 오류 최초 발생 시각, 같은 requestId의 흐름을 함께 봐야 합니다.
리소스system
CPU, 메모리, 디스크, 네트워크는 각각 병목 신호가 다르므로 한 명령으로 시스템 상태를 단정하지 않습니다.
LX-CS-06

Red Flags

신호왜 위험한가즉시 확인
rm -rf에 변수/와일드카드 사용대상 경로 오해 시 대량 삭제pwd, printf '%s\n' "$TARGET", dry-run
disk full인데 삭제 후 용량 회복 안 됨삭제된 파일을 프로세스가 열고 있을 수 있음lsof +L1, process restart
kill -9 먼저 사용flush/cleanup 없이 종료되어 상태가 손상될 수 있음kill, graceful shutdown log
권한 오류를 sudo로만 해결소유권/umask/deploy user 문제가 숨어 남음id, namei -l, stat
log rotation 무시보고 있는 파일과 앱이 쓰는 파일이 다를 수 있음tail -F, journalctl --since
LX-CS-07

Review Gate

작업차단 기준필수 증거
삭제 명령대상 경로 출력과 백업/복구 기준 없음dry-run, backup, owner 확인
서비스 재시작graceful shutdown, 영향 범위, rollback 미정systemd unit, health check, logs
권한 변경recursive chmod/chown 범위 불명확대상 tree, user/group, 최소 권한
디스크 정리큰 파일 원인과 재발 방지 없음du, df -h, df -i, logrotate
운영 조사시간대, host, service, request id가 기록되지 않음명령 로그, timestamp, hostname
LX-CS-08

Failure Playbook

DISK FULL TRIAGE1. df -h와 df -i로 block/inode 확인
2. du -xhd1 /path | sort -h로 큰 디렉터리 확인
3. lsof +L1로 삭제됐지만 열린 파일 확인
4. 로그면 logrotate와 retention 확인
5. 삭제 전 백업/owner/서비스 영향 확인
6. 정리 후 재발 방지 알림과 보존 정책 추가
운영 원칙

서버 명령은 빠르지만 되돌리기 어렵습니다. destructive command는 항상 현재 경로, 대상 확장 결과, 백업 가능성, 서비스 영향 범위를 확인한 뒤 실행합니다.

LX-CS-09

운영 명령 카드

Linux 명령은 결과를 얻는 속도보다 실수로 증거를 지우지 않는 것이 중요합니다. 삭제·재시작·권한 변경 전에는 항상 현재 위치와 대상 범위를 출력합니다.

상황명령정상 신호다음 행동
디스크 fulldf -h, df -i, du -xhd1 /path | sort -hblock과 inode 중 병목이 구분됨큰 파일, 작은 파일 폭증, 열린 삭제 파일을 분리
로그 조사journalctl -u service --since "30 min ago"배포 시각과 오류 최초 발생 시각이 맞춰짐request id 또는 error fingerprint로 좁힘
포트 충돌lsof -i :PORT, ss -tlnp프로세스와 bind address가 확인됨localhost/0.0.0.0/container port를 구분
권한 오류id, stat path, namei -l path파일과 상위 디렉터리 권한이 함께 보임sudo로 덮기 전 owner/group/umask 원인 수정
프로세스 종료kill PID 후 로그 확인graceful shutdown 로그와 health 회복 확인실패할 때만 kill -9 검토
LX-CS-10

안전 판정표

작업실행 전 확인위험 신호중단 기준
삭제pwd, 대상 경로 출력, 백업/복구 가능성변수, wildcard, 빈 문자열 확장대상 목록을 눈으로 확인하지 못하면 중단
권한 변경현재 owner/group, 재귀 범위, 서비스 사용자chmod -R 777, 광범위 chown최소 권한 설명 없으면 중단

로그 압축/전달시간 범위, 민감 정보, request idsecret/token 포함 가능마스킹 기준 없이 공유 금지
