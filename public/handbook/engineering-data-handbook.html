DB-01

관계형 모델과 정규화

관계형 DB는 1970년 Edgar Codd의 논문에서 나온 수학적 모델입니다. 핵심 발상 하나 — 데이터를 관계(relation, 곧 테이블)로 표현하고, 행 사이의 연결을 포인터가 아니라 값(key)으로 맺는다. 이 "값으로 잇는다"가 무결성과 유연한 조회의 출발점입니다.

테이블은 행(row, tuple)의 집합이고, 각 행은 열(column, attribute)로 구성됩니다. 행 사이엔 순서가 없고, 같은 행이 중복으로 들어갈 수 없다는 게 "집합"이라는 말의 뜻입니다 — 그래서 각 행을 유일하게 식별할 기본키(PK)가 필요합니다.

제약무엇을 보장하는가 / 왜 존재
PRIMARY KEY행을 유일하게 식별. NOT NULL + UNIQUE가 합쳐진 것. 한 테이블에 하나
FOREIGN KEY다른 테이블의 PK를 참조 — 참조 무결성. 존재하지 않는 부모를 가리키는 자식을 DB가 막음
UNIQUE중복 값 금지 (이메일 등). PK가 아닌 후보키에 사용
NOT NULL값 누락 금지. "모른다"와 "없다"를 코드가 아니라 DB가 강제
CHECK값의 도메인 제약 (예: price >= 0). 비즈니스 규칙의 마지막 방어선

정규화(normalization)는 이 구조를 단계적으로 다듬는 절차입니다. 목적은 단 하나 — 중복을 없애 갱신 이상(update anomaly)을 막는 것. 같은 사실이 여러 곳에 적혀 있으면, 하나만 고치고 나머지를 놓치는 순간 데이터가 거짓말을 합니다.

정규형규칙 / 제거하는 문제
1NF각 칸은 원자값 하나 — 한 칸에 콤마로 여러 값 금지, 반복 그룹 금지
2NF1NF + 비키 컬럼이 복합키 전체에 종속 (부분 종속 제거)
3NF2NF + 비키 컬럼끼리의 종속 제거 (이행적 종속 — A→B→C 꼴)
갱신 이상이 뭔가

주문 테이블에 customer_name을 매 행에 박아두면, 고객이 개명했을 때 그 고객의 모든 주문 행을 고쳐야 합니다(갱신 이상). 한 행이라도 놓치면 같은 고객이 두 이름을 갖습니다. 고객을 별도 테이블로 빼고 customer_id(FK)로 참조하면 이름은 한 곳에만 — 고치는 곳도 한 곳입니다. 이게 정규화가 실제로 사주는 것입니다.

비정규화는 측정 후

조인이 많아 느리다는 이유로 컬럼을 다시 합치는 비정규화(denormalization)는 "읽기 성능을 위해 쓰기 정합성을 포기"하는 거래입니다. 중복된 값을 동기화하는 책임이 애플리케이션으로 넘어옵니다. 먼저 인덱스·쿼리로 풀 수 있는지 보고(→ DB-05, DB-06), 실제 측정으로 병목이 조인임이 확인된 뒤에만 하세요. "조인이 비쌀 것 같아서"는 근거가 아닙니다.

DB-02

트랜잭션과 ACID

트랜잭션은 "전부 되거나, 전부 안 되거나"를 보장하는 작업 묶음입니다. 계좌 이체에서 출금은 됐는데 입금이 실패하면 돈이 증발합니다 — 트랜잭션은 이런 부분 실패를 원천 차단하기 위해 존재합니다. ACID는 그 보장의 네 가지 측면입니다.

속성의미 / 무엇을 막는가
Atomicity (원자성)묶음 안의 연산은 모두 성공하거나 모두 취소. 중간 상태가 남지 않음 — 부분 실패 방지
Consistency (일관성)트랜잭션 전후로 제약(PK·FK·CHECK)이 항상 유효. 무결성 규칙을 깨는 상태로 못 끝남
Isolation (격리성)동시에 도는 트랜잭션들이 서로의 중간 상태를 안 봄 — 마치 혼자 도는 것처럼(→ DB-03)
Durability (지속성)커밋된 결과는 장애·정전에도 살아남음. 디스크에 안전하게 기록됨

원자성과 지속성을 동시에 어떻게 보장할까요. 핵심 메커니즘이 WAL(Write-Ahead Logging)입니다 — 데이터 파일을 직접 고치기 전에, 변경 내용을 먼저 로그에 순차 기록(append)합니다. 로그는 순차 쓰기라 빠르고, 일단 로그에 커밋 기록이 남으면 그게 "지속성의 증거"입니다.

COMMIT 시점에 일어나는 일 (WAL) 1. 변경을 WAL 로그에 append (순차 쓰기 — 빠름)
2. COMMIT 레코드를 로그에 기록 → fsync (여기서부터 durable)
3. 실제 데이터 페이지는 나중에 천천히 디스크로 (background flush)

장애 복구 재시작 시 WAL을 재생(replay) → 커밋된 것은 redo, 미완은 버림

트랜잭션은 BEGIN으로 열고 COMMIT으로 확정하거나 ROLLBACK으로 통째로 취소합니다. JPA에서는 보통 @Transactional 메서드의 경계가 곧 트랜잭션 경계입니다 — 메서드가 정상 반환하면 commit, unchecked exception이 던져지면 자동 rollback.

트랜잭션 경계 = 비즈니스 단위

흔한 실패 모드 둘. ① 경계를 너무 넓게 잡아 외부 API 호출·이메일 전송까지 트랜잭션 안에 넣으면, DB 커넥션을 잡은 채 네트워크를 기다려 커넥션 풀이 고갈됩니다(→ DB-07). ② 경계를 너무 좁게 잡아 "원자적이어야 할 두 쓰기"를 별도 트랜잭션으로 쪼개면 부분 실패가 생깁니다. 기준은 단순합니다 — "함께 성공하거나 함께 실패해야 하는 한 덩어리"가 하나의 트랜잭션입니다. 외부 I/O는 트랜잭션 밖으로 빼세요.

DB-03

격리수준과 동시성 이상

완벽한 격리(모두가 한 줄로 줄 서서 차례로 실행)는 안전하지만 느립니다. 그래서 SQL 표준은 "어디까지 동시성 이상을 허용하고 그 대가로 얼마나 빠를 것인가"를 고르는 격리수준을 둡니다. 격리는 안전성과 성능 사이의 다이얼입니다.

먼저 막으려는 이상 현상(anomaly) 세 가지를 알아야 격리수준이 읽힙니다.

이상무슨 일
Dirty read다른 트랜잭션이 아직 커밋 안 한 값을 읽음. 그 트랜잭션이 롤백하면 존재한 적 없는 값을 본 셈
Non-repeatable read같은 행을 두 번 읽었는데 값이 다름 — 사이에 다른 트랜잭션이 그 행을 수정·커밋
Phantom read같은 조건으로 두 번 조회했는데 행의 개수가 다름 — 사이에 조건에 맞는 행이 추가·삭제됨

위로 갈수록 안전하지만 동시성(처리량)은 떨어집니다. 기본값은 DB마다 다릅니다 — PostgreSQL은 Read Committed, MySQL InnoDB는 Repeatable Read가 기본입니다. 이걸 모르고 옮기면 같은 코드가 다르게 동작합니다.

MVCC — 읽기는 락을 안 잡는다

표준 정의대로 락으로 격리를 구현하면 읽기가 쓰기를 막아 느립니다. 대신 PostgreSQL·MySQL(InnoDB)은 MVCC(Multi-Version Concurrency Control)를 씁니다 — 행을 수정하면 덮어쓰지 않고 새 버전을 만들고, 각 트랜잭션은 자기가 시작한 시점의 스냅샷을 읽습니다. 그래서 "읽는 사람은 쓰는 사람을 기다리지 않고, 쓰는 사람은 읽는 사람을 막지 않습니다". *PostgreSQL의 Repeatable Read(스냅샷 격리)는 표준과 달리 phantom도 대부분 막습니다. MVCC의 비용은 죽은 버전이 쌓이는 것이라, 정리 작업(PostgreSQL의 VACUUM)이 필요합니다.

흔한 실패 모드

Read Committed에서 "잔액을 읽고 → 계산해서 → 다시 쓰는" 패턴은 lost update를 부릅니다. 두 트랜잭션이 같은 잔액을 읽고 각자 갱신하면 한쪽 변경이 사라집니다. 격리수준을 올리거나, 낙관적/비관적 락으로 막아야 합니다(→ DB-04). "격리수준을 SERIALIZABLE로 올리면 다 해결"은 맞지만, 직렬화 실패(serialization failure)로 트랜잭션이 거부될 수 있어 재시도 로직이 필요합니다.

DB-04

락과 데드락

MVCC가 읽기를 자유롭게 해줘도, 같은 행을 동시에 쓰는 충돌은 결국 누군가 기다려야 합니다. 락은 그 충돌을 직렬화하는 도구이고, 데드락은 그 도구를 잘못 써서 서로를 무한정 기다리는 상태입니다.

구분의미
공유 락 (S, read)여러 트랜잭션이 동시에 가질 수 있음. 읽는 동안 변경을 막음
배타 락 (X, write)한 트랜잭션만 가짐. 쓰는 동안 다른 읽기·쓰기를 막음
행 락 (row)해당 행만 잠금 — 동시성 높음. 일반적 단위
테이블 락 (table)테이블 전체 잠금 — 동시성 낮음. DDL·대량 작업 시

충돌을 다루는 두 전략이 있습니다. 낙관적 락(optimistic)은 "충돌은 드물다"고 가정하고, 일단 진행한 뒤 커밋 시점에 충돌을 검사합니다. 비관적 락(pessimistic)은 "충돌이 잦다"고 보고, 처음부터 락을 잡아 남이 못 건드리게 합니다.

전략방식 / 언제
낙관적JPA @Version 컬럼 — 읽은 버전과 커밋 시 버전이 다르면 OptimisticLockException. 충돌 드물고 읽기 많을 때. 락을 안 잡아 빠르지만 재시도 필요
비관적SELECT ... FOR UPDATE (JPA PESSIMISTIC_WRITE) — 조회 즉시 배타 락. 충돌 잦고 재시도 비용이 클 때. 안전하지만 대기·데드락 위험
DEADLOCK — 서로 엇갈린 락 순서 T1: lock A ───▶ wants B (T2가 쥐고 있음 → 대기)
T2: lock B ───▶ wants A (T1이 쥐고 있음 → 대기)
→ 둘 다 영원히 대기. DB가 감지해 한쪽을 victim으로 abort
데드락 예방 — 락 순서를 일관되게

데드락의 거의 모든 원인은 트랜잭션마다 락을 잡는 순서가 다른 것입니다. 위 그림처럼 한쪽은 A→B, 다른 쪽은 B→A로 잡으면 엇갈립니다. 예방의 핵심은 모든 트랜잭션이 자원을 항상 같은 순서로 잠그는 것(예: 항상 작은 id부터). 부수적으로 ① 트랜잭션을 짧게 유지 ② 필요한 행만 좁게 잠그기 ③ 데드락은 완전히 못 막을 수 있으니 감지 시 재시도 로직을 두기. DB는 데드락을 감지하면 한 트랜잭션을 죽이고 에러를 던지니, 그걸 잡아 재시도하면 됩니다.

DB-05

인덱스 심화

인덱스가 없으면 DB는 원하는 행을 찾으려 테이블을 처음부터 끝까지 훑습니다(full table scan, O(n)). 인덱스는 특정 컬럼으로 정렬해둔 별도의 사본이라, 이진 탐색처럼 O(log n)에 위치를 찾습니다. 책 뒤의 색인과 같은 원리입니다.

대부분의 RDBMS 인덱스는 B-tree(정확히는 B+tree) 구조입니다 — 정렬된 균형 트리라 탐색·범위 조회·정렬에 모두 강합니다. "정렬되어 있다"는 게 핵심이고, 이 성질에서 인덱스의 거의 모든 규칙이 파생됩니다.

개념의미 / 왜 중요
복합 인덱스 선두 컬럼 규칙(a, b) 인덱스는 a 또는 a,b 조건엔 쓰이지만 b만 거는 조건엔 안 쓰임 — 전화번호부가 성→이름 순이라 이름만으론 못 찾는 것과 같음
커버링 인덱스쿼리가 필요한 컬럼이 인덱스에 다 있으면 테이블 본체를 안 봐도 됨 — 더 빠름
카디널리티컬럼 값의 다양성. 성별(2종)처럼 낮으면 인덱스 효과 적고, 이메일처럼 높으면 큼
클러스터드 vs 논클러스터드클러스터드는 테이블 데이터 자체가 그 키 순으로 정렬됨(테이블당 1개). 논클러스터드는 별도 구조 + 본체 포인터
인덱스를 무력화하는 패턴

인덱스는 "정렬된 컬럼 값 그대로"일 때만 쓰입니다. 그래서 ① 컬럼에 함수·연산을 걸면(WHERE YEAR(created_at) = 2026) 정렬이 깨져 인덱스를 못 씁니다 — created_at >= '2026-01-01' AND < '2027-01-01'처럼 컬럼을 가공하지 말고 범위로 바꿔야 합니다. ② 앞 와일드카드(LIKE '%keyword')도 마찬가지 — 정렬된 사본에서 "앞이 뭔지 모르는" 검색은 무용지물입니다. LIKE 'keyword%'(뒤 와일드카드)는 씁니다.

인덱스는 공짜가 아니다 — 쓰기 비용

인덱스는 읽기를 빠르게 하는 대신 쓰기를 느리게 합니다. 행을 INSERT/UPDATE/DELETE할 때마다 모든 관련 인덱스도 갱신해야 하고, 저장 공간도 더 씁니다. 그래서 "혹시 몰라서 다 거는" 인덱스는 안티패턴입니다. 실제 쿼리 패턴(WHERE·JOIN·ORDER BY에 자주 나오는 컬럼)과 카디널리티를 보고 거세요. 카디널리티가 낮은 컬럼(예: boolean 상태) 단독 인덱스는 대개 효과가 없습니다.

DB-06

실행계획과 쿼리 최적화

SQL은 "무엇을 원하는지"만 적고 "어떻게 가져올지"는 DB의 옵티마이저가 정합니다. 그 결정을 들여다보는 창이 EXPLAIN입니다. 느린 쿼리를 추측으로 고치지 말고, 실행계획을 읽어 병목을 짚으세요.

EXPLAIN(PostgreSQL은 EXPLAIN ANALYZE로 실제 실행까지)은 옵티마이저가 고른 접근 방법을 보여줍니다. 가장 먼저 볼 것은 스캔 방식과 예상 행 수입니다.

실행계획 항목읽는 법
Seq Scan (full scan)테이블 전체를 훑음. 작은 테이블이면 정상, 큰 테이블에서 나오면 인덱스 부재 신호
Index Scan인덱스로 좁혀 접근. 보통 원하는 그림
예상 행수 (rows)옵티마이저의 추정치. 실제(ANALYZE)와 크게 다르면 통계가 낡았다는 뜻 — 추정 오류가 나쁜 계획을 부름
cost상대적 비용 추정. 절대 시간이 아니라 계획 간 비교용

ORM 사용 시 가장 흔하고 비싼 함정은 N+1 문제입니다. 목록 1건을 조회한 뒤(쿼리 1번), 각 항목의 연관 엔티티를 가져오려 항목 수(N)만큼 추가 쿼리가 나가는 패턴입니다 — 100건이면 101번의 쿼리.

N+1 — JPA에서 흔히 // 주문 100건 조회 ──▶ SELECT * FROM orders (쿼리 1번)
// 각 주문의 customer 접근 ──▶ SELECT * FROM customer WHERE id=? × 100
→ 총 101 쿼리. 해결: fetch join 한 번으로
SELECT o FROM Order o JOIN FETCH o.customer (쿼리 1번)

해결책은 fetch join(연관을 한 번의 조인으로 함께 로딩)이나 batch fetching(IN 절로 N건을 한 번에)입니다. JPA에선 기본을 lazy로 두고 필요할 때 fetch join을 쓰는 게 정석입니다.

페이지네이션 — offset vs keyset

LIMIT 20 OFFSET 100000 방식은 뒤로 갈수록 느려집니다 — DB가 건너뛸 10만 행을 일단 다 읽고 버리기 때문입니다(O(offset)). 깊은 페이지가 필요하면 keyset 페이지네이션(cursor)으로 바꾸세요 — WHERE id > :last_id ORDER BY id LIMIT 20처럼 "마지막으로 본 값" 이후부터 인덱스로 바로 진입합니다. offset과 달리 페이지가 깊어져도 일정한 속도입니다. 트레이드오프는 "임의 페이지 번호로 점프"가 안 된다는 것 — 무한 스크롤·다음 버튼엔 keyset이 맞습니다.

DB-07

커넥션 풀

DB 커넥션 하나를 새로 맺는 건 비쌉니다 — TCP 핸드셰이크, 인증, 세션 설정이 매번 일어납니다(→ 백엔드 개요의 네트워킹). 요청마다 새로 맺으면 그 비용이 쌓입니다. 커넥션 풀은 미리 만들어둔 커넥션을 빌려주고 돌려받아 이 비용을 없앱니다.

애플리케이션이 시작할 때 N개의 커넥션을 만들어 풀에 담아두고, 요청이 오면 빌려서(borrow) 쓰고 끝나면 반납합니다(return). Spring Boot의 기본 풀은 HikariCP입니다.

설정 / 개념의미
maximumPoolSize풀이 유지할 최대 커넥션 수. 가장 중요하고 가장 자주 틀리는 값
connectionTimeout빈 커넥션을 기다리는 최대 시간. 초과 시 예외 — 고갈 신호
idleTimeout / maxLifetime유휴 커넥션 회수 시간 / 커넥션 최대 수명(주기적 재생성)
풀 크기 산정 원리 — 클수록 좋은 게 아니다

직관과 반대로, 풀을 크게 잡는다고 처리량이 오르지 않습니다. 커넥션은 결국 DB의 CPU·디스크라는 유한 자원을 두고 경쟁하고, 커넥션이 코어 수보다 훨씬 많으면 컨텍스트 스위칭·락 경합으로 오히려 느려집니다. HikariCP가 인용하는 경험식은 커넥션 수 ≈ (코어 수 × 2) + 유효 스핀들 수 수준 — 보통 수십 개면 충분하고, 수백 개는 거의 항상 잘못된 설정입니다. 병목이 커넥션 부족인지 DB 자체 처리량인지를 측정으로 먼저 구분하세요.

누수와 고갈 — 증상과 원인

증상은 connectionTimeout 초과 예외가 산발적으로 터지고, 부하가 올라가면 응답이 멈추는 것입니다. 원인은 둘 — ① 커넥션 누수: 빌린 커넥션을 반납 안 함(예외 경로에서 close 누락). 풀이 점점 비어 결국 고갈됩니다. ② 트랜잭션이 너무 김: 트랜잭션 안에서 외부 API 호출 같은 느린 I/O를 하면(→ DB-02) 커넥션을 오래 쥐고 있어 동시 요청이 풀을 다 써버립니다. @Transactional 메서드에서 외부 호출을 빼는 것만으로 많은 고갈이 사라집니다.

DB-08

스키마 설계와 마이그레이션

스키마는 코드보다 바꾸기 어렵습니다 — 이미 쌓인 데이터가 있고, 운영 중인 서비스가 그 위에서 돕니다. 그래서 설계 단계의 트레이드오프와, 변경을 안전하게 굴리는 절차가 둘 다 중요합니다.

설계의 기본 긴장은 정규화 vs 성능입니다(→ DB-01). 정규화는 정합성을, 비정규화는 읽기 속도를 삽니다. 출발은 정규화(3NF)이고, 측정으로 정당화될 때만 선택적으로 푸는 게 안전한 기본값입니다.

스키마 변경은 손으로 ALTER TABLE을 치는 게 아니라 버전 관리되는 마이그레이션으로 합니다 — 변경을 파일로 남겨 코드처럼 리뷰·재현하기 위해서입니다.

도구특징
FlywaySQL 파일에 버전 번호(V1__init.sql)를 매겨 순서대로 적용. 단순·명시적. SQL 그대로 작성
LiquibaseXML/YAML/SQL 등 changelog로 변경 기술. DB 독립적 추상화·rollback 정의 지원. 더 무겁고 유연
무중단 스키마 변경 — expand-contract

운영 중 컬럼 이름을 바꾸거나 NOT NULL을 추가하면, 변경 직후~배포 완료 사이에 구버전 코드와 신버전 스키마가 공존해 깨집니다. 그래서 변경을 한 번에 하지 않고 단계로 나눕니다(expand-contract): ① expand — 새 컬럼을 nullable로 추가(구코드와 호환) ② 코드가 양쪽에 쓰고 데이터를 backfill ③ 새 코드로 전환 ④ contract — 다 옮긴 뒤 구 컬럼 제거. 각 단계가 이전 버전과 호환되므로 무중단으로 굴러갑니다. "컬럼 rename 한 줄"처럼 보여도 운영에선 이 절차가 필요합니다.

DB-09

확장 — 복제·파티셔닝·샤딩

한 대의 DB로 부하를 못 감당할 때 세 가지 길이 있습니다 — 복사본을 더 두거나(복제), 한 테이블을 쪼개 한 서버 안에서 나누거나(파티셔닝), 데이터를 여러 서버로 분산하거나(샤딩). 뒤로 갈수록 강력하지만 복잡하고, 잃는 것도 많습니다.

전략해결 / 트레이드오프
복제 (replication)primary의 사본(replica)을 두고 읽기를 분산. 읽기 많은 워크로드에 효과적. 비용은 복제 지연
파티셔닝 (partitioning)큰 테이블을 기준(날짜·범위 등)으로 조각냄 — 한 서버 안에서. 오래된 파티션 관리·스캔 범위 축소에 유리
샤딩 (sharding)데이터를 여러 서버로 수평 분산(샤드 키 기준). 쓰기까지 확장. 가장 강력하고 가장 복잡
복제 지연 (replication lag)

읽기 복제본은 primary의 변경을 비동기로 따라잡습니다 — 그래서 방금 primary에 쓴 데이터가 replica엔 아직 없을 수 있습니다. "글을 쓰고 바로 새로고침했는데 안 보인다"가 전형적 증상입니다. 정합성이 중요한 읽기(쓰기 직후 본인 확인 등)는 primary로 보내거나, 복제가 따라잡을 때까지 기다리는 처리가 필요합니다. 모든 읽기를 무조건 replica로 보내면 안 됩니다.

샤딩의 진짜 비용 — 조인과 트랜잭션

샤딩은 데이터를 여러 서버로 갈라놓기 때문에, 단일 서버에선 공짜였던 두 가지가 어려워집니다. ① 샤드를 가로지르는 조인 — 서로 다른 서버의 데이터를 합쳐야 해 애플리케이션이 직접 모으거나 비싸집니다. ② 분산 트랜잭션 — 여러 샤드에 걸친 원자성 보장은 단일 노드 ACID로 안 되고, 복잡한 프로토콜(2PC)이나 saga 패턴이 필요합니다. 그래서 샤딩은 최후의 수단입니다 — 복제·파티셔닝·인덱스·캐시로 안 풀릴 때만, 그리고 샤드 키를 잘못 고르면 되돌리기 매우 어렵습니다. 쓰기 확장이 정말 필요한지부터 확인하세요.

DB-10

NoSQL은 언제

NoSQL은 "RDBMS의 상위 호환"이 아니라 다른 트레이드오프입니다. 관계형이 정합성과 유연한 조회를 위해 포기한 것들(수평 확장성, 스키마 유연성)을 가져오는 대신, 관계형이 공짜로 주던 것(조인, 강한 일관성, 트랜잭션)을 일부 내려놓습니다. "유행이라서"가 아니라 워크로드가 맞을 때 씁니다.

유형모델 / 대표 용도
DocumentJSON 형태 문서 (MongoDB). 스키마 유연, 한 문서로 읽는 집계 데이터
Key-Value키로 값 조회 (Redis). 캐시·세션·단순 초고속 조회
Wide-column행마다 컬럼이 다를 수 있는 대규모 분산 (Cassandra). 쓰기 폭주·시계열
Graph노드·간선으로 관계 자체를 1급 시민으로 (Neo4j). 소셜 그래프·추천·경로 탐색
CAP — 분산 시스템의 근본 제약

네트워크가 분리(Partition)될 수 있는 분산 시스템에선 일관성(Consistency)과 가용성(Availability)을 동시에 완벽히 만족할 수 없습니다(CAP 정리). 분리가 일어났을 때, 옛 데이터를 줄지언정 응답할 것인가(AP)와, 정확하지 않으면 차라리 거부할 것인가(CP) 중 골라야 합니다. 많은 NoSQL이 가용성·확장성을 위해 강한 일관성 대신 결과적 일관성(eventual consistency)을 택한 게 이 트레이드오프입니다 — 단, 이건 "분리 상황"에서의 선택이지 평상시 일관성을 늘 버린다는 뜻은 아닙니다.

선택 기준 — 폴리글랏 퍼시스턴스

현실의 답은 보통 "둘 중 하나"가 아니라 폴리글랏 퍼시스턴스(polyglot persistence) — 데이터 성격에 맞는 저장소를 섞어 씁니다. 핵심 거래·정합성이 중요한 데이터는 RDBMS, 캐시·세션은 Redis, 전문 검색은 Elasticsearch처럼. 기본값은 여전히 RDBMS로 두는 게 안전합니다 — 조인·트랜잭션·강한 일관성이 필요한지부터 묻고, 그게 필요 없고 특정 차원(쓰기 확장·스키마 유연·그래프 탐색)이 지배적일 때 NoSQL을 더하세요. 주문·정산 같은 거래·감사 추적 도메인은 RDBMS가 자연스러운 중심입니다.

DB-11

용어집

핸드북 전체에서 반복되는 핵심 용어. 정의를 한 줄로 잡아두면 본문이 빠르게 읽힙니다.

트랜잭션transaction
전부 성공하거나 전부 취소되는 작업 묶음. BEGIN~COMMIT/ROLLBACK. 경계는 "함께 성공·실패해야 할 비즈니스 단위"(→ DB-02)
ACIDatomicity·…
트랜잭션이 보장하는 네 속성 — 원자성·일관성·격리성·지속성. 부분 실패와 동시성 오류를 막음
WALwrite-ahead log
데이터 파일을 고치기 전에 변경을 로그에 먼저 순차 기록. 원자성·지속성과 장애 복구의 토대(→ DB-02)
격리수준isolation level
동시성 이상을 어디까지 허용할지의 다이얼. Read Committed(PG 기본)~Serializable. 안전성↔성능 트레이드오프(→ DB-03)
MVCCmulti-version cc
수정 시 새 버전을 만들고 트랜잭션은 시작 시점 스냅샷을 읽는 방식. 읽기가 쓰기를 막지 않음(→ DB-03)
phantom readanomaly

데드락deadlock

DB-13

락과 MVCC를 실제로 읽기

MVCC와 락은 정의를 외우면 쉬워 보이지만, 운영에서는 “왜 이 UPDATE가 멈췄는가”, “왜 방금 쓴 값이 안 보이는가”, “왜 vacuum이 밀리는가”를 설명해야 합니다.

MVCC는 한 행을 덮어쓰지 않고 새 버전을 만들어 읽기와 쓰기를 동시에 처리합니다. 읽는 트랜잭션은 자기 스냅샷에서 보이는 버전을 읽고, 쓰는 트랜잭션은 새 버전을 만듭니다. 그래서 읽기가 쓰기를 막지 않는 대신, 오래 열린 트랜잭션이 있으면 오래된 row version을 지울 수 없어 bloat가 커집니다.

현상겉으로 보이는 증상실제 확인
lock wait쿼리가 CPU를 안 쓰는데 오래 걸림blocking transaction, wait_event, 대기 중인 row/table lock
deadlock두 트랜잭션 중 하나가 강제로 실패서로 다른 순서로 같은 자원 잠금
long transactionvacuum 지연, table/index bloat 증가트랜잭션 시작 시각, idle in transaction
replica lag쓰기 직후 읽기에서 데이터가 안 보임primary와 replica replay lag, read-after-write 요구
LOCK TRIAGE QUESTION 1. 느린 쿼리가 실제로 실행 중인가, 락을 기다리는가?
2. 누가 blocking transaction인가?
3. 그 트랜잭션은 언제 시작했고 어떤 SQL을 실행했는가?
4. 같은 테이블을 항상 같은 순서로 잠그는가?
5. 실패시 재시도해도 안전한가?
면접 포인트

“MVCC는 읽기와 쓰기가 서로 막지 않게 합니다”에서 멈추면 얕습니다. 이어서 “대신 오래 열린 트랜잭션이 오래된 버전을 붙잡아 vacuum과 bloat에 영향을 주고, UPDATE끼리는 여전히 같은 row에 대해 lock 경합이 생긴다”고 말해야 실제 운영 이해가 드러납니다.

DB-14

실행계획 해석 훈련

실행계획은 DB가 쿼리를 어떻게 풀기로 결정했는지 보여주는 지도입니다. 인덱스를 탔는지만 보는 것이 아니라 예상 행 수, 실제 행 수, join 방식, sort, buffer 사용을 같이 봐야 합니다.

예를 들어 WHERE tenant_id = ? AND status = ? ORDER BY created_at DESC LIMIT 20 쿼리에는 보통 (tenant_id, status, created_at DESC) 형태의 복합 인덱스가 후보가 됩니다. 하지만 status 카디널리티가 낮고 tenant별 데이터 분포가 크게 다르면 순서와 partial index를 다시 봐야 합니다. 인덱스 설계는 “조건에 나온 컬럼을 다 넣기”가 아니라 선택도, 정렬, limit, 쓰기 비용을 함께 계산하는 일입니다.

면접 답변

느린 쿼리를 보면 먼저 EXPLAIN (ANALYZE, BUFFERS)로 실제 실행 시간을 확인하고, 예상 행 수와 실제 행 수 차이, scan 방식, join 방식, sort와 buffer hit/read를 봅니다. 인덱스 추가 전에는 쿼리 조건과 정렬, limit, 데이터 분포를 확인하고, 추가 후에는 쓰기 비용과 기존 인덱스 중복을 같이 검토합니다.

LAB-03p95/p99와 커넥션 풀 해석

p95는 “사용자 100명 중 느린 5명”, p99는 “가장 불쾌한 1명”에 가깝습니다. 평균이 좋아도 p99가 튀면 운영에서는 장애처럼 보입니다. 커넥션 풀은 tail latency를 키우는 대표적인 증폭기입니다. DB가 느려져 커넥션 반납이 늦어지고, 풀이 차고, 대기열이 생기고, 요청 timeout이 연쇄적으로 늘어납니다.

풀 크기 조정의 원칙

풀 크기는 “대기 요청을 없애기 위해” 키우는 값이 아니라 DB가 안정적으로 병렬 처리할 수 있는 동시 작업 수에 맞추는 값입니다. p99가 높을 때 먼저 볼 것은 pool size가 아니라 커넥션 점유 시간입니다. 트랜잭션 안의 외부 API, 느린 쿼리, lock wait를 줄이면 같은 pool size로 더 많은 요청을 처리합니다.

LAB-04Index trade-off experiment

인덱스 실험은 읽기 쿼리 하나만 보면 항상 성공처럼 보입니다. senior-level 판정은 읽기 이득, 쓰기 손실, 저장공간, 중복 인덱스, 데이터 분포 변화를 같이 봅니다. 특히 복합 인덱스는 컬럼 순서를 바꾸면 완전히 다른 물건이 됩니다.

INDEX TRADE-OFF LAB Candidate A: (tenant_id, status, created_at DESC)
Candidate B: (tenant_id, created_at DESC) WHERE status = 'READY'
Read result: A는 범용, B는 hot query에서 더 작고 빠름

실험에서 자주 놓치는 것

낮은 카디널리티 컬럼이라도 단독 인덱스가 약할 뿐, 복합 인덱스나 partial index에서는 강한 필터가 될 수 있습니다. 반대로 카디널리티가 높은 컬럼이라도 쿼리의 정렬 방향과 LIMIT을 만족하지 못하면 큰 sort가 남습니다. “이 컬럼은 카디널리티가 높다/낮다”가 아니라 전체 쿼리 모양으로 판단하세요.

PG-01

psql 기본·접속

psql은 PostgreSQL의 공식 CLI입니다. SQL을 치기 전에 접속하고, 무엇이 있는지 둘러보는 메타명령(\로 시작)부터 손에 익히세요. GUI 없이도 디버깅의 90%가 여기서 끝납니다.

접속 — 두 가지 방식 # 옵션 방식
psql -h localhost -p 5432 -U myuser -d mydb

PG-02

자주 쓰는 SQL

SQL 기본 문법은 표준이라 익숙할 겁니다. 여기선 PostgreSQL에서 실전에 매일 치는 형태를 모았습니다 — 필터링, JOIN, 그룹 집계까지.

SELECT + WHERE — 필터링 SELECT id, name, created_at
  FROM users
 WHERE status = 'active'
   AND created_at >= '2026-01-01'
 ORDER BY created_at DESC
 LIMIT 20;
JOIN — INNER vs LEFT -- 주문이 있는 유저만 (교집합)
SELECT u.name, o.total
  FROM users u
  JOIN orders o ON o.user_id = u.id;

-- 주문이 없어도 유저는 모두 (없으면 o.* 는 NULL)
SELECT u.name, o.total
  FROM users u
  LEFT JOIN orders o ON o.user_id = u.id;
GROUP BY + 집계 + HAVING SELECT user_id,
       count(*) AS order_count,
       sum(total) AS total_spent,
       avg(total) AS avg_order
  FROM orders
 GROUP BY user_id
HAVING count(*) >= 3
 ORDER BY total_spent DESC;
WHERE와 HAVING의 차이

WHERE는 그룹핑 전 개별 행을 거르고, HAVING은 그룹핑 후 집계 결과를 거릅니다. "주문 3건 이상인 유저"처럼 count(*) 같은 집계값으로 필터링하려면 HAVING을 써야 합니다 — WHERE count(*) >= 3은 에러입니다. 그리고 SELECT의 집계 아닌 컬럼은 모두 GROUP BY에 있어야 합니다.

PG-03

INSERT·UPDATE·upsert

데이터를 넣고 고치는 건 기본이지만, PostgreSQL의 진짜 무기는 ON CONFLICT입니다 — "있으면 업데이트, 없으면 삽입"을 한 문장으로, 경쟁 조건 없이. 멱등한 동기화 작업의 핵심입니다.

INSERT — 여러 행 한 번에 INSERT INTO users (email, name, status)
VALUES
  ('a@x.com', 'Aria', 'active'),
  ('b@x.com', 'Ben', 'active');
UPDATE — 조건 필수 UPDATE users
   SET status = 'inactive',
      updated_at = now()
 WHERE last_login < '2025-01-01';
UPSERT — ON CONFLICT DO UPDATE INSERT INTO users (email, name, status)
VALUES ('a@x.com', 'Aria Kim', 'active')
ON CONFLICT (email) DO UPDATE
  SET name = EXCLUDED.name,
     status = EXCLUDED.status;

-- 이미 있으면 그냥 무시 (중복 삽입 방지)
INSERT INTO users (email, name)
VALUES ('a@x.com', 'Aria')
ON CONFLICT (email) DO NOTHING;
EXCLUDED와 충돌 대상

ON CONFLICT (email)의 email은 UNIQUE 제약 또는 unique index가 걸려 있어야 동작합니다(없으면 에러). EXCLUDED는 "삽입하려다 충돌한 그 새 값"을 가리키는 특수 테이블입니다 — 즉 EXCLUDED.name은 방금 넣으려던 새 name. 충돌 시에만 일부 컬럼을 갱신하고 싶을 때 이 패턴을 씁니다. WHERE 절을 붙여 DO UPDATE ... WHERE users.status <> EXCLUDED.status처럼 조건부 갱신도 가능합니다.

WHERE 없는 UPDATE/DELETE

UPDATE users SET status='x';처럼 WHERE를 빼면 전체 행이 바뀝니다. 운영 DB에서 사고의 단골입니다. psql에서 위험한 문장 전엔 같은 조건으로 SELECT count(*)를 먼저 쳐서 몇 행이 잡히는지 눈으로 확인하세요. 트랜잭션으로 감싸 ROLLBACK 여지를 두는 것도 방법입니다(→ PG-05).

PG-04

인덱스와 EXPLAIN ANALYZE

"쿼리가 느려요"의 답은 거의 항상 인덱스와 실행계획에 있습니다. 인덱스를 만들고, EXPLAIN ANALYZE로 PostgreSQL이 실제로 어떻게 도는지 들여다보세요. 원리는 DB 핸드북(B-tree)에.

CREATE INDEX -- 단일 컬럼
CREATE INDEX idx_users_email ON users (email);

-- 복합 인덱스 (순서가 중요)
CREATE INDEX idx_orders_user_created
  ON orders (user_id, created_at DESC);

-- 운영 중 테이블 락 없이 (느리지만 안전)
CREATE INDEX CONCURRENTLY idx_orders_status
  ON orders (status);
EXPLAIN ANALYZE — 실제 실행 측정 EXPLAIN ANALYZE
SELECT * FROM orders
 WHERE user_id = 42
 ORDER BY created_at DESC
 LIMIT 10;

-- 출력에서 볼 것:
-- Seq Scan → 전체 훑기 (인덱스 못 씀, 큰 테이블이면 경고)
-- Index Scan → 인덱스로 바로 찾음 (좋음)
-- actual time / rows → 실제 걸린 시간·행 수
실행계획 노드의미
Seq Scan테이블 전체 순차 스캔. 작은 테이블엔 정상, 큰 테이블에서 자주 보이면 인덱스 검토 신호
Index Scan인덱스를 타고 해당 행만 읽음. 선택도 높은 조건에서 이상적
Bitmap Heap Scan인덱스로 여러 위치를 모아 한 번에 읽기 — 중간 선택도에서 등장
actual time=...그 노드의 실제 소요 시간(ms). 병목 노드 찾기의 핵심
rows=... (planned/actual)예상 행 수 vs 실제 행 수. 크게 어긋나면 통계가 오래됨 → ANALYZE
복합 인덱스는 leftmost prefix

(user_id, created_at) 인덱스는 WHERE user_id = ? 또는 WHERE user_id = ? AND created_at > ?엔 쓰이지만, WHERE created_at > ?만으로는 잘 안 쓰입니다 — 인덱스는 왼쪽 컬럼부터 차례로 타기 때문입니다. 그래서 컬럼 순서가 중요합니다. 그리고 EXPLAIN(ANALYZE 없이)은 계획만 보여주고 실행하지 않지만, EXPLAIN ANALYZE는 실제로 쿼리를 실행합니다 — UPDATE/DELETE에 쓰면 진짜 데이터가 바뀌니 트랜잭션으로 감싸세요.

PG-05

트랜잭션·격리

여러 변경을 전부 성공 또는 전부 취소로 묶는 게 트랜잭션입니다. 동시에 같은 행을 건드릴 때의 충돌은 격리수준과 행 잠금으로 다룹니다. 격리수준의 개념·이상현상은 DB 핸드북에.

BEGIN / COMMIT / ROLLBACK BEGIN;

UPDATE accounts SET balance = balance - 100
 WHERE id = 1;
UPDATE accounts SET balance = balance + 100
 WHERE id = 2;

-- 둘 다 맞으면 COMMIT;
-- 뭔가 틀렸으면 ROLLBACK;
격리수준 설정 BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- ... 쿼리들 ...
COMMIT;

-- 가능한 레벨:
-- READ COMMITTED (기본값)
-- REPEATABLE READ
-- SERIALIZABLE (가장 엄격)
SELECT ... FOR UPDATE — 행 잠금 BEGIN;
-- 이 행을 읽고, 커밋까지 다른 트랜잭션의 수정·잠금을 막음
SELECT balance FROM accounts
 WHERE id = 1
 FOR UPDATE;

UPDATE accounts SET balance = balance - 100
 WHERE id = 1;
COMMIT;
PostgreSQL 기본은 READ COMMITTED

별도 설정이 없으면 모든 트랜잭션은 READ COMMITTED로 돕니다 — 각 문장은 그 문장 시작 시점에 커밋된 데이터를 봅니다. "잔액 읽고 → 검사하고 → 차감"처럼 읽은 값에 의존해 쓰는 로직은 그 사이 다른 트랜잭션이 끼어들 수 있으니, SELECT ... FOR UPDATE로 행을 잠그거나 UPDATE accounts SET balance = balance - 100처럼 읽기 없이 원자적으로 계산하세요. 잠금을 오래 쥐면 다른 트랜잭션이 대기하니 트랜잭션은 짧게.

PG-06

JSONB — 반정형 데이터

스키마가 유동적인 데이터(설정값, 외부 API 응답, 메타데이터)는 jsonb 컬럼에 담습니다. json이 아니라 jsonb를 쓰세요 — 이진 저장이라 빠르고 인덱싱이 됩니다.

컬럼 정의 + 삽입 CREATE TABLE events (
  id   bigserial PRIMARY KEY,
  payload jsonb NOT NULL
);

INSERT INTO events (payload) VALUES
  ('{"type":"login","user":{"id":42,"plan":"pro"}}');
조회 — -> (객체 반환) vs ->> (텍스트 반환) -- -> 는 jsonb 를 반환 (중첩 접근 체이닝용)
SELECT payload -> 'user' ->> 'id' AS user_id
  FROM events;

-- ->> 는 text 를 반환 (비교·캐스팅용)
SELECT * FROM events
 WHERE payload ->> 'type' = 'login';

-- 숫자 비교는 캐스팅
SELECT * FROM events
 WHERE (payload -> 'user' ->> 'id')::int > 40;
@> 포함 연산자 + GIN 인덱스 -- payload 가 이 키-값을 "포함"하는가
SELECT * FROM events
 WHERE payload @> '{"type":"login"}';

-- @> 검색을 빠르게 하는 GIN 인덱스
CREATE INDEX idx_events_payload
  ON events USING GIN (payload);
연산자의미
->키/인덱스로 접근, jsonb 반환 (체이닝 가능)
->>키/인덱스로 접근, text 반환 (비교·캐스팅용)
@>좌변이 우변을 포함하는가 (containment)
?해당 키가 존재하는가 (top-level key)
#> · #>>경로(path)로 깊이 접근. payload #>> '{user,id}'
-> 와 ->> 를 혼동하지 말 것

가장 흔한 실수입니다. payload -> 'type' = 'login'은 jsonb와 text를 비교해 의도대로 안 됩니다 — 비교엔 ->>(text 반환)를 써야 합니다. 반대로 중첩 객체를 더 파고들 땐 ->로 jsonb를 유지하다가 마지막에 ->>로 꺼냅니다. 그리고 @> 포함 검색만 GIN 인덱스의 혜택을 받습니다 — ->> 등호 비교를 자주 한다면 ((payload ->> 'type'))에 대한 표현식 인덱스를 따로 만드세요.

PG-07

윈도우 함수 — 순위·누적

"유저별 최신 주문", "카테고리 내 매출 순위", "누적 합계" — GROUP BY는 행을 뭉치지만, 윈도우 함수는 행을 유지한 채 그 행 주변을 계산합니다. 한 번 익히면 복잡한 서브쿼리가 사라집니다.

ROW_NUMBER — 그룹 내 순번 SELECT id, user_id, total, created_at,
      ROW_NUMBER() OVER (
        PARTITION BY user_id
        ORDER BY created_at DESC
      ) AS rn
  FROM orders;

-- "유저별 최신 주문 1건"은 이걸 감싸서:
-- SELECT * FROM (위 쿼리) t WHERE rn = 1;
RANK vs ROW_NUMBER SELECT category, name, price,
      RANK() OVER (PARTITION BY category
                     ORDER BY price DESC) AS price_rank
  FROM products;

-- RANK: 동점이면 같은 순위, 다음은 건너뜀 (1,1,3)
-- ROW_NUMBER: 동점이어도 무조건 고유 (1,2,3)
SUM OVER — 누적 합계 (running total) SELECT created_at, total,
      SUM(total) OVER (
        ORDER BY created_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
      ) AS running_total
  FROM orders;
PARTITION BY는 "그룹", ORDER BY는 "정렬"

OVER (PARTITION BY user_id ORDER BY created_at)를 풀면 — user_id별로 묶고, 각 묶음 안에서 created_at 순으로 정렬한 뒤 계산하라는 뜻입니다. PARTITION BY를 빼면 전체가 하나의 윈도우가 됩니다. "유저별 최신 N건"은 ROW_NUMBER를 매기고 바깥에서 WHERE rn <= N으로 거르는 게 정석입니다 — 서브쿼리·상관 서브쿼리보다 훨씬 빠릅니다.

PG-08

페이지네이션 — OFFSET vs keyset

OFFSET 페이징은 쓰기 쉽지만 뒤로 갈수록 느려집니다 — DB가 건너뛸 행을 매번 다 세기 때문입니다. 무한 스크롤·대용량엔 keyset(cursor) 페이징이 정답입니다.

OFFSET / LIMIT — 쉽지만 한계 있음 -- 3페이지 (페이지당 20개) → 40개 건너뛰고 20개
SELECT * FROM orders
 ORDER BY created_at DESC, id DESC
 LIMIT 20 OFFSET 40;

-- 문제: OFFSET 100000 이면 10만 행을 세고 버림 → 느림
KEYSET — 마지막 본 값 이후부터 -- 첫 페이지
SELECT * FROM orders
 ORDER BY id DESC
 LIMIT 20;

-- 다음 페이지: 직전 페이지 마지막 id 가 1000 이었다면
SELECT * FROM orders
 WHERE id < 1000
 ORDER BY id DESC
 LIMIT 20;
방식장점단점
OFFSET"N페이지로 점프" 가능, 구현 쉬움뒤 페이지일수록 느림, 그 사이 삽입되면 행 중복·누락
keyset페이지 깊이와 무관하게 빠름(인덱스 활용), 결과 안정적임의 페이지 점프 불가(이전/다음만), 정렬 키가 인덱스에 있어야
정렬 키는 유일해야 한다

keyset 페이징의 정렬 컬럼은 유일(unique)해야 페이지 경계가 안전합니다. created_at처럼 중복될 수 있는 값으로만 자르면 같은 시각의 행이 잘려 누락·중복됩니다. 해결: ORDER BY created_at DESC, id DESC로 tie-breaker를 더하고, 조건도 WHERE (created_at, id) < ('2026-06-01 10:00', 1000)처럼 튜플 비교로 씁니다. 이 정렬 순서와 같은 복합 인덱스가 있어야 진짜 빠릅니다(→ PG-04).

PG-09

유용한 기능

매일 쓰진 않지만 알면 코드가 짧아지는 PostgreSQL 기능들. RETURNING, COALESCE, generate_series, 날짜 함수.

RETURNING — INSERT/UPDATE 결과 즉시 받기 -- 생성된 id 를 별도 SELECT 없이 바로
INSERT INTO users (email, name)
VALUES ('c@x.com', 'Cara')
RETURNING id, created_at;

-- UPDATE/DELETE 에도 — 바뀐 행을 돌려받음
UPDATE orders SET status = 'shipped'
 WHERE id = 7 RETURNING *;
COALESCE — NULL 대체 -- 첫 번째 NULL 아닌 값을 반환
SELECT name, COALESCE(nickname, name, '익명') AS display_name
  FROM users;

-- 집계 NULL 방어: 합계 없으면 0
SELECT COALESCE(sum(total), 0) FROM orders WHERE user_id = 99;
generate_series + 날짜 함수 — 빈 날짜 채우기 -- 6월 한 달의 모든 날짜 행 생성
SELECT generate_series(
  '2026-06-01'::date,
  '2026-06-30'::date,
  '1 day'::interval
) AS day;

-- 날짜 자르기·간격
SELECT date_trunc('month', created_at) AS month, count(*)
  FROM orders
 WHERE created_at >= now() - '30 days'::interval
 GROUP BY month ORDER BY month;
파티셔닝 — 한 줄 소개

로그·이벤트처럼 시간이 갈수록 무한히 쌓이는 거대 테이블은 선언적 파티셔닝(CREATE TABLE ... PARTITION BY RANGE (created_at))으로 월별·연도별 물리 테이블로 쪼갤 수 있습니다. 오래된 파티션을 통째로 DETACH/DROP하면 대량 삭제보다 훨씬 빠릅니다. 다만 설계·운영 부담이 있으니 테이블이 정말 커지기 전엔 과하게 도입하지 마세요 — 측정이 먼저입니다.

RETURNING은 PostgreSQL의 강점

많은 ORM이 INSERT 후 생성된 id를 얻으려 별도 쿼리를 날리지만, PostgreSQL은 RETURNING으로 한 번의 왕복(round-trip)에 끝낼 수 있습니다. upsert와 결합하면 INSERT ... ON CONFLICT ... DO UPDATE ... RETURNING id로 "있으면 그 id, 없으면 새 id"를 한 문장에 받습니다.

REDIS-01

Redis란·자료구조 개요

Redis는 메모리에 사는 key-value 저장소입니다. RDBMS가 디스크에 정규화된 행을 두는 것과 달리, Redis는 RAM 위에서 단순한 자료구조를 마이크로초 단위로 읽고 씁니다. "빠르게 읽혀야 하고, 잠깐 살아 있어도 되는" 데이터가 Redis 자리입니다.

프론트엔드에서 Map이나 localStorage를 쓰던 감각을 서버 바깥의 공유 메모리로 옮긴 것이라고 보면 빠릅니다. 단, value가 문자열만이 아니라 hash·list·set 같은 구조를 가질 수 있고, 모든 키에 만료(TTL)를 걸 수 있다는 점이 다릅니다.

REDIS = KEY → STRUCTURED VALUE (IN MEMORY) SET user:42:name "haeun" → string
HSET user:42 name "haeun" age "33" → hash(필드 묶음)
INCR page:home:views → string이지만 숫자로 원자적 증가
EXPIRE user:42:name 3600 → 1시간 뒤 자동 삭제
자료구조한 줄 정의대표 용도
string가장 단순한 값(텍스트·숫자·직렬화 JSON)캐시 값, 카운터, 토큰
hash하나의 key 아래 field→value 묶음객체 저장(user:42 의 필드들)
list순서 있는 값의 연결 리스트최근 항목, 간단한 큐(LPUSH/RPOP)
set중복 없는 값의 모음태그, 방문자 unique 집합
sorted setscore로 정렬되는 set랭킹·리더보드, 시간순 인덱스
pub/sub채널 기반 메시지 발행/구독실시간 알림 fan-out(저장은 안 함)
언제 쓰나 / 언제 안 쓰나

Redis는 primary DB가 아닙니다. RAM에 살기 때문에 빠르지만, 메모리는 비싸고 휘발 위험이 있습니다. "원본은 PostgreSQL, 빠른 사본·임시 상태는 Redis"가 기본 그림입니다. 영구 보존이 중요한 비즈니스 데이터(거래 기록 등)를 Redis에만 두는 건 위험합니다. 캐시·세션·카운터·락처럼 잃어도 재생성 가능하거나 본질적으로 임시인 데이터에 쓰세요.

REDIS-03

캐시 — Spring Cache 추상화

"이 메서드 결과를 캐싱하고 싶다"가 목적이라면, RedisTemplate을 직접 만지기 전에 Spring Cache 추상화부터 보세요. 어노테이션 몇 개로 메서드 결과를 Redis에 자동 캐싱하고, 비즈니스 코드는 캐시를 거의 의식하지 않습니다.

먼저 캐싱을 켜고, TTL을 가진 RedisCacheManager를 등록합니다.

CacheConfig.java — 캐싱 활성화 + TTL @Configuration
@EnableCaching
public class CacheConfig {
  @Bean
  public RedisCacheManager cacheManager(RedisConnectionFactory cf) {
    RedisCacheConfiguration config = RedisCacheConfiguration
      .defaultCacheConfig()
      .entryTtl(Duration.ofMinutes(10)) // 기본 TTL 10분
      .disableCachingNullValues();
    return RedisCacheManager.builder(cf)
      .cacheDefaults(config)
      .build();
  }
}

이제 서비스 메서드에 어노테이션만 붙이면 됩니다.

ProductService.java — 읽기/무효화/갱신 @Service
public class ProductService {

  // 캐시에 있으면 메서드를 건너뛰고 캐시 값 반환
  @Cacheable(value = "product", key = "#id")
  public Product findById(Long id) {
    return productRepository.findById(id).orElseThrow();
  }

  // 변경 후 해당 키의 캐시를 제거 → 다음 읽기에 다시 적재
  @CacheEvict(value = "product", key = "#product.id")
  public void update(Product product) {
    productRepository.save(product);
  }

  // 메서드는 항상 실행하고, 그 반환값으로 캐시를 갱신
  @CachePut(value = "product", key = "#result.id")
  public Product create(Product product) {
    return productRepository.save(product);
  }
}

키 "product" + id=42는 Redis에서 product::42 같은 키로 저장됩니다. redis-cli로 직접 확인해 보면 동작이 손에 잡힙니다.

redis-cli — 캐시 키 들여다보기 127.0.0.1:6379> KEYS product::*
1) "product::42"
127.0.0.1:6379> TTL product::42
(integer) 587 # 남은 만료 시간(초)
어노테이션 차이 — 헷갈리는 셋

@Cacheable은 "있으면 안 부른다"(읽기), @CacheEvict는 "지운다"(무효화), @CachePut은 "항상 부르고 결과를 덮어쓴다"(갱신). 흔한 실수는 생성/수정에 @Cacheable을 붙이는 것 — 그러면 메서드가 캐시 히트 시 아예 실행되지 않아 DB에 안 써집니다. 쓰기 경로엔 @CacheEvict/@CachePut이 맞습니다. 또 self-invocation(같은 빈 안에서 메서드 호출)에는 프록시가 안 걸려 캐시가 동작하지 않으니 주의하세요.

REDIS-04

cache-aside 직접 구현

어노테이션이 표현하기 어려운 캐시 로직(조건부 적재, 커스텀 키, 부분 갱신)이 필요하면 RedisTemplate으로 직접 짭니다. 가장 흔한 패턴이 cache-aside(lazy loading) — 캐시를 먼저 보고, 없으면 DB에서 읽어 캐시에 채운 뒤 반환.

1. 캐시 GET→2. 있으면 반환(hit)→3. 없으면 DB 조회(miss)→4. 캐시 SET(TTL)→5. 반환
ProductCacheRepository.java — get-or-load @Repository
public class ProductCacheRepository {

  private final RedisTemplate<String, Object> redisTemplate;
  private final ProductRepository productRepository;

  public Product getById(Long id) {
    String key = "product:" + id;
    ValueOperations<String, Object> ops = redisTemplate.opsForValue();

    // 1. 캐시 먼저
    Object cached = ops.get(key);
    if (cached != null) {
      return (Product) cached; // cache hit
    }

    // 2. miss → DB 조회
    Product product = productRepository.findById(id).orElseThrow();

    // 3. 캐시에 채우고 TTL 부여
    ops.set(key, product, Duration.ofMinutes(10));
    return product;
  }
}

opsForValue().set(key, value, ttl)은 내부적으로 Redis의 SETEX(또는 SET key value EX seconds)에 대응합니다 — 값과 만료를 한 번에 거는 것. 같은 동작을 redis-cli로 보면.

redis-cli — set + ttl을 한 번에 127.0.0.1:6379> SETEX product:42 600 "{...json...}"
OK # 600초 후 자동 만료
127.0.0.1:6379> GET product:42
"{...json...}"
127.0.0.1:6379> TTL product:42
(integer) 598
cache stampede — TTL은 꼭 걸어라

TTL 없이 set하면 키가 영원히 남아 메모리를 잠식하고, 원본이 바뀌어도 stale한 값이 박힙니다. cache-aside에서 TTL은 선택이 아니라 기본입니다. 또 인기 키가 동시에 만료되면 수많은 요청이 한꺼번에 DB로 몰리는 cache stampede가 생깁니다 — TTL에 약간의 랜덤(jitter)을 더하거나, 적재 구간에 짧은 락을 거는 식으로 완화합니다. 무효화 전략의 트레이드오프는 백엔드 개요를 참조하세요.

REDIS-05

세션 저장소

서버를 여러 대로 늘리는 순간 세션 문제가 터집니다. 로그인 세션이 서버 A의 메모리에만 있으면, 다음 요청이 서버 B로 가는 순간 "로그인 안 됨" 상태가 됩니다. 세션을 서버 바깥의 공유 저장소(Redis)에 두면 어느 서버로 가든 같은 세션을 봅니다.

왜 필요한가 — 분산 환경의 세션 [로컬 세션] 서버 메모리에 세션 저장
LB → 서버A(세션 보유) ✅   LB → 서버B(세션 없음) ❌

[Redis 세션] 공유 저장소에 세션 저장
LB → 서버A ──┐
LB → 서버B ──┼──▶ Redis (단일 세션 소스) ✅
LB → 서버C ──┘

Spring Session Data Redis를 쓰면 코드 변경 거의 없이 HttpSession을 Redis로 옮길 수 있습니다.

build.gradle implementation 'org.springframework.session:spring-session-data-redis'

application.yml spring:
  session:
    store-type: redis
    timeout: 30m # 세션 만료

이렇게만 하면 기존 session.setAttribute(...) 코드가 그대로 Redis에 저장됩니다. Redis에서는 세션이 hash로 들어갑니다.

redis-cli — 세션은 hash로 저장됨 127.0.0.1:6379> KEYS spring:session:*
1) "spring:session:sessions:2f9a...c1"
127.0.0.1:6379> TYPE spring:session:sessions:2f9a...c1
hash
127.0.0.1:6379> HGETALL spring:session:sessions:2f9a...c1
1) "sessionAttr:userId" 2) "42" ...
실전 — JWT vs Redis 세션

"분산 환경에서 인증 상태를 어떻게 공유하나"는 두 갈래입니다 — ① stateless JWT(서버가 상태를 안 가짐), ② Redis 세션(상태를 공유 저장소에 둠). JWT는 즉시 무효화(로그아웃·강제 차단)가 어렵고, Redis 세션은 DEL 한 번으로 끊을 수 있는 대신 매 요청마다 Redis를 조회합니다. 공공/B2B처럼 세션을 즉시 끊어야 하는 요구가 있으면 Redis 세션이 유리할 때가 많습니다. 선택은 요구사항에 달렸으니 단정하지 마세요.

REDIS-06

분산 락

여러 인스턴스가 동시에 같은 작업(예: 같은 주문의 중복 정산)을 하면 안 될 때, 서버 메모리의 synchronized는 무력합니다 — 다른 JVM은 그 락을 모릅니다. Redis로 여러 프로세스가 공유하는 락을 만들 수 있습니다.

핵심은 SET key value NX PX 한 줄입니다. NX는 "키가 없을 때만 set"(= 아무도 락을 안 잡았을 때만 획득), PX는 밀리초 단위 만료(= 락 잡은 프로세스가 죽어도 자동 해제).

redis-cli — 락 획득의 원자적 한 줄 127.0.0.1:6379> SET lock:order:42 "token-abc" NX PX 5000
OK # 획득 성공 (5초 뒤 자동 만료)
127.0.0.1:6379> SET lock:order:42 "token-xyz" NX PX 5000
(nil) # 이미 누가 잡음 → 실패
Java — RedisTemplate로 락 시도 (개념용) // value에 고유 토큰을 넣어, 내가 잡은 락만 풀 수 있게 함
String token = UUID.randomUUID().toString();
Boolean acquired = redisTemplate.opsForValue()
  .setIfAbsent("lock:order:42", token, Duration.ofSeconds(5));
// setIfAbsent = SET ... NX, Duration = PX

if (Boolean.TRUE.equals(acquired)) {
  try { /* 임계 구역 작업 */ }
  finally { /* 토큰 비교 후 해제 (아래 주의 참조) */ }
}
위험 — 직접 만든 분산 락은 안전하지 않다

위 코드는 동작하는 것처럼 보이지만 함정이 많습니다. ① 락 해제는 "내 토큰일 때만 DEL"이어야 하는데, GET 후 DEL은 원자적이지 않아 그 사이 만료된 락을 남이 잡으면 남의 락을 지웁니다 — 안전하게 하려면 Lua 스크립트로 비교+삭제를 원자적으로 해야 합니다. ② 작업이 TTL보다 오래 걸리면 락이 먼저 풀려 두 프로세스가 동시에 임계 구역에 들어갑니다. ③ Redis가 단일 노드가 아니면(복제·페일오버) 더 복잡해집니다.

직접 구현하지 말고 Redisson 같은 검증된 라이브러리를 쓰세요(RLock은 토큰 검증·watchdog 자동 연장·재진입을 처리). 분산 락은 "쉬워 보이지만 제대로 하기 어려운" 대표 주제입니다.

REDIS-07

rate limiting

"IP당 분당 100회"처럼 호출 횟수를 제한할 때, Redis의 INCR이 깔끔합니다. INCR은 키 값을 원자적으로 1 증가시키므로(동시 요청에도 카운트가 안 꼬임), 카운터 + 만료만으로 고정 윈도우 제한기가 됩니다.

redis-cli — 고정 윈도우 카운터 127.0.0.1:6379> INCR rate:ip:1.2.3.4:202606231830
(integer) 1 # 첫 요청 → 1
127.0.0.1:6379> EXPIRE rate:ip:1.2.3.4:202606231830 60
(integer) 1 # 이 분(minute) 키를 60초 뒤 삭제
127.0.0.1:6379> INCR rate:ip:1.2.3.4:202606231830
(integer) 2 # 같은 분의 두 번째 요청

키에 시간 윈도우(분)를 박아두면 분이 바뀔 때 자연히 새 카운터가 시작됩니다. Java로 옮기면.

RateLimiter.java — INCR + EXPIRE public boolean allow(String ip) {
  String minute = LocalDateTime.now()
    .format(DateTimeFormatter.ofPattern("yyyyMMddHHmm"));
  String key = "rate:ip:" + ip + ":" + minute;

  Long count = redisTemplate.opsForValue().increment(key); // INCR
  if (count != null && count == 1L) {
    // 첫 요청에만 TTL 부여 → 윈도우 끝나면 카운터 소멸
    redisTemplate.expire(key, Duration.ofMinutes(1));
  }
  return count != null && count <= 100; // 분당 100회 제한
}
고정 윈도우의 경계 문제

고정 윈도우는 단순하지만 경계에서 두 배가 샐 수 있습니다 — 한 분의 끝 0.5초에 100회, 다음 분의 시작 0.5초에 100회면 1초 안에 200회가 통과합니다. 더 매끄러운 제한이 필요하면 sliding window log(sorted set에 타임스탬프 저장)나 token bucket을 씁니다. 또 INCR과 EXPIRE가 두 명령이라, 첫 INCR 직후 프로세스가 죽으면 TTL 없는 키가 남을 수 있으니, 엄격하게는 Lua 스크립트로 묶는 게 안전합니다. 대부분의 API 보호엔 위 수준으로 충분합니다.

REDIS-08

TTL·만료·메모리

Redis를 캐시로 쓴다는 건 "이 데이터는 언젠가 사라진다"를 전제로 코드를 짠다는 뜻입니다. TTL은 메모리를 지키는 안전장치이자, stale 데이터를 자동 청소하는 수단입니다.

명령의미
EXPIRE key 60키를 60초 뒤 만료로 설정
TTL key남은 만료 시간(초). -1=만료 없음, -2=키 없음
PERSIST key만료 제거(영구화)
SET key v EX 60set과 동시에 TTL(초). PX는 밀리초
SETEX key 60 v위와 같은 의미의 전용 명령
redis-cli — TTL 상태 읽기 127.0.0.1:6379> SET token:abc "..." EX 30
OK
127.0.0.1:6379> TTL token:abc
(integer) 28
127.0.0.1:6379> TTL nonexistent
(integer) -2 # 키 자체가 없음

메모리가 한계에 닿으면 Redis는 maxmemory-policy에 따라 키를 쫓아냅니다(eviction). 캐시 용도라면 보통 LRU/LFU 계열을 씁니다.

eviction 정책동작
noeviction쫓아내지 않음 — 쓰기가 에러로 거부됨(기본값)
allkeys-lru모든 키 중 가장 오래 안 쓴 것부터 제거(범용 캐시에 무난)
allkeys-lfu가장 적게 쓴 것부터 제거(접근 빈도 기준)
volatile-ttlTTL 있는 키 중 만료가 임박한 것부터 제거
위험 — 캐시는 "있으면 좋은 것"이지 "있어야 하는 것"이 아니다

eviction이 기본 noeviction이면, 메모리가 차는 순간 모든 쓰기가 실패합니다 — 캐시로 쓸 거면 allkeys-lru 등으로 명시하세요. 더 중요한 건 코드 쪽 전제입니다: 캐시 키는 언제든 사라질 수 있으므로, 캐시 miss를 정상 경로로 다뤄야 합니다(REDIS-04의 get-or-load). "캐시에 분명히 넣었으니 있겠지"라고 가정하면, eviction/만료/재시작 후에 NPE로 터집니다.
