REDIS-01

Redis란·자료구조 개요

Redis는 메모리에 사는 key-value 저장소입니다. RDBMS가 디스크에 정규화된 행을 두는 것과 달리, Redis는 RAM 위에서 단순한 자료구조를 마이크로초 단위로 읽고 씁니다. "빠르게 읽혀야 하고, 잠깐 살아 있어도 되는" 데이터가 Redis 자리입니다.

프론트엔드에서 Map이나 localStorage를 쓰던 감각을 서버 바깥의 공유 메모리로 옮긴 것이라고 보면 빠릅니다. 단, value가 문자열만이 아니라 hash·list·set 같은 구조를 가질 수 있고, 모든 키에 만료(TTL)를 걸 수 있다는 점이 다릅니다.

REDIS = KEY → STRUCTURED VALUE (IN MEMORY) SET user:42:name "haeun" → string
HSET user:42 name "haeun" age "33" → hash(필드 묶음)
INCR page:home:views → string이지만 숫자로 원자적 증가
EXPIRE user:42:name 3600 → 1시간 뒤 자동 삭제
자료구조한 줄 정의대표 용도
string가장 단순한 값(텍스트·숫자·직렬화 JSON)캐시 값, 카운터, 토큰
hash하나의 key 아래 field→value 묶음객체 저장(user:42 의 필드들)
list순서 있는 값의 연결 리스트최근 항목, 간단한 큐(LPUSH/RPOP)
set중복 없는 값의 모음태그, 방문자 unique 집합
sorted setscore로 정렬되는 set랭킹·리더보드, 시간순 인덱스
pub/sub채널 기반 메시지 발행/구독실시간 알림 fan-out(저장은 안 함)
언제 쓰나 / 언제 안 쓰나

Redis는 primary DB가 아닙니다. RAM에 살기 때문에 빠르지만, 메모리는 비싸고 휘발 위험이 있습니다. "원본은 PostgreSQL, 빠른 사본·임시 상태는 Redis"가 기본 그림입니다. 영구 보존이 중요한 비즈니스 데이터(거래 기록 등)를 Redis에만 두는 건 위험합니다. 캐시·세션·카운터·락처럼 잃어도 재생성 가능하거나 본질적으로 임시인 데이터에 쓰세요.

REDIS-02

연결 설정

Spring Boot에서 Redis 연결은 의존성 하나와 설정 몇 줄이면 끝납니다. spring-boot-starter-data-redis가 커넥션 풀(기본 Lettuce 클라이언트)과 RedisTemplate 자동 설정을 가져옵니다.

build.gradle (Gradle) dependencies {
  implementation 'org.springframework.boot:spring-boot-starter-data-redis'
}

pom.xml (Maven) <dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>

다음은 접속 정보. Spring Boot 3.x 기준 프로퍼티는 spring.data.redis.* 아래입니다(과거 spring.redis.*는 deprecated).

application.yml spring:
  data:
    redis:
      host: localhost
      port: 6379
      password: ${REDIS_PASSWORD:} # 비밀번호는 환경변수로
      timeout: 2000ms

연결이 됐는지 확인하려면 로컬에서 redis-cli로 직접 핑을 쳐 봅니다.

redis-cli — 연결 확인 $ redis-cli -h localhost -p 6379 ping
PONG
$ redis-cli
127.0.0.1:6379> SET hello world
OK
127.0.0.1:6379> GET hello
"world"
실전 — 로컬 Redis 띄우기

개발용 Redis는 Docker 한 줄이 가장 빠릅니다: docker run -d -p 6379:6379 redis:7. password 같은 비밀값은 application.yml에 직접 박지 말고 ${REDIS_PASSWORD}처럼 환경변수로 주입하세요. 운영에서는 호스트·포트도 환경별 프로파일(application-prod.yml)로 분리합니다.

REDIS-03

캐시 — Spring Cache 추상화

"이 메서드 결과를 캐싱하고 싶다"가 목적이라면, RedisTemplate을 직접 만지기 전에 Spring Cache 추상화부터 보세요. 어노테이션 몇 개로 메서드 결과를 Redis에 자동 캐싱하고, 비즈니스 코드는 캐시를 거의 의식하지 않습니다.

먼저 캐싱을 켜고, TTL을 가진 RedisCacheManager를 등록합니다.

CacheConfig.java — 캐싱 활성화 + TTL @Configuration
@EnableCaching
public class CacheConfig {
  @Bean
  public RedisCacheManager cacheManager(RedisConnectionFactory cf) {
    RedisCacheConfiguration config = RedisCacheConfiguration
      .defaultCacheConfig()
      .entryTtl(Duration.ofMinutes(10)) // 기본 TTL 10분
      .disableCachingNullValues();
    return RedisCacheManager.builder(cf)
      .cacheDefaults(config)
      .build();
  }
}

이제 서비스 메서드에 어노테이션만 붙이면 됩니다.

ProductService.java — 읽기/무효화/갱신 @Service
public class ProductService {

  // 캐시에 있으면 메서드를 건너뛰고 캐시 값 반환
  @Cacheable(value = "product", key = "#id")
  public Product findById(Long id) {
    return productRepository.findById(id).orElseThrow();
  }

  // 변경 후 해당 키의 캐시를 제거 → 다음 읽기에 다시 적재
  @CacheEvict(value = "product", key = "#product.id")
  public void update(Product product) {
    productRepository.save(product);
  }

  // 메서드는 항상 실행하고, 그 반환값으로 캐시를 갱신
  @CachePut(value = "product", key = "#result.id")
  public Product create(Product product) {
    return productRepository.save(product);
  }
}

키 "product" + id=42는 Redis에서 product::42 같은 키로 저장됩니다. redis-cli로 직접 확인해 보면 동작이 손에 잡힙니다.

redis-cli — 캐시 키 들여다보기 127.0.0.1:6379> KEYS product::*
1) "product::42"
127.0.0.1:6379> TTL product::42
(integer) 587 # 남은 만료 시간(초)
어노테이션 차이 — 헷갈리는 셋

@Cacheable은 "있으면 안 부른다"(읽기), @CacheEvict는 "지운다"(무효화), @CachePut은 "항상 부르고 결과를 덮어쓴다"(갱신). 흔한 실수는 생성/수정에 @Cacheable을 붙이는 것 — 그러면 메서드가 캐시 히트 시 아예 실행되지 않아 DB에 안 써집니다. 쓰기 경로엔 @CacheEvict/@CachePut이 맞습니다. 또 self-invocation(같은 빈 안에서 메서드 호출)에는 프록시가 안 걸려 캐시가 동작하지 않으니 주의하세요.

REDIS-04

cache-aside 직접 구현

어노테이션이 표현하기 어려운 캐시 로직(조건부 적재, 커스텀 키, 부분 갱신)이 필요하면 RedisTemplate으로 직접 짭니다. 가장 흔한 패턴이 cache-aside(lazy loading) — 캐시를 먼저 보고, 없으면 DB에서 읽어 캐시에 채운 뒤 반환.

1. 캐시 GET→2. 있으면 반환(hit)→3. 없으면 DB 조회(miss)→4. 캐시 SET(TTL)→5. 반환
ProductCacheRepository.java — get-or-load @Repository
public class ProductCacheRepository {

  private final RedisTemplate<String, Object> redisTemplate;
  private final ProductRepository productRepository;

  public Product getById(Long id) {
    String key = "product:" + id;
    ValueOperations<String, Object> ops = redisTemplate.opsForValue();

    // 1. 캐시 먼저
    Object cached = ops.get(key);
    if (cached != null) {
      return (Product) cached; // cache hit
    }

    // 2. miss → DB 조회
    Product product = productRepository.findById(id).orElseThrow();

    // 3. 캐시에 채우고 TTL 부여
    ops.set(key, product, Duration.ofMinutes(10));
    return product;
  }
}

opsForValue().set(key, value, ttl)은 내부적으로 Redis의 SETEX(또는 SET key value EX seconds)에 대응합니다 — 값과 만료를 한 번에 거는 것. 같은 동작을 redis-cli로 보면.

redis-cli — set + ttl을 한 번에 127.0.0.1:6379> SETEX product:42 600 "{...json...}"
OK # 600초 후 자동 만료
127.0.0.1:6379> GET product:42
"{...json...}"
127.0.0.1:6379> TTL product:42
(integer) 598
cache stampede — TTL은 꼭 걸어라

TTL 없이 set하면 키가 영원히 남아 메모리를 잠식하고, 원본이 바뀌어도 stale한 값이 박힙니다. cache-aside에서 TTL은 선택이 아니라 기본입니다. 또 인기 키가 동시에 만료되면 수많은 요청이 한꺼번에 DB로 몰리는 cache stampede가 생깁니다 — TTL에 약간의 랜덤(jitter)을 더하거나, 적재 구간에 짧은 락을 거는 식으로 완화합니다. 무효화 전략의 트레이드오프는 백엔드 개요를 참조하세요.

REDIS-05

세션 저장소

서버를 여러 대로 늘리는 순간 세션 문제가 터집니다. 로그인 세션이 서버 A의 메모리에만 있으면, 다음 요청이 서버 B로 가는 순간 "로그인 안 됨" 상태가 됩니다. 세션을 서버 바깥의 공유 저장소(Redis)에 두면 어느 서버로 가든 같은 세션을 봅니다.

왜 필요한가 — 분산 환경의 세션 [로컬 세션] 서버 메모리에 세션 저장
LB → 서버A(세션 보유) ✅   LB → 서버B(세션 없음) ❌

[Redis 세션] 공유 저장소에 세션 저장
LB → 서버A ──┐
LB → 서버B ──┼──▶ Redis (단일 세션 소스) ✅
LB → 서버C ──┘

Spring Session Data Redis를 쓰면 코드 변경 거의 없이 HttpSession을 Redis로 옮길 수 있습니다.

build.gradle implementation 'org.springframework.session:spring-session-data-redis'

application.yml spring:
  session:
    store-type: redis
    timeout: 30m # 세션 만료

이렇게만 하면 기존 session.setAttribute(...) 코드가 그대로 Redis에 저장됩니다. Redis에서는 세션이 hash로 들어갑니다.

redis-cli — 세션은 hash로 저장됨 127.0.0.1:6379> KEYS spring:session:*
1) "spring:session:sessions:2f9a...c1"
127.0.0.1:6379> TYPE spring:session:sessions:2f9a...c1
hash
127.0.0.1:6379> HGETALL spring:session:sessions:2f9a...c1
1) "sessionAttr:userId" 2) "42" ...
실전 — JWT vs Redis 세션

"분산 환경에서 인증 상태를 어떻게 공유하나"는 두 갈래입니다 — ① stateless JWT(서버가 상태를 안 가짐), ② Redis 세션(상태를 공유 저장소에 둠). JWT는 즉시 무효화(로그아웃·강제 차단)가 어렵고, Redis 세션은 DEL 한 번으로 끊을 수 있는 대신 매 요청마다 Redis를 조회합니다. 공공/B2B처럼 세션을 즉시 끊어야 하는 요구가 있으면 Redis 세션이 유리할 때가 많습니다. 선택은 요구사항에 달렸으니 단정하지 마세요.

REDIS-06

분산 락

여러 인스턴스가 동시에 같은 작업(예: 같은 주문의 중복 정산)을 하면 안 될 때, 서버 메모리의 synchronized는 무력합니다 — 다른 JVM은 그 락을 모릅니다. Redis로 여러 프로세스가 공유하는 락을 만들 수 있습니다.

핵심은 SET key value NX PX 한 줄입니다. NX는 "키가 없을 때만 set"(= 아무도 락을 안 잡았을 때만 획득), PX는 밀리초 단위 만료(= 락 잡은 프로세스가 죽어도 자동 해제).

redis-cli — 락 획득의 원자적 한 줄 127.0.0.1:6379> SET lock:order:42 "token-abc" NX PX 5000
OK # 획득 성공 (5초 뒤 자동 만료)
127.0.0.1:6379> SET lock:order:42 "token-xyz" NX PX 5000
(nil) # 이미 누가 잡음 → 실패
Java — RedisTemplate로 락 시도 (개념용) // value에 고유 토큰을 넣어, 내가 잡은 락만 풀 수 있게 함
String token = UUID.randomUUID().toString();
Boolean acquired = redisTemplate.opsForValue()
  .setIfAbsent("lock:order:42", token, Duration.ofSeconds(5));
// setIfAbsent = SET ... NX, Duration = PX

if (Boolean.TRUE.equals(acquired)) {
  try { /* 임계 구역 작업 */ }
  finally { /* 토큰 비교 후 해제 (아래 주의 참조) */ }
}
위험 — 직접 만든 분산 락은 안전하지 않다

위 코드는 동작하는 것처럼 보이지만 함정이 많습니다. ① 락 해제는 "내 토큰일 때만 DEL"이어야 하는데, GET 후 DEL은 원자적이지 않아 그 사이 만료된 락을 남이 잡으면 남의 락을 지웁니다 — 안전하게 하려면 Lua 스크립트로 비교+삭제를 원자적으로 해야 합니다. ② 작업이 TTL보다 오래 걸리면 락이 먼저 풀려 두 프로세스가 동시에 임계 구역에 들어갑니다. ③ Redis가 단일 노드가 아니면(복제·페일오버) 더 복잡해집니다.

직접 구현하지 말고 Redisson 같은 검증된 라이브러리를 쓰세요(RLock은 토큰 검증·watchdog 자동 연장·재진입을 처리). 분산 락은 "쉬워 보이지만 제대로 하기 어려운" 대표 주제입니다.

REDIS-07

rate limiting

"IP당 분당 100회"처럼 호출 횟수를 제한할 때, Redis의 INCR이 깔끔합니다. INCR은 키 값을 원자적으로 1 증가시키므로(동시 요청에도 카운트가 안 꼬임), 카운터 + 만료만으로 고정 윈도우 제한기가 됩니다.

redis-cli — 고정 윈도우 카운터 127.0.0.1:6379> INCR rate:ip:1.2.3.4:202606231830
(integer) 1 # 첫 요청 → 1
127.0.0.1:6379> EXPIRE rate:ip:1.2.3.4:202606231830 60
(integer) 1 # 이 분(minute) 키를 60초 뒤 삭제
127.0.0.1:6379> INCR rate:ip:1.2.3.4:202606231830
(integer) 2 # 같은 분의 두 번째 요청

키에 시간 윈도우(분)를 박아두면 분이 바뀔 때 자연히 새 카운터가 시작됩니다. Java로 옮기면.

RateLimiter.java — INCR + EXPIRE public boolean allow(String ip) {
  String minute = LocalDateTime.now()
    .format(DateTimeFormatter.ofPattern("yyyyMMddHHmm"));
  String key = "rate:ip:" + ip + ":" + minute;

  Long count = redisTemplate.opsForValue().increment(key); // INCR
  if (count != null && count == 1L) {
    // 첫 요청에만 TTL 부여 → 윈도우 끝나면 카운터 소멸
    redisTemplate.expire(key, Duration.ofMinutes(1));
  }
  return count != null && count <= 100; // 분당 100회 제한
}
고정 윈도우의 경계 문제

고정 윈도우는 단순하지만 경계에서 두 배가 샐 수 있습니다 — 한 분의 끝 0.5초에 100회, 다음 분의 시작 0.5초에 100회면 1초 안에 200회가 통과합니다. 더 매끄러운 제한이 필요하면 sliding window log(sorted set에 타임스탬프 저장)나 token bucket을 씁니다. 또 INCR과 EXPIRE가 두 명령이라, 첫 INCR 직후 프로세스가 죽으면 TTL 없는 키가 남을 수 있으니, 엄격하게는 Lua 스크립트로 묶는 게 안전합니다. 대부분의 API 보호엔 위 수준으로 충분합니다.

REDIS-08

TTL·만료·메모리

Redis를 캐시로 쓴다는 건 "이 데이터는 언젠가 사라진다"를 전제로 코드를 짠다는 뜻입니다. TTL은 메모리를 지키는 안전장치이자, stale 데이터를 자동 청소하는 수단입니다.

명령의미
EXPIRE key 60키를 60초 뒤 만료로 설정
TTL key남은 만료 시간(초). -1=만료 없음, -2=키 없음
PERSIST key만료 제거(영구화)
SET key v EX 60set과 동시에 TTL(초). PX는 밀리초
SETEX key 60 v위와 같은 의미의 전용 명령
redis-cli — TTL 상태 읽기 127.0.0.1:6379> SET token:abc "..." EX 30
OK
127.0.0.1:6379> TTL token:abc
(integer) 28
127.0.0.1:6379> TTL nonexistent
(integer) -2 # 키 자체가 없음

메모리가 한계에 닿으면 Redis는 maxmemory-policy에 따라 키를 쫓아냅니다(eviction). 캐시 용도라면 보통 LRU/LFU 계열을 씁니다.

eviction 정책동작
noeviction쫓아내지 않음 — 쓰기가 에러로 거부됨(기본값)
allkeys-lru모든 키 중 가장 오래 안 쓴 것부터 제거(범용 캐시에 무난)
allkeys-lfu가장 적게 쓴 것부터 제거(접근 빈도 기준)
volatile-ttlTTL 있는 키 중 만료가 임박한 것부터 제거
위험 — 캐시는 "있으면 좋은 것"이지 "있어야 하는 것"이 아니다

eviction이 기본 noeviction이면, 메모리가 차는 순간 모든 쓰기가 실패합니다 — 캐시로 쓸 거면 allkeys-lru 등으로 명시하세요. 더 중요한 건 코드 쪽 전제입니다: 캐시 키는 언제든 사라질 수 있으므로, 캐시 miss를 정상 경로로 다뤄야 합니다(REDIS-04의 get-or-load). "캐시에 분명히 넣었으니 있겠지"라고 가정하면, eviction/만료/재시작 후에 NPE로 터집니다.
