DevOps
Cheat Sheet

배포와 운영 변경을 재현성, 추적성, 롤백 가능성, 관측 가능성 기준으로 빠르게 점검하는 참조입니다.

COMPILED : 2026-06 · FORMAT : QUICK REFERENCE
DO-CS-01

개요

DevOps의 핵심은 자동화 자체가 아니라 변경을 작게 만들고, 검증하고, 문제가 생기면 빠르게 멈추거나 되돌리는 운영 체계입니다.

상황먼저 확인할 것운영 질문
배포 전artifact digest, test gate, approval, rollback plan정확히 무엇이 운영에 올라가는가?
배포 중error rate, latency, saturation, health check어떤 지표가 나빠지면 멈출 것인가?
장애 대응recent change, blast radius, log/metric/trace사용자 영향 범위와 최근 변경은 무엇인가?
인프라 변경plan diff, drift, secret exposure, dependency이 변경이 어떤 리소스를 교체/삭제하는가?
Secret 변경rotation, consumer rollout, revocation, audit구버전 앱이 새 secret을 읽을 수 있는가?
DO-CS-02

핵심 명령/개념

개념압축 설명실무 체크
CImerge 전후 빌드, 테스트, 정적 검사로 통합 위험을 낮춤빠르고 결정적이며 실패 원인이 읽혀야 함
CD검증된 artifact를 환경에 배포하고 승격함환경별 재빌드보다 동일 artifact 승격이 바람직
Blue-green신/구 환경을 동시에 두고 트래픽을 전환DB schema와 background job 중복 실행 주의
Canary일부 트래픽부터 점진적으로 확대중단 기준과 자동/수동 rollback 기준 필요
REDRate, Errors, Duration 중심 서비스 지표API와 사용자 흐름의 품질 확인
USEUtilization, Saturation, Errors 중심 리소스 지표CPU, memory, disk, network 병목 확인
Drift코드에 선언된 인프라와 실제 상태의 차이수동 변경이 누적되면 재현성과 복구성이 깨짐
DO-CS-03

판단 기준

선택기준주의점
Rolling자원 추가가 어렵고 신/구 버전 공존이 안전함중간 상태에서 호환성이 깨지면 장애 발생
Blue-green빠른 전환과 빠른 rollback이 중요하고 자원 2배를 감당함DB와 queue consumer는 단순 전환이 어려움
Canary사용자 영향 반경을 작게 시작하고 지표 기반 확대가 가능함트래픽 분할과 지표 민감도가 충분해야 함
Manual gate운영 영향이 크고 사람의 판단이 필요한 release승인자가 볼 diff와 위험 요약이 있어야 함
IaC인프라 상태를 재현 가능하게 version control 해야 함plan 검토, state 보호, secret 분리 필요
Feature flag배포와 기능 공개를 분리해야 함flag cleanup을 안 하면 코드 경로가 복잡해짐
DO-CS-04

자주 나는 실수

  • CI가 통과했지만 어떤 artifact가 운영에 배포됐는지 추적하지 않는다. commit, image digest, config version이 연결돼야 한다.
  • rollback을 문서로만 두고 실제 리허설하지 않는다. DB migration, message schema, external side effect는 단순 되돌리기가 아니다.
  • secret을 이미지, 로그, build argument에 남긴다. secret은 빌드 산출물에 들어가면 회수 비용이 커진다.
  • 배포 후 볼 지표와 중단 기준 없이 canary를 시작한다. canary는 “조금 배포”가 아니라 “증거 기반 확대”다.
  • IaC plan diff를 검토하지 않고 apply한다. replacement, deletion, security rule broadening은 반드시 눈으로 확인한다.
  • 알람을 많이 만들면 안전하다고 생각한다. action 없는 알람은 피로만 만들고 진짜 장애 신호를 묻는다.
DO-CS-05

면접/리뷰용 한 줄 답변

CI/CDpipeline
CI/CD는 빠른 배포가 아니라 같은 입력에서 같은 artifact를 만들고 검증된 변경만 환경에 승격하는 체계입니다.
Rollbackrecovery
rollback은 버튼이 아니라 앱, DB, 메시지, 설정이 이전 버전과 호환되는지 확인한 운영 절차입니다.
관측성observability
로그, 메트릭, 트레이스는 장애가 난 뒤 원인을 좁히기 위한 증거 체계입니다.
IaCreproducibility
IaC는 클릭을 코드로 바꾸는 것이 아니라 인프라 변경을 리뷰, 재현, 복구 가능한 절차로 만드는 것입니다.
배포 전략release
배포 전략은 속도보다 영향 범위, 관측 가능성, 되돌릴 수 있는지를 기준으로 고릅니다.
DO-CS-06

Red Flags

DevOps 리스크는 자동화 여부보다 변경의 추적성, 롤백 가능성, 관측 가능성, secret 노출 여부로 판단합니다.

신호왜 위험한가즉시 확인
rollback 절차가 없거나 리허설되지 않음장애 때 배포 중단과 복구 판단이 늦어집니다.앱, DB, queue, config를 각각 되돌릴 수 있는지 확인합니다.
운영 artifact digest를 모름어떤 코드와 설정이 실행 중인지 추적할 수 없습니다.commit SHA, image digest, config version을 연결합니다.
운영 변경이 IaC 밖에서 수동으로 수행됨drift가 누적되어 재현과 복구가 깨집니다.plan diff와 실제 리소스 상태를 비교합니다.
secret이 CI 로그나 build artifact에 노출폐기와 재발급이 전체 공급망 이슈가 됩니다.log masking, build arg, image layer, rotation 이력을 봅니다.
알람에 owner와 action이 없음소음이 쌓이고 실제 장애 신호가 묻힙니다.알람별 runbook, severity, paging 기준을 정리합니다.
DO-CS-07

Review Gate

변경 유형merge/deploy 차단 기준통과 증거
Pipeline테스트 결과는 있으나 어떤 artifact가 승격되는지 불명확함동일 artifact promotion과 digest 기록이 있습니다.
Deployment중단 기준, rollback 기준, 사용자 영향 범위가 없음canary/rolling 지표와 복구 절차가 정의돼 있습니다.
Migration앱 구버전과 신버전이 동시에 동작할 수 없음expand/contract, backfill, rollback 계획이 있습니다.
Secretrotation 순서와 consumer rollout 순서가 없음dual-read, revocation, audit 기준이 있습니다.
Observability장애 때 봐야 할 dashboard와 log query가 없음RED/USE 지표와 trace 연결이 준비돼 있습니다.
DO-CS-08

Failure Playbook

DEPLOYMENT FAILURE TRIAGE배포 실패는 “재시도”보다 먼저 artifact, 변경 범위, 사용자 영향, 데이터 변경 여부를 고정합니다.
순서질문증거
1현재 운영 중인 SHA, image digest, config version은 무엇인가?deploy log, registry, release record
2사용자 영향은 error rate, latency, 특정 tenant 중 어디에 나타나는가?dashboard, alert, support ticket
3DB migration, queue schema, feature flag가 함께 변경됐는가?migration log, consumer lag, flag audit
4rollback이 data loss나 compatibility break를 만들지 않는가?schema 호환성, background job 상태, side effect
5rollback, canary pause, traffic drain, flag off 중 무엇을 먼저 실행할 것인가?runbook과 on-call 승인 기록
DO-CS-09

배포 실행 카드

DevOps 치트시트는 배포를 빠르게 하는 문서가 아니라, 배포를 멈춰야 할 순간을 빠르게 찾는 문서입니다.

시점바로 확인정상 신호멈출 기준
배포 전commit SHA, image digest, config version, migration 목록운영에 올라갈 artifact가 하나로 특정됨환경별 재빌드 또는 digest 미기록
Canary 시작error rate, p95 latency, saturation, business event중단 기준과 관찰 창이 정해져 있음지표 없이 일부 트래픽만 보내는 상태
DB 동반 배포expand/contract, backfill, old/new app 호환구버전과 신버전이 같은 schema에서 동작 가능rollback 시 데이터 shape가 깨짐
Secret rotationdual-read, consumer rollout, revoke 시점, audit구/신 secret 공존 기간과 폐기 기준이 있음한 번에 교체해 일부 consumer가 즉시 실패
배포 후release dashboard, logs, trace, support signal사용자 영향과 최근 변경 연결 가능성공 로그만 보고 관찰 종료
DO-CS-10

운영 판정표

신호위험 판정즉시 행동남길 기록
error rate 상승canary 또는 특정 route에서 기준 초과canary pause, flag off, rollback 후보 확인release id, route, error code, first seen

rollback 요청DB, queue, external side effect 동반단순 앱 rollback 전에 호환성 확인되돌릴 것과 되돌릴 수 없는 것 분리
alert noiseowner/action 없는 반복 알람severity, owner, runbook, silence 기준 정리알람 개선 PR 또는 runbook 링크
