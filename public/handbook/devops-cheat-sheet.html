DO-CS-01

핵심 명령·스니펫

kubectl — 상태 확인

kubectl get pods -o wide                 # 노드/IP까지 한눈에
kubectl get pods -w                      # 상태 변화 실시간 watch
kubectl logs -f <pod>                    # 로그 follow
kubectl logs <pod> --previous            # 크래시 직전 컨테이너 로그
kubectl describe pod <pod>               # 이벤트/스케줄링 실패 원인
kubectl get events --sort-by=.lastTimestamp   # 최근 이벤트 시간순
kubectl top pod                          # 실시간 CPU/mem (metrics-server 필요)

배포 상태·롤아웃

kubectl rollout status deploy/<x>       # 롤아웃 완료까지 block
kubectl rollout history deploy/<x>      # revision 목록
kubectl rollout undo deploy/<x>         # 직전 revision으로 롤백
kubectl rollout undo deploy/<x> --to-revision=<n>   # 특정 revision
kubectl rollout pause deploy/<x>        # 카나리 중 진행 정지
kubectl rollout restart deploy/<x>      # 이미지 태그 그대로 재기동

실행 중인 artifact 특정 (digest / commit)

git rev-parse --short HEAD                              # 현재 commit
kubectl get pod <pod> -o jsonpath='{..imageID}'        # 실행 중 image digest
kubectl get deploy <x> -o jsonpath='{..image}'         # 배포된 image 태그
kubectl get cm <name> -o yaml                           # 적용된 config 내용
주의

태그(:latest, :v1.2)는 재푸시되면 내용이 바뀔 수 있다. 무엇이 돌고 있는지 확정하려면 image digest(@sha256:...)를 봐라. digest 없이 "롤백했다"는 증거가 없다.

배포 검증 (health / 버전)

curl -f https://<host>/healthz          # 실패 시 non-zero exit (게이트용)
curl -fsS https://<host>/version         # 배포 버전이 기대값인지
curl -s -o /dev/null -w '%{http_code}\n' https://<host>/   # 상태코드만

helm

helm upgrade --install <release> <chart> -f values.yaml   # 없으면 설치, 있으면 갱신
helm diff upgrade <release> <chart> -f values.yaml        # 적용 전 diff (plugin)
helm history <release>                                    # revision 이력
helm rollback <release> <revision>                        # 특정 revision 복구
주의

helm rollback / kubectl rollout undo는 앱 코드만 되돌린다. DB migration, message schema, 외부 side effect가 함께 나갔다면 롤백은 안전하지 않다 — ch4로.

DO-CS-02

판단 한 줄

선택언제 쓰나주의
Rolling자원 여유 없고 구/신 공존이 안전할 때 (기본값)중간 상태에서 호환 깨지면 장애
Blue-green즉시 전환·즉시 롤백이 중요하고 자원 2배 감당DB·queue consumer는 단순 전환 불가
Canary영향 반경을 작게 시작해 지표로 확대중단 기준·관찰 창 없으면 그냥 "조금 배포"
Rollback직전 버전이 확실히 정상이고 데이터 변경 없음DB/schema 동반 시 roll-forward 검토
Roll-forward롤백이 data loss·호환 깨짐을 유발할 때수정 배포까지 시간 필요 — 그동안 완화책
Feature flag off문제 기능만 즉시 끄고 배포는 유지flag 없으면 불가 — redeploy가 유일
Redeployflag로 못 끄는 코드 경로 문제롤아웃 시간 + 재검증 필요
Config 변경동작만 바꾸면 되고 코드는 그대로config drift·미기록 시 재현 불가
Code 변경로직 자체를 고쳐야 할 때새 artifact = 새 digest = 재검증
DO-CS-03

자주 터지는 것 · 즉시 확인

신호왜 위험즉시 확인
어떤 artifact가 배포됐는지 모름 (digest 미확인)무엇이 돌고 있는지 추적 불가 → 롤백 대상도 불명kubectl get pod -o jsonpath='{..imageID}'로 digest, commit SHA 연결
rollback을 실행만 하고 검증 안 함"되돌렸다" 착각 — 태그 캐시/파드 미교체로 구 버전이 그대로rollout status + curl /version + digest 재확인
health check 없이 배포 성공 판정파드 Running이어도 앱은 죽어있을 수 있음readiness probe, curl -f /healthz exit code
config drift (수동 변경 누적)선언 상태와 실제가 달라 재현·복구 깨짐kubectl diff -f / helm diff로 실제 vs 선언 비교
secret이 로그·image·env에 노출폐기·재발급이 공급망 전체 이슈로 확대CI 로그 masking, image layer, get pod -o yaml env 점검
DB migration이 배포와 함께 나감롤백 시 데이터 shape 깨짐 — 단순 되돌리기 불가expand/contract 여부, 구/신 앱 동시 동작 가능한지
DO-CS-04

장애 증거 1카드

DEPLOYMENT FAILURE TRIAGE재시도·롤백 누르기 전에 먼저 무엇이 돌고 있고, 무엇이 함께 바뀌었는지 증거를 고정한다.
1. 실행 중 image digest · config version 확인
2. health check / 상태코드 확인
3. error rate · latency 확인
4. 데이터 변경(DB migration·schema) 동반 여부 확인
5. rollback 후보 결정 — 단, 데이터 변경 동반이면 rollback 보류
순서질문증거
1지금 운영에 돌고 있는 image digest와 config version은?kubectl get pod -o jsonpath='{..imageID}', cm/secret version, commit SHA
2앱이 실제로 살아있나? health가 통과하나?curl -f /healthz exit code, readiness probe, describe pod 이벤트
3사용자 영향은 어디에? error rate / latency 어디서 오르나?dashboard, 상태코드 분포, 특정 route/tenant
4이번 배포에 DB migration·schema·flag가 함께 나갔나?migration log, rollout history, flag audit
5rollback / pause / flag off 중 무엇을 먼저?runbook, on-call 승인 기록
복구·중단 기준

데이터 변경(DB migration·schema)이 동반된 배포는 rollback 보류 — 구 버전이 새 schema를 못 읽으면 롤백이 2차 장애가 된다. 이 경우 roll-forward(수정 배포) 또는 kubectl rollout pause로 확산부터 멈추고 호환성을 확인한 뒤 결정한다.
