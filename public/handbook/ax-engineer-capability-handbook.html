AX 엔지니어
역량 모델

AX 엔지니어는 AI를 잘 쓰는 사람이 아니라 업무를 AI가 안전하게 수행할 수 있는 실행 시스템으로 바꾸는 사람입니다. 핵심 역량은 프롬프트 작성보다 업무 분해, 컨텍스트 설계, 검증, 권한 통제, 효과 측정에 있습니다.

STANCE : AUTOMATION ARCHITECT · VERIFICATION OWNER · SOCIOTECHNICAL OPERATOR
CAP-01

역할 정의

AX 엔지니어의 산출물은 “AI가 만든 코드”가 아니라 “AI가 반복 가능한 업무를 수행하고, 실패하면 멈추고, 결과가 검증되는 운영 체계”입니다.

구분AI 사용자AX 엔지니어
문제 인식어떤 프롬프트를 넣을까이 업무는 자동화해도 되는가, 어디까지 맡길 수 있는가
입력 설계요청을 자연어로 설명목표, 비목표, 컨텍스트, 권한, 검증, 중단 조건을 명세화
실행 방식채팅 또는 IDE 보조하네스, 루프, 작업 큐, 리뷰 게이트, 감사 로그로 운영
품질 보증결과를 읽어보고 판단테스트, 정적 분석, 독립 리뷰, adversarial case, rollout metric 연결
조직 적용개인 생산성 향상팀의 업무 흐름, 정책, 지표, 지식 저장소, 교육 체계까지 설계

CAP-03

업무 분해 관점

업무 자동화는 “이 일을 AI에게 시킬 수 있나”가 아니라 “이 일을 어떤 단위로 나누면 각 단위가 검증 가능한가”에서 시작합니다.

WORKFLOW DECOMPOSITION Trigger: 언제 시작되는가
Input: 어떤 파일, 티켓, 로그, 지표, 정책을 읽는가
Decision: 어떤 판단이 필요한가
Action: 어떤 도구를 실행하고 무엇을 수정하는가
Verification: 무엇이 통과해야 하는가
Escalation: 언제 사람에게 넘기는가
Audit: 어떤 근거를 남기는가
예시 업무좋은 분해나쁜 분해
코드 리뷰risk classification → test gap → security check → finding triage이 PR 리뷰해줘
장애 분석symptom → blast radius → recent change → log/metric correlation → mitigation왜 장애 났는지 찾아줘
마이그레이션 검토schema diff → lock risk → rollback/forward → staging dry-run → approval이 migration 괜찮아?
문서 동기화API diff → affected docs → example validation → changelog update문서 최신화해줘
CAP-04

자동화 판단 기준

AX 엔지니어는 모든 업무를 자동화하려 하지 않습니다. 자동화가 적합한 업무, 보조만 해야 하는 업무, 금지해야 하는 업무를 구분합니다.

기준높으면 좋은가해석
반복성높을수록 자동화 후보같은 입력/출력 구조가 반복될수록 하네스화 가치가 큼
검증 가능성높을수록 자동화 후보테스트, lint, policy check, metric으로 판정 가능해야 함
컨텍스트 안정성높을수록 자동화 후보규칙과 자료 위치가 자주 바뀌지 않을수록 안정적
권한 위험낮을수록 자동화 후보DB write, deploy, customer data 접근은 승인 게이트 필요
실패 비용낮을수록 자동화 후보실패가 운영 장애나 보안 사고로 이어지면 자동 실행 금지
전문가 병목높으면 반자동 후보전문가 검토는 남기되 초안, 후보, 증거 수집을 자동화
판단 원칙

반복성과 검증 가능성이 높고 실패 비용이 낮은 업무부터 자동화합니다. 실패 비용이 높지만 반복성이 큰 업무는 “자동 실행”이 아니라 “자동 분석 + 사람 승인” 패턴으로 시작합니다.
