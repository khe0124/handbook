CD-00

pipeline stage

좋은 pipeline은 install, lint, typecheck, test, build, artifact publish, deploy, smoke test, metric gate가 분리되어 있다. 실패한 stage가 명확해야 재시도와 rollback 판단이 빨라진다.

운영 질문확인할 증거판단 기준
pipeline stage가 실패 위치와 재시도 기준을 명확히 드러내는가?install/lint/typecheck/test/build/publish/deploy/smoke/metric gate stage result배포 실패가 어느 stage의 문제인지 즉시 구분되어야 한다.
pipeline stage를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?CI run, artifact publish log, deploy marker, smoke result, metric gateCI green은 운영 성공이 아니라 운영 검증의 시작이다.
pipeline stage 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?deploy는 성공했지만 smoke test 없이 오류율 증가stage를 분리하고 metric gate와 rollback trigger를 추가한다.
CD-01

artifact

운영에 배포되는 단위는 branch가 아니라 artifact다. image digest, commit SHA, migration version, config version, build provenance를 묶어야 어떤 코드가 어디에 배포됐는지 추적된다.

운영 질문확인할 증거판단 기준
운영 배포 단위가 branch가 아니라 불변 artifact로 추적되는가?commit SHA, image digest, migration version, config version, SBOM/provenanceproduction에 무엇이 배포됐는지 digest 기준으로 답해야 한다.
artifact를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?release note, registry digest, deploy manifest, migration table, config diffbranch명이나 tag만으로는 재현성과 rollback 증거가 약하다.
artifact 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?어떤 이미지가 production에 있는지 모름artifact digest와 config version을 release note에 묶는다.
CD-02

environment

development, staging, production은 같은 artifact에 다른 environment config를 주입하는 구조가 안정적이다. secret, public config, runtime config를 분리하고 권한 있는 approval 없이 production 변경이 나가지 않게 한다.

운영 질문확인할 증거판단 기준
환경별 차이가 artifact가 아니라 runtime config로 관리되는가?environment config, secret source, approval gate, feature flag, public/runtime config split같은 artifact가 환경 설정만 바꿔 승격되어야 한다.
environment를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?config diff, secret reference, approval log, env variable inventoryproduction secret이 build-time bundle에 섞이면 경계 실패다.
environment 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?staging 정상, production만 config 오류config diff와 secret source를 검증하고 smoke test를 보강한다.
CD-03

rollback과 release strategy

rolling, blue-green, canary, feature flag는 모두 rollback 비용을 줄이기 위한 선택지다. DB 변경은 expand-contract로 나누고, rollback이 불가능한 변경은 forward fix와 feature off 절차를 먼저 준비한다.

운영 질문확인할 증거판단 기준
release strategy가 rollback 비용과 DB 호환성을 낮추는가?rolling/blue-green/canary/feature flag, migration compatibility, metric gate, rollback triggerDB 변경은 expand-contract와 forward fix 가능성을 먼저 본다.
rollback과 release strategy를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?canary metric, rollout status, feature flag state, migration version, rollback commandrollback 불가 변경은 사전 승인과 완화책이 필요하다.
rollback과 release strategy 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?migration 포함 배포 후 이전 artifact rollback 실패flag off, forward fix, expand-contract 재설계를 실행한다.

CHECK

실무 체크리스트

CI/CD·Artifact·Environment 체크리스트는 pipeline stage, artifact, environment, rollback과 release strategy의 변경 위험을 사전에 드러내기 위한 검토표다. 각 항목은 리뷰, 배포 승인, 운영 인수에서 같은 증거로 다시 확인되어야 한다.

  • install, test, build, artifact publish, deploy, smoke test를 stage로 분리한다.
  • commit SHA, image digest, migration version, config version을 release note에 묶는다.
  • environment별 approval과 secret 주입 경계를 둔다.
  • rollback trigger와 metric gate를 배포 전에 정한다.
작업 단계필수 산출물차단 기준
설계목적, 영향 범위, 비목표, owner영향 범위가 사용자·시스템·비용 기준으로 설명되지 않음
변경diff, 승인 기록, 검증 명령production 변경과 staging 검증의 차이가 설명되지 않음
복구rollback 또는 forward fix, 담당자, 예상 시간실패 시 되돌릴 수 없는데 사전 승인과 공지가 없음

CASE

장애/운영 시나리오

staging에서는 정상인데 production 배포 후 오류율이 증가한다. 배포된 artifact digest와 config version을 먼저 확인하고, feature flag를 끄거나 이전 artifact로 rollback한다. 이후 pipeline에 production smoke test와 error rate gate를 추가한다.

Triage frame

CI/CD·Artifact·Environment 장애는 증상 → 영향 범위 → 최근 변경 → Artifact 증거 → 완화책 → 복구 검증 → 재발 방지 순서로 기록한다. 이 흐름이 있어야 담당자가 바뀌어도 같은 결론에 도달한다.

처음 10분다음 30분종료 조건
commit SHA, image digest, SBOM/provenance, migration version 기준으로 영향 범위를 고정한다.release note, image digest, deploy marker, config version 비교에서 증거를 모아 완화책을 실행한다.같은 실패가 자동 차단되도록 한다.
Incident packet기록할 내용판정 기준
증상production 배포 후 오류율 상승artifact, config, migration, flag, dependency 변경을 대조한다.
완화flag off, canary stop, previous artifact rollback, traffic shift가장 작은 폭발 반경으로 되돌린다.
사후metric gate와 smoke test를 실패 케이스 기준으로 강화같은 실패가 자동 차단되도록 한다.
RISK

자주 틀리는 판단

CI/CD·Artifact·Environment에서 자주 틀리는 판단은 pipeline stage와 rollback과 release strategy의 책임 경계를 흐리는 데서 시작된다. 아래 신호가 보이면 설계, runbook, 답변 구조를 다시 점검한다.

  • branch 이름만 보고 어떤 artifact가 배포됐는지 모른다.
  • CI 통과를 운영 배포 가능성과 동일하게 본다.
  • DB migration rollback 가능성을 확인하지 않는다.
  • production secret을 build-time bundle에 섞는다.
Q&A

면접 답변 템플릿

CI/CD는 자동화가 아니라 변경 통제 시스템이라고 답합니다. 같은 artifact를 환경별 설정으로 승격하고, 배포 후 smoke test와 metric gate를 통과해야 하며, 실패 시 rollback 또는 feature off 절차가 있어야 합니다.

OPERATIONS ANSWER FRAME 1. CI/CD는 변경 통제와 복구 시스템이라고 설명한다.
2. artifact, environment, metric gate, rollback을 한 흐름으로 묶는다.
3. DB migration은 expand-contract와 호환성 검증으로 다룬다.
