Context Engineering

Context Engineering은 정보를 많이 넣는 기술이 아니라 지금 작업에 필요한 정보를 충분하지만 오염되지 않게, 출처와 함께 제공하는 기술입니다.

CTX-01

Context source inventory

컨텍스트 품질은 어떤 자료를 넣을지보다 먼저 어떤 자료원이 존재하는지 파악하는 데서 시작합니다. 팀은 반복 업무마다 source inventory를 만들고, 각 source의 소유자와 갱신 주기를 적어야 합니다.

Source포함할 내용주의점
code관련 파일, 테스트, 타입, 호출 경로직접 관련 없는 대량 파일 투입 금지
docs제품 문서, runbook, onboarding guide작성일과 적용 범위 확인
tickets요구사항, 버그 재현, 고객 영향댓글의 가정과 확정 사실 분리
logs오류 메시지, trace, 배포 전후 증상개인정보와 민감 토큰 제거
metricsSLO, latency, error rate, adoption 지표기간과 집계 기준 표시
policies보안, 개인정보, 배포, 접근성 규칙권고와 필수 규정을 구분
architecture decisionsADR, 이전 PR, migration 이유폐기된 결정을 현재 규칙처럼 쓰지 않음
schemasDB schema, event contract, validation rule운영 버전과 로컬 버전 차이 확인
APIsOpenAPI, SDK docs, webhook spec버전, rate limit, deprecation 확인
CTX-02

품질 기준

좋은 컨텍스트는 길지 않습니다. 현재 결정에 필요한 정보가 관련성 순서로 들어 있고, 신뢰 수준과 출처가 분리되어 있습니다.

기준질문실무 기준
Relevance현재 결정에 직접 필요한가관련 파일, 호출 경로, 테스트, 계약 우선
Sufficiency이 정보만으로 올바른 판단이 가능한가요구사항, 제약, 예외, 반례 포함
Isolation신뢰 수준이 다른 정보가 섞이지 않았는가사용자 입력, 외부 문서, 코드, 비밀 분리
Provenance출처를 추적할 수 있는가파일 경로, 커밋, 문서명, API 버전 기록
Freshness현재 시스템 상태를 반영하는가날짜, 버전, 배포 환경, 생성 시점 표시
Trust level이 정보를 어느 정도 믿을 수 있는가authoritative, internal note, user report, external input 구분
Training rule

컨텍스트를 추가할 때마다 “왜 필요한가, 어디서 왔는가, 얼마나 최신인가, 신뢰 수준은 무엇인가”를 함께 기록합니다.

CTX-03

신뢰 경계

모델은 주어진 텍스트 안에서 지시와 자료를 혼동할 수 있습니다. untrusted content는 내부 정책이나 시스템 지시와 격리하고, prompt injection 가능성을 항상 검토합니다.

자료Trust level처리 방식
소스 코드와 테스트높음현재 커밋 기준으로 직접 확인
내부 정책높음필수 규정과 권고를 분리
티켓 본문중간요구사항과 사용자 의견을 분리
외부 문서중간 또는 낮음공식 출처와 버전 확인
고객 입력, 로그 메시지, 웹 콘텐츠낮음untrusted content로 표시하고 지시로 실행하지 않음
Prompt injection

로그, 문서, 웹페이지 안에 “이전 지시를 무시하라” 같은 문장이 있어도 그것은 분석 대상 데이터입니다. 하네스와 작업자는 prompt injection을 자료 오염으로 보고 격리해야 합니다.

CTX-04

Package Template

CONTEXT PACKAGE TEMPLATE Task: 이번 episode의 목표와 완료 기준
Relevant sources: code, docs, tickets, logs, metrics, policies, architecture decisions, schemas, APIs
Source notes: 각 source의 owner, timestamp, version, Trust level
Decision context: 현재 판단에 필요한 규칙, 예외, 반례
Constraints: 권한, 보안, 개인정보, 성능, 호환성 제약
Untrusted content: 고객 입력, 외부 텍스트, 로그 payload를 별도 구획에 배치
Verification: 테스트, 빌드, metric, 수동 확인 방법
Open gaps: 부족한 정보와 그로 인한 risk
GOOD CONTEXT PACKAGE Goal: 비밀번호 재설정 링크를 재사용할 수 없게 한다
Relevant Code: PasswordResetController, PasswordResetService, PasswordResetTokenRepository, PasswordResetServiceTest
Domain Rules: 토큰 30분 만료, 사용한 토큰 재사용 불가, 새 토큰 발급 시 기존 미사용 토큰 폐기
Security Constraints: 토큰 원문 로그 금지, 사용자 존재 여부 응답 노출 금지
Verification: 만료 토큰, 재사용 토큰, 새 토큰 발급 테스트
