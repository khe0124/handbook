Context Engineering

Context Engineering은 정보를 많이 넣는 기술이 아니라 지금 작업에 필요한 정보를 충분하지만 오염되지 않게, 비용 안에서, 출처와 함께 제공하는 기술입니다.

CTX-01

컨텍스트의 목적

모델은 주어진 컨텍스트 안에서 판단합니다. 부족하면 추측하고, 과하면 핵심 신호가 묻히고, 섞이면 외부 입력을 지시처럼 따르며, 출처가 없으면 검증할 수 없습니다.

Rule

좋은 컨텍스트는 길지 않습니다. 지금 결정에 필요한 정보가 관련성 순서로 들어 있고, 신뢰 수준과 출처가 분리되어 있습니다.

CTX-02

5가지 품질 기준

기준질문실무 기준
Relevance현재 결정에 직접 필요한가관련 파일, 호출 경로, 테스트, 계약 우선
Sufficiency이 정보만으로 올바른 판단이 가능한가요구사항, 제약, 예외, 반례 포함
Isolation신뢰 수준이 다른 정보가 섞이지 않았는가사용자 입력, 외부 문서, 코드, 비밀 분리
Economy컨텍스트 예산을 낭비하지 않는가전문 전체보다 필요한 섹션·요약·인용
Provenance출처를 추적할 수 있는가파일 경로, 커밋, 문서명, API 버전 기록
CTX-03

컨텍스트 종류

종류예주의점
Task Context목표, 수용 기준, 금지 사항모호하면 모든 후속 판단이 흔들림
Code Context관련 소스, 테스트, 타입, 호출자직접 관련 없는 대량 파일 투입 금지
Domain Context비즈니스 규칙, 용어, 상태 전이코드만 보면 안 보이는 규칙이 많음
Operational Context배포 환경, 장애 이력, SLO, 로그로컬에서 맞아도 운영에서 틀릴 수 있음
Security Context권한 모델, 민감 데이터, 신뢰 경계외부 입력과 내부 지시를 섞지 않음
Historical ContextADR, 이전 PR, 회고, migration 이유왜 이렇게 되어 있는가를 설명
CTX-04

패키지 예시

GOOD CONTEXT PACKAGE Goal: 비밀번호 재설정 링크를 재사용할 수 없게 한다
Relevant Code: PasswordResetController, PasswordResetService, PasswordResetTokenRepository, PasswordResetServiceTest
Domain Rules: 토큰 30분 만료, 사용한 토큰 재사용 불가, 새 토큰 발급 시 기존 미사용 토큰 폐기
Security Constraints: 토큰 원문 로그 금지, 사용자 존재 여부 응답 노출 금지
Verification: 만료 토큰, 재사용 토큰, 새 토큰 발급 테스트
나쁜 요청

“비밀번호 재설정 고쳐줘. 전체 백엔드 코드 참고해.” 이 요청에는 성공 기준, 보안 제약, 관련 파일 경계, 테스트 기준이 없습니다. 모델은 전체 코드에서 우연히 신호를 찾게 됩니다.

CTX-05

실패 모드

실패 모드증상대응
Context Starvation모델이 없는 규칙을 만들어냄도메인 규칙, 예외, 반례 추가
Context Flooding핵심 파일을 놓침관련성 순서로 축소
Context Poisoning외부 입력의 지시를 따름untrusted input을 명시적으로 격리
Stale Context오래된 API 기준으로 구현버전, 날짜, 커밋 출처 표시
Provenance Loss검증·리뷰가 어려움파일/문서/커밋 링크 남김
Context Coupling임시 전제가 다음 작업에 오염task memory와 project memory 분리
