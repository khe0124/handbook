UX 사고와
문제 정의

UX 설계의 출발점은 화면이 아니라 문제입니다. 사용자가 무엇을 하려는지, 왜 실패하는지, 제품이 어떤 행동 변화를 만들고 싶은지를 명확히 해야 합니다.

UX-01

문제와 솔루션

“검색 필터를 추가하자”는 솔루션이고, “사용자가 원하는 항목을 찾지 못한다”가 문제입니다.

구분질문예시
문제사용자가 어디서 막히는가결제 전 배송비를 이해하지 못함
원인왜 막히는가배송비 정보가 마지막 단계에만 노출
솔루션무엇을 바꿀 것인가상품 상세와 장바구니에 예상 배송비 표시
검증좋아졌는지 어떻게 알 것인가결제 이탈률, 문의 감소, 테스트 관찰
솔루션 점프

팀이 바로 UI를 그리기 시작하면 문제를 잘못 풀 위험이 커집니다. 먼저 사용자의 실패 상황, 빈도, 영향, 현재 우회 방법을 확인합니다.

UX-02

사용자 이해

도구쓸 때주의
JTBD사용자가 제품을 “고용”하는 목적을 볼 때인구통계보다 상황과 동기를 봄
Persona팀이 사용자를 같은 언어로 이해할 때가상의 취향 묘사로 흐르지 않게 함
Scenario사용 맥락과 제약을 설명할 때happy path만 쓰지 않음
Journey map긴 여정의 감정과 접점을 볼 때실제 데이터와 인터뷰로 보정
실무 감각

업무 시스템에서는 “사용자”가 구매자, 승인자, 실무자, 관리자, 감사자처럼 여러 역할로 나뉩니다. 같은 화면도 역할별 목표와 권한이 다르면 완전히 다른 UX가 됩니다.

UX-03

가설과 지표

UX HYPOTHESIS 만약 [사용자 문제]를 [디자인 변경]으로 줄이면,
[사용자 행동]이 바뀌고,
[제품 지표]가 개선될 것이다.
지표예시주의
행동 지표완료율, 이탈률, 재시도율사용자 목표와 연결
품질 지표오류율, 문의량, 취소율나쁜 경험의 신호
속도 지표task completion time무조건 짧은 것이 좋은 것은 아님
만족 지표CSAT, NPS, 인터뷰 반응정량 행동과 함께 해석
UX-04

리서치 신호

신호강점한계
Analytics많은 사용자의 행동 패턴왜 그런지는 설명하지 못함
Session replay실제 사용 흐름 관찰개인정보 마스킹과 표본 편향 주의
Interview동기와 맥락 파악말과 실제 행동이 다를 수 있음
Support ticket반복 불만과 실패 지점문제 없는 사용자는 드러나지 않음
Usability test설계안의 이해 가능성 검증표본이 작으므로 패턴 중심 해석
UX-05

의사결정 프레임

Impact영향
얼마나 많은 사용자와 중요한 흐름에 영향을 주는가.
Confidence확신
데이터, 리서치, 실험으로 얼마나 뒷받침되는가.
Effort비용
디자인, 개발, QA, 운영 변경 비용이 어느 정도인가.
Risk리스크
잘못됐을 때 전환율, 신뢰, 데이터, 규정에 어떤 손상이 생기는가.
UX-06

문제 정의 심화

좋은 문제 정의는 사용자의 말, 행동 데이터, 비즈니스 손실, 기술 제약을 분리해서 봅니다.

증거 유형강점주의점
정성 인터뷰맥락과 감정, 우회 행동을 발견말한 선호를 그대로 요구사항으로 바꾸지 않음
행동 로그반복 패턴과 이탈 지점을 확인왜 그런 행동을 했는지는 설명하지 못함
고객 문의실제 고통과 언어를 포착목소리가 큰 일부 고객에 치우칠 수 있음
운영 비용수동 처리와 예외 대응 비용을 드러냄내부 효율만 보고 사용자 가치가 빠질 수 있음
Case study

“검색이 불편하다”는 요청을 그대로 검색창 확대로 해결하면 실패할 수 있습니다. 실제 문제를 보면 필터명이 조직 내부 용어와 달라 사용자가 올바른 조건을 못 고르거나, 결과가 0건일 때 다음 행동이 없어 멈추는 경우가 많습니다. 이때 해결책은 검색 UI 확대가 아니라 용어 정리, 추천 필터, empty state, 최근 사용 조건 저장일 수 있습니다.

Problem statement template 
When [상황]에서
[사용자 세그먼트]는 [목표]를 하려고 하지만
[장애물] 때문에 [측정 가능한 손실]이 발생한다.
우리는 [행동 변화]가 일어나면 문제가 개선됐다고 판단한다.
UX-07

Discovery 시스템

Discovery는 인터뷰 몇 번 하고 결론 내는 단계가 아닙니다. 제품이 풀 문제를 계속 갱신하기 위한 운영 시스템입니다. 정성 신호, 정량 신호, 지원 티켓, 세일즈 피드백, 사용 로그를 같은 문제 저장소로 모아야 합니다.

활동목적좋은 산출물나쁜 신호
Continuous discovery문제의 빈도와 변화 추적주간 learnings, opportunity backlog큰 프로젝트 전에만 리서치함
Interview synthesis발화가 아니라 행동 패턴 도출quote → interpretation → opportunity고객 말을 그대로 기능 요구로 바꿈
Behavior analytics규모와 반복성 확인funnel, cohort, segment 비교평균값만 보고 세그먼트 차이를 놓침
Support mining실패 언어와 예외 흐름 발견ticket taxonomy, severity, workaround문의량을 UX debt로 연결하지 않음
Research repository팀 기억을 자산화problem evidence, decision log, invalidated assumptions리서치 결과가 발표 자료로 끝남
Advanced practice

좋은 Discovery는 “무엇을 만들까”보다 “무엇을 더 이상 믿지 말아야 하는가”를 빠르게 업데이트합니다. 반증된 가정, 포기한 대안, 여전히 불확실한 리스크가 남아 있어야 다음 의사결정의 품질이 올라갑니다.

UX-08

가정·리스크 매핑

제품 디자인의 위험은 대부분 “사용자가 원할 것이다”, “이해할 것이다”, “운영 가능할 것이다”, “기술적으로 쉽게 될 것이다” 같은 검증되지 않은 가정에서 생깁니다. 가정은 명시하고, 위험도에 따라 검증 비용을 다르게 써야 합니다.

가정 유형질문검증 방법실패 신호
Value risk사용자가 이 문제를 충분히 중요하게 여기는가인터뷰, willingness-to-pay, 사용 빈도있으면 좋지만 지금 안 써도 됨
Usability risk사용자가 이 흐름을 이해하고 완료할 수 있는가prototype test, first-click test, task success설명 없이는 다음 행동을 못 찾음
Feasibility risk데이터, API, 권한, 성능 제약 안에서 가능한가technical spike, contract review디자인 상태가 시스템 상태와 맞지 않음
Viability risk비용, 정책, 운영 모델과 맞는가business review, support simulation출시 후 수동 운영 비용이 폭증
ASSUMPTION MAP High impact + low confidence: 지금 검증한다
High impact + high confidence: 설계 기준으로 고정한다
Low impact + low confidence: 작은 실험으로 처리한다
Low impact + high confidence: 과도한 리서치를 피한다
