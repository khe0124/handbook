GFX-01

SVG·Canvas·WebGL 선택

SVG, Canvas, WebGL은 렌더링 단위와 책임이 다릅니다. SVG는 DOM 기반 벡터, Canvas는 명령형 2D bitmap, WebGL은 GPU 파이프라인입니다. 기술 선택은 나중에 바꾸기 어렵기 때문에 “현재 구현 편의”보다 데이터 크기, hit testing, 접근성, 애니메이션 빈도, 메모리 상한으로 결정합니다.

기술가장 잘 맞는 경우강점실패 모드
CSS animation상태 전환, panel, tooltip, skeleton, transform/opacity 중심 모션브라우저 최적화, 접근성·DOM 유지, 구현 비용 낮음layout 속성을 매 프레임 바꾸면 reflow와 paint storm 발생
SVG아이콘, 다이어그램, 지도 일부, node 수가 제한된 인터랙티브 벡터해상도 독립, DOM event, label 연결, 스타일링 쉬움수천 node 이상에서 DOM 비용과 style recalculation 증가
Canvas 2D대량 점·선, 차트, drawing tool, heatmap, 게임형 2D UIDOM node 수와 무관하게 픽셀 단위 제어, draw batching 가능접근성 tree 없음, hit testing 직접 구현, 고해상도에서 fill-rate 비용 증가
WebGL3D, 많은 sprite/particle, shader 효과, GPU 병렬성이 필요한 화면GPU 가속, geometry·texture·shader 제어, 대규모 시각화 가능GPU 메모리 누수, context loss, 디버깅 난도, 저사양 fallback 필요
Three.js제품 3D, configurator, scene graph 기반 상호작용WebGL 추상화, loader·material·camera·controls 생태계추상화 때문에 dispose, draw call, material 수 같은 비용을 놓치기 쉬움
판단 질문SVG 쪽 신호Canvas/WebGL 쪽 신호
개별 요소가 focus, tooltip, semantic label을 가져야 하는가각 요소가 의미 있는 DOM 대상이면 SVG픽셀 덩어리이고 별도 DOM overlay로 충분하면 Canvas/WebGL
화면 요소 수가 얼마나 되는가수십~수백 개의 path/node수천~수십만 개의 점·선·sprite·triangle
변경 빈도가 높은가가끔 상태 변경, hover, selection매 frame 이동, pan/zoom, 물리·shader 애니메이션
품질 저하 없이 fallback이 가능한가정적 SVG 또는 HTML 표로 대체 가능정적 이미지, 저해상도 Canvas, 2D viewer, 제품 사진으로 대체 필요
팀이 디버깅할 수 있는가DOM Inspector와 CSS 지식으로 충분GPU trace, memory, shader, asset pipeline 운영 역량 필요
GRAPHICS DECISION GATE1. 상호작용 대상이 semantic DOM이어야 하는지 먼저 결정한다
2. 최악 데이터 크기와 화면 픽셀 수를 숫자로 적는다
3. 60fps가 필요한 구간과 정적 렌더가 충분한 구간을 나눈다
4. 저사양 기기 fallback과 no-WebGL 경로를 먼저 설계한다
5. 성능 예산을 통과한 prototype trace가 없으면 기술 선택을 확정하지 않는다
COMMON MISTAKE

SVG가 “벡터라 가볍다”거나 WebGL이 “GPU라 무조건 빠르다”는 식으로 고르면 실패합니다. DOM node 수, draw call 수, texture 크기, shader 비용, hit testing 방식 중 실제 병목이 어디 생기는지 prototype으로 확인해야 합니다.

GFX-02

Canvas 실무 패턴

Canvas는 React 컴포넌트처럼 선언적으로 관리되지 않습니다. 한 번 그린 픽셀은 DOM 요소가 아니므로 상태, invalidation, 좌표계, hit testing, 접근성을 직접 설계해야 합니다. 좋은 Canvas 구현은 매 frame 전체를 무작정 다시 그리지 않고, 무엇이 dirty인지 알고 그립니다.

패턴쓸 때구현 기준실패 모드
Retained model + redraw도형·레이어 상태를 JS model로 보관하고 필요 시 다시 그림scene model, dirty flag, deterministic draw ordermodel과 픽셀 상태가 어긋나 undo/selection이 깨짐
Dirty rectangle일부 영역만 자주 바뀌는 editor, chart cursor변경 전/후 bounding box를 합쳐 clear 후 redraw그림자·blur·lineWidth 여백을 빼먹어 잔상이 남음
Layered canvas배경, 데이터, hover, selection 갱신 빈도가 다를 때정적 layer와 동적 layer 분리, 같은 CSS size 유지canvas 여러 장이 memory와 composite 비용을 증가
OffscreenCanvas / worker렌더 계산이나 rasterization이 main thread를 막을 때message contract, cancellation, transfer cost 측정데이터 직렬화 비용이 계산 절감보다 커짐
DOM overlaytooltip, focus ring, form control, screen reader label이 필요할 때좌표 변환, z-index, keyboard focus 순서 정의Canvas pan/zoom과 overlay 위치가 drift

DPR TRAP

고해상도 디스플레이에서 Canvas를 CSS 크기만큼만 만들면 흐리고, device pixel ratio만큼 키우면 메모리와 fill-rate가 증가합니다. DPR을 무조건 3까지 쓰지 말고 화면 크기와 기기 등급에 따라 1.5 또는 2로 cap을 둡니다.

GFX-03

WebGL 기초 모델

WebGL은 DOM에 도형을 추가하는 API가 아니라 GPU에게 버퍼, 텍스처, shader, state를 넘겨 draw call을 실행하는 모델입니다. 문제를 빨리 잡으려면 “무엇을 그리는가”보다 “얼마나 자주 state를 바꾸고, 얼마나 많은 데이터를 GPU에 올리고, 몇 번 draw하는가”를 봐야 합니다.

개념역할실무 판단흔한 오류
Vertex buffer위치, normal, uv, color 같은 attribute 저장정적 geometry는 한 번 upload, 동적 데이터는 update 범위 최소화매 frame 새 buffer 생성으로 GPU memory와 GC 증가
Index buffer정점 재사용 순서 저장mesh가 크면 index 사용으로 vertex 중복 감소index type 한계와 WebGL extension 지원 확인 누락
Shadervertex와 fragment 계산fragment shader는 픽셀 수만큼 실행되므로 full-screen 효과는 특히 측정분기·고비용 함수·큰 texture lookup으로 저사양 GPU에서 급락
Texture이미지, normal map, environment map 등 GPU 이미지 데이터크기, 포맷, mipmap, anisotropy를 예산화4K texture 여러 장으로 모바일 GPU 메모리 초과
Draw callGPU에 한 묶음 그리기 명령 실행material·geometry·state 변경이 많으면 CPU overhead 증가오브젝트 수만 줄이고 material 수를 방치해 draw call이 줄지 않음
Render target화면이 아닌 texture에 렌더링post-processing, picking, reflection에 사용하되 해상도 cap 필요멀티패스 효과가 fill-rate와 메모리를 폭증
WebGL 장애사용자 증상원인 후보대응
Context loss3D 영역이 검거나 멈춤GPU 메모리 압박, 드라이버 reset, 탭 전환webglcontextlost/webglcontextrestored 처리, 자원 재생성 경로 테스트
Shader compile 실패특정 기기에서만 object가 안 보임precision, extension, GLSL 호환성compile log 수집, fallback material, device matrix QA
Texture upload stall모델 진입 순간 긴 freeze큰 이미지 decode/upload를 interaction 시점에 수행프리로드, KTX2/Basis 압축, 점진 로딩, skeleton scene
Overdraw투명 효과가 많은 장면에서 FPS 하락겹치는 transparent mesh, particle, post effect정렬, alpha test, particle 수 cap, 해상도 scale
GPU COST ORDERCPU: scene update → frustum culling → draw call submit
GPU upload: buffer/texture 변경 데이터 전송
Vertex: 정점 수와 skinning/morph 계산
Fragment: 화면 픽셀, overdraw, shader, post-processing
Memory: texture, geometry, render target, shadow map
MENTAL MODEL

WebGL 성능은 “triangle이 몇 개인가” 하나로 설명되지 않습니다. 같은 polygon 수라도 material이 200개면 draw call이 많아지고, 같은 draw call 수라도 4K texture와 bloom이 있으면 GPU memory와 fragment 비용이 커집니다.

GFX-04

Three.js 구조

Three.js는 WebGL을 scene graph, camera, material, loader, renderer로 다루게 해줍니다. 추상화는 빠른 제품 개발에 유리하지만, 자원 생명주기와 frame loop 비용은 여전히 개발자가 책임져야 합니다.

구성요소역할제품 코드 기준실패 모드
SceneObject3D 계층과 조명·환경을 담는 root제품 상태와 3D object mapping 규칙을 문서화React state와 scene mutation이 엇갈려 선택·undo가 깨짐
Camera사용자의 시점과 projectiondefault view, reset view, bounds fitting, clipping plane 정의near/far 값이 과도해 z-fighting 또는 depth precision 저하
Rendererscene을 canvas에 렌더링pixelRatio cap, antialias, toneMapping, shadow 옵션을 기기별 조정DPR 3, shadow 고해상도, post effect가 모바일에서 급락
Geometrymesh의 vertex 데이터공유 geometry, LOD, instancing 검토동일 mesh를 매번 clone해 메모리와 draw call 증가
Materialshader와 렌더링 상태 묶음material 수 cap, texture atlas, variant 전략색상 하나 때문에 material을 무한 생성
LoaderglTF, texture, HDRI 등 asset 로딩progress, cancellation, cache, dispose 경로 설계route 이동 뒤 loader promise가 완료되어 죽은 scene에 추가
Controls/Raycastorbit, drag, picking 상호작용pointer capture, keyboard 대체, hit target 우선순위 정의DOM overlay와 3D raycast가 서로 pointer event를 빼앗음
Frame loop 전략적합한 제품장점주의
Always render게임, particle, 계속 움직이는 scene상태 단순, 애니메이션 자연스러움정적 제품 viewer에서도 배터리와 CPU/GPU 사용 증가
Render on demand제품 configurator, 3D preview, static model viewer상호작용 때만 렌더링해 전력·열 감소texture load, resize, control damping, animation 종료 시 invalidate 필요
Hybrid정적 대부분 + 짧은 transition/hover animation체감 품질과 비용 균형render trigger가 누락되면 상태 변경이 화면에 반영되지 않음
THREE RESOURCE LIFECYCLEroute enter: renderer, scene, camera, controls 생성
asset load: geometry, material, texture cache 등록
interaction: raycast와 state mutation을 한 방향으로 흐르게 유지
route leave: animation frame 중단, listener 제거, geometry/material/texture/render target dispose
restore: context restored 또는 재진입 시 asset cache와 scene 재구성 검증
DISPOSE

Three.js object를 scene에서 제거해도 GPU 자원이 자동으로 모두 해제되는 것은 아닙니다. geometry, material, texture, render target, controls listener를 해제하지 않으면 route를 여러 번 오간 뒤 메모리와 FPS가 무너집니다.

GFX-05

3D 제품 경험 설계

3D 제품 경험은 모델을 화면에 띄우는 일이 아니라 사용자가 제품을 이해하고, 비교하고, 설정하고, 구매·승인 같은 다음 행동을 결정하게 만드는 흐름입니다. 카메라, 조명, 입력, 상태 표시, fallback은 같은 제품 경험의 일부입니다.

제품 패턴핵심 사용 가치필수 인터랙션품질 기준

Configurator색상·옵션 변경 결과 확인variant switch, price/spec sync, undo, shareable URL옵션 변경 후 시각 반영 200ms 내, 잘못된 조합 차단

UX 결정좋은 기본값나쁜 신호
카메라제품 전체가 보이는 hero angle, reset view, bounds clamp첫 화면에서 제품이 잘리거나 zoom 방향이 예측 불가
조명재질을 읽을 수 있는 균형 조명, 과한 bloom 제한분위기 때문에 실제 색상·표면 판단이 어려움
입력마우스, 터치, 키보드, trackpad 차이를 별도 테스트모바일에서 scroll과 orbit gesture가 충돌
상태 표시로딩 진행률, 저해상도 preview, 실패 메시지, retry검은 canvas 또는 무한 spinner만 노출
측정view start, model ready, first interaction, variant applied 이벤트3D가 전환율에 기여하는지 비용만 쓰는지 모름
3D PRODUCT REVIEW이 3D가 사용자의 어떤 결정을 더 빠르고 정확하게 만드는가
정적 이미지보다 나은 상호작용이 명확한가

PRODUCT RISK

3D는 강한 첫인상을 만들지만, 제품 과업을 느리게 만들기도 쉽습니다. 핵심 CTA가 모델 로딩 뒤로 밀리거나, 모바일에서 스크롤을 방해하거나, 색상 판단을 왜곡하면 3D는 경험 개선이 아니라 장애물이 됩니다.

GFX-06

정교한 3D 경험은 모델을 띄우는 수준을 넘어 좌표계, shader 비용, 색과 조명 파이프라인을 이해해야 안정됩니다. 클릭한 DOM 좌표가 어떤 3D object를 의미하는지, 왜 같은 색상이 기기마다 다르게 보이는지 설명할 수 있어야 합니다.

SENIOR GRAPHICS SIGNAL

정교한 3D 개발자는 “모델이 보인다”가 아니라 picking 좌표, shader 비용, color space, lighting, fallback을 함께 설명합니다. 특히 제품 configurator에서는 색과 재질이 구매 판단을 왜곡하지 않는 것이 성능만큼 중요합니다.

GFX-16

Canvas·WebGL 자동화

Canvas와 WebGL은 DOM snapshot만으로 품질을 검증하기 어렵습니다. 최소한 canvas가 비어 있지 않은지, fallback이 동작하는지, renderer.info가 반복 후 증가하지 않는지, reduced motion과 low-end tier가 적용되는지 자동화해야 합니다.

자동화목적방법차단 기준
Canvas nonblank검은/빈 canvas 회귀 탐지pixel sample 또는 screenshot diff의미 있는 픽셀 비율이 임계값 미만
Fallback routeWebGL 실패 시 제품 흐름 유지WebGL context 생성 실패 mockCTA가 사라지거나 무한 spinner
renderer.info regressiongeometry/texture 누수 탐지route 10회, variant 50회 뒤 counts 비교textures/geometries가 지속 증가
Asset budget모델/texture 크기 회귀 차단CI에서 asset manifest 검사예산 초과에 예외 근거 없음
Interaction smokeraycast, camera reset, option apply 검증Playwright pointer와 keyboard 병행선택/초기화/옵션 변경 실패
CANVAS NONBLANK 예시const pixels = await page.locator("canvas").evaluate((canvas) => {
  const ctx = canvas.getContext("2d") || canvas.getContext("webgl");
  // WebGL이면 테스트 전용 debug readback helper를 앱에 노출한다.
  return window.__readCanvasNonBlankRatio?.() ?? 0;
});

renderer.info regression 예시await openViewerAndWaitReady(page);

for (let i = 0; i < 10; i++) {

  await page.goto("/viewer");

TEST HOOK POLICY

그래픽 테스트를 위해 window.__threeInfo 같은 debug hook을 둘 수 있습니다. 단, production에서는 feature flag나 test build에서만 노출하고, 사용자 데이터나 원본 입력값을 노출하지 않아야 합니다.

GFX-17

근거 기반 그래픽 예산과 부수효과

Canvas와 WebGL 품질은 “GPU를 썼다”로 보장되지 않습니다. 픽셀 수, DPR, texture 해상도, draw call, context loss 복구, fallback 비용을 숫자로 계산해야 저사양 기기와 긴 세션에서 품질이 유지됩니다.

항목실질 숫자·계산근거와 의미부수효과대응
Canvas backing storewidth * height * DPR^2 * 4 bytes. 1200x800 DPR 2는 약 14.6MiB, 1920x1080 DPR 2는 약 31.6MiBCSS 크기가 같아도 DPR이 2면 픽셀 수와 메모리·fill-rate가 4배가 된다고해상도 canvas를 여러 개 유지하면 스크롤, tab 전환, mobile memory pressure에서 끊김이 커진다DPR cap, offscreen canvas 수 제한, resize debounce, hidden 상태 render 중지
Texture memoryRGBA 2048x2048은 16MiB, 4096x4096은 64MiB. mipmap을 포함하면 대략 33% 추가다운로드 용량이 작아도 GPU에 올라간 뒤에는 압축이 풀리거나 mip chain이 추가될 수 있다큰 texture 몇 장만으로도 mobile GPU budget을 압박하고 context loss 가능성이 높아진다KTX2/Basis 같은 GPU texture compression, texture atlas, LOD, max texture tier 적용
Frame budget60fps는 16.7ms/frame, mobile fallback 30fps는 33.3ms/framerender, physics, raycast, layout overlay, postprocessing이 모두 같은 frame 안에서 경쟁한다raycast나 postprocessing이 늘면 pointer hover가 늦어지고 camera 조작이 끊긴다dirty render, demand frameloop, BVH/spatial index, effect tier 분리
Draw call·material 수정답 숫자는 scene마다 다르지만, 증가 추세와 variant별 차이를 release gate로 본다draw call은 GPU뿐 아니라 CPU submit 비용도 만든다옵션을 바꿀 때 material을 계속 생성하면 긴 세션에서 memory slope와 stutter가 생긴다instancing, material reuse, geometry merge, dispose audit, renderer.info 회귀 테스트
WebGL context losswebglcontextlost 이벤트는 drawing buffer가 손실될 때 발생하며, WEBGL_lose_context로 시뮬레이션 가능브라우저·드라이버·메모리 압박 상황에서 실제로 canvas가 검게 비거나 렌더러가 멈출 수 있다복구 경로가 없으면 configurator, map, editor의 핵심 작업이 빈 화면으로 중단된다context lost/restore handler, asset reload, fallback UI, current selection 복구 테스트
Fallback 비용fallback 자체도 LCP, INP, accessibility tree에 포함된다3D 실패 시 단순 “지원하지 않음”보다 제품 행동을 계속할 수 있어야 한다fallback이 무거운 이미지 묶음이면 WebGL을 끈 사용자에게도 느린 경험을 준다poster image, option list, static preview, server-rendered summary, lazy asset load

근거 출처는 MDN의 webglcontextlost 문서와 WebGL 메모리 계산식을 기준으로 삼습니다. Canvas와 texture 예산은 압축 다운로드 크기가 아니라 실제 backing store와 GPU 업로드 후 비용을 기준으로 봅니다.

GRAPHICS BUDGET FORMULASCanvas memory MiB = width * height * DPR^2 * 4 / 1024 / 1024
RGBA texture MiB = textureWidth * textureHeight * 4 / 1024 / 1024
Mipmap 포함 texture MiB ~= base texture MiB * 1.33
Frame budget ms = 1000 / targetFPS
GPU 위험 신호 = texture count 증가 + context loss + long frame + fallback used
OPERATING STANDARD

그래픽 기능의 근거는 “Three.js로 구현했다”가 아니라 “DPR 2에서 backing store가 31.6MiB이고, 4096 texture 2장을 쓰며, route 10회 왕복 뒤 renderer.info가 증가하지 않고, context loss를 시뮬레이션해도 선택 상태가 복구된다” 같은 운영 문장이어야 합니다.

GFX-18

자주 봉착하는 문제와 해결

그래픽/3D 문제는 눈에 보이는 증상이 강해서 즉흥적으로 수정하기 쉽지만, 실제 원인은 GPU resource 수명, 좌표계 변환, asset pipeline, fallback 계약에 있는 경우가 많습니다. 해결은 shader나 camera 값을 찍어 맞추는 것이 아니라 재현 조건과 resource lifecycle을 고정하는 데서 시작합니다.

문제대표 증상주요 원인해결법검증
검은 canvas·빈 화면일부 기기에서 viewer가 비어 있거나 로딩이 끝나지 않음WebGL context 생성 실패, shader compile 실패, asset load 실패를 같은 spinner로 처리capability check, timeout, error boundary, static fallback, 실패 원인별 telemetry 분리WebGL off, asset 404, slow network, shader error mock
Context loss 복구 실패탭 전환이나 긴 세션 뒤 canvas가 멈춤context lost/restore listener가 없고 GPU resource 재업로드 경로가 없음webglcontextlost에서 기본 동작 제어, restore 시 geometry/texture/material 재생성, 현재 selection 복구WEBGL_lose_context로 loss/restore 자동화
GPU 메모리 누수route를 반복하면 점점 느려지고 mobile에서 crashgeometry, material, texture, render target dispose 누락scene cleanup checklist, resource registry, loader cache 정책, renderer.info 회귀 테스트viewer 10회 진입/이탈 후 textures/geometries 증가 여부 확인
Picking 좌표 오차클릭한 물체가 아닌 옆 object가 선택됨canvas rect, CSS transform, DPR, scroll offset, camera matrix를 혼동client 좌표에서 canvas rect를 빼고 NDC로 변환, overlay와 raycast 좌표계를 분리DPR 1/2, 스크롤된 컨테이너, resized canvas, mobile touch 테스트
Asset 용량·품질 균형 실패첫 진입이 느리거나 texture가 뭉개짐모델과 texture tier가 없고 모든 사용자에게 동일 asset 제공LOD, poster, progressive loading, KTX2/Basis compression, device tier별 texture cap 적용cold load, repeat load, low-end tier, visual QA capture 비교
Fallback이 제품 행동을 막음WebGL 실패 시 구매, 선택, 비교가 불가능함3D viewer를 상품 상태의 유일한 표현으로 설계DOM option list, static preview, text summary, CTA를 viewer 밖 독립 layer로 유지No-WebGL 환경에서 핵심 task completion 확인
Post-processing 과다bloom, blur, SSAO를 켜면 mobile 조작이 끊김full-screen pass와 render target이 frame budget을 초과effect tier, dynamic resolution, interaction 중 effect off, 중요 정보와 장식 효과 분리orbit/drag 중 frame time, effect on/off trace 비교
GRAPHICS DEBUG ORDER1. 재현 기기, DPR, GPU tier, WebGL version, reduced motion 여부를 기록한다
2. 실패가 context 생성, asset load, shader compile, render loop, interaction 중 어디인지 나눈다
3. renderer.info와 resource registry로 memory slope를 먼저 확인한다
4. 좌표 문제는 client, canvas, NDC, world, overlay를 단계별로 로그한다
5. fallback에서도 사용자가 같은 제품 결정을 완료할 수 있는지 확인한다
NO BLACK BOX

3D 라이브러리를 쓰더라도 resource 수명은 팀의 책임입니다. 모델이 보이는 데서 끝내면 운영에서 검은 화면, 메모리 누수, 선택 오차, fallback 실패가 뒤늦게 드러납니다. 정교한 그래픽 개발자는 렌더링 결과와 함께 실패 경로를 설계합니다.
