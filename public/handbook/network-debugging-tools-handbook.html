네트워크 디버깅

네트워크 장애는 계층별로 좁혀야 합니다. 바로 애플리케이션 로그를 보기보다 IP 도달성, 포트 연결성, HTTP/TLS 응답, DNS 해석을 분리해서 봅니다.

DBG-01

L3·L4·L7 순서

계층확인도구
L3 NetworkIP 경로와 도달성ping, traceroute, route table
L4 TransportTCP/UDP 포트 연결nc, telnet, tcpdump
L7 ApplicationHTTP, TLS, 앱 응답curl, browser, app logs
Name도메인 → IP 해석dig, nslookup
DBG-02

주소·경로 확인

L3 commands 
ip addr
ip route
ping 172.16.1.10
traceroute 172.16.1.10

ping 실패가 항상 네트워크 실패는 아닙니다. ICMP를 막는 조직도 많으므로 서비스 포트 확인과 함께 봐야 합니다.

DBG-03

포트·서비스 확인

L4/L7 commands 
nc -zv 172.16.1.10 5432
curl -v --connect-timeout 5 http://172.16.1.10:8080/health
curl -vk https://api.example.com/health
결과해석
Connection timed out라우팅 또는 방화벽에서 응답 없음
Connection refused서버 도달, 해당 포트 리스닝 없음 또는 reject
HTTP 5xx네트워크보다 앱/의존성 문제 가능성
TLS handshake 실패인증서, SNI, 프로토콜 불일치 가능성
DBG-04

DNS 확인

DNS commands 
dig db.customer.internal
dig @172.16.0.2 db.customer.internal
nslookup api.example.com

DNS 결과 IP가 예상한 사설 대역인지, public IP인지, resolver가 어디인지 먼저 확인합니다.

DBG-05

패킷 캡처

tcpdump commands 
sudo tcpdump -i eth0 host 172.16.1.10 -n
sudo tcpdump -i eth0 port 5432 -n
sudo tcpdump -i eth0 -w capture.pcap
Case study

nc가 timeout일 때 tcpdump로 SYN이 나가는지 확인하면 원인을 좁힐 수 있습니다. SYN이 나가지 않으면 우리 서버/라우팅 문제, SYN은 나가는데 SYN-ACK가 안 오면 중간 경로 또는 상대 방화벽 문제, SYN-ACK가 오는데 앱이 실패하면 L7 문제로 이동합니다.
