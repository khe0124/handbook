Java 백엔드 면접은 어노테이션 암기가 아니라 런타임, 프록시, 트랜잭션, 영속성 컨텍스트가 어떤 실패를 만드는지를 설명하는 자리입니다.

JSJ-01

Q. JVM, heap/stack, GC를 설명해보세요.

핵심 개념

JVM은 Java bytecode를 실행하는 런타임입니다. heap은 객체가 저장되는 공유 메모리이고, stack은 thread마다 method call frame과 지역 변수를 저장합니다. GC는 heap에서 더 이상 참조되지 않는 객체를 회수합니다.

개념 해부

JVM 메모리를 heap과 stack으로 나누는 이유는 객체 생명주기와 호출 생명주기가 다르기 때문입니다. stack frame은 method 호출이 끝나면 사라지지만, heap 객체는 더 이상 도달 가능한 참조가 없을 때 GC 대상이 됩니다. GC 문제는 단순히 메모리 부족이 아니라 allocation rate, 오래 살아남는 객체, stop-the-world pause가 응답 지연으로 드러나는 문제입니다.

30초 답변

Java 객체는 주로 heap에 저장되고 각 thread는 자기 stack을 가집니다. GC는 reachable object를 기준으로 사용하지 않는 heap 객체를 회수합니다. 실무에서는 stop-the-world, allocation rate, memory leak, GC log를 성능 문제와 연결해 봐야 합니다.

90초 답변

GC는 메모리를 자동으로 회수하지만 비용이 0은 아닙니다. 객체 생성이 과도하거나 큰 컬렉션을 오래 참조하면 GC 압박이 커집니다. memory leak은 객체가 진짜 필요 없는데도 collection, static field, ThreadLocal, cache 등이 계속 참조해 GC가 회수하지 못하는 상태입니다.

꼬리질문

질문답변 방향
StackOverflow와 OOM?StackOverflow는 call stack 과다, OOM은 heap/metaspace/native 등 메모리 부족입니다.
GC 튜닝부터?먼저 allocation과 leak, 객체 생명주기, heap dump를 봅니다.
ThreadLocal 위험?pool thread에서 값 제거 누락 시 오래 참조될 수 있습니다.
JSJ-02

Q. equals/hashCode는 왜 같이 구현해야 하나요?

핵심 개념

`equals`는 논리적 동등성, `hashCode`는 hash 기반 collection에서 bucket을 찾는 값입니다. equals가 true인 객체는 같은 hashCode를 가져야 합니다.

개념 해부

hash 기반 collection은 먼저 hash로 후보 위치를 좁히고, 그 안에서 equals로 실제 동등성을 확인합니다. 따라서 equals가 true인 두 객체의 hashCode가 다르면 같은 bucket 후보에 들어가지 못해 조회가 실패할 수 있습니다. entity에서는 persist 전 id, proxy class, mutable field 때문에 동등성 기준을 더 보수적으로 잡아야 합니다.

30초 답변

HashMap, HashSet은 hashCode로 후보 bucket을 찾고 equals로 최종 비교합니다. equals만 바꾸고 hashCode를 바꾸지 않으면 같은 값으로 여기는 객체가 set에서 중복되거나 map 조회가 실패할 수 있습니다.

90초 답변

JPA entity에서는 더 조심해야 합니다. DB id가 persist 전에는 null일 수 있고, proxy class와 equals 비교 문제가 생길 수 있습니다. mutable field를 hashCode에 넣으면 collection 안에서 값이 바뀐 뒤 찾지 못할 수 있습니다.

꼬리질문

질문답변 방향
hash collision?다른 객체가 같은 hashCode를 가질 수 있고 equals로 구분합니다.
record는?component 기반 equals/hashCode를 자동 생성합니다.
Entity equals?식별자, 생명주기, proxy를 고려해야 합니다.
JSJ-03

Q. Spring DI와 Bean lifecycle을 설명해보세요.

핵심 개념

DI는 객체가 필요한 의존성을 직접 만들지 않고 외부 컨테이너가 주입하는 방식입니다. Spring Bean은 생성, 의존성 주입, 초기화 callback, 사용, 소멸 lifecycle을 가집니다.

개념 해부

DI의 본질은 객체 생성 책임과 사용 책임을 분리해 변경 가능한 경계를 만드는 것입니다. Spring container는 bean definition을 보고 객체를 만들고 의존성을 연결한 뒤 초기화 callback과 proxy 후처리를 적용합니다. 생성자 주입은 필수 의존성을 객체 생성 시점에 드러내므로 null 상태와 테스트 어려움을 줄입니다.

30초 답변

Spring은 Bean을 생성하고 의존성을 주입해 객체 간 결합을 낮춥니다. 생성자 주입은 필수 의존성을 명확히 하고 테스트하기 쉽습니다. Bean lifecycle에는 생성, 주입, 초기화, 소멸 단계가 있고 scope에 따라 생명주기가 달라집니다.

90초 답변

DI의 핵심은 테스트 가능성과 변경 가능성입니다. field injection은 편해 보이지만 불변성과 테스트 측면에서 약합니다. Bean lifecycle을 이해하면 초기화 시점에 외부 연결을 열거나 proxy가 적용되기 전/후 차이에서 생기는 문제를 설명할 수 있습니다.

꼬리질문

질문답변 방향
singleton scope?Spring container 안에서 기본적으로 하나의 Bean instance를 공유합니다.
순환 참조?설계 냄새이며 생성자 주입에서는 더 빨리 드러납니다.
BeanPostProcessor?Bean 초기화 전후에 개입해 proxy 생성 등에 쓰입니다.
JSJ-04

Q. AOP, Filter vs Interceptor는 어떻게 다르나요?

핵심 개념

AOP는 횡단 관심사를 proxy 기반으로 분리합니다. Filter는 Servlet container 레벨, Interceptor는 Spring MVC handler 전후, AOP는 method call 전후에 주로 개입합니다.

개념 해부

Filter, Interceptor, AOP는 모두 중간에 끼어드는 구조지만 위치가 다릅니다. Filter는 Servlet container 경계에서 HTTP 요청을 다루고, Interceptor는 Spring MVC handler 전후를 다루며, AOP는 Spring bean method 호출을 proxy로 감쌉니다. proxy 기반 AOP는 proxy를 통과한 호출에만 적용되므로 같은 객체 내부 호출은 advice가 빠질 수 있습니다.

30초 답변

Filter는 Spring MVC에 들어오기 전 HTTP 요청/응답을 처리하고, Interceptor는 controller handler 전후에 동작합니다. AOP는 service method 같은 Bean method 호출에 proxy로 개입해 logging, transaction, security 같은 횡단 관심사를 처리합니다.

90초 답변

인증 토큰 파싱처럼 모든 요청에 필요한 처리는 Filter가 자연스럽고, handler 정보가 필요한 권한/로깅은 Interceptor가 맞을 수 있습니다. transaction은 보통 AOP proxy로 처리됩니다. proxy 기반이므로 같은 객체 내부에서 자기 메서드를 호출하는 self-invocation은 AOP가 적용되지 않는 문제가 생깁니다.

꼬리질문

질문답변 방향
proxy 종류?interface 기반 JDK dynamic proxy와 class 기반 CGLIB proxy가 있습니다.
self-invocation?proxy를 거치지 않아 AOP advice가 적용되지 않습니다.
순서?Filter → DispatcherServlet → Interceptor → Controller 흐름으로 설명합니다.
JSJ-05

Q. @Transactional propagation, rollback, self-invocation 문제를 설명해보세요.

핵심 개념

`@Transactional`은 Spring AOP proxy를 통해 transaction 경계를 열고 commit/rollback을 관리합니다. propagation은 기존 transaction이 있을 때 새 method가 어떻게 참여할지 결정합니다.

개념 해부

transaction은 SQL 묶음이 아니라 일관된 변경 단위와 실패 시 되돌림 경계입니다. Spring의 @Transactional은 proxy가 method 호출 전후로 transaction을 열고 닫는 모델이라 호출 경로와 예외 타입이 결과에 영향을 줍니다. 외부 API 호출이나 긴 계산을 transaction 안에 두면 DB connection과 lock을 오래 잡아 장애 전파 지점이 됩니다.

30초 답변

@Transactional은 proxy를 거쳐 호출될 때 적용됩니다. 기본 propagation은 REQUIRED라 기존 transaction이 있으면 참여하고 없으면 새로 엽니다. 기본 rollback은 unchecked exception 중심이므로 checked exception rollback은 별도 설정이 필요합니다.

90초 답변

self-invocation은 같은 클래스 안에서 transactional method를 직접 호출해 proxy를 거치지 않는 문제입니다. 또 transaction 안에서 외부 API를 호출하면 connection과 lock을 오래 잡아 장애를 키울 수 있습니다. readOnly는 쓰기 방지 보안장치라기보다 flush/최적화 힌트로 봐야 합니다.

꼬리질문

질문답변 방향
REQUIRES_NEW?기존 transaction을 보류하고 새 transaction을 엽니다.
rollbackOnly?내부에서 rollback-only가 표시되면 외부 commit 시 예외가 날 수 있습니다.
긴 transaction?lock, connection pool 고갈, 장애 전파 위험이 있습니다.
JSJ-06

Q. JPA persistence context, dirty checking, flush를 설명해보세요.

핵심 개념

persistence context는 entity를 관리하는 1차 cache이자 변경 감지 단위입니다. dirty checking은 관리 중인 entity의 변경을 transaction commit/flush 시점에 SQL로 반영하는 기능입니다.

개념 해부

persistence context는 entity identity map, 1차 cache, 변경 감지 단위가 합쳐진 작업 공간입니다. 같은 transaction 안에서 같은 id를 다시 조회하면 동일 객체를 돌려주고, commit/flush 시점에 snapshot과 현재 값을 비교해 SQL을 만듭니다. flush는 DB에 SQL을 보내는 동기화이고 commit은 transaction 확정이므로 둘을 구분해야 합니다.

30초 답변

JPA는 transaction 안에서 조회한 entity를 persistence context에 올리고 동일 id 조회는 같은 객체를 반환합니다. entity field가 바뀌면 dirty checking으로 변경을 감지하고 flush 시 SQL을 DB에 보냅니다.

90초 답변

flush는 DB에 SQL을 보내는 것이고 commit은 transaction을 확정하는 것입니다. flush가 됐다고 commit된 것은 아닙니다. persistence context가 너무 커지면 memory와 dirty checking 비용이 커질 수 있어 batch 처리에서는 flush/clear 전략이 필요합니다.

꼬리질문

질문답변 방향
detached entity?persistence context가 더 이상 관리하지 않는 entity입니다.
merge 주의?detached state를 복사해 managed entity를 반환하므로 반환값 사용에 주의합니다.
N+1과 관계?연관 loading 전략과 persistence context 동작을 이해해야 합니다.
JSJ-07

Q. lazy loading, Entity vs DTO, OSIV를 설명해보세요.

핵심 개념

lazy loading은 연관 객체 접근 시점에 query를 실행합니다. DTO는 API 응답이나 화면 요구에 맞춘 데이터 구조이고 Entity는 도메인/영속성 모델입니다. OSIV는 view rendering까지 persistence context를 열어두는 설정입니다.

개념 해부

lazy loading은 연관 데이터를 늦게 가져오는 최적화이지만 접근 시점에 SQL이 나간다는 사실을 숨깁니다. OSIV를 켜면 view/controller에서도 lazy loading이 가능해 편하지만 query가 service 경계 밖에서 발생하고 connection 보유 시간이 길어질 수 있습니다. Entity는 저장 모델이고 DTO는 API 계약이므로 응답 요구에 맞는 projection을 분리하는 편이 안전합니다.

30초 답변

lazy loading은 필요할 때 연관 데이터를 가져와 초기 조회를 줄이지만, transaction 밖에서 접근하면 LazyInitializationException이 나거나 N+1이 생길 수 있습니다. API 응답은 Entity를 그대로 노출하기보다 DTO로 필요한 필드만 명시하는 편이 안전합니다.

90초 답변

OSIV를 켜면 controller/view에서 lazy loading이 가능해 편하지만, DB connection이 오래 유지되고 query가 service boundary 밖에서 발생해 성능 추적이 어려워질 수 있습니다. 실무에서는 조회 use case에 맞게 fetch join, entity graph, DTO projection을 선택합니다.

꼬리질문

질문답변 방향
cascade/orphanRemoval?부모 entity 변경이 자식 생명주기에 어떤 영향을 주는지 관리합니다.
Entity 노출 위험?순환 참조, 과다 필드 노출, lazy loading, API 계약 불안정 문제가 있습니다.
fetch join 단점?컬렉션 pagination과 중복 row 문제가 생길 수 있습니다.
JSJ-08

Q. Spring Security 인증/인가 흐름은 어떻게 설명하나요?

핵심 개념

Spring Security는 filter chain에서 요청을 가로채 인증 정보를 만들고 SecurityContext에 저장한 뒤, 인가 단계에서 resource 접근 가능 여부를 판단합니다.

개념 해부

Spring Security는 controller 앞에서 끝나는 기능이 아니라 요청을 인증 객체로 바꾸고 SecurityContext에 보관한 뒤 여러 지점에서 인가 판단을 하게 하는 filter chain입니다. session 방식은 서버가 상태를 보관하고, JWT 방식은 매 요청 token 검증으로 상태를 재구성합니다. 401은 신원 확인 실패, 403은 신원은 알지만 권한이 부족한 상황입니다.

30초 답변

요청은 Security Filter Chain을 지나며 인증 토큰 파싱, Authentication 생성, SecurityContext 저장을 거칩니다. 이후 controller나 method security에서 현재 principal과 authority를 기준으로 인가를 판단합니다.

90초 답변

JWT 기반이면 stateless하게 매 요청 token을 검증하고 SecurityContext를 구성합니다. Session 기반이면 로그인 후 session에 인증 상태를 저장합니다. 401은 인증 실패, 403은 인증됐지만 권한 부족입니다. 중요한 것은 프론트 route guard가 아니라 서버에서 최종 인가를 한다는 점입니다.

꼬리질문

질문답변 방향
filter order?인증 filter 위치와 예외 처리 filter의 순서가 중요합니다.
method security?service method 단에서 권한을 검증해 controller 우회를 막습니다.
CSRF?cookie/session 기반 상태 변경 요청에서 방어가 필요합니다.
JSJ-09

Spring 프록시 실패 매트릭스

Spring 면접에서 어노테이션 이름만 말하면 얕습니다. 프록시가 어디에 생기고 어떤 호출은 프록시를 통과하지 않는지 설명해야 합니다.

SPRING PROXY FAILURE MATRIX Spring AOP 기반 기능은 대상 객체 자체가 아니라 proxy를 통해 호출될 때 동작합니다. 그래서 self-invocation, private/final method, Bean이 아닌 객체, 너무 이른 초기화 시점, 잘못된 proxy 타입은 transaction, cache, security, async advice가 적용되지 않는 원인이 됩니다.
실패 상황왜 실패하는가면접 답변 방향
같은 클래스 내부에서 transactional method 호출this 호출은 proxy를 거치지 않습니다.service 분리, 자기 proxy 주입보다 설계 경계 재검토를 우선 말합니다.
private/final method에 advice 기대proxy가 method override 또는 interface 호출을 감싸기 어렵습니다.AOP 적용 가능한 visibility와 proxy 방식을 구분합니다.
@Async가 동작하지 않음proxy 호출이 아니거나 executor 설정/return type이 맞지 않습니다.thread boundary, exception handling, transaction context 분리를 함께 말합니다.
@Cacheable이 갱신되지 않음key 설계, eviction 누락, self-invocation 문제가 있을 수 있습니다.cache key, TTL, eviction event, stale data 위험을 확인합니다.
security annotation이 기대와 다름method security 활성화와 proxy 적용 위치가 다릅니다.URL authorization과 method authorization의 책임을 분리합니다.

면접에서 말할 증거

proxy class 확인, AOP 적용 로그, transaction debug log, 호출 경로, Bean 등록 여부, test slice에서 proxy가 살아있는지 확인한 경험을 말합니다.

오답 방지

“@Transactional을 붙이면 됩니다”는 위험합니다. Spring은 선언을 읽고 proxy로 동작을 감싸는 모델이므로 호출 경로와 예외 처리까지 설명해야 합니다.

JSJ-10

JPA 조회 실패 매트릭스

JPA 면접은 ORM 장점보다 숨겨진 SQL, persistence context, transaction 경계가 만드는 실패를 이해하는지 봅니다.

JPA QUERY FAILURE MATRIX JPA 문제는 N+1, LazyInitializationException, OSIV로 인한 connection 장기 점유, 컬렉션 fetch join pagination, dirty checking 비용, batch flush/clear 누락처럼 나타납니다. 답변은 “fetch join을 씁니다”가 아니라 조회 use case와 데이터 규모에 맞는 전략을 비교해야 합니다.
문제발생 조건대응주의점
N+1목록 조회 후 lazy association을 반복 접근fetch join, entity graph, batch size, DTO projection모든 연관을 fetch join하면 row 폭증과 중복이 생깁니다.
LazyInitializationExceptiontransaction/persistence context 밖에서 lazy 접근service 경계 안에서 필요한 데이터 로딩, DTO 변환OSIV로 숨기면 query 위치가 흐려질 수 있습니다.
pagination 깨짐컬렉션 fetch join과 limit/offset 조합id page 조회 후 2차 조회, batch loading, projection메모리 pagination 경고를 놓치면 운영에서 터집니다.
dirty checking 비용 증가큰 persistence context, 대량 updatebulk query, batch 단위 flush/clear, readOnly transactionbulk update는 persistence context와 불일치할 수 있습니다.
lock 대기긴 transaction, 외부 API 호출, 불필요한 write locktransaction 범위 축소, timeout, optimistic lock, retry 정책retry는 idempotency와 사용자 안내가 함께 필요합니다.

면접에서 말할 증거

SQL 로그, query count, execution plan, Hibernate warning, transaction boundary, DTO projection 기준, batch 처리 flush/clear 단위를 말합니다.

흔한 오답

JPA를 쓰면 SQL을 몰라도 된다고 말하면 바로 신뢰가 떨어집니다. JPA는 SQL을 없애는 도구가 아니라 객체 변경을 SQL 실행으로 번역하는 도구입니다.

JSJ-13

Spring·JPA 개념 인과관계

Spring/JPA 답변이 얕아지는 이유는 어노테이션을 기능 스위치처럼 말하기 때문입니다. 실제로는 proxy, transaction boundary, persistence context, SQL 실행 시점이 연결되어 하나의 요청 경로를 만듭니다.

SPRING JPA CAUSAL MODEL@Transactional은 method에 마법을 거는 것이 아니라 proxy가 호출 전후에 transaction을 여닫는 구조입니다. 그래서 self-invocation은 proxy를 우회하고, checked exception은 기본 rollback 대상이 아니며, transaction 안에서 외부 API를 오래 기다리면 DB connection과 lock을 오래 점유합니다. JPA도 SQL을 없애는 것이 아니라 persistence context 안의 entity 변경을 flush 시점에 SQL로 번역합니다. 이 때문에 lazy loading, N+1, dirty checking 비용, bulk update 후 stale entity 문제가 발생합니다.
원인내부 동작실패 현상전문적인 답변 방향
같은 클래스 내부 transactional method 호출this 호출은 Spring proxy를 통과하지 않는다transaction, cache, async advice가 기대와 다르게 빠진다service 경계를 분리하고 proxy 적용 지점을 호출 경로로 설명한다
긴 transaction 안 외부 API 호출connection과 lock이 외부 응답 시간만큼 묶인다pool wait, lock wait, timeout, deadlock 가능성이 커진다transaction 범위 축소, outbox, afterCommit event, timeout을 비교한다
목록 조회 후 lazy association 반복 접근각 entity 접근 시 추가 SQL이 발생한다N+1, p95 latency 상승, DB 부하 증가fetch join, entity graph, batch size, DTO projection을 데이터 규모로 선택한다
bulk update 후 persistence context 유지bulk query가 1차 cache의 managed entity를 우회한다같은 transaction 안에서 이전 값이 보인다flush/clear, transaction 분리, clearAutomatically를 설명한다
면접 답변 전환

“N+1은 fetch join으로 해결합니다”에서 멈추지 말고 “목록 조회 후 연관 컬렉션에 접근할 때 entity 수만큼 query가 추가됩니다. fetch join은 round trip을 줄이지만 컬렉션 pagination과 row 중복 문제가 있어, 조회 목적에 따라 DTO projection이나 batch loading을 선택합니다”처럼 원인과 부작용을 함께 말합니다.

JSJ-14

실전 장애 디버깅: Spring 요청 하나를 끝까지 추적하기

전문가 수준의 Java/Spring/JPA 답변은 어노테이션 동작을 설명하는 데서 끝나지 않습니다. 느린 요청 하나를 HTTP thread, transaction proxy, Hibernate SQL, Hikari pool, DB lock, GC까지 같은 타임라인으로 추적해야 합니다.

EXPERT SPRING DEBUGGING CASE시나리오는 “주문 목록 API의 p99가 400ms에서 6초로 상승했고, CPU는 낮고 DB CPU도 아주 높지 않은데 사용자 timeout이 늘었다”입니다. 초보 답변은 “쿼리가 느린 것 같습니다”에서 멈춥니다. 전문가 답변은 요청 trace를 먼저 나눕니다. controller 진입 전 대기인지, service 내부 transaction이 긴지, SQL 실행이 느린지, connection pool에서 기다리는지, downstream API나 serialization이 느린지 확인합니다.

1. 요청 타임라인을 먼저 나눈다

APM trace나 structured log에서 request id를 기준으로 시간을 나눕니다. servlet thread가 요청을 받았는지, filter/security chain에서 지연이 있는지, controller 진입 후 service 호출까지 얼마인지, transaction 시작과 commit 사이가 얼마인지, SQL 개수와 각 SQL 시간이 얼마인지 봅니다. SQL 하나가 5초인 경우와 SQL 100개가 각각 50ms인 경우는 해결책이 다릅니다. 전자는 실행 계획과 lock을 보고, 후자는 N+1이나 반복 호출을 의심합니다.

2. HikariCP metric을 읽는다

Hikari의 active, idle, pending, max, connection timeout을 함께 봅니다. pending thread가 늘고 active가 max에 붙어 있으면 애플리케이션 thread가 DB connection을 기다리는 상태입니다. 이때 connection pool size를 무작정 늘리면 DB가 더 많은 동시 쿼리를 처리해야 해서 상황이 악화될 수 있습니다. 먼저 transaction duration, OSIV, slow query, 외부 API를 transaction 안에서 호출하는지 확인합니다. pool wait가 길고 SQL 시간은 짧다면 connection을 빌린 뒤 반납하지 않거나 transaction 범위가 과도하게 긴 구조일 수 있습니다.

3. Transaction proxy와 rollback 경계를 확인한다

@Transactional이 붙어 있어도 proxy를 통과하지 않는 self-invocation이면 transaction이 열리지 않을 수 있습니다. 반대로 transaction이 너무 넓으면 lock과 connection을 오래 잡습니다. checked exception은 기본 rollback 대상이 아니므로 비즈니스 실패가 commit되는 경우도 있습니다. 디버깅에서는 transaction active log, propagation, rollbackOnly, exception type을 봅니다. “transaction을 걸면 안전하다”가 아니라 “어떤 변경을 하나의 실패 단위로 묶고, 어떤 외부 부작용은 transaction 밖으로 뺄지”가 핵심입니다.

4. Hibernate SQL과 persistence context를 확인한다

Hibernate SQL log에서 같은 패턴의 select가 반복되면 N+1을 의심합니다. 다만 fetch join이 항상 정답은 아닙니다. 컬렉션 fetch join은 row 중복과 pagination 문제를 만들 수 있고, 여러 bag을 동시에 fetch하면 MultipleBagFetchException이 발생할 수 있습니다. 목록 API에서는 DTO projection, batch size, entity graph, 2단계 id 조회를 비교합니다. 대량 update에서는 persistence context가 커져 dirty checking 비용이 커지므로 batch 단위 flush/clear가 필요합니다. JPQL bulk update는 persistence context를 우회하므로 stale entity가 남을 수 있습니다.

5. Thread dump와 GC를 함께 본다

CPU가 낮은데 지연이 높으면 thread가 RUNNABLE이 아니라 WAITING/BLOCKED에 있을 수 있습니다. thread dump에서 Hikari connection borrow, synchronized lock, remote call, file I/O, blocking queue 대기를 확인합니다. GC pause가 길면 모든 thread가 멈춘 것처럼 보일 수 있으므로 GC log와 allocation rate도 확인합니다. 단, GC가 보인다고 바로 heap을 늘리는 것은 위험합니다. cache, ThreadLocal, 큰 DTO list, response buffering처럼 객체 생명주기와 retained path를 먼저 봅니다.

면접에서의 90초 답변

“저라면 먼저 trace id 기준으로 요청 시간을 filter, controller, service transaction, SQL, serialization, downstream으로 나눕니다. Hikari pending이 증가하면 connection pool wait와 transaction duration을 보고, SQL 개수와 실행 시간을 분리해 N+1인지 slow query인지 판단합니다. transaction 안에서 외부 API를 기다리거나 OSIV 때문에 connection이 오래 잡히면 경계를 줄입니다. Hibernate 쪽은 fetch join, DTO projection, batch size를 데이터 규모와 pagination 요구에 따라 고르고, 변경 후에는 p95/p99, query count, pool wait, DB plan을 비교합니다.”

전문가 기준

“로그를 봅니다”는 답변이 아닙니다. 어떤 로그, 어떤 metric, 어떤 dump, 어떤 plan을 보고 어떤 가설을 제거할지 말해야 합니다. Spring/JPA 면접의 깊이는 proxy와 ORM 내부 동작을 운영 지표로 연결할 때 드러납니다.
