LLM-00

AI NATIVE ROLE BOUNDARY

AI Native 개발자의 기준은 “프롬프트를 잘 쓴다”가 아닙니다. LLM을 권한, 데이터, 평가, 비용, 장애 대응이 필요한 소프트웨어 컴포넌트로 다루는 능력입니다.

FAIL

LLM FAILURE MODE MATRIX

LLM 시스템의 실패는 500 에러로만 드러나지 않습니다. 틀린 근거, 권한 누락, 비용 폭주, 스키마 깨짐처럼 정상 응답처럼 보이는 실패가 많습니다.

API

Messages API, Tool Use, Eval Control Surface

실무 LLM 앱의 중심 코드는 프롬프트 한 줄이 아니라 입력 계약, 도구 schema, tool_use 처리, tool_result 재전송, structured output schema 검증, eval gate 기록을 묶는 제어면입니다.

MARKER: artifact-ready — Messages tool loop skeleton

const response = await client.messages.create({
  model: process.env.CLAUDE_MODEL,
  max_tokens: 1200,
  system: [
    "Answer only from approved policy passages.",
    "Return JSON that matches the structured output schema.",
    "If evidence is missing, return refusal_reason."
  ].join(" "),
  messages: [{ role: "user", content: userQuestion }],
  tools: [{
    name: "lookup_policy",
    description: "Find approved policy passages by query.",
    input_schema: {
      type: "object",
      properties: {
        query: { type: "string" },
        top_k: { type: "integer", minimum: 1, maximum: 8 }
      },
      required: ["query"]
    }
  }]
});

const toolResults = [];
for (const block of response.content) {
  if (block.type === "tool_use" && block.name === "lookup_policy") {
    const passages = await lookupPolicy(block.input.query, block.input.top_k);
    toolResults.push({
      type: "tool_result",
      tool_use_id: block.id,
      content: JSON.stringify(passages)
    });
  }
}

const finalResponse = await client.messages.create({
  model: process.env.CLAUDE_MODEL,
  max_tokens: 1200,
  messages: [
    { role: "user", content: userQuestion },
    { role: "assistant", content: response.content },
    { role: "user", content: toolResults }
  ]
});

validateAgainstSchema(finalResponse, answerSchema);
recordEvalGate({
  prompt_version: "policy-answer@2026-07-02",
  dataset: "policy-golden-v3",
  schema_valid: true,
  groundedness_min: 0.9,
  release_gate: "pass"
});
{
  "name": "structured output schema",
  "type": "object",
  "required": ["answer", "citations", "confidence", "refusal_reason"],
  "properties": {
    "answer": { "type": "string" },
    "citations": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["document_id", "chunk_id"],
        "properties": {
          "document_id": { "type": "string" },
          "chunk_id": { "type": "string" }
        }
      }
    },
    "confidence": { "type": "number", "minimum": 0, "maximum": 1 },
    "refusal_reason": { "type": ["string", "null"] }
  }
}

DEPTH

AI Native 실무 전환 기준

MARKER: concept-depth — 모델 호출자에서 불확실성 관리자로

AI Native 개발자로 넘어간다는 것은 모델 API를 호출할 줄 안다는 뜻이 아닙니다. 같은 입력이어도 모델·프롬프트·검색 결과·도구 응답·샘플링 조건에 따라 출력이 달라질 수 있다는 사실을 제품 설계의 출발점으로 삼는다는 뜻입니다. 일반 API는 보통 명시된 입력과 코드 경로가 결과를 결정하지만, LLM 기능은 확률적 출력, 외부 문서, 권한 정책, 평가 기준이 함께 결과를 만듭니다. 그래서 이 로드맵의 핵심은 모델을 믿는 방법이 아니라, 모델이 흔들려도 제품이 무너지지 않도록 계약과 검증을 배치하는 방법입니다.

LLM 기능은 기존 백엔드 기능보다 실패가 조용합니다. 서버는 200을 반환했지만 citation이 틀릴 수 있고, JSON은 파싱됐지만 의미가 schema 의도와 다를 수 있으며, 검색 결과는 있었지만 현재 사용자가 볼 수 없는 문서였을 수 있습니다. 이런 실패는 로그에 에러로 남지 않기 때문에, 개발자는 처음부터 request_id, prompt_version, retrieval trace, schema validation, judge score, cost_per_successful_answer 같은 증거를 남겨야 합니다. 이 증거가 없으면 개선은 디버깅이 아니라 감각적 프롬프트 튜닝이 됩니다.

MARKER: failure-driven — 실패 모드로 학습 순서를 잡는 이유

학습 순서는 기술 유행이 아니라 실패 모드의 의존성으로 정해야 합니다. hallucination을 줄이려면 먼저 모델이 정답 DB가 아니라는 기초를 이해해야 하고, 그 다음 structured output으로 출력 계약을 고정하고, RAG로 근거를 공급하고, evaluation으로 실제 개선 여부를 확인해야 합니다. prompt injection은 보안 문서에서만 다룰 수 없습니다. RAG 문서가 비신뢰 입력을 prompt에 넣는 순간부터 이미 보안 문제가 시작되므로, 로드맵은 기초·프롬프팅·RAG·평가·보안을 분리된 과목이 아니라 하나의 요청 흐름으로 연결해야 합니다.

중급 개발자에게 필요한 신호는 “무엇을 배웠다”가 아니라 “어떤 설계 결정을 방어할 수 있다”입니다. 예를 들어 RAG를 배웠다는 증거는 벡터DB를 붙였다는 말이 아니라, 왜 chunk size를 이렇게 잡았는지, 권한 pre-filter를 어디서 적용했는지, recall@k가 낮을 때 parser와 query rewrite 중 무엇을 먼저 의심하는지 설명하는 것입니다. 평가를 배웠다는 증거도 LLM-as-judge를 호출했다는 말이 아니라, judge가 틀릴 수 있음을 알고 human-labeled holdout과 regression gate를 운영하는 것입니다.

MARKER: failure-driven — 로드맵을 기능 목록으로만 따라갈 때 생기는 실패

가장 흔한 실패는 structured output, RAG, agent, evaluation을 각각 독립 기능처럼 구현하고 끝내는 것입니다. 이 경우 포트폴리오에는 기능 이름이 많아지지만, 실제 리뷰에서 “왜 이 순서로 배웠는가”, “어떤 실패를 막기 위해 다음 단계로 넘어갔는가”, “품질이 좋아졌다는 증거는 무엇인가”에 답하지 못합니다. 로드맵은 체크박스가 아니라 위험을 줄이는 순서여야 합니다. Phase 1에서 schema validation과 repair loop를 만들지 않으면 Phase 2 RAG의 출력도 검증되지 않고, Phase 3 평가가 없으면 RAG 변경이 개선인지 퇴행인지 구분할 수 없습니다. 따라서 각 단계의 완료 기준은 기능 완성이 아니라 다음 단계의 실패를 줄이는 증거로 정의해야 합니다.
