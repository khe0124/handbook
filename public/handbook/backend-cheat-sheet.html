BE-CS-01

핵심 명령·스니펫

요청 찌르기 (curl / HTTPie)

# 헤더까지 보기 — status, 응답 헤더 확인 (가장 많이 씀)
curl -i https://api.example.com/orders/42

# 상태코드만 뽑기
curl -s -o /dev/null -w '%{http_code}\n' https://api.example.com/health

# POST + JSON + 멱등키 (재시도 안전)
curl -X POST \
  -H 'Content-Type: application/json' \
  -H 'Idempotency-Key: 8f14e45f-ceea-467a-9c3d-1b2c3d4e5f60' \
  -d '{"amount":1000,"currency":"KRW"}' \
  https://api.example.com/payments

# Bearer 토큰 인증
curl -H 'Authorization: Bearer <token>' https://api.example.com/me

# 타임/트레이스 프로파일 (DNS→connect→TTFB→total)
curl -s -o /dev/null \
  -w 'dns:%{time_namelookup} conn:%{time_connect} ttfb:%{time_starttransfer} total:%{time_total}\n' \
  https://api.example.com/orders
# HTTPie — 키=값이면 자동 JSON, 사람이 읽기 좋게 출력
http POST api.example.com/payments amount=1000 currency=KRW \
  Idempotency-Key:8f14e45f-ceea-467a Authorization:'Bearer <token>'

# 헤더만 보기
http --headers GET api.example.com/orders/42

응답 파싱 (jq)

# 특정 필드
curl -s api.example.com/orders/42 | jq '.status'

# 배열에서 골라 뽑기
curl -s api.example.com/orders | jq '.data[] | {id, status}'

# 조건 필터 (실패 건만)
curl -s api.example.com/orders | jq '.data[] | select(.status=="FAILED")'

# 개수 세기
curl -s api.example.com/orders | jq '.data | length'

부하 테스트

# hey — 10초 동안 동시 50 (p50/p95/p99 히스토그램)
hey -z 10s -c 50 https://api.example.com/orders

# hey — 총 1000요청, POST + body
hey -n 1000 -c 50 -m POST -T 'application/json' \
  -d '{"amount":1000}' https://api.example.com/payments

# ApacheBench — 총 1000요청, 동시 50
ab -n 1000 -c 50 https://api.example.com/orders

JWT 디코드 (payload 까보기)

# 2번째 파트(payload)만 base64 디코드 → exp, sub, role 확인
echo <jwt> | cut -d. -f2 | base64 -d | jq

# padding 에러 나면 base64 대신 (macOS)
echo <jwt> | cut -d. -f2 | base64 -D 2>/dev/null | jq

로그 추적

# requestId로 한 요청의 전 구간 따라가기
grep 'req-8f14e45f' app.log

# 5xx만 실시간
tail -f app.log | grep -E '"status":5[0-9]{2}'

# errorCode별 집계 top
grep -oE '"errorCode":"[^"]+"' app.log | sort | uniq -c | sort -rn | head
명령용도주의
curl -i <url>status·헤더 확인, 첫 진단-s로 progress 끄기
curl -sS -f <url>스크립트용: 4xx/5xx면 exit≠0-f는 body 안 보여줌
curl -L <url>redirect 따라가기POST가 GET으로 바뀔 수 있음
jq -r '.token'따옴표 없는 raw 값 추출파이프 대입 시 필수
hey -z 10s -c 50p95/p99 지연 측정prod 대상 금지
ab -n 1000 -c 50간단 처리량 측정keep-alive는 -k
주의

부하 명령(hey, ab)은 staging/local에만. prod에 걸면 self-DoS. JWT 디코드는 서명 검증이 아니라 payload 열람일 뿐이니 신뢰 판단에 쓰지 말 것.

BE-CS-02

판단 한 줄

선택언제 쓰나주의
Session즉시 폐기·서버 제어가 중요할 때scale-out 시 공용 session store 필요
JWTstateless·분산 검증이 중요할 때즉시 폐기 어려움 → 짧은 만료 + refresh
RBAC역할이 명확하고 조직이 단순할 때예외 늘면 role explosion
ABAC소유자·조직·상태 등 속성 기반 판단 필요정책 테스트·설명가능성 확보
Optimistic lock충돌이 드물고 재시도 UX가 가능할 때version 충돌 시 재시도 흐름 필요
Pessimistic lock충돌 비용이 크고 직렬 처리가 필요할 때lock wait·deadlock 관리
Sync 호출결과가 즉시 필요하고 빠를 때외부 장애가 요청 pool로 전파
Queue(async)부하 평탄화·재시도·장애 격리가 우선일 때중복 소비·순서 별도 설계
쿠키 옵션

인증 쿠키는 HttpOnly(JS 접근 차단) + Secure(HTTPS 전용) + SameSite(CSRF 완화) 세 개 세트로. SameSite=Lax가 기본 권장, 크로스사이트 전송이 꼭 필요하면 SameSite=None; Secure.

BE-CS-03

자주 터지는 것 · 즉시 확인

신호왜 위험즉시 확인
프론트 validation 있으니 server validation 생략클라이언트는 신뢰 경계 밖 — API 직접 호출로 우회server 측 형식·범위·상태전이 검증 존재 여부
권한 검사를 controller 일부 경로에만다른 진입점으로 우회 경로 생김service/data layer 기준 authorization + resource ownership
POST 재시도로 중복 생성/결제timeout·새로고침·모바일 재시도는 정상 조건idempotency key + unique constraint + 재시도 응답 정책
transaction 안에서 외부 API 호출lock 시간이 외부 장애 시간만큼 늘어남외부호출을 경계 밖으로 / outbox·사후 보상
에러를 전부 500으로 뭉갬클라이언트가 재시도/수정/권한요청을 구분 못함status + machine-readable code + retryable 여부
로그에 requestId/actor 없음장애 때 한 요청을 추적 불가requestId·actor·target·result 로깅(PII·secret 제외)
BE-CS-04

장애 증거 1카드

API INCIDENT TRIAGE 1. 어떤 route·status·errorCode·requestId가 늘었나 (증거 확보)
2. 상류 timeout인가 slow query인가 (upstream span vs DB metric)
3. 이 요청이 재시도 안전한가 (idempotency 있으면 재시도 OK)
4. 사용자 영향 범위는 (특정 tenant/actor/전체)
순서질문증거
1어떤 route·status·errorCode·requestId가 증가했나?gateway/app log, dashboard, grep errorCode
2상류(외부 API) timeout인가, 내부 slow query인가?trace span, upstream status vs DB slow query log
3이 요청은 재시도해도 안전한가?메서드(GET/PUT), idempotency key 유무
4사용자 영향 범위는? (특정 tenant/전체)actor·tenant 분포, 성공/실패 비율
복구/중단 기준

중복 생성·결제가 진행 중이면 쓰기 경로를 먼저 중단(flag off)하고 affected row를 식별한다. 단일 release 회귀면 rollback, 상류 장애면 traffic shedding·retry 축소로 격리. data migration이 얽힌 변경은 rollback 전에 정합성부터 확인.
