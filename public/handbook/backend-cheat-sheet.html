BE-CS-01

개요

백엔드 변경은 정상 응답보다 실패, 재시도, 권한, 정합성에서 깨집니다. 구현 전에 계약과 실패 모드를 먼저 고정합니다.

리뷰 대상먼저 볼 것깨지는 지점
신규 APIrequest/response schema, status code, error shape클라이언트가 실패 사유를 구분하지 못함
권한 기능authentication과 authorization 책임 분리로그인만 확인하고 소유권/역할 검사를 빠뜨림
쓰기 작업transaction boundary, retry, idempotencytimeout 후 재시도로 중복 생성/중복 결제 발생
비동기 처리timeout, backoff, duplicate message, dead letter실패 메시지가 무한 재시도되거나 조용히 유실됨
외부 연동rate limit, circuit breaker, fallback, audit log외부 장애가 전체 요청 지연으로 전파됨
BE-CS-02

핵심 명령/개념

개념압축 설명실무 체크
Authentication누구인지 확인세션/JWT 만료, 재발급, 폐기 정책 확인
Authorization무엇을 할 수 있는지 판단role뿐 아니라 resource ownership 확인
Validation입력 형식, 범위, 상태 전이 가능성 확인client 검증은 UX, server 검증은 신뢰 경계
Transaction함께 성공하거나 함께 실패해야 하는 변경 묶음외부 API 호출과 긴 작업을 경계 밖으로 빼기
Idempotency key같은 요청이 여러 번 와도 결과를 한 번처럼 만드는 식별자POST, 결제, 주문, 메시지 소비에 우선 적용
Backpressure처리량보다 입력이 클 때 시스템이 무너지지 않도록 제어queue length, rate limit, reject 전략 필요
Timeout응답을 무한히 기다리지 않기 위한 상한client, server, DB, 외부 API timeout 정렬
API CHECK읽기 API는 cache와 pagination, 쓰기 API는 validation, transaction, idempotency, audit log를 먼저 확인합니다.
BE-CS-03

판단 기준

선택기준주의점
Session서버 제어와 즉시 폐기가 중요하고 상태 저장 비용을 감당할 수 있음scale-out 시 session store 필요
JWT분산 검증과 stateless가 중요함폐기, claim drift, 짧은 만료/refresh 설계 필요
RBAC역할 단위 권한이 명확하고 조직이 단순함예외가 늘면 role explosion 발생
ABAC소유자, 조직, 상태, 시간 등 속성 기반 판단이 필요함정책 테스트와 설명 가능성 확보 필요
Optimistic lock충돌이 드물고 사용자에게 재시도 경험을 줄 수 있음충돌 메시지와 재시도 흐름 필요
Pessimistic lock충돌 비용이 크고 동시에 하나만 처리해야 함lock wait, deadlock, 처리 시간 관리 필요
Queue즉시 응답보다 평탄화, 재시도, 외부 연동 분리가 더 중요함중복 소비와 순서 보장을 별도로 설계
BE-CS-04

자주 나는 실수

  • 프론트 validation이 있으니 서버 validation을 생략한다. 클라이언트는 신뢰 경계 밖에 있다.
  • 권한 검사를 controller 일부 경로에만 넣고 service/data layer 기준을 만들지 않는다. 우회 경로가 생긴다.
  • POST 요청이 timeout 후 재시도될 수 있다는 점을 설계하지 않는다. 중복 요청은 예외가 아니라 정상 조건이다.
  • transaction 안에서 외부 API 호출이나 긴 작업을 수행한다. lock 시간이 길어지고 장애가 DB 정합성 문제로 번진다.
  • 로그에 requestId, actor, target, result가 없어 장애 분석이 어렵다. 개인정보와 secret은 남기지 않는다.
  • 에러 응답을 모두 500으로 뭉갠다. 클라이언트가 재시도/수정/권한 요청을 구분할 수 없다.
BE-CS-05

면접/리뷰용 한 줄 답변

API 계약contract
API는 함수가 아니라 팀 간 계약이므로 성공 응답보다 실패 응답과 호환성을 먼저 고정합니다.
멱등성idempotency
네트워크와 클라이언트 재시도는 정상 입력이므로 쓰기 작업은 중복 요청을 견뎌야 합니다.
트랜잭션transaction
트랜잭션 경계는 데이터 정합성 단위이며, 오래 잡을수록 lock과 장애 전파 비용이 커집니다.
인증/인가auth
인증은 신원 확인이고 인가는 특정 리소스에 대한 행동 권한 판단이라 둘을 분리해 설계해야 합니다.
비동기 처리async
큐는 일을 나중에 하게 만드는 도구가 아니라 부하 평탄화, 재시도, 장애 격리를 위한 경계입니다.
BE-CS-06

Red Flags

아래 신호가 보이면 구현 디테일보다 장애·권한·정합성 경계를 먼저 재검토합니다.

신호왜 위험한가즉시 확인
프론트에서 버튼을 숨기는 것으로 권한을 처리API를 직접 호출하면 우회됩니다.서버 authorization과 resource ownership 검사를 확인합니다.
transaction 안에서 외부 API 호출lock 시간이 외부 장애 시간만큼 늘어납니다.outbox, 사후 보상, 상태 전이 분리를 검토합니다.
쓰기 API에 idempotency 기준 없음timeout, 새로고침, 모바일 재시도에서 중복 생성됩니다.idempotency key, unique constraint, 재시도 응답 정책을 정합니다.
admin 작업 audit log 없음장애·보안 사고 때 누가 무엇을 바꿨는지 추적할 수 없습니다.actor, target, before/after, requestId, result를 남깁니다.
에러 응답 contract가 없음클라이언트가 재시도, 사용자 수정, 권한 요청을 구분하지 못합니다.status, machine-readable code, message, retryability를 고정합니다.
BE-CS-07

Review Gate

변경 유형merge/deploy 차단 기준통과 증거
인증·인가역할, 소유권, 조직 경계, 만료·폐기 정책이 불명확함권한 matrix와 negative test가 있습니다.
쓰기 API중복 요청, 부분 실패, 재시도 응답이 정의되지 않음idempotency와 transaction 경계가 문서화돼 있습니다.
비동기 처리중복 소비, poison message, dead letter 정책이 없음retry 횟수, backoff, DLQ, 재처리 절차가 있습니다.
외부 연동timeout, rate limit, circuit breaker, fallback이 없음연동 장애 때 사용자 영향과 복구 절차가 설명됩니다.
운영 관측requestId, actor, target, errorCode, latency가 추적되지 않음로그·메트릭·trace에서 하나의 요청을 따라갈 수 있습니다.
BE-CS-08

Failure Playbook

API INCIDENT TRIAGE장애 대응은 “최근 배포”만 보지 말고 요청 contract, 권한, DB, 외부 연동, retry를 같은 순서로 좁힙니다.
순서질문증거
1어떤 route, status, errorCode, requestId가 증가했는가?API gateway log, app log, dashboard
2특정 actor, tenant, resource, payload에서만 실패하는가?권한 로그, validation 실패, 샘플 요청
3DB lock, slow query, connection pool 고갈이 있는가?DB metric, query log, pool saturation
4외부 API timeout/rate limit이 내부 장애로 전파됐는가?trace span, upstream status, retry count
5rollback, feature flag off, traffic shedding 중 무엇이 가장 빠른가?변경 범위, data migration 여부, 사용자 영향
BE-CS-09

API 실행 카드

백엔드 치트시트는 API를 만들거나 고칠 때 정상 응답보다 실패, 권한, 재시도, 관측 증거를 먼저 고정하는 도구입니다.

작업필수 확인복붙 템플릿완료 기준
신규 API성공/실패 status, machine code, retryable 여부{ "code": "ORDER_ALREADY_PAID", "message": "...", "retryable": false }프론트가 오류별 UI 행동을 분기할 수 있음
쓰기 APIidempotency key, unique constraint, timeout 후 재시도Idempotency-Key: order-{clientRequestId}같은 요청 2회 호출 결과가 중복 생성으로 이어지지 않음
권한 변경role, ownership, tenant/org boundary, negative testactor=A, resource owned by B => 403UI 숨김 없이 API 직접 호출도 차단됨
외부 연동timeout, retry budget, fallback, circuit breaker, auditupstream_timeout_ms < server_timeout_ms외부 장애가 전체 요청 pool 고갈로 전파되지 않음
비동기 소비중복 메시지, poison message, DLQ, 재처리 절차messageId + consumerName 기준 dedupe실패 메시지를 추적하고 안전하게 재처리 가능
