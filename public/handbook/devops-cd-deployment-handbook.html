CD와
배포 전략

CD는 검증된 아티팩트를 환경에 안전하게 반영하는 체계입니다. 완전 자동 배포만이 답은 아니며, 서비스 위험도에 맞는 승인과 검증 모델을 설계해야 합니다.

CD-01

CD의 범위

개념의미
Continuous Delivery언제든 배포 가능한 상태로 유지하되 운영 반영은 승인 가능
Continuous Deployment검증 통과 후 운영까지 자동 반영
Deployment automation수동 명령을 스크립트나 pipeline으로 자동화
Release orchestration여러 서비스, DB, feature flag 변경을 순서 있게 조율
CD-02

승인 모델

승인은 배포 속도를 늦추는 장치가 아니라 위험이 큰 변경에 사람의 판단을 넣는 장치입니다.

모델적합한 변경주의
자동 승인작은 코드 변경, 강한 테스트관측과 자동 rollback 기준 필요
수동 승인운영 영향 큰 변경승인 기준이 명확해야 함
Change window고객사 연동, DB migration양쪽 담당자 대기 필요
Emergency deploy장애 핫픽스사후 리뷰와 기록 필수
CD-03

배포 전략

전략장점주의
Rolling간단하고 일반적구버전/신버전 공존 호환성 필요
Blue-Green빠른 전환과 rollback두 환경 비용과 DB 호환성
Canary일부 트래픽으로 위험 축소지표 기반 자동 판정 필요
Feature flag배포와 기능 노출 분리오래된 flag 정리 필요
실무 감각

배포 전략은 애플리케이션 호환성 전략과 같이 설계해야 합니다. rolling 배포에서는 동시에 구버전과 신버전이 떠 있으므로 API, DB schema, message format이 양방향 호환되어야 합니다.

progressive delivery

canary와 feature flag를 결합하면 progressive delivery가 됩니다. 새 버전을 일부 트래픽이나 특정 tenant에만 노출하고, 오류율·latency·핵심 비즈니스 지표가 기준을 넘으면 자동 중단하거나 이전 버전으로 traffic shift합니다.

CD-04

롤백과 복구

방식언제제약
Rollback이전 아티팩트로 되돌림DB migration이 되돌릴 수 있어야 함
Roll-forward빠른 수정 버전 배포원인 파악과 fix가 빨라야 함
Traffic switchBlue-Green, canary이전 환경 보존 필요
Feature flag off기능 오류flag 경계가 잘 설계되어야 함
DB 변경 주의

코드는 rollback 가능해도 데이터베이스 schema와 데이터 변경은 그렇지 않을 수 있습니다. expand-contract migration, backward-compatible schema, migration 검증이 필요합니다.

DB migration 단계운영 기준
Expand새 컬럼·테이블·인덱스를 기존 코드와 호환되게 추가
Dual write/read compatible구버전과 신버전이 동시에 떠도 데이터 계약이 깨지지 않게 유지
Contract더 이상 쓰지 않는 컬럼·경로를 지표로 확인한 뒤 제거
