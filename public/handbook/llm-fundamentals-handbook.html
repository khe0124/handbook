LLM 기초와
모델 동작 원리

LLM은 정답 DB가 아니라 다음 token의 확률 분포를 계산하는 모델입니다. 이 차이를 이해해야 prompt, RAG, evaluation, 운영 설계가 흔들리지 않습니다.

FND-00

LLM Mental Model

LLM은 “알고 있는 사실을 조회”하는 시스템이 아니라, 주어진 context에서 가장 그럴듯한 다음 token을 생성하는 시스템입니다. 그래서 정답성은 모델 내부가 아니라 입력 근거, 출력 계약, 평가 체계로 만들어야 합니다.

FND-EVAL-01 model prediction is not verification → source grounding is required
prompt quality is not enough → evaluation and validation close the loop
오해정확한 이해설계 결과
모델은 사실을 안다모델은 훈련 분포에서 그럴듯한 출력을 만든다최신·사내 지식은 RAG나 tool로 공급
긴 prompt면 안전하다긴 입력은 충돌·희석·비용을 만든다계약을 짧게 유지하고 근거를 선별
temperature 0이면 결정적이다시스템·모델·infra 변화로 완전 재현은 어렵다회귀 테스트와 허용 범위를 둔다
JSON으로 답하라고 하면 충분하다형식 요청과 schema 계약은 다르다structured output과 validation을 둔다
FND-01

FND-TOKEN-01 Tokenization

Token은 비용, 지연, context budget의 기본 단위입니다. 단어 수와 token 수는 같지 않으며, 한국어·코드·JSON·URL·로그는 예상보다 token이 늘어날 수 있습니다.

입력 유형왜 비싸지는가설계 규칙
한국어 문서조사·복합어·문장부호가 여러 token으로 나뉠 수 있음문서 전체 투입보다 chunk 검색 사용
JSON따옴표, key 반복, 중첩 구조가 token을 증가필요 필드만 추려 prompt에 넣음
코드·로그기호와 긴 식별자가 잘게 쪼개짐관련 stack trace와 주변 코드만 넣음
URL·base64의미 대비 token 소모가 큼원문 대신 metadata나 파일 참조 사용
FND-TOKEN-02

입력 token을 줄이는 것은 단순 비용 최적화가 아닙니다. 불필요한 context가 줄어들면 irrelevant context poisoning과 lost-in-the-middle 위험도 함께 줄어듭니다.

FND-ATTN-01

Attention은 완전한 메모리가 아니라 입력 token 간 관련도를 계산하는 메커니즘입니다. “context 안에 있으니 반드시 참고한다”는 가정은 위험합니다.

DEPTH

기초 개념을 설계로 옮기는 법

MARKER: concept-depth — 정답성은 모델 속성이 아니라 시스템 속성이다

LLM을 DB처럼 다루면 설계가 계속 어긋납니다. DB는 저장된 값을 조회하지만, LLM은 context 안의 token 관계를 바탕으로 그럴듯한 다음 출력을 생성합니다. 따라서 모델 내부 지식, 입력 context, 검색된 근거, 외부 tool, validator는 서로 다른 책임을 가져야 합니다. 최신 사내 정책은 모델이 “알고 있길” 기대하는 것이 아니라 retrieval이나 tool call로 공급해야 하고, 출력 형식은 모델의 선의에 맡기는 것이 아니라 structured output과 validator로 강제해야 합니다.

이 관점은 hallucination을 설명하는 데도 중요합니다. 환각은 모델이 악의적으로 거짓말하는 문제가 아니라, 근거가 없거나 애매한 상황에서도 그럴듯한 연속 token을 만드는 구조적 결과입니다. 그래서 “모르면 모른다고 해”라는 문장만으로는 부족합니다. no-answer policy, citation validation, claim-level groundedness, golden dataset이 함께 있어야 모델이 모르는 상황을 제품이 다룰 수 있습니다.

이 예산표가 보여주는 핵심은 긴 prompt가 항상 좋은 prompt가 아니라는 점입니다. system prompt가 길어지고 대화 기록을 무작정 붙이면 정작 필요한 검색 근거와 출력 schema가 들어갈 공간이 사라집니다. 출력 여유가 부족하면 JSON이 잘리고, 검색 chunk를 많이 넣으면 관련 없는 문서가 attention을 오염시키며, history가 길면 이전 지시와 현재 지시가 충돌합니다. token budget은 비용 계산이면서 동시에 품질 설계입니다.

MARKER: failure-driven — Context is not memory

Context window 안에 정보가 있다는 말은 모델이 그 정보를 반드시 정확하게 사용한다는 뜻이 아닙니다. 중요한 근거가 중간에 묻히거나, 마지막 사용자 메시지가 과하게 반영되거나, 같은 사실을 표현한 중복 chunk가 서로 경쟁할 수 있습니다. RAG에서 chunk ordering, deduplication, parent expansion, context compression을 설계해야 하는 이유가 여기에 있습니다. context는 저장소가 아니라 한 번의 추론을 위한 작업 공간이며, 작업 공간이 지저분하면 답변도 흔들립니다.

Sampling도 같은 방식으로 오해하면 안 됩니다. temperature를 낮추면 표현의 변동성은 줄어들 수 있지만, 근거 없는 주장을 사실로 바꾸지는 못합니다. 낮은 temperature의 오답은 더 일관된 오답이 될 수 있고, 높은 temperature의 답변은 더 다양한 표현을 만들지만 groundedness를 보장하지 않습니다. factuality는 sampling이 아니라 evidence supply, no-answer rule, validator, evaluation으로 통제해야 합니다.

MARKER: artifact-ready — Hallucination response architecture

missing_knowledge: detect retrieval_empty -> prevent with RAG/tool -> recover no-answer -> measure recall@k
unsupported_synthesis: detect unsupported claim -> prevent claim citation -> recover remove claim -> measure groundedness
stale_context: detect old document_version -> prevent freshness filter -> recover reindex -> measure stale_source_rate
unsafe_guessing: detect risk tier -> prevent refusal/human review -> recover escalation -> measure policy_violation_rate

MARKER: failure-driven — Sampling을 사실성 제어 장치로 오해하는 실패

temperature를 낮추면 답이 더 차분해지고 반복 가능성은 올라가지만, 근거가 없는 답이 사실이 되는 것은 아닙니다. 반대로 temperature를 높이면 표현은 다양해질 수 있지만, 누락된 지식이나 오래된 context를 보완하지는 못합니다. 초급 구현에서는 “temperature를 0으로 두면 환각이 줄어든다”는 식으로 문제를 닫아버리기 쉽습니다. 미들급 설계에서는 sampling이 variance와 style을 제어한다는 점을 분리하고, 사실성은 retrieval evidence, tool result, validator, no-answer policy로 다룹니다. 같은 질문에 대해 낮은 temperature 출력이 틀렸다면 모델 설정 문제가 아니라 근거 공급과 검증 경계가 실패한 것입니다.
