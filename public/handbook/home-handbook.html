Dev Handbook

이 핸드북은 프론트엔드 기반 개발자가 백엔드, 인프라, DevOps, AX, 디자인까지 연결해서 제품을 끝까지 책임지는 역량을 기르기 위한 작업 지도입니다. 각 문서는 지식을 외우기보다 판단 기준, 실패 모드, 운영 관점을 익히도록 구성되어 있습니다.

STANCE : 개념 → 구조 → 운영 → 검증 → 실전 반복 · UPDATED 2026-06
HOME-01

전체 로드맵

전체 흐름은 화면을 만드는 능력에서 출발해, 데이터와 시스템 경계, 배포와 운영, AI 기반 작업 루프, 제품 경험 설계로 확장됩니다.

프론트엔드→ 백엔드→ 네트워크 인프라→ DevOps→ AX→ 디자인·실무
축핵심 질문도달해야 할 상태
제품 화면사용자가 빠르고 명확하게 목적을 달성하는가렌더링, 접근성, 성능, 테스트, 배포를 함께 판단
서버와 데이터요청, 상태, 트랜잭션, 권한이 안전하게 설계되었는가API 계약, DB, 인증, 아키텍처, 장애 대응을 연결
인프라 경계패킷이 어떤 주소, 경로, 정책을 거쳐 이동하는가VPC, 라우팅, 방화벽, DNS, VPN 문제를 설명하고 디버깅
전달 시스템변경이 반복 가능하고 되돌릴 수 있게 배포되는가CI/CD, IaC, 컨테이너, 관측 가능성, 신뢰성을 운영
작업 증폭AI와 도구가 검증 가능한 변경을 만들도록 통제되는가하네스, 컨텍스트, 루프, 멀티 에이전트, 거버넌스 활용
경험 설계기능이 실제 사용 흐름과 의사결정 구조에 맞는가UX 사고, 정보구조, UI 패턴, 디자인 시스템을 실무에 반영
HOME-02

학습 순서

순서대로 읽되, 모든 문서를 한 번에 끝내려 하지 말고 현재 프로젝트에서 바로 검증할 수 있는 주제부터 실험합니다.

단계읽을 메뉴목표실전 과제
1프론트엔드브라우저, 렌더링, 성능, 접근성, 테스트의 기준 세우기현재 화면 하나를 Lighthouse, DevTools, 접근성 체크로 진단
2백엔드API, DB, 인증, 비동기, 관측 가능성의 연결 이해기능 하나를 API 계약, 트랜잭션, 실패 응답까지 설계
3네트워크 인프라주소, 경로, 정책, 이름 해석, VPN 디버깅의 기본기 확보서비스 요청이 브라우저에서 서버까지 가는 경로를 그림으로 설명
4DevOps변경을 빌드, 테스트, 배포, 관측, 복구 가능한 단위로 만들기작은 앱에 CI, preview deploy, rollback checklist 구성
5AXAI 작업을 프롬프트가 아니라 하네스와 검증 루프로 다루기작업 하나를 Plan → Act → Verify → Reflect episode로 기록
6디자인요구사항을 사용 흐름, 정보구조, 컴포넌트 품질로 번역폼 또는 대시보드 하나를 오류 상태, 빈 상태, 권한 상태까지 재설계
7예시 사례·실무 가이드도구와 언어별 감각을 실제 명령과 코드로 고정새 환경 설정부터 배포까지 재현 가능한 체크리스트 작성
HOME-03

메뉴별 목적

백엔드system core
요청 처리, 데이터 정합성, 인증·인가, 아키텍처 경계, 장애 대응을 통해 기능을 운영 가능한 시스템으로 바꾸는 메뉴입니다.
프론트엔드product surface
브라우저 동작, 렌더링, 접근성, 성능, 보안, 테스트, 배포를 기준으로 사용자와 맞닿은 품질을 다룹니다.
네트워크 인프라packet path
VPC, CIDR, 라우팅, 방화벽, DNS, VPN을 통해 서비스 간 통신이 실제로 어떻게 성립하는지 설명합니다.
DevOpsdelivery system
빌드, 테스트, 배포, IaC, 컨테이너, 오케스트레이션, 관측 가능성, 복구 체계를 하나의 전달 시스템으로 봅니다.
AXAI execution
AI가 만든 산출물을 신뢰 가능한 변경으로 만들기 위해 하네스, 컨텍스트, 루프, 검증, 권한 경계를 설계합니다.
디자인experience design
문제 정의, 정보구조, 사용자 흐름, UI 패턴, 디자인 시스템, QA를 통해 기능을 쓰기 좋은 경험으로 정리합니다.
예시 사례concrete syntax
Java, JPA, Spring Boot, Docker, nginx, Redis, PostgreSQL처럼 자주 쓰는 기술을 실제 코드와 명령으로 익힙니다.
실무 가이드operating playbook
새 PC 셋업, 패키지 매니저, 환경변수, DB 시딩, CI/CD, 로그 도구처럼 반복되는 일을 재현 가능한 절차로 만듭니다.
HOME-04

핸드북 사용법

이 문서는 읽는 자료라기보다 작업 전에 확인하고, 작업 중에 기준으로 삼고, 작업 후에 회고하는 체크리스트에 가깝습니다.

상황사용 방식남길 산출물
새 기능 시작관련 메뉴의 개요와 실패 모드를 먼저 읽고 범위를 정한다요구사항, 비목표, 수용 기준, 리스크 목록
설계 판단각 메뉴의 트레이드오프 표를 보고 선택 근거를 적는다ADR, API 계약, 데이터 모델, 운영 가정
구현 중예시 사례와 실무 가이드의 명령·패턴을 참고한다작은 diff, 테스트, 검증 로그
리뷰 전보안, 접근성, 성능, 관측 가능성, 배포 위험을 점검한다리뷰 체크리스트, 남은 리스크, 롤백 방법
장애·버그네트워크, 백엔드, 프론트엔드, DevOps 문서를 따라 계층별로 좁힌다재현 조건, 원인 가설, 검증 결과, 재발 방지책
운영 기준

좋은 개발자는 “내 코드가 맞다”가 아니라 “이 변경이 어떤 조건에서 안전하고, 어떤 조건에서 깨지며, 깨졌을 때 어떻게 알 수 있는가”를 설명합니다. 이 핸드북의 모든 메뉴는 그 설명을 더 잘하기 위한 재료입니다.

실전 루프

한 주 단위로 아래 루프를 반복하면 읽은 지식이 프로젝트 판단으로 전환됩니다.

2. 관련 메뉴 2개 이상을 연결해 읽는다
3. 작업 전 가정과 수용 기준을 적는다
4. 구현 또는 분석을 진행한다
5. 테스트·빌드·수동 시나리오로 검증한다
6. 놓친 지식, 실패한 가정, 다음에 볼 문서를 기록한다
