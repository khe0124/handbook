Dev Handbook

이 핸드북은 개발자가 CS 기본기, 시스템 사고, 제품 개발, 인프라 운영, AI Native 실행, 디자인 실무, 커리어 증거까지 연결해서 제품을 끝까지 책임지는 역량을 기르기 위한 작업 지도입니다. 대메뉴는 성격이 같은 문서끼리 묶고, 각 묶음 안에서 목적별로 찾도록 정리했습니다.

STANCE : 개념 → 구조 → 운영 → 검증 → 실전 반복 · UPDATED 2026-06
HOME-01

AI Native 개발자 역량 지도

이 핸드북의 큰그림은 많은 기술을 나열하는 것이 아니라, 기능 하나를 끝까지 책임지는 개발자를 만드는 것입니다. 기본기, 제품 구현, 운영 책임, AI Native 실행, 사용자 경험, 반복 작업 자동화, 커리어 증거가 서로 끊기지 않아야 합니다.

역량 층담당하는 질문이 핸드북에서 확인할 문서남겨야 할 증거
기본기왜 이 구조가 맞고 어디서 깨지는가CS 기초, 컴퓨터 시스템, 언어 런타임, 응용 수학·측정복잡도 판단, resource trace, benchmark evidence, 불변식 설명
제품 구현사용자 흐름이 UI, API, DB 상태로 안전하게 닫히는가프론트엔드 핵심, 백엔드 핵심, 데이터 계층, Java·Spring·JPAUI state model, API contract, transaction boundary, 테스트 결과
운영 책임배포 후 문제가 생기면 어떻게 감지하고 되돌리는가인프라·운영, 런타임 품질, 플랫폼 도구runbook, log/metric/trace, rollback plan, incident note
AI Native 실행AI를 빠른 초안이 아니라 검증 가능한 작업 시스템으로 쓰는가LLM, AX 실무task spec, context package, eval dataset, security fixture, cost/latency trace
제품 경험기능이 실제 사용 흐름과 의사결정 구조에 맞는가디자인 실무, 프론트엔드 인터랙션·품질IA, user flow, accessibility check, QA gate, before/after 화면
증거화내가 한 판단과 결과를 면접·포트폴리오에서 방어할 수 있는가면접·커리어, 실무 도구PR, ADR, metric, 장애기록, 포트폴리오 narrative, 30초·90초 답변

기능 하나를 끝까지 책임지는 작동 모델

실무에서는 메뉴가 따로 움직이지 않습니다. 요구사항을 받는 순간부터 사용자 흐름, UI 상태, API 계약, 데이터·권한, 배포·관측, AI 보조, 커리어 증거가 하나의 작업 단위로 이어져야 합니다.

요구사항→ 사용자 흐름→ UI 상태→ API 계약→ 데이터·권한→ 배포·관측→ AI 보조→ 커리어 증거
작업 순간연결할 핸드북확인 질문
요구사항을 받았을 때디자인 실무 · 개발 핸드북 · LLM 작업 표준사용자 문제, 비목표, 수용 기준, AI에 줄 context package가 분리됐는가
구현을 시작할 때프론트엔드 · 백엔드 · 데이터 · Java/Spring화면 상태, 요청/응답, 트랜잭션, 권한, 실패 응답이 같은 모델을 공유하는가
리뷰를 요청할 때품질·릴리스 · LLM 평가 · AX 실행 루프테스트, 접근성, 성능, eval, security fixture, 비용/지연 근거가 있는가
배포할 때인프라·운영 · 플랫폼 도구 · 런타임 품질환경 차이, rollback, 로그·메트릭·트레이스, 장애 시나리오가 준비됐는가
경험을 정리할 때면접·커리어문제, 선택지, 버린 대안, 결과 지표, 후속 개선을 증거로 말할 수 있는가
읽는 기준

어떤 문서를 보든 “이 지식이 기능의 어느 경계에서 쓰이는가”를 먼저 묻습니다. CS는 설계 근거가 되고, 프론트엔드는 사용자 흐름이 되고, 백엔드는 상태와 권한이 되고, 운영은 신뢰성이 되고, AI Native는 작업 속도와 검증 체계가 되고, 커리어 문서는 그 모든 판단을 증거로 바꿉니다.

HOME-02

전체 로드맵

전체 흐름은 CS·시스템·런타임·측정 기본기에서 출발해, 화면과 서버, 데이터와 운영, LLM/AX 실행 체계, 제품 경험 설계, 커리어 증거로 확장됩니다.

홈→ 개발 핸드북→ LLM→ 인프라·운영→ AX 실무→ 디자인 실무→ 실무 도구→ 면접·커리어
축핵심 질문도달해야 할 상태
기본기복잡도, 시스템 자원, 런타임, 측정을 판단 근거로 쓰는가CS, OS, 네트워크, 언어 런타임, 통계·측정으로 설계 선택을 설명
제품 화면사용자가 빠르고 명확하게 목적을 달성하는가렌더링, 접근성, 성능, 테스트, 배포를 함께 판단
서버와 데이터요청, 상태, 트랜잭션, 권한이 안전하게 설계되었는가API 계약, DB, 인증, 아키텍처, 장애 대응을 연결
인프라 경계패킷이 어떤 주소, 경로, 정책을 거쳐 이동하는가VPC, 라우팅, 방화벽, DNS, VPN 문제를 설명하고 디버깅
전달 시스템변경이 반복 가능하고 되돌릴 수 있게 배포되는가CI/CD, IaC, 컨테이너, 관측 가능성, 신뢰성을 운영
AI Native 실행AI와 LLM 기능이 검증 가능한 변경과 제품 기능으로 닫히는가context package, eval dataset, security fixture, cost/latency trace, approval gate 활용
디자인 실무기능이 실제 사용 흐름과 의사결정 구조에 맞는가문제 정의, IA, user flow에서 UI 시스템, AX 인터랙션, QA까지 연결
실무 도구반복 작업을 빠르고 안전하게 처리하는 절차가 있는가치트시트, 환경 설정, 빌드, 설정, 릴리스 절차를 재사용
HOME-03

학습 순서

순서대로 읽되, 모든 문서를 한 번에 끝내려 하지 말고 현재 프로젝트에서 바로 검증할 수 있는 주제부터 실험합니다.

단계읽을 메뉴목표실전 과제
1개발 핸드북CS, 시스템, 런타임, 측정 기본기를 제품 개발 판단으로 전환자료구조 선택, syscall/resource trace, benchmark evidence packet을 하나씩 작성
2개발 핸드북프론트엔드와 백엔드의 화면·API·DB·보안 경계 이해기능 하나를 UI 상태, API 계약, 트랜잭션, 실패 응답까지 설계
3LLMLLM 기능을 prompt, RAG, eval, security, operations 체계로 제품화AI Native Definition of Done으로 task spec, eval dataset, cost/latency trace를 제출
4인프라·운영네트워크 경로, 배포, 관측, 장애 대응의 기본기 확보서비스 요청이 브라우저에서 서버까지 가는 경로와 배포 후 확인 절차를 설명
5AX 실무AI 작업을 검증 가능한 실행 루프와 거버넌스로 통제작업 하나를 Plan → Act → Verify → Reflect episode로 기록
6디자인 실무문제 정의·IA·Flow와 Layout·Form·Component·System·QA를 연결핵심 흐름 하나를 IA, user flow, interaction checklist, QA gate로 검증
7실무 도구치트시트, 환경 설정, 빌드·릴리스 절차를 작업 루프로 정리반복 명령과 리뷰 체크리스트를 프로젝트 runbook에 반영
8면접·커리어앞 단계의 산출물을 답변, 포트폴리오, 직무별 포지셔닝으로 정리대표 질문 하나를 30초·90초 답변과 꼬리질문으로 녹음
HOME-04

메뉴별 목적

홈starting point
전체 로드맵과 학습 순서를 확인하는 입구입니다. 세부 문서를 바로 찾기보다 어떤 묶음에서 시작할지 결정합니다.
면접·커리어interview & positioning
기술면접, 프로젝트 심층 답변, 컬처·압박 대응, 개인화 포지셔닝을 한곳에 묶었습니다. 면접 준비와 직무별 답변 조정은 여기서 시작합니다.
개발 핸드북product engineering
CS 기초, 컴퓨터 시스템, 언어 런타임, 응용 측정에서 시작해 프론트엔드, 백엔드, 데이터, 아키텍처, 런타임 품질, Java·Spring·JPA와 백엔드 예시를 제품 개발의 한 흐름으로 봅니다.
LLMai native systems
LLM 기능을 모델 호출이 아니라 prompt contract, RAG, eval, security, cost/latency operations, AI Native Definition of Done으로 다룹니다.
인프라·운영delivery & reliability
네트워크 인프라와 DevOps를 묶었습니다. 요청 경로, VPC, DNS, VPN, CI/CD, 컨테이너, 관측 가능성, 장애 대응을 한 운영 흐름으로 정리합니다.
AX 실무ai execution
AI 작업을 조직, 자동화, 컨텍스트, 하네스, 루프, 거버넌스 관점으로 관리합니다. 생성 결과보다 검증 가능한 실행 체계를 다룹니다.
디자인 실무product experience
문제 정의, 정보구조, 사용자 흐름을 먼저 잡고 레이아웃, 인터랙션, 폼, 컴포넌트, 디자인 시스템, AX 인터랙션, 접근성, QA로 실행 품질을 검증합니다.
실무 도구working tools
치트시트, 환경 설정, 빌드, 설정, DB 시딩, CI/CD, 로그처럼 매일 반복하는 작업 절차를 빠르게 찾는 메뉴입니다.
HOME-05

핸드북 사용법

이 문서는 읽는 자료라기보다 작업 전에 확인하고, 작업 중에 기준으로 삼고, 작업 후에 회고하는 체크리스트에 가깝습니다.

상황사용 방식남길 산출물
새 기능 시작관련 메뉴의 개요와 실패 모드를 먼저 읽고 범위를 정한다요구사항, 비목표, 수용 기준, 리스크 목록
설계 판단각 메뉴의 트레이드오프 표를 보고 선택 근거를 적는다ADR, API 계약, 데이터 모델, 운영 가정
구현 중개발 핸드북의 기본기와 예시, LLM 작업 표준, AX 실무의 루프, 디자인 실무의 상태 기준, 실무 도구의 명령어를 참고한다작은 diff, 테스트, eval report, 검증 로그
리뷰 전보안, 접근성, 성능, 관측 가능성, 배포 위험을 점검한다리뷰 체크리스트, 남은 리스크, 롤백 방법
장애·버그네트워크, 백엔드, 프론트엔드, DevOps 문서를 따라 계층별로 좁힌다재현 조건, 원인 가설, 검증 결과, 재발 방지책
운영 기준

좋은 개발자는 “내 코드가 맞다”가 아니라 “이 변경이 어떤 조건에서 안전하고, 어떤 조건에서 깨지며, 깨졌을 때 어떻게 알 수 있는가”를 설명합니다. 이 핸드북의 모든 메뉴는 그 설명을 더 잘하기 위한 재료입니다.

실전 루프

한 주 단위로 아래 루프를 반복하면 읽은 지식이 프로젝트 판단으로 전환됩니다.

2. 관련 메뉴 2개 이상을 연결해 읽는다
3. 작업 전 가정과 수용 기준을 적는다

5. 테스트·빌드·수동 시나리오로 검증한다
6. 놓친 지식, 실패한 가정, 다음에 볼 문서를 기록한다
훈련 주제연결할 메뉴완료 기준
로그인 기능프론트엔드 보안 · 인증과 보안 · 테스트 전략세션, 토큰, 오류 상태, 회귀 테스트를 함께 설명

VPN 연동네트워크 인프라 · IPsec VPN · 네트워크 디버깅L3/L4/L7 순서로 통신 실패 지점을 설명

HOME-07
