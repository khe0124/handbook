IP 주소·CIDR·서브넷

네트워크 설계의 첫 질문은 “어떤 주소 범위를 쓸 것인가”입니다. CIDR을 잘못 잡으면 서버 생성은 되더라도 VPN, 라우팅, 확장, 고객사 연동에서 되돌리기 어려운 문제가 생깁니다.

IP-01

IP 주소와 사설 대역

클라우드 VPC는 보통 인터넷에 직접 라우팅되지 않는 사설 IP 대역을 사용합니다.

대역CIDR실무 사용
10.0.0.0 대역10.0.0.0/8큰 조직, 클라우드 VPC에서 자주 사용
172.16.0.0 대역172.16.0.0/12온프레미스·사내망에서 자주 사용
192.168.0.0 대역192.168.0.0/16가정/소규모 네트워크에서 흔함
RFC 1918

위 세 범위가 RFC 1918 사설 IP 대역입니다. 인터넷 라우터는 이 주소를 공인 인터넷으로 전달하지 않습니다. 외부 인터넷과 통신하려면 NAT 또는 공인 IP가 필요합니다.

IP-02

CIDR 읽는 법

CIDR의미사용 가능 IP 감각용도
/16앞 16비트 고정약 65KVPC 전체
/20앞 20비트 고정약 4K중간 규모 VPC 또는 큰 subnet
/24앞 24비트 고정254개 수준일반 subnet
/28앞 28비트 고정14개 수준관리용 작은 subnet
/32단일 IP1개특정 서버 허용

클라우드별로 subnet의 일부 IP는 reserved IP로 예약됩니다. 예를 들어 네트워크 주소, 브로드캐스트 성격의 주소, 게이트웨이, DNS 용도 IP가 빠질 수 있으므로 문서상 사용 가능 개수와 실제 할당 가능 개수가 다를 수 있습니다.

IP-03

서브넷 설계

서브넷은 보안, 라우팅, 가용 영역, 역할 경계를 표현합니다.

서브넷예시 CIDR배치 리소스라우팅 성격
Public10.0.1.0/24Load Balancer, bastionInternet Gateway 경로 보유
Private App10.0.2.0/24API, WAS, workerNAT 또는 VPN 경로 사용
Private DB10.0.3.0/24DB, Redis인터넷 직접 경로 없음
Management10.0.10.0/28관리 장비, endpoint제한적 접근
IP-04

대역 충돌

VPN에서 가장 치명적인 초기 설계 오류는 양쪽 네트워크가 같은 CIDR을 쓰는 것입니다.

충돌 예시 
우리 VPC:      10.0.0.0/16
고객사 VPC:    10.0.0.0/16

10.0.2.10으로 가는 패킷은 우리 쪽 서버인가,
고객사 서버인가? 라우터가 구분할 수 없다.
상황대응
새 VPC 설계 전고객사, 사내망, 다른 클라우드 VPC 대역을 먼저 수집
이미 충돌한쪽 CIDR 재설계, NAT 기반 우회, 프록시/중계망 검토
여러 고객사 연동고객사별 격리 VPC 또는 충분한 주소 계획 필요
IP-05

실무 기준

Case study

VPN 프로젝트에서 우리 VPC를 10.0.0.0/16으로 만들었는데 고객사도 같은 대역을 쓰면, 터널을 만들어도 트래픽이 어디로 가야 하는지 결정할 수 없습니다. 실제 프로젝트에서는 VPC 생성 전에 고객사 사설 대역, 사내망 대역, 다른 클라우드 대역을 표로 받아 충돌 검사를 먼저 해야 합니다.

주소 계획address plan
지금 필요한 IP 수가 아니라 향후 AZ, 환경, 고객사 연동까지 고려해 잡습니다.
서브넷 크기capacity
너무 작게 잡으면 확장 때 재구성이 필요하고, 너무 크게 잡으면 대역 낭비와 충돌 가능성이 커집니다.
