AX 개요

AX는 프롬프트 작성 기술이 아니라 업무를 검증 가능한 실행 시스템으로 바꾸는 운영 역량입니다. 핵심은 더 긴 요청문이 아니라 더 좋은 작업 명세, 컨텍스트, 권한, 검증 루프입니다.

STANCE : 생성보다 검증 · 프롬프트보다 컨텍스트 · 자동화보다 책임 경계 · COMPILED 2026-06
AX-01

AX의 정의

AX는 AI Experience, Agent Experience, AI-augmented eXecution을 모두 포괄하지만, 실무에서의 의미는 분명합니다. AX는 업무를 AI에게 말로 맡기는 방식이 아니라 사람이 책임지는 실행 체계 안에서 AI가 작업하고, 검증받고, 기록되도록 만드는 운영 방식입니다.

관점낮은 수준의 AI 활용AX 관점
목표코드를 빨리 생성검증 가능한 변경을 반복적으로 생산
입력자연어 요청목표, 비목표, 컨텍스트, 제약, 성공 기준
실행단발 프롬프트Plan → Act → Verify → Reflect 루프
검증AI의 자기 요약테스트, 리뷰, trace, verification report
책임AI가 해줬다사람이 하네스와 승인 경계를 운영한다
핵심 문장

AX의 목표는 AI에게 더 많은 일을 시키는 것이 아닙니다. 작업을 더 작게 나누고, 검증을 더 명확히 하며, 실패했을 때 더 빨리 멈추는 것입니다.

AX-02

AX 시스템 모델

AI 개발 성과는 모델 단독 능력이 아니라 명세, 컨텍스트, 권한, 도구, 검증, 기록이 결합된 시스템의 결과입니다. 같은 모델도 어떤 업무 구조 위에 놓이느냐에 따라 위험한 자동완성 도구가 되기도 하고, 통제 가능한 실행 파트너가 되기도 합니다.

기존의 Model·Harness·Environment 관점은 그대로 유지하되, 조직 도입에서는 이를 task spec, context package, tool permission, verification report, audit trail로 운영 가능한 산출물까지 내려야 합니다.

AX-03

도입 학습 지도

AX 도입 학습 지도는 개인이 프롬프트를 더 잘 쓰게 만드는 과정이 아니라 조직이 반복 업무를 발견하고, 작은 파일럿으로 검증하고, 표준 산출물로 남기는 과정입니다.

단계학습 질문산출물
조직 진단어떤 업무가 반복되고 검증 가능하며 실패 비용이 낮은가workflow inventory, risk baseline
파일럿작은 팀에서 read-only 또는 low-risk 업무를 안전하게 실험할 수 있는가pilot brief, guarded workflow
표준 산출물성공한 업무를 다른 팀이 재사용할 수 있는 형태로 남겼는가task spec, context package, eval checklist
확산교육, 권한, 지표, governance가 함께 준비되었는가rollout note, 운영 지표, audit rule
학습 원칙

도입은 “AI를 써보자”에서 시작하지 않습니다. 조직 진단으로 후보 업무를 고르고, 파일럿으로 실패 비용을 제한하며, 표준 산출물로 재현 가능한 학습을 남깁니다.

AX-04

전문가의 책임 경계

전문가는 AI보다 더 많은 문장을 쓰는 사람이 아니라 AI가 실행할 수 있는 경계와 멈춰야 할 조건을 정하는 사람입니다. AX 책임 원칙은 명확합니다. AI는 작업자가 될 수 있지만 책임자는 될 수 없습니다.

책임 영역AI가 할 수 있는 일전문가가 남겨야 하는 일
판단후보안, 비교표, 위험 신호 제안최종 의사결정과 승인 기준 설정
실행제한된 권한 안에서 diff, 문서, 분석 작성권한 범위와 destructive action 승인
검증테스트 실행, 로그 요약, 실패 후보 분류검증 충분성 판단과 미검증 위험 공개
책임작업 흔적과 근거 제공품질, 보안, 법무, 고객 영향에 대한 책임
AX 책임 원칙

AI는 작업자가 될 수 있지만 책임자는 될 수 없습니다. 책임은 작업을 명세화하고, 권한을 부여하고, 검증 결과를 승인한 사람과 조직에 남습니다.

AX-05

표준 산출물

AX는 말로 전파되지 않습니다. 반복 가능한 산출물로 전파됩니다. 산출물은 AI가 다음 episode에서 같은 품질로 일하도록 만드는 최소 운영 단위입니다.

산출물내용사용 시점
task spec목표, 범위, 비목표, 성공 기준, 중단 조건작업 시작 전
context package관련 파일, 정책, 데이터 계약, 예외 사례모델 실행 전
tool permission허용 도구, 금지 도구, 승인 필요 작업자동화 수준 결정 시
verification report실행한 검증, 실패한 검증, 남은 위험리뷰와 승인 전
audit trail요청, 판단, 도구 호출, 변경, 승인 내역사후 분석과 governance review
AX-06

도입 체크포인트

AX 도입은 도구 설치 여부가 아니라 업무가 검증 가능한 실행 시스템으로 바뀌었는지로 판단합니다.

ADOPTION CHECKPOINTS 1. 반복 업무가 업무 단위와 episode 단위로 분해되었는가
2. 각 업무에 task spec과 context package가 있는가
3. tool permission과 승인 게이트가 위험도에 맞는가
4. verification report가 리뷰의 입력으로 쓰이는가
5. audit trail이 사후 학습과 책임 추적에 충분한가
6. 표준 산출물이 교육과 확산에 재사용되는가
운영 기준

좋은 AX 도입은 더 많은 AI 사용량을 만드는 것이 아니라, 더 적은 혼란으로 더 많은 검증 가능한 업무 결과를 만드는 것입니다.

AX 엔지니어
역량 모델

AX 엔지니어는 AI를 잘 쓰는 사람이 아니라 업무를 AI가 안전하게 수행할 수 있는 실행 시스템으로 바꾸는 사람입니다. 핵심 역량은 프롬프트 작성보다 업무 분해, 컨텍스트 설계, 검증, 권한 통제, 효과 측정에 있습니다.

STANCE : AUTOMATION ARCHITECT · VERIFICATION OWNER · SOCIOTECHNICAL OPERATOR
CAP-01

역할 정의

AX 역량은 개인 사용자, 업무 책임자, 플랫폼 리더가 서로 다른 책임을 나누어 가질 때 조직 역량이 됩니다. 한 사람이 모든 것을 해내는 구조가 아니라 역할별 산출물과 승인 경계를 명확히 하는 구조가 필요합니다.

역할책임대표 산출물
Practitioner자신의 반복 업무를 episode로 기록하고 작은 자동화를 실험task spec, episode log, verification note
Workflow Owner팀 업무를 표준 workflow로 정의하고 품질 기준을 관리eval rubric, automation scorecard, rollout note
Platform/Adoption Lead공통 하네스, 권한 정책, 교육 과정, governance를 설계governance memo, training path, platform standard
역할 기준

AX 엔지니어는 AI 산출물을 대신 믿어주는 사람이 아닙니다. 업무를 명세화하고 검증 가능하게 만들며, 팀이 재사용할 수 있는 운영 자산으로 남기는 사람입니다.

CAP-02

역량 지도

역량 지도는 프롬프트 기술 목록이 아니라 업무 분해 관점에서 반복 업무를 실행 가능한 단위로 바꾸는 능력 목록입니다.

역량설명확인 질문
Workflow Mining반복 업무를 단계, 입력, 출력, 승인 지점으로 분해이 업무의 반복 패턴과 예외를 설명할 수 있는가
Task SpecificationAI가 실행 가능한 단위로 업무 명세 작성성공 조건과 중단 조건이 테스트 가능하게 쓰였는가
Context Engineering필요한 지식과 불필요한 잡음을 분리모델이 봐야 할 파일, 정책, 로그, 계약이 선별되었는가
Harness Design도구, 권한, 상태, 메모리, 검증을 묶은 실행 계층 설계AI가 어떤 도구를 어떤 순서와 권한으로 쓰는가
Evaluation자동화 결과를 episode 단위로 평가정확도, 비용, 시간, rework, escaped defect를 측정하는가
Governance보안, 개인정보, 라이선스, 감사, 승인 경계 관리데이터와 권한이 업무 위험도에 맞게 제한되는가
Change Management팀의 일하는 방식을 점진적으로 바꿈파일럿, 교육, 피드백, 확산 계획이 있는가
CAP-03

성장 레벨

AX 성장 레벨은 더 복잡한 모델을 쓰는 순서가 아니라 더 넓은 책임 경계를 다루는 순서입니다. 개인 episode에서 팀 workflow, 조직 governance로 이동할수록 산출물의 품질과 추적 가능성이 중요해집니다.

레벨초점증거
L1 Practitioner개인 업무의 task spec과 검증 루틴 작성episode log, 작은 diff, 검증 기록
L2 Workflow Owner팀 반복 업무를 표준 workflow로 전환eval rubric, automation scorecard, 팀 템플릿
L3 Platform/Adoption Lead권한, 교육 과정, 지표, governance를 연결governance memo, rollout note, adoption metric
LEVEL-UP SIGNAL 개인이 AI를 잘 쓰는가 → 팀이 같은 업무를 같은 방식으로 검증하는가 → 조직이 위험과 효과를 같은 언어로 판단하는가
CAP-04

훈련 루브릭

훈련은 도구 기능 소개보다 실제 업무 episode를 기준으로 평가해야 합니다. 교육 과정은 실습자가 자신의 업무를 가져와 명세, 컨텍스트, 검증, 기록으로 바꾸는 방식이 가장 효과적입니다.

훈련 항목초급숙련평가 산출물
Workflow Mining업무를 큰 요청으로 설명trigger, input, action, verification, escalation으로 분해workflow map
Task Specification원하는 결과만 작성비목표, 중단 조건, 승인 기준까지 포함task spec
Context Engineering많은 파일을 한 번에 제공필수 근거와 제외 근거를 구분context package
Harness DesignAI에게 자유롭게 실행시킴권한과 tool permission을 위험도에 맞춤harness checklist
Evaluation결과를 읽고 감으로 판단eval rubric과 반례로 판정eval rubric
Governance정책을 사후에 확인security, legal, data rule을 사전에 반영governance memo
Change Management잘 된 프롬프트를 공유교육 과정과 rollout note로 확산rollout note
CAP-05

포트폴리오 산출물

AX 포트폴리오는 데모 영상보다 운영 증거가 중요합니다. 어떤 업무를 선택했고, 어떻게 권한을 제한했으며, 어떤 검증으로 승인했는지 보여주는 산출물이 역량을 증명합니다.

산출물담는 내용보여주는 역량
episode log목표, 컨텍스트, 도구 사용, 결과, 실패 분류작업 추적과 회고
eval rubric성공 기준, 반례, 허용 오차, 중단 기준Evaluation
automation scorecard반복성, 검증 가능성, 실패 비용, 권한 위험자동화 판단
governance memo데이터 등급, 승인 경계, 감사 필요성Governance
rollout note대상 팀, 교육 과정, 지원 채널, 확산 조건Change Management
포트폴리오 기준

좋은 AX 포트폴리오는 “AI가 만든 결과”가 아니라 “AI가 어떤 경계 안에서 일했고 사람이 어떤 근거로 승인했는지”를 설명합니다.

AX 조직 적용 패턴

AX 도입은 도구 배포가 아니라 소프트웨어 전달 시스템에 AI 실행 계층을 추가하는 조직 변화입니다. 규모가 커질수록 하네스보다 더 중요한 것은 정책, 지식, 측정, 교육, 책임 경계입니다.

ORG-01

조직 진단

조직 진단은 “어떤 도구를 살 것인가”가 아니라 어떤 업무가 반복되고, 어떤 검증이 가능하며, 어떤 위험 때문에 자동화를 멈춰야 하는지 확인하는 단계입니다. baseline이 없으면 파일럿 성과도 비용도 판단할 수 없습니다.

진단 영역baseline 질문증거
업무 흐름반복 업무와 병목은 어디에 있는가lead time, review latency, WIP
품질결함, 재작업, 롤백은 어디서 발생하는가rework rate, escaped defect, revert rate
지식AI가 참고할 문서와 정책은 최신인가ADR, runbook, API spec, data contract
권한어떤 업무가 쓰기, 배포, 고객 데이터 접근을 요구하는가permission map, security review
책임최종 승인자와 data owner는 누구인가RACI, stakeholder map
ORG-02

파일럿 설계

파일럿은 성공 사례를 과시하는 행사가 아니라 위험을 제한한 상태에서 조직 학습을 얻는 실험입니다. 처음부터 전사 확산을 목표로 하지 말고 guarded workflow를 한두 개 만들고 실패 조건을 먼저 정합니다.

설계 요소권장 방식피해야 할 방식
범위한 팀, 한 업무, 2~4주전 직원 동시 도입
권한read-only 분석 또는 low-risk patchproduction write와 배포 자동화
검증테스트, 리뷰, 정책 체크를 pilot brief에 포함사용자 만족도만 측정
기록episode log와 audit trail을 남김좋은 프롬프트만 공유
확산 조건team standard와 platform support 필요 항목을 정리파일럿 종료 즉시 강제 확산
PILOT MATURITY PATH baseline
guarded workflow
team standard
platform support
scaled governance
ORG-03

이해관계자 지도

AX 도입은 개발팀만의 변화가 아닙니다. 데이터, 보안, 법무, 플랫폼, 운영, 현업 책임자가 각기 다른 위험과 승인 기준을 가집니다. stakeholder map은 확산 전에 반드시 작성해야 합니다.

이해관계자관심사필요한 합의
engineering lead품질, 생산성, 리뷰 부담업무별 성공 기준과 중단 조건
securitysecret, 권한, 외부 전송, tool permission금지 데이터와 승인 게이트
legal개인정보, 저작권, 규제, 책임 소재입력 가능 데이터와 기록 보존 기준
data owner데이터 의미, 접근 권한, 품질 책임데이터 사용 목적과 최소 접근 원칙
platform team공통 하네스, 모델 게이트웨이, 비용 관측지원 범위와 표준 템플릿
AI Platform Team공통 모델 gateway, prompt/tool policy, eval infrastructure팀별 golden workflow를 플랫폼 표준으로 승격
business owner업무 효과, 고객 영향, 운영 리스크확산 우선순위와 rollback 기준
ORG-04

교육 경로

교육 경로는 도구 기능 나열이 아니라 역할별 책임을 학습시키는 구조여야 합니다. Practitioner는 episode를 남기고, Workflow Owner는 평가 기준을 만들며, Platform/Adoption Lead는 governance와 지원 체계를 연결해야 합니다.

대상교육 내용수료 증거
Practitionertask spec, context package, verification note 작성자신의 업무 episode 3개
Workflow Ownerworkflow mining, eval rubric, automation scorecard팀 표준 workflow 1개
ReviewerAI 산출물 위험 분류, test gap, policy check리뷰 finding과 미검증 위험 기록
Platform/Adoption Lead권한 모델, audit, rollout, 운영 지표governance memo와 rollout note
교육 원칙

교육은 “잘 쓰는 법”보다 “어디서 멈추고 누구에게 승인받을지”를 먼저 다뤄야 합니다. 이 기준이 없으면 파일럿은 생산성 실험이 아니라 책임 공백이 됩니다.

ORG-05

확산 게이트

AX 확산은 사용량이 아니라 안정성 기준을 통과할 때 진행합니다. team standard가 생기고 platform support가 준비되며 scaled governance가 감당 가능한 속도일 때만 다음 단계로 넘어갑니다.

게이트통과 기준확산 중단 기준
품질rework와 escaped defect가 baseline 대비 악화되지 않음AI 산출물로 인한 반복 결함 증가
보안security 승인 경계와 감사 로그가 작동secret 노출, 미승인 tool call, 데이터 외부 전송
법무legal 검토가 필요한 데이터와 기록 보존 기준 확정개인정보 또는 라이선스 위험 미분류
운영지원 채널, 장애 대응, rollback 방식이 준비플랫폼 장애 시 업무 중단 또는 우회 사용 증가
역량교육 경로 이수자와 Workflow Owner가 충분함리뷰 부담이 특정 전문가에게 집중
확산 중단 기준

품질, 보안, 법무, 운영 지표 중 하나라도 baseline보다 지속적으로 나빠지면 확산을 멈추고 업무 범위와 권한을 낮춰야 합니다.

ORG-06

운영 지표

조직 단위 AX 지표는 개인 생산성보다 흐름과 품질을 봐야 합니다. 사용량이 늘어도 review burden, rework, risk event가 함께 늘면 성공이 아닙니다.

지표군예해석
Flowlead time, review latency, wait time, WIPAI가 흐름을 빠르게 하는지 병목을 옮기는지 확인
Qualityescaped defects, revert rate, rework rate, flaky test rate산출량 증가가 품질 비용을 만들지 않는지 확인
Adoptionactive users, workflow usage, acceptance rate도구 사용률보다 승인된 workflow 사용률을 봄
Riskpolicy violation, secret exposure, unapproved tool call보안과 감사 관점의 선행 지표
Capabilityepisode quality, failure taxonomy coverage, training completion팀이 AI 결과를 이해하고 검증하는 능력
Costtoken cost, tool runtime, human review cost자동화 단위가 경제적으로 타당한지 판단
