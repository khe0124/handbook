LLM 포트폴리오
프로젝트

좋은 LLM 포트폴리오는 데모 영상보다 품질을 증명하는 자료가 강합니다. API 계약, RAG 평가, 비용·지연, 실패 케이스를 함께 보여줘야 합니다.

PORT-00

weak portfolio vs strong portfolio

LLM 포트폴리오의 목표는 “모델 호출 경험”이 아니라 “불확실한 모델을 제품 시스템으로 통제한 증거”를 보여주는 것입니다.

약한 포트폴리오강한 포트폴리오
챗 UI와 API 호출만 있음problem statement, user scenario, success criteria가 있음
답변 예시 스크린샷만 있음retrieval trace, citation precision, failure case log가 있음
평가가 없음golden dataset, hit rate, MRR, groundedness, faithfulness 측정
보안 설명이 API key 숨김뿐permission filtering, prompt injection, PII 처리 설명
운영 관점이 없음latency p95, cost per request, fallback model, budget guardrail, runbook 포함

PORT-02

DEPTH

제출 가능한 포트폴리오 증거 패키지

MARKER: concept-depth · MARKER: portfolio-evidence — 강한 포트폴리오는 데모가 아니라 증거다

LLM 포트폴리오에서 챗 화면은 가장 약한 증거입니다. 화면은 잘 된 한 번의 결과만 보여주지만, 채용자나 리뷰어가 알고 싶은 것은 실패를 어떻게 발견하고 줄였는지입니다. 강한 포트폴리오는 API contract, prompt contract, retrieval trace, golden dataset, regression report, cost/latency log, security fixture, failure case log를 함께 보여줍니다. 즉 “모델을 붙였다”가 아니라 “불확실한 모델을 제품 시스템으로 통제했다”는 증거를 제시해야 합니다.

중급 개발자용 포트폴리오는 설계 결정을 방어할 수 있어야 합니다. 왜 pgvector를 썼는지, 왜 chunk size를 그렇게 잡았는지, 왜 top-k 뒤에 reranker를 넣었는지, 왜 no-answer를 허용했는지, LLM-as-judge를 어떻게 검증했는지 말할 수 있어야 합니다. README는 기능 목록이 아니라 이런 판단의 기록이어야 하며, 각 판단은 평가 결과나 실패 로그와 연결되어야 합니다.

MARKER: failure-driven — Failure case log

실패 케이스 로그는 포트폴리오에서 가장 강한 신뢰 신호 중 하나입니다. 예를 들어 “환불은 언제까지 가능해?”라는 질문에 모델이 14일이라고 답했지만 근거 문서는 7일이라고 되어 있었다면, 단순히 프롬프트를 고쳤다고 쓰면 부족합니다. 실패 입력, 실제 출력, 기대 동작, 원인 분류, 수정한 설계, 추가한 regression case를 남겨야 합니다. 이렇게 해야 리뷰어는 이 프로젝트가 데모가 아니라 개선 가능한 시스템이라는 것을 확인할 수 있습니다.

failed_input: "환불은 언제까지 가능해?"
actual_output: "14일 이내 가능합니다."
expected_behavior: "7일 이내 또는 근거 없음"
root_cause: wrong chunk ranked above refund policy
design_change: add reranker and stricter citation validation
regression_added: golden/q_refund_period_03.jsonl

면접에서는 “LLM 챗봇을 만들었다”보다 설계 긴장을 설명해야 합니다. pgvector는 기존 Postgres 권한·백업·트랜잭션과 묶기 쉬워 선택했고, hallucination은 no-answer policy와 claim-level citation으로 줄였으며, LLM-as-judge는 human-labeled holdout으로 calibration했다고 말할 수 있어야 합니다. retrieval miss가 생기면 parser, chunking, metadata filter, query rewrite, hybrid search, reranker 순서로 원인을 나눈다는 답변까지 준비되어야 합니다.

MARKER: mid-level-signal 이 포트폴리오의 합격 기준은 기능 수가 아니라 설계 결정, 실패 분석, 평가 리포트, 운영 로그를 통해 품질을 방어할 수 있는가입니다.

MARKER: failure-driven — 데모 영상은 좋지만 재현 증거가 없는 실패

LLM 포트폴리오에서 흔한 약점은 화면 녹화나 예쁜 채팅 UI만 있고, 같은 입력을 다시 넣었을 때 어떤 근거로 같은 품질을 확인할 수 있는지 남아 있지 않은 것입니다. 이 경우 면접관이 “검색 결과가 틀렸을 때 어떻게 알았나”, “prompt를 바꾸면 품질이 떨어지지 않는다는 증거가 있나”, “권한 없는 문서가 검색되면 어디서 막히나”를 물었을 때 답이 흐려집니다. 포트폴리오 제출물은 재현 가능한 request sample, retrieval trace, evaluation report, failure log, 운영 지표 캡처를 포함해야 합니다. 데모는 관심을 끌 수 있지만, 미들급 역량을 증명하는 것은 실패를 발견하고 고친 기록입니다.
