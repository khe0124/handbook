네트워크 인프라
개요

프론트엔드와 Docker 경험을 가진 개발자가 VPC, 서브넷, 라우팅, 방화벽, IPsec VPN을 실무에서 다룰 수 있도록 연결하는 입문서입니다. 목표는 네트워크 이론 암기가 아니라 VPN 인프라 세팅과 장애 대응에 필요한 판단력을 만드는 것입니다.

TARGET : Cloud VPN · NCP · KT Cloud · Private Network
STYLE : 쉬운 비유 → 실무 설정 → 고급 장애 포인트
NET-01

전체 지도

클라우드 VPN 구축은 여러 메뉴를 누르는 작업처럼 보이지만, 실제로는 다섯 가지 개념을 올바른 순서로 연결하는 일입니다.

IP 대역→ 서브넷→ 라우팅→ 방화벽→ VPN 터널
구성요소역할실무 질문
VPC클라우드 안의 격리된 사설 네트워크우리 서비스가 쓸 전체 IP 대역은 무엇인가
Subnet역할별로 나눈 IP 구간외부 공개 영역과 내부 서버 영역을 어떻게 나눌 것인가
Route Table목적지별 다음 경로고객사 대역으로 가는 패킷을 VPN Gateway로 보낼 것인가
ACG/NACL트래픽 허용 규칙어느 대역에서 어느 포트까지 허용할 것인가
IPsec VPN인터넷 위 암호화 터널양쪽 장비가 같은 암호화 파라미터로 합의했는가
NET-02

Docker 경험과 VPC

Docker Compose의 network를 이해하고 있다면 VPC의 절반은 이미 알고 있는 셈입니다. 차이는 범위와 운영 책임입니다.

DOCKER COMPOSE NETWORK networks:
  frontend:
    subnet: 172.20.1.0/24
  backend:
    subnet: 172.20.2.0/24
  db:
    subnet: 172.20.3.0/24

VPC는 위 `networks` 블록 전체에 가깝고, 각 subnet이 public/private/db 서브넷에 해당합니다. Docker는 한 서버 안의 가상 네트워크이고, VPC는 클라우드 전체를 아우르는 가상 네트워크입니다.

실무 감각

VPC 설계에서 가장 먼저 할 일은 멋진 구조를 그리는 것이 아니라 고객사와 IP 대역이 겹치지 않는지 확인하는 것입니다. VPN은 서로 다른 사설망을 라우팅으로 이어 붙이는 구조라서, 양쪽이 모두 10.0.0.0/16을 쓰면 목적지를 구분할 수 없습니다.

NET-03

VPN 구축 흐름

VPN 구축은 “터널 생성” 하나가 아니라 네트워크 설계, 양쪽 협의, 라우팅, 방화벽, 검증이 묶인 작업입니다.

VPC 설계→ VPN Gateway→ IKE 합의→ Route 추가→ 검증
단계해야 할 일실패 시 흔한 원인
1VPC·서브넷 CIDR 결정고객사 대역과 충돌
2VPN Gateway와 Customer Gateway 정보 교환Peer IP 오기입
3Phase 1/2 파라미터 합의암호화, DH group, lifetime 불일치
4Route Table과 ACG 설정터널은 UP인데 패킷이 VPN으로 가지 않음
5ping, nc, curl, tcpdump로 검증ICMP 차단을 네트워크 장애로 오판
NET-04

실무 책임 범위

VPN 프로젝트는 클라우드 담당자 혼자 끝낼 수 없습니다. 우리 쪽, 고객사 쪽, 애플리케이션 팀의 책임이 분리됩니다.

역할책임확인해야 할 산출물
우리 클라우드VPC, 서브넷, VPN Gateway, route, ACG터널 상태, 라우팅 테이블, 방화벽 규칙
고객사 네트워크Peer VPN 장비, 고객사 방화벽, 내부 라우팅Peer IP, 내부 대역, 허용 포트, return route
애플리케이션 팀서비스 포트, health endpoint, DNS, TLS접속 URL, 포트, 인증 방식, timeout 정책
고급 포인트

VPN 장애는 “우리 쪽은 정상”과 “상대 쪽은 정상”이 동시에 참일 수 있습니다. 예를 들어 요청 패킷은 고객사에 도달하지만 응답 패킷의 return route가 없으면 애플리케이션에서는 timeout으로 보입니다. 그래서 장애 대응은 항상 양방향 패킷 흐름으로 판단해야 합니다.
