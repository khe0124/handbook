네트워크 인프라
로드맵과 패킷 사고

네트워크 인프라는 콘솔 메뉴를 외우는 일이 아니라 패킷이 어디서 출발해 어떤 주소, 경로, 이름 해석, 보안 정책을 지나 목적지에 도착하는지 설명하는 일입니다. VPN도 결국 IP 대역, route table, DNS, 방화벽, 디버깅의 조합입니다.

TARGET : 프론트엔드·Docker 경험자가 클라우드/VPN 인프라를 단계적으로 이해하기 위한 기준 지도
NET-01

학습 경로

필수지식 로드맵은 기술 이름이 아니라 의존 관계를 따라야 합니다. 주소 계획을 모르면 라우팅을 설명할 수 없고, 라우팅을 모르면 방화벽이나 VPN 장애를 제대로 좁힐 수 없습니다.

단계핵심 질문문서
00. 로드맵패킷을 주소, 경로, 이름, 정책, 서비스로 나눠 설명할 수 있는가로드맵과 패킷 사고
01. 주소 계획이 서버와 네트워크는 어떤 CIDR 범위에 있는가IP·CIDR·서브넷 설계
02. 배치 구조이 리소스는 인터넷에서 직접 접근 가능한가VPC와 Public·Private 배치
03. 경로목적지 IP로 가는 다음 홉은 무엇인가Route Table·NAT·IGW
04. 이름 해석서비스 이름이 어떤 IP로 해석되는가DNS와 이름 해석
05. 보안 경계경로가 있어도 정책이 source, destination, port를 허용하는가SG·NACL·방화벽
06. 하이브리드 연결클라우드와 고객사/사내망을 어떤 gateway로 연결하는가클라우드 VPN Gateway
07. 터널 협상IPsec Phase 1/2와 traffic selector가 양쪽에서 일치하는가IPsec VPN 협상 원리
08-10. 운영L3/L4/L7 중 어디서 막혔고, 어떤 체크리스트로 재현할 것인가디버깅·트러블슈팅·체크리스트
NET-02

주소→ 경로→ 이름→ 정책→ 서비스
단계확인할 것대표 도구
주소source/destination IP, CIDR, 대역 충돌ip addr, 콘솔 서브넷
경로route table, default route, return pathtraceroute, route table

서비스프로세스, 포트, HTTP/TLS 응답curl, logs
메뉴를 읽는 방법

처음 학습할 때는 00부터 10까지 순서대로 읽고, 장애 대응 중에는 08 디버깅 도구에서 시작해 주소·경로·DNS·정책 문서로 거슬러 올라가면 됩니다. VPN 작업은 06 Gateway에서 구축 절차를 잡고 07 IPsec에서 협상 파라미터를 맞춥니다.

NET-04

VPN 구축 흐름

VPN 구축은 터널 생성이 전부가 아닙니다. 터널이 UP이어도 라우팅, 방화벽, DNS, 서비스 포트가 맞지 않으면 통신은 실패합니다.

단계확인 질문연결 문서
1. 대역 협의우리 CIDR과 고객사 CIDR이 겹치지 않는가01 IP·CIDR·서브넷 설계
2. Gateway 구성peer public IP, local/remote CIDR, tunnel 수가 맞는가06 클라우드 VPN Gateway
3. 터널 협상Phase 1/Phase 2 파라미터가 일치하는가07 IPsec VPN 협상 원리
4. 라우팅고객사 대역으로 가는 route와 return path가 있는가03 Route Table·NAT·IGW
5. DNS/방화벽이름 해석과 양쪽 source/destination/port 허용이 맞는가04 DNS, 05 보안 경계
6. 검증L3, L4, L7 순서로 어디까지 되는가08-10 운영 문서
NET-05

운영 체크포인트

Case study

VPN 터널은 UP인데 고객사 DB 접속이 안 되는 사례의 대부분은 IPsec 자체보다 route table 또는 방화벽 문제입니다. 우리 private subnet에서 고객사 CIDR로 가는 route가 없거나, 고객사 방화벽이 우리 NAT IP만 허용하고 실제 source CIDR을 허용하지 않은 경우가 많습니다. 터널 상태와 서비스 통신은 별개로 검증해야 합니다.

필수 순서roadmap
로드맵 → IP/CIDR → VPC/Subnet → route table → DNS → firewall → VPN Gateway → IPsec → debugging 순서로 학습합니다.
핵심 질문packet path
이 패킷의 source, destination, next hop, policy, response path를 설명할 수 있는가.
운영 진입점ops
장애 대응은 디버깅 도구에서 시작하고, 원인 후보에 따라 주소·경로·DNS·정책 문서로 되돌아갑니다.
