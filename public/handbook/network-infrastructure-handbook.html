네트워크 인프라
필수지식 로드맵

네트워크 인프라는 콘솔 메뉴를 외우는 일이 아니라 패킷이 어디서 출발해 어떤 경로와 보안 정책을 지나 목적지에 도착하는지 설명하는 일입니다. VPN도 결국 IP 대역, route table, 방화벽, DNS, 디버깅의 조합입니다.

TARGET : 프론트엔드·Docker 경험자가 클라우드/VPN 인프라를 이해하기 위한 필수 순서
NET-01

필수지식 로드맵

학습 순서는 기술 이름이 아니라 의존 관계를 따라야 합니다. IP를 모르면 CIDR을 못 잡고, CIDR을 모르면 라우팅과 VPN 협의를 못 합니다.

순서필수 질문다음 문서
1. IP·CIDR·서브넷이 서버와 네트워크는 어떤 주소 범위에 있는가IP 주소·CIDR·서브넷
2. VPC와 Public/Private이 리소스는 인터넷에서 직접 접근 가능한가VPC와 Public·Private 구조
3. 라우팅·NAT·IGW목적지 IP로 가는 다음 홉은 무엇인가라우팅·NAT·Internet Gateway
4. 방화벽경로가 있어도 정책이 허용하는가방화벽·Security Group·NACL
5. DNS이름이 어떤 IP로 해석되는가DNS와 이름 해석
6. VPN사설 대역 간 트래픽을 암호화 터널로 보낼 수 있는가IPsec VPN 원리
7. 운영장애 시 L3/L4/L7 중 어디서 막혔는가네트워크 디버깅·트러블슈팅

NET-04

VPN 구축 지도

VPN 구축은 터널 생성이 전부가 아닙니다. 터널이 UP이어도 라우팅, 방화벽, DNS, 서비스 포트가 맞지 않으면 통신은 실패합니다.

작업확인 질문실패 시 흔한 원인
대역 협의우리 CIDR과 고객사 CIDR이 겹치지 않는가양쪽 모두 10.0.0.0/16 사용
터널 협상Phase 1/Phase 2 파라미터가 일치하는가PSK, DH group, lifetime 불일치
라우팅고객사 대역으로 가는 route가 VPN을 향하는가route table 누락, return path 누락
방화벽출발지/목적지/포트가 양쪽에서 허용되는가우리 ACG 또는 고객사 방화벽 차단
검증L3, L4, L7 순서로 어디까지 되는가ping만 보고 서비스 장애로 오판
NET-05

실무 체크포인트

Case study

VPN 터널은 UP인데 고객사 DB 접속이 안 되는 사례의 대부분은 IPsec 자체보다 route table 또는 방화벽 문제입니다. 우리 private subnet에서 고객사 CIDR로 가는 route가 없거나, 고객사 방화벽이 우리 NAT IP만 허용하고 실제 source CIDR을 허용하지 않은 경우가 많습니다. 터널 상태와 서비스 통신은 별개로 검증해야 합니다.

필수 순서roadmap
IP/CIDR → VPC/Subnet → route table → firewall → DNS → VPN → debugging 순서로 학습합니다.
