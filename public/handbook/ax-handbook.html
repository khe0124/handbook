AX 개요

AX는 AI 도구 사용법이 아니라 AI가 개발 작업 안에서 안전하게 실행되고, 검증되고, 기록되도록 만드는 운영 역량입니다. 핵심은 더 긴 프롬프트가 아니라 더 좋은 작업 시스템입니다.

STANCE : 생성보다 검증 · 프롬프트보다 컨텍스트 · 자동화보다 책임 경계 · COMPILED 2026-06
AX-01

AX의 정의

AX는 AI Experience, Agent Experience, AI-augmented eXecution으로 해석할 수 있습니다. 이름보다 중요한 것은 관점입니다. AI를 채팅창으로 보면 프롬프트 품질만 보이지만, 실무에서는 명세, 컨텍스트, 도구, 권한, 검증, 기록이 함께 결과를 만듭니다.

관점낮은 수준의 AI 활용AX 관점
목표코드를 빨리 생성검증 가능한 변경을 반복적으로 생산
입력자연어 요청목표, 비목표, 컨텍스트, 제약, 성공 기준
실행단발 프롬프트Plan → Act → Verify → Reflect 루프
검증AI의 자기 요약테스트, 리뷰, trace, episode package
책임AI가 해줬다개발자가 하네스와 승인 경계를 운영한다
핵심 문장

AX의 목표는 AI에게 더 많은 일을 시키는 것이 아닙니다. 작업을 더 작게 나누고, 검증을 더 명확히 하며, 실패했을 때 더 빨리 멈추는 것입니다.

AX-02

Model·Harness·Environment

AI 개발 성과는 모델 단독 능력이 아니라 모델, 하네스, 환경의 합성 결과입니다. 같은 모델도 어떤 작업 명세와 도구, 어떤 검증 루프 위에 놓이느냐에 따라 완전히 다른 개발자가 됩니다.

구성요소의미실무에서 보는 것
Model추론·생성 능력코드 이해, 요약, 계획, 패턴 인식, 언어 생성
Harness모델을 감싸는 작업 실행 계층작업 명세, 컨텍스트 선택, 도구 호출, 권한, 검증, 관측
Environment모델이 작동하는 실제 세계코드베이스, 테스트, CI, DB, 배포 환경, 조직 규칙, 보안 경계
AX SYSTEM MODEL task spec + context + tools + permissions
→ model
→ patch / analysis / decision
→ verification + review + audit trail
→ accepted change or classified failure
AX-03

고급 개발자의 역할

고급 개발자는 AI에게 일을 많이 시키는 사람이 아니라, AI가 하는 일을 제한된 위험 안에서 반복 가능한 산출물로 수렴시키는 사람입니다.

역할하는 일산출물
작업 설계자목표, 범위, 금지사항, 성공 기준 정의task spec, acceptance criteria
컨텍스트 큐레이터관련 파일·문서·규칙을 선별하고 출처 기록context package
권한 관리자read/write/network/exec/deploy 경계 설정permission policy
검증 책임자테스트, 리뷰, 수동 검증, 보안 체크 연결verification report
실패 분석자모델 오류, 컨텍스트 누락, 도구 실패, 테스트 부재를 분리failure taxonomy
AX-04

실패 모드

실패실제 원인예방
그럴듯하지만 틀린 구현도메인 규칙과 반례가 컨텍스트에 없음상태 전이, 예외, 수용 기준을 함께 제공
지나친 리팩터링작업 경계가 불명확함수정 범위와 건드리지 않을 것을 명시
False green테스트는 통과하지만 핵심 케이스가 없음AI가 만든 테스트와 독립 검증을 분리
권한 오남용도구와 실행 권한이 과함destructive action, network, deploy에 승인 게이트
감사 불가AI 개입 내역과 검증 결과가 남지 않음episode package와 audit log 저장
운영 기준

“AI가 틀렸다”는 분석이 아닙니다. 작업 명세가 모호했는지, 컨텍스트가 부족했는지, 도구가 실패했는지, 검증이 약했는지까지 분리해야 다음 루프가 좋아집니다.

AX-05

업무 자동화 관점

AX를 실무에 적용한다는 것은 업무를 “AI에게 질문하기 좋은 형태”가 아니라 “AI가 제한된 권한 안에서 실행하고 검증할 수 있는 형태”로 바꾸는 것입니다.

질문낮은 수준의 접근AX 자동화 접근
무엇을 자동화할까귀찮은 업무를 AI에게 맡김반복성, 검증 가능성, 실패 비용, 권한 위험으로 후보 선정
어디까지 맡길까AI가 할 수 있으면 맡김read-only, proposal, patch, PR, deploy 단계별 Autonomy ladder 운영
어떻게 안전하게 할까사람이 마지막에 대충 확인tool permission, policy gate, independent review, episode audit 적용
효과를 어떻게 볼까시간이 줄었는지 감으로 판단lead time, review latency, rework rate, defect escape, cost per episode 측정
어떻게 확장할까잘 된 prompt를 공유업무별 golden workflow, context package, eval set, rollout playbook으로 확산
핵심 전환

AX 엔지니어링은 prompt engineering을 포함하지만 거기에 머물지 않습니다. 실제 성과는 업무 분해, 검증 가능한 산출물, 권한 경계, 조직 내 확산 구조가 함께 있을 때 나옵니다.
