AX 개요

AX는 프롬프트 작성 기술이 아니라 업무를 검증 가능한 실행 시스템으로 바꾸는 운영 역량입니다. 핵심은 더 긴 요청문이 아니라 더 좋은 작업 명세, 컨텍스트, 권한, 검증 루프입니다.

STANCE : 생성보다 검증 · 프롬프트보다 컨텍스트 · 자동화보다 책임 경계 · COMPILED 2026-06
AX-01

AX의 정의

AX는 AI Experience, Agent Experience, AI-augmented eXecution을 모두 포괄하지만, 실무에서의 의미는 분명합니다. AX는 업무를 AI에게 말로 맡기는 방식이 아니라 사람이 책임지는 실행 체계 안에서 AI가 작업하고, 검증받고, 기록되도록 만드는 운영 방식입니다.

관점낮은 수준의 AI 활용AX 관점
목표코드를 빨리 생성검증 가능한 변경을 반복적으로 생산
입력자연어 요청목표, 비목표, 컨텍스트, 제약, 성공 기준
실행단발 프롬프트Plan → Act → Verify → Reflect 루프
검증AI의 자기 요약테스트, 리뷰, trace, verification report
책임AI가 해줬다사람이 하네스와 승인 경계를 운영한다
핵심 문장

AX의 목표는 AI에게 더 많은 일을 시키는 것이 아닙니다. 작업을 더 작게 나누고, 검증을 더 명확히 하며, 실패했을 때 더 빨리 멈추는 것입니다.

AX-02

AX 시스템 모델

AI 개발 성과는 모델 단독 능력이 아니라 명세, 컨텍스트, 권한, 도구, 검증, 기록이 결합된 시스템의 결과입니다. 같은 모델도 어떤 업무 구조 위에 놓이느냐에 따라 위험한 자동완성 도구가 되기도 하고, 통제 가능한 실행 파트너가 되기도 합니다.

기존의 Model·Harness·Environment 관점은 그대로 유지하되, 조직 도입에서는 이를 task spec, context package, tool permission, verification report, audit trail로 운영 가능한 산출물까지 내려야 합니다.

AX-03

도입 학습 지도

AX 도입 학습 지도는 개인이 프롬프트를 더 잘 쓰게 만드는 과정이 아니라 조직이 반복 업무를 발견하고, 작은 파일럿으로 검증하고, 표준 산출물로 남기는 과정입니다.

단계학습 질문산출물
조직 진단어떤 업무가 반복되고 검증 가능하며 실패 비용이 낮은가workflow inventory, risk baseline
파일럿작은 팀에서 read-only 또는 low-risk 업무를 안전하게 실험할 수 있는가pilot brief, guarded workflow
표준 산출물성공한 업무를 다른 팀이 재사용할 수 있는 형태로 남겼는가task spec, context package, eval checklist
확산교육, 권한, 지표, governance가 함께 준비되었는가rollout note, 운영 지표, audit rule
학습 원칙

도입은 “AI를 써보자”에서 시작하지 않습니다. 조직 진단으로 후보 업무를 고르고, 파일럿으로 실패 비용을 제한하며, 표준 산출물로 재현 가능한 학습을 남깁니다.

AX-04

전문가의 책임 경계

전문가는 AI보다 더 많은 문장을 쓰는 사람이 아니라 AI가 실행할 수 있는 경계와 멈춰야 할 조건을 정하는 사람입니다. AX 책임 원칙은 명확합니다. AI는 작업자가 될 수 있지만 책임자는 될 수 없습니다.

책임 영역AI가 할 수 있는 일전문가가 남겨야 하는 일
판단후보안, 비교표, 위험 신호 제안최종 의사결정과 승인 기준 설정
실행제한된 권한 안에서 diff, 문서, 분석 작성권한 범위와 destructive action 승인
검증테스트 실행, 로그 요약, 실패 후보 분류검증 충분성 판단과 미검증 위험 공개
책임작업 흔적과 근거 제공품질, 보안, 법무, 고객 영향에 대한 책임
AX 책임 원칙

AI는 작업자가 될 수 있지만 책임자는 될 수 없습니다. 책임은 작업을 명세화하고, 권한을 부여하고, 검증 결과를 승인한 사람과 조직에 남습니다.

AX-05

표준 산출물

AX는 말로 전파되지 않습니다. 반복 가능한 산출물로 전파됩니다. 산출물은 AI가 다음 episode에서 같은 품질로 일하도록 만드는 최소 운영 단위입니다.

산출물내용사용 시점
task spec목표, 범위, 비목표, 성공 기준, 중단 조건작업 시작 전
context package관련 파일, 정책, 데이터 계약, 예외 사례모델 실행 전
tool permission허용 도구, 금지 도구, 승인 필요 작업자동화 수준 결정 시
verification report실행한 검증, 실패한 검증, 남은 위험리뷰와 승인 전
audit trail요청, 판단, 도구 호출, 변경, 승인 내역사후 분석과 governance review
AX-06

도입 체크포인트

AX 도입은 도구 설치 여부가 아니라 업무가 검증 가능한 실행 시스템으로 바뀌었는지로 판단합니다.

ADOPTION CHECKPOINTS 1. 반복 업무가 업무 단위와 episode 단위로 분해되었는가
2. 각 업무에 task spec과 context package가 있는가
3. tool permission과 승인 게이트가 위험도에 맞는가
4. verification report가 리뷰의 입력으로 쓰이는가
5. audit trail이 사후 학습과 책임 추적에 충분한가
6. 표준 산출물이 교육과 확산에 재사용되는가
운영 기준

좋은 AX 도입은 더 많은 AI 사용량을 만드는 것이 아니라, 더 적은 혼란으로 더 많은 검증 가능한 업무 결과를 만드는 것입니다.
