AX 개요

AX는 AI를 잘 쓰는 기술이 아니라, 업무를 AI가 수행해도 사람이 검증하고 책임질 수 있는 구조로 바꾸는 방식입니다. 핵심은 긴 프롬프트가 아니라 명세, 컨텍스트, 권한, 검증, 기록입니다.

STANCE : SPEC FIRST · CONTEXT CONTROL · PERMISSION BOUNDARY · VERIFICATION EVIDENCE
AX-01

한 문장 정의

AX는 사람이 책임지는 업무 체계 안에서 AI가 작업하고, 검증받고, 기록되도록 만드는 운영 방식입니다. AI Experience, Agent Experience, AI-augmented eXecution이라는 표현보다 실무에서는 이 한 문장이 더 중요합니다.

구분AI 활용AX
목표결과를 빠르게 얻는다검증 가능한 결과를 반복해서 만든다
입력자연어 요청task spec, context package, 제약, 성공 기준
실행대화 또는 단발 생성Plan → Act → Verify → Reflect 루프
검증결과를 읽고 판단테스트, 리뷰, 반례, verification report
책임사용자 개인의 감권한 경계와 승인 기준을 운영하는 사람과 조직
핵심

AX의 품질은 모델 답변의 유창함이 아니라 업무가 명세화되고, 권한이 제한되고, 완료 증거가 남는지로 판단합니다.

AX-02

실행 모델

AX의 기본 흐름은 Spec → Context → Permission → Execution → Verification → Audit입니다. 기존의 Model·Harness·Environment 관점을 실제 운영에서는 아래 여섯 산출물로 내려야 합니다.

AX EXECUTION MODEL 1. Spec: 목표, 비목표, 성공 기준, 중단 조건을 쓴다
2. Context: 필요한 파일, 정책, 로그, 반례만 선별한다
3. Permission: 읽기, 쓰기, 실행, 네트워크, 배포 권한을 제한한다
4. Execution: 하네스와 도구 안에서 작은 단위로 실행한다
5. Verification: 테스트, 리뷰, 수동 확인, 남은 위험을 기록한다
6. Audit: 요청, 판단, 도구 호출, 변경, 승인 내역을 남긴다
단계좋은 상태실패 신호
Spec완료 여부를 독립적으로 판정할 수 있음“개선해줘”처럼 열린 요청만 있음
Context필수 근거와 제외 근거가 분리됨파일을 많이 넣었지만 핵심 규칙이 없음
Permission위험도에 맞게 tool permission이 제한됨쓰기, 배포, 데이터 접근이 암묵적으로 열림
Verification완료 증거와 미검증 위험이 함께 남음AI의 자기 요약만 있음
Auditepisode를 나중에 재구성할 수 있음왜 바꿨고 무엇을 믿었는지 추적 불가
AX-03

적용 판단 기준

AX는 모든 업무에 적용하는 구호가 아닙니다. 먼저 Workflow Mining으로 반복 episode를 모으고, 검증 가능성과 실패 비용을 기준으로 적용 후보를 고릅니다.

AX 적용 후보판단 질문
반복된다최근 2~4주 안에 같은 패턴의 episode가 여러 번 있었는가
입출력이 안정적이다업무 시작 자료와 결과물이 일정한 형식으로 정리되는가
검증 가능하다테스트, diff, 체크리스트, 리뷰 기준으로 성공을 판정할 수 있는가
실패 비용이 낮거나 되돌릴 수 있다잘못되어도 리뷰나 롤백으로 피해를 제한할 수 있는가
컨텍스트를 모을 수 있다필요한 코드, 정책, 로그, 도메인 규칙을 제공할 수 있는가
AX 적용 보류이유
판단 기준이 불명확하다좋은 결과를 판정할 사람이 없으면 자동화가 아니라 위임이 된다
고객, 운영, 보안 영향이 크다먼저 read-only 분석과 reviewer handoff로 제한해야 한다
검증 경로가 없다완료 증거가 없으면 빠른 생성이 빠른 rework가 된다
권한이 넓다DB write, 배포, 민감 데이터 접근은 기본 자동화 대상이 아니다
예외가 대부분이다업무 정의와 정책을 먼저 정리해야 한다
판단 원칙

“AI가 할 수 있는가”보다 “사람이 검증하고 책임질 수 있는가”가 먼저입니다. 검증 가능성이 낮으면 자동 실행이 아니라 제안과 분석까지만 허용합니다.

AX-04

최소 산출물

AX는 말로 전파되지 않고 표준 산출물로 전파됩니다. 처음부터 큰 플랫폼을 만들 필요는 없지만, 최소한 task spec, context package, verification report는 남겨야 합니다.

TASK SPEC MINI Goal: 이번 episode의 완료 상태
Non-goals: 건드리지 않을 범위
Success criteria: 테스트, 리뷰, 동작, 정책 기준
Permission boundary: 읽기, 쓰기, 실행, 네트워크, 배포 가능 범위
Stop condition: missing context, unsafe permission, 반복 실패, 요구 불명확
CONTEXT PACKAGE MINI Relevant sources: 파일, 티켓, 로그, 정책, API 계약
Decision rules: 판단 기준, 예외, 반례
Trust level: 내부 근거와 untrusted content 구분
Open gaps: 부족한 정보와 그로 인한 risk
VERIFICATION REPORT MINI Executed checks: 테스트, 빌드, lint, 수동 시나리오
Review evidence: diff, 근거, 반례, 승인 기준
Residual risk: 검증하지 못한 항목과 이유
Audit link: 요청, 도구 호출, 변경, 승인 기록
AX-05

책임 경계

AI는 작업자가 될 수 있지만 책임자는 될 수 없습니다. AX 전문가의 핵심 역할은 AI에게 더 많은 문장을 쓰는 것이 아니라 실행 권한과 중단 조건을 정하는 것입니다.

책임 영역AI가 할 수 있는 일전문가가 남겨야 하는 일
판단후보안, 비교표, 위험 신호 제안최종 의사결정과 승인 기준 설정
실행제한된 권한 안에서 diff, 문서, 분석 작성권한 범위와 destructive action 승인
검증테스트 실행, 로그 요약, 실패 후보 분류검증 충분성 판단과 미검증 위험 공개
책임작업 흔적과 근거 제공품질, 보안, 법무, 고객 영향에 대한 책임
AX 책임 원칙

AI는 작업자가 될 수 있지만 책임자는 될 수 없습니다. 책임은 작업을 명세화하고, 권한을 부여하고, 검증 결과를 승인한 사람과 조직에 남습니다.

AX-06

첫 도입 순서

AX 도입은 도구 설치에서 시작하지 않습니다. 조직 진단으로 후보 업무를 고르고, 작은 파일럿에서 실패 비용을 제한한 뒤, 검증된 표준 산출물만 확산합니다.

FIRST AX ROLLOUT 1. 조직 진단: 반복 업무, 실패 비용, 검증 가능성을 조사한다
2. 후보 선정: 적용 후보와 보류 업무를 분리한다
3. 파일럿 설계: read-only 분석 또는 작은 patch부터 시작한다
4. 산출물 고정: task spec, context package, verification report 형식을 통일한다
5. 권한 제한: tool permission과 approval gate를 업무 위험도에 맞춘다
6. 운영 지표: lead time, review latency, rework, defect escape, cost per episode를 본다
7. 확산 판단: 표준 산출물이 교육과 재사용에 충분할 때만 넓힌다
운영 기준

좋은 AX 도입은 AI 사용량을 늘리는 일이 아닙니다. 같은 업무를 더 작은 위험, 더 명확한 증거, 더 낮은 rework로 반복하게 만드는 일입니다.
