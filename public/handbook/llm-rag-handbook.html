RAG·임베딩·벡터DB

RAG는 모델에게 모든 문서를 넣는 방식이 아니라, 질문에 필요한 근거 조각을 찾아 모델 입력으로 주입하는 검색-생성 시스템입니다.

RAG-01

EMBEDDING MODEL CONTRACT

임베딩은 텍스트 의미를 벡터로 바꾼 표현입니다. 운영에서는 “어떤 모델로, 어떤 차원으로, 어떤 거리 함수로, 언제 재색인할지”가 계약이어야 합니다.

RAG-03

PGVECTOR CHROMA DECISION

벡터DB 선택은 성능만의 문제가 아닙니다. 권한, 백업, 운영자 경험, migration, index tuning, 비용까지 같이 봐야 합니다.

선택장점비용추천 상황
pgvectorPostgreSQL과 권한·백업·트랜잭션 통합대규모 ANN 튜닝과 vacuum/index 운영 필요이미 Postgres를 쓰고 tenant/ACL을 SQL로 처리할 때
Chroma로컬 실험과 Python 생태계 접근 쉬움운영 DB 통합과 권한 정책은 별도 설계빠른 PoC와 notebook 실험
Managed vector DB스케일, 관리형 index, 운영 기능비용, vendor lock-in, 데이터 이전대량 문서와 운영 SLA가 중요할 때
운영 질문확인할 것

재색인embedding_model 변경 시 reindex job과 rollback이 있는가

RAG-05

Permission Filter와 Context Policy

RAG 보안의 기본은 “문서가 존재하는가”가 아니라 “현재 actor가 이 chunk를 볼 권한이 있는가”입니다. 권한 없는 chunk는 prompt에 들어가기 전에 제거되어야 합니다.

RAG ACCESS CONTROL RULE actor scope 생성 → tenant_id/document_acl_version으로 pre-filter → retrieval → rerank → chunk provenance 기록 → answer citation 검증
위험증상방어
permission leak다른 tenant 문서가 답변에 섞임pre-filter, row-level security, tenant isolation test
stale permission권한이 회수된 문서가 검색됨document_acl_version과 cache invalidation

DEPTH

RAG를 검색·근거·권한 시스템으로 다루기

MARKER: concept-depth — RAG는 벡터 검색이 아니라 근거 관리 시스템이다

RAG의 핵심은 “비슷한 문서를 찾는다”가 아니라 “현재 질문에 답할 수 있는 근거를, 현재 사용자가 볼 수 있는 범위 안에서, 답변에 인용 가능한 형태로 공급한다”입니다. 같은 모델과 같은 프롬프트를 써도 정답 chunk가 context에 없으면 답변은 우연이 됩니다. 그래서 RAG 디버깅은 생성 답변을 먼저 고치는 것이 아니라, 정답 근거가 검색 후보에 있었는지, rerank 후에도 살아남았는지, context packing에서 빠지지 않았는지부터 확인해야 합니다.

Chunking은 이 시스템의 첫 번째 설계 결정입니다. 작은 chunk는 검색 precision과 citation에는 유리하지만, 법률 조항의 조건문이나 정책의 예외 조항처럼 주변 문맥이 필요한 경우 답변을 잘못 만들 수 있습니다. 큰 chunk는 문맥을 보존하지만 top-k에 들어온 순간 token cost와 noise를 크게 늘립니다. overlap은 경계 손실을 줄이지만 같은 근거가 여러 번 들어와 모델이 특정 내용을 과신하게 만들 수 있습니다. parent-child chunking은 작은 child로 찾고 큰 parent로 읽게 만드는 절충안입니다.

MARKER: failure-driven — Embedding contract가 깨지는 순간

임베딩 모델을 바꾸는 것은 단순 성능 개선이 아니라 검색 공간을 바꾸는 일입니다. dimension이 달라지면 DB column과 index가 맞지 않고, normalization 정책이 달라지면 cosine score 분포가 바뀌며, parser나 chunker가 바뀌면 같은 문서도 다른 벡터 집합이 됩니다. 이 상태에서 옛 벡터와 새 벡터를 섞으면 recall@k가 떨어져도 원인을 찾기 어렵습니다. embedding_model, parser_version, chunker_version, content_hash를 함께 저장하고, 변경 시 reindex와 rollback 경로를 갖춰야 합니다.

Hybrid retrieval은 성능 욕심이 아니라 실패 모드 보완입니다. vector search는 “환불 가능한 기간”과 “취소 기한”처럼 표현이 다른 의미를 잘 찾지만, 정책 번호, SKU, 날짜, 에러 코드 같은 식별자에는 약할 수 있습니다. BM25는 이런 정확 키워드에 강하지만 표현이 다르면 놓칩니다. RRF는 서로 다른 ranking을 결합해 한쪽 검색 방식의 약점을 줄이지만, 후보 자체가 나쁘면 reranker도 구할 수 없습니다.

DEBUGGING FLOW — 정답 chunk가 top-20에도 없을 때

정답 chunk가 top-20에도 없다면 프롬프트를 고치기 전에 검색 계층을 분해해야 합니다. 먼저 parser가 원문을 누락했는지 확인하고, 다음으로 chunk boundary가 질문의 의미 단위를 잘랐는지 봅니다. 그 다음 metadata/ACL pre-filter가 정답 문서를 과도하게 제외했는지, embedding_model과 index_version이 섞였는지, query rewrite가 사용자의 의도를 바꿨는지 확인합니다. 마지막으로 식별자나 날짜가 중요한 질문이라면 hybrid search를 추가하고, 후보는 있는데 순위가 낮다면 reranker 실험으로 넘어갑니다.

MARKER: failure-driven — 권한 필터를 생성 단계에서만 처리하는 실패

RAG에서 “모델에게 민감 정보는 답하지 말라고 지시한다”는 방식은 보안 경계가 아닙니다. 이미 금지된 chunk가 prompt에 들어갔다면 모델이 답변을 거절하더라도 데이터는 모델 context와 log, trace, cache 경로에 노출된 것입니다. 이 실패는 데모에서는 잘 보이지 않지만 멀티테넌트 서비스에서는 치명적입니다. 권한 필터는 retrieval candidate를 prompt에 넣기 전에 적용되어야 하고, trace에는 어떤 chunk가 왜 제외되었는지 남아야 합니다. RAG 품질을 recall만으로 보면 이런 사고를 놓치므로, permission-filtered recall과 forbidden-context exposure를 별도 지표로 두어야 합니다.
