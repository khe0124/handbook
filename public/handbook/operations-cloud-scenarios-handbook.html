CLD-00

웹 서비스 기본 구성

AWS와 Azure 서비스는 이름을 1:1로 외우면 안 된다. CloudFront와 Azure Front Door는 edge 계층에서 캐시, WAF, 전역 라우팅을 담당하고, Application Load Balancer와 Application Gateway는 regional L7 진입점 역할을 한다. 런타임, 데이터, identity, observability는 각 클라우드의 책임 경계와 장애 신호를 따로 확인한다.

운영 질문확인할 증거판단 기준
AWS와 Azure 구성 요소를 서비스명이 아니라 책임 경계로 설명할 수 있는가?Route 53/Azure DNS, CloudFront/Azure Front Door, ALB/Application Gateway, ECS/EKS/AKS/App Service, RDS/Azure SQLedge, regional ingress, runtime, data, identity, observability를 분리해 그릴 수 있어야 한다.
웹 서비스 기본 구성을 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?CloudFront/Front Door request log, ALB/App Gateway target health, runtime health, DB metricsedge 403, regional 5xx, runtime failure, DB pressure를 같은 오류로 묶지 않는다.
웹 서비스 기본 구성 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?CloudFront 또는 Front Door 403 증가와 origin 5xx가 동시에 관찰됨WAF/edge 차단과 origin 연결 실패를 분리해 완화책을 선택한다.
CLD-01

사설 네트워크와 데이터 경계

AWS는 VPC, public/private subnet, NAT Gateway, Security Group, VPC Endpoint, AWS PrivateLink로 경계를 만든다. Azure는 VNet, subnet, NAT Gateway, Network Security Group, Private Endpoint, Private Link, Private DNS Zone을 조합한다. 핵심은 DB와 관리형 서비스가 인터넷을 거치지 않게 하는 것이다.

운영 질문확인할 증거판단 기준
DB와 관리형 서비스가 인터넷을 거치지 않는다는 것을 어떤 증거로 확인하는가?route table, private endpoint, Private DNS, Security Group/NSG, flow log, public access disabledDNS가 private endpoint를 가리키고 route/policy/flow log가 사설 경로를 증명해야 한다.
사설 네트워크와 데이터 경계를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?route table readback, private DNS resolution, flow log ACCEPT/REJECT, DB public access settingprivate endpoint를 만들었어도 DNS가 public endpoint를 가리키면 사설 경로가 아니다.
사설 네트워크와 데이터 경계 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?private subnet 앱이 DB 또는 storage 접근 실패, 또는 public egress 비용 급증DNS override, endpoint policy, route table, NSG/Security Group, egress path를 순서대로 본다.
예시 신호관찰값해석
정상Private DNS가 사설 IP를 반환하고 flow log가 private endpoint 경로를 기록인터넷을 거치지 않는 경로로 판단한다.
비정상Private Endpoint는 있지만 DNS가 public endpoint를 반환DNS override 또는 resolver rule을 먼저 수정한다.
CLD-02

Secret과 배포

AWS에서는 IAM role, Secrets Manager, Parameter Store, ECR, CodePipeline 또는 GitHub Actions를 사용한다. Azure에서는 Microsoft Entra ID, Managed Identity, Key Vault, Azure Container Registry, Azure DevOps 또는 GitHub Actions를 연결한다. secret은 build artifact에 넣지 않고 runtime 주입과 rotation 절차를 둔다.

운영 질문확인할 증거판단 기준
secret과 배포 권한이 artifact와 runtime 경계에서 분리되어 있는가?IAM/Managed Identity, Secrets Manager/Key Vault, runtime injection, rotation record, deploy approvalsecret이 build artifact, image, frontend bundle, log에 남으면 운영 경계 실패로 본다.
Secret과 배포를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?role assignment, secret version, deployment environment, audit log, image/config diff배포 주체와 런타임 주체의 권한이 분리되어야 영향 범위를 줄일 수 있다.
Secret과 배포 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?secret rotation 직후 app 인증 실패 또는 배포 pipeline 권한 오류revoke, 재발급, runtime 재시작, 영향 범위 확인을 절차로 나눈다.
CLD-03

장애와 비용 사례

CloudFront나 Azure Front Door 캐시 설정 오류는 stale content를 만들고, NAT Gateway 비용 폭증은 private subnet egress 설계가 잘못됐다는 신호일 수 있다. RDS/Azure SQL connection pool 고갈, ALB/Application Gateway health probe 실패, AKS/EKS rollout 실패는 지표와 로그를 같은 릴리스 단위로 묶어야 빠르게 좁힌다.

운영 질문확인할 증거판단 기준
장애 신호와 비용 신호를 같은 운영 timeline에서 설명할 수 있는가?WAF rule change, cache policy, target health, DB pressure, NAT bytes, retry storm, log ingestion비용 급증도 retry, egress route, cross-AZ traffic 같은 운영 장애 신호일 수 있다.
장애와 비용 사례를 원인 후보에서 배제하려면 어떤 확인 결과가 필요한가?edge log, target health, DB metric, NAT/egress cost, retry rate, deployment marker최근 변경과 비용/장애 신호를 같은 시간축에 놓고 원인 후보를 줄인다.
장애와 비용 사례 장애에서 임시 완화와 영구 수정은 어떻게 나누는가?NAT 비용 4배 증가와 외부 API timeout/retry 증가retry 제한, egress allowlist, route 수정, cost anomaly alert를 함께 남긴다.

실무 체크리스트

AWS·Azure 실전 시나리오 체크리스트는 웹 서비스 기본 구성, 사설 네트워크와 데이터 경계, Secret과 배포, 장애와 비용 사례의 변경 위험을 사전에 드러내기 위한 검토표다. 각 항목은 리뷰, 배포 승인, 운영 인수에서 같은 증거로 다시 확인되어야 한다.

  • AWS와 Azure에서 같은 역할을 하는 서비스를 1:1로 외우지 말고 edge, network, runtime, data, identity, observability 책임으로 매핑한다.
  • Route 53/Azure DNS, CloudFront/Azure Front Door, ALB/Application Gateway, ECS Fargate·EKS/AKS·App Service, RDS/Azure SQL 조합을 요청 경로로 그린다.
  • Secrets Manager 또는 Key Vault, IAM role 또는 Managed Identity, PrivateLink 또는 Private Endpoint를 보안 경계에 포함한다.
  • CloudWatch 또는 Azure Monitor에서 RED 지표, DB connection, health probe, deployment version을 같은 대시보드에 둔다.

CASE

장애/운영 시나리오

B2B SaaS 관리자 서비스를 클라우드에 배포한다. AWS안은 Route 53 → CloudFront/WAF → Application Load Balancer → ECS Fargate → RDS Multi-AZ와 ElastiCache로 구성하고, secret은 Secrets Manager, 이미지는 ECR, 로그와 알림은 CloudWatch로 둔다. Azure안은 Azure DNS → Azure Front Door/WAF → Application Gateway → AKS 또는 App Service → Azure SQL zone redundant 구성과 Azure Cache for Redis로 구성하고, secret은 Key Vault, 이미지는 ACR, 로그와 알림은 Azure Monitor로 둔다.

Triage frame

AWS·Azure 실전 시나리오 장애는 증상 → 영향 범위 → 최근 변경 → Edge / Ingress 증거 → 완화책 → 복구 검증 → 재발 방지 순서로 기록한다. 이 흐름이 있어야 담당자가 바뀌어도 같은 결론에 도달한다.

처음 10분다음 30분종료 조건
Route 53/Azure DNS, CloudFront/Front Door, WAF, ALB/App Gateway 기준으로 영향 범위를 고정한다.Route 53 answer, CloudFront cache/WAF log, ALB target health, ECS task/RDS metrics 항목에서 증거를 모아 완화책을 실행한다.서비스명 암기가 아니라 책임 경계로 정리한다.
Incident packet기록할 내용판정 기준
증상edge 403 증가 또는 NAT 비용 4배 증가WAF rule, origin policy, egress route, retry storm을 본다.
완화WAF rule rollback, cache policy rollback, retry 제한, egress allowlist보안과 가용성의 trade-off를 기록한다.
사후cloud-specific runbook과 cost anomaly alert 추가서비스명 암기가 아니라 책임 경계로 정리한다.
RISK

자주 틀리는 판단

AWS·Azure 실전 시나리오에서 자주 틀리는 판단은 웹 서비스 기본 구성와 장애와 비용 사례의 책임 경계를 흐리는 데서 시작된다. 아래 신호가 보이면 설계, runbook, 답변 구조를 다시 점검한다.

  • AWS 서비스명과 Azure 서비스명을 단순 번역해서 같은 장애 특성을 가진다고 가정한다.
  • CloudFront나 Azure Front Door 뒤에 origin timeout, cache policy, WAF rule 변경 이력을 남기지 않는다.
  • NAT Gateway 또는 egress 비용을 보지 않아 private subnet의 외부 호출 비용이 폭증한다.
  • RDS/Azure SQL 백업은 켜져 있지만 restore drill과 connection pool 장애 대응이 없다.

면접 답변 템플릿

실제 클라우드 구성 질문에는 먼저 역할을 나눠 답합니다. DNS는 Route 53 또는 Azure DNS, edge는 CloudFront 또는 Azure Front Door, L7 진입점은 Application Load Balancer 또는 Application Gateway, 런타임은 ECS Fargate/EKS 또는 AKS/App Service, 데이터는 RDS 또는 Azure SQL, secret은 Secrets Manager 또는 Key Vault로 설명합니다. 이후 private network, observability, rollback, 비용과 DR까지 연결합니다.

OPERATIONS ANSWER FRAME 1. AWS/Azure 서비스명을 역할로 매핑해 설명한다.
2. edge, network, runtime, data, identity, observability 책임을 나눈다.
3. 장애와 비용 신호를 같은 운영 timeline에서 다룬다.
