브라우저 동작 원리
핸드북

프레임워크는 브라우저 위에서 동작합니다. 성능 문제, 렌더링 버그, 입력 지연을 다루려면 React 이전에 브라우저가 어떤 순서로 일을 처리하는지 알아야 합니다.

AUDIENCE : 실무 프론트엔드 개발자 · React/Vite 기반 제품 개발
BR-01

브라우저는 앱 런타임이다

브라우저는 HTML 뷰어가 아니라 네트워크, 파서, JS 엔진, 렌더링 엔진, 저장소, 보안 모델을 가진 애플리케이션 런타임입니다.

영역하는 일프론트엔드 판단
파싱HTML/CSS/JS를 읽어 구조화blocking script와 critical CSS 관리
JS 엔진코드 실행, microtask 처리긴 작업을 쪼개 input delay를 줄임
렌더링 엔진layout, paint, composite비싼 스타일 변경과 layout thrash 회피
네트워크요청 우선순위, cache, connection 관리preload, lazy loading, cache header 이해
핵심 관점

브라우저는 한 번에 모든 일을 하지 않습니다. JS가 main thread를 오래 점유하면 클릭, 스크롤, 렌더링이 밀립니다. UI 성능은 "코드가 빠른가"보다 브라우저에게 렌더링할 기회를 주는가에 가깝습니다.

BR-02

이벤트 루프와 태스크 큐

이벤트 루프는 JS 실행, Promise callback, timer, 사용자 입력, 렌더링을 순서대로 조율합니다.

EVENT LOOP ORDERtask 하나 실행 (click, timer, network callback 등)
→ microtask queue 모두 비움 (Promise, queueMicrotask)
→ 필요하면 style/layout/paint/composite
→ 다음 task 선택
도구특징사용 기준
Promise.thenmicrotask. 렌더링 전에 몰아서 실행짧은 후속 처리
setTimeouttask. 다음 turn으로 미룸작업 분할, UI 반응성 확보
requestAnimationFrame다음 repaint 전에 애니메이션 콜백 실행을 요청애니메이션과 layout 읽기 타이밍
requestIdleCallbackidle period에 낮은 우선순위 작업 예약. 지원 범위와 timeout 필요성 확인필수 작업, 입력 직후 피드백, animation에는 사용하지 않음
MICROTASK STARVATION

Promise 체인을 무한히 이어가면 렌더링 기회가 밀립니다. microtask는 작고 빠르게 끝나야 합니다. 무거운 작업은 task로 쪼개거나 worker로 옮깁니다.

BR-03

DOM·CSSOM·렌더 트리

화면은 DOM과 CSSOM이 합쳐진 렌더 트리에서 시작합니다. 이후 style, layout, paint, composite를 거쳐 픽셀이 됩니다.

단계비용대표 트리거
styleselector 계산class 변경, stylesheet 변경
layout박스 크기와 위치 계산width, height, top, font-size 변경
paint픽셀 그리기color, shadow, border 변경
composite레이어 합성transform, opacity 변경

성능이 중요한 애니메이션은 가능한 transform과 opacity를 사용합니다. layout을 유발하는 속성을 매 프레임 바꾸면 60fps를 유지하기 어렵습니다.

COMPOSITE COST

transform과 opacity는 레이아웃을 피하기 쉬워 애니메이션 기본값으로 좋지만 공짜는 아닙니다. 별도 compositing layer가 생기면 GPU 메모리와 layer 관리 비용이 늘 수 있으므로 DevTools의 Layers와 Performance에서 실제 composite 비용을 확인합니다.

BR-04

네트워크와 로딩 우선순위

초기 로드는 HTML, CSS, JS, font, image가 경쟁하는 과정입니다. 무엇을 먼저 받게 할지 결정하는 것이 성능 설계입니다.

기법용도주의
preload현재 페이지에 꼭 필요한 리소스를 빨리 받음남용하면 우선순위 경쟁을 악화
prefetch다음 페이지 후보를 낮은 우선순위로 미리 받음데이터 요금과 cache 압박 고려
deferHTML 파싱을 막지 않고 script 실행DOM 준비 이후 실행됨
lazy image아래쪽 이미지를 늦게 로드hero image에는 쓰지 않음
BR-05

저장소와 worker

브라우저 저장소와 worker는 강력하지만 보안과 동기화 비용을 동반합니다.

기능적합한 데이터주의
localStorage작은 사용자 설정동기 API, XSS에 읽힘
sessionStorage탭 단위 임시 상태탭을 닫으면 사라짐
IndexedDB큰 구조화 데이터, offline cacheAPI 복잡, migration 필요
Web WorkerCPU 무거운 계산DOM 접근 불가, 직렬화 비용
Service Workeroffline, cache, push캐시 무효화와 업데이트 UX 중요
TOKEN STORAGE

인증 토큰을 localStorage에 저장하면 XSS가 읽을 수 있습니다. 저장소 선택은 보안 위협 모델과 함께 판단해야 합니다.
