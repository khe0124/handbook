브라우저 동작 원리
핸드북

프레임워크는 브라우저 위에서 동작합니다. 성능 문제, 렌더링 버그, 입력 지연을 다루려면 React 이전에 브라우저가 어떤 순서로 일을 처리하는지 알아야 합니다.

AUDIENCE : 실무 프론트엔드 개발자 · React/Vite 기반 제품 개발
BR-01

브라우저는 앱 런타임이다

브라우저는 HTML 뷰어가 아니라 네트워크, 파서, JS 엔진, 렌더링 엔진, 저장소, 보안 모델을 가진 애플리케이션 런타임입니다.

영역하는 일프론트엔드 판단
파싱HTML/CSS/JS를 읽어 구조화blocking script와 critical CSS 관리
JS 엔진코드 실행, microtask 처리긴 작업을 쪼개 input delay를 줄임
렌더링 엔진layout, paint, composite비싼 스타일 변경과 layout thrash 회피
네트워크요청 우선순위, cache, connection 관리preload, lazy loading, cache header 이해
핵심 관점

브라우저는 한 번에 모든 일을 하지 않습니다. JS가 main thread를 오래 점유하면 클릭, 스크롤, 렌더링이 밀립니다. UI 성능은 "코드가 빠른가"보다 브라우저에게 렌더링할 기회를 주는가에 가깝습니다.

BR-02

이벤트 루프와 태스크 큐

이벤트 루프는 JS 실행, Promise callback, timer, 사용자 입력, 렌더링을 순서대로 조율합니다.

EVENT LOOP ORDERtask 하나 실행 (click, timer, network callback 등)
→ microtask queue 모두 비움 (Promise, queueMicrotask)
→ 필요하면 style/layout/paint/composite
→ 다음 task 선택
도구특징사용 기준
Promise.thenmicrotask. 렌더링 전에 몰아서 실행짧은 후속 처리
setTimeouttask. 다음 turn으로 미룸작업 분할, UI 반응성 확보
requestAnimationFrame다음 repaint 전에 애니메이션 콜백 실행을 요청애니메이션과 layout 읽기 타이밍
requestIdleCallbackidle period에 낮은 우선순위 작업 예약. 지원 범위와 timeout 필요성 확인필수 작업, 입력 직후 피드백, animation에는 사용하지 않음

시간 예산으로 보는 이벤트 루프

기준의미실무 판단
16.7ms60fps에서 한 프레임에 쓸 수 있는 전체 시간JS, style, layout, paint를 모두 포함하므로 JS만 16ms를 써도 이미 위험
50ms 초과Long Task 후보입력 지연과 INP 악화 원인. 작업 분할 또는 worker 이동 검토
100ms 이상사용자가 반응 지연을 체감하기 쉬움클릭 핸들러, validation, formatter, 대량 렌더링을 우선 조사
500ms 이상UI가 멈췄다고 느끼는 구간동기 루프, 큰 JSON parse, 차트 초기화, hydration 비용을 flame chart로 확인
MICROTASK STARVATION

Promise 체인을 무한히 이어가면 렌더링 기회가 밀립니다. microtask는 작고 빠르게 끝나야 합니다. 무거운 작업은 task로 쪼개거나 worker로 옮깁니다.

BR-03

DOM·CSSOM·렌더 트리

화면은 DOM과 CSSOM이 합쳐진 렌더 트리에서 시작합니다. 이후 style, layout, paint, composite를 거쳐 픽셀이 됩니다.

단계비용대표 트리거
styleselector 계산class 변경, stylesheet 변경
layout박스 크기와 위치 계산width, height, top, font-size 변경
paint픽셀 그리기color, shadow, border 변경
composite레이어 합성transform, opacity 변경

성능이 중요한 애니메이션은 가능한 transform과 opacity를 사용합니다. layout을 유발하는 속성을 매 프레임 바꾸면 60fps를 유지하기 어렵습니다.

강제 layout을 만드는 패턴

패턴왜 비싼가대응
DOM 쓰기 직후 offsetHeight 읽기브라우저가 최신 layout을 즉시 계산해야 함읽기와 쓰기를 분리하고 requestAnimationFrame으로 묶음
scroll handler에서 크기 계산 반복스크롤마다 style/layout 계산이 누적passive listener, throttle, IntersectionObserver 검토
큰 테이블에서 행 높이 동적 측정많은 DOM node의 위치 계산이 연쇄 발생고정 높이, virtualization, 측정 cache 사용
애니메이션에서 top/left/width 변경매 프레임 layout과 paint를 유발transform, opacity 중심으로 변경
COMPOSITE COST

transform과 opacity는 레이아웃을 피하기 쉬워 애니메이션 기본값으로 좋지만 공짜는 아닙니다. 별도 compositing layer가 생기면 GPU 메모리와 layer 관리 비용이 늘 수 있으므로 DevTools의 Layers와 Performance에서 실제 composite 비용을 확인합니다.

BR-04

네트워크와 로딩 우선순위

초기 로드는 HTML, CSS, JS, font, image가 경쟁하는 과정입니다. 무엇을 먼저 받게 할지 결정하는 것이 성능 설계입니다.

기법용도주의
preload현재 페이지에 꼭 필요한 리소스를 빨리 받음남용하면 우선순위 경쟁을 악화
prefetch다음 페이지 후보를 낮은 우선순위로 미리 받음데이터 요금과 cache 압박 고려
deferHTML 파싱을 막지 않고 script 실행DOM 준비 이후 실행됨
lazy image아래쪽 이미지를 늦게 로드hero image에는 쓰지 않음
LOADING WATERFALL REVIEW1. HTML이 늦는가, JS/CSS가 늦는가, 데이터 API가 늦는가를 분리한다.
2. LCP 후보 이미지가 preload 또는 높은 우선순위를 받는지 확인한다.
3. route chunk와 data fetch가 직렬 waterfall을 만들지 않는지 본다.
4. cache-control, immutable asset, index.html 재검증 정책을 함께 확인한다.
캐시 대상권장 방향실패 신호
해시가 붙은 JS/CSS asset길게 cache, immutable새 배포마다 같은 URL을 덮어써 사용자가 낡은 코드 실행
index.html짧게 cache 또는 재검증HTML은 새 버전인데 chunk는 오래된 버전이거나 반대 상황 발생
API 응답데이터 신선도와 사용자 작업 기준으로 설정권한·가격·재고처럼 민감한 데이터가 오래 남음
font/image크기, 포맷, 우선순위를 명시폰트 교체로 layout shift가 발생하거나 hero image가 늦게 로드
BR-05

저장소와 worker

브라우저 저장소와 worker는 강력하지만 보안과 동기화 비용을 동반합니다.

기능적합한 데이터주의
localStorage작은 사용자 설정동기 API, XSS에 읽힘
sessionStorage탭 단위 임시 상태탭을 닫으면 사라짐
IndexedDB큰 구조화 데이터, offline cacheAPI 복잡, migration 필요
Web WorkerCPU 무거운 계산DOM 접근 불가, 직렬화 비용
Service Workeroffline, cache, push캐시 무효화와 업데이트 UX 중요

저장소·worker 실패 모드

증상원인 후보확인대응
새 배포 후 일부 사용자만 흰 화면Service Worker 또는 CDN이 낡은 chunk 제공Application 탭 cache, release id, chunk 404SW update 전략, asset hash, index 재검증 정리
큰 파일 처리 중 UI 멈춤main thread에서 parse/encode 수행Performance flame chart의 긴 JS taskWorker 이동, transferable object, chunk 처리
오프라인 데이터가 깨짐IndexedDB schema migration 누락DB version, migration 실패 로그versioned migration과 fallback 초기화 설계
저장 버튼이 간헐적으로 실패storage quota 또는 private mode 제약QuotaExceededError, 브라우저 모드저장 실패 UI, 서버 동기화, 최소 로컬 저장
TOKEN STORAGE

인증 토큰을 localStorage에 저장하면 XSS가 읽을 수 있습니다. 저장소 선택은 보안 위협 모델과 함께 판단해야 합니다.

BR-06

브라우저 문제 진단 루틴

브라우저 문제는 추측으로 고치기 쉽지만, 원인 계층을 잘못 잡으면 해결이 늦어집니다. 증상에서 시작해 main thread, network, rendering, storage를 순서대로 좁힙니다.

증상먼저 볼 도구확인할 증거주요 처방
클릭 후 반응이 늦음Performance, Event Loginput 이후 long task, handler 비용, render commit작업 분할, worker, optimistic feedback, 불필요 렌더 제거
스크롤이 끊김Performance, Rendering FPSscroll handler, forced layout, paint 비용passive listener, virtualization, layout read/write 분리
첫 화면이 늦음Network, Lighthouse, CoverageHTML TTFB, render-blocking CSS/JS, LCP resource timingcritical resource 우선순위, code splitting, image 최적화
배포 후 일부 화면만 깨짐Network, Applicationchunk 404, service worker cache, mixed release asset캐시 정책 수정, reload 안내, SW update flow
메모리 사용량 증가Memory, Performancedetached DOM, listener 누수, cache 무한 증가cleanup, weak reference 검토, cache size 제한
EVIDENCE FIRST

브라우저 이슈 PR에는 "느려서 수정"보다 flame chart 캡처, waterfall, affected route, 브라우저 버전, 재현 조건을 남깁니다. 그래야 최적화가 실제 병목을 줄였는지 비교할 수 있습니다.

BR-08

치트시트

브라우저 파트의 마지막 참조 장입니다. 본문을 읽은 뒤 진단 중 자주 확인하는 용어만 모았습니다.

Long Task>50ms
main thread를 오래 막아 입력 지연을 유발하는 작업입니다.
Layout Thrashread/write mix
layout 읽기와 쓰기를 반복해 브라우저가 강제 layout을 여러 번 수행하는 현상입니다.
Critical Pathinitial render
첫 화면을 그리기 위해 반드시 필요한 리소스와 작업의 경로입니다.
Forced Reflowsync layout
DOM 변경 후 layout 값을 즉시 읽어 브라우저가 렌더링 파이프라인을 앞당겨 수행하는 상황입니다.
Hydration GapSSR interactivity
HTML은 보이지만 클라이언트 JS 연결 전이라 클릭과 입력이 아직 동작하지 않는 시간입니다.
