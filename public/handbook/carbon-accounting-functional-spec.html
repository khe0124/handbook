탄소회계
UI/UX 기능정의서

조직의 온실가스 인벤토리를 세고·검증받고·공시하는 과정을 소프트웨어로 옮기면 어떤 화면이 필요한가. 탄소회계 도메인 핸드북의 개념(경계·Scope·계수·검증)을 화면 인벤토리와 화면별 기능정의로 번역한 기획 문서입니다. 일반 B2B SaaS(기업 지속가능성·ESG 팀용)를 가정합니다.

SCOPE : 기업 단위 GHG 인벤토리 산정·관리·검증·공시 SaaS
ASSUMPTION : 특정 고객사·특정 플랫폼 비종속 — 도메인 표준 기반 일반 기획안
COMPANION : 탄소회계 도메인 핸드북 · LCA 기능정의서
FS-00

문서 개요·전제

무엇을 만드는 문서인지, 무엇은 다루지 않는지를 먼저 못 박습니다. 기획 문서의 경계도 회계의 경계만큼 중요합니다.

항목내용
목적조직 온실가스 인벤토리를 산정·관리·검증·공시하는 제품의 화면과 기능을 정의
대상 제품기업 지속가능성·ESG 담당 팀이 쓰는 B2B SaaS. 다(多)사업장·다년도 인벤토리 운영을 전제
준거 표준GHG Protocol(Corporate·Scope 2 Guidance·Scope 3) · ISO 14064-1/3 · 공시 프레임워크
포함경계 설정 · 데이터 수집 · 계수·산정 · Scope 1/2/3 작업 · 검증 워크플로 · 공시·내보내기 · 감사추적
제외특정 고객 비즈니스 규칙(ENUM·승인단계 수 등) · 청구·과금 · 특정 국가 규제의 세부 서식. 단, 외부 사용자(공급사·검증인)를 받는 제품이므로 인증·테넌트 격리·암호화 등 보안 기반은 제외가 아니라 FS-14에서 정의
전제·가정아래 모든 화면·규칙은 도메인 표준 기반 제안이며, 실제 도입 시 고객별 비즈니스 규칙으로 확정 필요(미확정 항목은 본문에 명시)
기획자 노트 — 추측하지 않은 것

승인 단계 수, 검증 보증수준 의무 여부, 공시 대상 규제(K-ETS/CBAM/지속가능성 공시 중 무엇), 조직경계 기준(지분 vs 통제) 같은 항목은 고객·규제 환경에 따라 갈리는 비즈니스 규칙이라 임의로 채우지 않았습니다. 화면은 이 값들을 설정 가능한 파라미터로 받도록 설계했습니다 — 정책을 코드가 아니라 설정으로.

FS-01

페르소나·역할

탄소회계는 한 사람의 작업이 아니라 작성→검토→검증→승인의 분업입니다. 역할 분리(SoD)가 회계 무결성의 전제라, 권한 모델이 화면 설계를 좌우합니다.

PRIMARY · PREPARER
인벤토리 작성자 (지속가능성 실무자)

활동자료를 수집·입력하고 산정을 실행한다. 화면에서 가장 오래 머무는 사용자 — 데이터 입력 효율과 계수 추적성이 핵심 니즈.

REVIEWER
내부 검토자 (팀 리드)

작성된 산정의 완전성·일관성을 검토하고 보완을 요청한다. 변경 이력과 핫스팟 요약을 빠르게 본다.

VERIFIER · 3RD PARTY
제3자 검증인 (외부)

ISO 14064-3 기반으로 표본을 추출해 근거를 확인하고 finding을 남긴다. 읽기·코멘트 권한 중심, 원본 변경 불가.

APPROVER
승인자 (경영·공시 책임)

최종 인벤토리를 승인·확정하고 공시본을 발행한다. 요약·리스크·검증 결과만 본다.

SUPPLIER · EXTERNAL
공급사 담당자 (외부)

요청받은 제품탄소발자국(PCF)·활동자료를 제출한다. 제한된 단일 목적 화면만 접근.

ADMIN
시스템 관리자

조직·사용자·역할, 계수 라이브러리, 기준연도 정책을 관리한다.

설계 함의

작성자와 검증인이 같은 데이터를 다른 권한으로 봅니다. 즉 화면을 역할별로 따로 만들기보다, 하나의 화면 + 역할 기반 액션·필드 가시성으로 설계합니다(→ FS-07). 검증인의 "변경 불가, 코멘트 가능"은 데이터 레이어가 아니라 권한 레이어에서 강제합니다.

FS-02

정보구조(IA)·화면맵

제품을 여섯 영역으로 나눕니다. 데이터가 흐르는 순서(경계→수집→산정→결과→거버넌스)가 곧 내비게이션 순서입니다.

① 설정·경계

  • 조직경계 설정 지분/통제·법인·사업장
  • 보고기간·기준연도 재계산 정책
  • 배출원 등록 시설별 소스 목록

② 데이터 수집

  • 활동자료 입력 수동·업로드·연동
  • 공급사 PCF 수집 요청·수취 포털
  • 데이터 품질 출처·티어·완전성

③ 계수·산정

  • 배출계수 라이브러리 출처·버전·GWP
  • 산정 엔진 AD×EF×GWP 실행
  • 불확실성 페디그리·분포

④ Scope 작업공간

  • Scope 1 고정·이동·공정·냉매
  • Scope 2 위치/시장 이중장부
  • Scope 3 15 카테고리

⑤ 결과·관리

  • 인벤토리 대시보드 집계·핫스팟
  • 감축목표·시나리오 기준연도 대비
  • 크레딧·상쇄 회계 분리 계상

⑥ 거버넌스

  • 검증·보증 워크플로 finding
  • 공시·내보내기 규제 서식
  • 감사추적·버전 재현성
  • 사용자·권한 RBAC

여섯 영역은 좌측 글로벌 내비게이션의 1차 분류가 되고, Scope 작업공간(④)은 한 보고연도를 선택한 뒤 들어가는 컨텍스트 종속 영역입니다 — 항상 "어느 조직·어느 보고연도"라는 컨텍스트가 상단에 고정됩니다.

FS-04

핵심 화면 기능정의

P0·P1 화면을 기능정의 카드로 상술합니다. 각 카드는 목적 · 핵심 기능 · 데이터/상태/검증 · 핸드북 연계로 구성됩니다.

상술 범위 — 솔직한 표기

이 장은 전 화면이 아니라 선별 상술입니다. 카드가 없는 화면(SC-02·03·06 산정엔진·07 Scope 1·11·13·18 RBAC 등)도 FS-03 인벤토리에 정의돼 있으며, 잔여 P0 카드(특히 SC-06 산정엔진의 계산식·보정·라운딩, SC-18 권한·SoD 구현)는 상세설계 단계에서 완성합니다. 수용 기준은 FS-16, 빠진 도메인 개념은 FS-13으로 보강했습니다.

SC-01조직경계 설정P0

"보안·DR·마이그레이션이 비었다"FS-14에서 보강(테넌트 격리·암호화·tamper-evident 로그·DR·PII·소유권). 과거 데이터 마이그레이션·기준연도 backfill도 포함
