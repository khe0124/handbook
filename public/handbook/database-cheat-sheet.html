Cheat Sheet

쿼리와 스키마 변경을 빠르게 점검하기 위한 참조입니다. SQL 문법보다 인덱스, 실행 계획, lock, 마이그레이션 안전성을 우선합니다.

DB-CS-01

개요

DB 문제는 대부분 잘못된 접근 경로, 과한 lock, 되돌릴 수 없는 migration에서 시작합니다. 변경 전 실행 계획과 rollback을 확인합니다.

상황먼저 확인할 것위험 신호
느린 쿼리filter, join, sort, row estimate, actual rowsactual rows가 추정치와 크게 다름, sort/hash가 디스크 사용
인덱스 추가WHERE/JOIN/ORDER BY와 column order쓰기 많은 테이블에 사용 빈도 낮은 인덱스 추가
스키마 변경table rewrite, lock level, default, backfill큰 테이블에 NOT NULL/default/backfill을 한 번에 수행
정합성 이슈transaction boundary, isolation, lost update읽고 계산한 뒤 업데이트하는 로직에 충돌 제어 없음
페이지네이션정렬 기준, index, offset 크기, cursor 후보큰 offset에서 응답 시간이 선형 증가

선택기준주의점
정규화쓰기 정합성과 중복 제거가 중요함조회 시 join 비용과 쿼리 복잡도 증가

DB-CS-04

자주 나는 실수

  • 인덱스를 추가하면 모든 쿼리가 빨라진다고 생각한다. 읽기 패턴과 선택도가 맞지 않으면 쓰기 비용만 늘어난다.
  • 실행 계획의 estimated rows와 actual rows 차이를 보지 않는다. 통계가 틀리면 optimizer가 잘못된 경로를 고른다.
  • 큰 테이블에 default와 not null을 한 번에 추가해 lock 위험을 만든다. nullable 추가, backfill, 검증, NOT NULL 순서로 나눠야 한다.
  • 마이그레이션 rollback은 코드 rollback과 별개라는 점을 놓친다. 데이터 변경은 되돌릴 스크립트나 보정 절차가 필요하다.
  • pagination에서 offset이 커질수록 비용이 커지는 문제를 무시한다. 무한 스크롤과 대량 목록은 cursor 방식을 검토한다.
  • foreign key와 unique constraint를 앱 로직으로만 보장한다. 경쟁 조건에서는 DB 제약이 최후의 방어선이다.
DB-CS-05

면접/리뷰용 한 줄 답변

인덱스index
인덱스는 읽기를 빠르게 하지만 쓰기 비용과 저장 공간을 늘리므로 쿼리 패턴에 맞춰 설계해야 합니다.
실행 계획plan
실행 계획은 DB가 데이터를 어떤 경로로 읽고 조인하고 정렬하는지 보여주는 비용 지도입니다.
마이그레이션migration
운영 마이그레이션은 DDL이 아니라 lock, backfill, rollback, 신구 코드 호환성까지 포함한 변경 절차입니다.
트랜잭션isolation
트랜잭션은 원자성만이 아니라 동시에 실행될 때 어떤 값을 볼 수 있는지까지 포함해 설계해야 합니다.
제약조건constraint
중복, 참조, 필수값 같은 핵심 정합성은 애플리케이션 검증뿐 아니라 DB 제약으로도 막아야 합니다.

변경merge/deploy 차단 기준필수 증거

DB-CS-08

Failure Playbook

SLOW QUERY FIRST LOOK1. 느린 route와 query fingerprint 확인
2. EXPLAIN (ANALYZE, BUFFERS)로 실제 실행 확인
3. estimated rows와 actual rows 차이 확인
4. sort/hash spill, lock wait, sequential scan 분리
5. index 후보를 쿼리 조건/정렬 순서와 맞춤
6. 변경 전후 p95/p99와 write latency 기록
운영 원칙

DB 변경은 코드 변경보다 되돌리기 어렵습니다. "쿼리가 빨라질 것"이 아니라 어떤 실행 계획이 어떻게 바뀌고, lock과 rollback 위험이 어떤지 증거로 남겨야 합니다.

DB-CS-09

쿼리·마이그레이션 카드

DB 치트시트는 SQL 문법보다 운영 안전성 판단에 씁니다. 쿼리 변경은 실행 계획으로, 스키마 변경은 lock과 rollback으로 닫습니다.

작업바로 실행/확인정상 신호위험 신호
느린 SELECTEXPLAIN (ANALYZE, BUFFERS) SELECT ...예상 rows와 실제 rows가 크게 다르지 않고 sort spill 없음Seq Scan 반복, rows 오차 큼, temp read/write 발생
인덱스 후보WHERE 동등 조건, 범위 조건, ORDER BY 순서 정리쿼리 패턴과 컬럼 순서가 설명됨사용 빈도 모름, 쓰기 많은 테이블에 무조건 추가
DDL 변경row count, table size, lock level, rewrite 여부expand/deploy/backfill/validate/contract 순서가 있음대용량 테이블에 NOT NULL/default/backfill 일괄 실행
Backfillbatch size, idempotency, progress, verification query중단 후 재시작 가능하고 affected rows 추적 가능한 transaction으로 전체 update, 진행률/검증 없음
삭제/보정transaction, WHERE, LIMIT, SELECT dry-run, backup대상 row를 먼저 조회하고 복구 방법이 있음운영에서 조건 없는 UPDATE/DELETE
