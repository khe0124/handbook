DB-CS-01

핵심 명령·스니펫

실행계획 → 인덱스 → 락 → 통계 순으로 본다. 아래 스니펫은 대부분 복붙 후 테이블/컬럼명만 바꾸면 된다.

실행계획 (EXPLAIN)

# 실제 실행 + 버퍼까지 (진짜 느린 원인 찾을 때 이거)
EXPLAIN (ANALYZE, BUFFERS) SELECT ... FROM orders WHERE user_id = 42;

# 계획만 (쿼리 실제 실행 안 함, 안전)
EXPLAIN SELECT ... FROM orders WHERE user_id = 42;

# 포맷 옵션
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) SELECT ...;

# MySQL:  EXPLAIN ANALYZE SELECT ...;   /  EXPLAIN FORMAT=JSON SELECT ...;
# MySQL 8: EXPLAIN ANALYZE 는 실제 실행됨 (Postgres와 동일 주의)
주의

ANALYZE는 쿼리를 실제로 실행한다. UPDATE/DELETE/INSERT에 붙이면 데이터가 바뀐다. 쓰기 쿼리는 BEGIN; EXPLAIN (ANALYZE) ...; ROLLBACK; 로 감싼다.

인덱스 확인·생성

# 테이블의 인덱스/컬럼/제약 한눈에 (psql)
\d orders
\d+ orders

# 인덱스 목록만
SELECT * FROM pg_indexes WHERE tablename = 'orders';

# 인덱스 정의 + 크기
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'orders';
SELECT pg_size_pretty(pg_relation_size('idx_orders_user_id'));

# 락 없이 인덱스 생성 (운영 테이블은 반드시 CONCURRENTLY)
CREATE INDEX CONCURRENTLY idx_orders_user_id ON orders (user_id);

# 복합 인덱스 (컬럼 순서 = 좌측 우선 규칙)
CREATE INDEX idx_orders_user_created ON orders (user_id, created_at DESC);

# 부분 인덱스 (조건 걸린 행만 색인 → 작고 빠름)
CREATE INDEX idx_orders_active ON orders (user_id) WHERE status = 'ACTIVE';

# 안 쓰이는 인덱스 찾기 (idx_scan = 0)
SELECT relname, indexrelname, idx_scan
FROM pg_stat_user_indexes ORDER BY idx_scan ASC;
주의

CREATE INDEX CONCURRENTLY는 transaction block 안에서 못 쓴다. 실패하면 INVALID 인덱스가 남으니 DROP INDEX 후 재시도. MySQL은 ALGORITHM=INPLACE, LOCK=NONE.

블로킹 락 / 활성 쿼리

# 지금 잡혀 있는 락 (누가 무엇을)
SELECT * FROM pg_locks WHERE NOT granted;

# 무엇이 무엇을 막는지 (blocking → blocked) — 가장 실전적
SELECT blocked.pid   AS blocked_pid,
       blocking.pid  AS blocking_pid,
       blocked.query AS blocked_query,
       blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocking
  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE blocked.wait_event_type = 'Lock';

# 오래 도는 쿼리 (5초 이상)
SELECT pid, now() - query_start AS runtime, state, query
FROM pg_stat_activity
WHERE state != 'idle' AND now() - query_start > interval '5 seconds'
ORDER BY runtime DESC;

# 문제 세션 죽이기 (cancel = 쿼리만, terminate = 연결까지)
SELECT pg_cancel_backend(<pid>);
SELECT pg_terminate_backend(<pid>);
주의

pg_terminate_backend는 연결을 끊는다. 트랜잭션 중이면 rollback된다. 운영에서 남의 세션 죽이기 전에 무슨 쿼리인지 반드시 확인.

느린 쿼리 통계 (pg_stat_statements)

# 확장 켜기 (한 번, 재시작 필요할 수 있음)
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

# 총 소요시간 상위 쿼리 (범인 후보)
SELECT query, calls, total_exec_time, mean_exec_time, rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC LIMIT 20;

# 평균 느린 쿼리
SELECT query, calls, mean_exec_time
FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 20;

# 통계 리셋
SELECT pg_stat_statements_reset();

# MySQL: SELECT * FROM sys.statement_analysis ORDER BY total_latency DESC;
#        performance_schema.events_statements_summary_by_digest

통계·정리 (VACUUM / ANALYZE)

# 통계 갱신 (실행계획이 이상하면 제일 먼저)
VACUUM (ANALYZE) orders;
ANALYZE orders;              -- 통계만, 락 가벼움

# dead tuple / 마지막 vacuum 시점 확인
SELECT relname, n_dead_tup, last_autovacuum, last_analyze
FROM pg_stat_user_tables ORDER BY n_dead_tup DESC;

# 테이블/인덱스 크기
SELECT pg_size_pretty(pg_total_relation_size('orders'));
위험

VACUUM FULL은 테이블을 ACCESS EXCLUSIVE 락으로 완전히 잠그고 재작성한다. 대형 테이블에 운영 중 실행 금지. 공간 회수는 pg_repack 고려.

안전한 마이그레이션 패턴 (NOT NULL 컬럼 추가)

# 1) nullable로 먼저 추가 (즉시, 락 짧음)
ALTER TABLE orders ADD COLUMN region text;

# 2) 배치로 backfill (한 번에 UPDATE 금지 — 락+WAL 폭증)
UPDATE orders SET region = 'KR'
WHERE id BETWEEN 1 AND 10000 AND region IS NULL;
-- id 범위 올려가며 반복 / 사이에 sleep

# 3) 기본값·제약은 backfill 끝난 뒤
ALTER TABLE orders ALTER COLUMN region SET DEFAULT 'KR';

# 4) NOT NULL 은 CHECK로 먼저 검증 후 승격 (풀스캔 락 회피)
ALTER TABLE orders ADD CONSTRAINT region_nn CHECK (region IS NOT NULL) NOT VALID;
ALTER TABLE orders VALIDATE CONSTRAINT region_nn;   -- 락 가벼움
ALTER TABLE orders ALTER COLUMN region SET NOT NULL;
주의

대형 테이블 ALTER TABLE은 ACCESS EXCLUSIVE 락을 잡는다. 배포 시 SET lock_timeout = '3s'; 를 먼저 걸어 락 대기가 서비스를 막지 않게 한다.

DB-CS-04

장애 증거 1카드

SLOW QUERY FIRST LOOK 1. EXPLAIN (ANALYZE, BUFFERS) 로 실제 실행계획을 뜬다.
2. 인덱스 사용 여부 — Seq Scan인가 Index Scan인가. 예상 인덱스가 왜 안 먹나.
3. row 수 / 버퍼 — estimated vs actual rows 괴리(통계 낡음?), read 버퍼량.
4. lock 대기 — 느린 게 아니라 막힌 것일 수 있다. pg_blocking_pids 확인.
5. 그다음 쿼리 → 인덱스 → 스키마 순으로 고친다. 스키마 변경이 가장 무겁고 마지막.
순서질문증거
1실제로 무엇이 느린가EXPLAIN (ANALYZE, BUFFERS) 출력, 가장 비싼 노드
2인덱스가 쓰였나Index Scan / Seq Scan / Bitmap 여부, \d orders 인덱스 목록
3추정과 실제가 맞나estimated vs actual rows, last_analyze 시점
4느린 게 아니라 막힌 건 아닌가pg_stat_activity wait_event, pg_blocking_pids
5반복적으로 비싼가pg_stat_statements total/mean_exec_time 순위
복구·중단 기준

인덱스 추가는 CONCURRENTLY로 무중단. 원인이 특정 세션의 blocking이면 pg_cancel_backend로 완화. DDL이 락으로 서비스를 막고 있으면 즉시 취소하고 배치·CONCURRENTLY로 재설계 — 운영 중 대형 테이블 락은 롤백이 최우선.
