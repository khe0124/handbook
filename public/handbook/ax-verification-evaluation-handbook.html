검증과 평가

AI 결과의 신뢰는 그럴듯함에서 나오지 않습니다. 어떤 입력으로, 어떤 도구를 써서, 어떤 검증을 통과했는지가 남아야 합니다.

EVAL-01

검증의 목적

AI output은 초안입니다. 신뢰는 독립 검증, 재현 가능한 trace, 실패 원인 분류에서 나옵니다. “테스트 통과”와 “충분히 검증됨”은 다릅니다.

평가 축질문
기능 정확성요구한 동작을 실제로 만족하는가
회귀 없음기존 계약과 사용자 흐름을 깨지 않았는가
보안 경계권한, secret, 신뢰 경계가 약화되지 않았는가
재현 가능성다른 사람이 같은 검증을 다시 실행할 수 있는가
설명 가능성왜 이 변경이 필요한지 근거가 남아 있는가
TASK SPEC SAMPLE : AUTH CHANGE goal: refresh token 재사용 탐지 실패 시 모든 active session을 폐기한다
scope: auth/session service, token rotation tests, incident log event
non-goals: 로그인 UI 개편, token schema rename, OAuth provider 변경
acceptance: 재사용된 refresh token으로 재발급 요청 시 401, session revocation event 1건 기록, 정상 rotation은 유지
invariants: 기존 access token TTL, cookie 옵션, mobile client contract 불변
required evidence: 실패하는 regression test 먼저 제시, 통과한 unit/integration test, 수동 재사용 시나리오 로그
approval gate: 인증 담당자와 온콜 담당자가 위험 설명과 rollback path를 확인
TASK SPEC SAMPLE : COPY / CONTENT CHANGE goal: 결제 실패 안내 문구를 사용자 행동 중심으로 수정한다
scope: checkout error message, localization key, screenshot evidence
non-goals: 결제 provider retry 정책, pricing page, analytics schema 변경
acceptance: 카드 거절, 네트워크 실패, provider timeout의 문구가 서로 구분됨
invariants: error code mapping, support deep link, existing locale fallback 불변
required evidence: affected states screenshot, localization diff, QA checklist
approval gate: PM 또는 CS owner가 고객 영향 문구를 승인
EVAL-02

Episode Package

Episode package는 하나의 AI 작업을 재현 가능한 평가 단위로 저장한 것입니다. 최종 diff만 저장하면 왜 그런 변경이 나왔는지, 무엇을 검증했는지 알 수 없습니다.

EPISODE PACKAGE episode_id: ax-auth-refactor-2026-06-24
goal: refresh token rotation 리팩터링
input_context: 관련 파일, 요구사항, 금지사항
agent_actions: 읽은 파일, 수정한 파일, 실행한 테스트
tool_trace: commands, exit codes, relevant outputs
expected_invariants: 기존 로그인 플로우 유지, 재사용 탐지, 만료 정책 불변
verification: unit, integration, manual attack scenario
failure_notes: 놓친 edge case, 추가 검증 필요 영역
EVAL RUBRIC eval set: 대표 episode, 회귀 episode, adversarial episode를 섞어 구성
functional check: 요구 동작이 실제 사용자 흐름에서 맞는지 확인
regression check: 기존 계약, public API, 저장 데이터 호환성 확인
security check: 인증, 권한, secret 노출, injection 위험 확인
policy check: 라이선스, 데이터 사용, 조직 승인 정책 확인
human-review check: reviewer가 근거와 남은 위험을 확인
training cohort: 같은 eval rubric으로 실습하고 판정 차이를 보정
Context package gate통과 기준거절 기준
요구사항goal, scope, non-goal, acceptance가 분리되어 있음“개선”, “정리”, “적당히”처럼 판정 불가능한 표현만 있음
시스템 경계관련 owner, upstream/downstream, runtime 환경이 명시됨agent가 파일 검색으로 경계를 추측해야 함
불변조건깨지면 안 되는 API, 데이터, UX, 보안 정책이 적힘새 동작만 있고 기존 계약이 빠져 있음
검증 명령필수 테스트와 수동 확인 절차가 구체적임“테스트 실행”만 있고 어떤 테스트인지 없음
증거 보존trace, 주요 output, screenshot, reviewer note 위치가 정해짐최종 diff 외에는 재검토할 자료가 없음
EPISODE LOG EXAMPLE 09:12 task accepted: checkout timeout 문구 분리, provider retry 정책은 비목표
09:17 context read: CheckoutError.tsx, payment-errors.ts, ko.json, existing QA note
09:26 hypothesis: timeout과 declined가 같은 fallback key를 공유해 사용자가 다음 행동을 알 수 없음
09:41 change: error reason별 message key 분리, unknown은 기존 fallback 유지
09:48 verification: npm test -- checkout-errors, locale key snapshot, desktop/mobile screenshot
09:55 reviewer challenge: provider가 timeout을 declined로 매핑하는 edge case 확인 요청
10:07 follow-up: provider fixture 추가, timeout/declined/unknown 모두 독립 assertion 통과
10:12 residual risk: 실제 provider sandbox 장애는 재현하지 못함. release 후 payment_error_reason 분포를 24시간 모니터링
EVAL-03

Failure Classification

분류의미다음 액션
Spec drift요청하지 않은 리팩터링이나 기능 변경 수행작업 명세와 비목표 강화
False green테스트는 통과하지만 핵심 케이스가 없음수용 기준과 독립 테스트 추가
Tool hallucination실행하지 않은 명령을 실행한 것처럼 요약tool trace와 exit code 저장
Context truncation중요 파일이나 정책을 읽지 않고 일반론으로 구현context package 재구성
Verification theater검증 완료라고 하지만 실제로는 lint만 실행작업 유형별 필수 검증 매트릭스 운영
Judge biasLLM judge가 그럴듯한 답변에 높은 점수실행 가능한 테스트와 사람 리뷰 병행
환각 / 도구 오용 유형관찰 신호차단 장치
Invented API존재하지 않는 helper, config, endpoint를 사용참조 파일 line evidence 요구, compile/test 필수
Phantom command실행하지 않은 명령을 검증 결과로 보고command, exit code, 핵심 output을 episode에 저장
Stale memory현재 repo가 아니라 일반 프레임워크 지식으로 구현수정 전 관련 파일과 호출부 read receipt 확인
Over-broad edit요청 범위를 넘어 포맷팅, rename, 구조 변경 수행owned files와 forbidden files를 task spec에 명시
Unsafe tool choice운영 DB, secret, destructive command를 편의상 사용human approval gate와 dry-run branch 우선 원칙 적용
Result laundering실패한 테스트를 “환경 문제”로 축소하고 통과처럼 요약실패 로그, 재시도 조건, 미해결 위험을 별도 필드로 기록
EVAL-04

Adversarial Verification

검증자는 결과를 통과시키려 하지 말고 깨뜨리려 해야 합니다. 특히 인증, 결제, 권한, 마이그레이션, 캐싱, 보안 변경은 adversarial case가 필요합니다.

작업깨뜨려 볼 질문
인증만료, 재사용, 탈취, 다른 사용자 토큰으로 깨지는가
인가소유자가 아닌 사용자가 id만 바꿔 접근할 수 있는가
마이그레이션구버전 앱과 신버전 DB가 동시에 존재해도 안전한가
캐싱stale data, stampede, 권한별 캐시 오염이 생기는가
프론트엔드loading, empty, error, permission denied, offline 상태가 정의됐는가
Human approval gate필수 승인자승인 전 증거
권한 / 인증 변경security owner, service ownerattack scenario, negative test, rollback path
데이터 마이그레이션DB owner, on-call ownerbackfill plan, lock 위험, 구버전/신버전 동시 동작 검증
결제 / 과금billing owner, PM금액 계산 fixture, refund/duplicate charge 방지 검증
고객 커뮤니케이션PM, support owner상태별 screenshot, localization review, 민감 표현 확인
운영 자동화SRE 또는 platform ownerdry-run output, rate limit, failure rollback, alert 영향
승인 원칙

사람 승인은 “AI가 맞는지”를 감으로 판단하는 절차가 아닙니다. 승인자는 남은 위험, 검증 증거, rollback 가능성, owner 책임을 확인합니다. 증거가 부족하면 재작업을 요청하고, 시간이 없으면 범위를 줄입니다.

EVAL-05

평가 체크리스트

  • 최종 산출물만 보지 않고 trace를 확인했는가?
  • 요구사항, 금지사항, 불변조건이 episode에 명시됐는가?
  • 테스트 실패 시 원인 분석이 남았는가?
  • AI가 직접 만든 테스트만 믿지 않고 독립 검증을 했는가?
  • adversarial case가 최소 1개 이상 있는가?
  • tool trace에 command, exit code, 핵심 output이 남았는가?
  • 실패한 검증과 건너뛴 검증을 성공 결과와 분리해 기록했는가?
  • human approval gate가 필요한 변경인지 판단했는가?
  • reviewer가 볼 증거가 diff 안팎에 충분히 연결돼 있는가?
  • 통과와 충분히 검증됨을 구분했는가?
Rubric 항목0점1점2점
Spec 충실도요구를 추측하거나 범위를 벗어남핵심 요구는 맞지만 non-goal 또는 invariant 누락goal, scope, non-goal, invariant를 모두 지킴
Context 사용관련 파일과 정책을 읽은 증거 없음주요 파일은 읽었지만 호출부나 운영 맥락 누락관련 코드, 정책, owner 맥락을 근거로 판단
검증 깊이lint 또는 빌드만 실행happy path 테스트 중심회귀, negative, adversarial case까지 확인
Failure handling실패를 숨기거나 성공처럼 요약실패는 기록했지만 원인과 다음 액션이 약함원인, 영향, 재시도 조건, 남은 위험을 분리
Reviewer burden리뷰어가 처음부터 재조사해야 함근거는 있으나 산발적임episode log와 증거 링크로 핵심 판단이 빠름
EVAL-06

자동화 성과 평가

AX 자동화는 “AI가 만든 양”이 아니라 업무 흐름이 실제로 좋아졌는지로 평가합니다. 특히 리뷰 부담과 재작업은 생성 속도 뒤에 숨어 있는 비용입니다.

지표좋아지는 신호나빠지는 신호
lead time작업 완료 시간이 줄고 대기 시간이 줄어듦생성은 빠른데 리뷰와 수정이 길어짐
review latency리뷰어가 핵심 위험에 더 빨리 도달AI finding이 많아져 triage 비용 증가
acceptance rateAI 제안 중 실제 반영되는 비율이 안정적으로 상승초안은 많지만 대부분 폐기됨
rework rateAI 산출물 수정·되돌림 비율 감소초안 품질이 낮아 반복 수정 증가
defect escape운영 결함 또는 회귀가 증가하지 않음AI 변경 이후 rollback, hotfix, incident 증가
cost per episode토큰·도구·사람 검토 비용이 업무 가치보다 낮음작은 일에 agent orchestration 비용이 과도함
learning retention팀원이 변경 이유와 원리를 설명 가능AI가 처리한 영역을 팀이 이해하지 못함
운영 지표계산 예시해석
review burdenreview minutes per accepted episode, reviewer comment count, clarification round 수생성 속도가 빨라도 리뷰 시간이 늘면 자동화 이익이 줄어듦
rework depthAI diff 중 사람이 다시 쓴 line 비율, follow-up commit 수, revert 비율초안 채택률보다 실제 수정 깊이가 품질을 더 잘 보여줌
defect escape rateAI-assisted change 이후 7/30일 내 bug, rollback, incident, support ticket 비율검증 체계가 약하면 결함이 리뷰 밖으로 빠져나감
spec clarification load작업 시작 후 요구사항 재질문 수, blocked time, missing context countcontext package 품질이 낮을수록 agent와 reviewer가 같은 질문을 반복
approval gate hit rate고위험 변경 중 승인 gate를 실제 통과한 비율낮으면 프로세스가 우회되고, 높기만 하면 과승인으로 병목 가능
evidence completenessepisode 중 spec, trace, verification, residual risk가 모두 있는 비율자동화 성과를 사후 감사와 학습으로 연결하는 기반
실무 해석

공개 연구들은 AI가 단순 구현 과제에서는 큰 시간 절감을 보일 수 있지만, 실제 협업 환경에서는 통합 시간과 유지보수 부담이 늘 수 있음을 보여줍니다. 따라서 AX 평가는 개인 체감 속도보다 팀 단위 흐름, 품질, 리뷰 부담을 함께 봐야 합니다.
