시스템 설계 면접

시스템 설계 면접은 완벽한 구조를 바로 그리는 시험이 아닙니다. 요구사항을 묻고, 제약을 세우고, trade-off를 설명하며, 위험을 검증 가능한 단위로 줄이는 과정을 보여주는 자리입니다.

SD-01

Q. 관리자 대시보드를 설계해보세요.

핵심 개념

관리자 대시보드는 단순 목록 화면이 아니라 검색, 필터, 권한, 감사 로그, 대량 데이터, 실패 상태, 운영 액션을 포함한 업무 도구입니다.

개념 해부

관리자 대시보드는 사용자가 많지 않아도 한 번의 오조작 영향이 큰 시스템입니다. 검색과 목록뿐 아니라 권한, audit log, bulk action, dry-run, 실패 복구가 설계 대상입니다. 면접에서는 UI 배치보다 actor, resource, action, evidence가 어떻게 남는지 말해야 깊이가 납니다.

30초 답변

먼저 사용자 역할, 데이터 규모, 검색/필터 요구, 권한별 action, 실시간성 필요 여부를 확인합니다. 이후 URL state 기반 필터, cursor 또는 page pagination, 서버 filtering/sorting, 권한별 action guard, audit log, loading/empty/error 상태를 설계합니다.

90초 답변

데이터 모델은 user/admin role, resource, action, audit event를 분리합니다. API는 목록 조회, 상세 조회, 상태 변경, bulk action을 나누고, 목록 API는 pagination과 filter contract를 명확히 합니다. 프론트는 URL state로 검색 조건을 공유 가능하게 하고, 서버 상태와 선택/편집 UI 상태를 분리합니다. 운영 관점에서는 action 실패, 권한 거부, 중복 클릭, 부분 성공을 처리해야 합니다.

꼬리질문

질문답변 방향
대량 데이터는?서버 pagination/filtering, virtualization, export job 분리를 고려합니다.
권한은?프론트는 UX guard, 서버는 최종 인가와 audit log를 담당합니다.
관측성은?action 실패율, latency, audit event, traceId를 남깁니다.

오답/주의점

주의

차트와 카드 배치만 말하면 디자인 답변이 됩니다. 데이터 계약과 운영 액션을 함께 말하세요.

SD-02

Q. 대형 Data Grid 입력 시스템을 설계해보세요.

핵심 개념

대형 grid는 렌더링 문제와 데이터 정합성 문제가 동시에 있습니다. 화면 cell state와 저장 가능한 domain model을 분리해야 합니다.

개념 해부

대형 Data Grid는 표가 아니라 수천 개의 작은 입력 폼과 검증 결과가 동시에 있는 시스템입니다. 원본 데이터, draft, dirty state, validation error, selection, focus가 섞이면 저장 취소와 실패 복구가 어려워집니다. 렌더링 최적화와 데이터 정합성 설계를 동시에 말해야 합니다.

30초 답변

행 수, 컬럼 수, 편집 방식, 저장 단위, validation 위치를 먼저 확인합니다. grid는 virtualization으로 렌더링 비용을 줄이고, serverData와 draftData, validationErrors, dirtyRows를 분리해 저장 취소와 실패 복구가 가능하게 설계합니다.

90초 답변

column definition은 표시 설정이 아니라 formatter, parser, editor, validator를 가진 입력 계약으로 봅니다. cell-level validation은 즉시 피드백을 주고, row-level validation은 cross-field 규칙을 확인하며, submit-level validation은 중복이나 서버 상태 의존 규칙을 처리합니다. 대량 저장은 전체 submit보다 changed rows 기반 patch나 batch job을 고려합니다.

꼬리질문

질문답변 방향
모든 row를 React state에 두면?렌더링과 메모리 비용이 커질 수 있어 변경분 중심 상태와 virtualization을 봅니다.
Excel 업로드는?parse, mapping, preview, validation, save를 단계로 나눕니다.
저장 실패는?row/cell 단위 오류를 매핑하고 성공/실패 partial result를 보여줍니다.

오답/주의점

주의

Data Grid 라이브러리 이름만 말하지 말고 상태 구조와 validation 책임을 말해야 합니다.

SD-03

Q. 실시간 대시보드를 설계해보세요.

핵심 개념

실시간은 항상 WebSocket이 답이 아닙니다. 정확도, 지연 허용치, 업데이트 빈도, 사용자 수, reconnect 비용으로 polling, SSE, WebSocket을 고릅니다.

개념 해부

실시간성은 지연 시간이 0이라는 뜻이 아니라 허용 가능한 freshness와 전달 방식의 선택입니다. polling은 단순하고 견고하지만 지연과 낭비가 있고, SSE는 단방향 stream에 적합하며, WebSocket은 양방향 상호작용에 강하지만 연결 운영 비용이 큽니다. 연결 상태와 데이터 stale 상태를 UI가 설명해야 합니다.

30초 답변

먼저 어느 정도 지연을 허용하는지, 양방향 통신이 필요한지, 데이터 손실이 치명적인지 확인합니다. 단방향 알림이나 지표는 SSE, 낮은 빈도 갱신은 polling, 양방향 협업이나 즉시성이 강한 기능은 WebSocket을 검토합니다.

90초 답변

프론트는 connection state, stale state, reconnect, backoff, 마지막 업데이트 시각을 보여줘야 합니다. 백엔드는 fan-out, rate limit, backpressure, message ordering을 고려합니다. 지표 대시보드는 데이터가 늦을 수 있음을 사용자에게 표시하고, 알림은 중복과 누락을 막기 위해 event id나 cursor를 둘 수 있습니다.

꼬리질문

질문답변 방향
polling 단점은?불필요한 요청과 지연이 생기지만 단순하고 운영이 쉽습니다.
WebSocket 단점은?연결 관리, scale-out, 인증 갱신, reconnect가 복잡합니다.
데이터가 stale하면?마지막 업데이트 시각과 동기화 상태를 명확히 보여줍니다.

오답/주의점

주의

“실시간이니까 WebSocket”이라고 바로 답하지 마세요. 요구사항을 먼저 나눠야 합니다.

SD-04

Q. 복잡한 multi-step form을 설계해보세요.

핵심 개념

복잡한 form은 입력 필드 묶음이 아니라 상태 전이와 validation, 임시 저장, 권한, 서버 검증이 결합된 workflow입니다.

개념 해부

multi-step form은 field collection이 아니라 상태 전이 모델입니다. 단계별 입력 가능 조건, 임시 저장, 뒤로가기, 서버 검증 실패, 권한 변경이 모두 사용자 흐름을 바꿉니다. state machine까지 쓰지 않더라도 가능한 상태와 전이를 명확히 두면 예외 처리가 안정됩니다.

30초 답변

단계 의존성, 임시 저장, 뒤로 가기, validation 범위, 제출 후 수정 가능 여부를 먼저 확인합니다. 상태는 step state, draft data, validation errors, server snapshot으로 나누고, schema validation과 서버 validation을 모두 고려합니다.

90초 답변

각 step이 독립인지, 이전 입력이 다음 step 계산에 영향을 주는지에 따라 state machine을 검토할 수 있습니다. 사용자가 이탈해도 draft를 복구해야 하면 local persistence나 서버 draft 저장이 필요합니다. validation은 field, section, submit 단계로 나누고 서버 오류는 사용자가 수정할 수 있는 위치에 매핑해야 합니다.

꼬리질문

질문답변 방향
dirty check는?server snapshot과 draft를 비교하거나 dirty field/row를 추적합니다.
임시 저장은?저장 주기, 충돌, 만료, 복구 UX를 함께 설계합니다.
권한이 step별로 다르면?서버 권한과 UI action visibility를 분리합니다.

오답/주의점

주의

폼 라이브러리 이름보다 상태 전이와 실패 복구를 먼저 말하세요.

SD-05

Q. 인증 기반 앱을 설계해보세요.

핵심 개념

인증 기반 앱은 로그인 화면만이 아니라 session lifecycle, token refresh, 권한별 route/action, 만료 UX, 보안 정책을 포함합니다.

개념 해부

인증 기반 앱의 핵심은 로그인 성공 화면이 아니라 session lifecycle입니다. access token 만료, refresh 실패, 401/403 구분, 권한별 action, 로그아웃, 탭 간 동기화가 모두 사용자 경험과 보안에 영향을 줍니다. client guard와 server authorization의 역할을 분리해야 합니다.

30초 답변

로그인, 세션 유지, 토큰 갱신, 로그아웃, 권한별 route/action 보호, 만료 시 복구 흐름을 설계합니다. 프론트는 사용자 경험과 route guard를 담당하고, 서버는 인증과 인가를 최종 검증합니다.

90초 답변

JWT를 쓸지 session cookie를 쓸지는 보안, revoke, scale, 운영 복잡도를 기준으로 판단합니다. access token은 짧게, refresh token은 rotation과 탈취 감지를 고려할 수 있습니다. 권한은 route뿐 아니라 API action마다 확인해야 하고, 401과 403을 UI에서 다르게 처리해야 합니다.

꼬리질문

질문답변 방향
401과 403 차이는?401은 인증 필요/만료, 403은 인증됐지만 권한 부족입니다.
토큰 갱신 실패는?요청 큐 정리, 로그인 이동, unsaved draft 보호를 봅니다.
CSRF는?cookie 기반 인증이면 SameSite, CSRF token, origin 검증을 고려합니다.

오답/주의점

주의

프론트 route guard를 보안의 전부로 말하면 위험합니다. 서버 최종 검증을 반드시 말하세요.

SD-06

Q. Excel 업로드와 validation 시스템을 설계해보세요.

핵심 개념

Excel 업로드는 파일 처리 기능이 아니라 외부 비정형 입력을 도메인 데이터로 안전하게 변환하는 workflow입니다.

개념 해부

Excel 업로드는 신뢰할 수 없는 외부 입력을 도메인 데이터로 변환하는 pipeline입니다. raw file, parsed row, normalized row, validation result를 나누면 오류 위치와 복구 흐름이 명확해집니다. 대용량이면 synchronous request가 아니라 job, progress, partial failure 모델이 필요합니다.

30초 답변

먼저 파일 크기, row 수, template version, validation 범위, 저장 단위를 확인합니다. 업로드는 parse, header mapping, preview, validation, error correction, submit 단계로 나누고, 오류는 row/cell 단위로 사용자가 수정할 수 있게 보여줍니다.

90초 답변

데이터 모델은 raw file, parsed row, normalized domain row, validation result를 분리합니다. API는 upload session 생성, parse result 조회, validation 실행, 저장 요청으로 나눌 수 있습니다. 대용량은 synchronous request로 처리하지 말고 job과 progress를 둡니다. 보안상 파일 타입, 크기, formula injection, 개인정보 로그 노출도 봐야 합니다.

꼬리질문

질문답변 방향
template 변경은?template version과 column mapping을 관리합니다.
부분 성공은?성공 row와 실패 row를 분리하고 재시도 경로를 둡니다.
대용량은?비동기 job, progress, chunk 처리, timeout 정책을 봅니다.

면접에서 말할 증거

row/cell 오류 매핑 화면, validation report, parse 실패 유형, 저장 job 상태 모델을 말합니다.

오답/주의점

주의

파일을 읽어서 저장한다고만 말하면 입력 신뢰성과 오류 복구 설계가 빠집니다.

SD-07

Q. 무한 스크롤 피드를 설계해보세요.

핵심 개념

무한 스크롤은 pagination UX가 아니라 정렬 안정성, cursor, 중복 제거, 캐시, 접근성, SEO를 함께 보는 설계 문제입니다.

개념 해부

무한 스크롤은 더 불러오기 UI가 아니라 정렬 안정성 문제입니다. offset은 중간 삽입으로 중복과 누락이 생길 수 있고, cursor는 특정 정렬 기준을 계약으로 삼습니다. 뒤로가기 복원, 접근성, SEO 필요 여부에 따라 무한 스크롤이 오히려 나쁜 선택일 수 있습니다.

30초 답변

먼저 정렬 기준, 새 글 삽입 방식, 뒤로가기 복원, SEO 필요 여부를 확인합니다. offset보다 cursor pagination을 우선 검토하고, client에서는 page cache, dedupe, loading/error/retry, scroll restoration을 설계합니다.

90초 답변

시간순 피드는 중간에 새 데이터가 들어오면 offset이 밀려 중복이나 누락이 생길 수 있습니다. cursor는 createdAt+id 같은 안정적인 기준을 씁니다. 접근성 측면에서는 keyboard 이동, footer 도달성, “더 보기” 대체 버튼을 고려합니다. 검색 노출이 중요한 콘텐츠는 SSR/SSG와 페이지형 pagination이 더 적합할 수 있습니다.

꼬리질문

질문답변 방향
중복 제거는?id 기반 dedupe와 cursor 안정성을 같이 봅니다.
뒤로가기 복원은?scroll position과 page cache를 보존합니다.
실시간 새 글은?상단 새 글 알림으로 사용자 선택을 유도합니다.

면접에서 말할 증거

cursor contract, scroll restoration 테스트, dedupe 로직, 접근성 대체 흐름을 말합니다.

오답/주의점

주의

IntersectionObserver만 말하면 데이터 정합성과 UX 복원 문제가 빠집니다.

SD-08

Q. 운영 장애 복구용 admin tool을 설계해보세요.

핵심 개념

운영 도구는 강력한 권한을 가진 위험한 UI이므로 권한, 감사 로그, dry-run, 승인, rollback 가능성이 핵심입니다.

개념 해부

운영 복구 도구는 일반 사용자 기능보다 더 강한 권한과 더 큰 피해 반경을 가집니다. 그래서 execute API보다 preview/dry-run, approval, audit, idempotency, rollback 가능성 설계가 먼저입니다. 잘못 누른 한 번의 action이 데이터 손상으로 이어지지 않도록 방어선을 여러 겹 둡니다.

30초 답변

먼저 어떤 장애를 복구하는 도구인지, 누가 실행할 수 있는지, 잘못 실행했을 때 피해가 무엇인지 묻습니다. 모든 action은 권한, 입력 검증, dry-run, confirmation, audit log, rate limit, rollback 절차를 포함해야 합니다.

90초 답변

데이터 모델은 operation, target, actor, approval, result, audit event를 분리합니다. API는 preview/dry-run과 execute를 나누고 idempotency key를 둡니다. 프론트는 영향 범위, 변경 전후, 위험도, 실행 결과를 명확히 보여줘야 합니다. 관측성은 action success/failure와 traceId를 남깁니다.

SD-09

Q. 글로벌 다국어 서비스를 설계해보세요.

핵심 개념

다국어 서비스는 문자열 치환만이 아니라 locale, timezone, currency, number format, fallback, 번역 workflow를 포함합니다.

개념 해부

다국어 서비스는 문자열 치환이 아니라 locale-aware product입니다. 날짜, 숫자, 통화, 정렬, plural, timezone, URL, SEO, 번역 QA가 모두 포함됩니다. SSR이 있다면 서버와 클라이언트의 locale 입력이 달라 hydration mismatch가 생기지 않게 해야 합니다.

30초 답변

먼저 지원 locale, 기본 언어, fallback, SEO, 동적 콘텐츠 번역 여부를 확인합니다. 프론트는 i18n key와 namespace, lazy loading, date/number format을 관리하고, 백엔드는 사용자 locale과 콘텐츠 locale을 명확히 저장합니다.

90초 답변

SSR이 있는 서비스는 서버와 클라이언트 locale이 달라 hydration mismatch가 생기지 않게 해야 합니다. 번역 key는 제품 영역별로 관리하고 미번역 fallback과 QA workflow를 둡니다. 검색/SEO가 중요하면 locale별 URL, hreflang, canonical을 설계해야 합니다.

SD-10

화이트보드 진행 순서

시스템 설계 면접은 바로 컴포넌트를 그리는 시험이 아닙니다. 면접관 신호를 보면서 요구사항을 좁히고, 비기능 요구사항과 운영 리스크를 구조화하는 과정입니다.

순서말할 내용면접관 신호
1. 요구사항 정리사용자, 핵심 시나리오, 제외 범위, 성공 기준을 질문한다“그 부분은 가정해도 됩니다”라고 하면 다음 단계로 이동
2. 비기능 요구사항트래픽, 지연, 정합성, 보안, 접근성, 운영 권한을 확인한다성능이나 장애를 물으면 운영 관점으로 확장
3. 데이터 모델핵심 entity, 상태 전이, 소유권, 감사 로그 필요성을 정리한다DB나 API 질문이 나오면 계약과 제약 조건을 구체화
4. API와 화면 흐름읽기/쓰기 API, pagination, validation, error recovery를 연결한다프론트 중심 질문이면 상태 구조와 UX 복구를 깊게 설명
5. 병목과 장애 대응캐시, 큐, retry, idempotency, rollback, observability를 붙인다scale 질문이 나오면 병목 하나를 골라 trade-off를 말한다
답변 흐름 요구사항 정리 → 비기능 요구사항 → 데이터 모델 → API와 화면 흐름 → 병목과 장애 대응 → 검증 계획 순서로 말합니다.
화이트보드에는 모든 것을 그리지 말고, 선택 기준이 드러나는 경계와 데이터 흐름만 남깁니다.
SD-11

Capacity estimation

시스템 설계에서 Capacity estimation은 정확한 숫자 맞히기가 아니라, 어떤 가정으로 병목을 추정하고 어떤 지점부터 구조가 달라지는지 말하는 훈련입니다.

항목질문답변 포인트
QPS평균과 peak traffic은 얼마인가DAU, 사용자당 요청 수, peak factor로 대략 산정합니다.
read/write ratio읽기와 쓰기 비율은 어떻게 다른가읽기가 많으면 cache/replica, 쓰기가 많으면 partition/queue/lock을 봅니다.
storage size하루 데이터 증가량은 얼마인가row 크기 × 생성량 × 보관 기간으로 추정하고 index 크기도 고려합니다.
latency budget사용자가 허용할 응답 시간은 얼마인가frontend, gateway, app, DB, external API 예산으로 나눕니다.
DB connection pool동시 요청을 DB가 감당할 수 있는가요청 시간, 쿼리 시간, pool 대기, transaction 길이를 함께 봅니다.
queue worker 수비동기 작업이 밀리지 않는가생산 속도, 처리 시간, queue lag 목표로 worker 수를 계산합니다.
고가용성·DR장애가 났을 때 얼마나 빨리 복구해야 하는가SPOF를 제거하고 RPO/RTO, active-active 필요성, failover 절차를 정합니다.
load shedding과부하 때 무엇을 포기할 수 있는가비핵심 요청 차단, degraded mode, queue backpressure로 핵심 흐름을 보호합니다.
간단 산정 예시 DAU 100,000명 × 사용자당 20 요청 = 2,000,000 requests/day
평균 QPS ≈ 2,000,000 / 86,400 = 23 QPS, peak factor 10이면 약 230 QPS
쓰기 비율이 10%라면 peak write는 약 23 QPS입니다. 이 정도는 단일 DB와 cache로 충분할 수 있지만, query가 느리거나 transaction이 길면 connection pool이 먼저 병목이 됩니다.
HA/DR 답변 고가용성·DR 질문에서는 먼저 SPOF를 찾고, RPO/RTO 목표를 정한 뒤 active-active가 필요한지 판단합니다. 모든 요청을 지키려 하기보다 load shedding으로 핵심 사용자 흐름을 보호할 수 있는지도 함께 말합니다.
SD-12

고가용성·DR

고가용성·DR 질문은 “서버를 여러 대 둔다”보다 어떤 실패를 가정하고, 어느 수준의 중단과 데이터 손실을 허용하는지 설명하는 문제입니다. 먼저 SPOF를 찾고 RPO/RTO를 숫자로 합의한 뒤 구조를 선택합니다.

주제면접에서 말할 기준주의점
SPOF 제거load balancer, app instance, DB primary, cache, queue, DNS, external API 중 단일 장애 지점을 찾습니다.컴퓨트만 이중화하고 DB, 배포, 운영 권한을 빼면 답변이 얕아집니다.
active-passive평상시 한쪽이 처리하고 장애 시 failover합니다. 운영이 단순하고 비용이 낮지만 전환 시간이 있습니다.자동 failover 기준과 split-brain 방지를 설명해야 합니다.
active-active여러 region 또는 zone이 동시에 트래픽을 처리합니다. 가용성은 높지만 데이터 정합성, routing, 비용이 복잡합니다.쓰기 충돌, session affinity, global rate limit을 함께 봅니다.
RPO/RTORPO는 허용 가능한 데이터 손실 시간, RTO는 복구까지 허용 가능한 시간입니다.“장애가 나면 복구합니다”가 아니라 수치 기준을 먼저 말해야 합니다.
backup vs replicationreplication은 빠른 전환에 유리하고 backup은 논리적 삭제·오염 복구에 필요합니다.replication만 있으면 잘못 삭제된 데이터도 빠르게 복제됩니다.
부하 방어rate limit, backpressure, graceful degradation, load shedding으로 핵심 기능을 보호합니다.모든 요청을 끝까지 처리하려 하면 장애가 연쇄적으로 커집니다.
30초 답변 고가용성 설계에서는 먼저 SPOF를 찾고, 서비스가 허용하는 RPO/RTO를 정합니다. 그 기준에 따라 multi-AZ active-passive, multi-region active-active, backup/restore, replication, failover를 선택합니다. 장애 중에는 graceful degradation, rate limit, backpressure, load shedding으로 핵심 경로를 보호합니다.
90초 답변 예를 들어 B2B admin 서비스라면 결제 승인처럼 즉시성이 큰 기능과 리포트 다운로드처럼 지연 가능한 기능을 나눕니다. 앱 서버는 multi-AZ로 수평 확장하고, DB는 primary/replica와 automated backup을 둡니다. RTO가 짧아야 하면 failover 자동화와 health check를 강화하고, RPO가 거의 0이어야 하면 동기 복제나 쓰기 경로 제약을 검토합니다. 반대로 비용이 중요하면 일부 기능은 read-only mode나 degraded mode로 낮춰 운영할 수 있습니다.
꼬리질문답변 방향
replication lag가 있으면?read-after-write가 필요한 요청은 primary로 보내고, lag metric을 보고 replica routing을 제한합니다.
DB primary 장애는?failover, connection 재시도, idempotency, transaction 중단 복구, 운영 알림을 함께 설계합니다.
외부 API 장애는?timeout, circuit breaker, fallback, queueing, 사용자 고지, 재처리 runbook을 둡니다.
장애 훈련은?backup restore drill, failover rehearsal, runbook review, alert threshold 검증을 정기적으로 합니다.

SD-14

Trade-off Matrix

면접관은 특정 기술을 골랐는지가 아니라, 왜 그 선택이 현재 제약에서 합리적인지 봅니다.

선택지유리한 경우위험·비용면접 답변 포인트
동기 API vs Queue즉시 결과가 필요하면 동기 API, 지연 허용·부하 흡수가 필요하면 queuequeue는 중복·순서·DLQ 운영이 필요합니다.사용자 대기 가능 시간과 재처리 필요성을 기준으로 고릅니다.
Cache vs DB 직조회읽기 반복과 계산 비용이 크면 cachestale data, invalidation, stampede가 생깁니다.정합성 요구와 TTL, invalidation 주체를 함께 말합니다.
SQL vs NoSQL관계·transaction·ad-hoc query는 SQL, 대규모 key-value나 유연 스키마는 NoSQLNoSQL은 쿼리 유연성과 transaction 경계가 제한될 수 있습니다.데이터 접근 패턴이 먼저고 저장소 이름은 나중입니다.
Monolith vs MSA작은 팀과 빠른 일관성은 monolith, 독립 배포와 조직 분리가 필요하면 MSAMSA는 네트워크, 관측성, 분산 트랜잭션 비용이 큽니다.팀 구조와 운영 성숙도를 같이 봅니다.
정규화 vs Denormalization쓰기 정합성이 중요하면 정규화, 읽기 성능과 projection이 중요하면 denormalizationdenormalization은 동기화와 재생성 비용이 있습니다.source of truth와 projection 재생성 방법을 말합니다.
Strong vs Eventual consistency돈·권한·재고는 strong에 가깝게, 피드·검색·통계는 eventual 가능강한 정합성은 latency와 가용성 비용이 있습니다.업무 영향과 사용자 신뢰 기준으로 나눕니다.
TRADE-OFF MATRIX 설계 답변에서는 “A를 쓰겠습니다”보다 “A는 이 제약에서 좋지만 이런 실패 모드가 있어 B와 C로 보완합니다”가 전문적입니다. 선택의 단점까지 먼저 말하면 꼬리질문에서 방어가 쉬워집니다.
SD-15

SYSTEM DESIGN PRESSURE DRILL

압박 질문에서는 예쁜 아키텍처보다 숫자 가정, 병목 추정, 정합성 선택, 실패 대응, rollout/backfill/rollback 순서를 빠르게 닫는 능력을 봅니다.

DAU→QPS→DB CONNECTION/CACHE/QUEUE/WORKER DAU 500,000명 × 사용자당 30 요청 = 15,000,000 requests/day입니다. 평균 QPS는 약 174이고 peak factor 8이면 약 1,400 QPS입니다. DB에 직접 가는 비율이 30%이고 cache hit가 70%라면 peak DB read는 약 420 QPS입니다. 요청당 DB 점유 시간이 80ms이면 필요한 동시 connection은 대략 34개지만, transaction 지연과 p95를 고려해 pool wait metric을 봅니다. 비동기 작업이 초당 300개 생성되고 worker 1개가 초당 15개 처리하면 최소 20개 worker가 필요하며, queue lag 목표와 재시도율을 보고 여유를 둡니다.
압박 질문답변 방향추가로 붙일 증거
DAU만 주어졌을 때 QPS를 어떻게 계산하나요?DAU, 사용자당 일 요청 수, 평균 QPS, peak factor, read/write ratio, payload 크기를 가정하고 병목 후보를 좁힙니다.계산식과 가정 변경 시 구조가 바뀌는 임계점을 말합니다.
DB connection이 먼저 터지면 어떻게 하나요?pool wait, slow query, transaction 길이, N+1, cache hit, read replica, queue 분리를 순서대로 봅니다.connection 수를 무작정 늘리면 DB가 더 불안정해질 수 있음을 말합니다.
cache hit가 낮으면 무엇을 확인하나요?key 설계, TTL, invalidation, user-specific cache, stampede, cacheable endpoint와 non-cacheable endpoint를 나눕니다.hit ratio, stale 신고, origin QPS, p95 latency 전후를 봅니다.
queue lag가 늘면 worker를 몇 개로 늘리나요?producer rate, worker 처리율, retry율, poison message, downstream limit을 보고 필요한 worker 수와 backpressure를 계산합니다.worker만 늘리면 외부 API quota나 DB write lock이 터질 수 있음을 설명합니다.
consistency는 어디서 강하게 잡나요?돈·권한·재고·중복 주문은 strong consistency 또는 idempotency/unique constraint를 우선하고, 피드·검색·통계는 eventual consistency를 허용합니다.사용자에게 pending/synced/failed 상태를 어떻게 보여줄지 말합니다.
failure mode는 무엇을 먼저 보나요?DB failover, cache 장애, queue 적체, external API timeout, region/AZ 장애, 배포 실패를 dependency별로 나눕니다.timeout, circuit breaker, degraded mode, runbook, alert를 붙입니다.
rollout/backfill/rollback은 어떤 순서인가요?expand schema, dual-write 또는 backward-compatible write, backfill, read switch, old path 제거 순서로 진행하고 feature flag와 canary로 blast radius를 줄입니다.rollback은 코드, config, migration, data backfill의 되돌림 가능성을 분리합니다.
면접관 압박 질문

“그 숫자는 어디서 나왔나요?”, “cache hit가 20%로 떨어지면?”, “DB connection이 100개 제한이면?”, “worker를 늘렸는데 lag가 그대로면?”, “backfill 중 구버전 앱이 쓰면?”, “rollback했는데 데이터는 이미 바뀌었으면?”에 답할 준비를 합니다.

30초 압축 답변 “먼저 DAU와 요청 수로 peak QPS를 잡고 read/write ratio, cache hit, DB connection, queue lag, worker 처리율로 병목을 좁힙니다. 그 다음 consistency가 필요한 경로와 eventual로 둘 경로를 나누고, dependency failure와 rollout/backfill/rollback 순서를 붙입니다. 숫자는 정답이 아니라 구조가 바뀌는 지점을 찾기 위한 가정입니다.”

SD-17

시스템 설계 개념 인과관계

시스템 설계 답변이 외운 티가 나는 이유는 기술 선택이 원인 없이 튀어나오기 때문입니다. 좋은 답변은 요구사항의 압력이 어떤 병목을 만들고, 그 병목 때문에 어떤 구조를 선택하며, 그 선택이 다시 어떤 비용을 만드는지 설명합니다.

SYSTEM DESIGN CAUSAL MODEL사용자가 늘면 요청 수가 늘고, 요청 수가 늘면 DB connection, slow query, cache miss, queue lag 중 하나가 먼저 병목이 됩니다. 읽기 병목이면 cache, replica, projection을 검토하고, 쓰기 병목이면 transaction 길이, lock 범위, queue 분리, idempotency를 봅니다. 하지만 cache는 stale data를 만들고, queue는 중복 처리와 지연을 만들며, replica는 read-your-writes 문제를 만듭니다. 그래서 설계 답변은 “cache를 둔다”가 아니라 “읽기 반복이 많고 stale을 몇 초 허용할 수 있어서 cache를 두되 invalidation과 fallback을 이렇게 둔다”로 이어져야 합니다.
원인메커니즘겉으로 보이는 현상선택 기준
읽기 요청이 쓰기보다 훨씬 많다같은 데이터를 반복 조회해 DB와 network round trip이 누적된다DB CPU, connection wait, p95 latency가 오른다cache, read replica, materialized view를 쓰되 stale 허용 시간을 먼저 정한다
쓰기 요청이 동시에 같은 row를 바꾼다lock wait, lost update, deadlock 가능성이 커진다timeout, retry 증가, transaction rollback이 보인다optimistic lock, unique constraint, idempotency, queue 직렬화를 비교한다
외부 API가 느리다thread와 connection이 대기 상태로 묶인다CPU는 낮은데 응답이 느리고 pool wait가 증가한다timeout budget, circuit breaker, async job, degraded mode를 둔다
데이터 마이그레이션이 크다schema lock, backfill 부하, 구버전 코드와 신버전 코드 공존 문제가 생긴다배포 중 오류, 쓰기 실패, rollback 불가 상태가 생긴다expand-contract, dual-write/read, canary, backfill 검증 순서로 닫는다
면접 답변 전환

“트래픽이 많으면 캐시를 씁니다”라고 말하지 말고 “읽기 반복이 많아 origin DB가 병목이 되고, 이 데이터는 30초 stale을 허용할 수 있으므로 cache-aside를 둡니다. 대신 권한과 결제 상태는 stale 허용치가 낮아 cache하지 않거나 짧은 TTL과 명시적 purge를 둡니다”처럼 인과관계와 예외를 함께 말합니다.

SD-18

완성형 시스템 설계 사례: 대용량 B2B 관리자 데이터 처리

전문가 수준의 시스템 설계 답변은 여러 주제를 따로 말하지 않습니다. 하나의 문제를 두고 요구사항, 트래픽, 데이터 모델, API, DB, 캐시, 큐, 장애, 배포를 같은 인과관계 안에서 연결합니다.

EXPERT SYSTEM DESIGN CASE예제 요구사항은 “B2B 고객사가 수십만 건의 거래 데이터를 업로드하고, 운영자가 검증 오류를 수정하며, 대시보드에서 필터·집계·다운로드를 수행하는 관리자 시스템”입니다. 이 문제는 단순 CRUD가 아닙니다. 업로드는 비정형 입력을 도메인 데이터로 바꾸는 pipeline이고, 검증은 row/cell 오류를 사용자에게 되돌려야 하며, 조회는 대량 데이터와 권한, audit, export job을 함께 다뤄야 합니다. 따라서 설계는 UI 화면이 아니라 데이터가 어떤 상태를 거쳐 신뢰 가능한 데이터가 되는지부터 시작합니다.

1. 요구사항을 닫는 순서

먼저 사용자와 성공 기준을 좁힙니다. 운영자는 파일을 업로드하고, 시스템은 template version을 확인하고, parsing 결과를 preview로 보여주고, validation error를 row/cell 단위로 표시하며, 사용자가 수정한 뒤 저장합니다. 비기능 요구는 더 중요합니다. 업로드 파일은 최대 몇 MB인지, row 수는 1만인지 100만인지, 검증이 즉시 끝나야 하는지, 일부 성공을 허용하는지, tenant별 데이터가 완전히 격리되어야 하는지, audit log가 법적 증거인지 확인합니다. 이 질문을 먼저 하지 않으면 이후 기술 선택은 근거 없는 나열이 됩니다.

2. 데이터 모델과 상태 전이

핵심 entity는 upload_session, raw_file, parsed_row, validation_error, domain_record, audit_event로 나눕니다. raw file은 원본 증거이고, parsed row는 아직 신뢰할 수 없는 중간 표현이며, domain record는 검증을 통과해 업무 규칙을 만족한 최종 데이터입니다. 이 구분이 없으면 “파일은 올라갔는데 저장은 실패한 상태”, “일부 row만 성공한 상태”, “검증 규칙이 바뀐 뒤 재검증해야 하는 상태”를 설명할 수 없습니다. 상태 전이는 UPLOADED → PARSED → VALIDATED_WITH_ERRORS | VALIDATED → COMMITTED | FAILED처럼 명시하고, 각 전이에 actor와 timestamp를 남깁니다.

3. 동기 API와 비동기 job의 경계

파일 업로드 요청에서 parsing과 validation을 끝내려 하면 request timeout, memory pressure, DB connection 점유가 발생합니다. 그래서 업로드 API는 session을 만들고 파일을 object storage에 저장한 뒤 job id를 반환합니다. parsing worker는 파일을 읽어 row 단위로 저장하고, validation worker는 rule engine 또는 validator set을 적용합니다. 사용자는 polling, SSE, WebSocket 중 요구사항에 맞는 방식으로 progress를 봅니다. 여기서 queue를 넣는 이유는 “비동기가 멋져서”가 아니라 사용자 요청 경로에서 긴 작업을 분리하고, 실패를 재시도·DLQ·운영자 확인이 가능한 상태로 바꾸기 위해서입니다.

4. Capacity estimation과 병목 추론

예를 들어 하루 1,000개 tenant가 각각 100,000 row 파일을 2개씩 올리면 하루 2억 row를 parsing해야 합니다. 평균으로 나누면 작아 보이지만 업무 시간 peak에 몰릴 수 있습니다. row당 validation이 3ms면 단일 worker는 초당 약 333 row를 처리하고, 100,000 row 파일 하나는 약 5분이 걸립니다. 100개 파일이 동시에 들어오면 worker 수, DB write throughput, object storage read bandwidth, validation rule CPU가 병목 후보가 됩니다. 이때 worker만 늘리면 DB write lock이나 connection pool이 먼저 터질 수 있으므로 queue lag, worker throughput, DB write latency, connection pool wait를 함께 봅니다.

5. 조회와 인덱스 설계

운영 목록 화면은 tenant, status, date range, error type, uploader 같은 조건으로 조회합니다. (tenant_id, status, created_at) 같은 복합 인덱스는 equality 조건과 range/order 조건 순서를 보고 설계합니다. 하지만 모든 필터 조합에 인덱스를 만들면 쓰기 비용과 저장 공간이 폭증합니다. 자주 쓰는 조회 경로는 execution plan으로 검증하고, export는 화면 조회 API와 분리해 background job으로 처리합니다. offset pagination은 뒤 페이지에서 비용이 커질 수 있으므로 운영 목록은 cursor 또는 keyset pagination을 검토합니다.

6. 정합성과 실패 복구

저장 단계에서 “검증된 row만 domain_record로 commit”하는 것은 transaction 경계를 신중히 잡아야 합니다. 파일 하나 전체를 하나의 transaction으로 묶으면 lock과 undo/WAL이 커지고 실패 시 복구가 어렵습니다. batch 단위 commit을 쓰면 부분 성공이 생기므로 session 상태와 failed batch를 추적해야 합니다. 같은 파일을 사용자가 두 번 제출할 수 있으므로 upload checksum, idempotency key, unique constraint를 둡니다. commit 후 알림이나 downstream 이벤트가 필요하면 DB 변경과 이벤트 발행 사이 간극을 줄이기 위해 outbox를 둡니다.

7. 보안과 권한

프론트에서 tenant id를 보내더라도 서버는 principal 기준으로 tenant scope를 재계산해야 합니다. 파일에는 formula injection, 개인정보, 악성 payload가 들어올 수 있으므로 파일 크기, MIME, 확장자, sheet 수, formula escape, virus scan 필요성을 검토합니다. 다운로드 URL은 presigned URL을 쓰더라도 TTL, scope, audit log가 필요합니다. 관리자 action은 actor, tenant, resource, action, before/after, result, trace id를 audit event로 남깁니다.

8. 배포와 마이그레이션

validation rule 저장 구조를 바꾸거나 upload 상태 enum을 추가할 때는 구버전 worker와 신버전 worker가 동시에 떠 있을 수 있습니다. 먼저 nullable column이나 새 테이블을 추가하고, 구버전이 읽어도 깨지지 않는 형태로 dual-write 또는 backward-compatible write를 적용합니다. backfill 후 read path를 전환하고, 지표가 안정되면 옛 필드를 제거합니다. rollback은 코드만 되돌리는 것이 아니라 이미 생성된 새 상태와 데이터가 구버전에서 어떻게 보이는지까지 정의해야 합니다.

전문가 답변의 핵심

이 사례에서 좋은 답변은 “S3, SQS, Redis, RDS를 씁니다”가 아닙니다. “파일 업로드는 긴 작업이므로 요청 경로에서 분리하고, row 상태를 분리해 오류 복구를 가능하게 하며, tenant 격리와 audit을 서버에서 보장하고, worker 확장은 DB write throughput과 queue lag를 함께 보며, schema 변경은 expand-contract로 배포합니다”처럼 제약에서 구조가 도출되는 답변입니다.
