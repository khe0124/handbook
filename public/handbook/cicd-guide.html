CICD-01

CI/CD가 하는 일

한 줄로: CI는 "합칠 때마다 자동으로 빌드·테스트해 깨졌는지 검증", CD는 "검증 통과한 코드를 자동으로 배포". 사람이 손으로 하던 반복을 기계에 넘기는 것입니다.

CI(Continuous Integration)는 여러 사람이 각자 작업한 코드를 자주 합치되, 합칠 때마다 자동으로 빌드하고 테스트를 돌려 깨진 채로 합쳐지는 것을 막습니다. CD(Continuous Delivery/Deployment)는 그렇게 검증된 산출물을 스테이징·프로덕션으로 자동 배포합니다. 둘은 하나의 흐름이라 보통 CI/CD라 붙여 부릅니다.

수동 배포의 문제CI/CD가 주는 것
매번 사람이 빌드·복사·재시작 → 단계 빠뜨림·오타스크립트로 고정 → 항상 같은 순서로 실행
"내 컴퓨터에선 됐는데" → 재현 불가깨끗한 runner에서 매번 동일하게 빌드
테스트를 깜빡하고 배포 → 운영 장애테스트 통과해야만 배포(게이트, → CICD-06)
배포 기록이 사람 머릿속에만 있음누가·언제·무엇을 배포했는지 로그로 남음

전형적인 파이프라인은 푸시 한 번으로 다음 단계들이 줄줄이 실행됩니다. 각 단계는 앞 단계가 성공해야만 다음으로 넘어갑니다 — 테스트가 깨지면 거기서 멈추고 이미지·배포로 가지 않습니다.

git push→ 빌드→ 테스트→ 이미지 빌드·푸시→ 배포
이 가이드의 범위

여기선 파이프라인을 어떻게 짜는가만 다룹니다. Docker 이미지의 내부 구조, 컨테이너 런타임, 배포 대상(서버·쿠버네티스) 자체의 동작은 깊이 들어가지 않습니다(→ Docker·배포 핸드북). CI/CD는 "그 도구들을 자동으로 호출하는 오케스트레이션 레이어"라고 보면 됩니다.

CICD-02

GitHub Actions 기본

GitHub Actions의 작업 단위는 workflow YAML 파일 하나입니다. 리포지토리 안 정해진 위치에 두면, GitHub가 트리거 조건에 맞춰 알아서 실행합니다.

워크플로우 파일은 반드시 .github/workflows/ 디렉터리 안에 *.yml(또는 *.yaml)로 둬야 합니다 — 이 경로가 아니면 GitHub가 인식하지 않습니다. 파일 하나가 워크플로우 하나이고, 한 리포에 여러 개를 둘 수 있습니다(예: ci.yml, deploy.yml).

WORKFLOW 의 골격 — .github/workflows/ci.yml name: CI # Actions 탭에 표시될 이름
on: # 언제 돌릴지 — 트리거
  push:
    branches: [main]
jobs: # 무엇을 할지 — 하나 이상의 job
  build: # job 이름(자유)
    runs-on: ubuntu-latest # 어떤 runner(가상머신)에서
    steps: # 순서대로 실행되는 단계들
      - uses: actions/checkout@v4 # 남이 만든 action 호출
      - run: echo "hello" # 셸 명령 직접 실행
키워드의미 / 역할
on트리거 — 언제 워크플로우를 돌릴지. push, pull_request, workflow_dispatch(수동), schedule(주기) 등
jobs워크플로우가 하는 일의 묶음. job끼리는 기본 병렬, needs로 순서 지정
runs-onjob이 돌 runner 지정 — ubuntu-latest가 표준. 매 실행마다 깨끗한 새 VM
stepsjob 안에서 위에서 아래로 순차 실행. 하나라도 실패하면 그 job 중단
uses마켓플레이스의 재사용 action 호출 (예: actions/checkout@v4)
runrunner 셸에서 명령 직접 실행 (예: ./gradlew build)
runner = 매번 새로 만들어지는 깨끗한 가상머신

runner는 job이 시작될 때 생성되고 끝나면 버려지는 일회용 VM입니다. 그래서 "내 로컬에 깔려 있던 것"에 의존할 수 없고, 필요한 런타임(JDK 등)은 매번 step에서 설치해야 합니다. 이게 CI의 핵심 가치인 재현성의 출처입니다 — 항상 동일한 빈 상태에서 시작하니까요. 대신 매번 새로 설치하면 느려지므로 캐시로 보완합니다(→ CICD-03).

CICD-03

빌드·테스트 워크플로우

가장 먼저 만들 워크플로우는 "푸시·PR마다 빌드하고 테스트를 돌리는" CI입니다. Spring Boot + Gradle 기준으로 실제 동작하는 ci.yml을 통째로 봅니다.

.github/workflows/ci.yml — 빌드 & 테스트 name: CI

on:
  push:
    branches: [main]
  pull_request: # PR이 열릴 때마다 검증 → 깨진 코드 머지 차단

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4 # 리포 코드를 runner로 가져옴

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          distribution: temurin # Eclipse Temurin JDK 배포판
          java-version: '21'
          cache: gradle # Gradle 의존성 캐시 자동 — 재설치 시간 단축

      - name: Grant execute permission
        run: chmod +x ./gradlew # 래퍼 실행 권한 부여

      - name: Build & Test
        run: ./gradlew build # build는 test를 포함 — 컴파일+테스트 한 번에
step무엇을 / 왜
actions/checkout@v4리포 소스를 runner로 클론. 이게 없으면 코드가 없어 아무것도 못 함. 거의 모든 워크플로우의 첫 step
actions/setup-java@v4지정 JDK를 runner에 설치하고 JAVA_HOME 설정. distribution: temurin이 가장 흔함
cache: gradlesetup-java 내장 옵션. ~/.gradle 의존성을 캐싱해 다음 실행을 단축. 왜: runner는 매번 비어 있어서 캐시 없으면 매번 전부 다운로드
./gradlew buildGradle wrapper로 컴파일+테스트. build는 test를 의존하므로 테스트 실패 시 전체 실패 → 빨간 X
왜 gradlew(wrapper)인가

전역 gradle이 아니라 리포에 포함된 ./gradlew를 쓰는 이유는 Gradle 버전을 리포가 고정하기 때문입니다 — runner든 동료 노트북이든 같은 Gradle로 빌드되어 재현성이 보장됩니다. test만 따로 돌리려면 ./gradlew test, 빌드 캐시 문제를 의심하면 --no-daemon을 붙이는 정도만 알아두면 됩니다.

CICD-04

이미지 빌드·푸시

테스트가 통과하면 배포할 산출물을 만듭니다. 컨테이너 기반 배포라면 그 산출물은 Docker 이미지이고, 레지스트리(이미지 저장소)에 푸시합니다.

흐름은 셋입니다 — ① 레지스트리에 로그인, ② 이미지 빌드, ③ 푸시. GitHub Actions에선 공식 docker/build-push-action이 빌드와 푸시를 한 step으로 묶어줍니다. 로그인은 docker/login-action을 씁니다.

.github/workflows/deploy.yml — 이미지 빌드 & 푸시 (job 일부)   docker:
    needs: build # build job 성공 후에만 — 테스트 통과 게이트
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Log in to registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io # GitHub Container Registry
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }} # 시크릿 — 평문 금지(→CICD-05)

      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: | # 두 태그를 동시에 — 추적용 sha + 최신 latest
            ghcr.io/${{ github.repository }}:${{ github.sha }}
            ghcr.io/${{ github.repository }}:latest
요소의미 / 왜
needs: buildbuild job이 성공해야만 이 job 시작 → 테스트 깨지면 이미지 안 만듦
docker/login-action@v3레지스트리 인증. credential은 반드시 ${{ secrets.X }}로 주입
docker/build-push-action@v6빌드+푸시를 한 번에. context는 Dockerfile 위치, push: true면 레지스트리로 전송
:${{ github.sha }}커밋 SHA 태그 — 어떤 코드로 만든 이미지인지 정확히 추적. 롤백·디버깅의 핵심
:latest"가장 최신" 포인터. 편하지만 어떤 버전인지 모호 — sha 태그와 병행
latest 태그만 쓰지 말 것

:latest 하나만 쓰면 "지금 운영에 떠 있는 게 정확히 어떤 커밋인지" 알 수 없고, 롤백할 과거 버전을 가리킬 수도 없습니다. 항상 ${{ github.sha }} 같은 불변 태그를 함께 박으세요 — 사고가 나면 "그 sha로 다시 배포"가 가능해집니다. 이미지 빌드 자체(Dockerfile 작성·레이어 최적화)는 → Docker 핸드북.

CICD-05

시크릿·환경

토큰·비밀번호·DB 접속정보는 절대 YAML이나 코드에 평문으로 쓰지 않습니다. GitHub repo Secrets에 저장하고, 워크플로우에선 ${{ secrets.X }}로 참조만 합니다.

저장 위치는 리포의 Settings → Secrets and variables → Actions입니다. 여기 등록한 값은 암호화되어 보관되고, 워크플로우 실행 시점에만 주입되며, 로그에 찍히면 ***로 자동 마스킹됩니다.

시크릿 참조 — 평문 금지 # 나쁜 예 — 절대 금지
  password: ghp_aBcD1234realtoken # 평문 노출 = 즉시 유출

# 좋은 예 — Secrets 참조
  password: ${{ secrets.DOCKER_TOKEN }}

# step에 환경변수로 주입
  - name: Deploy
    env:
      DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
    run: ./deploy.sh
종류의미 / 언제
${{ secrets.X }}리포에 등록한 암호화 시크릿. credential·토큰·키 전부 이걸로
${{ secrets.GITHUB_TOKEN }}GitHub가 자동 제공하는 토큰. 같은 리포의 ghcr 푸시 등에 별도 등록 없이 사용
Environment secretsenvironment별로 분리 저장(staging/production). 같은 이름 다른 값 + 승인 게이트(→CICD-06)
${{ vars.X }}비밀 아닌 설정값(Variables). API URL·플래그 등. 마스킹 안 됨
절대 하지 말 것

① 토큰·비밀번호를 YAML·소스코드·커밋 메시지에 평문으로 쓰지 마세요 — 한 번 푸시되면 git 히스토리에 영구히 남아, 지워도 유출로 간주하고 즉시 폐기·재발급해야 합니다. ② 시크릿을 echo "$TOKEN"으로 일부러 출력하지 마세요 — 마스킹을 우회하면 로그에 새어 나갑니다. ③ PR을 fork에서 받는 공개 리포는 시크릿 접근 정책을 따로 확인하세요(악의적 PR이 시크릿을 빼갈 수 있는 표면).

CICD-06

배포 트리거·전략

"언제 배포할 것인가"를 정하는 게 CD 설계의 핵심입니다. main 푸시마다? 태그를 달 때만? 사람 승인을 거쳐서? — 트리거와 게이트로 표현합니다.

트리거 패턴들 — on: # 1) main에 머지되면 staging 배포
on:
  push:
    branches: [main]

# 2) v1.2.3 같은 버전 태그를 달면 production 배포
on:
  push:
    tags: ['v*'] # 의도적 릴리스만 트리거 → 실수 배포 방지

# 3) 버튼으로 수동 실행
on: workflow_dispatch

배포 job에 environment를 지정하면 GitHub의 승인 게이트를 걸 수 있습니다 — production environment에 "required reviewers"를 설정해두면, 배포 job이 그 지점에서 멈추고 지정된 사람이 승인해야 진행됩니다.

승인 게이트 + 테스트 통과 게이트   deploy-prod:
    needs: [build, docker] # 빌드·테스트·이미지 다 성공해야만
    runs-on: ubuntu-latest
    environment: production # 이 환경에 승인 게이트 설정 → 사람 승인 대기
    steps:
      - run: ./deploy.sh ${{ github.sha }}
전략 / 장치의미 / 왜
branch 트리거main 머지 = 자동 staging 배포. 빠른 피드백, 낮은 위험
tag 트리거v* 태그 = production 배포. 의도적 릴리스만 — 실수로 운영 안 나감
승인 게이트environment + required reviewers. 운영 배포 전 사람의 마지막 확인
테스트 통과 게이트needs:로 배포를 빌드·테스트 job 뒤에 묶음 → 깨진 코드는 배포 불가
롤백과거 sha 이미지로 재배포. 그래서 sha 태그가 필수(→CICD-04)
롤백은 "이전 sha로 다시 배포"

새 배포가 잘못되면, 빌드를 새로 하지 않고 직전에 잘 돌던 커밋 sha 이미지를 다시 배포하는 게 가장 빠르고 안전한 롤백입니다 — 이미 검증된 산출물이니까요. 그래서 모든 이미지에 불변 sha 태그를 박아두는 게 롤백 전략의 전제 조건입니다. blue-green·canary 같은 무중단 배포 전략은 → 배포 핸드북.

CICD-07

파이프라인 베스트 프랙티스

파이프라인은 짜는 것보다 유지하는 것이 일입니다. 느리거나 자주 깨지는 CI는 사람들이 우회하기 시작하면서 가치가 사라집니다. 몇 가지 원칙.

원칙왜 / 어떻게
빠른 피드백 우선빠른 검사(lint·unit test)를 앞에, 느린 것(E2E·이미지 빌드)을 뒤에. 깨질 거면 빨리 깨져야 개발자 대기 시간이 준다
캐시로 시간 단축의존성(cache: gradle)·Docker 레이어를 캐싱. runner는 매번 비어서 캐시 없으면 매번 풀다운로드
job 병렬화독립적인 검사(테스트·lint·보안 스캔)는 별도 job으로 동시 실행. 전체 시간 단축
실패 시 후속 차단needs:로 의존을 명시 → 앞이 깨지면 뒤(배포)는 시작 안 함. 깨진 걸 내보내지 않는 안전장치
비밀 노출 주의시크릿은 항상 ${{ secrets.X }}, 로그에 출력 금지, fork PR 권한 확인(→CICD-05)
워크플로우도 코드YAML 변경도 PR·리뷰 대상. action은 버전 핀(@v4)으로 고정해 갑작스런 동작 변화 방지
병렬 검사 + 게이트 결합 예시 jobs:
  test: # ┐
    runs-on: ubuntu-latest # │ 둘이 동시에 돌아(병렬)
  lint: # ┘ 전체 시간 = 둘 중 느린 쪽
    runs-on: ubuntu-latest
  deploy:
    needs: [test, lint] # 둘 다 통과해야만 배포 — 게이트
    runs-on: ubuntu-latest
느린 CI는 죽은 CI

CI가 매번 20분 걸리면 개발자는 결과를 안 기다리고 다음 작업으로 넘어가버리고, 깨진 빌드가 방치됩니다. 캐시·병렬화로 핵심 검사를 몇 분 안에 끝내는 걸 목표로 하세요. 그리고 flaky한(가끔 실패하는) 테스트는 즉시 격리하거나 고치세요 — "또 그 테스트네" 하고 빨간 X를 무시하기 시작하면 CI 신뢰가 통째로 무너집니다.

CICD-08

치트시트 — GitHub Actions 핵심

실무에서 반복 참조하는 조각들. 전부 외우지 말고 "이런 게 있다"만 알아두고 필요할 때 찾으세요.

워크플로우 위치where
.github/workflows/*.yml — 이 경로 아니면 GitHub가 인식 안 함. 파일 하나 = 워크플로우 하나
트리거on
on: push / on: pull_request / on: workflow_dispatch(수동) / on: { push: { tags: ['v*'] } }(태그)
job 골격jobs / runs-on
jobs.build.runs-on: ubuntu-latest + steps:. job끼리 기본 병렬, needs:로 순서
코드 체크아웃checkout
- uses: actions/checkout@v4 — 거의 모든 워크플로우의 첫 step. 없으면 코드가 없음
JDK 설치setup-java
actions/setup-java@v4 + with: { distribution: temurin, java-version: '21', cache: gradle }
Gradle 빌드·테스트gradlew
./gradlew build (test 포함) / ./gradlew test. wrapper로 버전 고정 → 재현성
의존성 캐시cache: gradle
setup-java 내장 cache: gradle. runner는 매번 비어서 캐시로 재설치 시간 단축
레지스트리 로그인login-action
docker/login-action@v3 + registry / username / password: ${{ secrets.X }}
이미지 빌드·푸시build-push-action
docker/build-push-action@v6 + push: true, tags:에 sha와 latest 병행
시크릿 참조secrets
${{ secrets.NAME }} — Settings→Secrets에 등록. 평문 금지, 로그 자동 마스킹. ${{ secrets.GITHUB_TOKEN }}은 자동 제공
실행 순서·게이트needs
needs: [build, test] — 지정 job 성공 후에만 실행. "테스트 통과해야만 배포"의 핵심
환경·승인 게이트environment
environment: production + required reviewers → 운영 배포 전 사람 승인 대기. env별 시크릿 분리
버전 태그 트리거tags
on: { push: { tags: ['v*'] } } — 의도적 릴리스만 production 트리거
롤백rollback
과거 :${{ github.sha }} 이미지로 재배포. 새 빌드 없이 검증된 산출물로 복귀
학습 순서 제안

한 번에 다 만들지 마세요. ① 먼저 ci.yml로 빌드·테스트만 자동화(CICD-03) — 여기서 PR마다 빨간/초록이 뜨는 것만으로도 가치가 큽니다. ② 그다음 이미지 빌드·푸시(CICD-04)와 시크릿(CICD-05)을 붙이고, ③ 마지막으로 배포 트리거·승인 게이트(CICD-06)를 올립니다. 각 단계가 독립적으로 동작하니 점진적으로 쌓으세요. 이미지·배포 대상 자체의 동작은 → Docker·배포 핸드북에서 이어집니다.
