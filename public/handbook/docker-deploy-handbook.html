컨테이너는 "제 PC에선 됩니다"를 끝냅니다 — 코드·런타임·의존성을 한 이미지로 봉인해 어디서나 같게 돕니다. 이 핸드북은 명령어 나열이 아니라 왜 이렇게 도는가를 먼저 잡습니다 — 레이어 캐시가 왜 그 순서를 강요하는지, 배포 전략 사이의 트레이드오프가 무엇인지, 어디서 깨지는지.

OPS-01

컨테이너란

컨테이너의 핵심은 가상화가 아니라 격리입니다 — 하나의 OS 커널을 공유하면서, 프로세스가 자기만의 파일시스템·네트워크·프로세스 공간을 가진 것처럼 속이는 기술입니다. VM과 헷갈리면 무게도 한계도 잘못 잡게 됩니다.

VM은 하이퍼바이저 위에 guest OS를 통째로 올립니다 — 커널, 시스템 데몬, 전부. 그래서 부팅에 수십 초가 걸리고 이미지가 기가바이트 단위입니다. 컨테이너는 그게 없습니다. host의 커널을 그대로 빌려 쓰고, Linux 커널의 두 기능으로 격리만 만듭니다 — namespace(무엇을 볼 수 있는가: PID·네트워크·마운트 공간 분리)와 cgroup(얼마나 쓸 수 있는가: CPU·메모리 한도). 그래서 컨테이너는 사실 격리된 일반 프로세스일 뿐입니다.

축가상머신(VM)컨테이너
격리 단위guest OS 통째로 (커널 포함)프로세스 (namespace·cgroup)
커널VM마다 독립 커널host 커널 공유
크기GB 단위MB 단위 (베이스에 따라 수~수백 MB)
기동 시간수십 초 (OS 부팅)수백 ms (프로세스 시작)
격리 강도강함 (하드웨어 수준)약함 (커널 공유 → 커널 취약점 공유)
이식성하이퍼바이저 의존이미지 하나로 어디서나 동일
실패 모드 — 커널 공유의 함정

컨테이너는 host 커널을 공유하므로, 커널이 다른 OS는 native로 못 돕니다. macOS·Windows에서 Docker가 도는 건 사실 가벼운 Linux VM을 한 겹 깔고 그 안에서 컨테이너를 돌리는 것입니다 — 그래서 로컬(맥)과 서버(리눅스)의 동작이 미묘하게 다를 수 있습니다. 또 격리가 VM만큼 강하지 않으니, 멀티테넌트 환경에서 신뢰 못 할 코드를 돌릴 땐 컨테이너만으로 보안 경계를 삼지 마세요.

OPS-02

이미지와 레이어

세 단어를 정확히 구분하세요 — 이미지는 불변 스냅샷(설계도), 레이어는 그 이미지를 쌓아 올린 캐시 단위, 컨테이너는 이미지를 실행한 인스턴스입니다. 클래스와 인스턴스의 관계를 떠올리면 정확합니다.

이미지는 읽기 전용 레이어의 스택입니다. Dockerfile의 명령 하나(RUN, COPY 등)가 대체로 레이어 하나를 만들고, 각 레이어는 직전 상태에서 바뀐 파일만 담습니다. 컨테이너를 실행하면 이 읽기 전용 스택 위에 얇은 쓰기 가능 레이어가 한 장 얹히고, 실행 중 생기는 변경은 전부 거기로 갑니다. 컨테이너를 지우면 그 레이어도 사라집니다 — 그래서 상태는 컨테이너 안에 두면 안 됩니다(→ OPS-07).

LAYER STACK — 위로 갈수록 자주 바뀜 ┌─────────────────────────────┐
│ RW 레이어 (컨테이너 실행 중 변경) │ ← 컨테이너마다 1장
├─────────────────────────────┤
│ COPY . . (앱 소스) │ ← 매 빌드 바뀜
│ RUN npm ci (의존성) │ ← 가끔 바뀜
│ COPY package.json │ ← 드물게 바뀜
│ FROM node:22-slim (베이스) │ ← 거의 안 바뀜
└─────────────────────────────┘ ↑ 읽기 전용, 공유·캐시됨

레이어가 캐시 단위라는 점이 빌드 속도의 전부를 결정합니다. Docker는 빌드 시 각 레이어를 위에서부터 검사해 입력이 그대로면 캐시를 재사용합니다. 그러나 한 레이어가 깨지면 그 아래(이후) 레이어는 전부 다시 빌드됩니다. 그래서 "자주 바뀌는 것을 아래쪽(나중)에" 두는 순서가 중요합니다(→ OPS-03). 여러 이미지가 같은 베이스 레이어를 쓰면 디스크에서 그 레이어를 한 번만 저장합니다 — content-addressable(해시 기반) 저장 덕입니다.

멘탈 모델

이미지 : 컨테이너 = 클래스 : 인스턴스. 한 이미지에서 컨테이너 100개를 띄워도 읽기 전용 레이어는 공유되고, 각자 얇은 RW 레이어만 따로 가집니다. docker images로 이미지를, docker ps로 실행 중 컨테이너를, docker history <image>로 레이어 구성을 봅니다.

OPS-03

Dockerfile 기초·베스트 프랙티스

Dockerfile은 이미지를 만드는 레시피입니다. 명령 몇 개만 알면 동작은 하지만, 순서와 선택이 빌드 속도·이미지 크기·보안을 가릅니다.

명령역할 / 주의
FROM베이스 이미지 지정. 모든 Dockerfile의 첫 줄. 태그를 명시(node:22-slim)
WORKDIR이후 명령의 작업 디렉터리 설정(없으면 생성). cd 대신 이걸 쓴다
COPYhost 파일을 이미지로 복사. ADD는 URL·자동 압축해제까지 하지만 의외성 때문에 보통 COPY 권장
RUN빌드 중 셸 명령 실행(레이어 생성). 패키지 설치·빌드 등
CMD컨테이너 기동 시 기본 명령. docker run 인자로 덮어쓸 수 있음
ENTRYPOINT고정 실행 진입점. CMD는 그 인자로 들어감 — 둘을 조합

가장 큰 레버는 레이어 캐시를 살리는 명령 순서입니다. 의존성 설치(npm ci)는 소스 코드가 바뀌어도 package.json이 그대로면 다시 할 필요가 없습니다. 그래서 의존성 매니페스트만 먼저 COPY → 설치 → 그다음 전체 소스 COPY 순으로 짭니다. 이러면 코드 한 줄 고친 재빌드에서 의존성 설치 레이어가 캐시로 건너뛰어집니다.

DOCKERFILE — Node 앱 (캐시·보안 고려) FROM node:22-slim
WORKDIR /app
# 1) 의존성 매니페스트만 먼저 — 코드보다 드물게 바뀜
COPY package.json package-lock.json ./
RUN npm ci --omit=dev
# 2) 그다음 소스 — 자주 바뀌므로 캐시 경계를 여기 둔다
COPY . .
# 3) 비root 유저로 강등 (slim 이미지는 node 유저 내장)
USER node
EXPOSE 3000
CMD ["node", "server.js"]

두 가지를 더 챙깁니다. 첫째, .dockerignore로 node_modules·.git·로컬 빌드 산출물을 빌드 컨텍스트에서 제외하세요 — 안 그러면 거대한 컨텍스트를 daemon으로 통째로 전송하고, 로컬 node_modules가 이미지로 섞여 들어가 깨집니다. 둘째, USER로 비root로 떨어뜨리세요. 기본 root로 도는 컨테이너는 탈출 시 공격면이 커집니다.

CMD vs ENTRYPOINT — 자주 틀리는 곳

CMD는 "기본값", ENTRYPOINT는 "고정 실행파일"입니다. docker run myimg echo hi를 하면 CMD는 통째로 echo hi로 교체되지만, ENTRYPOINT가 있으면 echo hi는 그 인자로 붙습니다. 또 CMD node server.js(셸 형식)는 /bin/sh -c로 감싸져 신호(SIGTERM)가 앱까지 안 전달될 수 있습니다 — graceful shutdown이 깨지는 단골 원인입니다. CMD ["node","server.js"](exec 배열 형식)을 쓰세요(→ OPS-05의 신호 처리).

OPS-04

멀티스테이지 빌드와 이미지 최적화

빌드에 필요한 것과 실행에 필요한 것은 다릅니다. 컴파일러·devDependencies·빌드 캐시는 실행 시점엔 쓰레기입니다. 멀티스테이지 빌드는 이 둘을 분리해 최종 이미지에서 빌드 도구를 통째로 들어냅니다.

핵심 문법은 하나입니다 — Dockerfile 안에 FROM을 여러 번 써서 단계(stage)를 나누고, COPY --from=<stage>로 앞 단계의 결과물만 다음 단계로 가져옵니다. 빌드 단계는 무거운 SDK를 다 깔아 빌드하고, 런타임 단계는 가벼운 베이스에 산출물만 얹습니다. 빌드 단계는 최종 이미지에 포함되지 않습니다.

MULTI-STAGE — 빌드 / 런타임 분리 # --- 1단계: 빌드 (무거운 SDK 포함) ---
FROM node:22 AS build
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build # → /app/dist 생성

# --- 2단계: 런타임 (가벼운 베이스, 산출물만) ---
FROM node:22-slim
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
USER node
CMD ["node", "dist/server.js"]

베이스 이미지 선택이 크기와 공격면을 함께 줄입니다. node:22(full, ~1GB)보다 node:22-slim(~200MB)이, 더 극단적으로는 distroless(셸·패키지매니저조차 없는 최소 런타임)나 Alpine 계열이 있습니다. 작을수록 다운로드·기동이 빠르고, 깔린 패키지가 적으니 CVE 노출도 줄어듭니다.

베이스크기 감트레이드오프
node:22 (full)크다 (~1GB)디버깅 편함(셸·도구 다 있음), 그러나 무겁고 공격면 큼
node:22-slim중간 (~200MB)실전 기본값. glibc 기반이라 호환성 무난
Alpine작다 (~50MB)musl libc — native 모듈 빌드 이슈가 가끔 터짐
distroless가장 작다셸 없음 → docker exec 디버깅 불가, 보안 최상
왜 효과가 큰가

멀티스테이지의 진짜 이득은 크기보다 분리된 관심사입니다 — 빌드 도구(컴파일러·devDeps)가 production 이미지에 절대 못 들어가니, 공급망 공격면이 구조적으로 줄어듭니다. 디버깅이 어려운 distroless가 부담되면 slim에서 출발하고, 이미지 취약점은 docker scout나 trivy 같은 스캐너로 CI에서 점검하세요(→ OPS-08).

OPS-05

컨테이너 실행과 구성

이미지를 만들었으면 이제 돌립니다. 실행 시점의 4대 구성 축 — 포트·볼륨·환경변수·네트워크 — 만 잡으면 로컬에서 운영까지 같은 그림으로 다룹니다.

축플래그 / 의미
포트 매핑-p 3000:3000 — host:container. 컨테이너 안의 포트를 밖으로 노출
볼륨-v ./data:/app/data — 컨테이너 밖에 상태 보존(named volume / bind mount)
환경변수-e NODE_ENV=production 또는 --env-file .env — 설정 주입(→ OPS-07)
네트워크--network mynet — 같은 네트워크의 컨테이너끼리 이름으로 통신
실행/정리-d(백그라운드) --rm(종료 시 삭제) --name(이름 부여)

로컬에서 DB·API·캐시 여러 서비스를 동시에 띄워야 하면 docker run을 손으로 여러 번 치는 대신 Docker Compose를 씁니다 — compose.yaml에 서비스·네트워크·볼륨을 선언하고 docker compose up 한 번으로 전부 띄웁니다. Compose가 자동으로 네트워크를 만들어, 서비스끼리 서비스 이름을 호스트명처럼 부를 수 있습니다(예: 앱에서 DB 호스트를 db로).

compose.yaml — 앱 + Postgres services:
  app:
    build: .
    ports: ["3000:3000"]
    environment:
      DATABASE_URL: postgres://app:pw@db:5432/app
    depends_on: [db]
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: pw
    volumes: ["pgdata:/var/lib/postgresql/data"] # 상태 보존
volumes:
  pgdata:
신호와 graceful shutdown

오케스트레이터(또는 docker stop)는 컨테이너에 SIGTERM을 보내고, 유예 시간(기본 10초) 안에 안 죽으면 SIGKILL로 강제 종료합니다. 앱이 PID 1로 돌면서 신호를 직접 처리해야 진행 중 요청을 마저 끝내고 연결을 정리합니다. CMD를 셸 형식으로 쓰면 신호가 sh에서 막혀 앱까지 안 갑니다(→ OPS-03). exec 배열 형식을 쓰고, 필요하면 --init으로 가벼운 init을 PID 1에 두세요.

OPS-06

레지스트리와 태깅

레지스트리는 이미지의 원격 저장소입니다 — git의 GitHub에 해당합니다. 로컬에서 빌드한 이미지를 push하고, 서버·CI가 pull해 실행합니다. 그 사이를 잇는 건 태그이고, 태그 전략이 배포 추적성을 좌우합니다.

동작명령 / 의미
로그인docker login <registry> — Docker Hub, GHCR, AWS ECR 등
태깅docker tag app:local ghcr.io/org/app:1.4.2 — 레지스트리 경로 + 버전
업로드docker push ghcr.io/org/app:1.4.2 — 레이어 단위로 전송(있는 건 건너뜀)
다운로드docker pull ghcr.io/org/app:1.4.2 — 서버·CI가 가져와 실행

태그 이름은 레지스트리/네임스페이스/이미지:태그 구조입니다. 태그는 같은 이미지에 여러 개 붙일 수 있는 가변 포인터입니다 — 진짜 불변 식별자는 내용 해시인 digest(@sha256:...)입니다. 버전 태그는 사람이 읽기 위한 것이고, "정확히 이 이미지"를 못 박으려면 digest로 고정합니다.

안티패턴 — latest 태그

latest는 "최신"을 보장하지 않습니다 — 그저 태그를 안 붙이면 붙는 기본 문자열일 뿐입니다. production에서 latest로 배포하면 ① 지금 도는 게 어떤 커밋인지 추적 불가 ② 재시작·스케일아웃 시 노드마다 다른 이미지를 당겨와 버전이 갈라짐 ③ 롤백 대상이 모호해집니다. 커밋 SHA나 시맨틱 버전(1.4.2)으로 불변 태깅하고, production 배포는 digest 고정을 권합니다. latest는 로컬 편의용으로만.

OPS-07

12-factor 앱

12-factor는 "컨테이너·클라우드에서 잘 도는 앱"의 설계 원칙 모음입니다. 전부 외울 필요는 없고, 컨테이너 배포에 직결되는 네 가지만 몸에 익히면 대부분의 사고를 피합니다.

원칙무엇을 / 왜
설정 = 환경변수DB URL·시크릿·플래그를 코드 밖 환경변수로. 같은 이미지를 dev/stage/prod에 그대로 재사용
stateless 프로세스프로세스는 상태를 안 가짐 — 세션·파일은 외부(DB·오브젝트 스토리지)로. 아무 인스턴스나 죽고 늘어도 무방
로그 = stdout 스트림앱은 파일에 안 쓰고 stdout으로 흘려보냄. 수집·라우팅은 플랫폼이 담당
빌드/릴리즈/실행 분리코드→이미지(빌드), 이미지+설정(릴리즈), 그 릴리즈 실행(실행)을 단계로 분리 → 롤백은 이전 릴리즈로

왜 이게 컨테이너와 맞물리나. 컨테이너의 RW 레이어는 휘발성입니다(→ OPS-02) — 컨테이너가 죽으면 안에 쓴 파일은 사라집니다. 그러니 상태를 안에 두면 안 되고(stateless), 로그를 파일로 쓰면 수집 전에 증발합니다(stdout). 설정을 이미지에 굽으면 환경마다 이미지를 새로 빌드해야 하니, 환경변수로 빼서 한 이미지를 모든 환경에서 재사용합니다. 이 셋이 맞물려야 수평 확장·무중단 배포·빠른 롤백이 성립합니다.

실패 모드 — 상태를 컨테이너에 둔 경우

업로드 파일을 컨테이너 로컬 디스크에 저장하면, 스케일아웃해 인스턴스가 2개가 되는 순간 "방금 올린 파일이 안 보임"이 됩니다 — 요청이 다른 인스턴스로 갔기 때문입니다. 배포로 컨테이너가 교체되면 통째로 사라지기도 합니다. 파일은 S3 같은 외부 스토리지로, 세션은 Redis·DB로 빼야 합니다. 시크릿을 환경변수로 넣을 땐 docker inspect·로그에 노출되지 않게 secret manager 사용을 고려하세요(→ OPS-08의 시크릿 주입).

OPS-08

CI/CD 파이프라인

CI(지속적 통합)는 코드가 합쳐질 때마다 자동으로 빌드·테스트하는 것, CD(지속적 배포)는 그 결과물을 자동으로 환경에 올리는 것입니다. 컨테이너가 CI/CD와 잘 맞는 이유는 빌드 산출물이 이미지 하나로 떨어져 그대로 배포되기 때문입니다 — 빌드와 배포 사이에 변형이 없습니다.

commit / PR→ 빌드→ 테스트 · 스캔→ 이미지 빌드 · push→ 배포 (deploy)→ 헬스체크

각 단계 사이에는 게이트(gate)가 있습니다 — 테스트가 깨지면 거기서 멈추고 배포로 못 넘어갑니다. production 배포 앞에는 보통 사람의 수동 승인 게이트를 둡니다. 단계가 통과할 때마다 신뢰가 쌓이고, 깨지면 가장 싼 단계(테스트)에서 빨리 잡힙니다 — 이게 게이트의 핵심 가치입니다.

단계하는 일
빌드의존성 설치·컴파일. 캐시로 빠르게
테스트unit·integration. 실패 시 즉시 중단(게이트)
스캔이미지 취약점(trivy)·lint·SAST. 보안 게이트
이미지 빌드·push커밋 SHA로 태깅해 레지스트리에 push(→ OPS-06)
배포그 이미지를 환경에 롤아웃(전략은 → OPS-09)

GitOps는 한 걸음 더 갑니다 — 배포 상태를 명령으로 밀어넣는 대신, 원하는 상태를 git 저장소에 선언하고 에이전트(예: Argo CD·Flux)가 실제 클러스터를 그 선언에 맞춰 끌어옵니다. git이 단일 진실 공급원이 되어, 배포 이력이 곧 git 이력이고 롤백은 revert입니다.

시크릿 주입 — 절대 이미지에 굽지 마라

API 키·DB 비밀번호를 Dockerfile의 ENV나 COPY .env로 넣으면 이미지 레이어에 영구히 박혀, 레지스트리에 접근 가능한 누구나 docker history로 꺼낼 수 있습니다 — 나중에 덮어써도 이전 레이어에 남습니다. 시크릿은 ① CI의 secret store(GitHub Actions Secrets 등)에서 빌드 시점에만 주입하거나 ② 런타임에 secret manager(Vault·클라우드 KMS)·오케스트레이터 시크릿으로 환경변수에 넣습니다. 빌드 중 일시적으로 필요하면 BuildKit의 --mount=type=secret으로 레이어에 안 남게 하세요.

OPS-09

배포 전략

"새 버전으로 어떻게 갈아끼우는가"의 문제입니다. 핵심 목표는 셋 — 무중단(사용자가 끊김을 못 느낌), 빠른 롤백(문제 시 즉시 되돌림), 위험 통제(나쁜 버전의 폭발 반경 최소화). 전략마다 이 셋의 균형이 다릅니다.

전략방식장점 / 트레이드오프
롤링(rolling)인스턴스를 몇 개씩 순차 교체추가 자원 거의 없음 / 교체 중 신·구 버전 공존, 롤백이 느림
블루그린구(blue)·신(green) 환경을 통째로 띄워두고 트래픽을 한 번에 전환즉시 전환·즉시 롤백(되돌리기=스위치 back) / 자원 2배, DB 마이그레이션 호환 필요
카나리신 버전에 트래픽을 5%→25%→100%로 점증, 지표 보며 확대폭발 반경 최소·실데이터 검증 / 설정 복잡, 관측·자동 롤백 인프라 필요

세 전략 모두 헬스체크가 안전판입니다. 새 인스턴스가 트래픽을 받기 전, 오케스트레이터는 readiness 프로브로 "받을 준비 됐는지"를 확인합니다 — 준비 안 된 인스턴스로 트래픽이 가지 않게. liveness 프로브는 "죽었는지"를 보고 죽었으면 재시작합니다. 카나리의 자동 확대·롤백은 이 헬스체크와 에러율·레이턴시 지표에 묶입니다(→ 로깅·모니터링·장애대응).

트레이드오프 — DB 마이그레이션이 발목을 잡는다

무중단 배포의 진짜 난점은 코드가 아니라 스키마입니다. 신·구 버전이 잠깐 공존하므로(롤링·카나리·블루그린 전환 순간), DB 스키마 변경은 양쪽 버전과 모두 호환돼야 합니다 — 컬럼 삭제·이름변경 같은 파괴적 변경을 코드 배포와 한 번에 하면 한쪽이 깨집니다. expand-and-contract 패턴(컬럼 추가→양쪽 호환 코드 배포→구 컬럼 제거를 다음 릴리즈로)으로 나눠야 안전합니다. 블루그린에서 green이 prod DB를 공유하면 "환경 통째 분리"의 환상이 여기서 깨집니다(→ 백엔드 개요의 트랜잭션·마이그레이션).

OPS-10

오케스트레이션 개요

컨테이너 하나를 돌리는 건 docker run으로 끝입니다. 그러나 수십~수백 개를 여러 서버에 걸쳐 띄우고·살피고·죽으면 되살리고·확장하고·배포하는 건 손으로 못 합니다. 이걸 자동화하는 게 오케스트레이션이고, 사실상 표준이 Kubernetes(K8s)입니다.

K8s를 왜 쓰나 — 핵심은 선언적 자가복구입니다. "이 앱을 3개 띄워라"라고 원하는 상태를 선언하면, K8s가 실제 상태를 거기에 맞춰 계속 끌어옵니다. 노드가 죽어 컨테이너가 2개로 줄면 자동으로 1개를 다른 노드에 다시 띄웁니다. 명령(어떻게)이 아니라 상태(무엇을)를 다룬다는 점이 OPS-08의 GitOps와 맞물립니다.

리소스역할 (개념만)
Pod배포의 최소 단위 — 컨테이너 1개(때로 몇 개)를 묶음. 가장 작은 "실행 단위"
DeploymentPod를 몇 개 유지할지 선언·관리. 롤링 업데이트·롤백을 담당
Service바뀌는 Pod들 앞의 안정적 진입점(가상 IP·DNS) — 로드밸런싱
Ingress클러스터 외부 HTTP(S) 트래픽을 내부 Service로 라우팅(도메인·경로 기반)
범위 — 여기까지만

K8s의 깊이(스케줄링·네트워킹·오퍼레이터·헬름·RBAC 등)는 이 핸드북의 범위 밖입니다. 지금 필요한 건 왜 존재하고 핵심 개념이 무엇인지까지입니다 — Pod/Deployment/Service/Ingress 네 단어의 관계만 잡으면 팀의 매니페스트를 읽고 대화에 참여할 수 있습니다. 작은 서비스라면 K8s가 과합니다 — 관리형 PaaS(Cloud Run·ECS·Fly.io 등)나 Compose로 충분한 경우가 많고, 복잡성 비용을 정당화할 규모가 됐을 때 K8s로 갑니다. 실제 운영 디테일은 → 로깅·모니터링·장애대응, 백엔드 개요와 함께 보세요.

OPS-11

용어집

이 핸드북에서 반복된 핵심 용어. 정의를 한 줄로 못 줄이면 아직 이해 못 한 것입니다 — 막힐 때 여기로.

컨테이너container
namespace·cgroup로 격리된 프로세스. host 커널을 공유하는 이미지의 실행 인스턴스(→ OPS-01)
이미지image
코드·런타임·의존성을 봉인한 불변(read-only) 스냅샷. 컨테이너의 "클래스"에 해당(→ OPS-02)
레이어layer
이미지를 구성하는 읽기 전용 스택의 한 장 = 캐시·공유 단위. Dockerfile 명령 하나가 대체로 한 장(→ OPS-02)
Dockerfilebuild recipe
이미지를 만드는 명령 스크립트. FROM/RUN/COPY/CMD 등. 순서가 캐시·크기를 결정(→ OPS-03)
.dockerignorebuild context
빌드 컨텍스트에서 제외할 파일 목록. node_modules·.git 제외로 전송량·오염 방지(→ OPS-03)
멀티스테이지 빌드multi-stage
한 Dockerfile에 빌드·런타임 단계를 나눠, 산출물만 COPY --from으로 가져옴. 크기·공격면↓(→ OPS-04)
distroless / slimminimal base
패키지를 최소화한 베이스 이미지. 작고 안전하지만 디버깅 도구가 적거나 없음(→ OPS-04)
볼륨volume
컨테이너 밖에 상태를 보존하는 마운트. 휘발성 RW 레이어 문제를 우회(→ OPS-05, OPS-07)
Composedocker compose
여러 컨테이너를 compose.yaml로 선언해 한 번에 띄우는 로컬 다중 서비스 도구(→ OPS-05)
레지스트리registry
이미지의 원격 저장소. push/pull로 주고받음(Docker Hub·GHCR·ECR)(→ OPS-06)
태그 / digesttag · @sha256
태그는 가변 포인터(1.4.2), digest는 내용 해시로 고정한 불변 식별자. latest는 안티패턴(→ OPS-06)
12-factorconfig·stateless·logs
클라우드 친화 앱 설계 원칙. 설정=환경변수, stateless, 로그=stdout, 빌드/릴리즈/실행 분리(→ OPS-07)
CI/CDpipeline
빌드→테스트→이미지→배포를 게이트로 자동화. 산출물=이미지 하나라 변형 없이 배포(→ OPS-08)
GitOpsdeclarative deploy
원하는 상태를 git에 선언하고 에이전트가 클러스터를 동기화. 롤백=revert(→ OPS-08, OPS-10)
롤링rolling update
인스턴스를 순차 교체하는 무중단 배포. 자원 절약하나 신·구 공존·롤백 느림(→ OPS-09)
블루그린blue-green
신·구 환경을 통째로 띄워두고 트래픽을 한 번에 전환. 즉시 롤백, 자원 2배(→ OPS-09)
카나리canary
신 버전에 트래픽을 점증하며 지표 관찰. 폭발 반경 최소, 관측 인프라 필요(→ OPS-09)
헬스체크readiness · liveness
readiness=트래픽 받을 준비됐나, liveness=살아있나. 무중단 배포의 안전판(→ OPS-09)
KubernetesK8s
선언적 자가복구 오케스트레이터. 원하는 상태를 선언하면 실제 상태를 맞춰줌(→ OPS-10)
Pod / DeploymentK8s 리소스
Pod=실행 최소 단위(컨테이너 묶음), Deployment=Pod 개수·롤링·롤백 관리(→ OPS-10)
Service / IngressK8s 네트워킹
Service=Pod 앞 안정적 진입점·LB, Ingress=외부 HTTP를 내부 Service로 라우팅(→ OPS-10)
학습 순서 제안

전부 한 번에 익히려 하지 마세요. 먼저 손에 붙일 것: docker build → docker run -p → 멀티스테이지 Dockerfile 한 장 → docker compose up으로 로컬 풀스택. 이 네 단계로 OPS-01~05가 몸에 들어옵니다. 그다음 CI에서 이미지를 빌드·push해보고(OPS-06·08), 배포 전략(OPS-09)은 팀의 실제 파이프라인을 읽으며 익히세요. K8s(OPS-10)는 규모가 정당화할 때 갑니다 — 그전엔 개념만으로 충분합니다. 런타임 관측은 → 로깅·모니터링·장애대응, 데이터 계층은 → 백엔드 개요와 함께.

OPS-12

운영 가능한 배포 심화

배포는 이미지를 올리는 일이 아니라 트래픽 전환, 종료 처리, 설정 일관성, 롤백 가능성을 관리하는 일입니다.

주제실무 기준실패 신호
graceful shutdownSIGTERM을 받으면 새 요청을 막고 진행 중 요청을 마무리배포 때 간헐적인 502/connection reset 발생
readiness트래픽을 받아도 되는 상태인지 확인앱 시작 직후 DB 연결 전부터 트래픽 유입
liveness프로세스가 복구 불능 상태인지 확인느린 의존성 때문에 정상 앱이 계속 재시작
config drift환경별 설정 차이를 코드/템플릿으로 추적운영만 수동 hotfix 설정이 남아 재현 불가
rollback앱, DB, 메시지 schema가 되돌릴 수 있는지 사전 확인배포 후 DB 변경 때문에 이전 버전이 실행 불가
Case study

롤링 배포 중 새 컨테이너가 뜨자마자 readiness가 성공하면 아직 connection pool이 준비되지 않은 상태에서 요청이 들어올 수 있습니다. startup probe, readiness probe, graceful shutdown, termination grace period를 함께 설계해야 배포 중 오류율을 낮출 수 있습니다.
