DK-CS-01

핵심 명령·스니펫

컨테이너 상태 · 로그

docker ps                    # 실행 중인 컨테이너
docker ps -a                 # 종료된 것 포함 전체 + EXIT CODE
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
docker logs -f <c>            # 로그 실시간 추적 (가장 자주 씀)
docker logs --tail 100 <c>   # 마지막 100줄만
docker logs --since 10m <c>   # 최근 10분치만
docker stats                 # CPU/MEM/NET 실시간 (Ctrl+C로 종료)
docker stats --no-stream     # 한 번만 찍고 종료

내부 진입 · 검사

docker exec -it <c> sh       # 컨테이너 내부 shell (alpine)
docker exec -it <c> bash     # bash 있는 이미지
docker exec <c> env          # 주입된 env 덤프
docker inspect <c>           # 전체 설정(mount/env/network) JSON
docker inspect --format '{{.State.ExitCode}}' <c>
docker inspect --format '{{json .Mounts}}' <c>
docker inspect --format '{{.Id}}' <img>   # 이미지 digest 고정 확인
docker port <c>              # 포트 매핑 확인

이미지 · 빌드

docker build -t app:tag .            # 태그 박아서 빌드 (latest 금지)
docker build --no-cache -t app:tag . # 캐시 무시하고 처음부터
docker build --target builder -t x . # multi-stage 특정 stage만
docker history app:tag               # layer별 크기 · 명령 추적
docker images                        # 로컬 이미지 목록
docker run --rm -it --entrypoint sh app:tag  # 시작 못하는 이미지 진입

Compose

docker compose up -d          # 백그라운드로 전체 기동
docker compose up -d --build   # 이미지 재빌드 후 기동
docker compose logs -f         # 전체 서비스 로그 추적
docker compose logs -f <svc>   # 특정 서비스만
docker compose ps              # 서비스 상태
docker compose config          # .env 병합된 최종 설정 확인
docker compose down            # 컨테이너+네트워크 정리
docker compose down -v         # 볼륨까지 삭제 (데이터 날아감)
docker compose exec <svc> sh    # 서비스 내부 진입
명령용도주의
docker ps -a종료 컨테이너 EXIT CODE 확인STATUS 컬럼이 원인 첫 단서
docker logs -f <c>앱 로그 실시간앱이 stdout/stderr로 찍어야 보임
docker exec -it <c> sh실행 중 컨테이너 진입distroless엔 shell 없음
docker cp <c>:/path ./컨테이너 파일 밖으로 복사디버깅용, 영속 수단 아님
docker rm -f <c>강제 삭제volume 안 붙었으면 데이터 소실
주의

docker system prune -af 는 중지된 컨테이너·미사용 이미지·네트워크·빌드 캐시를 전부 지웁니다. --volumes 를 붙이면 미사용 named volume(=DB 데이터)까지 삭제됩니다. 운영 근처에서 함부로 실행 금지.

판단 한 줄

선택언제 쓰나주의
bind mount로컬 개발, 소스 변경 즉시 반영host 권한·성능 영향, 운영엔 부적합
named volumeDB·업로드 등 컨테이너 교체 뒤에도 보존삭제 전 소유자·백업 확인, down -v 조심
CMD기본 실행 인자, 오버라이드 가능하게docker run 뒤 인자로 통째 교체됨
ENTRYPOINT컨테이너를 실행파일처럼 고정인자 교체엔 --entrypoint 필요
multi-stage build빌드 도구와 런타임 산출물 분리빌드 stage의 secret/cache가 최종에 안 남게
alpine작고 shell 있음, 일반 서비스musl libc 호환·DNS 이슈 간혹
distroless최소 공격면, 운영 배포shell·패키지매니저 없음, 디버깅 불편
restart policyunless-stopped=재부팅 후 자동, on-failure=비정상만always는 수동 중지도 되살림

신호왜 위험즉시 확인
latest 태그로 배포rollback·감사 불가, 재배포 때 다른 이미지immutable tag, docker inspect --format '{{.Id}}' digest 고정

DK-CS-04

장애 증거 1카드

CONTAINER WON'T START1. docker ps -a → STATUS/EXIT CODE 확인 (0=정상종료, 137=OOM/kill, 1=앱오류)
2. docker logs --tail 100 <c> → 첫 에러 로그
3. docker inspect <c> → mount / env / entrypoint 확인
4. docker inspect --format '{{json .State.Health}}' <c> → 헬스체크 실패 여부
5. docker inspect --format '{{.Id}}' <img> → 실행 이미지 digest가 의도한 것인지
6. docker run --rm -it --entrypoint sh <img> → 이미지 자체 진입해 파일·bind 검증
순서질문증거
1어떻게 죽었나ps -a EXIT CODE + STATUS
2앱이 뭘 남겼나logs 첫 에러 라인
3설정이 맞나inspect env·mount·entrypoint
4준비됐다고 보고했나healthcheck State.Health
5맞는 이미지인가image digest(.Id)
복구/중단

EXIT 137이면 메모리 한계(--memory)나 외부 OOM kill부터 본다. logs·inspect로 원인을 짚기 전엔 재빌드 반복 금지 — 이미지가 아니라 env·mount·network 설정 문제인 경우가 많다.
