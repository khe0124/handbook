Docker
Cheat Sheet

Docker 실행 오류를 이미지, 컨테이너, 포트, 볼륨, 네트워크, Compose 기준으로 빠르게 좁히는 참조입니다.

COMPILED : 2026-06 · FORMAT : QUICK REFERENCE
DK-CS-01

개요

이미지는 불변 실행 패키지이고 컨테이너는 그 이미지를 실행한 프로세스입니다. 런타임 문제는 build와 run 단계를 분리해 봅니다.

상황먼저 확인할 것
빌드 실패Dockerfile 순서, build context, cache, secret 노출
실행 실패entrypoint, env, port, volume, working directory
연결 실패container network, service name, port mapping
데이터 사라짐container layer와 volume/bind mount 구분
DK-CS-02

핵심 명령/개념

명령압축 설명
docker build -t app .현재 context로 이미지 빌드
docker run --rm -p 3000:3000 app컨테이너 실행 후 종료 시 삭제
docker ps -a실행/종료 컨테이너 확인
docker logs -f name컨테이너 로그 추적
docker exec -it name sh실행 중 컨테이너 내부 진입
docker compose up여러 서비스를 선언 파일 기준으로 실행
DK-CS-03

판단 기준

선택기준
bind mount로컬 개발에서 파일 변경을 즉시 반영해야 함
named volumeDB 데이터처럼 컨테이너 교체 뒤에도 보존해야 함
multi-stage build빌드 도구와 런타임 산출물을 분리해야 함
slim image운영 이미지 크기와 취약점 노출을 줄여야 함
compose앱, DB, 캐시 등 로컬 다중 서비스를 함께 띄워야 함
DK-CS-04

자주 나는 실수

  • .dockerignore 없이 node_modules와 .git을 build context에 포함한다.
  • Dockerfile에 secret을 굽거나 build arg로 남긴다.
  • container port와 host port 방향을 헷갈린다.
  • 컨테이너 내부 파일을 영속 데이터로 믿는다.
  • latest 태그를 production 배포 식별자로 사용한다.
DK-CS-05

면접/리뷰용 한 줄 답변

이미지와 컨테이너model
이미지는 불변 스냅샷이고 컨테이너는 그 이미지를 실행한 격리된 프로세스입니다.
레이어 캐시cache
Dockerfile은 자주 바뀌지 않는 의존성 레이어를 앞에 두고 자주 바뀌는 소스를 뒤에 둬야 빌드가 빨라집니다.
볼륨volume
컨테이너 파일시스템은 휘발성이므로 보존해야 하는 상태는 volume이나 외부 스토리지로 빼야 합니다.
