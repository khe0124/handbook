설정과 비밀 정보 관리

설정은 환경마다 달라지는 값이고, 비밀 정보는 노출되면 권한 탈취로 이어지는 값입니다. 둘을 같은 방식으로 다루면 재현성과 보안이 동시에 무너집니다.

SEC-01

Config와 Secret

구분예시관리 방식
ConfigAPI base URL, feature flag, timeout환경별 값으로 버전 관리 가능
SecretDB password, API key, private key암호화 저장소와 접근 통제
Runtime metadatacommit SHA, release ID빌드/배포 시스템에서 주입
원칙

Secret은 코드, 이미지, 로그, 테스트 fixture, 릴리스 노트에 들어가면 안 됩니다. 유출된 secret은 삭제가 아니라 폐기와 재발급이 우선입니다.

SEC-02

주입 시점

시점예시주의
Build-time프론트엔드 public configbundle에 박히므로 secret 금지
Deploy-timemanifest, Helm values환경별 차이 추적 필요
Runtimeenv var, mounted secret, metadata API교체와 재시작 전략 필요
실무 감각

프론트엔드 빌드에 들어간 값은 사용자 브라우저에 배포된 공개 정보입니다. 이름이 API_KEY여도 bundle에 포함되면 secret이 아닙니다.

SEC-03

Secret 저장소

방식장점주의
CI Secret파이프라인 주입이 쉬움job 권한과 로그 마스킹 확인
Cloud Secret ManagerIAM, rotation, audit 연동앱 권한 경계 설계 필요
Vault동적 secret, 강한 정책운영 복잡도와 가용성 관리 필요
Kubernetes Secret클러스터와 통합etcd 암호화와 RBAC 필수
SEC-04

Rotation

Secret rotation은 새 값을 발급하고, 양쪽을 전환하고, 옛 값을 폐기하는 절차입니다.

새 secret 발급→ 앱 주입→ 동작 확인→ 옛 secret 폐기
대상rotation 주의
DB password새 계정/비밀번호 병행 기간 필요
API token권한 범위와 만료일 확인
VPN PSK양쪽 동시 변경 창 필요
TLS certificate만료 전 자동 갱신과 reload 확인
SEC-05

유출 대응

즉시 폐기revoke
Git에서 지우는 것보다 노출된 권한을 먼저 무효화합니다.
영향 범위blast radius
권한 범위, 사용 로그, 접근 가능한 리소스를 확인합니다.
히스토리 정리cleanup
필요하면 git history를 정리하지만 이미 노출된 값은 신뢰하지 않습니다.
재발 방지prevention
secret scanning, pre-commit hook, CI gate, 권한 최소화를 적용합니다.
