IO-01

Q. 브라우저에서 서버까지 요청 흐름을 설명해보세요.

핵심 개념

웹 요청은 함수 호출이 아니라 DNS, TCP, TLS, HTTP, CDN/LB, 서버, DB, 캐시를 지나는 실패 가능한 경로입니다.

개념 해부

요청 흐름은 한 번의 함수 호출이 아니라 여러 계층의 계약이 연쇄적으로 성공해야 완성되는 경로입니다. DNS는 이름을 IP로 바꾸고, TCP는 연결을 만들며, TLS는 암호화와 서버 인증을 제공하고, HTTP는 애플리케이션 의미를 전달합니다. 장애 대응에서는 “서버 문제”라고 뭉개지 말고 어느 계층에서 실패했는지 증거로 좁혀야 합니다.

30초 답변

브라우저가 URL을 해석하고 DNS로 IP를 찾은 뒤 TCP 연결과 TLS handshake를 거쳐 HTTP 요청을 보냅니다. 요청은 CDN이나 Load Balancer를 지나 애플리케이션 서버에 도착하고, 서버는 인증, 라우팅, 비즈니스 로직, DB/캐시 조회 후 응답을 반환합니다.

90초 답변

이 흐름을 알아야 장애를 계층별로 좁힐 수 있습니다. DNS가 틀리면 서버까지 도달하지 못하고, TLS가 실패하면 인증서나 SNI 문제일 수 있으며, LB health check가 실패하면 앱은 살아 있어도 트래픽을 받지 못합니다. API가 느리면 네트워크, 서버 CPU, DB 대기, 외부 API timeout을 분리해야 합니다.

꼬리질문

질문답변 방향
TCP와 TLS 차이는?TCP는 전송 연결, TLS는 그 위에서 암호화와 서버 인증을 제공합니다.
timeout은 어디에 두나요?client, gateway, app, DB, 외부 API마다 목적에 맞게 두고 전체 예산을 맞춥니다.
느린 요청은 어떻게 좁히나요?waterfall, server log, trace, DB query, external dependency 순서로 분해합니다.

오답/주의점

주의

“브라우저가 서버에 요청합니다”에서 멈추면 얕습니다. 계층별 실패 지점을 말해야 합니다.

IO-02

Q. CDN과 cache-control은 어떻게 설계하나요?

핵심 개념

CDN은 사용자 가까운 edge에서 정적 자산이나 cacheable response를 제공해 latency와 origin 부하를 줄입니다. 핵심은 무엇을 얼마나 오래 캐시해도 되는지입니다.

개념 해부

CDN 캐시는 origin 부하와 거리 비용을 줄이는 대신 최신성 책임을 edge로 분산시킵니다. content hash가 붙은 asset은 URL이 내용의 버전이 되므로 오래 캐시해도 안전하지만, HTML이나 사용자별 응답은 같은 정책을 쓰면 배포 꼬임이나 개인정보 노출 위험이 생깁니다. Cache-Control은 성능 옵션이 아니라 데이터 신뢰성 계약입니다.

30초 답변

hash가 붙은 JS/CSS asset은 immutable로 길게 캐시하고, HTML은 짧게 캐시하거나 revalidate하게 둡니다. API response는 데이터 최신성 요구에 따라 private/public, max-age, stale-while-revalidate를 신중히 선택합니다.

90초 답변

캐시는 계층별로 다르게 동작합니다. 브라우저 캐시, CDN 캐시, 애플리케이션 캐시가 같은 데이터를 서로 다른 TTL로 보관하면 사용자는 오래된 화면을 볼 수 있습니다. 배포에서는 HTML이 새 asset을 참조하는데 asset이 아직 CDN에 없거나, 반대로 HTML이 오래 캐시되어 구버전을 계속 보는 문제가 흔합니다.

꼬리질문

질문답변 방향
immutable asset은 왜 안전한가요?파일명에 content hash가 있어 내용이 바뀌면 URL도 바뀌기 때문입니다.
stale-while-revalidate는?오래된 값을 우선 주고 백그라운드에서 새 값을 갱신하는 전략입니다.
캐시 버그는 어떻게 확인하나요?response header, CDN status, asset hash, hard reload, purge 이력을 확인합니다.

오답/주의점

주의

CDN을 “빠르게 해주는 것”으로만 말하지 말고 무효화와 최신성 리스크를 같이 말하세요.

IO-03

Q. Docker와 Kubernetes를 어떻게 설명하나요?

핵심 개념

Docker는 애플리케이션과 런타임 의존성을 이미지로 패키징하고, Kubernetes는 컨테이너를 스케줄링, 복구, 확장, 서비스 디스커버리, rollout하는 플랫폼입니다.

개념 해부

Docker 이미지는 실행 환경을 포장하지만 프로세스 생명주기와 운영 정책을 자동으로 해결하지는 않습니다. Kubernetes는 pod를 원하는 상태로 유지하려고 재시작, 스케줄링, 서비스 디스커버리, rollout을 수행합니다. 따라서 health check, resource request/limit, graceful shutdown, secret 주입 방식까지 알아야 운영 답변이 됩니다.

30초 답변

Docker는 “어디서나 같은 방식으로 실행되는 이미지”를 만드는 도구이고, Kubernetes는 그 이미지를 여러 노드에 배치하고 장애 시 재시작하며 서비스로 노출하고 rollout/rollback을 관리하는 오케스트레이션 시스템입니다.

90초 답변

운영 관점에서는 Dockerfile보다 이미지 크기, secret 주입, health check, graceful shutdown, 로그 출력 방식이 중요합니다. Kubernetes에서는 readiness와 liveness를 구분해야 합니다. readiness는 트래픽을 받을 준비가 됐는지, liveness는 프로세스를 재시작해야 할 정도로 죽었는지 판단합니다.

꼬리질문

질문답변 방향
readiness와 liveness 차이는?readiness는 트래픽 수신 여부, liveness는 재시작 여부입니다.
graceful shutdown은 왜 필요한가요?배포/종료 중 처리 중인 요청을 안전하게 마무리하기 위해 필요합니다.
request와 limit은?request는 스케줄링 기준, limit은 사용할 수 있는 상한입니다.

오답/주의점

주의

Kubernetes를 “Docker 여러 개 관리”로만 말하면 약합니다. health, rollout, service discovery, resource 관리까지 말해야 합니다.

IO-04

Q. CI/CD와 무중단 배포는 어떻게 보나요?

핵심 개념

CI/CD는 자동 배포 버튼이 아니라 변경을 검증하고, 추적하고, 안전하게 사용자에게 전달하고, 나빠지면 되돌리는 시스템입니다.

개념 해부

CI/CD는 배포 자동화가 아니라 변경을 작게 검증하고 추적 가능한 artifact로 사용자에게 전달하는 시스템입니다. lint/test/build만 통과해도 운영 지표가 나빠질 수 있으므로 배포 후 health, error rate, latency를 확인해야 합니다. 무중단 배포의 핵심은 새 버전과 구버전, DB schema가 잠시 공존할 수 있는 호환성입니다.

30초 답변

CI는 lint, typecheck, test, build로 변경을 빠르게 검증하고, CD는 artifact를 환경에 배포하며 metric과 health check로 안전성을 확인합니다. 무중단 배포는 rolling, blue-green, canary 같은 전략과 rollback 기준이 함께 있어야 합니다.

90초 답변

배포에서 중요한 것은 어떤 commit과 image digest, migration version, config가 배포됐는지 추적 가능해야 한다는 점입니다. DB migration은 코드 배포와 함께 깨지기 쉬우므로 expand-contract 방식으로 하위 호환성을 유지해야 합니다. canary는 일부 트래픽에 먼저 배포하고 error rate, latency, business metric을 보고 확대합니다.

꼬리질문

질문답변 방향
rollback 기준은?5xx, latency, 핵심 전환율, error budget burn rate 같은 지표로 정합니다.
DB migration은?새 컬럼 추가, 양쪽 코드 호환, backfill, old column 제거를 단계로 나눕니다.
artifact 추적은?commit SHA, image digest, release tag, config version을 묶어 남깁니다.

오답/주의점

주의

“GitHub Actions로 배포합니다”는 도구 설명입니다. 검증, 승인, 추적, rollback을 말해야 합니다.

IO-05

Q. 로그, 메트릭, 트레이스는 각각 어떤 역할을 하나요?

핵심 개념

로그는 사건의 상세 기록, 메트릭은 시스템 상태의 숫자 흐름, 트레이스는 요청이 여러 서비스와 계층을 지나간 경로입니다.

개념 해부

로그, 메트릭, 트레이스는 같은 문제를 다른 해상도로 봅니다. 메트릭은 증상의 크기와 시점을 보여주고, 로그는 개별 사건의 문맥을 주며, 트레이스는 요청이 어느 dependency에서 늦거나 실패했는지 보여줍니다. 좋은 관측성은 장애 후 검색 가능한 흔적이 아니라 장애 중 의사결정 가능한 신호입니다.

30초 답변

장애 대응에서는 먼저 영향 범위와 지표를 보고, 로그로 구체적 사건을 확인하고, trace로 요청이 어디서 늦거나 실패했는지 좁힙니다. 로그만 보는 것이 아니라 RED, USE, Golden Signals 같은 지표와 함께 봐야 합니다.

90초 답변

RED는 request rate, error rate, duration으로 서비스 관점 지표를 봅니다. USE는 utilization, saturation, errors로 리소스 관점을 봅니다. 좋은 로그에는 traceId, user/action/resource, error code, dependency가 있어야 하지만 고카디널리티 필드를 무분별하게 metric label로 넣으면 비용과 성능 문제가 생길 수 있습니다.

꼬리질문

질문답변 방향
장애가 나면 무엇부터?영향 범위, 최근 변경, metric, log, trace, rollback 가능성을 순서대로 봅니다.
postmortem에는?타임라인, 영향, 원인, 감지 지연, 완화, 재발 방지 action을 남깁니다.
alert 기준은?증상 기반, 사용자 영향 기반, action 가능한 기준으로 잡습니다.

오답/주의점

주의

“로그를 봅니다”로 끝내지 마세요. 어떤 지표로 영향 범위를 보고 어떤 trace로 병목을 좁히는지 말해야 합니다.

IO-06

Q. DNS, TCP, TLS, HTTP는 요청에서 어떤 역할을 하나요?

핵심 개념

웹 요청은 이름 해석, 연결 수립, 암호화 협상, 애플리케이션 프로토콜 처리로 나뉩니다.

개념 해부

DNS, TCP, TLS, HTTP를 구분하면 같은 “접속 안 됨”도 원인이 달라집니다. DNS TTL과 resolver cache, TCP 방화벽/라우팅, TLS 인증서/SNI, HTTP gateway/upstream 오류는 확인 도구와 담당 영역이 다릅니다. 면접에서는 dig, curl -v, browser timing, gateway log처럼 계층별 증거를 말할 수 있어야 합니다.

30초 답변

DNS는 도메인을 IP로 해석하고, TCP는 신뢰성 있는 전송 연결을 만들며, TLS는 암호화와 서버 인증을 제공합니다. HTTP는 그 위에서 method, header, body, status code로 애플리케이션 요청과 응답을 표현합니다.

90초 답변

장애도 단계별로 다릅니다. DNS 문제면 특정 ISP나 resolver에서만 실패할 수 있고, TCP 문제면 방화벽이나 라우팅, port open 여부를 봅니다. TLS 문제는 인증서 만료, SNI, protocol/cipher mismatch가 원인일 수 있고, HTTP 문제는 gateway timeout, auth, routing, upstream failure로 나눠 봅니다.

꼬리질문

질문답변 방향
HTTP/2 장점은?multiplexing, header compression, connection 효율을 말합니다.
TLS handshake 비용은?초기 연결 비용이 있어 keep-alive, session resumption을 봅니다.
DNS TTL은?변경 전파 속도와 cache 효율의 trade-off입니다.

면접에서 말할 증거

`dig`, `curl -v`, browser timing, 인증서 검사, gateway log로 계층별 확인한 사례를 말합니다.

오답/주의점

주의

네트워크를 한 덩어리로 말하면 장애를 좁히는 능력이 보이지 않습니다.

IO-07

Q. CORS는 왜 생기고 어떻게 해결하나요?

핵심 개념

CORS는 브라우저가 다른 origin 요청을 제한하는 Same-Origin Policy를 서버가 명시적으로 완화하는 메커니즘입니다.

개념 해부

CORS는 서버가 요청을 거절하는 기능이 아니라 브라우저가 cross-origin 응답 접근을 제한하는 보안 모델입니다. 서버는 허용할 origin, method, header, credential 정책을 응답 헤더로 명시하고, 브라우저는 그 정책에 맞지 않으면 JavaScript에 응답을 넘기지 않습니다. 인증·인가는 CORS와 별개의 서버 책임입니다.

30초 답변

CORS 오류는 서버가 해당 origin, method, header, credential 사용을 허용하는 응답 헤더를 주지 않을 때 브라우저가 막는 문제입니다. 서버에서 정확한 origin allowlist, preflight 대응, credential 정책을 설정해야 합니다.

90초 답변

단순히 `*`로 열면 credential 요청과 맞지 않거나 보안 범위가 넓어집니다. Authorization header나 JSON POST는 preflight OPTIONS 요청이 먼저 갈 수 있으므로 gateway와 app이 OPTIONS를 처리해야 합니다. CORS는 브라우저 정책이지 서버 인증/인가를 대체하지 않습니다.

꼬리질문

질문답변 방향

IO-08

Q. readiness와 liveness는 어떻게 다르나요?

핵심 개념

readiness는 트래픽 수신 가능 여부, liveness는 프로세스를 재시작해야 할 정도로 비정상인지 판단합니다.

개념 해부

readiness와 liveness를 구분하지 않으면 장애를 키울 수 있습니다. readiness는 트래픽을 받을 준비가 되었는지라서 실패 시 endpoint에서 빠지고, liveness는 재시작해야 할 정도로 프로세스가 망가졌는지라서 실패 시 컨테이너가 재시작됩니다. 외부 DB가 잠깐 느린 것을 liveness 실패로 만들면 재시작 폭풍이 생길 수 있습니다.

30초 답변

readiness가 실패하면 pod는 살아 있지만 Service endpoint에서 빠져 트래픽을 받지 않습니다. liveness가 실패하면 kubelet이 컨테이너를 재시작합니다. 둘을 같은 체크로 두면 장애 때 재시작 루프를 만들 수 있습니다.

90초 답변

readiness는 DB 연결, migration 완료, warm-up처럼 요청 처리 준비 상태를 볼 수 있습니다. liveness는 deadlock이나 event loop 정지처럼 재시작이 필요한 상태만 봐야 합니다. 외부 의존성이 잠시 느리다고 liveness를 실패시키면 전체 pod가 재시작되어 장애가 커질 수 있습니다.

꼬리질문

질문답변 방향
startup probe는?초기 기동이 긴 앱에서 liveness 시작 전 여유를 주기 위해 씁니다.
DB 체크를 liveness에?보통 위험합니다. 외부 장애가 재시작 폭풍으로 번질 수 있습니다.
graceful shutdown과 관계?종료 전 readiness를 내리고 in-flight 요청을 마무리합니다.

면접에서 말할 증거

probe 설정, rollout 중 endpoint 변화, 재시작 횟수, readiness 실패 로그를 설명합니다.

오답/주의점

주의

health check를 단순 `/health` 하나로 끝내면 운영 리스크가 남습니다.

IO-09

Q. 무중단 배포와 rollback은 어떻게 설계하나요?

핵심 개념

무중단 배포는 새 버전을 천천히 투입하고, 문제가 보이면 빠르게 되돌릴 수 있도록 배포 단위와 호환성을 관리하는 것입니다.

개념 해부

무중단 배포는 서버 인스턴스를 둘로 나누는 문제가 아니라 사용자 요청, 배포 artifact, DB schema, cache, feature flag가 호환되는 상태를 만드는 문제입니다. rolling이나 canary는 새 버전의 blast radius를 줄이는 전략이고, rollback은 코드뿐 아니라 migration과 config까지 되돌릴 수 있는지 따져야 합니다.

30초 답변

rolling, blue-green, canary를 상황에 맞게 선택하고 readiness, metric, log, alert로 새 버전 상태를 확인합니다. rollback은 image만 되돌리는 것이 아니라 DB migration, config, feature flag 호환성까지 고려해야 합니다.

90초 답변

가장 깨지기 쉬운 부분은 코드와 DB schema가 동시에 바뀌는 경우입니다. expand-contract 방식으로 새 컬럼 추가, 양쪽 코드 호환, backfill, 읽기 전환, old column 제거를 나눕니다. canary는 일부 트래픽에서 5xx, latency, business metric을 보고 확대합니다.

꼬리질문

질문답변 방향
blue-green 장점?전환과 rollback이 빠르지만 비용과 데이터 호환성 문제가 있습니다.
canary 기준?error rate, latency, 핵심 전환율, error budget burn을 봅니다.
migration rollback?파괴적 변경은 되돌리기 어려우므로 단계적 호환 배포가 핵심입니다.

면접에서 말할 증거

배포 대시보드, release tag, image digest, migration version, rollback runbook을 말합니다.

오답/주의점

주의

무중단 배포를 “서버 두 대로 번갈아 배포”로만 말하면 DB와 상태 호환성을 놓칩니다.

IO-10

Q. RED, USE, Golden Signals는 무엇인가요?

핵심 개념

관측 지표를 서비스 관점과 리소스 관점으로 나눠 장애를 빠르게 감지하고 좁히는 프레임입니다.

개념 해부

RED, USE, Golden Signals는 지표를 외우는 표가 아니라 관측 대상을 나누는 프레임입니다. RED는 서비스가 사용자 요청을 어떻게 처리하는지 보고, USE는 리소스가 포화되는지 봅니다. 평균 latency보다 p95/p99가 중요한 이유는 소수의 느린 요청이 실제 사용자 경험과 장애 신호를 더 잘 드러내기 때문입니다.

30초 답변

RED는 request rate, error rate, duration으로 서비스 상태를 보고, USE는 utilization, saturation, errors로 리소스 상태를 봅니다. Golden Signals는 latency, traffic, errors, saturation을 중심으로 사용자 영향을 봅니다.

90초 답변

서비스 장애 대응에서는 먼저 사용자 영향이 있는 증상 기반 지표를 봐야 합니다. CPU가 높아도 사용자가 영향받지 않을 수 있고, CPU가 낮아도 downstream timeout 때문에 에러가 날 수 있습니다. alert는 action 가능한 기준으로 잡고, 고카디널리티 label을 metric에 남발하지 않아야 합니다.

꼬리질문

질문답변 방향
p95/p99는 왜?평균이 숨기는 tail latency를 보기 위해서입니다.
증상 기반 alert?사용자 영향이 있는 error/latency 중심 alert입니다.
고카디널리티 문제?metric label 폭증으로 비용과 성능 문제가 생깁니다.

면접에서 말할 증거

대시보드, SLO, p95 latency, error budget burn alert, traceId 연계를 말합니다.

오답/주의점

주의

CPU, 메모리만 보면 사용자 영향과 서비스 품질을 놓칠 수 있습니다.

IO-11

Q. 장애가 나면 무엇부터 확인하나요?

핵심 개념

장애 초동 대응은 원인 추측보다 영향 범위 파악, 완화, 최근 변경 확인, 계층별 분리 순서가 중요합니다.

개념 해부

장애 초동 대응은 원인을 맞히는 퀴즈가 아니라 피해를 줄이는 절차입니다. 먼저 영향 범위와 사용자 증상을 확인하고, 최근 변경과 rollback 가능성을 본 뒤, metric/log/trace로 계층을 좁힙니다. 원인 분석이 길어질 때는 feature flag off, traffic 우회, degraded mode처럼 완화 조치를 먼저 선택할 수 있어야 합니다.

30초 답변

먼저 사용자 영향과 범위를 확인합니다. 전체 장애인지 특정 기능/지역/버전인지 보고, 최근 배포와 설정 변경을 확인합니다. 동시에 rollback, feature flag off, traffic 우회 같은 완화 가능성을 검토합니다.

90초 답변

그 다음 metric으로 언제부터 나빠졌는지 보고, log와 trace로 실패 지점을 좁힙니다. DB, cache, external dependency, gateway, app 중 어디가 병목인지 나누고, 원인 분석보다 사용자 영향 완화가 먼저일 때는 rollback을 우선합니다. 대응 중에는 타임라인과 의사결정을 기록합니다.

주의

초동 대응에서 범위와 완화를 놓치고 원인 찾기에만 몰두하면 위험합니다.

IO-12

Q. postmortem에는 무엇을 남겨야 하나요?

핵심 개념

postmortem은 비난 문서가 아니라 장애를 조직 지식으로 바꾸고 재발 가능성을 줄이는 문서입니다.

개념 해부

postmortem은 누구의 실수인지 찾는 문서가 아니라 어떤 방어선이 없었거나 늦게 작동했는지 기록하는 문서입니다. 좋은 action item은 “주의”가 아니라 alert 조정, 배포 가드, 테스트 추가, runbook 개선처럼 owner와 검증 방법이 있는 작업입니다. 타임라인은 다음 장애에서 탐지와 의사결정 속도를 높이는 자료가 됩니다.

좋은 postmortem은 “무엇이 고장났나”보다 “왜 늦게 발견했나”, “왜 blast radius가 컸나”, “어떤 자동화와 테스트가 없었나”를 봅니다. action item은 관측성 개선, 배포 가드, runbook, 테스트, alert 조정처럼 실행 가능한 형태여야 합니다.

Service와 Ingress를 어떻게 나누나요?Service는 pod 집합에 대한 안정적인 내부 접근과 discovery를 제공하고, Ingress는 host/path 기반 L7 외부 진입과 TLS, routing 정책을 담당한다고 설명합니다.Ingress가 pod를 직접 찾는다고 말하거나, Service type과 Ingress controller 책임을 섞는 답변입니다.
HPA/VPA를 동시에 쓰면 무엇을 조심하나요?HPA는 replica 수를 조정하고 VPA는 request/limit 권고 또는 조정을 담당합니다. CPU metric만 보지 말고 queue lag, p95 latency, DB pool wait도 scale signal로 봅니다.pod만 늘리면 DB connection, lock, downstream limit까지 같이 늘어 장애가 커질 수 있습니다.
PDB가 있는 서비스에서 node drain은 어떻게 진행하나요?PDB가 허용하는 unavailable 수 안에서 eviction되며, readiness를 먼저 내리고 terminationGracePeriod와 preStop으로 in-flight 요청을 마무리합니다.PDB를 “pod가 절대 내려가지 않게 하는 설정”으로 이해하면 오답입니다.
network policy는 어디까지 보호하나요?namespace/pod selector 기반 east-west traffic 허용 범위를 줄입니다. 기본 deny 정책, DNS egress, monitoring, ingress controller 경로를 함께 검토합니다.인증·인가나 TLS를 network policy가 대체한다고 말하면 위험합니다.
IAM trust/resource policy와 KMS는 장애 때 어떻게 보나요?trust policy는 누가 role을 assume할 수 있는지, resource policy는 특정 리소스 접근 주체와 조건을 제한합니다. KMS key policy와 grant가 막히면 secret 복호화, object read, DB 암호화 작업이 실패할 수 있습니다.권한 오류를 애플리케이션 500으로만 보고 audit log와 denied event를 확인하지 않는 것입니다.
DR failback과 game day는 어떻게 설명하나요?failover 후 원래 region으로 돌아가는 failback은 데이터 방향, DNS TTL, write freeze, replication 재구성, config/secret 차이를 검증해야 합니다. game day는 runbook을 실제로 실행해 RTO/RPO와 누락된 의존성을 확인하는 훈련입니다.백업 존재를 DR 완료로 말하거나 failover만 있고 failback 절차가 없는 답변입니다.
90초 답변 구조 “트래픽은 Ingress에서 Service로 들어오고, Service는 readiness를 통과한 pod에만 보냅니다. 부하가 늘면 HPA/VPA를 보되 DB connection과 queue lag까지 같이 봅니다. 점검 때는 PDB와 node drain으로 가용 replica를 유지하고, network policy와 IAM/KMS로 blast radius를 줄입니다. DR은 failover뿐 아니라 failback과 game day 결과까지 있어야 운영 가능한 설계입니다.”

분산 시스템 면접

DS-01

Q. 캐시는 왜 어렵나요?

핵심 개념

캐시는 응답을 빠르게 하지만 최신성 문제를 만듭니다. 어려운 지점은 캐시 적중률보다 무효화와 일관성입니다.

개념 해부

캐시는 원본 데이터가 아니라 특정 시점의 복사본입니다. 그래서 캐시 설계의 핵심 질문은 “얼마나 빨라지는가”보다 “얼마나 오래 틀려도 되는가”입니다. 권한, 결제, 재고처럼 틀리면 손상되는 데이터와 통계, 추천처럼 잠시 늦어도 되는 데이터를 같은 정책으로 다루면 장애가 납니다.

30초 답변

캐시는 빠르지만 오래된 값을 보여줄 수 있기 때문에 어렵습니다. 그래서 어떤 데이터는 stale해도 되는지, 어떤 데이터는 반드시 최신이어야 하는지 먼저 나누고 TTL, invalidation, version key, write-through 같은 전략을 선택합니다.

90초 답변

캐시는 브라우저, CDN, 애플리케이션, DB 근처 등 여러 계층에 있을 수 있고 각 계층의 TTL이 다르면 문제를 찾기 어렵습니다. 목록 캐시, 상세 캐시, 권한별 캐시가 섞이면 사용자가 수정 직후 이전 값을 볼 수도 있습니다. 결제 상태나 권한처럼 틀리면 안 되는 데이터는 캐시보다 정합성을 우선해야 합니다.

꼬리질문

질문답변 방향
cache invalidation은?TTL, 명시적 purge, 이벤트 기반 무효화, versioned key를 상황별로 씁니다.
cache stampede는?동시에 많은 요청이 origin으로 몰리는 문제로 lock, jitter, stale-while-revalidate로 줄입니다.
권한 데이터 캐시는?권한 변경 즉시 반영 필요성이 높아 TTL과 invalidation을 매우 보수적으로 잡습니다.

오답/주의점

주의

캐시를 “성능 향상”으로만 말하지 말고 stale data와 무효화 실패를 반드시 언급하세요.

DS-02

Q. 비동기 처리와 큐는 언제 쓰나요?

핵심 개념

비동기 처리는 즉시 응답이 필요 없는 작업을 요청-응답 경로에서 분리해 latency와 결합도를 낮추는 선택입니다. 대신 운영 복잡도가 생깁니다.

개념 해부

큐는 작업을 사라지게 하는 장치가 아니라 처리 시점을 뒤로 미루고 실패를 관찰 가능한 상태로 바꾸는 장치입니다. 요청자는 빠르게 응답받을 수 있지만 consumer 실패, 중복 소비, 순서 역전, lag가 새 문제가 됩니다. 따라서 메시지에는 idempotency key, retry 정책, DLQ, 처리 상태 추적이 함께 따라와야 합니다.

30초 답변

이메일, 알림, OCR, 리포트 생성처럼 사용자가 즉시 기다리지 않아도 되는 작업은 큐로 분리할 수 있습니다. 하지만 큐를 넣으면 중복 소비, 순서 보장, retry, DLQ, consumer lag 모니터링을 함께 설계해야 합니다.

90초 답변

큐는 시스템을 빠르게만 만드는 도구가 아니라 실패 방식을 바꾸는 도구입니다. producer는 메시지를 보냈다고 생각하지만 consumer가 실패할 수 있고, 같은 메시지가 두 번 처리될 수 있으며, 순서가 바뀔 수 있습니다. 그래서 consumer는 idempotent해야 하고, 처리 실패는 DLQ나 retry policy로 관찰 가능해야 합니다.

꼬리질문

질문답변 방향
at-least-once란?최소 한 번 전달을 보장하지만 중복 가능성이 있어 idempotency가 필요합니다.
DLQ는 왜 필요한가요?계속 실패하는 poison message를 격리하고 원인을 분석하기 위해 필요합니다.
consumer lag은?producer 속도를 consumer가 따라가지 못해 처리 지연이 쌓이는 상태입니다.

오답/주의점

주의

“비동기로 하면 빨라집니다”는 절반만 맞습니다. 사용자 응답은 빨라질 수 있지만 전체 완료는 더 복잡해집니다.

DS-03

Q. Retry, timeout, idempotency는 어떻게 설계하나요?

핵심 개념

분산 호출은 실패할 수 있으므로 timeout과 retry가 필요하지만, retry는 중복 실행을 만들 수 있어 idempotency가 함께 필요합니다.

개념 해부

timeout은 기다림의 상한이고, retry는 실패를 다시 시도하는 정책이며, idempotency는 재시도 때문에 같은 부작용이 반복되지 않게 하는 계약입니다. 세 가지는 함께 설계해야 합니다. timeout 없이 retry하면 대기열이 쌓이고, idempotency 없이 retry하면 주문·결제·차감이 중복될 수 있습니다.

30초 답변

모든 외부 호출에는 timeout이 있어야 하고, 재시도는 exponential backoff와 jitter를 둡니다. 다만 결제나 주문 생성처럼 부작용이 있는 요청은 idempotency key로 같은 요청이 여러 번 와도 한 번만 처리되게 해야 합니다.

90초 답변

timeout이 없으면 장애가 대기열로 전파되고, retry가 과하면 장애 중인 시스템에 더 큰 부하를 줍니다. retry는 transient failure에만 의미가 있고 validation error 같은 영구 실패에는 하면 안 됩니다. idempotency는 request key, unique constraint, 처리 상태 저장으로 구현할 수 있습니다.

꼬리질문

질문답변 방향
jitter는 왜?동시에 재시도하는 thundering herd를 줄이기 위해 랜덤성을 넣습니다.
멱등한 메서드는?GET, PUT, DELETE는 의미상 멱등이어야 하지만 구현이 실제로 안전해야 합니다.
retry하면 안 되는 경우?권한 오류, validation 오류, 중복 결제 위험이 있는 요청은 신중해야 합니다.

오답/주의점

주의

retry를 안정성으로만 말하면 안 됩니다. retry storm과 중복 부작용을 같이 말해야 합니다.

DS-04

Q. Consistency와 availability는 어떻게 판단하나요?

핵심 개념

모든 데이터를 항상 강하게 일치시키는 것은 비쌉니다. 도메인별로 즉시 맞아야 하는 데이터와 eventual consistency를 허용할 수 있는 데이터를 나눠야 합니다.

개념 해부

일관성은 시스템 전체의 추상 목표가 아니라 사용자와 도메인이 허용할 수 있는 불일치의 범위입니다. 강한 일관성은 이해하기 쉽지만 latency와 availability 비용이 있고, eventual consistency는 확장에 유리하지만 UX가 상태를 설명해야 합니다. “처리 중”, “동기화 중”, “마지막 갱신 시각”은 기술적 불일치를 사용자에게 정직하게 보여주는 장치입니다.

30초 답변

결제, 재고, 정산은 강한 정합성이 중요하고, 피드, 추천, 통계는 잠시 늦게 맞아도 되는 경우가 많습니다. 그래서 consistency와 availability는 기술 선택이 아니라 비즈니스가 어떤 불일치를 허용할 수 있는지의 판단입니다.

90초 답변

분산 시스템에서는 네트워크 지연과 장애가 기본값입니다. 모든 요청에 강한 정합성을 요구하면 latency와 장애 전파가 커질 수 있습니다. 반대로 eventual consistency를 쓰면 사용자에게 처리 중, 동기화 중, 마지막 업데이트 시각 같은 상태를 보여줘야 혼란이 줄어듭니다.

꼬리질문

질문답변 방향
eventual consistency UX는?pending, processing, synced, failed 같은 상태를 명확히 보여줍니다.
읽기-your-writes는?사용자가 방금 쓴 값을 다시 읽을 수 있게 세션/primary read 등을 고려합니다.
CAP를 어떻게 설명하나요?네트워크 partition 상황에서 consistency와 availability 사이 선택이 필요하다는 모델로 설명합니다.

오답/주의점

주의

CAP 정의만 외우지 말고 결제, 재고, 통계 같은 도메인 예시로 판단해야 합니다.

DS-05

Q. Saga와 Outbox는 왜 쓰나요?

핵심 개념

서비스가 분리되면 하나의 DB 트랜잭션으로 모든 변경을 묶기 어렵습니다. Saga와 Outbox는 분산된 변경의 정합성을 현실적으로 관리하는 패턴입니다.

개념 해부

Outbox는 DB 변경과 이벤트 발행 사이의 원자성 간극을 줄이는 패턴이고, Saga는 여러 로컬 트랜잭션을 보상 가능한 workflow로 묶는 패턴입니다. 둘 다 분산 트랜잭션을 마법처럼 없애는 것이 아니라 실패 지점을 명시하고 재처리 가능하게 만드는 방법입니다. 따라서 event id, 상태 저장, 보상 동작, 관측성이 핵심입니다.

30초 답변

Outbox는 DB 변경과 이벤트 발행 사이의 간극을 줄이기 위해 같은 DB 트랜잭션에 이벤트 기록을 남기고 나중에 발행하는 패턴입니다. Saga는 여러 서비스에 걸친 작업을 로컬 트랜잭션과 보상 트랜잭션으로 관리합니다.

90초 답변

예를 들어 주문 생성, 결제 승인, 재고 차감, 알림 발송이 여러 서비스에 나뉘면 중간 실패가 생길 수 있습니다. 강한 분산 트랜잭션은 결합과 장애 포인트가 커질 수 있어, 각 단계의 완료와 실패를 이벤트로 관리하고 실패 시 보상 동작을 수행하는 방식이 필요합니다. 다만 Saga는 설계와 관측이 복잡하므로 단일 서비스에서 충분한 경우에는 과할 수 있습니다.

꼬리질문

질문답변 방향
Outbox가 막는 문제는?DB commit은 됐지만 이벤트 발행이 실패하는 간극을 줄입니다.
보상 트랜잭션은?이미 완료된 단계를 비즈니스적으로 되돌리는 별도 동작입니다.
MSA는 언제 과한가요?팀 경계, 데이터 소유권, 배포 독립성이 없으면 네트워크 비용만 늘 수 있습니다.

오답/주의점

주의

Saga와 Outbox를 최신 패턴처럼 말하지 말고, 어떤 실패 간극을 줄이는지 설명하세요.

DS-06

Q. cache invalidation 전략은 어떻게 잡나요?

핵심 개념

cache invalidation은 캐시에 무엇을 언제 지울지 정하는 문제이며 최신성, 성능, 구현 복잡도의 trade-off입니다.

개념 해부

cache invalidation은 쓰기 경로와 읽기 경로가 만나는 지점입니다. 상세 캐시만 지워도 목록 캐시가 남으면 사용자는 이전 값을 볼 수 있고, 이벤트 기반 무효화는 이벤트 유실과 순서 문제를 만듭니다. TTL은 단순하지만 stale window를 허용한다는 명시적 선택입니다.

30초 답변

데이터별로 stale 허용 시간을 먼저 정합니다. 비교적 덜 중요한 데이터는 TTL로 단순화하고, 권한이나 가격처럼 민감한 데이터는 명시적 purge, versioned key, event 기반 invalidation을 씁니다.

90초 답변

목록과 상세 캐시가 함께 있으면 하나만 지워서는 사용자에게 오래된 값이 보일 수 있습니다. write-through는 쓰기 경로가 복잡해지고, write-behind는 유실 위험을 봐야 합니다. 캐시 stampede를 막기 위해 jitter, lock, stale-while-revalidate도 고려합니다.

꼬리질문

질문답변 방향
TTL만 쓰면?단순하지만 변경 직후 stale 데이터가 보일 수 있습니다.
event 기반은?정교하지만 이벤트 유실과 순서 문제를 고려해야 합니다.
versioned key는?무효화 대신 새 key로 전환하지만 오래된 key 정리가 필요합니다.

면접에서 말할 증거

cache hit ratio, stale 데이터 신고, purge 로그, Redis key 설계, TTL 정책 표를 말합니다.

오답/주의점

주의

캐시를 추가하는 순간 최신성 책임도 함께 생깁니다.

DS-07

Q. 메시지 순서 보장은 어떻게 다루나요?

핵심 개념

분산 메시징에서 순서 보장은 partition, key, consumer 병렬성, retry 정책에 의해 깨질 수 있습니다.

개념 해부

메시지 순서 보장은 보통 전체 시스템 순서가 아니라 특정 entity 단위 순서의 문제입니다. 전체 순서를 보장하려 하면 병렬성과 처리량을 크게 잃습니다. 주문 id, 사용자 id 같은 key 기준으로 partition을 정하고, event version이나 sequence로 out-of-order 적용을 막는 것이 현실적입니다.

30초 답변

순서가 필요한 범위를 먼저 정합니다. 전체 순서가 필요한지, 주문 id나 사용자 id 단위 순서만 필요한지 나눈 뒤 같은 key를 같은 partition으로 보내고 consumer 병렬성을 조정합니다.

90초 답변

전체 순서 보장은 처리량을 크게 낮출 수 있습니다. 대부분은 entity별 순서면 충분합니다. retry나 DLQ 처리 과정에서 나중 이벤트가 먼저 적용될 수 있으므로 event version, sequence number, idempotent consumer가 필요합니다.

질문답변 방향

DS-08

Q. circuit breaker는 언제 필요한가요?

핵심 개념

circuit breaker는 실패 중인 의존성에 계속 요청을 보내 장애를 증폭시키지 않도록 호출을 빠르게 차단하는 패턴입니다.

개념 해부

circuit breaker는 실패한 의존성을 고치는 장치가 아니라 실패 중인 의존성으로 장애가 확산되는 것을 막는 장치입니다. timeout과 retry만 있으면 느린 서비스에 요청이 계속 쌓일 수 있고, breaker는 일정 실패율에서 빠른 실패 또는 fallback으로 전환합니다. half-open은 회복 여부를 제한적으로 확인하는 상태입니다.

30초 답변

외부 API나 downstream 서비스가 느리거나 실패할 때 timeout과 retry만으로는 부족할 수 있습니다. 실패율이 임계치를 넘으면 circuit을 열어 빠르게 실패시키고, 일정 시간 뒤 half-open으로 일부 요청만 보내 회복을 확인합니다.

90초 답변

retry는 transient failure에는 도움이 되지만 장애 중인 서비스에는 부하를 더할 수 있습니다. circuit breaker는 bulkhead, timeout, fallback과 같이 써야 효과가 있습니다. 다만 임계치가 과하면 정상 요청도 막을 수 있어 metric과 알림이 필요합니다.

꼬리질문

질문답변 방향
fallback은?캐시된 값, 제한된 기능, 안내 메시지를 제공하는 방식입니다.
half-open은?회복 확인을 위해 일부 요청만 통과시키는 상태입니다.
retry와 관계?retry storm을 막기 위해 timeout/backoff와 함께 조정합니다.

면접에서 말할 증거

downstream failure rate, timeout budget, fallback UX, breaker state metric을 말합니다.

오답/주의점

주의

circuit breaker는 장애를 없애는 기능이 아니라 장애 전파를 줄이는 장치입니다.

DS-09

Q. degraded mode는 어떻게 설계하나요?

핵심 개념

degraded mode는 일부 의존성이 실패해도 핵심 기능을 제한된 품질로 계속 제공하는 전략입니다.

개념 해부

degraded mode는 장애를 숨기는 것이 아니라 핵심 기능을 보호하기 위해 품질이나 기능 범위를 의도적으로 낮추는 운영 모드입니다. 추천, 통계, 알림은 제한하고 주문, 로그인, 조회 같은 핵심 경로를 살릴 수 있습니다. 사용자가 보는 데이터가 stale하거나 제한된 기능이라면 UI가 그 상태를 명확히 말해야 합니다.

30초 답변

모든 기능을 같은 중요도로 보지 않고 핵심 경로와 부가 기능을 나눕니다. 추천, 통계, 알림이 실패해도 주문이나 로그인 같은 핵심 기능은 유지하도록 fallback, cached data, read-only mode를 설계합니다.

90초 답변

사용자에게는 기능이 제한되었음을 명확히 보여줘야 합니다. stale 데이터라면 마지막 갱신 시각을 표시하고, 쓰기 기능이 위험하면 read-only로 전환합니다. degraded mode도 운영자가 켜고 끌 수 있어야 하며, 어떤 지표에서 발동할지 기준이 필요합니다.

꼬리질문

질문답변 방향

사용자 안내는?제한 상태와 재시도 가능성을 명확히 보여줍니다.

feature flag, fallback 화면, stale 표시, 장애 모드 runbook을 말합니다.

오답/주의점

주의

장애 때 모든 기능을 계속 정상처럼 보이게 하면 데이터 신뢰를 잃을 수 있습니다.

DS-10

Q. microservices는 언제 적절하고 언제 과한가요?

핵심 개념

MSA는 기술 구조가 아니라 팀 경계, 데이터 소유권, 배포 독립성, 장애 격리를 위한 조직적 선택입니다.

개념 해부

microservices의 비용은 HTTP 호출이 늘어나는 정도가 아니라 데이터 소유권, 배포, 관측성, 장애 복구 방식이 모두 분산된다는 점입니다. 경계가 약한 상태에서 서비스를 나누면 독립 배포는 못 하고 네트워크 비용만 갖는 분산 모놀리스가 됩니다. 먼저 bounded context와 팀 소유권이 분명한지 확인해야 합니다.

30초 답변

도메인 경계가 명확하고 팀이 독립 배포와 운영을 감당할 수 있을 때 microservices가 유리합니다. 반대로 데이터 소유권이 불분명하고 팀 규모가 작으면 네트워크, 배포, 관측성, 정합성 비용만 늘 수 있습니다.

90초 답변

서비스 분리는 DB join을 API 호출로 바꾸고, 트랜잭션을 saga나 eventual consistency 문제로 바꿉니다. 그래서 먼저 모듈러 모놀리스로 경계를 명확히 하고, 실제 병목이나 팀 독립성이 필요할 때 분리하는 접근이 현실적입니다.

꼬리질문

질문답변 방향
공유 DB는?서비스 독립성을 깨므로 데이터 소유권을 분리해야 합니다.
분리 기준은?bounded context, 변경 빈도, 팀 소유권, scaling 요구를 봅니다.
분산 모놀리스는?배포는 나뉘었지만 강한 동기 의존으로 독립성이 없는 상태입니다.

면접에서 말할 증거

서비스 경계도, 배포 독립성, 장애 격리 사례, 데이터 소유권 표를 말합니다.

오답/주의점

주의

MSA를 확장성의 기본 정답처럼 말하면 운영 비용을 과소평가한 것입니다.

DS-11

Q. read-your-writes는 왜 중요하고 어떻게 보장하나요?

핵심 개념

read-your-writes는 사용자가 방금 쓴 데이터를 바로 다시 읽을 수 있게 하는 사용자 체감 정합성입니다.

개념 해부

read-your-writes는 이론적 정합성보다 사용자 신뢰에 가까운 요구사항입니다. 저장 성공 후 바로 이전 값이 보이면 사용자는 실패했다고 느낍니다. replica lag가 있는 구조에서는 쓰기 직후 일정 시간 primary를 읽거나, session 단위 라우팅, optimistic UI와 rollback, pending 상태 표시를 조합합니다.

30초 답변

replica lag이나 비동기 동기화가 있으면 저장 성공 후 바로 조회했는데 이전 값이 보일 수 있습니다. 이 경우 일정 시간 primary read, session stickiness, optimistic UI, 동기화 상태 표시를 고려합니다.

90초 답변

모든 읽기를 primary로 보내면 비용이 커지고 replica 사용 이점이 줄어듭니다. 그래서 쓰기 직후 사용자나 특정 resource에 한해 primary를 읽거나, write timestamp보다 최신 replica만 선택하는 방식이 있습니다. 사용자에게 처리 중 상태를 보여주는 것도 UX 보정입니다.

꼬리질문

질문답변 방향

언제 중요한가요?프로필 수정, 결제 상태, 권한 변경처럼 사용자 신뢰가 걸린 경우입니다.

면접에서 말할 증거

오답/주의점

주의

outbox CDC vs polling은 어떻게 고르나요?CDC는 DB log 기반으로 지연과 부하가 낮고 누락 가능성이 작지만 운영 복잡도가 있습니다. polling은 단순하지만 지연, 중복, scan 부하, lock 경합을 관리해야 합니다.outbox를 쓰면 재처리와 idempotency가 필요 없다고 말합니다.
inbox/processed table은 왜 두나요?consumer가 처리한 event id와 결과를 저장해 retry·중복 delivery에도 side effect가 한 번만 적용되게 합니다. TTL, unique constraint, transaction boundary가 중요합니다.consumer 코드가 멱등하다고만 말하고 저장소 기반 증거를 두지 않습니다.
reconciliation은 무엇을 검증하나요?source of truth와 projection, broker offset과 DB 상태, 결제 provider와 내부 주문 상태를 주기적으로 비교해 누락·중복·불일치를 찾아 보정합니다.eventual consistency라 시간이 지나면 알아서 맞는다고 말합니다.
90초 답변 구조 “돈·권한·재고처럼 성공 직후 신뢰가 필요한 경로는 linearizability나 강한 transaction 경계를 우선합니다. 그 외 projection, 검색, 통계는 eventual consistency를 허용하되 outbox CDC 또는 polling으로 이벤트를 발행하고, consumer에는 inbox/processed table을 둡니다. exactly-once는 broker 용어와 비즈니스 부작용을 분리해서 설명하고, 마지막에는 reconciliation으로 수렴 여부를 검증합니다.”

DS-17

분산 정합성 사례: 주문·결제·재고 플로우

분산 시스템을 전문가처럼 설명하려면 패턴 이름이 아니라 실패가 실제로 어디서 생기는지 말해야 합니다. 주문·결제·재고 플로우는 retry, idempotency, outbox, saga, reconciliation을 한 번에 설명하기 좋은 사례입니다.

EXPERT DISTRIBUTED CONSISTENCY CASE사용자가 주문 버튼을 누르면 주문 생성, 결제 승인, 재고 차감, 알림 발송이 이어집니다. 이 네 단계가 하나의 DB 안에 있으면 transaction으로 묶을 수 있지만, 결제 provider와 재고 서비스, 알림 서비스가 분리되면 네트워크 실패와 부분 성공이 기본값이 됩니다. 결제는 성공했는데 주문 DB update가 실패할 수 있고, 주문은 생성됐지만 이벤트 발행이 실패할 수 있으며, consumer가 재시작하면서 같은 메시지를 두 번 처리할 수 있습니다.

1. Idempotency는 API와 consumer 양쪽에 필요하다

사용자가 버튼을 두 번 누르거나 client timeout 후 재시도하면 주문 생성 요청이 두 번 들어올 수 있습니다. API는 idempotency key, payload hash, status, response snapshot을 저장해 같은 요청이 같은 결과를 돌려주도록 합니다. 하지만 이것만으로 충분하지 않습니다. 메시지 broker가 at-least-once delivery를 제공하면 consumer도 같은 event를 두 번 받을 수 있습니다. 그래서 consumer에는 inbox 또는 processed_message table을 두고 event id에 unique constraint를 걸어 side effect가 한 번만 실행되게 합니다.

2. Outbox는 DB commit과 event publish 사이 간극을 줄인다

주문 DB에 주문을 저장한 뒤 broker에 OrderCreated를 publish하는 구조에서는 DB commit은 성공했지만 publish가 실패하는 간극이 생깁니다. Outbox는 주문 저장과 outbox event 저장을 같은 DB transaction으로 묶습니다. 별도 publisher가 outbox row를 읽어 broker에 발행합니다. 여기서도 발행 중복은 가능하므로 event id와 consumer idempotency가 필요합니다. Outbox는 exactly-once 마법이 아니라 “누락을 찾고 재발행할 수 있는 durable record”를 만드는 패턴입니다.

3. Saga는 rollback이 아니라 보상 가능한 workflow다

결제가 승인된 뒤 재고 차감이 실패하면 DB transaction처럼 자동 rollback할 수 없습니다. 이미 외부 결제 provider에 부작용이 발생했기 때문입니다. Saga는 각 단계의 상태를 저장하고, 실패 시 보상 동작을 실행합니다. 예를 들어 재고 차감 실패 시 결제 취소 요청을 보내고 주문 상태를 PAYMENT_CANCEL_REQUESTED로 바꿉니다. 보상도 실패할 수 있으므로 manual review queue와 운영자 runbook이 필요합니다. “Saga를 쓰면 해결됩니다”가 아니라 “보상 실패를 어디에 저장하고 누가 처리하는가”까지 답해야 합니다.

4. Exactly-once의 오해

Kafka나 특정 broker가 exactly-once semantics를 제공한다고 해도 외부 결제 API, 이메일 발송, DB update 같은 business side effect가 정확히 한 번만 일어난다는 뜻은 아닙니다. broker 내부의 producer/consumer transaction과 외부 시스템의 부작용은 다른 경계입니다. 실무 답변은 at-least-once delivery를 전제로 idempotency, dedupe key, unique constraint, processed table, reconciliation을 둔다고 말하는 편이 더 안전합니다.

5. Reconciliation은 eventual consistency의 안전망이다

eventual consistency는 시간이 지나면 자동으로 맞는다는 뜻이 아닙니다. 맞도록 만드는 작업이 있어야 합니다. reconciliation job은 결제 provider의 승인 내역과 내부 주문 상태, 재고 차감 이벤트와 재고 ledger, outbox 발행 상태와 consumer processed 상태를 비교합니다. 불일치가 있으면 재발행, 보상, manual review로 보냅니다. 운영 지표는 outbox backlog, DLQ count, consumer lag, reconciliation mismatch count, compensation failure count입니다.

면접에서의 90초 답변

“주문·결제·재고가 분리된 시스템에서는 네트워크 실패와 중복 처리를 기본값으로 둡니다. 주문 생성 API에는 idempotency key를 저장하고, 주문 DB 변경과 OrderCreated 이벤트는 outbox로 같은 transaction에 기록합니다. consumer는 processed table로 중복 event를 막고, 결제 승인 후 재고 실패처럼 부분 성공이 생기면 saga state와 보상 트랜잭션을 실행합니다. exactly-once라는 표현은 broker 내부 보장과 비즈니스 부작용을 분리해서 봐야 하며, 최종적으로 reconciliation job으로 provider 상태와 내부 상태가 수렴했는지 검증합니다.”

분산 시스템 답변에서 “큐를 쓰면 됩니다”, “Kafka면 됩니다”, “eventual consistency입니다”는 초보 신호입니다. 실패 기록이 어디에 남고, 중복을 어떤 key로 막고, 보상 실패를 누가 처리하며, 수렴 여부를 어떤 job과 지표로 확인하는지 말해야 합니다.

개념 해부

클라우드는 서버를 빌리는 것이 아니라 물리 인프라 일부를 provider에게 넘기고, 네트워크·권한·비용·가용성 설계 책임을 더 명시적으로 갖는 모델입니다. managed service도 설정, 접근 제어, 백업 정책, 장애 대응은 사용자 책임으로 남습니다. 면접에서는 shared responsibility를 기준으로 어디까지 직접 책임지는지 말해야 합니다.

CLD-02

Q. VPC, subnet, public/private 배치를 설명해보세요.

핵심 개념

VPC는 클라우드 안의 논리적 네트워크 경계이고, subnet은 그 경계 안에서 CIDR과 가용 영역 단위로 리소스를 배치하는 영역입니다. public/private 구분은 이름이 아니라 route table과 노출 정책으로 결정됩니다.

개념 해부

VPC는 논리적으로 격리된 네트워크 경계이고, subnet은 그 안에서 주소 범위와 routing 성격을 나누는 단위입니다. public/private은 리소스 자체 속성이 아니라 route table과 internet-facing 경로가 있는지로 결정됩니다. DB를 private subnet에 두는 이유는 인터넷에서 직접 도달하지 못하게 blast radius를 줄이기 위해서입니다.

30초 답변

VPC는 서비스 네트워크의 큰 경계이고 subnet은 그 안의 주소 대역과 배치 단위입니다. public subnet은 인터넷 게이트웨이로 나가는 경로를 갖고 외부 진입점인 load balancer나 bastion을 둡니다. private subnet에는 app server, DB, cache처럼 외부에서 직접 접근하면 안 되는 리소스를 둡니다.

90초 답변

좋은 설명은 “어디에 무엇을 둔다”보다 요청 흐름을 말하는 것입니다. 사용자는 인터넷에서 load balancer로 들어오고, load balancer는 private subnet의 app으로 전달하며, app은 private DB에 접근합니다. outbound가 필요한 private 리소스는 NAT Gateway나 VPC endpoint를 사용합니다. 고가용성을 위해 subnet은 여러 AZ에 나누고, 각 AZ 장애 때도 최소 용량을 유지할 수 있게 설계합니다.

CLD-03

Q. Internet Gateway, NAT Gateway, route table은 어떻게 다르나요?

핵심 개념

Internet Gateway는 VPC와 인터넷 사이의 출입구이고, NAT Gateway는 private subnet 리소스의 outbound 인터넷 접근을 제공합니다. route table은 패킷을 어디로 보낼지 정하는 규칙입니다.

개념 해부

Internet Gateway는 VPC가 인터넷과 통신할 출입구이고, NAT Gateway는 private subnet의 outbound 인터넷 접근을 대리합니다. route table은 목적지 CIDR별로 다음 hop을 결정하는 규칙표입니다. private resource가 외부 패키지를 받거나 API를 호출할 수는 있어도 외부에서 직접 들어오지 못하게 만드는 것이 NAT의 핵심입니다.

30초 답변

public subnet은 route table에 0.0.0.0/0 → Internet Gateway 경로가 있어 인터넷과 직접 통신할 수 있습니다. private subnet은 외부에서 직접 들어오는 경로를 열지 않고, 필요할 때 0.0.0.0/0 → NAT Gateway로 outbound만 허용합니다. 그래서 NAT는 private 리소스를 공개하는 장치가 아니라 외부로 나가는 길입니다.

90초 답변

운영에서는 NAT 비용과 단일 장애 지점도 봐야 합니다. AZ별 NAT를 두지 않으면 한 AZ 장애 또는 cross-AZ routing 비용이 문제가 될 수 있습니다. 패키지 설치, 외부 API 호출, 이미지 pull 같은 outbound가 많은 서비스는 NAT 비용이 커질 수 있으므로 VPC endpoint, private registry, egress 제한 정책을 함께 검토합니다.

route 충돌은?가장 구체적인 prefix가 우선되므로 peering, VPN, endpoint route를 함께 확인합니다.

오답/주의점

주의

NAT Gateway를 “private subnet에 접속하게 해주는 장치”라고 말하면 방향을 반대로 이해한 것입니다.

CLD-04

Q. Security Group과 NACL은 어떻게 다르고 어떻게 설계하나요?

핵심 개념

Security Group은 리소스 단위의 stateful firewall이고, NACL은 subnet 단위의 stateless 방화벽입니다. 대부분의 애플리케이션 접근 제어는 SG를 중심으로 설계하고, NACL은 넓은 네트워크 방어선으로 보조합니다.

개념 해부

Security Group은 instance나 ENI에 붙는 stateful 방화벽이고, NACL은 subnet 경계에서 동작하는 stateless 규칙입니다. stateful이라는 말은 허용된 inbound에 대한 응답 outbound를 따로 열지 않아도 된다는 뜻입니다. 설계는 넓게 열고 막는 방식보다 필요한 source, port, protocol을 최소화하는 방식이어야 합니다.

30초 답변

Security Group은 인스턴스나 DB 같은 리소스에 붙고 stateful이라 inbound를 허용하면 응답 outbound는 자동으로 허용됩니다. NACL은 subnet에 적용되고 stateless라 inbound와 outbound 규칙을 모두 명시해야 합니다. DB는 0.0.0.0/0에 열지 않고 app security group을 source로 지정합니다.

90초 답변

면접에서는 “포트를 열었다”가 아니라 어떤 신뢰 경계에서 어떤 리소스로만 접근 가능한지 말해야 합니다. 예를 들어 ALB는 443을 인터넷에 열 수 있지만 app은 ALB SG에서 오는 트래픽만 받고, DB는 app SG에서 오는 5432만 받습니다. 운영 접근은 개인 IP 허용보다 VPN, SSM, role, MFA, audit log를 우선합니다.

꼬리질문

질문답변 방향
SG를 너무 많이 만들면?추적이 어려워지므로 계층별 표준 SG와 태그, IaC 리뷰가 필요합니다.
NACL은 언제 유용한가요?subnet 단위 차단, 임시 격리, 명시적 deny가 필요한 경계에서 유용합니다.
운영자가 DB에 접근해야 하면?직접 공개하지 않고 VPN/SSM/bastion, 권한 승인, 감사 로그, 시간 제한을 둡니다.

오답/주의점

주의

보안 그룹을 “클라우드 방화벽”으로만 말하면 얕습니다. source를 CIDR이 아니라 다른 SG로 제한하는 설계를 말해야 합니다.

CLD-05

Q. Load Balancer는 단순 트래픽 분산 외에 어떤 역할을 하나요?

핵심 개념

Load Balancer는 트래픽 분산뿐 아니라 health check, TLS termination, routing, rollout 안정성, 장애 격리의 중심 컴포넌트입니다.

개념 해부

Load Balancer는 단순 분산기가 아니라 health check를 통해 비정상 target을 제외하고 TLS termination, path routing, sticky session, connection 관리까지 담당할 수 있습니다. LB가 정상이어도 target group health가 실패하면 사용자는 장애를 봅니다. 따라서 app health endpoint와 readiness 기준이 배포 안정성의 일부가 됩니다.

30초 답변

Load Balancer는 여러 app instance로 요청을 분산하고 health check에 실패한 target을 제외합니다. TLS 인증서를 관리하고 path나 host 기반 routing을 처리하며, rolling 배포 중 준비되지 않은 인스턴스에 트래픽이 가지 않도록 도와줍니다.

90초 답변

ALB 같은 L7 LB는 host/path routing, header 기반 처리, sticky session, WebSocket 지원 등을 볼 수 있고, NLB 같은 L4 LB는 낮은 지연과 TCP/UDP 처리에 유리합니다. health check는 너무 단순하면 죽은 앱을 살았다고 보고, 너무 무거우면 의존성 장애가 전체 장애로 번질 수 있어 readiness 성격을 명확히 해야 합니다.

꼬리질문

질문답변 방향
TLS termination은 어디서 하나요?LB에서 종료하면 인증서와 암호 정책 관리가 쉬워지지만 내부 구간 암호화 요구도 확인합니다.
sticky session은 좋은가요?상태ful 설계를 숨길 수 있어 가능하면 session store 분리를 우선합니다.
health check endpoint는?프로세스 생존, 주요 dependency, readiness를 목적에 맞게 분리합니다.

면접에서 말할 증거

target group health, 5xx 원인 구분, 배포 중 deregistration delay, access log를 봤던 경험을 연결합니다.

CLD-06

Q. IAM과 Secret 관리는 어떻게 설명하나요?

핵심 개념

IAM은 누가 어떤 리소스에 어떤 action을 할 수 있는지 정의하는 권한 시스템입니다. secret 관리는 비밀값을 코드, 이미지, 로그에 노출하지 않고 저장·주입·회전·감사하는 운영 체계입니다.

개념 해부

IAM은 사람이 아니라 workload와 service까지 포함해 누가 무엇을 할 수 있는지 정하는 권한 시스템입니다. access key를 코드에 넣는 대신 role과 temporary credential을 쓰는 이유는 유출 범위와 수명을 줄이기 위해서입니다. least privilege는 이상론이 아니라 사고 시 blast radius를 제한하는 운영 장치입니다.

30초 답변

IAM은 최소 권한 원칙으로 사용자, 서비스, CI/CD, batch worker의 역할을 분리합니다. 장기 access key보다 role 기반 임시 credential을 우선하고, secret은 repository나 container image에 넣지 않고 secret manager에서 관리합니다. rotation, audit log, 접근 권한 검토도 함께 봅니다.

90초 답변

실무에서는 “admin 권한 하나로 다 한다”가 가장 위험합니다. CI/CD는 배포에 필요한 권한만, app은 특정 DB secret read나 object storage prefix 접근만, 운영자는 승인된 시간에 필요한 role만 사용하게 합니다. secret rotation이 가능한 구조인지, rotation 중 구버전과 신버전이 공존 가능한지, 로그와 crash dump에 비밀이 남지 않는지도 확인합니다.

꼬리질문

질문답변 방향
access key가 유출되면?즉시 revoke/rotate, CloudTrail 같은 audit 확인, 영향 범위 산정, 탐지 룰 보강을 말합니다.
권한을 어떻게 리뷰하나요?IaC diff, permission boundary, last used, role assumption log, 정기 access review를 봅니다.
환경변수는 안전한가요?주입 방식일 뿐이며 저장소, 로그, 프로세스 노출, rotation 정책이 더 중요합니다.

오답/주의점

주의

secret을 “환경변수로 빼면 끝”이라고 말하면 운영 수준이 낮아 보입니다. 저장, 접근, 주입, 회전, 감사까지 말해야 합니다.

CLD-07

Q. Managed DB를 쓸 때 무엇을 여전히 직접 책임져야 하나요?

핵심 개념

Managed DB는 운영 부담을 줄이지만 데이터 모델, 쿼리 성능, 권한, 백업 정책, 복구 검증, 마이그레이션 안정성은 애플리케이션 팀의 책임으로 남습니다.

개념 해부

Managed DB는 patching, failover, backup 자동화를 제공하지만 schema, query, index, connection pool, migration 위험은 여전히 애플리케이션 팀 책임입니다. 장애가 provider 영역인지, 설정/쿼리/락 문제인지 구분할 수 있어야 합니다. managed라고 해서 데이터 모델링과 운영 검증이 사라지지 않습니다.

30초 답변

Managed DB는 패치, 백업 기능, replication, failover 같은 운영 기능을 제공하지만 slow query, index, schema migration, connection pool, 권한 설계, 데이터 복구 훈련은 직접 챙겨야 합니다. managed라고 해서 DB 운영 리스크가 사라지는 것은 아닙니다.

90초 답변

면접에서는 DB를 private subnet에 둔다에서 멈추면 안 됩니다. connection 수가 app autoscaling과 맞는지, backup retention과 PITR이 요구사항을 만족하는지, read replica lag를 허용할 수 있는지, failover 중 connection 재시도가 안전한지, migration이 lock을 오래 잡지 않는지까지 말해야 합니다.

꼬리질문

질문답변 방향
RPO/RTO는?얼마나 잃을 수 있는지와 얼마나 빨리 복구해야 하는지를 수치로 정의합니다.
read replica는 언제 쓰나요?읽기 부하 분산에 쓰지만 stale read와 lag monitoring이 필요합니다.
DB failover 후 앱은?connection 재생성, retry, idempotency, pool 설정을 확인합니다.

면접에서 말할 증거

slow query 대응, connection pool 조정, backup restore drill, migration dry-run 경험을 말하면 신뢰도가 올라갑니다.

CLD-08

Q. Object Storage와 CDN은 언제 쓰고 어떤 보안 이슈가 있나요?

핵심 개념

Object Storage는 파일과 정적 객체 저장에 적합하고, CDN은 사용자 가까운 edge에서 콘텐츠를 캐싱해 latency와 origin 부하를 줄입니다. 핵심은 공개 범위와 캐시 무효화 정책입니다.

개념 해부

Object Storage는 파일 시스템이 아니라 key 기반 object 저장소이고, CDN은 그 object나 응답을 edge에 캐시합니다. 공개 bucket, presigned URL 만료, cache-control, content-type, 개인정보 포함 여부가 핵심 보안 지점입니다. 파일 URL 하나가 권한 우회 통로가 되지 않게 접근 경계를 설계해야 합니다.

30초 답변

Object Storage는 이미지, 첨부파일, 백업, 정적 자산 저장에 쓰고 CDN은 public read 콘텐츠의 지연과 origin 부하를 줄이는 데 씁니다. private 파일은 bucket public open을 피하고 presigned URL, 짧은 만료, content-type 검증, 권한 scope를 관리해야 합니다.

90초 답변

CDN은 성능 도구이면서 운영 리스크도 만듭니다. cache-control을 잘못 잡으면 배포 후 오래된 JS가 남거나 민감한 응답이 edge에 캐시될 수 있습니다. 업로드 파일은 확장자보다 MIME 검증, 크기 제한, 악성 파일 스캔, path traversal 방지, tenant별 prefix 권한을 함께 봐야 합니다.

꼬리질문

질문답변 방향
presigned URL은 안전한가요?짧은 만료, 권한 범위, 파일명/타입 검증, 로그 추적이 있어야 안전합니다.
CDN invalidation은 언제?HTML이나 mutable asset은 invalidation이 필요하고, hashed asset은 장기 캐시가 가능합니다.
private bucket인데 유출되면?정책, ACL, public access block, signed URL 발급 로그, object access log를 확인합니다.

오답/주의점

주의

“S3에 올리고 URL을 저장합니다”로 끝내면 권한, 만료, 캐시, 파일 검증 리스크를 놓칩니다.

CLD-09

Q. Queue와 Event 서비스를 클라우드에서 쓰는 이유는 무엇인가요?

핵심 개념

Queue와 Event 서비스는 요청 처리와 후속 작업을 분리해 peak 부하를 흡수하고, 실패한 작업을 재시도하며, 시스템 간 결합도를 낮춥니다.

개념 해부

Queue와 Event 서비스는 서비스 간 시간 결합을 끊지만 중복, 지연, 순서 문제를 새로 만듭니다. producer가 메시지를 보냈다는 사실과 consumer가 처리했다는 사실은 다릅니다. DLQ, retry, idempotent consumer, lag metric이 없으면 비동기 처리는 장애를 늦게 발견하게 만들 수 있습니다.

30초 답변

사용자 요청에서 오래 걸리거나 실패 가능성이 높은 작업은 queue로 분리할 수 있습니다. 예를 들어 이메일 발송, 파일 변환, 외부 API 동기화는 메시지로 넘기고 worker가 처리합니다. 이때 retry, DLQ, idempotency, visibility timeout을 같이 설계해야 합니다.

90초 답변

클라우드 managed queue는 운영 부담을 줄이지만 중복 전달, 순서 보장 제한, poison message, retry 폭주 같은 분산 시스템 문제가 남습니다. 이벤트 기반 설계에서는 “발행하면 끝”이 아니라 consumer 실패, schema 변경, 재처리, 관측성을 함께 설계해야 합니다.

꼬리질문

질문답변 방향
메시지가 중복 처리되면?idempotency key, 처리 상태 저장, unique constraint, deduplication을 둡니다.
DLQ는 왜 필요한가요?반복 실패 메시지를 격리해 전체 queue 지연을 막고 원인을 분석합니다.
queue lag가 늘면?worker autoscaling, 외부 dependency 지연, poison message, retry 정책을 봅니다.

면접에서 말할 증거

queue depth, age of oldest message, retry count, DLQ 처리 runbook을 운영 지표로 말합니다.

CLD-10

Q. 고가용성과 DR은 어떻게 설명하나요?

핵심 개념

고가용성은 평상시 장애를 견디는 설계이고, DR은 큰 장애 후 서비스를 복구하는 계획입니다. 둘 다 RTO와 RPO 기준이 있어야 합니다.

개념 해부

고가용성은 instance 수가 아니라 failure domain을 나눠 한 영역 장애가 전체 장애로 번지지 않게 하는 설계입니다. DR은 데이터 손실 허용치인 RPO와 복구 시간 목표인 RTO를 먼저 정해야 현실적인 구조가 나옵니다. active-active는 강력하지만 비용과 데이터 정합성 복잡도가 크므로 항상 정답은 아닙니다.

30초 답변

고가용성은 multi-AZ 배치, health check, autoscaling, failover, stateless app 설계로 단일 장애 지점을 줄이는 것입니다. DR은 region 장애나 데이터 손상 같은 큰 사건에서 어떤 순서로 복구하고 어느 정도 데이터 손실을 허용할지 정하는 계획입니다.

90초 답변

RTO는 복구까지 걸리는 목표 시간, RPO는 잃어도 되는 데이터 양입니다. 모든 서비스를 active-active로 만들 필요는 없습니다. 비즈니스 중요도에 따라 backup/restore, pilot light, warm standby, active-active를 선택합니다. 중요한 것은 설계 문서보다 복구 훈련입니다. 복구 절차를 실행해보지 않으면 실제 장애 때 작동한다고 보기 어렵습니다.

꼬리질문

질문답변 방향
multi-AZ와 multi-region 차이는?AZ 장애 대응과 region 장애 대응은 비용과 복잡도가 다릅니다.
active-active는 항상 좋은가요?데이터 일관성, 운영 복잡도, 비용이 커져 요구 수준이 높을 때만 적절합니다.
DR 훈련은 무엇을 보나요?복구 시간, 데이터 검증, DNS 전환, secret/config, runbook 누락을 확인합니다.

오답/주의점

주의

“백업이 있습니다”는 DR 답변이 아닙니다. 복구 목표, 절차, 검증 결과가 있어야 합니다.

CLD-11

Q. 클라우드 비용이 갑자기 증가하면 어떤 순서로 확인하나요?

핵심 개념

클라우드 cost estimation은 compute뿐 아니라 NAT, egress, managed DB I/O, object request, log ingest/storage, autoscaling, retry 폭주까지 함께 계산하는 작업입니다.

개념 해부

클라우드 비용 증가는 트래픽 증가, 캐시 실패, NAT egress, 로그 폭증, 잘못된 autoscaling, orphan resource처럼 여러 계층에서 생깁니다. 비용은 사후 청구서가 아니라 metric으로 관측해야 합니다. 태그, 예산 알림, 서비스별 cost attribution이 없으면 원인과 소유자를 찾기 어렵습니다.

30초 답변

먼저 비용 리포트에서 서비스별 증가분을 봅니다. compute인지, DB I/O인지, NAT/egress인지, 로그 저장인지 분리하고 최근 배포, 트래픽 변화, retry 증가, autoscaling, batch 작업을 확인합니다. 비용 알림과 태그 기반 attribution이 없으면 원인 추적이 늦어집니다.

90초 답변

비용 최적화는 무조건 줄이는 것이 아니라 성능과 안정성을 해치지 않는 범위에서 낭비를 제거하는 것입니다. 사용하지 않는 리소스 정리, right sizing, reserved/savings plan, lifecycle policy, log retention, CDN cache hit rate, VPC endpoint로 NAT 우회, autoscaling 기준 조정 같은 방법을 씁니다.

핵심 개념

클라우드 장애 대응은 공급자 장애인지, 우리 설정/배포 문제인지, 네트워크/권한/용량 문제인지 계층별로 좁히는 과정입니다.

개념 해부

클라우드 장애 대응은 provider status page 확인만으로 끝나지 않습니다. region/AZ 영향인지, 우리 계정 quota·권한·route·security group·최근 배포 문제인지 나눠야 합니다. 우회, failover, degraded mode, 고객 공지 기준을 runbook으로 갖고 있어야 실제 대응이 됩니다.

30초 답변

먼저 사용자 영향 범위와 최근 변경을 확인합니다. 특정 region, AZ, 서비스, endpoint만 문제인지 보고, cloud provider status, LB 5xx, app log, DB metric, queue lag, IAM denied log를 함께 봅니다. 원인 분석보다 rollback, traffic 우회, scale out 같은 완화가 먼저일 수 있습니다.

90초 답변

관측성은 metric, log, trace, audit log를 연결해야 합니다. 예를 들어 502가 늘면 LB target health, app readiness, upstream timeout, deploy version을 보고, 403이 늘면 IAM policy 변경이나 token scope를 봅니다. 장애 대응 후에는 postmortem에서 감지 지연, blast radius, runbook 누락, alert 품질을 개선합니다.

CLD-15

클라우드 개념 인과관계

클라우드 면접에서 서비스 이름을 나열하면 얕게 들립니다. 트래픽이 어떤 경계를 통과하고, 어떤 책임이 cloud provider에서 application team으로 넘어오는지 설명해야 합니다.

CLOUD CAUSAL MODEL사용자 요청은 DNS, CDN 또는 Load Balancer, public subnet, security group, private subnet workload, managed DB를 거치며 각 계층에서 다른 실패와 권한 문제가 생깁니다. private subnet에 DB를 둬도 잘못된 route, 보안 그룹, bastion, secret 유출이 있으면 접근될 수 있습니다. IAM은 “누가 무엇을 할 수 있나”의 선언이고, KMS는 “어떤 key로 누가 복호화할 수 있나”의 경계입니다. DR은 백업 파일이 있다는 뜻이 아니라 실제 restore와 failback까지 검증했다는 뜻입니다.

IAMtrust policy와 permission policy를 혼동한다배포 role assume 실패 또는 과권한 부여who can assume, what can do, on which resource, under what condition을 분리한다

비용트래픽 경로와 저장 기간을 모르고 리소스를 켠다NAT, 로그, egress, snapshot 비용이 예측보다 커진다request path, data transfer, retention, scale metric으로 산정한다
면접 답변 전환

“DB는 private subnet에 둡니다”에서 멈추지 말고 “DB는 public route가 없는 subnet에 두고, app security group에서 DB port만 허용합니다. 다만 운영 접속, migration, backup restore, secret rotation 경로도 필요하므로 bastion/SSM, IAM, audit log, break-glass 절차를 함께 설계합니다”라고 말합니다.
