JAVA-01

변수·타입·var

Java는 정적 타입 언어 — 변수마다 타입이 박힙니다. 핵심 구분 하나: 기본형(primitive)은 값 자체를, 참조형(reference)은 객체 주소를 담습니다. JS의 let/const와 대비하면 감이 빨리 잡힙니다.

PRIMITIVE vs REFERENCE // 기본형 — 값을 직접 담음 (소문자로 시작)
int count = 10;
long big = 9_000_000_000L; // L 접미사, _ 가독성
double price = 19.99;
boolean active = true;
char grade = 'A';

// 참조형 — 객체 (대문자로 시작, null 가능)
String name = "haeun";
Integer boxed = 42; // int의 객체 wrapper, null 가능
int[] nums = {1, 2, 3};

int은 절대 null이 될 수 없지만 Integer(wrapper)는 가능합니다. DB 컬럼이 nullable이면 entity 필드는 Integer/Long을 씁니다 — 여기서 JS 출신이 자주 헷갈립니다.

1.1var 타입 추론 · final

Java 10+의 var는 TS의 타입 추론과 비슷하지만, 지역 변수에서만 쓸 수 있고 우변으로 타입이 즉시 정해집니다(런타임 동적 타입이 아님). final은 JS의 const에 해당 — 재할당 금지.

var & final var users = new ArrayList<String>(); // ArrayList<String>로 추론
var total = 0; // int로 추론

final int MAX = 100; // 재할당 불가 (= JS const)
final var config = loadConfig(); // final + var 조합 가능
// MAX = 200; → 컴파일 에러
var 주의

var x;처럼 초기값 없이 선언하거나, 필드(클래스 멤버)·메서드 파라미터·리턴 타입에는 var를 못 씁니다. 우변이 타입을 명확히 드러내지 않으면(예: var x = null;) 컴파일 에러입니다. JS의 let x; 감각으로 쓰면 막힙니다.

JAVA-02

컬렉션

JS의 배열·Map·Set이 Java에선 List·Map·Set 인터페이스로 나뉩니다. 인터페이스(List)로 선언하고 구현체(ArrayList)로 생성하는 게 관례입니다.

생성 — 수정 가능 vs 불변 // 수정 가능한 빈 리스트
List<String> names = new ArrayList<>();
names.add("kim");
names.add("lee");

// 불변 리스트 — add 하면 UnsupportedOperationException
List<String> fixed = List.of("a", "b", "c");

// Map / Set
Map<String, Integer> scores = new HashMap<>();
scores.put("haeun", 95);
Map<String, Integer> m = Map.of("x", 1, "y", 2); // 불변
Set<String> tags = new HashSet<>(List.of("order", "payment"));

2.1순회

ITERATION // List 순회 — for-each (JS for...of)
for (String name : names) {
  System.out.println(name);
}
names.forEach(System.out::println); // 메서드 참조

// Map 순회 — entrySet (JS for...of Object.entries)
for (Map.Entry<String, Integer> e : scores.entrySet()) {
  System.out.println(e.getKey() + " = " + e.getValue());
}
scores.forEach((key, value) -> System.out.println(key + value));
자주 쓰는 메서드의미 / JS 대응
list.size()크기 (JS arr.length)
list.get(0) / list.set(0, v)인덱스 접근/수정 (arr[0])
list.contains(v)포함 여부 (arr.includes)
list.isEmpty()비었는지 (arr.length === 0)
map.getOrDefault(k, def)없으면 기본값 (map.get(k) ?? def)
map.computeIfAbsent(k, x -> ...)없을 때만 계산해 넣고 반환
map.containsKey(k)키 존재 여부 (k in obj)
선택 기준

순서 유지가 필요하면 ArrayList, 중복 제거는 HashSet, 키-값은 HashMap. 입력 순서를 보존하는 Map이 필요하면 LinkedHashMap. List.of()/Map.of()로 만든 건 불변이라 나중에 수정할 거면 new ArrayList<>(List.of(...))로 감싸세요.

JAVA-03

스트림 API

JS의 .filter().map().reduce() 체이닝이 Java에선 Stream입니다. 차이 하나: 컬렉션에 .stream()으로 진입하고, .collect()로 빠져나와야 결과가 나옵니다(lazy).

filter · map · collect // JS: users.filter(u => u.age >= 18).map(u => u.name)
List<String> adults = users.stream()
  .filter(u -> u.age() >= 18)
  .map(User::name)
  .collect(Collectors.toList());

// Java 16+ : .toList() 로 더 짧게 (불변 리스트 반환)
List<String> names = users.stream().map(User::name).toList();

3.1reduce · groupingBy

집계 // 합계 — JS reduce((a, b) => a + b, 0)
int totalAge = users.stream()
  .mapToInt(User::age)
  .sum();

int product = nums.stream().reduce(1, (a, b) -> a * b);

// 그룹핑 — Map<도시, List<User>> (JS엔 기본 내장 없음)
Map<String, List<User>> byCity = users.stream()
  .collect(Collectors.groupingBy(User::city));

// 그룹별 개수 — Map<도시, Long>
Map<String, Long> countByCity = users.stream()
  .collect(Collectors.groupingBy(User::city, Collectors.counting()));
Stream 연산JS 배열 대응
.filter(p).filter()
.map(f).map()
.sorted() / .sorted(comparator).sort()
.distinct()[...new Set(arr)]
.limit(n) / .skip(n).slice()
.anyMatch(p) / .allMatch(p).some() / .every()
.findFirst().find() (Optional 반환)
.count().length (필터 후)
스트림은 일회용

한 번 소비한 Stream은 재사용 못 합니다 — 두 번 .collect()하면 IllegalStateException. 또 .stream()만 호출하고 terminal 연산(collect/forEach/count 등)을 안 붙이면 아무 일도 일어나지 않습니다(lazy). JS 배열 메서드처럼 매번 새 배열을 즉시 반환하는 게 아닙니다.

JAVA-04

Optional

"값이 있을 수도, 없을 수도" 를 타입으로 표현 — JS의 ?./??가 메서드로 들어온 형태입니다. null 체크를 강제해 NPE를 줄입니다.

생성 & 꺼내기 // 생성
Optional<User> found = repo.findById(1L); // 보통 라이브러리가 반환
Optional<String> opt = Optional.ofNullable(maybeNull);

// 기본값 — JS: maybeNull ?? "guest"
String name = opt.orElse("guest");
String lazy = opt.orElseGet(() -> computeDefault()); // 필요할 때만 계산

// 없으면 예외 던지기
User user = found.orElseThrow(() -> new NotFoundException("없음"));
map · ifPresent — chaining // JS: user?.profile?.email ?? "no-email"
String email = found
  .map(User::profile)
  .map(Profile::email)
  .orElse("no-email");

// 값이 있을 때만 실행 — JS: if (x != null) {...}
found.ifPresent(u -> System.out.println(u.name()));
found.ifPresentOrElse(
  u -> log.info("찾음: " + u.name()),
  () -> log.warn("못 찾음")
);
안티패턴 — Optional.get()

opt.get()은 값이 없으면 NoSuchElementException을 던집니다 — opt.isPresent()로 먼저 확인하지 않고 get()을 부르는 건 NPE를 다른 예외로 바꾼 것뿐입니다. orElse/orElseThrow/map으로 처리하세요. 그리고 Optional은 리턴 타입용입니다 — 필드나 메서드 파라미터에 쓰지 마세요.

JAVA-05

record와 불변 객체

데이터만 담는 클래스(DTO, value object)는 record 한 줄이면 끝 — 생성자·getter·equals·hashCode·toString이 자동 생성됩니다. TS의 readonly 객체 타입에 가깝습니다.

RECORD — 한 줄 불변 객체 // 이게 전부. 필드는 자동 final, getter는 name() 형태
public record User(String name, int age, String city) {}

// 사용
var u = new User("haeun", 31, "gwangmyeong");
u.name(); // "haeun" (getName 아님 — name())
u.age(); // 31
// u.age = 32; → 불가능 (불변)

// 검증·정규화는 compact 생성자에
public record Credit(String id, int amount) {
  public Credit {
    if (amount < 0) throw new IllegalArgumentException("음수 불가");
  }
}

5.1빌더 패턴

필드가 많거나 선택적 값이 섞이면 생성자 인자 순서가 헷갈립니다. 이때 빌더로 "이름 붙은 인자"처럼 씁니다(JS의 옵션 객체 {name, age} 감각).

BUILDER public class Order {
  private final String id;
  private final int qty;
  private final String memo;

  private Order(Builder b) {
    this.id = b.id; this.qty = b.qty; this.memo = b.memo;
  }

  public static class Builder {
    private String id;
    private int qty = 1;
    private String memo = "";
    public Builder id(String v){ this.id = v; return this; }
    public Builder qty(int v){ this.qty = v; return this; }
    public Builder memo(String v){ this.memo = v; return this; }
    public Order build(){ return new Order(this); }
  }
}

// 사용 — 순서 무관, 읽기 쉬움
var order = new Order.Builder().id("A1").qty(5).build();
실무 팁

빌더를 손으로 다 짜는 건 보일러플레이트입니다. 실무에선 Lombok의 @Builder/@Value 애너테이션으로 자동 생성하거나, 단순 DTO는 record로 끝내는 경우가 많습니다. record는 빌더가 필요 없을 만큼 필드가 적을 때, 빌더는 선택 필드가 많을 때.

JAVA-06

예외 처리

Java 예외는 두 갈래 — checked(컴파일러가 처리를 강제)와 unchecked(런타임, 강제 안 함). JS에는 이 구분이 없어서 처음엔 낯섭니다.

구분예시 / 처리
checkedIOException, SQLException — try-catch 또는 throws로 선언 강제
uncheckedNullPointerException, IllegalArgumentException — 처리 강제 없음(보통 버그)
try-with-resources — 자동 close // 괄호 안 자원은 끝나면 자동 close (JS엔 없는 패턴)
try (var reader = Files.newBufferedReader(path)) {
  return reader.readLine();
} catch (IOException e) {
  // expected failure — 회복 가능한 곳에서 처리
  log.error("파일 읽기 실패: " + path, e);
  throw new ConfigLoadException("설정 로드 실패", e);
}
커스텀 예외 클래스 // RuntimeException 상속 = unchecked (요즘 도메인 예외 관례)
public class CreditNotFoundException extends RuntimeException {
  public CreditNotFoundException(String id) {
    super("크레딧 없음: " + id); // 메시지를 부모로
  }
}

// 던지기
if (credit == null) throw new CreditNotFoundException(id);
예외를 삼키지 마라

빈 catch {}는 버그를 숨기는 최악의 패턴입니다. 잡았으면 ① 회복하거나 ② 로그를 남기고 의미 있는 예외로 다시 던지세요(throw new ...(msg, e) — 원인 e를 꼭 전달해 stack trace 보존). 회복 못 할 거면 애초에 잡지 말고 위로 흘려보내는 게 낫습니다.

JAVA-07

제네릭

TS의 <T>와 같은 개념 — 타입을 파라미터로 받아 재사용 가능한 코드를 만듭니다. 차이: Java 제네릭은 컴파일 후 타입 정보가 지워집니다(type erasure).

제네릭 메서드 & 클래스 // 제네릭 메서드 — <T>를 리턴 타입 앞에
public static <T> T firstOrNull(List<T> list) {
  return list.isEmpty() ? null : list.get(0);
}

// 제네릭 클래스 — 결과 래퍼 같은 것
public class ApiResult<T> {
  private final T data;
  private final String error;
  public ApiResult(T data, String error) {
    this.data = data; this.error = error;
  }
  public T data() { return data; }
}

ApiResult<User> res = new ApiResult<>(user, null);

7.1와일드카드 ? extends

"User든 그 하위 타입이든 다 받겠다"를 표현 — 읽기 전용으로 여러 하위 타입을 함께 다룰 때 씁니다.

WILDCARD // ? extends Number — Number의 하위 타입 리스트 모두 허용
public double sum(List<? extends Number> nums) {
  double total = 0;
  for (Number n : nums) total += n.doubleValue();
  return total;
}

// 둘 다 호출 가능
sum(List.of(1, 2, 3)); // List<Integer>
sum(List.of(1.5, 2.5)); // List<Double>
PECS 기억법

"Producer Extends, Consumer Super" — 값을 꺼내 읽기만 하면 ? extends T, 값을 넣기만 하면 ? super T. foundational 단계에선 ? extends(읽기) 하나만 확실히 알아도 충분합니다. erasure 때문에 런타임에 List<String>인지 List<Integer>인지 구분 못 한다는 것만 기억하세요.

JAVA-08

동시성 기초

JS는 단일 스레드 + 이벤트 루프지만, Java는 진짜 멀티스레드입니다. 스레드를 직접 만들기보다 ExecutorService(스레드 풀)와 CompletableFuture(JS Promise에 해당)를 씁니다.

ExecutorService — 스레드 풀 var pool = Executors.newFixedThreadPool(4);

// 작업 제출 → Future로 결과 받기
Future<Integer> future = pool.submit(() -> heavyCompute());
Integer result = future.get(); // 끝날 때까지 블로킹

pool.shutdown(); // 반드시 종료 (안 하면 JVM 안 끝남)
CompletableFuture — JS Promise 대응 // supplyAsync = Promise 시작, thenApply = .then()
CompletableFuture<String> cf = CompletableFuture
  .supplyAsync(() -> fetchUser(id)) // 비동기 실행
  .thenApply(user -> user.name()) // .then(u => u.name)
  .thenApply(String::toUpperCase)
  .exceptionally(e -> "기본값"); // .catch()

String name = cf.join(); // 결과 대기 (await 비슷)

// 둘을 합치기 — Promise.all 비슷
CompletableFuture<String> combined = cf1
  .thenCombine(cf2, (a, b) -> a + b);
JS와 다른 점

JS의 await는 이벤트 루프에 양보하지만, Java future.get()/join()은 실제로 스레드를 블로킹합니다. 공유 상태를 여러 스레드가 동시에 건드리면 race condition이 생기니 — foundational 단계에선 가능하면 불변 객체를 넘기고, 공유 가변 상태를 피하세요. 스레드 풀은 shutdown() 누락이 단골 버그입니다.

JAVA-09

날짜·시간

옛날 Date/Calendar는 잊으세요 — Java 8+의 java.time이 표준입니다. 모두 불변이라 JS의 가변 Date보다 안전합니다.

java.time — 핵심 타입 LocalDate today = LocalDate.now(); // 날짜만 2026-06-23
LocalDate d = LocalDate.of(2026, 6, 23);
LocalDateTime now = LocalDateTime.now(); // 날짜+시간
Instant ts = Instant.now(); // UTC 타임스탬프

// 연산 — 불변이라 새 객체 반환
LocalDate next = today.plusDays(7);
LocalDate prev = today.minusMonths(1);
boolean before = d.isBefore(today);
Duration · Period · 포맷 // 두 시각 사이 간격
Duration dur = Duration.between(start, end);
long mins = dur.toMinutes();

// 두 날짜 사이 일수
long days = ChronoUnit.DAYS.between(d1, d2);

// 포맷팅 — JS의 toLocaleDateString 대응
var fmt = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm");
String text = now.format(fmt);

// 파싱 — 문자열 → 날짜
LocalDate parsed = LocalDate.parse("2026-06-23");
타임존

저장·전송은 Instant(UTC)로, 사용자 표시할 때만 ZonedDateTime으로 타임존을 입히는 게 안전합니다 — instant.atZone(ZoneId.of("Asia/Seoul")). LocalDateTime은 타임존 정보가 없으니, "한국 시각 그대로" 같은 의미가 필요할 때만 쓰세요.

JAVA-10

자주 쓰는 이디엄

매일 마주치는 작은 패턴들. equals/hashCode, enum, switch 표현식, 문자열 처리 — 여기서 JS 습관과 가장 자주 충돌합니다.

10.1equals / hashCode

동등성 — == 아님 // JS: a === b 와 다름! ==는 참조(주소) 비교
String a = new String("hi");
String b = new String("hi");
a == b; // false — 다른 객체
a.equals(b); // true — 값 비교 (이걸 써야 함)

// null 안전 비교
Objects.equals(a, b); // a가 null이어도 NPE 안 남

10.2enum · switch 표현식

ENUM & SWITCH EXPRESSION (Java 14+) public enum CreditStatus { ISSUED, TRADED, RETIRED }

// switch가 값을 반환 — 화살표 문법, break 불필요
String label = switch (status) {
  case ISSUED -> "발행됨";
  case TRADED -> "거래됨";
  case RETIRED -> "소멸됨";
}; // 모든 case 처리 안 하면 컴파일 에러 (안전)

10.3문자열 처리

STRING String.join(", ", list); // JS arr.join(", ")
"a,b,c".split(","); // → String[]
" hi ".trim(); // "hi"
"Hello".contains("ell"); // true
String.format("%s = %d", k, v); // JS 템플릿 리터럴 대응

// 여러 줄 — text block (Java 15+)
String json = """
    { "name": "haeun" }

== 함정

JS 출신이 가장 많이 당하는 버그 — Java에서 객체 비교는 항상 .equals(), ==는 참조 비교입니다. 특히 String, Integer를 ==로 비교하면 값이 같아도 false가 나올 수 있습니다(literal 캐싱 때문에 가끔 우연히 맞기도 해서 더 위험). 직접 만든 클래스를 HashMap 키나 Set에 넣으려면 equals+hashCode를 같이 구현해야 합니다(record는 자동).
