컨테이너와 런타임

컨테이너는 애플리케이션과 실행 환경을 이미지로 묶어 배포하는 방식입니다. DevOps 관점에서는 이미지 재현성, 리소스 제한, 보안 경계, 운영 관측성이 핵심입니다.

CTR-01

이미지와 레이어

컨테이너 이미지는 여러 레이어로 구성된 불변 아티팩트입니다.

구성의미실무 포인트
Base imageOS/runtime 기반태그보다 digest 고정 검토
Dependency layer패키지 설치 결과lockfile 기반 설치
Application layer빌드 산출물불필요 파일 제외
Metadataentrypoint, env, labelscommit SHA, source repo 기록
실무 감각

이미지는 “서버에 배포할 파일 묶음”이 아니라 registry에 저장되는 배포 아티팩트입니다. 태그는 움직일 수 있으므로 운영 추적에는 image digest가 더 강합니다.

CTR-02

격리의 의미

격리역할주의
Process namespace프로세스 시야 분리VM 수준 격리는 아님
Network namespace컨테이너별 네트워크 스택port mapping과 service discovery 필요
Filesystem이미지 레이어와 writable layer컨테이너 삭제 시 로컬 변경 손실
cgroupsCPU/memory 제한제한 없으면 node 전체 영향
PID 1과 종료 처리

컨테이너 안의 메인 프로세스는 PID 1로 동작하며 신호 처리와 zombie process 회수 책임을 가집니다. 앱이 SIGTERM을 받고 요청을 마무리하도록 구현하고, orchestrator의 termination grace period 안에 정상 종료되도록 shutdown hook과 readiness 전환을 맞춥니다.

네트워크와 스토리지

영역핵심운영 질문

Volume컨테이너 밖 지속 저장소삭제와 재시작 후 데이터가 보존되는가

Logstdout/stderr 수집로그가 중앙 수집 시스템으로 가는가
CTR-04

리소스와 Health

CPU limitthrottle
CPU 제한은 성능 저하로 나타날 수 있습니다. latency 지표와 함께 봅니다.
Memory limitOOM
메모리 초과는 OOM kill로 이어집니다. JVM/Node heap 설정과 함께 조정합니다.
Readinesstraffic ready
트래픽을 받을 준비가 되었는지 판단합니다.
Livenessrestart
프로세스를 재시작해야 하는 죽은 상태인지 판단합니다.
