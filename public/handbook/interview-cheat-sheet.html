IV-CS-01

즉답 구조 템플릿

질문을 받으면 키워드부터 쏟지 말고 아래 구조를 머릿속에 세운다. 결론 먼저, 근거는 조건과 함께.

30초 답변 — 정의 · 차이 · 가벼운 확인 1. 결론 한 문장: "핵심은 X입니다."
2. 근거 한 문장: "왜냐하면 Y이기 때문입니다."
3. 경계 한 문장: "다만 Z 조건에서는 달라집니다."
예) "REST와 GraphQL의 핵심 차이는 데이터를 서버가 정하느냐 클라이언트가 정하느냐입니다. 그래서 화면마다 필요한 필드가 크게 다르면 GraphQL이 유리합니다. 다만 캐싱과 러닝커브 비용은 감수해야 합니다."
90초 답변 — 개념 · 기준 · trade-off · 예시 1. 개념 정의 + 왜 필요한가
2. 선택 기준 2~3개 (트래픽, 일관성, 운영 비용 등)
3. trade-off: 무엇을 얻고 무엇을 포기하나
4. 실무 예시 한 줄 (직접/개념 경계 명시)
예) "Redis 캐시는 반복 조회 비용을 줄이려고 씁니다. 판단 기준은 읽기 빈도, 데이터 신선도 요구, 캐시 무효화 비용입니다. 속도는 얻지만 정합성 경계를 직접 관리해야 하는 게 비용입니다. 제가 맡았던 조회 API에서는 TTL과 write-through로 처리했습니다."
STAR — 행동 질문 (협업 · 갈등 · 실패 · 리더십) Situation: 언제, 어떤 맥락에서 (한 문장)
Task: 내가 맡은 과제와 제약
Action: 내가 한 결정과 행동 (여기가 핵심, 가장 길게)
Result: 결과 지표 + 배운 점 / 바꾼 행동
예) "배포 후 장애가 반복됐고(S), 저는 원인 파악과 재발 방지를 맡았습니다(T). 로그를 추적해 특정 쿼리 타임아웃을 찾고, 배포 전 부하 체크를 파이프라인에 넣었습니다(A). 이후 같은 유형 장애가 사라졌고, 배포 전 검증 습관이 팀에 남았습니다(R)."
Trade-off 답변 프레임 "A는 X 때문에 좋고, B는 Y 때문에 좋습니다. 그래서 우리 상황에선 Z 기준이 더 중요하니 [선택]입니다."
예) "MSA는 팀 독립 배포 때문에 좋고, 모놀리식은 운영 단순함 때문에 좋습니다. 그래서 팀이 작고 도메인 경계가 아직 흐린 지금은 배포 속도보다 단순함이 중요해서 모놀리식으로 시작하겠습니다."

IV-CS-02

질문 유형별 접근

질문 유형을 먼저 분류하면 무엇을 첫 문장으로 말할지 정해진다.

질문 유형먼저 말할 것예시 시작 문장
경험형문제와 내 역할부터. 스택 나열 금지."제가 맡은 문제는 ~였고, 제 역할은 ~였습니다."
설계형요구사항·제약을 먼저 되묻고 좁힌다."트래픽 규모와 일관성 요구를 먼저 확인하고 싶습니다."
트레이드오프형양쪽 장점 → 우리 상황 기준 → 선택."A는 ~ 때문에, B는 ~ 때문에 좋습니다. 우리 상황에선 ~"
디버깅형증상 → 가설 → 좁히는 순서. 재현부터."먼저 재현하고, 범위를 로그로 좁히겠습니다."
모르는 질문아는 경계 명시 → 원리 추론 → 확인 방법."직접 경험은 없지만 원리로 추론하면 ~"
압박 질문방어 대신 전제·기준·증거로 재구성."그 지적의 전제를 먼저 정리하면 ~"
IV-CS-03

감점 신호 · 즉시 확인

전문성은 말의 양이 아니라 경계·기준·증거에서 드러난다. 아래는 면접관이 즉시 감점하는 신호.

감점 신호왜 위험한가즉시 확인 / 보정
결론 없이 장황판단 구조가 안 보이고 시간만 쓴다.첫 문장을 결론으로. "핵심은 X입니다"로 시작.
직접 경험과 이론을 같은 톤으로꼬리질문에서 신뢰가 무너진다."직접 해본 부분"과 "설계한다면"을 분리해 말함.
기술명만 나열키워드는 있고 적용 판단이 없다.정의 → 선택 기준 → trade-off로 재구성.
trade-off 없이 정답 단정"항상/무조건"은 조건을 무시한 답.규모·latency·consistency·비용 조건을 붙임.
모른다는 걸 숨김추측을 사실처럼 말하면 바로 들킨다.모르는 범위 인정 + 확인 방법 제시로 닫음.
성과가 팀 단위 표현뿐본인 기여와 의사결정이 흐려진다.내 역할·내가 바꾼 선택·결과 지표를 분리.
실패에 재발 방지가 없음학습 능력 대신 변명으로 들린다.관측·테스트·리뷰 기준을 어떻게 바꿨는지 말함.
IV-CS-04

모르는 질문 1카드

STUCK ANSWER TRIAGE 모르는 질문에서 조급하게 키워드를 쏟지 않는다. 아래 순서로 회복한다.
1. 질문 재정의 — "운영 기준인지, 개념 비교 기준인지 먼저 나누겠습니다."
2. 아는 경계 명시 — "직접 운영 경험은 없지만, 원리는 이렇게 이해합니다."
3. 원리로 추론 — "일관성·latency·복구 비용 중 무엇이 중요하냐에 따라 갈립니다."
4. 검증 방법 제안 — "그래서 작은 부하 테스트와 rollback 리허설로 확인하겠습니다."
5. 솔직히 모른다 + 학습 계획 — "이 부분은 확실치 않아 끝나고 문서로 확인하겠습니다."
순서질문말할 증거 / 예시
1이 질문이 진짜 묻는 건 뭔가?"운영 트래픽 기준인지 개념 비교인지 나누겠습니다."
2내가 확실히 아는 경계는?"직접 해본 건 API 계약과 장애 재현입니다."
3원리로 어디까지 추론되나?"CAP상 일관성을 택하면 가용성을 일부 포기합니다."
4어떻게 검증할 수 있나?"작은 부하 테스트로 병목 지점부터 확인합니다."
5모르는 건 어떻게 닫나?"확실치 않으니 확인 후 정확히 답 드리겠습니다."
중단 기준

추측을 사실처럼 이어 붙이는 순간 멈춘다. 모르면 "여기까지는 알고, 그다음은 확인이 필요합니다"로 경계를 긋는 게 지어내는 답보다 항상 낫다.
