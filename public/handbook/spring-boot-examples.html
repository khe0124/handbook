예시 사례

컨트롤러부터 예외 처리·검증·테스트까지 — 한 기능을 끝까지 짜는 레시피. 개념 설명보다 실제로 동작하는 코드를 먼저 보고, 계층(Controller / Service / Repository)이 각자 무슨 책임을 지는지 손에 익히는 게 목표입니다. 프론트엔드에서 넘어온 감각으로 읽히게 썼습니다.

NOTE : Spring Boot 3.x 기준 — 패키지는 jakarta.*(구 javax.* 아님). 깊은 개념은 (→ 백엔드 개요), (→ 아키텍처 패턴) 참조

SPRING-01

프로젝트 구조와 시작

Spring Boot 앱의 출발점은 main 메서드 하나입니다 — 거기서 컨테이너가 뜨고, 패키지를 스캔해 객체들을 알아서 엮습니다. 프론트의 index.tsx처럼 "여기가 시작"인 파일을 먼저 잡으세요.

Application.java — 진입점 @SpringBootApplication
public class AdminWorkflowApplication {
  public static void main(String[] args) {
    SpringApplication.run(AdminWorkflowApplication.class, args);
  }
}

@SpringBootApplication은 세 어노테이션을 묶은 것입니다 — @SpringBootConfiguration(설정 클래스), @EnableAutoConfiguration(의존성 보고 자동 설정), @ComponentScan(이 클래스가 있는 패키지 아래를 훑어 빈 등록). 그래서 진입점은 보통 최상위 패키지에 둡니다 — 아래에 있는 컴포넌트를 다 발견하도록.

1.1패키지 계층

한 도메인을 네 층으로 나눕니다. 요청이 위에서 아래로 흐릅니다.

controller→service→repository→domain (DB)
패키지책임
controllerHTTP 요청·응답만. URL 매핑, 요청 파싱, 상태코드 결정. 비즈니스 로직 없음
service비즈니스 로직과 트랜잭션 경계. "무엇을 한다"가 여기 모임
repositoryDB 접근. 저장·조회 쿼리 (→ JPA 예시)
domain엔티티·도메인 모델. 데이터의 모양과 규칙
의존성 — starter

Spring Boot는 개별 라이브러리 대신 starter 묶음으로 의존성을 받습니다. spring-boot-starter-web(REST·내장 Tomcat), spring-boot-starter-data-jpa(JPA·Hibernate), spring-boot-starter-validation(Bean Validation), spring-boot-starter-test(JUnit·Mockito·MockMvc). starter 하나가 호환되는 버전들을 한꺼번에 가져와서, 프론트의 package.json보다 버전 충돌 걱정이 적습니다.

SPRING-02

REST 컨트롤러

컨트롤러는 HTTP의 경계입니다 — URL과 메서드를 받아 service에 넘기고, 결과를 상태코드와 함께 돌려줍니다. Express의 라우터 핸들러와 같은 자리지만, 어노테이션으로 선언합니다.

UserController.java @RestController
@RequestMapping("/api/users")
public class UserController {

  private final UserService userService;

  @GetMapping("/{id}")
  public ResponseEntity<UserResponse> getUser(@PathVariable Long id) {
    UserResponse user = userService.findById(id);
    return ResponseEntity.ok(user);  // 200
  }

  @GetMapping
  public List<UserResponse> list(@RequestParam(defaultValue = "0") int page) {
    return userService.findAll(page);
  }

  @PostMapping
  public ResponseEntity<UserResponse> create(@RequestBody CreateUserRequest req) {
    UserResponse created = userService.create(req);
    return ResponseEntity.status(HttpStatus.CREATED).body(created);  // 201
  }
}
어노테이션역할
@RestController@Controller + @ResponseBody — 반환값을 JSON 본문으로 직렬화
@RequestMapping("/api/users")클래스 공통 prefix. 메서드 매핑이 이어붙음
@GetMapping / @PostMapping / @PutMapping / @DeleteMappingHTTP 메서드별 매핑. 조회·생성·교체·삭제
@PathVariableURL 경로의 값 (/users/{id}의 id)
@RequestParam쿼리 파라미터 (?page=0). defaultValue·required 지정 가능
@RequestBody요청 JSON 본문을 객체로 역직렬화
ResponseEntity vs 객체 직접 반환

상태코드·헤더를 직접 정해야 하면 ResponseEntity<User>를, 항상 200이면 객체(UserResponse · List<UserResponse>)를 그냥 반환하면 됩니다. 생성은 201 Created, 본문 없는 삭제는 ResponseEntity.noContent().build()(204)가 관례입니다. 상태코드는 의미를 담으니 다 200으로 뭉뚱그리지 마세요(→ API 설계 원칙).

SPRING-03

DI와 빈

JS에서는 new UserService()로 직접 객체를 만들지만, Spring에서는 컨테이너가 객체를 만들어 들고 있다가 필요한 곳에 넣어줍니다. 이게 의존성 주입(DI)입니다. 당신은 "이게 필요해"라고 선언만 합니다.

생성자 주입 — 권장 방식 @Service
public class UserService {

  private final UserRepository userRepository;

  // 생성자가 하나면 @Autowired 생략 가능 — 컨테이너가 알아서 주입
  public UserService(UserRepository userRepository) {
    this.userRepository = userRepository;
  }
}

생성자 주입이 권장입니다 — 필드를 final로 둘 수 있어 불변이고, 객체가 만들어질 때 의존성이 반드시 채워지며(누락 시 컴파일·기동 단계에서 발각), 테스트에서 new UserService(mockRepo)로 쉽게 갈아끼웁니다. 필드 주입(@Autowired 필드)은 이 장점들을 잃으니 피하세요.

스테레오타입붙이는 곳
@Component일반 빈. 아래 셋의 부모 — 특정 역할이 없을 때
@Service비즈니스 로직 계층. 의미상 service임을 표시
@Repository데이터 접근 계층. DB 예외를 Spring 예외로 변환해주는 효과도
@RestController / @Controller웹 계층 (위 셋과 같은 @Component 계열)

3.1직접 만드는 빈 — @Configuration + @Bean

내가 코드를 소유하지 않은 외부 라이브러리 객체(예: RestClient)는 어노테이션을 못 붙입니다. 이럴 때 메서드로 빈을 직접 만들어 등록합니다.

AppConfig.java @Configuration
public class AppConfig {

  @Bean
  public RestClient paymentApiClient() {
    return RestClient.builder()
      .baseUrl("https://api.example.com")
      .build();
  }
}
사고 전환

프론트에서는 모듈을 import하면 그 인스턴스를 내가 책임지고 만들고 관리합니다. Spring에서는 객체의 생애주기를 컨테이너에 위임합니다 — 기본적으로 빈은 싱글톤 하나로 관리되고, 어디서 주입받든 같은 인스턴스입니다. "누가 이 객체를 만들고 들고 있나?"의 답이 항상 컨테이너라는 점이 핵심 차이입니다(→ 아키텍처 패턴).

SPRING-04

설정

환경마다 다른 값(DB URL, 외부 키, 포트)을 코드에서 분리합니다. .env의 Spring 버전이 application.yml이고, 환경 분리는 profile로 합니다.

application.yml spring:
  application:
    name: payment-service
  profiles:
    active: ${SPRING_PROFILES_ACTIVE:local}  # 환경변수 없으면 local

payment:
  trading:
    max-credit-per-tx: 10000
    api-key: ${PARTNER_API_KEY}  # secret은 환경변수로

4.1값 읽기 — @Value vs @ConfigurationProperties

방식언제
@Value("${payment.service.api-key}")값 한두 개를 필드에 바로 꽂을 때. 간단하지만 흩어짐
@ConfigurationProperties("payment.service")관련 값 묶음을 타입 안전한 객체로. 묶음이 커지면 이쪽
@ConfigurationProperties — 묶음을 객체로 @ConfigurationProperties(prefix = "payment.service")
public record TradingProperties(
  int maxCreditPerTx,
  String apiKey
) {}

// 진입점이나 @Configuration에 @EnableConfigurationProperties(TradingProperties.class)
profile별 파일 분리

application-{profile}.yml 규칙으로 환경을 나눕니다 — application-local.yml, application-prod.yml. 공통값은 application.yml에 두고, 활성 profile 파일이 같은 키를 덮어씁니다. 활성화는 spring.profiles.active 또는 실행 시 --spring.profiles.active=prod / 환경변수 SPRING_PROFILES_ACTIVE.

SECRET

API 키·DB 비밀번호를 application.yml에 직접 박지 마세요. ${PARTNER_API_KEY}처럼 환경변수나 secret manager를 참조하고, 실제 값은 저장소 밖에 둡니다. yml에 평문으로 커밋된 secret은 git 히스토리에 영원히 남습니다.

SPRING-05

서비스·트랜잭션

service 계층이 비즈니스 로직의 집입니다. 그리고 여기서 트랜잭션 경계를 긋습니다 — "이 메서드 안의 DB 작업은 전부 성공하거나, 전부 롤백된다".

UserService.java — 트랜잭션 경계 @Service
public class UserService {

  private final UserRepository userRepository;
  // 생성자 생략

  @Transactional(readOnly = true)
  public UserResponse findById(Long id) {
    User user = userRepository.findById(id)
      .orElseThrow(() -> new UserNotFoundException(id));
    return UserResponse.from(user);
  }

  @Transactional  // 쓰기 — 중간에 실패하면 전부 롤백
  public UserResponse create(CreateUserRequest req) {
    if (userRepository.existsByEmail(req.email())) {
      throw new DuplicateEmailException(req.email());
    }
    User saved = userRepository.save(req.toEntity());
    return UserResponse.from(saved);
  }
}
요소의미
@Transactional메서드 시작~끝을 한 트랜잭션으로. 정상 종료 시 commit, 예외 전파 시 rollback
readOnly = true조회 전용 — 쓰기 안 함을 명시. 성능 힌트(flush 생략 등)이자 의도 표현
throw로 롤백기본은 unchecked 예외(RuntimeException)에서 롤백. service에서 던진 도메인 예외가 롤백을 유발
트랜잭션은 service에, controller에 두지 말 것

트랜잭션 경계는 "하나의 비즈니스 작업" 단위여야 합니다 — 그래서 service 메서드에 둡니다. controller에 @Transactional을 붙이면 HTTP·직렬화까지 트랜잭션 안에 들어와 경계가 흐려집니다. 또 같은 클래스 내부 메서드 호출은 @Transactional이 안 먹습니다 — Spring AOP 프록시가 외부 진입점에서만 가로채기 때문입니다(→ 아키텍처 패턴).

SPRING-06

예외 처리

각 컨트롤러에서 try-catch를 반복하지 마세요. service는 도메인 예외를 던지고, 한 곳에서 잡아 일관된 에러 응답으로 변환합니다. 프론트의 전역 에러 바운더리에 가깝습니다.

ErrorResponse.java — 표준 에러 DTO public record ErrorResponse(
  String code,
  String message,
  Instant timestamp
) {
  public static ErrorResponse of(String code, String message) {
    return new ErrorResponse(code, message, Instant.now());
  }
}
GlobalExceptionHandler.java @RestControllerAdvice
public class GlobalExceptionHandler {

  @ExceptionHandler(UserNotFoundException.class)
  public ResponseEntity<ErrorResponse> handleNotFound(UserNotFoundException e) {
    return ResponseEntity.status(HttpStatus.NOT_FOUND)  // 404
      .body(ErrorResponse.of("USER_NOT_FOUND", e.getMessage()));
  }

  @ExceptionHandler(DuplicateEmailException.class)
  public ResponseEntity<ErrorResponse> handleDuplicate(DuplicateEmailException e) {
    return ResponseEntity.status(HttpStatus.CONFLICT)  // 409
      .body(ErrorResponse.of("DUPLICATE_EMAIL", e.getMessage()));
  }
}
요소역할
@RestControllerAdvice모든 컨트롤러에 걸치는 전역 예외 처리. 응답은 JSON
@ExceptionHandler(X.class)특정 예외 타입을 잡아 응답으로 변환하는 메서드
4xx (클라이언트 잘못)잘못된 입력·없는 리소스·중복 — 도메인 예외를 매핑
5xx (서버 잘못)예기치 못한 버그·외부 장애. 마지막에 Exception.class로 받아 500
에러를 삼키지 말 것

5xx로 내려갈 예외는 클라이언트에 내부 stack trace·SQL을 노출하지 말고, 로그에는 전부 남기세요 — 사용자에게 보여줄 메시지와 로그 메시지를 분리합니다. expected failure(중복 이메일 같은 도메인 규칙)는 4xx로 명확히, bug(NPE 등)는 5xx로 잡되 알람이 울리게. 빈 catch로 예외를 묻으면 디버깅이 불가능해집니다.

SPRING-07

요청 검증

잘못된 입력은 service에 닿기 전에 막습니다. DTO에 제약을 어노테이션으로 선언하고, 컨트롤러에서 @Valid로 발동시키면 Spring이 자동으로 검사합니다.

CreateUserRequest.java — Bean Validation public record CreateUserRequest(

  @NotBlank  // null·빈문자·공백 모두 거부
  @Size(max = 50)
  String name,

  @NotBlank
  @Email  // 이메일 형식
  String email,

  @NotNull
  @Positive  // 0보다 큰 값
  Integer creditBalance
) {}
컨트롤러에서 @Valid로 발동 @PostMapping
public ResponseEntity<UserResponse> create(
  @Valid @RequestBody CreateUserRequest req  // 검증 실패 시 자동으로 예외
) {
  return ResponseEntity.status(HttpStatus.CREATED).body(userService.create(req));
}
제약의미
@NotNullnull만 거부 (빈 문자열·0은 통과)
@NotBlank문자열 전용 — null·""·공백만 모두 거부
@Size(min=, max=)문자열 길이·컬렉션 크기 범위
@Email이메일 형식
@NotNull + @Positive / @Min숫자 범위. @Positive는 0 초과
검증 실패도 GlobalExceptionHandler에서 처리

@Valid 검증이 실패하면 MethodArgumentNotValidException이 던져집니다. 이것도 ch6의 @RestControllerAdvice에서 한 번에 잡아 400으로 변환하면, 검증 에러까지 같은 ErrorResponse 포맷으로 일관됩니다.

검증 에러 → 400으로 변환 @ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException e) {
  String message = e.getBindingResult().getFieldErrors().stream()
    .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
    .collect(Collectors.joining(", "));
  return ResponseEntity.badRequest()  // 400
    .body(ErrorResponse.of("VALIDATION_FAILED", message));
}
SPRING-08

데이터 접근 연동

service는 DB를 직접 만지지 않고 repository에 위임합니다. repository는 인터페이스만 선언하면 Spring Data JPA가 구현체를 자동으로 만들어줍니다 — 여기서는 연동 흐름만 보고, 쿼리 작성은 (→ JPA 예시).

UserRepository.java — 인터페이스만 @Repository
public interface UserRepository extends JpaRepository<User, Long> {

  // 메서드 이름만으로 쿼리 생성 — 구현 불필요
  boolean existsByEmail(String email);
  Optional<User> findByEmail(String email);
}

JpaRepository<User, Long>를 확장하면 save·findById·findAll·delete 같은 기본 CRUD가 공짜로 생깁니다(엔티티 User, PK 타입 Long). service에서는 이걸 주입받아 그냥 호출하면 됩니다.

UserController→UserService→UserRepository→DB
service에서 repository 사용 (ch5 재방문) @Transactional(readOnly = true)
public List<UserResponse> findAll(int page) {
  return userRepository.findAll(PageRequest.of(page, 20))
    .map(UserResponse::from)
    .getContent();
}
엔티티를 컨트롤러로 흘리지 말 것

DB 엔티티(User)를 그대로 응답으로 반환하지 말고 UserResponse 같은 DTO로 변환해 내보내세요 — 엔티티를 노출하면 내부 필드·연관관계가 새어 나가고, JSON 직렬화 중 lazy loading 문제가 생깁니다. 쿼리 메서드 작명 규칙·연관관계·N+1 같은 본격적인 내용은 (→ JPA 예시)에서 다룹니다.

SPRING-09

테스트

전부 띄울지(@SpringBootTest), 웹 계층만 잘라 빠르게 볼지(@WebMvcTest)를 먼저 고릅니다. 컨트롤러 검증·매핑 테스트는 슬라이스가 빠르고 충분합니다.

UserControllerTest.java — @WebMvcTest 슬라이스 @WebMvcTest(UserController.class)  // 웹 계층만 로드 — 빠름
class UserControllerTest {

  @Autowired MockMvc mockMvc;
  @MockBean UserService userService;  // service는 가짜로 대체

  @Test
  void 존재하는_id로_조회하면_200과_유저를_반환한다() throws Exception {
    given(userService.findById(1L))
      .willReturn(new UserResponse(1L, "haeun", "a@b.com"));

    mockMvc.perform(get("/api/users/1"))
      .andExpect(status().isOk())
      .andExpect(jsonPath("$.name").value("haeun"));
  }
}
방식로드 범위 / 언제

SPRING-10

운영 — Actuator

배포한 앱이 살아있는지, 어떤 상태인지를 들여다보는 창입니다. spring-boot-starter-actuator를 추가하면 운영용 엔드포인트가 열립니다 — 로드밸런서·쿠버네티스 헬스체크의 표적이 됩니다.

application.yml — 노출 엔드포인트 선택 management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics  # 필요한 것만 명시
  endpoint:
    health:
      show-details: when-authorized
엔드포인트역할
GET /actuator/health앱·DB·디스크 상태. {"status":"UP"} — 헬스체크의 핵심
GET /actuator/info빌드·버전 등 임의 메타 정보
GET /actuator/metricsJVM 메모리·요청 수·응답 시간 등 지표. 모니터링 연동
엔드포인트 노출 범위 주의

Actuator 엔드포인트는 운영 정보를 담으니 전부 공개하지 마세요. 기본은 health·info만 노출되고, include: "*"로 전체를 열면 env·heapdump 같은 민감 정보까지 새어나갑니다. 필요한 것만 명시하고, 외부 노출 시에는 인증으로 보호하세요(→ 백엔드 개요의 관측가능성).

SPRING-12

프록시·트랜잭션·테스트 심화

Spring을 깊게 이해한다는 것은 어노테이션을 많이 아는 것이 아니라, 컨테이너가 만든 프록시 객체를 통해 부가기능이 적용된다는 사실을 이해하는 것입니다. @Transactional, 보안, 캐시, 로깅 AOP가 모두 이 사고 위에 있습니다.

개념정확한 의미면접 단골 함정
BeanSpring container가 생성·관리하는 객체new로 만든 객체에는 DI/AOP가 적용되지 않음
Proxy실제 객체 앞에서 트랜잭션 같은 부가기능을 감싸는 대리 객체같은 클래스 내부 메서드 호출(self-invocation)은 프록시를 거치지 않음
@Transactional프록시가 메서드 전후로 transaction begin/commit/rollback을 처리private 메서드, 내부 호출, checked exception rollback 정책 오해
readOnly읽기 트랜잭션 힌트, flush 최적화와 의도 표현DB가 모든 쓰기를 자동 차단한다고 착각
@ControllerAdvice예외를 HTTP error contract로 번역하는 경계모든 예외를 500으로 뭉개거나 문자열 message만 반환
Slice Test웹, JPA 등 특정 계층만 로드하는 빠른 테스트모든 테스트를 @SpringBootTest로 돌려 느리고 깨지기 쉬움
TRANSACTION PROXY FLOW client → proxy.findUser()
proxy: begin transaction
target service method 실행
proxy: 정상 return이면 commit, runtime exception이면 rollback
this.otherTransactionalMethod() 는 proxy를 거치지 않으므로 새 트랜잭션 속성이 적용되지 않는다.
rollback 정책

Spring의 기본 @Transactional은 unchecked exception(RuntimeException, Error)에서 rollback하고 checked exception은 기본적으로 rollback하지 않습니다. 비즈니스 예외를 checked로 만들었다면 rollbackFor를 명시해야 할 수 있습니다. 면접에서 “예외가 나면 롤백됩니다”로 끝내면 부족하고, 어떤 예외인지와 트랜잭션 경계가 어디인지까지 말해야 합니다.

SPRING DEPTH ANSWERSpring의 @Transactional은 메서드에 마법이 걸리는 게 아니라 Spring bean을 감싼 proxy가 호출 전후로 트랜잭션을 시작하고 종료하는 구조입니다. 그래서 같은 클래스 내부 호출은 proxy를 거치지 않아 트랜잭션 속성이 적용되지 않을 수 있고, 기본 rollback은 unchecked exception 기준입니다. 테스트도 이 구조를 기준으로 domain은 순수 단위 테스트, web은 @WebMvcTest, repository는 @DataJpaTest, wiring은 소수의 @SpringBootTest로 나눕니다.
