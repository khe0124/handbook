DEEP요청 처리 경로와 트랜잭션 경계

백엔드 개념을 외운 답변처럼 보이게 만드는 가장 큰 이유는 요청 하나가 서버 안에서 어떻게 흘러가는지 설명하지 못하기 때문입니다. 좋은 백엔드 답변은 HTTP 요청이 컨트롤러, 애플리케이션 서비스, 도메인, 저장소, 외부 시스템을 통과하며 어디서 실패할 수 있는지까지 따라갑니다.

컨트롤러는 HTTP를 애플리케이션 명령으로 번역하는 얇은 경계입니다. 여기서 입력 형식, 상태코드, 응답 포맷, 요청 주체 전달을 다룹니다. 애플리케이션 서비스는 유스케이스를 조율하고 트랜잭션 경계를 잡습니다. 도메인은 “주문은 취소 가능한 상태에서만 취소된다”, “재고는 음수가 될 수 없다” 같은 불변식을 보호합니다. 저장소는 영속성 세부사항이고, 외부 API 클라이언트는 실패와 지연이 기본값인 의존성입니다.

BE-02

HTTP·REST·API 계약

API는 코드가 아니라 계약입니다. 한 번 공개하면 수많은 클라이언트가 의존하고, 함부로 못 바꿉니다. 설계의 무게가 내부 함수와 다른 이유입니다.

2.1메서드 의미와 멱등성

HTTP 메서드는 단순 동사가 아니라 안전성(safe)과 멱등성(idempotent)의 약속입니다. GET은 안전(상태 변경 없음)하고 멱등하며, PUT·DELETE는 멱등(여러 번 호출해도 결과 동일), POST는 비멱등(호출마다 새 자원). 이 약속이 중요한 이유는 재시도 때문입니다 — 네트워크가 끊겨 응답을 못 받았을 때 멱등한 요청만 안전하게 다시 보낼 수 있습니다.

IDEMPOTENCY KEY

결제·주문 같은 비멱등 POST를 재시도 안전하게 만들려면 멱등성 키를 씁니다 — 클라이언트가 요청마다 고유 키를 헤더로 보내고, 서버는 같은 키의 두 번째 요청을 "이미 처리됨"으로 같은 응답을 돌려줍니다. "결제 버튼 두 번 눌러서 두 번 결제됨"을 막는 표준 장치입니다.

2.2상태코드 — 4xx와 5xx의 경계

상태코드는 의미군입니다: 2xx 성공, 3xx 리다이렉트, 4xx 클라이언트 잘못(요청이 틀림 — 재시도해도 같은 결과), 5xx 서버 잘못(서버가 못 함 — 재시도하면 될 수도). 이 경계가 운영을 가릅니다 — 입력 검증 실패를 500으로 던지면 알람이 울리고 온콜이 깨지지만, 실제로는 400이어야 합니다. "누구의 잘못인가"를 상태코드로 정직하게 말하는 것이 API의 기본 예의입니다.

3.1ACID — 트랜잭션의 4가지 약속

  • 원자성(Atomicity) — 전부 되거나 전부 안 된다. 중간 실패 시 롤백.
  • 일관성(Consistency) — 제약(외래키·유니크·체크)을 깨는 상태로 못 간다.
  • 격리성(Isolation) — 동시 트랜잭션이 서로의 중간 상태를 안 본다 (수준은 조절 가능 → 3.3).
  • 지속성(Durability) — 커밋되면 장애가 나도 남는다 (WAL·로그).

4.1영속성 컨텍스트 — JPA의 심장

JPA의 핵심은 영속성 컨텍스트(persistence context)라는 1차 캐시입니다. 트랜잭션 동안 조회한 엔티티를 들고 있다가, 변경을 감지(dirty checking)해 커밋 시점에 자동으로 UPDATE를 날립니다 — 그래서 save()를 명시 안 해도 필드만 바꾸면 반영됩니다. 이게 강력하면서 동시에 혼란의 원천입니다: "내가 안 부른 쿼리가 왜 나가지?"의 답이 대개 dirty checking과 flush 타이밍입니다.

4.2지연 로딩과 트랜잭션 경계

  • 기본은 lazy — 필요할 때만 로딩. @ManyToOne의 기본이 eager인 함정 주의.
  • 화면에서 함께 쓸 연관은 fetch join으로 한 번에 (N+1 회피).
  • 트랜잭션 경계를 서비스 계층에 명확히 — 컨트롤러·뷰에 영속성 컨텍스트를 끌고 가지 않기(OSIV 논쟁의 핵심).

5.1경쟁 상태와 갱신 손실

두 트랜잭션이 같은 값을 읽고(read), 각자 계산하고, 각자 쓰면(write) — 나중 쓰기가 먼저 쓰기를 덮습니다(lost update). 애플리케이션 코드의 if (재고 > 0) 재고--는 동시성 하에서 거의 항상 틀립니다. 해결은 두 갈래입니다.

CORE INSIGHT

동시성 버그의 공통 뿌리는 "읽고-판단하고-쓰는" 사이의 틈입니다. 그 틈에 남이 끼어듭니다. 해법은 항상 둘 중 하나 — 틈을 잠그거나(비관적), 틈에서 바뀌었는지 나중에 확인하거나(낙관적). DB의 원자적 연산(UPDATE ... SET n=n-1 WHERE n>0) 한 줄로 틈 자체를 없애는 것이 가장 단순한 정답일 때도 많습니다.

낙관적 락은 “대부분 충돌하지 않는다”는 가정에서 좋습니다. 충돌하면 실패시키고 다시 읽어 재시도합니다. 비관적 락은 “충돌이 자주 나고, 기다리는 것이 실패보다 싸다”는 상황에서 씁니다. 하지만 락을 오래 잡으면 대기열이 생기고 데드락 가능성이 커집니다. 고급 답변은 “어떤 락을 쓰겠다”가 아니라 “충돌 빈도, 대기 비용, 사용자 경험, 재시도 가능성으로 고르겠다”입니다.

NET외부 호출과 런타임 네트워크

백엔드는 네트워크 위에 삽니다. 그리고 네트워크의 8가지 오류(fallacy) 중 첫 번째가 "네트워크는 신뢰할 수 있다"입니다 — 네트워크는 느리고, 실패하고, 가끔 거짓말합니다.

8.1TCP·DNS·TLS — 한 번의 요청에 숨은 비용

  • TCP는 연결을 3-way handshake로 맺습니다 — 연결 수립 자체가 왕복 비용입니다. 그래서 매 요청마다 새 연결을 맺지 않고 커넥션 풀로 재사용합니다(DB·HTTP 클라이언트 모두).
  • DNS는 도메인 → IP 변환이며 TTL 동안 캐시됩니다. 배포·장애조치 시 DNS TTL이 길면 트래픽 전환이 늦어집니다.
  • TLS는 추가 handshake로 암호화 채널을 맺고 인증서로 신원을 증명합니다 — HTTPS의 토대. 인증서 만료는 흔하고 치명적인 운영 사고입니다.

Circuit Breaker

다른 서비스를 호출할 때 반드시 갖춰야 할 셋입니다. 타임아웃 — 무한 대기는 스레드·커넥션을 고갈시켜 한 서비스의 느림이 전체 장애로 번지게 합니다(연결·읽기 타임아웃 둘 다). 재시도 — 일시적 실패엔 효과적이지만, 즉시·고정 간격 재시도는 죽어가는 서버를 더 때립니다 → 지수 백오프 + 지터, 그리고 멱등한 요청만. 서킷브레이커 — 연속 실패가 임계를 넘으면 회로를 열어 한동안 호출 자체를 차단하고 빠르게 실패시켜, 회복할 시간을 줍니다.

FAILURE MODE — 캐스케이딩 장애

타임아웃 없는 호출 하나가 전체를 무너뜨리는 전형: A가 느려진 B를 무한 대기 → A의 스레드풀 고갈 → A도 응답 불능 → A를 호출하던 C도... 한 군데의 느림이 도미노로 번집니다. 타임아웃과 격벽(bulkhead)은 선택이 아니라 분산 시스템의 기본 위생입니다.

백엔드 설계의 최종 시험은 "정답을 아는가"가 아니라 "트레이드오프를 설명하는가"입니다. 시스템 설계엔 정답이 없고, 제약에 맞는 선택과 그 비용의 명시가 있을 뿐입니다.

12.1먼저 묻는 질문 — 비기능 요구

기능("무엇을")보다 비기능 요구("얼마나")가 설계를 결정합니다: 예상 트래픽(읽기:쓰기 비율, peak), 지연 목표(p99), 가용성 목표, 데이터 규모와 증가율, 일관성 요구(강한가 최종이어도 되나). 이걸 안 묻고 그린 설계는 추측입니다 — 당신의 작업 원칙 "요구사항이 모호하면 채우지 말고 질문하라"가 그대로 적용됩니다.

BACKEND DESIGN QUESTION FLOW 요구사항: 사용자가 실제로 끝내야 하는 행동과 실패 시 손실은 무엇인가
트래픽: read/write 비율, peak QPS, p95·p99 latency 목표, 데이터 증가율은 어느 정도인가
일관성: 즉시 맞아야 하는 값과 나중에 맞아도 되는 값은 무엇인가
데이터 모델: aggregate root, 고유 제약, 상태 전이, 삭제·보존 정책은 어디에 있는가
API 계약: request/response, error code, idempotency key, pagination, versioning을 어떻게 고정할 것인가
트랜잭션 경계: 한 번에 커밋해야 하는 변경과 이벤트로 분리할 변경은 무엇인가
장애 모드: 외부 API, DB, cache, queue가 느리거나 죽었을 때 사용자는 무엇을 보게 되는가
확장 경로: 지금은 단순하게 두되, 어느 지표가 넘으면 cache, queue, read model, sharding을 검토할 것인가
관측 지표: 성공률, 에러율, p99, lock wait, queue lag, retry count를 어디서 볼 것인가

12.2확장과 일관성의 거래

  • 수직 확장(서버 키우기)은 단순하지만 한계·단일 장애점. 수평 확장(서버 늘리기)은 무한하지만 무상태(stateless) 설계와 데이터 분산을 요구합니다.
  • CAP — 네트워크 분단(P)은 피할 수 없으니, 분단 시 일관성(C)과 가용성(A) 중 무엇을 포기할지의 선택입니다. 은행 잔액은 C, 좋아요 수는 A.
  • 일관성 모델 — 강한 일관성(항상 최신)은 비싸고, 최종 일관성(잠깐 어긋나도 결국 수렴)은 싸고 빠릅니다. 도메인이 결정합니다.
설계 결정단순하게 시작분리하는 신호주의할 비용
모듈 vs 서비스한 코드베이스 안의 명확한 module boundary배포 주기, 데이터 소유권, 장애 격리가 실제로 갈라질 때분산 트랜잭션, 네트워크 실패, 관측 복잡도
동기 API vs 비동기 메시지사용자 응답에 필요한 작업만 동기 처리느린 외부 호출, 후처리, fan-out, 재시도가 필요할 때중복 메시지, 순서 보장, 보상 처리
RDB vs 캐시·검색·큐정합성의 기준은 RDB에 둔다읽기 병목, 검색 요구, 비동기 처리량이 명확해질 때무효화, 재색인, lag, 운영 장애 지점 증가
강한 일관성 vs 최종 일관성돈, 재고, 권한은 강한 일관성알림, 추천, 집계처럼 잠깐 틀려도 되는 화면사용자에게 보이는 상태 불일치와 보정 로직
단일 트랜잭션 vs outbox/saga한 DB 안의 핵심 상태는 단일 트랜잭션외부 시스템 발행, 여러 서비스 변경, 장기 작업중간 상태, 재시도, 보상 이벤트, 모니터링
DESIGN DECISION MATRIX 결정은 기술 선호가 아니라 제약의 결과여야 합니다. 설계 리뷰에서는 "왜 이 기술인가"보다 "어떤 제약 때문에 이 선택이 최소 비용인가", "어떤 지표가 나오면 다음 단계로 바꿀 것인가", "되돌리거나 확장할 수 있는가"를 먼저 설명하세요.

12.3장애를 전제로 설계

"무엇이 깨질 수 있는가"를 먼저 그리는 것이 실무 설계입니다 — 단일 장애점(SPOF) 제거, 우아한 성능저하(graceful degradation: 추천이 죽어도 상품은 보이게), 백프레셔(과부하 시 받지 않고 거절), 격벽(한 기능의 자원 고갈이 다른 기능을 안 죽이게).

측정 없는 최적화 금지

"이게 더 빠를 것 같아서"는 근거가 아닙니다 — 병목은 측정한 뒤 잡으세요(프로파일링·메트릭). 명확성과 성능이 충돌하면 명확성을 먼저 택하고, 측정이 정당화할 때 최적화하세요. 예외는 알고리즘 복잡도 — O(n²)가 될 게 명백하면 처음부터 피합니다. 용량 산정은 정밀할 필요 없이 back-of-envelope(자릿수 추정)면 충분합니다: "초당 만 건 × 1KB = 10MB/s, 하루 ~860GB" 수준의 감각이 설계 방향을 가릅니다.

12.4Case study — 주문 생성 설계

예시 요구사항: 사용자는 상품을 주문하고, 재고는 중복 차감되면 안 되며, 결제와 알림은 실패해도 재시도되어야 합니다. 이때 설계의 핵심은 "주문 생성"과 "후처리"를 같은 무게로 다루지 않는 것입니다.

ORDER CREATION DESIGN WALKTHROUGH API: POST /orders는 idempotency key를 받아 같은 요청의 중복 주문을 막는다
Data: orders, order_items, stock_reservations, outbox_events를 같은 DB에 둔다
Transaction: 주문 생성, 재고 예약, outbox 기록은 하나의 트랜잭션으로 커밋한다
Concurrency: 상품 재고 행은 optimistic lock 또는 조건부 update로 음수 재고를 막는다
Async: 결제 요청, 알림, 검색 색인은 outbox consumer가 at-least-once로 처리한다
Idempotency: producer는 idempotency key, consumer는 event id inbox로 중복 처리를 방어한다
Failure: 결제 실패는 주문을 취소하거나 결제대기 만료 작업으로 회수한다
Observability: 주문 생성 성공률, 재고 예약 실패율, 결제 retry count, outbox lag, consumer error를 대시보드에 둔다

이 사례에서 "처음부터 마이크로서비스"가 답은 아닙니다. 먼저 모듈 경계를 선명하게 두고, 결제·재고·알림의 실패 모드와 데이터 소유권이 실제로 갈라질 때 서비스 분리를 검토합니다.

BE-15

용어집

멱등성Idempotency
같은 요청을 여러 번 해도 결과가 같은 성질. 재시도·중복 메시지 안전의 핵심. PUT/DELETE는 멱등, POST는 아님.
ACIDAtomicity·Consistency·Isolation·Durability
트랜잭션의 4가지 약속 — 전부-또는-전무, 제약 유지, 동시 격리, 커밋 영속.
격리수준Isolation Level
동시 트랜잭션이 서로를 얼마나 보는지의 등급. 높을수록 정합성↑ 동시성↓. Read Committed / Repeatable Read / Serializable.
N+1 문제N+1 Query
목록 1번 조회 후 각 항목마다 추가 쿼리 N번. ORM의 대표 성능 함정 — fetch join·배치 로딩으로 해결.
영속성 컨텍스트Persistence Context
JPA의 1차 캐시. 엔티티 변경을 감지(dirty checking)해 커밋 시 자동 UPDATE. 트랜잭션 경계와 생명주기를 공유.
낙관적/비관적 락Optimistic / Pessimistic Lock
동시 수정 방어 — 버전 검사 후 충돌 시 재시도(낙관적) vs 읽을 때 행 잠금(비관적). 충돌 빈도가 선택 기준.
Cache-asideLazy Loading Cache
앱이 캐시 먼저 보고 없으면 DB→캐시 채움. 가장 흔한 캐싱 패턴. 무효화는 앱 책임.
Cache StampedeThundering Herd
인기 키 동시 만료로 요청이 DB에 몰리는 현상. single-flight 잠금·TTL 지터·사전 갱신으로 방어.
At-least-onceDelivery Guarantee
메시지가 최소 한 번(중복 가능) 전달. 실무 표준 = at-least-once + 멱등 소비자.
Outbox 패턴Transactional Outbox
DB 저장과 메시지 발행의 원자성 확보 — 이벤트를 같은 트랜잭션의 outbox 테이블에 쓰고 별도 프로세스가 발행.
커넥션 풀Connection Pool
연결 수립 비용을 피하려 미리 맺은 연결을 재사용하는 풀. DB·HTTP 클라이언트의 기본.
서킷브레이커Circuit Breaker
연속 실패 시 호출을 차단해 빠르게 실패시키고 회복 시간을 주는 패턴. 캐스케이딩 장애 방어.
