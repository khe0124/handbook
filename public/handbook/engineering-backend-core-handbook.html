백엔드 엔지니어링 핸드북

백엔드 역량의 핵심은 기능 구현을 운영 가능한 시스템으로 바꾸는 설계 판단입니다. 같은 CRUD라도 트랜잭션 경계·인덱스·실패 처리·관측가능성에서 품질이 갈립니다. 이 핸드북은 "되게 만드는 법"이 아니라 틀려도 안전하고, 바뀌어도 견디고, 터지면 추적 가능한 백엔드를 만드는 판단의 지도입니다.

BE-01

무엇이 실무형 백엔드를 만드는가

실무형 백엔드 엔지니어는 더 많은 기능을 짜는 사람이 아니라, 모든 선택에 대해 "이 결정의 비용은 무엇인가"를 묻는 사람입니다. 코드가 도는 것은 시작점일 뿐입니다.

프론트엔드 경험의 자산은 그대로 옮겨옵니다 — 계약(인터페이스) 사고, 상태 관리, 사용자 관점은 백엔드의 API 설계·일관성·에러 응답으로 직결됩니다. 새로 장착할 근육은 지속성(persistence)과 동시성입니다: 프론트는 대개 한 사용자의 한 세션을 다루지만, 백엔드는 같은 데이터를 동시에 만지는 수천 개의 요청과 영구 저장된 상태를 다룹니다. 이 핸드북의 무게중심이 DB(BE-03·04)와 동시성(BE-05)에 쏠린 이유입니다.

CORE STANCE

숙련된 백엔드의 표식은 트레이드오프를 언어화하는 능력입니다. "JWT 쓸게요"가 아니라 "stateless라 확장은 쉽지만 즉시 무효화가 안 되니, 짧은 access + refresh로 가겠습니다"라고 말할 수 있는가. 정답을 외우는 게 아니라 비용과 대안을 같이 제시하는 것 — 면접도 실무도 여기서 갈립니다.

BE-02

HTTP·API 설계 — 계약으로서의 인터페이스

API는 코드가 아니라 계약입니다. 한 번 공개하면 수많은 클라이언트가 의존하고, 함부로 못 바꿉니다. 설계의 무게가 내부 함수와 다른 이유입니다.

2.1메서드 의미와 멱등성

HTTP 메서드는 단순 동사가 아니라 안전성(safe)과 멱등성(idempotent)의 약속입니다. GET은 안전(상태 변경 없음)하고 멱등하며, PUT·DELETE는 멱등(여러 번 호출해도 결과 동일), POST는 비멱등(호출마다 새 자원). 이 약속이 중요한 이유는 재시도 때문입니다 — 네트워크가 끊겨 응답을 못 받았을 때 멱등한 요청만 안전하게 다시 보낼 수 있습니다.

IDEMPOTENCY KEY

결제·주문 같은 비멱등 POST를 재시도 안전하게 만들려면 멱등성 키를 씁니다 — 클라이언트가 요청마다 고유 키를 헤더로 보내고, 서버는 같은 키의 두 번째 요청을 "이미 처리됨"으로 같은 응답을 돌려줍니다. "결제 버튼 두 번 눌러서 두 번 결제됨"을 막는 표준 장치입니다.

2.2상태코드 — 4xx와 5xx의 경계

상태코드는 의미군입니다: 2xx 성공, 3xx 리다이렉트, 4xx 클라이언트 잘못(요청이 틀림 — 재시도해도 같은 결과), 5xx 서버 잘못(서버가 못 함 — 재시도하면 될 수도). 이 경계가 운영을 가릅니다 — 입력 검증 실패를 500으로 던지면 알람이 울리고 온콜이 깨지지만, 실제로는 400이어야 합니다. "누구의 잘못인가"를 상태코드로 정직하게 말하는 것이 API의 기본 예의입니다.

2.3인증과 인가 — authN vs authZ

개념질문수단
인증 (authN)"너 누구냐"세션 쿠키, JWT, OAuth2/OIDC(소셜·SSO)
인가 (authZ)"이걸 할 권한 있냐"RBAC(역할), ABAC(속성), 자원 소유권 검사

둘은 다른 층입니다. 인증을 통과해도 이 자원에 접근할 권한은 따로 확인해야 합니다 — 이걸 빠뜨리면 BE-10의 IDOR(남의 id로 남의 데이터 조회)가 됩니다.

SESSION vs JWT

세션: 서버가 상태를 가짐 → 즉시 무효화 가능, 확장 시 세션 저장소(Redis) 공유 필요. JWT: stateless, 서명만 검증 → 수평 확장 쉽지만 만료 전 강제 로그아웃이 어렵다(토큰이 이미 발급됨). 실무 절충: access token은 짧게(분 단위) + refresh token으로 갱신, 블랙리스트는 refresh에만. "정답"이 아니라 무효화 요구 강도 vs 확장 편의의 트레이드오프입니다.

2.4진화 가능한 API

계약은 깨지지 않게 진화해야 합니다 — 버저닝(URL /v1 또는 헤더), 필드 추가는 OK·삭제/의미변경은 breaking, 페이지네이션은 offset(간단하지만 깊은 페이지·삽입에 취약)보다 대규모에선 cursor(키 기반)가 안정적입니다. 에러 응답도 계약입니다 — {code, message, details}처럼 구조를 표준화(RFC 9457 problem+json)해 클라이언트가 기계적으로 처리하게 하세요.

BE-03

관계형 DB — 트랜잭션·인덱스·격리

백엔드 실무의 진짜 시험장. 성능 문제의 대부분과 정합성 버그의 거의 전부가 여기서 납니다. ORM 뒤에 숨지 말고 직접 봐야 하는 층입니다.

3.1ACID — 트랜잭션의 4가지 약속

  • 원자성(Atomicity) — 전부 되거나 전부 안 된다. 중간 실패 시 롤백.
  • 일관성(Consistency) — 제약(외래키·유니크·체크)을 깨는 상태로 못 간다.
  • 격리성(Isolation) — 동시 트랜잭션이 서로의 중간 상태를 안 본다 (수준은 조절 가능 → 3.3).
  • 지속성(Durability) — 커밋되면 장애가 나도 남는다 (WAL·로그).

3.2인덱스 — B-tree가 빠른 이유

인덱스는 정렬된 사본입니다. B-tree는 균형 트리라 O(log n)에 값을 찾고, 정렬되어 있어 범위 검색(BETWEEN, ORDER BY)과 접두 매칭에도 강합니다. 실무에서 자주 밟는 함정:

규칙이유
복합 인덱스는 선두 컬럼부터(a,b) 인덱스는 a 또는 a,b 검색엔 쓰이지만 b 단독엔 안 쓰인다 (전화번호부 정렬과 같음)
컬럼에 함수·연산을 걸면 인덱스 무력화WHERE DATE(created_at)=...는 인덱스를 못 탄다 → 범위 조건으로 재작성
앞 와일드카드 LIKE '%foo'는 못 탄다정렬의 시작점을 모르므로. 'foo%'는 탄다
커버링 인덱스필요한 컬럼이 인덱스에 다 있으면 테이블을 안 읽는다 (가장 빠른 읽기)
인덱스는 공짜가 아니다쓰기마다 인덱스도 갱신 → 과다 인덱스는 쓰기·저장 비용. 필요한 것만

판단의 도구는 EXPLAIN(실행계획)입니다 — "이 쿼리가 인덱스를 타는가, 풀스캔하는가, 몇 행을 훑는가"를 추측하지 말고 보세요. 측정 없는 인덱스 추가는 BE-12의 "측정 없는 최적화"와 같은 실수입니다.

N+1 — ORM의 대표 함정

목록 100건을 가져온 뒤 각 건의 연관 데이터를 위해 쿼리를 100번 더 날리는 것 = 1+100 쿼리. 단위 테스트·로컬에선 안 보이다가 운영 부하에서 터집니다. 해법은 fetch join·배치 로딩·명시적 IN 조회 — 핵심은 "이 코드가 쿼리를 몇 번 날리는가"를 항상 의식하는 것입니다(→ BE-04).

BE-04

ORM·JPA — 추상화의 비용

ORM은 객체와 테이블 사이의 임피던스 불일치를 가립니다 — 가리지만 없애지 않습니다. 생성되는 SQL을 모르면, ORM은 편의가 아니라 지뢰밭이 됩니다.

4.1영속성 컨텍스트 — JPA의 심장

JPA의 핵심은 영속성 컨텍스트(persistence context)라는 1차 캐시입니다. 트랜잭션 동안 조회한 엔티티를 들고 있다가, 변경을 감지(dirty checking)해 커밋 시점에 자동으로 UPDATE를 날립니다 — 그래서 save()를 명시 안 해도 필드만 바꾸면 반영됩니다. 이게 강력하면서 동시에 혼란의 원천입니다: "내가 안 부른 쿼리가 왜 나가지?"의 답이 대개 dirty checking과 flush 타이밍입니다.

4.2지연 로딩과 트랜잭션 경계

@Transactional의 범위 = 영속성 컨텍스트의 생명주기입니다. 지연 로딩(lazy)된 연관을 트랜잭션 밖에서 접근하면 그 유명한 LazyInitializationException이 납니다 — 컨텍스트가 이미 닫혀서 추가 쿼리를 못 날리는 거죠. 실무 판단:

  • 기본은 lazy — 필요할 때만 로딩. @ManyToOne의 기본이 eager인 함정 주의.
  • 화면에서 함께 쓸 연관은 fetch join으로 한 번에 (N+1 회피).
  • 트랜잭션 경계를 서비스 계층에 명확히 — 컨트롤러·뷰에 영속성 컨텍스트를 끌고 가지 않기(OSIV 논쟁의 핵심).
RULE

ORM을 SQL 없이 쓰지 마라. JPA를 배우는 가장 빠른 길은 SQL 로그를 켜놓고(show_sql, p6spy) "내 코드 한 줄이 어떤 쿼리를 만드는가"를 눈으로 확인하는 것입니다. 복잡한 조회·통계는 ORM을 고집하지 말고 네이티브 쿼리·QueryDSL로 내려가는 게 실무 판단입니다 — 추상화는 도구지 신앙이 아닙니다.

BE-05

동시성·일관성

단일 요청에선 멀쩡하던 코드가 동시 요청에서 무너집니다. "재고 1개 남았는데 두 명이 동시에 주문"은 백엔드의 통과의례입니다.

5.1경쟁 상태와 갱신 손실

두 트랜잭션이 같은 값을 읽고(read), 각자 계산하고, 각자 쓰면(write) — 나중 쓰기가 먼저 쓰기를 덮습니다(lost update). 애플리케이션 코드의 if (재고 > 0) 재고--는 동시성 하에서 거의 항상 틀립니다. 해결은 두 갈래입니다.

전략방식언제
비관적 락
Pessimistic		읽을 때 행을 잠금(SELECT ... FOR UPDATE) — 남이 못 건드림충돌이 잦고, 충돌 비용이 큰 경우(재고·잔액). 락 대기·데드락 위험
낙관적 락

5.2멱등성 — 분산의 안전벨트

네트워크·메시지큐·재시도가 있는 한 "정확히 한 번"은 보장되지 않습니다(→ BE-07). 그래서 백엔드는 같은 작업이 두 번 실행돼도 결과가 같도록 설계합니다 — 멱등성 키(BE-02), 유니크 제약으로 중복 삽입 차단, "이미 처리된 주문인가" 상태 검사. 멱등성은 기능이 아니라 분산 환경의 기본 자세입니다.

CORE INSIGHT

동시성 버그의 공통 뿌리는 "읽고-판단하고-쓰는" 사이의 틈입니다. 그 틈에 남이 끼어듭니다. 해법은 항상 둘 중 하나 — 틈을 잠그거나(비관적), 틈에서 바뀌었는지 나중에 확인하거나(낙관적). DB의 원자적 연산(UPDATE ... SET n=n-1 WHERE n>0) 한 줄로 틈 자체를 없애는 것이 가장 단순한 정답일 때도 많습니다.

BE-06

캐싱 — 속도와 정합성의 거래

캐시는 가장 손쉬운 성능 개선이자 가장 흔한 정합성 버그의 원천입니다. "캐시 무효화는 컴퓨터 과학의 두 어려운 문제 중 하나"라는 농담은 진담입니다.

패턴동작특징
Cache-aside (가장 흔함)앱이 캐시 먼저 조회 → 없으면 DB 읽고 캐시에 채움구현 단순·범용. 첫 요청은 느림(cold), 무효화는 앱 책임
Read-through캐시 계층이 DB 읽기를 대행앱 코드 단순. 캐시 라이브러리 의존
Write-through쓸 때 캐시와 DB 동시 갱신일관성↑, 쓰기 지연↑
Write-back캐시에 먼저 쓰고 나중에 DB 반영쓰기 빠름, 장애 시 유실 위험

핵심 결정은 무효화(invalidation)와 TTL입니다. 데이터가 바뀌었는데 캐시가 옛 값을 들고 있으면(stale) 사용자는 틀린 걸 봅니다 — 짧은 TTL은 신선도를, 긴 TTL은 부하 절감을 삽니다. 정답은 데이터 특성에 따라 다릅니다(가격은 짧게, 카테고리 목록은 길게).

CACHE STAMPEDE

인기 키의 캐시가 동시에 만료되면, 수천 요청이 한꺼번에 DB로 몰립니다(thundering herd) — 캐시를 깔았는데 DB가 더 위험해집니다. 방어: 만료 직전 한 요청만 갱신하게 잠금(mutex/single-flight), TTL에 무작위 지터 추가, 백그라운드 사전 갱신. 캐시는 "있으면 빠른" 게 아니라 없을 때 무슨 일이 나는가까지 설계해야 합니다.

BE-07

메시징·비동기

동기 호출은 두 서비스를 시간으로 묶습니다 — 하나가 느리거나 죽으면 다른 하나도 같이 멈춥니다. 큐는 그 결합을 끊어, 시간과 장애를 분리합니다.

도구 유형모델대표강점
메시지 큐작업을 소비하면 사라짐 (work queue)RabbitMQ, AWS SQS작업 분산·부하 평탄화
이벤트 스트림로그에 쌓이고 여러 소비자가 각자 읽음Apache Kafka이벤트 소싱·재처리·다중 구독

전달 보장은 세 단계입니다: at-most-once(유실 가능), at-least-once(중복 가능), exactly-once(이상적). 실무의 현실은 거의 항상 at-least-once + 멱등 소비자입니다 — 메시지가 중복 도착할 수 있다고 가정하고, 소비 측에서 "이미 처리했나"를 검사해 중복을 무해하게 만듭니다(→ BE-05 멱등성).

OUTBOX PATTERN

"DB에 주문을 저장하고 + 메시지큐에 이벤트를 발행한다"를 원자적으로 하려는 순간 문제가 생깁니다 — DB 커밋은 됐는데 발행이 실패하면(또는 반대) 둘이 어긋납니다(이중 쓰기 문제). 해법인 outbox 패턴: 이벤트를 같은 DB 트랜잭션 안의 outbox 테이블에 함께 쓰고, 별도 프로세스가 그 테이블을 읽어 발행합니다. "DB 트랜잭션 하나"라는 원자성에 메시지 발행을 얹는 영리한 우회입니다.

DLQ · 재시도

소비가 계속 실패하는 메시지는 무한 재시도하면 큐를 막습니다 — 지수 백오프 + 지터로 재시도하되, N회 실패하면 DLQ(dead letter queue)로 보내 격리하고 사람이 보게 하세요. 실패를 삼키지도, 무한 반복하지도 않는 중간 길입니다.

BE-08

네트워킹 — 분산의 토대

백엔드는 네트워크 위에 삽니다. 그리고 네트워크의 8가지 오류(fallacy) 중 첫 번째가 "네트워크는 신뢰할 수 있다"입니다 — 네트워크는 느리고, 실패하고, 가끔 거짓말합니다.

8.1TCP·DNS·TLS — 한 번의 요청에 숨은 비용

  • TCP는 연결을 3-way handshake로 맺습니다 — 연결 수립 자체가 왕복 비용입니다. 그래서 매 요청마다 새 연결을 맺지 않고 커넥션 풀로 재사용합니다(DB·HTTP 클라이언트 모두).
  • DNS는 도메인 → IP 변환이며 TTL 동안 캐시됩니다. 배포·장애조치 시 DNS TTL이 길면 트래픽 전환이 늦어집니다.
  • TLS는 추가 handshake로 암호화 채널을 맺고 인증서로 신원을 증명합니다 — HTTPS의 토대. 인증서 만료는 흔하고 치명적인 운영 사고입니다.

8.2외부 호출의 3종 안전장치

타임아웃
Timeout→ 재시도
Retry+백오프→ 서킷브레이커
Circuit Breaker

다른 서비스를 호출할 때 반드시 갖춰야 할 셋입니다. 타임아웃 — 무한 대기는 스레드·커넥션을 고갈시켜 한 서비스의 느림이 전체 장애로 번지게 합니다(연결·읽기 타임아웃 둘 다). 재시도 — 일시적 실패엔 효과적이지만, 즉시·고정 간격 재시도는 죽어가는 서버를 더 때립니다 → 지수 백오프 + 지터, 그리고 멱등한 요청만. 서킷브레이커 — 연속 실패가 임계를 넘으면 회로를 열어 한동안 호출 자체를 차단하고 빠르게 실패시켜, 회복할 시간을 줍니다.

FAILURE MODE — 캐스케이딩 장애

타임아웃 없는 호출 하나가 전체를 무너뜨리는 전형: A가 느려진 B를 무한 대기 → A의 스레드풀 고갈 → A도 응답 불능 → A를 호출하던 C도... 한 군데의 느림이 도미노로 번집니다. 타임아웃과 격벽(bulkhead)은 선택이 아니라 분산 시스템의 기본 위생입니다.

E2E smoke: 로그인, 핵심 생성/조회/취소 같은 소수 happy path만 확인
Regression: 장애나 버그가 재발하지 않도록 실패 사례를 테스트로 고정
TDD as DESIGN

버그를 고칠 땐 먼저 그 버그를 재현하는 실패 테스트를 쓰세요 — 고친 뒤 그 테스트가 통과하면 "정말 고쳤다"의 증거이자, 재발을 막는 회귀 방어선이 됩니다. 테스트를 먼저 쓰면 자연스럽게 "이 코드를 어떻게 호출할까(인터페이스)"를 먼저 설계하게 됩니다 — TDD의 진짜 가치는 커버리지가 아니라 설계 압력입니다.

BE-12

시스템 설계 사고

백엔드 설계의 최종 시험은 "정답을 아는가"가 아니라 "트레이드오프를 설명하는가"입니다. 시스템 설계엔 정답이 없고, 제약에 맞는 선택과 그 비용의 명시가 있을 뿐입니다.

12.1먼저 묻는 질문 — 비기능 요구

기능("무엇을")보다 비기능 요구("얼마나")가 설계를 결정합니다: 예상 트래픽(읽기:쓰기 비율, peak), 지연 목표(p99), 가용성 목표, 데이터 규모와 증가율, 일관성 요구(강한가 최종이어도 되나). 이걸 안 묻고 그린 설계는 추측입니다 — 당신의 작업 원칙 "요구사항이 모호하면 채우지 말고 질문하라"가 그대로 적용됩니다.

BACKEND DESIGN QUESTION FLOW 요구사항: 사용자가 실제로 끝내야 하는 행동과 실패 시 손실은 무엇인가
트래픽: read/write 비율, peak QPS, p95·p99 latency 목표, 데이터 증가율은 어느 정도인가
일관성: 즉시 맞아야 하는 값과 나중에 맞아도 되는 값은 무엇인가
데이터 모델: aggregate root, 고유 제약, 상태 전이, 삭제·보존 정책은 어디에 있는가
API 계약: request/response, error code, idempotency key, pagination, versioning을 어떻게 고정할 것인가
트랜잭션 경계: 한 번에 커밋해야 하는 변경과 이벤트로 분리할 변경은 무엇인가
장애 모드: 외부 API, DB, cache, queue가 느리거나 죽었을 때 사용자는 무엇을 보게 되는가
확장 경로: 지금은 단순하게 두되, 어느 지표가 넘으면 cache, queue, read model, sharding을 검토할 것인가
관측 지표: 성공률, 에러율, p99, lock wait, queue lag, retry count를 어디서 볼 것인가

12.2확장과 일관성의 거래

  • 수직 확장(서버 키우기)은 단순하지만 한계·단일 장애점. 수평 확장(서버 늘리기)은 무한하지만 무상태(stateless) 설계와 데이터 분산을 요구합니다.
  • CAP — 네트워크 분단(P)은 피할 수 없으니, 분단 시 일관성(C)과 가용성(A) 중 무엇을 포기할지의 선택입니다. 은행 잔액은 C, 좋아요 수는 A.
  • 일관성 모델 — 강한 일관성(항상 최신)은 비싸고, 최종 일관성(잠깐 어긋나도 결국 수렴)은 싸고 빠릅니다. 도메인이 결정합니다.
설계 결정단순하게 시작분리하는 신호주의할 비용
모듈 vs 서비스한 코드베이스 안의 명확한 module boundary배포 주기, 데이터 소유권, 장애 격리가 실제로 갈라질 때분산 트랜잭션, 네트워크 실패, 관측 복잡도
동기 API vs 비동기 메시지사용자 응답에 필요한 작업만 동기 처리느린 외부 호출, 후처리, fan-out, 재시도가 필요할 때중복 메시지, 순서 보장, 보상 처리
RDB vs 캐시·검색·큐정합성의 기준은 RDB에 둔다읽기 병목, 검색 요구, 비동기 처리량이 명확해질 때무효화, 재색인, lag, 운영 장애 지점 증가
강한 일관성 vs 최종 일관성돈, 재고, 권한은 강한 일관성알림, 추천, 집계처럼 잠깐 틀려도 되는 화면사용자에게 보이는 상태 불일치와 보정 로직
단일 트랜잭션 vs outbox/saga한 DB 안의 핵심 상태는 단일 트랜잭션외부 시스템 발행, 여러 서비스 변경, 장기 작업중간 상태, 재시도, 보상 이벤트, 모니터링
DESIGN DECISION MATRIX 결정은 기술 선호가 아니라 제약의 결과여야 합니다. 설계 리뷰에서는 "왜 이 기술인가"보다 "어떤 제약 때문에 이 선택이 최소 비용인가", "어떤 지표가 나오면 다음 단계로 바꿀 것인가", "되돌리거나 확장할 수 있는가"를 먼저 설명하세요.

12.3장애를 전제로 설계

"무엇이 깨질 수 있는가"를 먼저 그리는 것이 실무 설계입니다 — 단일 장애점(SPOF) 제거, 우아한 성능저하(graceful degradation: 추천이 죽어도 상품은 보이게), 백프레셔(과부하 시 받지 않고 거절), 격벽(한 기능의 자원 고갈이 다른 기능을 안 죽이게).

측정 없는 최적화 금지

"이게 더 빠를 것 같아서"는 근거가 아닙니다 — 병목은 측정한 뒤 잡으세요(프로파일링·메트릭). 명확성과 성능이 충돌하면 명확성을 먼저 택하고, 측정이 정당화할 때 최적화하세요. 예외는 알고리즘 복잡도 — O(n²)가 될 게 명백하면 처음부터 피합니다. 용량 산정은 정밀할 필요 없이 back-of-envelope(자릿수 추정)면 충분합니다: "초당 만 건 × 1KB = 10MB/s, 하루 ~860GB" 수준의 감각이 설계 방향을 가릅니다.

12.4Case study — 주문 생성 설계

예시 요구사항: 사용자는 상품을 주문하고, 재고는 중복 차감되면 안 되며, 결제와 알림은 실패해도 재시도되어야 합니다. 이때 설계의 핵심은 "주문 생성"과 "후처리"를 같은 무게로 다루지 않는 것입니다.

ORDER CREATION DESIGN WALKTHROUGH API: POST /orders는 idempotency key를 받아 같은 요청의 중복 주문을 막는다
Data: orders, order_items, stock_reservations, outbox_events를 같은 DB에 둔다
Transaction: 주문 생성, 재고 예약, outbox 기록은 하나의 트랜잭션으로 커밋한다
Concurrency: 상품 재고 행은 optimistic lock 또는 조건부 update로 음수 재고를 막는다
Async: 결제 요청, 알림, 검색 색인은 outbox consumer가 at-least-once로 처리한다
Idempotency: producer는 idempotency key, consumer는 event id inbox로 중복 처리를 방어한다
Failure: 결제 실패는 주문을 취소하거나 결제대기 만료 작업으로 회수한다
Observability: 주문 생성 성공률, 재고 예약 실패율, 결제 retry count, outbox lag, consumer error를 대시보드에 둔다

이 사례에서 "처음부터 마이크로서비스"가 답은 아닙니다. 먼저 모듈 경계를 선명하게 두고, 결제·재고·알림의 실패 모드와 데이터 소유권이 실제로 갈라질 때 서비스 분리를 검토합니다.

리뷰 축좋은 설계의 증거위험 신호

BE-13

용어집

멱등성Idempotency
같은 요청을 여러 번 해도 결과가 같은 성질. 재시도·중복 메시지 안전의 핵심. PUT/DELETE는 멱등, POST는 아님.
ACIDAtomicity·Consistency·Isolation·Durability
트랜잭션의 4가지 약속 — 전부-또는-전무, 제약 유지, 동시 격리, 커밋 영속.
격리수준Isolation Level
동시 트랜잭션이 서로를 얼마나 보는지의 등급. 높을수록 정합성↑ 동시성↓. Read Committed / Repeatable Read / Serializable.
N+1 문제N+1 Query
목록 1번 조회 후 각 항목마다 추가 쿼리 N번. ORM의 대표 성능 함정 — fetch join·배치 로딩으로 해결.
영속성 컨텍스트Persistence Context
JPA의 1차 캐시. 엔티티 변경을 감지(dirty checking)해 커밋 시 자동 UPDATE. 트랜잭션 경계와 생명주기를 공유.
낙관적/비관적 락Optimistic / Pessimistic Lock
동시 수정 방어 — 버전 검사 후 충돌 시 재시도(낙관적) vs 읽을 때 행 잠금(비관적). 충돌 빈도가 선택 기준.
Cache-asideLazy Loading Cache
앱이 캐시 먼저 보고 없으면 DB→캐시 채움. 가장 흔한 캐싱 패턴. 무효화는 앱 책임.
Cache StampedeThundering Herd
인기 키 동시 만료로 요청이 DB에 몰리는 현상. single-flight 잠금·TTL 지터·사전 갱신으로 방어.
At-least-onceDelivery Guarantee
메시지가 최소 한 번(중복 가능) 전달. 실무 표준 = at-least-once + 멱등 소비자.
Outbox 패턴Transactional Outbox
DB 저장과 메시지 발행의 원자성 확보 — 이벤트를 같은 트랜잭션의 outbox 테이블에 쓰고 별도 프로세스가 발행.
커넥션 풀Connection Pool
연결 수립 비용을 피하려 미리 맺은 연결을 재사용하는 풀. DB·HTTP 클라이언트의 기본.
서킷브레이커Circuit Breaker
연속 실패 시 호출을 차단해 빠르게 실패시키고 회복 시간을 주는 패턴. 캐스케이딩 장애 방어.
지수 백오프Exponential Backoff + Jitter
재시도 간격을 점점 늘리고(2배씩) 무작위 지터를 더해, 죽어가는 서버에 몰리는 재시도 폭주를 방지.
상관관계 IDCorrelation / Trace ID
요청 진입점에서 발급해 모든 로그·하위 호출에 전파하는 식별자. 분산 추적의 실타래.
RED 메트릭Rate·Errors·Duration
서비스 건강의 3대 지표. 평균이 아니라 p95·p99로 꼬리 지연을 본다.
liveness/readinessHealth Probe
살아있나(죽으면 재시작) vs 요청 받을 준비됐나(아니면 트래픽 차단). 합치면 사고.
IDORInsecure Direct Object Reference
남의 식별자로 남의 자원에 접근하는 인가 누락. 가장 흔한 실제 침해. authN≠authZ.
Parameterized QueryPrepared Statement
값을 코드와 분리해 바인딩하는 쿼리. SQL injection의 근본 방어 — 문자열 결합 금지.
CAP 정리Consistency·Availability·Partition
네트워크 분단 시 일관성과 가용성 중 하나를 포기해야 한다는 분산 시스템의 제약.
최종 일관성Eventual Consistency
잠깐 어긋나도 결국 수렴하는 일관성 모델. 강한 일관성보다 싸고 빠름 — 도메인이 허용하면 선택.
TestcontainersIntegration Test
테스트 시 진짜 DB·브로커를 컨테이너로 띄워 통합 테스트. mock DB의 거짓 안심을 피함.
BE-14

실무 API 설계 심화

백엔드 API는 기능 호출점이 아니라 장기적으로 유지되는 제품 계약입니다. 클라이언트가 늘고 배포 주기가 갈라질수록 backward compatibility가 핵심 품질이 됩니다.

주제실무 기준실패 신호
idempotency key결제, 주문, 포인트 적립처럼 중복 실행이 위험한 POST에 적용네트워크 timeout 후 사용자가 재시도하면 중복 생성됨
paginationoffset은 단순 목록, cursor는 대량·실시간 목록에 사용페이지 이동 중 데이터 삽입으로 누락/중복 발생
error contractcode, message, field, retryable, traceId를 구조화프론트가 문자열 파싱으로 분기
API versioning필드 추가는 호환, 의미 변경과 삭제는 버전 또는 migration 필요모바일 앱 구버전이 갑자기 깨짐
API CONTRACT MINI Endpoint / method: 자원과 행동을 URL, method, status code로 명확히 표현
Request DTO validation: 필수값, 형식, 범위, cross-field rule을 서버에서 검증
Response schema: 성공 응답 field, nullable, enum, timestamp format을 고정
Error contract: code, message, field, retryable, traceId를 구조화
Idempotency rule: 중복 위험 POST는 idempotency key와 request fingerprint 저장
Pagination rule: offset, cursor, keyset 중 데이터 특성에 맞게 선택
AuthZ rule: role뿐 아니라 resource ownership과 tenant boundary 확인
Backward compatibility note: 필드 삭제, 의미 변경, enum 추가의 consumer 영향 기록
Case study

주문 생성 API가 timeout을 냈을 때 클라이언트는 성공인지 실패인지 알 수 없습니다. 이때 같은 idempotency key로 재요청하면 서버는 이전 처리 결과를 반환해야 합니다. key 저장소에는 요청 fingerprint, 응답, 만료 시간을 함께 저장하고, 같은 key에 다른 payload가 오면 충돌로 거절해야 합니다.

BE-15

대규모 운영 판단

시스템이 커지면 “어떤 기술을 쓸까”보다 “어디까지 단순하게 버티고, 언제 복잡도를 받아들일까”가 중요해집니다. 고급 백엔드 판단은 규모, 위험, 팀 역량, 운영 비용을 함께 놓고 결정합니다.

상황먼저 할 일복잡도를 받아들일 신호
조회가 느림EXPLAIN, slow query, cardinality, index hit ratio 확인인덱스와 쿼리 재작성 후에도 p95 목표를 못 맞춤

Capacity thinking

용량 산정은 완벽한 예측이 아니라 자릿수 감각입니다. 예를 들어 초당 2,000요청, 평균 응답 20KB면 egress만 약 40MB/s입니다. 요청당 DB 쿼리 5개면 DB는 초당 10,000쿼리를 받습니다. 이 숫자가 connection pool, CPU, index, cache, network 비용을 압박하는지 먼저 계산해야 합니다.

Case study

관리자 검색 화면이 느려졌다고 바로 Elasticsearch를 붙이면 운영 복잡도가 커집니다. 먼저 실제 검색 조건, 정렬 기준, total count 필요 여부, keyset pagination 가능 여부, 부분 인덱스와 covering index를 확인합니다. 검색어 형태가 전문 검색이고 필터 조합이 많으며 DB가 OLTP 부하를 침해할 때 별도 검색 엔진을 검토합니다.

BE-16

요청 처리 경로와 트랜잭션 경계

백엔드 개념을 외운 답변처럼 보이게 만드는 가장 큰 이유는 요청 하나가 서버 안에서 어떻게 흘러가는지 설명하지 못하기 때문입니다. 좋은 백엔드 답변은 HTTP 요청이 컨트롤러, 애플리케이션 서비스, 도메인, 저장소, 외부 시스템을 통과하며 어디서 실패할 수 있는지까지 따라갑니다.

REQUEST PATH client → controller → request validation → authentication → authorization
→ application service (transaction boundary)
→ domain rule → repository / external client → commit or rollback
→ response mapping → structured log / metric / trace

컨트롤러는 HTTP를 애플리케이션 명령으로 번역하는 얇은 경계입니다. 여기서 인증, 입력 형식, 상태코드, 응답 포맷을 다룹니다. 애플리케이션 서비스는 유스케이스를 조율하고 트랜잭션 경계를 잡습니다. 도메인은 “주문은 취소 가능한 상태에서만 취소된다”, “재고는 음수가 될 수 없다” 같은 불변식을 보호합니다. 저장소는 영속성 세부사항이고, 외부 API 클라이언트는 실패와 지연이 기본값인 의존성입니다.

면접에서 이렇게 말하기

90초 답변백엔드 요청은 controller에서 HTTP와 입력을 검증하고, application service에서 유스케이스와 트랜잭션 경계를 잡고, domain에서 불변식을 보호한 뒤 repository로 저장됩니다. 저는 트랜잭션 안에는 반드시 함께 성공해야 하는 DB 변경만 넣고, 외부 API나 메시지 발행은 outbox, 상태 전이, 멱등성 키로 분리해 부분 실패를 다룹니다. 그래서 좋은 백엔드 설계는 기능이 되는 것보다 실패했을 때 어떤 상태로 남는지를 먼저 정하는 일이라고 봅니다.
BE-17

동시성 사고와 멱등성 설계

백엔드에서 “가끔 두 번 처리된다”, “재고가 음수가 된다”, “같은 쿠폰이 여러 번 쓰인다”는 버그는 대부분 동시성, 재시도, 중복 요청을 정상 입력으로 보지 않았기 때문에 생깁니다.

문제왜 생기는가방어선

가장 강한 방어선은 애플리케이션 if문이 아니라 DB 제약입니다. “이미 쓴 쿠폰인지 조회하고 없으면 저장”은 두 요청이 동시에 들어오면 둘 다 조회를 통과할 수 있습니다. 반대로 UNIQUE(user_id, coupon_id) 제약을 두고 insert를 먼저 시도하면 둘 중 하나는 DB가 거절합니다. 애플리케이션은 그 예외를 “이미 사용됨”으로 번역하면 됩니다.

ATOMIC STOCK DECREASE UPDATE product
SET stock = stock - :qty
WHERE id = :id AND stock >= :qty;
affected rows == 1 이면 성공, 0이면 재고 부족 또는 조건 불만족
락 선택 기준

낙관적 락은 “대부분 충돌하지 않는다”는 가정에서 좋습니다. 충돌하면 실패시키고 다시 읽어 재시도합니다. 비관적 락은 “충돌이 자주 나고, 기다리는 것이 실패보다 싸다”는 상황에서 씁니다. 하지만 락을 오래 잡으면 대기열이 생기고 데드락 가능성이 커집니다. 고급 답변은 “어떤 락을 쓰겠다”가 아니라 “충돌 빈도, 대기 비용, 사용자 경험, 재시도 가능성으로 고르겠다”입니다.

ARCH-01

좋은 아키텍처란

좋은 아키텍처의 정의는 "빠르다"도 "확장된다"도 아니다. 변경 비용이 낮은 구조다 — 요구가 바뀌었을 때 고쳐야 할 코드의 범위가 작고, 그 영향이 예측 가능한 것. 거의 모든 설계 원칙은 이 한 문장의 따름정리다.

두 가지 축이 변경 비용을 지배한다. 결합도(coupling)는 한 모듈이 바뀔 때 함께 바뀌어야 하는 다른 모듈의 양이다 — 낮을수록 좋다. 응집도(cohesion)는 한 모듈 안의 요소들이 같은 목적을 향하는 정도다 — 높을수록 좋다. 결합도가 높으면 작은 변경이 멀리까지 번지고, 응집도가 낮으면 하나의 변경이 여러 모듈에 흩어진다. 둘 다 "함께 바뀌는 것을 한곳에, 따로 바뀌는 것을 떨어뜨려라"는 같은 원칙의 두 얼굴이다.

COUPLING × COHESION — 변경의 전파 범위 [ 모듈 A ] ───strong─── [ 모듈 B ] ← A 변경 시 B도 변경 (전파)
[ 모듈 A ] - - weak - - [ 모듈 B ] ← A 변경이 B에 안 닿음 (격리)

GOAL low coupling + high cohesion = 변경이 국소적(local)으로 멈춤

실무에서 아키텍처가 좋은지 판별하는 가장 정직한 질문은 셋이다 — "이 부분을 따로 이해할 수 있는가, 따로 변경할 수 있는가, 따로 테스트할 수 있는가." 셋 다 "예"면 그 경계는 건강하다. 하나라도 "아니오"면 결합이 숨어 있다는 신호다. 그리고 의존성의 방향이 결정적이다 — 자주 바뀌는 것(세부사항: DB, 프레임워크, 외부 API)이 거의 안 바뀌는 것(정책: 비즈니스 규칙)에 의존해야지, 그 반대가 되면 규칙이 세부사항에 인질로 잡힌다(→ ARCH-03 의존성 역전).

판별 질문"예"면 의미하는 것
독립적으로 이해 가능한가응집도가 높다 — 한 모듈을 읽을 때 다른 모듈을 펼칠 필요가 없다
독립적으로 변경 가능한가결합도가 낮다 — 변경이 경계 밖으로 새지 않는다
독립적으로 테스트 가능한가의존성이 주입 가능하다 — 외부 없이 정책만 검증할 수 있다
측정의 함정

결합도·응집도는 정량 지표가 아니라 설계 감각이다. "이 파일을 고치려면 평소 몇 개의 다른 파일을 같이 열게 되는가"가 실무적 결합도 측정에 가깝다. 추상적인 점수보다, 최근 한 달의 PR에서 한 변경이 건드린 파일의 분포를 보라 — 무관해 보이는 모듈들이 매번 함께 바뀌면 거기에 숨은 결합이 있다.

ARCH-02

계층형 아키텍처

가장 흔하고, 그래서 가장 잘못 쓰이는 구조. 코드를 역할별 층(layer)으로 쌓고, 위층이 아래층에만 의존하게 한다. Spring 프로젝트의 controller / service / repository가 바로 이 패턴의 축약형이다.

계층책임 / Spring 대응
presentation외부 요청 수신·응답 직렬화. HTTP·검증·인증. @RestController
application유스케이스 조율(orchestration)·트랜잭션 경계. @Service
domain비즈니스 규칙·엔티티·불변식. 프레임워크에 무지해야 함
infrastructureDB·메시지큐·외부 API 등 기술 세부. @Repository 구현
LAYERED — 의존성은 위에서 아래로 한 방향 presentation ──depends on──▶ application
application   ──depends on──▶ domain
domain       ──depends on──▶ infrastructure ✗ 누수!
(도메인이 DB를 알게 되는 순간 규칙이 기술에 종속)

계층형의 가치는 관심사 분리와 낮은 진입 장벽이다 — 새 팀원도 "이건 어느 층 일인가"로 위치를 잡는다. 함정은 마지막 줄에 있다. 도메인 계층이 JPA @Entity나 EntityManager에 직접 의존하기 시작하면 계층 누수(layer leakage)가 일어난다. 비즈니스 규칙을 테스트하려는데 DB를 띄워야 하고, ORM을 바꾸려는데 도메인 로직이 함께 무너진다. 의존 방향이 "정책 → 세부사항"으로 뒤집힌 것이다.

안티패턴 — Anemic Domain Model

계층형이 흔히 빠지는 함정: 도메인 객체가 getter/setter만 가진 데이터 자루가 되고, 모든 로직이 service에 몰린다(빈혈 도메인 모델). 그러면 "객체지향"은 이름뿐이고 사실상 절차적 코드다. 규칙은 그 규칙이 보호하는 데이터와 같은 객체에 있어야 한다 — order.cancel()이 orderService.cancel(order)보다 응집도가 높다. 이 경계를 진지하게 다루는 것이 ARCH-03·04로 가는 동기다.

ARCH-03

헥사고날·클린 아키텍처

계층형의 누수 문제를 정면으로 푸는 구조. 핵심 발상 하나 — 도메인을 안쪽에 두고, DB·웹·메시지큐 같은 외부를 바깥에 두되, 의존성이 항상 바깥에서 안쪽으로만 향하게 한다. 헥사고날(포트 & 어댑터)과 클린 아키텍처는 같은 아이디어의 두 표현이다.

도구는 의존성 역전 원칙(DIP)이다. 도메인이 "DB에 저장한다"는 능력을 필요로 할 때, 도메인은 인터페이스(port)를 자기 쪽에 정의하고, infrastructure가 그 인터페이스를 구현(adapter)한다. 그러면 컴파일 의존성은 infrastructure → domain 방향이 되어, 런타임 호출 방향(domain이 저장을 호출)과 반대가 된다. 화살표가 뒤집혔기 때문에 "역전"이다. 도메인은 자신을 누가 구현하는지 모른 채 정책만 표현한다.

PORTS & ADAPTERS — 의존성은 항상 안쪽을 향한다 [web adapter] ──▶ ( inbound port ) ──▶ [ DOMAIN ]
│ 정책·규칙
[ DOMAIN ] ──▶ ( outbound port ) ◀── [db adapter]
interface 소유=domain 구현=infra ← DIP

RULE 안쪽 = 정책(policy) · 바깥 = 세부(detail) · 안쪽은 바깥을 모른다

얻는 것: 도메인을 DB·프레임워크 없이 순수 단위테스트로 검증할 수 있고, 어댑터를 갈아끼우면 PostgreSQL을 인메모리로, REST를 gRPC로 바꿔도 정책은 그대로다. 비용도 명확하다 — 인터페이스와 구현이 분리되면서 간접 계층이 늘고, 작은 CRUD에는 과한 의식(ceremony)이 된다. 트레이드오프는 "이 도메인의 규칙이 오래 살아남고, 외부 기술이 바뀔 가능성이 높은가"로 판단한다(→ ARCH-09).

실무 번역 — Spring에서

domain 패키지에 OrderRepository 인터페이스를 두고, infrastructure 패키지의 JpaOrderRepository가 그것을 구현한다. service는 인터페이스 타입에만 의존하고, 스프링이 구현체를 주입한다. 이미 익숙한 DI(의존성 주입)가 DIP의 실행 메커니즘이다 — 새 개념이 아니라, 인터페이스를 누가 소유하느냐(도메인 쪽)를 의식적으로 정하는 규율의 문제다.

ARCH-04

DDD 기초

Domain-Driven Design은 "복잡한 비즈니스를 코드로 옮길 때, 도메인 전문가의 언어를 그대로 코드의 언어로 삼자"는 접근이다. 두 층위가 있다 — 객체를 어떻게 모델링하는가(전술적 패턴)와, 시스템을 어떻게 나누는가(전략적 설계).

전술 패턴정의 / 식별 기준
엔티티(Entity)고유 식별자(ID)로 구분되는 객체. 속성이 바뀌어도 같은 것 (예: 주문)
값 객체(Value Object)식별자 없이 값으로만 동등성 판단. 불변(immutable) (예: 금액, 주소)
애그리거트(Aggregate)함께 변경되어야 하는 객체 묶음. 루트를 통해서만 접근·일관성 경계
리포지토리(Repository)애그리거트의 영속성 추상화. 컬렉션처럼 다루는 인터페이스

가장 자주 오해되는 것이 애그리거트다. 애그리거트는 "트랜잭션 일관성의 경계"다 — 한 번의 트랜잭션에서 불변식이 깨지지 않아야 하는 객체들의 묶음. 예컨대 Order와 그 OrderLine들은 "주문 총액 = 라인 합계"라는 불변식을 공유하므로 한 애그리거트다. 외부는 OrderLine을 직접 만지지 못하고 루트인 Order를 통해서만 바꾼다. 애그리거트는 작게 유지하는 것이 핵심 — 너무 크면 동시성 충돌과 로딩 비용이 커진다.

STRATEGIC vs TACTICAL — 두 층위 [ STRATEGIC ] 시스템을 어떻게 나누는가
  Bounded Context  ·  Ubiquitous Language  ·  Context Map
│ 경계 안에서만
[ TACTICAL ] 경계 안의 객체를 어떻게 모델링하는가
  Entity  ·  Value Object  ·  Aggregate  ·  Repository

전략적 설계의 핵심은 바운디드 컨텍스트(Bounded Context)다. 같은 단어가 맥락마다 다른 의미를 가진다 — "고객"이 영업 컨텍스트에서는 잠재 계약자이고, 배송 컨텍스트에서는 주소와 수령인이다. 하나의 거대 모델로 통합하려 하면 누구도 정확하지 않은 모델이 된다. 대신 각 컨텍스트가 자기 모델과 유비쿼터스 언어(그 경계 안에서 일관되게 쓰는 용어)를 갖고, 경계 사이는 명시적으로 번역한다. 이 경계가 나중에 서비스 분리의 후보가 된다(→ ARCH-06).

DDD를 시작하는 곳

전술 패턴(엔티티·값객체)부터 손대고 싶겠지만, DDD의 진짜 레버리지는 전략적 설계에 있다. 패턴만 적용하고 경계를 안 그으면 "값 객체를 쓰는 빈혈 모델"이 될 뿐이다. 먼저 도메인 전문가와 대화하며 바운디드 컨텍스트와 언어를 합의하라 — 코드 패턴은 그 경계가 정해진 다음의 구현 디테일이다. DDD는 기술이 아니라 대화의 규율이다.

ARCH-05

모놀리스 vs 마이크로서비스

MSA는 유행이 아니라 비용을 옮기는 결정이다. 모놀리스의 코드 복잡도를 분산 시스템의 운영 복잡도로 바꾼다. 그 거래가 이득인 경우는 생각보다 드물다 — 기본값은 모듈러 모놀리스다.

관점모듈러 모놀리스마이크로서비스
경계 위반컴파일러가 못 막음(규율 필요)네트워크가 강제
트랜잭션로컬 ACID 그대로분산 → 결과적 일관성·saga
배포한 단위로 단순독립 배포 가능, 그러나 오케스트레이션 필요
장애한 프로세스 안부분 장애·네트워크 단절을 항상 고려
확장전체를 함께병목 서비스만 선택적으로

분산이 도입하는 비용은 셋이다. 네트워크: 함수 호출이 네트워크 호출이 되면 지연·실패가 일상이 된다(타임아웃·재시도가 필수 — → ARCH-08). 데이터 일관성: 서비스마다 DB가 분리되면 하나의 트랜잭션으로 묶을 수 없어, 결과적 일관성과 보상 트랜잭션(saga)으로 풀어야 한다. 운영 복잡도: 분산 추적, 서비스 디스커버리, 버전 호환성, CI/CD 파이프라인이 서비스 수만큼 곱해진다.

DECISION — MSA로 가야 하는 신호 기본값: Modular Monolith (경계는 패키지/모듈로, 한 프로세스)
─────────────────────────────────────────────
MSA 정당화 조건 (대부분 충족해야)
  □ 팀이 여럿 & 독립 배포가 조직 병목
  □ 특정 부분만 확장 부하가 극단적으로 다름
  □ 도메인 경계가 이미 안정적으로 검증됨
→ 셋 다 아니면 분산 비용만 떠안는다
분산 모놀리스 — 최악의 조합

경계가 불안정한 채로 서비스를 쪼개면 분산 모놀리스가 된다 — 서비스들이 동기 호출로 촘촘히 엮여, 하나를 배포하려면 다른 것도 함께 배포해야 하고, 한 곳의 장애가 전체로 번진다. 모놀리스의 강결합에 분산의 운영 비용까지 더한, 두 세계의 단점만 가진 구조다. 경계가 자주 바뀌는 초기 도메인일수록 모놀리스 안에서 경계를 먼저 단단히 만든 뒤 쪼개라. "쪼개기 쉬운 모놀리스"가 "합치기 어려운 MSA"보다 낫다.

ARCH-06

서비스 경계와 통신

서비스를 어디서 쪼갤지는 기술이 아니라 비즈니스 능력(business capability)으로 정한다. "주문" "결제" "배송"처럼 독립적으로 변하는 단위가 경계의 후보다 — 바운디드 컨텍스트(→ ARCH-04)가 그대로 서비스 경계로 이어지는 경우가 많다.

잘못된 경계의 신호는 데이터 중심 분할이다. "User 서비스" "Product 서비스"처럼 명사(테이블)로 쪼개면, 하나의 유스케이스가 여러 서비스를 동기로 들락거리며 강결합과 분산 트랜잭션을 만든다. 좋은 경계는 동사(능력)로 나뉘어, 한 요청이 가능한 한 한 서비스 안에서 완결된다.

통신 방식성격 / 적합한 상황
동기 REST요청-응답 즉시 필요·범용. 단순하나 호출자가 응답까지 대기(결합)
동기 gRPC서비스 간 저지연·강타입 계약(protobuf). 내부 통신에 효율적
비동기 이벤트"일이 일어났다"를 발행. 발신자가 수신자를 모름 → 느슨한 결합
주문 서비스 ──동기 호출──▶ 결제 서비스 ⏳ 응답 대기 (강결합·장애 전파)
주문 서비스 ──이벤트 발행──▶ [ 메시지 큐 ] ──▶ 결제 · 배송 · 알림

동기 호출은 직관적이지만 강결합을 만든다 — 호출 대상이 느리거나 죽으면 호출자도 함께 막힌다(캐스케이딩 장애 — → ARCH-08). 비동기 이벤트는 발신자와 수신자를 시간·지식 양면에서 분리한다. "주문이 생성됨" 이벤트를 발행하면, 그것을 누가 구독하는지 주문 서비스는 알 필요가 없다 — 결제·배송·알림이 각자 반응한다. 대신 결과적 일관성과 메시지 순서·중복 처리(idempotency)라는 새 복잡도가 붙는다(→ 비동기·메시지 큐 핸드북).

동기 호출 사슬을 경계하라

A → B → C → D로 동기 호출이 이어지면, 전체 지연은 합이 되고 가용성은 곱이 된다(각 99.9%면 4단계는 약 99.6%). 한 곳의 느려짐이 사슬 전체를 막는다. "이 호출이 정말 지금 당장 응답이 필요한가, 아니면 나중에 처리해도 되는가"를 물어라. 후자라면 이벤트로 떼어내 결합과 장애 전파를 동시에 끊는다.

ARCH-07

CQRS와 이벤트 소싱

두 패턴은 자주 함께 언급되지만 별개다. CQRS는 읽기 모델과 쓰기 모델을 분리하는 것, 이벤트 소싱은 상태 대신 상태 변화(이벤트)를 저장하는 것. 둘 다 강력하지만, 둘 다 대부분의 경우 과하다.

CQRS(Command Query Responsibility Segregation)의 동기는 단순하다 — 쓰기는 정규화된 도메인 모델로 일관성을 지키고 싶고, 읽기는 화면에 맞춰 비정규화된 형태로 빠르게 가져오고 싶다. 하나의 모델로 둘 다 만족시키려면 타협이 생긴다. 그래서 명령(상태를 바꾸는 것)과 조회(상태를 읽는 것)를 다른 모델로 분리한다. 가벼운 버전은 같은 DB에서 읽기 전용 쿼리 모델만 따로 두는 것이고, 무거운 버전은 읽기 전용 DB를 별도로 운영하며 이벤트로 동기화한다.

CQRS — 명령과 조회의 분리 Command (쓰기) ──▶ [ Write Model ] ──▶ 정규화 DB · 일관성
│ 이벤트/동기화
Query (읽기)   ◀── [ Read Model ]  ◀── 비정규화 뷰 · 빠른 조회

EVENT SOURCING — 상태가 아니라 변화를 저장 전통: balance = 100 (현재 값만)
ES : +100, -30, +50, -20 → 재생(replay)하면 100. 전 이력이 진실

이벤트 소싱은 "잔액 100"을 저장하는 대신 "입금 100, 출금 30, ..."이라는 이벤트의 연속을 저장하고, 현재 상태는 이벤트를 재생해서 계산한다. 얻는 것은 완전한 감사 이력(audit trail), 과거 어느 시점으로든 복원, "왜 이 상태가 됐는가"의 추적성이다. 회계·금융·규제 도메인에서 강력하다. 비용은 가파르다 — 이벤트 스키마 버저닝, 스냅샷, 재생 성능, 결과적 일관성, 그리고 팀 전체의 사고방식 전환.

언제 과한가 — 거의 항상

CQRS와 이벤트 소싱은 특정 통증이 실제로 있을 때의 약이다. 읽기/쓰기 부하나 모델 형태가 극단적으로 다르거나(CQRS), 모든 상태 변화의 이력이 비즈니스 요구사항일 때(이벤트 소싱). 그 통증 없이 "확장성 좋다더라"로 도입하면, 단순 CRUD에 결과적 일관성과 디버깅 지옥을 떠안는다. 기본은 단순 CRUD + 단일 모델. 통증이 측정될 때 그 부분에만 국소적으로 적용하라(→ ARCH-09 YAGNI).

ARCH-08

회복탄력성 패턴

분산 시스템에서 실패는 예외가 아니라 상수다. 네트워크는 끊기고, 의존 서비스는 느려지고, 인스턴스는 죽는다. 회복탄력성 패턴의 목표는 실패를 없애는 게 아니라, 한 곳의 실패가 전체로 번지지 않게 가두는 것이다.

패턴막는 실패 모드
타임아웃(Timeout)무한 대기. 응답 없는 호출에 매달려 자원을 소진하는 것
재시도 + 지수 백오프일시적 장애. 단, 간격을 늘려야 함(고정 재시도는 부하 폭증)
서킷 브레이커죽은 서비스에 계속 호출. 빠르게 실패해 회복 시간을 줌
격벽(Bulkhead)한 의존성의 포화가 전체 자원풀을 잠식하는 것
CIRCUIT BREAKER — 세 상태 CLOSED (정상 통과) ──실패율 임계 초과──▶ OPEN (즉시 차단·빠른 실패)
▲ │ 일정 시간 후
└──성공── HALF-OPEN (시험 호출 몇 개만 통과) ◀──┘
└──또 실패하면 다시 OPEN

서킷 브레이커의 발상은 전기 차단기와 같다. 호출 실패율이 임계를 넘으면 회로를 열어(OPEN) 그 의존성으로의 호출을 즉시 막는다 — 죽은 서비스를 계속 두드리며 자원을 낭비하고 타임아웃을 쌓는 대신, 곧바로 실패를 반환하고 폴백(fallback)으로 넘긴다. 일정 시간 후 반열림(HALF-OPEN)으로 시험 호출을 보내, 회복됐으면 닫고 아니면 다시 연다. 격벽은 배의 격벽처럼, 의존성마다 별도의 스레드풀/커넥션풀을 할당해 한 곳의 포화가 다른 호출의 자원까지 삼키지 못하게 한다.

캐스케이딩 장애 — 가장 흔한 전면 장애

흔한 시나리오: 하위 서비스가 느려진다 → 호출자의 스레드가 응답을 기다리며 쌓인다 → 스레드풀이 고갈된다 → 호출자도 응답 불능이 된다 → 그 호출자의 호출자도... 한 서비스의 지연이 시스템 전체를 무너뜨린다. 막는 조합은 타임아웃(매달리지 않기) + 서킷 브레이커(죽은 곳 차단) + 격벽(자원 격리)이다. 재시도는 신중하게 — 지수 백오프 없는 재시도는 죽어가는 서비스에 부하를 더해 장애를 가속한다(→ 비동기·메시지 큐 핸드북의 재시도·DLQ 설계와 함께 볼 것).

ARCH-09

패턴 선택 기준

이 핸드북의 모든 패턴은 비용을 동반한 도구다. 중요한 것은 패턴을 아는 것이 아니라, 지금 이 패턴이 필요한지를 판단하고 그 판단을 설명할 수 있는 것이다.

지배 원칙은 YAGNI(You Aren't Gonna Need It)다. "나중에 필요할까 봐" 미리 넣은 추상화의 대부분은 끝내 쓰이지 않고, 대신 매일의 인지 부하로 청구된다. 추상화는 공짜가 아니다 — 모든 간접 계층은 코드를 읽는 사람이 따라가야 할 점프 한 번을 더한다. 복잡도 예산이라는 관점이 유용하다: 팀이 감당할 수 있는 복잡도의 총량은 유한하고, 불필요한 곳에 쓰면 정작 필요한 곳에 쓸 예산이 없다.

상황권장피할 것
초기 제품·불확실한 도메인모듈러 모놀리스 + CRUDMSA, 이벤트 소싱
규칙이 복잡·장수하는 핵심 도메인헥사고날 + DDD 전술 패턴핵심까지 빈혈 모델로 방치
읽기/쓰기 부하·형태가 극단적으로 다름그 부분에만 CQRS전 시스템에 일괄 적용
분산 호출이 이미 존재타임아웃·서킷 브레이커회복탄력성 없는 동기 사슬
지금 통증이 있나? ─아니오─▶ 가장 단순한 것 유지 ／ 예 ▼ 통증을 측정 ──▶ 그 부분에만 패턴 적용
설계 결정을 설명하는 법

좋은 아키텍트는 "이게 best practice니까"라고 말하지 않는다. "우리 제약은 X(팀 규모·도메인 안정성·부하 특성)이고, 그래서 Y를 택했고, 그 대가로 Z를 포기했다"고 말한다. 패턴 이름을 외우는 것보다, 한 패턴을 도입할 때 무엇을 얻고 무엇을 잃는지를 한 문단으로 쓸 수 있어야 한다. 결정을 ADR(Architecture Decision Record)로 짧게 남기면, 미래의 너와 팀이 "왜 이렇게 했지"에 답할 수 있다(→ 백엔드 개요 핸드북과 함께 볼 것).

ARCH-10

용어집

반복해서 등장하는 핵심 용어를 한자리에. 정의보다 "왜 중요한가"에 무게를 둔다.

결합도coupling
한 모듈 변경이 다른 모듈 변경을 강제하는 정도. 낮을수록 변경이 국소적으로 멈춘다. 변경 비용의 절반을 지배한다.
응집도cohesion
한 모듈 안 요소들이 같은 목적을 향하는 정도. 높을수록 "함께 바뀌는 것이 한곳에" 모여 이해·변경이 쉽다.
계층형 아키텍처layered
코드를 역할별 층(presentation/application/domain/infra)으로 쌓고 위→아래 단방향 의존. 흔하지만 계층 누수에 취약.
계층 누수layer leakage
하위 계층의 세부(예: JPA)가 상위 정책 계층으로 새어드는 것. 도메인이 DB를 알게 되면 규칙을 DB 없이 테스트할 수 없게 된다.
헥사고날ports & adapters
도메인을 안쪽에, 외부(DB·웹)를 바깥에 두고 의존성이 항상 안을 향하게 하는 구조. 클린 아키텍처와 같은 아이디어.
의존성 역전DIP
인터페이스를 도메인이 소유하고 인프라가 구현해, 컴파일 의존을 "인프라→도메인"으로 뒤집는 원칙. 헥사고날의 동력.
DDDdomain-driven design
도메인 전문가의 언어를 코드의 언어로 삼는 설계. 전술(객체 모델링)과 전략(경계 나누기) 두 층위로 구성.
애그리거트aggregate
함께 변경되어야 하는 객체 묶음 = 트랜잭션 일관성의 경계. 루트를 통해서만 접근. 작게 유지하는 것이 핵심.
값 객체value object
식별자 없이 값으로만 동등성을 판단하는 불변 객체(금액·주소 등). 엔티티와 달리 ID로 구분하지 않는다.
바운디드 컨텍스트bounded context
하나의 모델과 용어가 일관되게 통하는 경계. 같은 단어가 맥락마다 다른 의미를 가짐을 인정하고 경계를 긋는다. 서비스 분리의 후보.
유비쿼터스 언어ubiquitous language
한 바운디드 컨텍스트 안에서 코드·대화·문서가 공유하는 용어. 번역 손실을 없애 도메인과 코드의 간극을 좁힌다.
모듈러 모놀리스modular monolith
한 프로세스 안에서 모듈로 경계를 긋는 구조. 분산 비용 없이 응집·결합을 관리. 대부분 프로젝트의 합리적 기본값.
MSAmicroservices
독립 배포 가능한 작은 서비스로 시스템을 분할. 코드 복잡도를 운영·분산 복잡도로 교환. 조직·확장 필요가 명확할 때만.
분산 모놀리스distributed monolith
서비스로 쪼갰으나 동기 호출로 촘촘히 엮여 함께 배포·함께 장애가 나는 안티패턴. 두 세계의 단점만 가짐.
CQRScommand query separation
상태를 바꾸는 명령 모델과 읽는 조회 모델을 분리. 읽기/쓰기 요구가 극단적으로 다를 때 유효, 그 외엔 과함.
이벤트 소싱event sourcing
현재 상태 대신 상태 변화(이벤트)를 저장하고 재생으로 상태를 계산. 완전한 감사 이력을 얻지만 복잡도가 가파르다.
결과적 일관성eventual consistency
즉시 일치하지 않지만 시간이 지나면 수렴하는 일관성. 분산·비동기의 대가. saga·idempotency 설계가 따라온다.
서킷 브레이커circuit breaker
실패율이 임계를 넘으면 호출을 차단(OPEN)해 죽은 의존성을 두드리지 않게 한다. 캐스케이딩 장애의 1차 방어선.
격벽bulkhead
의존성마다 자원풀(스레드·커넥션)을 격리해, 한 곳의 포화가 전체 자원을 삼키지 못하게 가두는 패턴.
YAGNIyou aren't gonna need it
"나중에 필요할까 봐" 미리 만든 추상화의 대부분은 안 쓰이고 인지 부하만 청구된다. 통증이 측정될 때 추가하라.
이 핸드북을 덮으며

패턴 이름 20개를 외웠다고 아키텍트가 되지 않는다. 한 가지만 남기자 — 모든 구조는 무언가를 쉽게 만드는 대가로 다른 무언가를 어렵게 만든다. 좋은 결정은 "지금 우리 제약에서 무엇을 쉽게 만들어야 하는가"에서 나온다. 단순함을 기본값으로 두고, 통증이 측정될 때 그 부분에만 패턴을 쓰고, 그 트레이드오프를 한 문단으로 설명하라(→ 백엔드 개요 · 비동기·메시지 큐 · DB 핸드북과 함께 볼 것).

ARCH-11

경계 설계와 점진적 전환

아키텍처의 핵심은 유행하는 구조를 고르는 것이 아니라 변경의 경계를 찾고, 기존 시스템을 끊지 않으면서 개선하는 것입니다.

주제실무 기준위험 신호
bounded context같은 단어가 같은 의미를 갖는 도메인 경계를 찾음주문, 결제, 정산이 하나의 Order 모델을 서로 다르게 해석
anti-corruption layer외부/레거시 모델이 내부 도메인을 오염시키지 않게 번역 계층을 둠외부 API 필드명이 핵심 도메인 객체에 그대로 침투
modular monolith배포는 하나지만 모듈 경계와 의존 방향을 강제서비스 분리 없이도 내부 결합이 이미 마이크로서비스보다 복잡
strangler fig레거시 기능을 라우팅 단위로 점진 대체한 번에 재작성하다가 비즈니스 기능이 장기간 동결
Case study

정산 로직을 주문 서비스에서 바로 분리하려 하면 데이터 소유권과 배포 순서가 얽힙니다. 먼저 정산 읽기 모델을 만들고, 주문 이벤트를 outbox로 발행하고, anti-corruption layer에서 레거시 상태를 새 도메인 언어로 변환한 뒤, 트래픽을 strangler fig 방식으로 일부 기능부터 넘기는 편이 안전합니다.

SEC-01

인증 vs 인가

보안 사고의 큰 갈래 하나는 이 둘을 뭉뚱그리는 데서 시작합니다. authentication은 "너 누구냐"를 확인하는 것, authorization은 "이걸 할 권한이 있냐"를 판정하는 것 — 순서도 책임도 다릅니다.

로그인에 성공했다고 해서 모든 자원에 접근할 수 있는 게 아닙니다. authentication은 신원(identity)을 세우고, authorization은 그 신원이 특정 자원·작업에 대해 권한이 있는지를 매번 따집니다. 둘은 별개의 단계이며, 인증이 끝났다고 인가를 건너뛰면 그게 바로 사고입니다. 흔한 약어로 authentication은 authN, authorization은 authZ로 적습니다.

구분authN (인증)authZ (인가)
질문"너 누구냐?""이걸 할 권한 있냐?"
판정 대상신원(identity) — 자격증명 검증권한(permission) — 자원·작업별
시점보통 한 번 (로그인)요청마다 반복
실패 시 응답401 Unauthorized403 Forbidden
예비밀번호·토큰·OTP 확인"이 주문은 네 것이냐" 검사
HTTP 상태코드의 함정

이름이 헷갈립니다 — 401 Unauthorized는 사실 인증 실패(누군지 모름·자격증명 없음)이고, 403 Forbidden이 인가 실패(누군진 알지만 권한 없음)입니다. 약어 표기로는 401=authN 실패, 403=authZ 실패로 기억하세요. 신원은 확인됐는데 권한이 없으면 401이 아니라 403을 돌려줘야 클라이언트가 "다시 로그인"이 아니라 "권한 없음"으로 옳게 반응합니다(→ 백엔드 개요).

SEC-02

세션 기반 인증

전통적이고 여전히 견고한 방식입니다. 서버가 상태(session)를 쥐고, 브라우저엔 그 세션을 가리키는 식별자만 쿠키로 건넵니다. 핵심 가치는 "서버가 통제권을 쥔다"는 점.

로그인에 성공하면 서버는 세션 객체(사용자 id, 만료 시각 등)를 세션 저장소에 만들고, 임의의 추측 불가능한 session ID를 발급해 Set-Cookie로 내려보냅니다. 이후 브라우저는 같은 도메인 요청마다 그 쿠키를 자동으로 실어 보내고, 서버는 ID로 저장소를 조회해 누군지 압니다. 민감한 데이터는 전부 서버에 남고, 클라이언트는 의미 없는 식별자만 들고 있습니다.

SESSION FLOW [로그인] client ──▶ server : id/pw 검증
server ──▶ store : 세션 생성(sid → user)
server ──▶ client : Set-Cookie: sid=...; HttpOnly; Secure; SameSite
[이후 요청] client ──▶ server : Cookie: sid=... (자동 첨부)
server ──▶ store : sid 조회 → 신원 복원

가장 큰 장점은 즉시 무효화입니다. "로그아웃", "모든 기기에서 로그아웃", "탈취된 세션 강제 종료"가 저장소에서 레코드 하나 지우는 것으로 끝납니다. 단점은 확장성 — 서버가 상태를 쥐므로, 인스턴스가 여러 대면 어느 서버로 요청이 가든 같은 세션을 봐야 합니다. 그래서 세션을 메모리가 아니라 Redis 같은 공유 저장소에 두는 게 정석입니다.

쿠키 속성역할 / 왜 필요한가
HttpOnlyJavaScript에서 쿠키 접근 차단 → XSS로 세션 탈취 방어(→ SEC-09)
SecureHTTPS에서만 전송 → 평문 네트워크 노출 방지
SameSite크로스 사이트 요청에 쿠키 첨부 제한 → CSRF 완화(Lax/Strict)
Max-Age / Expires만료. 없으면 세션 쿠키(브라우저 종료 시 소멸)
세션과 CSRF의 관계

세션 쿠키는 브라우저가 자동으로 첨부하기 때문에, 공격자가 사용자를 속여 악성 폼을 제출하게 하면 쿠키가 딸려 가 정상 요청처럼 보입니다 — 이게 CSRF입니다(→ SEC-09). 그래서 세션 방식은 거의 항상 CSRF 방어(토큰 또는 SameSite 쿠키)와 함께 가야 합니다. 반대로 토큰을 Authorization 헤더로 직접 실어 보내는 방식(→ SEC-03)은 자동 첨부가 없어 CSRF에 구조적으로 덜 취약합니다.

SEC-03

토큰 기반 — JWT

JWT(JSON Web Token)는 서명된 자기 완결적 토큰입니다. 서버에 세션을 저장하지 않고, 토큰 자체에 신원 정보를 담아 서명으로 위변조를 막습니다. 핵심 트레이드오프 한 줄: stateless를 얻는 대신, 만료 전 무효화를 잃는다.

JWT는 점(.)으로 구분된 세 부분 header.payload.signature입니다. header와 payload는 단순 Base64URL 인코딩일 뿐 — 암호화가 아닙니다. 누구나 디코드해 내용을 읽을 수 있으므로 비밀번호 같은 민감 정보를 payload(claims)에 담으면 안 됩니다. 마지막 signature는 서버의 비밀키로 header+payload를 서명한 값이고, 서버는 받은 토큰을 같은 키로 재서명해 일치하는지로 위변조 여부를 검증합니다.

JWT STRUCTURE header.payload.signature
{alg, typ} | {sub, exp, role...} | HMAC/RSA(...)

VERIFY sig == sign(header + payload, secret) ? 신뢰 : 거부
※ payload는 Base64일 뿐 — 누구나 읽을 수 있음(암호화 아님)

stateless라서 서버는 토큰만 검증하면 되고, 저장소 조회가 없어 수평 확장에 유리합니다. 그런데 바로 그 점이 약점입니다 — 서버가 토큰을 기억하지 않으니, 만료 전에 특정 토큰을 무효화하기가 어렵습니다. 탈취된 토큰은 만료될 때까지 유효합니다. 그래서 실무 정석은 두 종류로 나눕니다.

토큰수명역할 / 저장 위치
access token짧게 (분 단위)API 호출에 첨부. 짧아서 탈취돼도 피해 창이 작음
refresh token길게 (일~주)access 재발급용. 서버가 저장·추적 → 무효화 가능

access는 짧게 만들어 탈취 위험을 줄이고, 만료되면 refresh token으로 새 access를 받습니다. refresh token은 보통 서버가 저장소에 보관해 폐기(rotation·revocation)할 수 있게 합니다 — 결국 무효화가 필요한 부분만 stateful로 되돌리는 절충입니다. 저장 위치도 트레이드오프입니다.

저장 위치장점위험
localStorage다루기 쉬움, 헤더로 전송XSS로 읽힘 → 탈취(→ SEC-09)
HttpOnly 쿠키JS 접근 불가 → XSS에 강함자동 첨부 → CSRF 대비 필요
치명적 실수 — alg:none과 키 혼동

JWT 검증에서 반복되는 사고: ① alg: none을 받아들이는 라이브러리 설정 — 서명 없는 토큰을 통과시켜 누구나 위조 가능. ② alg를 토큰이 시키는 대로 따르다 RS256(공개키 검증)을 HS256(대칭키)로 바꿔치기당해, 공개된 키를 비밀키로 오용하는 공격. 검증 알고리즘은 서버가 고정하고 토큰의 alg를 신뢰하지 마세요. 그리고 payload는 평문이니 거기에 비밀을 담지 마세요.

SEC-04

OAuth2 / OIDC

"구글로 로그인"의 뒤편입니다. 둘은 다른 일을 합니다 — OAuth2는 인가 위임(delegated authorization) 프로토콜이고, OIDC는 그 위에 인증(authentication) 층을 얹은 것입니다. 이걸 섞으면 설계가 어긋납니다.

OAuth2의 본래 목적은 "비밀번호를 넘기지 않고 제3자 앱에게 내 자원의 일부 접근을 허용"하는 것입니다 — 즉 인가 위임이지 "누구인지 알려주기"가 아닙니다. 그래서 "OAuth2로 로그인"은 엄밀히는 오용이고, 로그인(=신원 확인)이 필요하면 OAuth2 위에 표준 신원 계층을 더한 OIDC(OpenID Connect)를 씁니다. OIDC는 사용자의 신원을 담은 ID Token(JWT 형식)을 추가로 발급합니다.

역할(role)의미
Resource Owner자원의 주인 = 보통 사용자 본인
Client접근을 원하는 앱 (우리 서비스)
Authorization Server동의를 받고 토큰을 발급 (구글·카카오 등)
Resource Server실제 자원을 가진 API 서버

가장 권장되는 플로우는 Authorization Code + PKCE입니다. 사용자를 authorization server로 보내 로그인·동의를 받으면, 서버는 짧은 일회성 authorization code를 redirect로 돌려줍니다. 클라이언트는 그 code를 다시 서버 백채널로 보내 access token(과 OIDC면 ID token)으로 교환합니다. 토큰이 브라우저 URL에 직접 노출되지 않는 게 핵심입니다.

4. client → auth server : code + code_verifier (백채널 교환)
5. auth server → client : access token (+ id token)
※ PKCE: code 가로채기 공격 방어 — verifier를 아는 자만 교환 성공

access token으로 "로그인 됐다"고 판단하지 마세요 — access token은 인가용이라 누구인지를 보장하지 않습니다. 로그인 여부는 OIDC의 ID token으로 판정합니다. 또 redirect URI는 화이트리스트로 정확히 등록·검증해야 합니다 — 느슨하면 토큰을 공격자 주소로 빼돌리는 open redirect 사고가 납니다. SSO(한 번 로그인으로 여러 서비스)는 이 OIDC 위에서 구성됩니다.

SEC-05

비밀번호와 해싱

비밀번호 저장의 제1원칙: "원문을 복원할 수 없게 만든다." 그래서 암호화가 아니라 해싱을 씁니다. 둘을 혼동하는 것이 평문 저장 다음으로 흔한 사고입니다.

암호화(encryption)는 키로 되돌릴 수 있는 양방향 변환입니다. 해싱(hashing)은 단방향 — 같은 입력은 같은 출력을 내지만, 출력에서 입력을 복원할 수 없습니다. 비밀번호는 우리조차 원문을 알 필요가 없으니(로그인 시 입력값을 같은 방식으로 해싱해 비교만 하면 됨) 해싱이 맞습니다. DB가 털려도 원문이 드러나지 않게 하는 게 목적입니다.

그런데 아무 해시나 되는 게 아닙니다. MD5·SHA-1·SHA-256은 빠르도록 설계된 함수라 비밀번호용으로는 부적합합니다 — 빠르다는 건 공격자가 초당 수십억 개를 대입(brute force)할 수 있다는 뜻이기 때문입니다. 비밀번호엔 일부러 느린 전용 함수를 씁니다.

요소역할 / 왜
salt사용자마다 다른 랜덤값을 비밀번호에 더해 해싱 → 같은 비밀번호도 다른 해시. rainbow table·동일 비번 노출 방어
work factor해싱 반복 비용(cost). 하드웨어가 빨라지면 올려서 brute force를 계속 비싸게 유지
bcrypt검증된 표준. salt·work factor 내장. 무난한 기본 선택
scrypt메모리도 많이 쓰게 해 GPU 병렬 공격에 강함
argon2최신 권장(argon2id). 시간·메모리·병렬도 조절 가능
절대 하지 말 것

① 평문 저장 — 논외. ② MD5/SHA 단독 해싱 — 빠른 해시라 무용지물. ③ salt 없이 해싱 — 같은 비밀번호가 같은 해시가 되어 rainbow table에 뚫림. ④ 직접 구현 — 암호 함수를 손으로 짜지 말고 검증된 라이브러리(bcrypt/argon2)를 쓰세요. ⑤ 로그에 비밀번호를 남기지 마세요(→ SEC-10). Spring Security라면 PasswordEncoder(예: BCryptPasswordEncoder)를 표준으로 씁니다.

SEC-06

인가 모델

인증을 통과한 다음의 세계입니다. "이 사용자가 이 자원에 이 작업을 해도 되는가"를 일관되게 판정하는 구조 — 그리고 그 판정을 빠뜨리는 것이 가장 흔한 인가 사고입니다.

가장 널리 쓰이는 두 모델입니다. RBAC(Role-Based)는 사용자에게 역할(role)을 주고 역할에 권한을 묶습니다 — "admin은 삭제 가능, member는 읽기만". 단순하고 관리가 쉽습니다. ABAC(Attribute-Based)는 사용자·자원·환경의 속성으로 규칙을 만듭니다 — "같은 부서이고 근무시간 내일 때만 접근". 더 세밀하지만 복잡합니다.

모델판정 기준장점 / 한계

ABAC속성(attribute)세밀·유연 / 규칙이 복잡해지고 디버깅 어려움
ownership 검사자원의 소유자 == 요청자?RBAC/ABAC와 함께 — 역할만으론 못 막는 사고 방어

관통하는 원칙은 최소 권한(least privilege)입니다 — 기본은 거부, 필요한 만큼만 허용. 그리고 역할 검사만으로 끝내면 안 됩니다. "member 역할이면 주문 조회 가능"까지만 보고 그 주문이 이 사용자의 것인지(자원 소유권)를 확인하지 않으면, 남의 주문 id로 바꿔치기해 조회·수정할 수 있습니다.

IDOR — 인가 누락의 대표 사고

IDOR(Insecure Direct Object Reference)는 /orders/1024의 id를 1025로 바꾸기만 해도 남의 자원이 열리는 취약점입니다. 원인은 단 하나 — 요청한 자원이 이 사용자 소유인지 확인하는 인가 검사를 빠뜨린 것. 인증됐다고, 역할이 맞다고 끝이 아닙니다. 모든 자원 접근에서 소유권·권한을 명시적으로 검사하세요. "id는 추측 못 할 거야"(추측 불가 식별자)는 방어가 아니라 가림막일 뿐입니다(→ SEC-07 Broken Access Control).

SEC-08

Injection과 입력 검증

Injection의 본질은 한 문장입니다 — "데이터로 와야 할 것이 코드로 실행된다." 사용자 입력을 코드·쿼리 문자열에 그대로 이어붙이는 순간, 입력이 명령이 됩니다.

대표는 SQL Injection입니다. 입력을 쿼리 문자열에 연결하면, ' OR '1'='1 같은 입력이 쿼리의 구조를 바꿔 인증을 우회하거나 테이블을 통째로 빼냅니다(DROP TABLE까지). 근본 해법은 입력을 거르는 게 아니라 데이터와 코드를 애초에 분리하는 것 — parameterized query / prepared statement입니다. 자리표시자(? 또는 named param)에 값을 바인딩하면, DB가 그 값을 항상 데이터로만 다루고 절대 구문으로 해석하지 않습니다.

왜 parameterized query인가 // 취약 — 입력이 구문이 됨
"SELECT * FROM users WHERE id = '" + input + "'"
input = " ' OR '1'='1 " → 항상 참, 전체 노출

// 안전 — 값은 끝까지 데이터
"SELECT * FROM users WHERE id = ?" + bind(input)

JPA를 쓴다고 안전한 게 아니다

JPA/Hibernate의 @Query와 메서드 쿼리는 파라미터를 바인딩하므로 기본적으로 SQLi에 안전합니다. 하지만 JPQL·native query를 문자열로 직접 조립하면("... where name = '" + name + "'") 그대로 뚫립니다. 항상 :param 바인딩이나 setParameter()를 쓰세요. SQL만이 아니라 OS 명령 실행에 입력을 넘기면 command injection도 같은 원리로 발생합니다 — 가능하면 셸을 거치지 말고 인자 배열로 직접 호출하세요.

SEC-09

XSS와 CSRF

이름이 비슷해 자주 헷갈리지만 정반대 방향의 공격입니다. XSS는 "악성 스크립트를 피해자 브라우저에서 실행"시키고, CSRF는 "피해자의 인증을 빌려 원치 않는 요청을 보내게" 합니다. 방어법도 다릅니다.

9.1XSS — Cross-Site Scripting

공격자가 주입한 JavaScript가 다른 사용자의 브라우저에서 실행되는 것입니다. 실행되면 세션 쿠키 탈취(→ HttpOnly로 완화), 화면 변조, 키 입력 가로채기까지 가능합니다. 근본 원인은 신뢰할 수 없는 데이터를 HTML/JS 컨텍스트에 이스케이프 없이 출력한 것입니다.

유형경로
Stored(저장형)악성 스크립트가 DB에 저장돼 다른 사용자에게 매번 전달 (댓글 등). 영향 큼
Reflected(반사형)요청 파라미터가 응답에 즉시 반사 (검색어·에러 메시지). 악성 링크 유도
DOM-based서버 거치지 않고 클라이언트 JS가 입력을 위험하게 DOM에 삽입

방어의 핵심은 출력 이스케이프(context-aware escaping)입니다 — 데이터가 HTML 본문/속성/JS/URL 중 어디로 나가는지에 맞춰 인코딩합니다. React·Vue 같은 프레임워크는 기본적으로 출력을 이스케이프하므로 안전하지만, dangerouslySetInnerHTML(React)이나 v-html(Vue)로 우회하면 그 보호가 사라집니다. 보조 방어로 CSP(Content-Security-Policy) 헤더로 실행 가능한 스크립트 출처를 제한합니다.

9.2CSRF — Cross-Site Request Forgery

CSRF는 브라우저가 쿠키를 자동 첨부한다는 신뢰를 악용합니다. 로그인된 사용자가 공격자의 페이지를 열면, 그 페이지가 우리 사이트로 요청을 보내고 브라우저는 세션 쿠키를 딸려 보냅니다 — 서버는 정상 요청으로 착각합니다(→ SEC-02). 사용자는 아무것도 모릅니다.

방어의미
CSRF 토큰폼·요청마다 예측 불가 토큰을 심고 서버가 검증. 공격 페이지는 토큰을 모름
SameSite 쿠키크로스 사이트 요청에 쿠키 첨부 제한(Lax/Strict). 현대 브라우저 기본 방어
헤더 토큰 방식Authorization 헤더 토큰은 자동 첨부 안 됨 → CSRF에 구조적으로 강함
한 줄 비교 — 방향이 반대다

XSS: 공격자 코드가 피해자 브라우저에서 실행 → 방어는 출력 이스케이프 + CSP. CSRF: 피해자의 인증으로 원치 않는 요청 전송 → 방어는 CSRF 토큰 + SameSite. 주의: XSS가 뚫리면 CSRF 방어는 무력화됩니다(스크립트가 토큰을 읽어버림). 그래서 XSS 방어가 더 근본적입니다 — 입력 신뢰 금지의 원칙으로 되돌아옵니다.

SEC-10

전송·저장 보안과 secret 관리

데이터는 두 상태에서 보호받아야 합니다 — 이동 중(in transit)과 저장 중(at rest). 그리고 그 보호의 열쇠인 secret 자체를 어떻게 다루느냐가 전체 보안의 급소입니다.

전송 중: TLS(HTTPS)로 클라이언트-서버 구간을 암호화해 도청·변조(man-in-the-middle)를 막습니다. 평문 HTTP로 자격증명·토큰을 보내는 건 그대로 노출입니다. HSTS 헤더로 브라우저가 항상 HTTPS만 쓰도록 강제할 수 있습니다(→ DB 핸드북·백엔드 개요의 TLS 핸드셰이크 참고). 저장 중: 민감 데이터(개인정보 등)는 디스크에서도 암호화(encryption at rest)하고, 비밀번호는 암호화가 아니라 해싱입니다(→ SEC-05).

secret 관리가 가장 자주 사고나는 지점입니다. API 키·DB 비밀번호·서명 키를 코드에 하드코딩하거나 git에 커밋하면, 저장소 접근 권한이 곧 모든 비밀의 노출입니다. git 히스토리는 영구적이라 한 번 커밋된 secret은 나중에 지워도 과거 커밋에 남습니다 — 노출 즉시 키 폐기·재발급(rotation)이 정답입니다.

secret은 코드·git·로그에 절대 두지 않는다

SEC-13

Security Threat Modeling

위협 모델링은 "무서운 것을 상상하는 회의"가 아니라 설계가 어떤 신뢰 가정 위에 서 있는지 드러내는 작업입니다. 기능을 만들기 전에 데이터 흐름, 신뢰 경계, 권한 경계, 감사 가능성을 놓고 "어디서 공격자가 개입할 수 있는가"를 구체적으로 묻습니다.

좋은 위협 모델은 문서가 길지 않습니다. 대신 자산(asset), 행위자(actor), 진입점(entry point), 신뢰 경계(trust boundary), 실패 시 영향이 분명합니다. 로그인, 결제, 파일 업로드, 웹훅, 관리자 기능, 테넌트 데이터 접근처럼 실패 비용이 큰 기능은 티켓이 아니라 설계 단계에서 먼저 모델링해야 합니다.

13.1STRIDE로 빠르게 훑기

STRIDE는 위협을 여섯 질문으로 나누는 실무 프레임입니다. 모든 항목에 대단한 답을 쓰는 것이 목적이 아니라, "이 기능에서 빠뜨리면 사고가 되는 질문"을 빠르게 누락 없이 통과하는 것이 목적입니다.

Elevation of Privilege일반 사용자가 관리자·소유자 권한을 얻는 경로가 있는가?권한 중앙화, step-up auth, dual control, 정책 테스트
THREAT MODEL MINI LOOP 1. 무엇을 보호하나? 자산: 계정, 결제수단, 개인정보, 파일, API key
2. 누가 접근하나? anonymous, user, tenant admin, internal admin, partner service
3. 어디서 경계가 바뀌나? browser → API → worker → DB → external provider
4. 무엇을 신뢰하지 않나? client input, URL id, headers, file metadata, webhook payload
5. 실패하면 어떻게 알 수 있나? audit log, alert, anomaly query, replayable request id

13.2Trust Boundary는 선으로 그린다

신뢰 경계는 데이터가 다른 신뢰 수준으로 넘어가는 지점입니다. 브라우저에서 온 값, 모바일 앱에서 온 값, 파트너가 보낸 웹훅, 내부 큐 메시지, 관리자 콘솔 입력은 모두 별도 경계입니다. "우리 프론트엔드가 보내니까 안전하다"는 말은 경계 밖의 입력을 신뢰한다는 뜻입니다.

경계위험 질문필수 통제
Browser → API사용자가 hidden field, price, role, ownerId를 바꾸면?서버 측 권한 판정, 값 재계산, schema validation
API → DB쿼리 조건에 tenantId/ownerId가 항상 붙는가?repository 정책, RLS 또는 mandatory predicate, 테스트
API → Worker/Queue큐 메시지를 내부 신뢰로 처리해도 되는가?message schema, idempotency key, actor/context 포함
Service → External외부 응답·callback을 그대로 상태 변경에 쓰는가?서명 검증, allowlist, timeout, replay 방어
Admin Console → Core Data관리자 실수·악용이 고객 데이터에 즉시 반영되는가?승인 단계, reason code, audit log, 민감 작업 재인증
경계 밖 데이터는 다시 검증한다

한 번 검증한 데이터라도 다른 경계를 넘으면 다시 검증합니다. 예를 들어 API에서 검증한 업로드 작업도 worker가 파일을 열기 전 크기·MIME·확장자·바이러스 스캔 상태를 다시 확인해야 합니다. 큐, 캐시, 웹훅은 "내부"처럼 보이지만 공격자가 간접적으로 값을 넣을 수 있으면 신뢰 경계 밖입니다.

13.3Tenant Isolation은 필터가 아니라 불변식

멀티테넌트 서비스의 보안 목표는 단순합니다. 한 테넌트의 사용자가 다른 테넌트의 데이터를 읽거나, 바꾸거나, 존재 여부를 추론할 수 없어야 합니다. 이를 컨트롤러마다 where tenant_id = ?를 기억하는 방식으로만 운영하면 언젠가 빠집니다.

계층설계 기준리뷰 질문
식별자모든 테넌트 소유 리소스에 tenant_id를 필수 보유테넌트 없는 row가 생성될 수 있는가?
쿼리조회·수정·삭제 조건에 tenant predicate를 강제id만으로 findById 하는 경로가 있는가?
권한role은 tenant scope 안에서만 의미를 가짐tenant A admin이 tenant B에 admin인가?
캐시캐시 키에 tenant scope 포함user:123 같은 전역 키가 섞이지 않는가?

DB 방어가능하면 RLS, view, repository guard 등 2차 방어애플리케이션 버그 하나로 전 테넌트가 노출되는가?

13.4IDOR 테스트 케이스

IDOR는 "id를 바꿔본다"에서 시작하지만, 실무 테스트는 더 넓어야 합니다. 단건 조회뿐 아니라 목록, 검색, export, bulk update, soft delete, 첨부파일 다운로드, websocket subscription까지 객체 참조가 들어가는 모든 표면을 대상으로 봅니다.

케이스테스트 방법기대 결과
단건 조회user A 토큰으로 user B의 /orders/{id} 요청403 또는 존재 숨김이 필요한 경우 404
수정·삭제본문의 ownerId, tenantId, role을 다른 값으로 변경서버가 본문 값을 무시하고 권한 컨텍스트로 판정
목록·검색filter에 다른 tenant/user id를 직접 지정허용된 scope 밖 row가 0건
파일 다운로드첨부파일 id 또는 signed URL을 다른 계정에서 재사용객체 권한 재검증, URL 만료·scope 확인
Bulk API요청 배열에 내 리소스와 남의 리소스를 섞음부분 성공 정책 명확, unauthorized 항목 차단·기록
Export/Report비동기 리포트 생성 요청에 다른 tenant id 삽입worker 단계에서도 tenant scope 재검증
테스트가 통과해야 하는 기준

컨트롤러에서 403을 한 번 돌려주는 것만으로 충분하지 않습니다. repository, service, worker, export job처럼 같은 데이터에 접근하는 다른 경로에서도 같은 정책이 적용되어야 합니다. 인가 테스트는 "성공 케이스"보다 "다른 사용자의 자원을 섞었을 때 실패하는 케이스"가 더 중요합니다.

13.5Audit Log Design

감사 로그는 디버그 로그가 아닙니다. 침해 대응, 내부자 악용 조사, 고객 문의, 규제 대응에서 행위의 증거가 되는 별도 데이터 모델입니다. 그래서 읽기 좋게 남기는 것보다 변조 어렵고, 검색 가능하고, 민감정보를 과하게 담지 않는 것이 중요합니다.

필드기준주의
actoruser id, admin id, service account, impersonation 여부이메일만 남기면 변경·재사용 추적이 약함
actionuser.role.changed, invoice.refunded처럼 안정적 이벤트명자유문장 로그는 집계·alert가 어려움
resourcetype, id, tenant id, owner id테넌트 scope 누락 시 조사 불가
before/after민감 필드는 마스킹한 변경 diff비밀번호, token, secret, 전체 PII를 남기지 않음
contextrequest id, ip, user-agent, auth method, reason code프록시 환경의 실제 client ip 해석 기준 필요
integrityappend-only 저장, 보존 기간, 관리자 삭제 제한관리자가 흔적을 지울 수 있으면 감사가 아님
AUDIT EVENT SHAPE {
  "event": "tenant.member.role_changed",
  "actor": {"type":"user","id":"u_123","tenant":"t_1"},
  "resource": {"type":"member","id":"m_456","tenant":"t_1"},
  "change": {"role":{"from":"viewer","to":"admin"}},
  "context": {"request_id":"req_...","ip":"203.0.113.10","reason":"support_ticket"}
}
※ secret, raw token, password, full card number는 감사 로그에도 저장하지 않는다

13.6File Upload와 Webhook 위험

파일 업로드와 웹훅은 공격자가 서버 안쪽으로 데이터를 밀어 넣는 대표 경로입니다. 둘 다 "사용자가 보낸 입력"보다 위험할 수 있습니다. 파일은 저장소·백신·썸네일러·다운로드 경로를 지나고, 웹훅은 외부 서비스 이름을 달고 내부 상태 변경을 트리거하기 때문입니다.

표면주요 위험실무 방어
파일 업로드웹셸, polyglot 파일, MIME 속임, zip bomb, 악성 이미지 parser exploit크기 제한, 확장자·MIME allowlist, magic byte 확인, AV scan, 이미지 재인코딩
파일 저장경로 조작, 공개 버킷, 원본 파일명 XSS, tenant 간 파일 섞임서버 생성 파일명, private bucket, metadata escape, tenant prefix 분리
파일 다운로드signed URL 재사용, 권한 변경 후 접근 유지, content sniffing짧은 만료, 다운로드 직전 인가, Content-Disposition, nosniff
Webhook 수신서명 없는 위조 이벤트, replay, 이벤트 순서 뒤집힘, SSRF callbackHMAC 서명 검증, timestamp window, idempotency, event id 저장, allowlist
Webhook 처리외부 payload를 내부 권한으로 상태 변경상태 전이 검증, 서버 측 조회로 사실 확인, 실패 큐·재시도 제한
Webhook은 인증된 API처럼 다룬다

웹훅 endpoint가 공개 URL이라는 점을 잊지 마세요. provider 이름이 path에 들어있어도 인증이 아닙니다. 서명 검증, timestamp 검증, replay 방지, idempotency를 통과한 뒤에도 "이 이벤트가 현재 리소스 상태에서 가능한 전이인가"를 서버가 다시 판단해야 합니다.

13.7Admin Abuse Cases

관리자 기능은 "권한이 높으니 괜찮다"가 아니라 권한이 높아서 더 엄격해야 하는 표면입니다. 외부 공격자만 위협 행위자가 아닙니다. 실수한 운영자, 과도한 권한을 가진 CS 담당자, 탈취된 관리자 계정, 악의적 내부자도 모델에 포함해야 합니다.

남용 케이스위험통제
사용자 impersonation고객 계정으로 임의 작업, 부인 가능성읽기 전용 기본값, 명시적 사유, 배너 표시, 모든 행위 audit
권한 상승·역할 변경동료 또는 본인에게 과도 권한 부여self-grant 금지, dual approval, step-up auth, break-glass 절차
PII 조회·export대량 개인정보 유출필드 마스킹, 목적 제한, export 승인, anomaly alert
결제 취소·환불금전 손실, 고객 분쟁금액 threshold, 2인 승인, ledger 불변 기록
감사 로그 삭제침해 흔적 제거append-only 저장, 별도 권한, 보존 정책, 삭제 대신 redaction workflow

13.8Security Review Rubric

리뷰는 "보안 문제 없어 보임"이 아니라 구체적인 질문에 답하는 과정이어야 합니다. 아래 rubric은 PR 설명이나 설계 문서에 붙여도 됩니다. 답이 "해당 없음"이면 왜 해당 없는지까지 짧게 남기는 것이 좋습니다.

영역리뷰 질문통과 기준
AuthN/AuthZ새 endpoint/job/subscription이 인증과 객체 단위 인가를 모두 수행하는가?권한 정책 위치와 negative test가 명확함
Tenant Isolation모든 query, cache key, export, background job에 tenant scope가 있는가?id-only 접근 경로가 없음
Input/Output외부 입력이 schema validation을 통과하고 출력 컨텍스트에 맞게 escape되는가?allowlist, binding, escaping이 코드로 확인됨
Secrets/PIIsecret, token, PII가 코드·로그·URL·analytics에 남지 않는가?redaction과 보관 기간이 정의됨
State Change중요 변경에 CSRF/replay/idempotency/동시성 방어가 있는가?중복 요청과 순서 뒤집힘이 안전함
Files/Webhooks업로드·웹훅이 서명, 크기, 타입, 재시도, replay를 다루는가?비동기 처리 단계에서도 재검증함
Admin관리자 기능에 최소 권한, 사유, 재인증, 감사 로그가 있는가?내부자·계정 탈취 시 피해가 제한됨
Observability침해나 남용을 나중에 발견할 수 있는 이벤트가 남는가?audit event와 alert query가 설계됨
리뷰의 결론은 액션이어야 한다

위협 모델링의 산출물은 멋진 다이어그램이 아니라 코드와 운영 통제입니다. 예를 들어 "tenant isolation 위험"을 찾았다면 repository guard, RLS, negative test, audit event 중 무엇을 이번 변경에 넣을지 결정해야 합니다. 남은 위험은 "known risk"로 기록하고 owner와 만료일을 둡니다. 보안 리뷰가 결정으로 이어지지 않으면 회의록일 뿐입니다.
