Network
Cheat Sheet

연결 장애를 DNS, 경로, 포트, TLS, HTTP 단계로 좁히기 위한 빠른 참조입니다. 한 번에 추측하지 않고 계층별로 확인합니다.

COMPILED : 2026-06 · FORMAT : QUICK REFERENCE
NW-CS-01

개요

네트워크 디버깅은 source, destination, name, route, policy, protocol 순서로 좁히면 빠릅니다. ping 성공은 서비스 성공을 의미하지 않습니다.

증상먼저 확인할 것
도메인이 안 열림DNS record, TTL, authoritative response
포트 연결 실패route table, security group, firewall, listening port
TLS 오류certificate chain, SNI, hostname, expiration
HTTP 오류status code, header, proxy, upstream log

NW-CS-03

판단 기준

선택기준
Public subnet인터넷에서 직접 들어와야 하고 public route가 필요함
Private subnet외부 직접 접근을 막고 NAT나 내부 경로만 허용함
NATprivate 자원이 외부로 나가야 하지만 외부에서 들어오면 안 됨

NW-CS-04

자주 나는 실수

  • DNS가 맞는지 확인하지 않고 애플리케이션 문제로 단정한다.
  • outbound는 열었지만 return path나 inbound rule을 확인하지 않는다.
  • TLS 인증서의 CN/SAN과 실제 접속 hostname이 다른 문제를 놓친다.
  • 보안 그룹과 NACL의 stateful/stateless 차이를 혼동한다.
  • VPN tunnel up을 서비스 통신 성공과 같은 의미로 본다.
NW-CS-05

면접/리뷰용 한 줄 답변

요청 흐름request path
브라우저 요청은 DNS, TCP, TLS, HTTP, proxy, application 순서로 지나가며 각 단계의 실패 신호가 다릅니다.
방화벽policy
네트워크 정책은 출발지, 목적지, 포트, 방향, 상태 추적 여부를 함께 봐야 합니다.
디버깅debug
연결 장애는 추측보다 계층별 증거로 좁혀야 하며, ping 하나로 서비스 정상 여부를 판단할 수 없습니다.
