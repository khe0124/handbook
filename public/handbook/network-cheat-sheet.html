NW-CS-01

핵심 명령·스니펫

DNS 해석

dig +short example.com          # A record만 간결하게
dig example.com A               # A record + TTL + 권위 응답
dig example.com CNAME           # CNAME chain 확인
dig +short example.com @1.1.1.1 # 특정 resolver로 질의 (내부/외부 비교)
dig +trace example.com          # root부터 authoritative까지 추적
nslookup example.com            # 빠른 확인 (레거시, 출력은 dig가 정확)

연결·HTTP

nc -vz example.com 443          # TCP 도달성만 빠르게 (succeeded/refused/timeout)
curl -v https://example.com     # DNS→connect→TLS→HTTP 전 흐름 한 번에
curl -vkI https://example.com/  # 헤더만 (-k: 인증서 무시, -I: HEAD)
curl -w 'dns:%{time_namelookup} conn:%{time_connect} tls:%{time_appconnect} total:%{time_total}\n' -o /dev/null -s https://example.com
                                # 단계별 소요 시간 분해 (병목 위치 특정)

리슨 소켓·경로

ss -tlnp                        # 서버에서 listening TCP 소켓 (bind address:port)
ss -tnp state established        # 현재 맺힌 연결
traceroute example.com          # hop별 경로 추정 (방화벽/ICMP로 불완전)
mtr example.com                 # traceroute + ping 연속 (packet loss/hop 관찰)

TLS·패킷

openssl s_client -connect example.com:443 -servername example.com
                                # SNI 지정 handshake, chain/SAN/만료 확인
openssl s_client -connect example.com:443 -servername example.com 2>&1 | openssl x509 -noout -dates
                                # 인증서 만료일만 추출
sudo tcpdump -ni any port 443   # 443 패킷 실시간 (인터페이스 any, 이름 해석 끔)
명령용도주의
dig +short example.comA record 간결 확인캐시된 값일 수 있음, TTL은 안 보임
nc -vz host 443TCP 도달성HTTP 이전 계층만, 앱 정상 여부 아님
curl -v https://host전 계층 흐름어느 단계에서 멈추는지 로그를 읽어라
ss -tlnplisten 소켓root 아니면 프로세스명 안 보일 수 있음
openssl s_client -connect host:443 -servername hostTLS handshake-servername 빼면 SNI 미전송으로 다른 인증서
주의

tcpdump는 sudo 필요, 트래픽에 PII/토큰이 캡처될 수 있으니 파일 저장·공유 주의. traceroute/mtr는 방화벽·ICMP rate limit로 hop이 *로 비어도 정상일 수 있어 경로 단정 금지.

NW-CS-03

자주 터지는 것 · 즉시 확인

신호왜 위험즉시 확인
ping 성공만으로 정상 판단ICMP 성공과 TCP/TLS/HTTP 성공은 별개nc -vz host port, curl -v, 서버 access log
return path/방화벽 미확인outbound만 열고 응답 경로·inbound 규칙을 안 봄양쪽 firewall, route table, stateless 장비 양방향 rule
DNS 캐시/TTL 간과변경했는데 캐시된 옛 record로 접속dig host @resolver로 내부·외부 비교, TTL 확인
TLS SNI/체인 누락SNI 미전송·중간 인증서 누락으로 handshake 실패openssl s_client -connect host:443 -servername host
출발지·목적지·포트 미특정"열었다"만으로는 검증 불가, 장애 시간 증가source/dest/port/direction 4개를 명령에 박아 재현
localhost bind0.0.0.0 대신 127.0.0.1 bind로 외부 연결 거부ss -tlnp로 bind address 확인
NW-CS-04

장애 증거 1카드

L3-L7 TRIAGE1. DNS — dig +short host @resolver로 이름 해석 확인 (내부/외부 비교)
2. TCP — nc -vz host port로 도달성 확인
3. TLS — openssl s_client -connect host:443 -servername host로 chain/SAN 확인
4. HTTP — curl -v https://host/path로 status/redirect/header 확인
5. 서버 — ss -tlnp로 bind, access log/target health 대조
순서질문증거
DNS기대 IP로 해석되나, resolver별로 같나dig record + TTL, 내부/외부 일치
TCP포트까지 패킷이 닿나nc -vz succeeded (아니면 refused/timeout 구분)
TLS인증서 신뢰·hostname 일치하나openssl chain 검증, SAN에 host 포함, 미만료
HTTP앱이 응답하나, 어느 계층 오류인가curl -v status code, request id, upstream log
중단 기준

출발지·목적지·포트·방향을 특정하기 전엔 "네트워크 정상"으로 판단 금지. 한 계층에서 증거가 확정되면 다음 계층으로 내려가고, 상위 계층 수정 전 하위 계층 증거를 먼저 확보한다.
