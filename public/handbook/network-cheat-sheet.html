Network
Cheat Sheet

연결 장애를 DNS, 경로, 포트, TLS, HTTP 단계로 좁히기 위한 빠른 참조입니다. 한 번에 추측하지 않고 계층별로 확인합니다.

COMPILED : 2026-06 · FORMAT : QUICK REFERENCE
NW-CS-01

개요

네트워크 디버깅은 source, destination, name, route, policy, protocol 순서로 좁히면 빠릅니다. ping 성공은 서비스 성공을 의미하지 않습니다.

증상먼저 확인할 것분리 기준
도메인이 안 열림DNS record, TTL, authoritative response이름이 틀린 문제인지 IP 이후 문제인지 분리
포트 연결 실패route table, security group, firewall, listening port클라이언트에서 못 나가는지 서버가 안 듣는지 확인
TLS 오류certificate chain, SNI, hostname, expiration인증서 신뢰 문제인지 앱 HTTP 문제인지 분리
HTTP 오류status code, header, proxy, upstream log네트워크는 됐고 앱/프록시 계층 문제로 이동
간헐 장애DNS cache, connection pool, load balancer target health특정 AZ, 특정 upstream, 특정 클라이언트인지 비교

NW-CS-04

자주 나는 실수

  • DNS가 맞는지 확인하지 않고 애플리케이션 문제로 단정한다. 같은 이름이 내부/외부에서 다르게 해석될 수 있다.
  • outbound는 열었지만 return path나 inbound rule을 확인하지 않는다. stateless 장비에서는 양방향 규칙이 필요하다.
  • TLS 인증서의 CN/SAN과 실제 접속 hostname이 다른 문제를 놓친다. IP로 접속하면 인증서 검증이 실패할 수 있다.
  • 보안 그룹과 NACL의 stateful/stateless 차이를 혼동한다. ephemeral port 때문에 한쪽만 열어서는 실패할 수 있다.
  • VPN tunnel up을 서비스 통신 성공과 같은 의미로 본다. routing, DNS, firewall, app bind까지 별도 확인해야 한다.
  • proxy나 load balancer timeout을 앱 timeout보다 짧게 둔다. 요청은 처리됐는데 클라이언트는 실패로 보고 재시도할 수 있다.
NW-CS-05

면접/리뷰용 한 줄 답변

요청 흐름request path
브라우저 요청은 DNS, TCP, TLS, HTTP, proxy, application 순서로 지나가며 각 단계의 실패 신호가 다릅니다.
방화벽policy
네트워크 정책은 출발지, 목적지, 포트, 방향, 상태 추적 여부를 함께 봐야 합니다.
디버깅debug
연결 장애는 추측보다 계층별 증거로 좁혀야 하며, ping 하나로 서비스 정상 여부를 판단할 수 없습니다.
CIDRaddressing
CIDR은 IP 범위의 언어라서 subnet, route, firewall rule을 설계할 때 가장 먼저 맞아야 합니다.
TLScertificate
TLS 문제는 인증서 체인, 만료, hostname, SNI를 나눠 보면 애플리케이션 오류와 빠르게 분리됩니다.

NACL/stateless 장비 변경응답 ephemeral port가 막혀 한 방향만 성공할 수 있음inbound/outbound 양방향 rule

NW-CS-08

Failure Playbook

L3-L7 TRIAGE1. dig host로 이름 해석 확인
2. nc -zv host port로 TCP 도달성 확인
3. curl -vk https://host로 TLS/HTTP 확인
4. 서버에서 ss -tlnp로 listen 확인
5. LB/proxy/access log와 target health 비교
6. route table, firewall, return path를 양쪽에서 대조
운영 원칙

네트워크 장애는 "열었다"가 아니라 어떤 출발지에서 어떤 목적지의 어떤 포트를 어떤 방향으로 검증했는지로 기록합니다. 증거 없는 네트워크 정상 판단은 장애 시간을 늘립니다.

NW-CS-09

연결 진단 카드

네트워크 치트시트는 계층을 건너뛰지 않는 데 씁니다. 항상 source, destination, port, protocol, time을 기록합니다.

증상명령정상 신호다음 행동
DNS 의심dig host +short, dig host @resolver기대 record와 TTL이 resolver별로 일치내부/외부 resolver, CNAME chain, split horizon 확인
TCP 연결 실패nc -vz host 443succeeded 또는 openroute, firewall, security group, server listen 순서로 확인
TLS 실패openssl s_client -servername host -connect host:443chain 검증 가능, SAN에 hostname 포함인증서 chain, SNI, 만료, LB listener 확인
HTTP 오류curl -vkI https://host/pathstatus, redirect, header, request id 확인 가능proxy/LB/access log와 upstream app log 대조
서버 bind 의심ss -tlnp앱이 기대 address:port에서 listenlocalhost bind, container port, host firewall 확인
