CI 파이프라인

CI의 목적은 main에 통합되는 변경이 빠르게 검증되도록 만드는 것입니다. 좋은 CI는 실패를 숨기지 않고, 실패 원인을 빠르게 좁히며, 배포 가능한 아티팩트를 일관되게 만듭니다.

CI-01

CI의 목적

CI는 테스트 자동화 그 자체가 아니라 통합 비용을 낮추는 피드백 시스템입니다.

목적설명
빠른 피드백오류를 PR 단계에서 발견
통합 안정성main branch를 배포 가능 상태로 유지
재현 가능한 빌드개발자 PC 차이를 줄임
품질 기준 강제lint, typecheck, test, scan을 gate로 사용
CI-02

최소 구성

MINIMUM CI STAGES checkout → install dependencies → lint/typecheck → test → build → upload artifact
Stage실패 시 의미
Installlockfile, registry, package manager 문제
Lint정적 규칙 위반
Typecheck컴파일 전 타입 계약 위반
Test행동 회귀
Build산출물 생성 실패
Artifact배포 후보 저장 실패
PR 게이트

CI 결과는 branch protection과 연결되어야 의미가 있습니다. main/release branch에는 필수 check, 최소 review 수, stale review dismissal, linear history 같은 규칙을 걸고, 관리자 예외는 감사 로그가 남는 break-glass 절차로 제한합니다.

CI-03

테스트 단계

테스트목적CI 배치
Unit작은 로직 검증모든 PR
IntegrationDB, queue, API 경계 검증PR 또는 nightly
E2E사용자 핵심 흐름 검증merge 전 또는 staging
Contract서비스 간 API 계약 검증consumer/provider 변경 시
Security scan취약한 의존성·이미지 탐지PR + release
느린 CI의 함정

CI가 너무 느리면 개발자는 결과를 기다리지 않고 다음 변경을 쌓습니다. 빠른 테스트를 앞에 두고, 느린 테스트는 병렬화하거나 필요한 경로에서만 실행합니다.

CI-04

의존성과 캐시

의존성 설치는 CI 시간과 재현성에 큰 영향을 줍니다. lockfile과 package manager 버전을 고정해야 합니다.

항목원칙
LockfilePR에서 diff를 리뷰하고 CI는 lockfile 기준 설치
Package manager버전을 고정해 개발자와 CI 차이를 줄임
Cache keylockfile hash를 포함해 잘못된 캐시 사용 방지
Private registry토큰 권한과 scope registry 설정 검토
CI 인증

장기 access token을 CI secret에 넣는 방식은 회수와 범위 통제가 어렵습니다. 가능한 경우 GitHub Actions, GitLab CI, cloud provider 사이에 OIDC federation을 사용해 짧은 수명의 임시 자격 증명을 발급하고, job별 권한을 최소화합니다.

CI-05

실패 처리

Fail fast빠른 중단
명확한 실패가 나오면 뒤 stage를 생략해 피드백 시간을 줄입니다.
Flaky test불안정 테스트
재시도로 숨기기보다 격리하고 원인을 추적합니다. flaky는 신뢰를 무너뜨립니다.
Required check필수 게이트
main merge 전에 반드시 통과해야 할 check를 branch rule로 고정합니다.
Artifact retention증거 보존
테스트 리포트, screenshot, trace, coverage, build log를 일정 기간 보존합니다.
