웹접근성
핸드북

접근성은 특정 사용자만을 위한 부가 기능이 아니라 브라우저와 보조기술이 이해할 수 있는 UI 계약입니다. 올바른 HTML은 접근성, 테스트 안정성, SEO, 유지보수성을 동시에 개선합니다.

STANDARD : WCAG 사고방식 · 실무 컴포넌트 기준
A11Y-01

접근성은 UI의 계약

화면에 보이는 것과 DOM이 말하는 의미가 다르면 UI는 거짓말을 합니다. 접근성의 출발점은 시각 표현이 아니라 의미 구조입니다.

품질나쁜 기준좋은 기준
탐색마우스로만 가능키보드만으로 모든 기능 접근
의미div와 span으로 전부 구현button, a, nav, main, form 등 의미 요소 사용
상태색으로만 알림텍스트, aria, focus, live region으로 전달
오류빨간 테두리만 표시오류 문구와 필드를 연결하고 수정 방법 제공
PRODUCT QUALITY

접근성은 법적 요구이기 전에 제품 품질입니다. 키보드 탐색, 명확한 label, 충분한 contrast는 장애 사용자뿐 아니라 빠른 작업자, 모바일 사용자, 임시 장애 상황의 사용자에게도 직접적인 개선입니다.

A11Y-02

Semantic HTML

가장 좋은 ARIA는 필요 없는 ARIA입니다. 먼저 올바른 HTML 요소를 선택합니다.

상황사용피함
액션 실행<button type="button"><div onClick>
페이지 이동<a href>button으로 route 이동
주요 영역header, nav, main, footer의미 없는 wrapper 중첩
제목 구조순서 있는 h1~h6크기 때문에 heading 건너뛰기

Accessible name 설계

요소좋은 이름나쁜 이름비고
아이콘 버튼aria-label="검색", aria-label="닫기"아이콘만 있고 이름 없음툴팁은 보조 설명일 뿐 accessible name을 대체하지 않음
입력 필드보이는 label과 input 연결placeholder만 사용placeholder는 입력하면 사라지고 이름으로 충분하지 않음
링크대상이 드러나는 텍스트여기를 클릭, 자세히링크 목록만 읽어도 목적지가 이해되어야 함
상태 배지텍스트와 색을 함께 사용초록색/빨간색만 표시색에 의존하지 않고 의미가 전달되어야 함
A11Y-03

키보드와 focus

키보드 사용자는 focus를 통해 UI의 현재 위치를 파악합니다. focus가 사라지거나 예측 불가능하게 이동하면 UI 맥락이 끊깁니다.

컴포넌트키보드 기준focus 기준
ModalEsc 닫기, Tab trap열릴 때 첫 actionable 또는 제목으로 이동, 닫히면 트리거로 복귀
MenuArrow navigation, Enter/Space 선택열린 메뉴 안에서 현재 항목 표시
TabsArrow로 tab 이동선택된 tab과 panel 연결
Toast키보드 작업 방해 금지중요 상태는 live region 고려
패턴구현 기준주의
Dialogrole="dialog" 또는 native <dialog>, aria-modal, aria-labelledby, 배경 inert, focus trap/restore닫힌 뒤 focus를 원래 트리거로 돌려보내야 함
Menu buttonaria-haspopup, aria-expanded, roving tabindex 또는 active descendant단순 사이트 내비게이션에 menu role을 남용하지 않음
Tabsrole="tablist/tab/tabpanel", aria-selected, aria-controls, keyboard arrow 처리일반 링크 목록을 tabs로 꾸미지 않음
Disclosurebutton + aria-expanded + 연결된 panel상태가 시각적으로만 드러나지 않게 함
KEYBOARD WALKTHROUGH1. 주소창에서 Tab을 눌러 첫 focus가 skip link 또는 주요 내비게이션으로 가는지 본다.
2. 핵심 작업을 Tab, Shift+Tab, Enter, Space, Esc, Arrow만으로 완료한다.
3. 모달·메뉴·드롭다운을 닫은 뒤 focus가 원래 트리거로 돌아오는지 확인한다.
4. focus가 화면 밖, 숨김 요소, disabled처럼 보이는 요소에 머물지 않는지 본다.
FOCUS RING

outline을 지우는 것은 접근성 회귀입니다. 디자인에 맞춰 focus style을 새로 정의할 수는 있지만, 키보드 사용자가 현재 위치를 알 수 있어야 합니다.

A11Y-04

ARIA 사용 기준

ARIA는 의미를 보강하는 도구이지 잘못된 HTML을 고치는 만능 패치가 아닙니다.

원칙설명
가능하면 native 사용button을 만들 수 있으면 role="button"을 만들지 않음
role은 행동까지 책임role만 붙이면 키보드 동작은 자동으로 생기지 않음
이름을 제공아이콘 버튼에는 aria-label 필요
관계를 연결aria-controls, aria-describedby로 상태/오류 연결

ARIA 실패 모드

패턴문제수정
role="button"만 붙인 divEnter/Space, disabled, form 동작이 자동으로 생기지 않음가능하면 native button 사용
aria-hidden="true"를 부모에 적용내부 focus 가능한 요소가 보조기술에서 사라짐숨김 영역 안에 focusable 요소가 남지 않게 처리
동적 메시지를 시각적으로만 표시스크린 리더 사용자가 저장 성공·실패를 알 수 없음중요 상태는 적절한 live region으로 전달
복잡한 combobox를 직접 구현role, active descendant, 키보드, announcement가 빠지기 쉬움검증된 headless 컴포넌트 또는 충분한 테스트를 둠
A11Y-05

폼·오류·상태 전달

폼 접근성은 label, 설명, 오류, 필수 여부, 제출 상태를 사용자가 이해할 수 있게 연결하는 일입니다.

FORM ACCESSIBILITY CHECKlabel과 input은 htmlFor/id로 연결
오류 문구는 aria-describedby로 연결
오류 상태는 aria-invalid로 표시
제출 실패 후 첫 오류 필드로 focus 이동
필수값은 시각 표시와 텍스트 설명을 함께 제공
상황구현 기준놓치기 쉬운 점
필수 입력label 또는 설명에 필수 여부를 텍스트로 제공별표만 표시하면 의미가 전달되지 않음
필드 오류aria-invalid와 aria-describedby로 오류 문구 연결오류 문구 id가 매 렌더마다 바뀌면 연결이 깨짐
서버 오류필드 오류와 전역 오류를 분리모든 오류를 상단 toast로만 보여주면 수정 위치를 잃음
제출 중버튼 pending 상태와 중복 제출 방지disabled만 두면 이유를 모를 수 있어 상태 텍스트도 고려
제출 실패첫 오류 필드 또는 오류 요약으로 focus 이동화면 위쪽 오류를 보지 못한 채 같은 입력을 반복
ERROR UX

오류는 "틀렸습니다"가 아니라 "무엇을 어떻게 고쳐야 하는지"를 알려야 합니다. 서버 오류와 클라이언트 검증 오류도 구분합니다.

A11Y-06

컴포넌트별 접근성 패턴

접근성 품질은 페이지 단위보다 컴포넌트 단위에서 반복적으로 무너집니다. 공통 컴포넌트는 의미, 키보드, focus, 상태 전달을 API 계약으로 가져야 합니다.

컴포넌트필수 계약회귀 신호
Dialog이름 제공, 배경 inert, focus trap, Esc 닫기, 닫힌 뒤 트리거로 focus 복귀Tab이 배경으로 빠지거나 닫힌 뒤 focus가 body로 감
Dropdown/Menu열림 상태 전달, Arrow 이동, Enter/Space 선택, 외부 클릭·Esc 닫기마우스로만 선택 가능하거나 menu role을 내비게이션에 남용
Tabstablist/tab/tabpanel 관계, 선택 상태, Arrow 이동, panel 이름 연결선택된 탭과 패널이 보조기술에서 연결되지 않음
Combobox입력값, 목록 열림, active option, 선택 결과를 명확히 전달검색 결과 개수와 현재 항목을 알 수 없음
Toast작업을 가로막지 않고 중요한 상태는 live region으로 전달오류가 사라져 사용자가 복구 방법을 잃음
Data grid표 의미, 헤더 연결, 정렬 상태, 행 선택 상태, keyboard strategy 정의시각적으로 표지만 DOM은 div 목록이고 셀 맥락을 잃음
Tooltip보조 설명으로만 사용, hover와 focus 모두 지원필수 정보를 tooltip 안에만 숨김
HEADLESS COMPONENT RISK

headless 컴포넌트는 스타일 자유도가 높지만 접근성 책임이 사라지는 것은 아닙니다. 호출자가 label, description, focus target, disabled reason을 빠뜨릴 수 없게 API를 설계해야 합니다.

A11Y-07

검증 루틴

접근성은 자동 검사만으로 충분하지 않습니다. 패턴을 설계한 뒤 자동화와 수동 키보드 검사를 함께 둡니다.

도구잡는 것못 잡는 것
Lighthouse기본 label, contrast, landmark 일부실제 키보드 흐름의 자연스러움
axe규칙 기반 접근성 위반제품 맥락의 설명 품질
키보드 테스트Tab 순서, focus trap, hidden focus스크린 리더 발화 품질 일부
스크린 리더실제 보조기술 경험모든 브라우저 조합 자동 커버
수동 검증기준
Keyboard only핵심 플로우를 Tab/Shift+Tab/Enter/Space/Esc/Arrow로 완료
Screen readerSafari + VoiceOver, Chrome/Firefox + NVDA 조합 중 최소 주요 플로우 확인
Mobile assistive techiOS VoiceOver 또는 Android TalkBack으로 주요 화면 탐색
Display settingszoom 200%, reduced motion, 고대비 모드, text spacing에서 레이아웃 유지
Touch target작은 화면에서 주요 터치 대상 크기와 간격 확인
PR CHECK

새 상호작용 컴포넌트는 keyboard-only 플로우, focus visible, label/name, 오류 메시지 연결, 모바일 화면 겹침 여부를 PR 체크리스트에 포함합니다. 자동 axe/Lighthouse 통과만으로 완료 처리하지 않습니다.

Accessible Name이름
보조기술이 요소를 부를 때 사용하는 이름입니다. 버튼 텍스트나 aria-label에서 옵니다.
Live Region상태 알림
동적으로 바뀐 메시지를 스크린 리더에게 알리는 영역입니다.
A11Y-08

접근성 회귀 방지

접근성은 한 번 감사해서 끝나는 품질이 아닙니다. 공통 컴포넌트와 핵심 플로우에 자동 검사, 수동 검사, 리뷰 질문을 반복 가능하게 넣어야 합니다.

단계확인 항목증거
설계상호작용 컴포넌트의 role, keyboard, focus restore 정의컴포넌트 명세 또는 Storybook notes
구현label, accessible name, describedby, live region 연결DOM 캡처, 테스트 쿼리 이름
테스트keyboard-only 플로우, 오류 상태, zoom 200%, reduced motion 확인체크리스트와 실패 재현 링크
릴리즈새 공통 컴포넌트가 기존 화면의 focus 순서를 깨지 않는지 확인시각 회귀와 접근성 회귀 결과
ACCESSIBILITY PR QUESTIONS1. 이 화면의 주요 작업은 마우스 없이 끝나는가?
2. 보이는 label과 보조기술의 이름이 같은 의미인가?
3. 오류, 저장 성공, 로딩 상태가 시각 정보 외 경로로도 전달되는가?
4. 모달·메뉴·페이지 전환 후 focus 위치가 사용자의 맥락을 보존하는가?
5. 자동 검사 통과 외에 수동으로 확인한 핵심 플로우가 있는가?
