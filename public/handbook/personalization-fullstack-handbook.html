FS-PER-02

답변 템플릿

Q. 풀스택 개발자로 볼 수 있는 근거는 무엇인가요?

30초서버 구현을 주도한 백엔드 개발자라고 말하기보다는, 프론트엔드에서 서버와 맞닿는 경계를 많이 다뤄온 개발자라고 설명하는 것이 정확합니다. 인증, 권한, API request/response, pagination/filter, 파일 업로드/다운로드, 금액 정밀도, 운영자용 관리 화면을 구현하며 서버 계약과 데이터 흐름을 계속 맞췄습니다.

Q. API 계약을 어떻게 다뤘나요?

90초목록 화면에서는 query parameter, filter, pagination, sorting, reset 조건을 명확히 맞췄고, 상세/상태 변경 화면에서는 버튼 활성 조건과 API 응답 전 disable 처리로 중복 요청을 막았습니다. 인증이 필요한 요청은 axios interceptor로 token 부착과 만료 처리를 공통화했고, 오류 응답은 사용자가 복구 가능한 위치에 메시지를 노출했습니다.

Q. 데이터 정합성을 어떻게 보나요?

90초금융성 데이터에서는 JavaScript number만 믿지 않고 BigNumber.js로 소수 정밀도와 금액 표기를 처리했습니다. UI에서는 중복 클릭, 잘못된 상태 전이, 필터 초기화 누락, undefined 값 표시 같은 문제가 실제 정합성 신뢰를 깨뜨리기 때문에 서버 정합성과 별개로 프론트에서도 방어 로직을 둡니다.

강점 전환 문장

풀스택 답변의 핵심은 프론트 경험을 그대로 나열하지 않고, 경험을 서버 협업 역량으로 전환해 말하는 것입니다.

프론트 경험서버 협업 역량으로 전환
Data Grid와 검색 필터조회 API의 pagination/filter, 정렬 조건, 인덱스 후보를 함께 생각할 수 있습니다.
인증 만료와 route guardJWT/refresh token, 401/403 구분, 서버 최종 권한 검증 필요성을 이해합니다.
파일 업로드/다운로드multipart, presigned URL, 비동기 job, 실패 복구 UX를 API 계약과 연결합니다.
오류 상태 UI서버 error code, retryability, 사용자 복구 가능 메시지의 필요성을 압니다.
ANSWER“제 강점은 서버를 혼자 다 만들었다는 것이 아니라, 프론트에서 서버 계약이 깨지는 지점을 많이 겪었다는 점입니다. 그래서 API 응답 구조, 인증 만료, 권한, pagination/filter, 파일 처리, 오류 복구를 백엔드와 같은 언어로 논의할 수 있습니다.”
FS-PER-06

서버 협업 역량

풀스택 면접에서는 프론트 경험을 서버 흉내로 바꾸지 말고, 경험을 서버 협업 역량으로 전환해서 말합니다.

프론트 경험강점 전환 문장
목록·필터·pagination“목록 화면을 만들 때 서버 query pattern과 사용자의 복원 가능한 검색 조건을 함께 봤기 때문에, API 계약을 화면 상태와 분리해서 생각합니다.”
인증 만료 UX“token refresh를 프론트에서 처리하며 인증 만료, 요청 복구, logout 정리를 다뤘고, 서버에서는 revoke와 rotation이 함께 필요하다고 이해합니다.”
대량 파일 처리“업로드와 다운로드는 UI 이벤트가 아니라 비동기 작업, 오류 응답, 재시도, 진행 상태가 필요한 서버 협업 기능으로 봅니다.”
금액·수량 표기“프론트 표기 정확성을 다루며 서버 계산 결과를 그대로 보여주는 것만으로는 부족하고, decimal precision과 통화/시간대 규칙을 계약에 포함해야 한다는 점을 배웠습니다.”
FS-PER-07

압박 질문 연결

풀스택 포지션에서는 인프라·분산·클라우드·시스템 설계 질문을 “직접 전부 운영했다”로 과장하지 말고, 프론트 경험이 닿은 API 계약과 운영 경계에서 정확히 연결합니다.

SYSTEM DESIGN PRESSURE DRILL숫자 압박 질문은 DAU, QPS, connection, cache hit ratio, queue lag, worker 수로 가정을 쪼개고, 프론트 경험은 pagination/filter, progress UI, 캐시 가능 조회와 쓰기 경로 분리로 연결합니다.
DISTRIBUTED CONSISTENCY SENIOR DRILL분산 정합성 질문은 중복 클릭 방지 수준에서 끝내지 않고 idempotency key, outbox/inbox, processed table, reconciliation이 왜 필요한지까지 연결해 답합니다.
압박 질문안전한 포지셔닝말할 증거
Kubernetes나 Cloud 운영을 얼마나 이해하나요?Service/Ingress, HPA/VPA, PDB, node drain, network policy는 직접 운영 범위를 구분해 설명하고, 배포 중 readiness·rollback·환경변수·로그 확인 경험과 연결합니다.배포 스크립트, 환경별 설정, 장애 화면, 릴리즈 확인, 운영자 콘솔 이슈 대응
IAM, KMS, secret 관리는 어떻게 보나요?서버 권한을 직접 설계했다고 과장하지 않고, token/refresh/RBAC 경험에서 IAM trust/resource policy, KMS, secret rotation 필요성을 학습해 연결합니다.axios interceptor, 401/403 처리, 권한별 메뉴, secret을 코드에 두지 않은 배포 관행
분산 정합성 질문을 받으면?결제·권한·상태 변경처럼 사용자 신뢰가 큰 경로는 idempotency, 중복 클릭 방지, processed 상태, reconciliation 관점으로 답하고 exactly-once 오해를 피합니다.중복 요청 방지, 상태 전이 버튼 제어, API error 재시도 정책, 사용자에게 pending/failed 표시
시스템 설계 숫자 압박은?DAU→QPS→cache hit→DB connection→queue lag→worker 수 순서로 가정을 놓고, 정확한 숫자보다 병목 후보와 질문 목록을 명확히 합니다.목록 API pagination/filter, 캐시 가능한 조회와 쓰기 구분, 대량 업로드 job/progress 경험
프로젝트 증거를 요구하면?PROJECT EVIDENCE QUALITY RUBRIC 기준으로 PR/ADR/metric/before-after/장애기록/QA ticket/사용자 피드백 중 실제로 가진 증거만 말합니다.PR, QA ticket, before-after 화면, 사용자 피드백, 버그 재현 steps, 수정 전후 로그
ANSWER“풀스택 질문에 답할 때는 제가 직접 운영한 범위와 학습으로 설명할 범위를 나눕니다. 직접 경험은 API 계약, 인증 만료, 권한별 UI, 대량 입력, 오류 복구이고, 여기서 이어지는 서버·인프라 주제는 어떤 지표와 정책을 확인해야 하는지까지 말할 수 있게 준비했습니다.”
